* POO Object Family Syntax Scenario

This scenario records the repair from repeated hand-written POO object
accessors and projections to a generated object-family declaration.

- =input.ss= keeps the old shape: one predicate, one accessor per slot, and one
  projection hand-written for every object family.
- =expected.ss= keeps the same object shape and public projection result, but
  declares the repeated family through =defpoo-object-family=.

The expected form is the project style for large stable POO object families:
slot ownership stays explicit, runtime payloads remain boundary projections, and
mechanical accessor/projection duplication is generated by Gerbil syntax.

The performance gate constructs and projects 1,000 POO objects with integer
references.  Integer references deliberately keep string and symbol creation
outside the measured path: the gate is evidence about native POO construction,
generated accessors, and boundary projection rather than text allocation or a
runtime adapter.  It is run through =gxpkg env gxtest= and must remain below
the scenario time and RSS limits declared in =benchmark.ss=.

On 2026-07-09, three independent 1,000-object runs measured 2.326-5.595 ms
for the dynamic benchmark receipt.  Process RSS ranged from 136,527,872 to
136,626,176 bytes, a 96 KiB spread.  The benchmark fixture still contains
legacy =observed*= contract fields; they are harness acceptance inputs, not
dynamic phase telemetry.  Consumers must use receipt =elapsedMs= for the
measured duration.  Per-phase construction, selection, and projection telemetry
belongs in the benchmark harness receipt before it can be claimed as measured
evidence; macro expansion requires a separate build-stage measurement.

The harness now emits one dynamic =testing-phase= receipt for each runtime
phase.  The object-family scenario verifies =object-construction=,
=boundary-projection=, and =object-selection= receipts, each with its own
=elapsedMs=.  Compile-time macro expansion remains a build-stage metric; it is
not represented as a fabricated runtime phase.

The test output is intentionally compact: one top-level gate projection shows
feature, rule, dynamic elapsed time, threshold, and status; one line per
runtime phase shows its phase symbol and dynamic elapsed time.  Static fixture
metadata is retained for harness contract validation but is not printed as a
performance report.
