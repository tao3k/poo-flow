* 10.07 Functional Flow Kernel

** 10.07.00 Purpose

This document records the implementation route for making the Funflow kernel
feel like functional programming in Gerbil Scheme while keeping the project
boundary unchanged:

- Gerbil Scheme owns flow composition, POO category metadata, policy
  selection, and lightweight validation runs.
- Rust or another runtime owns Docker, CAS, subprocesses, durable state,
  parallel scheduling, and artifact materialization.

This is not a DSL track. It does not introduce an alist or record language for
workflow authoring. Gerbil Scheme remains the host language, and Gerbil POO is
the semantic object system.

** 10.07.10 Reference Evidence

The local Gerbil POO checkout at =.data/gerbil-poo/fun.ss= defines functional
programming concepts as type objects:

- =Category.= has =Arrow=, =domain=, =codomain=, =compose=, and =identity=.
- =Functor.= and =ParametricFunctor.= have =.ap= and =.map= style slots.
- =Identity= supplies identity mapping and bind-like behavior as object
  semantics.

The Funflow checkout at =.data/funflow/funflow/src/Funflow/Flow.hs= exposes the
user-facing kernel as:

- =Flow input output= and =ExtendedFlow=;
- =pureFlow= and =ioFlow= smart constructors;
- =toFlow= for task-family lifting;
- =returnFlow= as =returnA=;
- public re-export of Arrow composition such as =>>>=.

The Scheme implementation should therefore model the same core ideas as
Category/Arrow operations over =flow= declarations, not as a separate config
syntax.

** 10.07.20 Code Landing

The current landing keeps =src/core/flow.ss= as the owner because that file
already owns the stable =flow= declaration structure and descriptor
classification.

Implemented functional kernel surface:

- =flow-arr= lifts a Scheme procedure into a pure flow.
- =flow-identity= aliases the unit/identity flow.
- =flow-map= maps over a flow's output by composing a pure arrow after it.
- =flow-fanout= exposes =flow-branch= as Arrow-style fanout.
- =flow-first= applies a flow to the first element of a pair-shaped value.
- =flow-second= applies a flow to the second element of a pair-shaped value.
- =flow-category-prototype= and =default-flow-category= expose these
  operations as POO metadata slots.
- =flow-category-compose=, =flow-category-identity=, =flow-category-arr=,
  =flow-category-map=, and =flow-category-fanout= call the selected POO
  category operations.
- =src/core/flow-syntax.ss= owns hygienic Gerbil authoring macros:
  =defpoo-flow-arr=, =defpoo-flow-identity=, =defpoo-flow-compose=,
  =defpoo-flow-map=, =defpoo-flow-fanout=, =defpoo-flow-first=, and
  =defpoo-flow-second=.

The stable =flow= struct is unchanged. The POO category object is metadata and
operation selection over existing declarations, not a replacement runtime
representation.

** 10.07.30 Boundary Rules

1. Category/Arrow composition stays in =src/core/flow.ss=.
2. Planning stays in =src/core/plan.ss= and strategy-owned planner selection.
3. Local execution proofs stay in =src/core/runner.ss=.
4. Runtime handoff stays behind runtime adapters and extension descriptors.
5. Authoring conveniences may use hygienic Gerbil macros later, but those
   macros must expand to the functional kernel and POO objects. They must not
   define a new alist/record DSL.
6. The landed =defpoo-flow-*= macros define Gerbil bindings over existing
   combinators. Flow names become quoted symbolic names; procedures, contracts,
   and source flows remain caller expressions.

** 10.07.120 User-Facing Functional DAG Authoring Landing
This slice exposes a compact Gerbil macro surface for users who want to write a
Funflow-style DAG through functional composition.
- =defpoo-flow-dag= defines a DAG flow as Arrow fanout over two already-defined
  flow expressions.
- =defpoo-flow-dag-artifacts= names the report-only DAG receipt and Marlin
  runtime manifest for that flow.
- Both macros expand to public =flow= and =plan= combinators. They do not
  introduce an alist/record workflow language.
Example:
#+begin_src scheme
(defpoo-flow-arr parse-input
  parse-input
  (lambda (x) x)
  'text
  'tokens)

(defpoo-flow-arr score-input
  score-input
  (lambda (x) x)
  'text
  'score)

(defpoo-flow-dag analyze-input
  analyze-input
  parse-input
  score-input)

(defpoo-flow-dag-artifacts analyze-input-receipt
  analyze-input-manifest
  analyze-input
  'analyze-input-request)
#+end_src
Boundary result:
- Users author with normal Scheme bindings and procedures.
- The DAG receipt and manifest are named evidence artifacts.
- Runtime work remains outside Scheme and is owned by =marlin-agent-core=.
