#+title: RFC 139: Feature Composition into the Existing Domain Case Algebra
#+status: implemented

* Decision

Phase 3D introduces =FeatureDomainCaseAssembly= as an immutable POO receipt that
connects =FeatureCompositionPlan= to the existing Domain Case algebra.

It does not define a second Domain Case representation.  Component validation,
parent ordering, type inheritance, slot/method contract closure, role overlay,
projection catalog construction, canonical identity, and caching remain owned by
=src/module-system/domain-case.ss= and =poo-flow-domain-case-close=.

* Existing owner contract

The Gerbil module owner already exports POO-native values and operations for:

- =poo-flow-case-component=;
- case type, slot, and method contracts;
- case projections;
- component composition;
- Domain Case close, instantiate, and project operations;
- canonical descriptors and module-owned closure caches.

A Case component binds these distinct layers in one POO value:

1. component identity and version;
2. role prototype;
3. type contract;
4. slot contracts;
5. method contracts;
6. projections;
7. optional parent component identities;
8. optional policy and strategy algebras.

Contract and type information is therefore attached to the component, but it is
not an inheritance method and is not replaced by policy or strategy metadata.

* Assembly contract

=feature-domain-case-assembly= receives four explicit POO/control-plane values:

1. a module-owned =poo-flow-domain-case-cache=;
2. the semantic Domain Case id;
3. the Domain Case schema version;
4. an accepted =FeatureCompositionPlan=.

It sends the plan's resolver-ordered components directly to
=poo-flow-domain-case-close=.  The resulting constant-slot assembly receipt
contains:

- the composition plan and ordered components;
- the existing Domain Case closure receipt;
- accepted/rejected status, canonical key, diagnostics, and closed Domain Case;
- policy and strategy contributions as separate ordered values;
- adapter requirements and projection contributions as separate ordered values.

The declarative module-level spelling is:

#+begin_src scheme
(defpoo-feature-domain-case-assembly agent-case-assembly
  (using-cache agent-case-cache)
  (domain-case-id 'agent-case)
  (domain-case-version 1)
  (from-plan agent-composition-plan))
#+end_src

=require-feature-domain-case-assembly= converts a rejected receipt into an
explicit build failure carrying the Domain Case diagnostics.

* Layer boundary

The assembler never injects plan-level policy or strategy contributions into
component inheritance.  It never interprets adapter requirements and never runs
projection or runtime code.  Those values remain visible on the assembly receipt
for later, explicitly owned binders.

This keeps four facts connected without collapsing their semantics:

- object/component composition is closed by the Domain Case role algebra;
- inheritance is expressed by parent component and parent type identities;
- contracts and types are validated by their dedicated Domain Case owners;
- policy and strategy remain explicit composition inputs for a later binder.

* Ordering invariant

The Domain Case owner rejects a parent that appears after its child.  Therefore
every component inheritance edge must also be represented by a corresponding
Feature dependency edge.  =FeatureCompositionPlan= then presents parents before
dependants using the existing graph resolver.

If the Feature edge is absent, the bundle may still be graph-valid, but Domain
Case closure returns structured =missing-or-forward-parent-component= and
=missing-or-forward-parent-type= diagnostics.  The assembly preserves those
diagnostics unchanged instead of silently sorting or repairing the input.

* Cache and reuse

The cache is explicit and module-owned.  Reassembling the same case id, schema
version, and canonical components returns the same closed Domain Case object with
=cache-hit?= equal to =#t=.  No global Feature or expander registry is introduced.

This is the Gerbil single-instantiation reuse boundary: declarations construct
pure values once, while the Domain Case owner reuses canonical closure results.

* Qualification gates

The implemented test suite proves:

- Feature dependency order closes component and type inheritance successfully;
- Case component POO objects are passed to the existing owner without conversion;
- contract/type closure remains separate from policy, strategy, and adapters;
- repeated assembly reuses the module-owned Domain Case cache by object identity;
- missing Feature dependency preserves both structured forward-parent errors;
- a 256-feature component/type inheritance chain closes under the Scheme memory
  guard;
- rejected assembly and malformed macro declarations exit non-zero;
- source and isolated AOT execution require no stale local artifact.

* Next mainline stage

Phase 3E defines explicit policy/strategy binding over an accepted assembly
receipt.  It must reuse the existing algebra owners and produce a new immutable
receipt.  Cedar evaluation, adapter I/O, runtime execution, and evidence emission
remain outside the pure assembly layer.
