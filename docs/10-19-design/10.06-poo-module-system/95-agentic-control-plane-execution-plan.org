* Status
Status: *AC-03 five-facet Bundle and cross-facet validation implemented*

RFC 90–94 are accepted for implementation planning. This document turns their P0–P6 migration into bounded work packages with explicit owners, deliverables, validation, stop conditions, and commit boundaries.

The current checkout contains a large pre-existing staged tranche. No implementation package may start in that checkout until AC-00 either establishes an isolated worktree or confirms that the staged tranche has been completed and the implementation paths are clean.

* Purpose
This plan is operational rather than aspirational.

It answers:

- what starts first;
- which existing owners are reused;
- which proposed owners require confirmation before creation;
- which scenario proves each layer;
- which commands validate each slice;
- what blocks progression;
- when a compatibility path must be removed;
- which changes belong in one commit.

It does not authorize a monolithic rewrite. Each package must leave a reviewable, testable repository state.

* Operating Principles
1. Public authoring remains 100% POO-native and functional.
2. The Bundle is compiler output, not a new user-written record or alist DSL.
3. C3 resolves definitions; facet-specific algebra decides semantics.
4. The implementation begins with a read-only shadow Bundle and one vertical slice.
5. No external effect is executed twice for comparison.
6. Existing ABI and Python surfaces are classified before they are changed.
7. Python production execution must migrate to CFFI out-of-line API mode; =ctypes= may remain only as an explicitly diagnostic ABI probe after cutover.
8. Downstream Rust, TypeScript, and other runtimes are not implemented in this plan.
9. Every package has a real scenario, focused tests, and a machine-checkable receipt.
10. Performance is measured on the actual C/Gerbil/Python hot path with complete data.
11. A new canonical route retires its obsolete public predecessor after the cutover gate.
12. Never run =gxpkg uninstall= in this workspace.

* Implementation Workspace Gate
Before AC-01 changes code, AC-00 must select exactly one safe mode:

** Isolated Worktree
- Create a dedicated worktree/branch from the intended source revision.

** Completed Current Tranche
- The user finishes or commits the current staged tranche.
- The implementation paths named by AC-01 are clean.
- A baseline receipt records the resulting source revision.

Implementation MUST NOT begin by resetting, unstaging, overwriting, or absorbing the existing staged changes.

* AC-01 — First Vertical-Slice Contract
** Purpose
Freeze one bounded scenario before creating the Bundle kernel.

** Scenario
A parent agent delegates one restricted tool capability to a child. The child receives filtered context and follows start, effect, cancel, checkpoint, and restore protocol paths. Allow, deny, timeout, retry, stale epoch, and evidence outcomes are explicit.

** Tasks
- Express the scenario through the current POO/session authoring route.
- Identify parent/child Agent, Role, Principal, Capability, Resource, Artifact, Protocol, transition, effect, and obligation identities.
- Define input and expected canonical facts without choosing a final binary encoding.
- Add negative fixtures for authority widening, context leakage, missing evidence, illegal transition, stale epoch, and replay.
- Bind every fixture to focused tests in 10.06.96.

** Deliverables
- scenario fixture and expected semantic facts;
- typed negative fixture catalog;
- no runtime behavior change.

** Exit Criteria
- Every entity and invariant has one unambiguous meaning.
- Existing authoring remains POO-native.
- The scenario can be evaluated without executing an external effect.

* AC-02 — Bundle Kernel
** Purpose
Implement deterministic identity, manifest, lineage, canonicalization, and digest before full facet semantics.

** Tasks
- Add typed stable identities and kind validation.
- Add Bundle schema/version, compiler provenance, epoch, parent, facet-digest slots, and Bundle digest.
- Implement deterministic ordering and canonical encoding.
- Reject ambient time, path, process, random, and map-order influence unless declared semantic input.
- Produce typed validation and digest receipts.

** Focused Tests
- identical input produces byte-identical output;
- kind mismatch, duplicate identity, missing parent, non-increasing epoch, and digest mutation fail;
- historical identity remains interpretable under its original epoch.

** Exit Criteria
- The vertical slice produces a deterministic manifest and shared identity universe.
- No partial invalid Bundle is activatable.
- The Bundle remains internal and shadow-only.

* AC-03 — Five Facets And Cross-Facet Validation
** Purpose
Implement Organization, Authority, Context, Protocol, and Evidence as typed facets with distinct algebras.

** Tasks
- Implement responsibility/ownership conflict rules for Organization.
- Implement issuer provenance, attenuation, lease, delegation, and revocation for Authority.
- Implement information labels, provenance, restriction intersection, redaction, lease, and explicit declassification for Context.
- Implement lifecycle/refinement, effect intent, cancel, recovery, checkpoint, and restore for Protocol.
- Implement monotonic obligation accumulation and typed discharge for Evidence.
- Implement every RFC 91 cross-facet invariant.
- Prove through tests that C3 cannot authorize, declassify, delete evidence, or hide incompatible transitions.

** Exit Criteria
- Positive and negative facet fixtures pass.
- Every external effect closes Protocol–Authority–Evidence.
- Every context transfer closes Context–Protocol–Evidence.
- Ambiguity is a typed compile error, not precedence behavior.

* AC-04 — Read-Only Shadow Projection
** Purpose
Compare the Bundle path with current behavior before changing runtime execution.

** Tasks
- Project the Bundle into current graph, policy, lifecycle, context, composition, and receipt shapes.
- Run current and Bundle-derived paths against the same scenario.
- Record each difference as expected correction, current defect, new defect, or unresolved semantic question.
- Keep all external effects disabled in the comparison lane.
- Prevent dual writes and dual durable truths.

** Exit Criteria
- Every difference is classified and reviewed.
- No unexplained difference is accepted for compatibility.
- Shadow mode has a named removal owner and cannot execute effects.

* AC-05 — C ABI Control Lane And C Harness
** Purpose
Introduce negotiated, typed lifecycle semantics without optimizing prematurely.

** Tasks
- Map the current graph ABI to the accepted Bundle/runtime contract.
- Decide, from evidence, whether the next surface is a compatible extension or an ABI-major successor.
- Add version, schema, capability, ownership, concurrency, batch, and payload negotiation.
- Add typed opaque handles, lifecycle operations, errors, cancellation, checkpoint, restore, and close disposition.
- Preserve the current standalone probe as a regression lane where its semantics remain valid.
- Build a pure C harness that does not depend on private Gerbil symbols.

** Exit Criteria
- Installed headers/libraries are sufficient for the C harness.
- Public types expose no =___SCMOBJ=, Scheme heap address, Python object, or private generated symbol.
- Wrong-kind, stale, cross-instance, double-release, and incompatible-version cases fail safely.

* AC-06 — Batched Hot Lane And Bulk Buffer Lane
** Purpose
Close the performance architecture with evidence from Gerbil/Gambit and gerbil-mysql.

** Tasks
- Repair the gerbil-mysql owner-item evidence lane and inspect exact result, row, string, prepared-buffer, and release owners.
- Benchmark direct primitive, Gerbil wrapper, batch, callback, string, byte-buffer, owned, borrowed, caller-buffer, and arena paths.
- Add =poll_batch= and =submit_batch= semantics.
- Add contiguous headers, offsets, lengths, capacities, payload arenas, and partial-acceptance results.
- Add explicit owned, borrowed, caller-provided, and negotiated-arena lifetimes.
- Prohibit Scheme-heap zero-copy without verified pinning.
- Freeze bytes covered by authorization during their lease.
- Differentially validate native fast views against canonical packets.

** Performance Matrix
- batches: =1, 8, 32, 128, 1024=;
- payloads: empty, 1 KiB, 64 KiB, 1 MiB;
- paths: allow, deny, success, error, timeout, cancel, restore;
- metrics: crossings, allocations, copied bytes, GC time, p50, p99, throughput, RSS, seal overhead.

** Exit Criteria
- Batch size reduces crossings as designed.
- No hidden per-item string conversion remains on the production hot path.
- Every optimized path passes baseline conformance and fault tests.
- ABI freeze remains blocked if owner evidence or performance receipts are incomplete.

* AC-07 — Python Production CFFI Runtime
** Purpose
Prove the public ABI can support a complete production runtime without Python becoming a semantic owner.

** Tasks
- Add an out-of-line CFFI API-mode binding generated from the installed header.
- Port the production hot path from =ctypes= to CFFI.
- Retain =ctypes= only as an explicitly diagnostic ABI smoke path if it remains useful.
- Ensure graph plans, checkpoints, stores, context, messages/tools, subgraphs, builder, and prebuilt loops remain runtime-local strategies and do not become competing organization semantics.
- Preserve C ABI plan digest and receipt binding through execution results.
- Validate wheels without Gerbil using published projection artifacts.
- Add long-run, memory-pressure, crash/restart, lease, and exception tests.

** Exit Criteria
- Python uses only installed headers and libraries.
- Required runtime behavior has no private Gerbil bypass.
- C and Python traces agree after declared nondeterminism normalization.
- Production benchmark and wheel gates pass.

* AC-08 — Cedar, Tokens, And Evidence Roots

* Stop Conditions
Stop the current package and report the exact blocker when:

- owner evidence points to a different semantic or contract owner;
- the implementation workspace overlaps unknown staged changes;
- the vertical-slice semantics are ambiguous;
- shadow paths produce an unexplained difference;
- a runtime requires a private Gerbil or Python-only bypass;
- a claimed zero-copy path lacks verified lifetime stability;
- gerbil-mysql owner evidence remains unavailable at the ABI-freeze gate;
- performance regresses without a reviewed explanation;
- Cedar policy, proof, Bundle, or runtime digests do not bind;
- an effect may have occurred but the design lacks an =indeterminate= recovery path;
- a proof claim exceeds its modeled assumptions;
- a compatibility alias would create a second public semantic route.

Do not work around these conditions with raw source fallback, silent compatibility, skipped evidence, or weaker claims.

* First Advancement Tranche
The first implementation request after this plan is accepted is AC-00 only.

It performs:

1. isolated-worktree or clean-tranche selection;
2. current ABI and Python-surface classification;
3. exact owner/test matrix confirmation;
4. current build/probe/test/benchmark receipt capture;
5. gerbil-mysql provider-index recovery plan.

It does not add Bundle code, change runtime behavior, or freeze ABI layout.
