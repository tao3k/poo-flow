#+subtitle: Recovering prototype reuse without changing wire or receipt contracts
#+author: POO Flow Contributors

#+status: Second bounded migration implemented

* Problem

The AC-01--AC-11 implementation used POO objects extensively but reused little
prototype behavior.  Manifest, version, readiness, evidence, and receipt owners
often behaved as independent record families with repeated slot validation.

This design restores POO as a composition system while preserving functional
control-plane projection and immutable boundary values.

Each capability is a parameterized POO parent prototype and owns a disjoint
slot set.  Domain objects use multiple parents through C3-backed =.mix= via the
project's =poo-core-role-object= macro.  Existing accessors continue to resolve
slots through =.ref=, so callers do not learn where a slot is inherited.

* Safety rules

- Capability slot ownership MUST be disjoint.  Duplicate capability identities
  and cross-capability slot collisions fail before object construction.
- Validators are named per capability and aggregated explicitly.  No five-way
  competing =validate= method override is introduced.
- Local domain slots may intentionally override an inherited default, and the
  override must be covered by a focused test.
- C3 is a reuse and dispatch mechanism, not an authorization boundary.
- Prototype metadata and parent order never enter JSON, ABI, digest, or receipt
  projections.

* First migration

=ReleaseVersionMatrix= now composes Versioned, OwnerBound, and DecisionState.
=CutoverReadinessInput= composes Versioned, RevisionBound, EvidenceBound, and
DecisionState.  Their public constructors, predicates, accessors, schemas,
diagnostics, JSON key order, and fixed safety flags remain unchanged.

Wire vectors, hot buffers, cryptographic tokens, evidence snapshots, and Bundle
facet algebras are deliberately outside this inheritance tranche.

* Second migration

=ReleaseAssuranceManifest= now composes Versioned and RevisionBound only.  The
existing =release.source-revision= projection remains the canonical receipt and
digest source; validation requires it to equal the inherited
=source-revision=.  This deliberate duplicate projection preserves the v1
contract while making revision-bound behavior reusable.

OwnerBound is not composed because the manifest does not expose the same
=owner-artifacts= domain.  EvidenceBound is not composed because claims,
evidence references, TCBs, and their ordering/inclusion rules form a stronger
proof-linked contract.  DecisionState is not composed because assurance claim
validity and ABI review are not AC-11 cutover decision/freeze/delete state.
The release, environment, assurance, claim, TCB, evidence, gate, and ABI
subobjects therefore retain their current owners in this tranche.

The normalized form remains an explicit functional projection.  It does not
materialize prototype markers, =schema-id=, or =schema-version=.  The frozen
fixture digest remains
=871e2303fcabbaffb4b3a95dfbafdef4fb65768ad9a5ae4c75e409b20bd73de5=.

* Qualification

Focused tests cover:

- three-parent inherited slot resolution;
- parent-order stability for orthogonal capabilities;
- intentional local override precedence;
- independent instance composition;
- missing-slot rejection;
- duplicate capability and cross-capability slot collision rejection;
- existing release-version and cutover-readiness positive and negative suites;
- inherited manifest schema/revision resolution and instance isolation;
- rejection of a flat manifest without capability parents;
- rejection when inherited and release-projected revisions diverge;
- byte-identical JSON receipts before and after migration.

* Next decision

Review repeated behavior in the release/environment/assurance subobjects only
when a semantic capability contract, rather than a shared field name, is
proven.  Do not migrate proof-linked evidence or Bundle facets merely because
they resemble generic qualification slots; their validation and composition
algebras remain distinct.
