#+title: Scheme / Marlin Runtime Boundary for POO Flow
#+date: 2026-07-05

* Boundary

POO Flow's Scheme layer is not a LangGraph runtime clone and must not grow into
one.  The Scheme layer owns the strategy object model, profile composition,
policy contracts, graph intent, durable intent, proof facts, and user interface
syntax.  Marlin, Rust, or another long-running runtime language owns process
lifetime, async execution, scheduler loops, durable stores, tool IO, sandbox IO,
network IO, and recovery.

This distinction is architectural, not merely implementation detail:

- Scheme/POO Flow should answer: what is allowed, what is composed, what must
  happen before what, what scope is visible, what policy gates must hold, and
  what proof facts can be generated.
- Marlin/runtime should answer: how long-running work is executed, persisted,
  resumed, retried, streamed, cancelled, observed, and isolated in production.

* Why Scheme Does Not Own Runtime

The value of Scheme here is the control plane:

- POO objects provide reusable policy and profile fragments.
- Declarative and procedural macros provide a small user interface over
  reusable object families.
- Functional composition keeps profiles, strategy bundles, and graph facts as
  inspectable values.
- Lean-facing facts provide a proof boundary before a runtime consumes the
  plan.

The value of Marlin or another runtime language is the data plane:

- durable process supervision;
- async task execution;
- persistent checkpoints and stores;
- streaming events;
- sandbox and tool IO;
- subagent lifecycle;
- runtime observability;
- retries, timeouts, cancellation, and recovery.

Conflating these two layers would make Scheme reproduce a production runtime
that already belongs in a long-running language.  The correct interface is a
verified control-plane receipt that a runtime can consume.

* Durable Means Durable Policy Intent Here

When a user asks for durable behavior in POO Flow, the Scheme layer should model
durable policy intent, not implement a full durable runtime.

Examples of Scheme-owned durable facts:

- this composition requires checkpointing;
- this checkpoint is required before human approval;
- this human approval must be resume-safe;
- this tool action must not execute before sandbox scope is established;
- this subagent handoff requires a persisted session identity;
- this recovery path must preserve the policy preconditions.

Examples of runtime-owned durable work:

- writing checkpoint records;
- choosing memory, sqlite, postgres, object storage, or another backend;
- resuming a thread from a checkpoint cursor;
- retrying failed node attempts;
- streaming partial events;
- enforcing wall-clock timeouts;
- shutting down and draining tasks.

* LangGraph Scenario Target

LangGraph is useful as a scenario because it demonstrates production agent
orchestration requirements: graph nodes and edges, conditional routing,
superstep-style coordination, channels, checkpointers, interrupts, streaming,
subgraphs, retries, timeouts, and deployment surfaces.  It is not the semantic
center of POO Flow.

POO Flow must not map its own policy system onto LangGraph as if LangGraph were
the source of truth.  The direction is the opposite: POO Flow's Policy,
Durable, Checkpoint, Graph, Loop, and Composition capabilities should be strong
enough that a LangGraph-like system is only one composition scenario among many.
When LangGraph exposes a missing capability, the gap belongs under =src= as a
general platform capability, not under =user-interface= as a special case.

The Scheme control plane should make the general strategy contract explicit:

#+begin_src latex
\[
\operatorname{SchemePlan}
  = \langle P, G, S, D, A, H, F\rangle
\]
#+end_src

where:

- \(P\) is the reusable POO policy/profile composition;
- \(G\) is the graph or loop topology intent;
- \(S\) is the sandbox and tool scope policy;
- \(D\) is durable checkpoint/store intent;
- \(A\) is agent/subagent lifecycle intent;
- \(H\) is human approval or handoff intent;
- \(F\) is the Lean-facing fact contract.

The runtime receives this plan and must either implement it or reject it with a
typed incompatibility.

#+begin_src latex
\[
\operatorname{RuntimeAccepts}(R, \operatorname{SchemePlan})
  \Rightarrow
  \operatorname{RuntimeObeys}(R, P \land G \land S \land D \land A \land H)
\]
#+end_src

The proof obligation is not "Scheme executes the runtime."  The proof
obligation is "the plan given to the runtime is internally sound, and the
runtime receipt can be checked against the plan."

* Scenario Gap Rule

LangGraph, LangChain, community contributed profiles, production agent flows,
and domain-specific workflow libraries all follow the same rule:

- =src= owns the reusable Graph, Loop, Durable, Checkpoint, Policy, Composition,
  proof, and handoff primitives.
- =user-interface= owns only reusable profile libraries and scenario
  compositions over those primitives.
- Lean owns the mathematical contracts proving that the selected composition is
  internally sound.
- Marlin/runtime owns execution and returns receipts that can be checked against
  the Scheme plan.

If implementing a LangGraph-like scenario requires special runtime semantics in
=user-interface=, the scenario has found a missing =src= capability.  The fix is
to strengthen the platform primitive, then express the scenario as ordinary
composition.

* Proof Boundary

Lean should prove the control-plane facts that Scheme generates:

- branch targets are declared;
- graph or loop progress is bounded or has an explicit handoff;
- a policy cannot start before required preconditions hold;
- sandbox/tool scopes are monotone or explicitly narrowed;
- durable intent exists before human-in-the-loop resume points;
- subagent handoff requires session identity and scope facts;
- profile overrides preserve required contracts;
- generated runtime receipts satisfy the declared plan.

Lean should not prove every implementation detail of Marlin, Rust async IO, or a
third-party persistence library inside the Scheme proof model.  That would be
the wrong proof boundary.  Instead, Lean should model the interface contract:

#+begin_src latex
\[
\operatorname{GeneratedFacts}(scheme)
  \land
  \operatorname{RuntimeReceipt}(runtime)
  \land
  \operatorname{ReceiptMatchesPlan}
  \Rightarrow
  \operatorname{PolicySoundRun}
\]
#+end_src

* Runtime Language Projection
The runtime side must be implemented by a runtime language. Python is the first concrete runtime language in this project because it can host agent graph execution, event streaming, checkpoint/resume behavior, tool/action bindings, and C ABI validation without forcing Scheme to become an executor.

This is the intended direction:
#+begin_src text
POO Flow Scheme control plane
  -> profile composition
  -> graph/loop/policy/proof facts
  -> runtime handoff receipt
  -> Python RuntimeGraphProgram / C ABI
  -> runtime execution receipt
  -> Scheme/Lean receipt check
#+end_src

The same boundary applies to LangGraph, CrewAI, and Pi Agent core style systems. POO Flow should not clone those runtimes in Scheme. Instead, POO Flow should make their core capabilities ordinary composition scenarios, then lower the verified plan into Python or another runtime language.

For example:
- LangChain-like capability maps to model/tool loop policy, prompt/model/parser ordering, middleware/tool constraints, and runtime provider bindings.
- LangGraph-like capability maps to graph routing, long graph execution, stream events, interrupts, and checkpoint/resume in the Python runtime.
- CrewAI-like capability maps to agent/task/crew/flow profiles plus a runtime receipt that executes the selected graph/process plan.
- Pi Agent core-like capability maps to session state, tool calling, agent loop progress, attachment/message state, and provider/tool runtime bindings in the Python runtime.

The proof obligation is:
#+begin_src latex
\[
\operatorname{SoundSchemePlan}(p)
  \land
\operatorname{RuntimeLanguageAccepts}(\texttt{python}, p)
  \land
\operatorname{ReceiptMatchesPlan}(r, p)
  \Rightarrow
\operatorname{PolicySoundRun}(r)
\]
#+end_src

This keeps implementation cost low: the Scheme layer composes and proves the policy, while Python implements the operational behavior. If a Python runtime can already execute a long graph, then a LangGraph-like, CrewAI-like, or Pi Agent core-like scenario should be mostly a composition and receipt-lowering problem, not a new Scheme runtime feature.
* Runtime Contract Shape

The Scheme layer should produce a POO-native and proof-friendly runtime handoff
contract.  It should contain at least:

- composition id;
- stage id;
- selected profiles and overrides;
- graph nodes, edges, branch targets, and loop bounds;
- durable checkpoint requirements;
- sandbox/tool scopes;
- subagent lifecycle requirements;
- human approval and resume requirements;
- expected handoff target;
- generated Lean fact module or fact rows;
- observability labels for runtime receipts.

The runtime should return a receipt with:

- accepted or rejected plan id;
- executed node trace or superstep trace;
- checkpoint writes;
- sandbox/tool grants actually used;
- subagent handoffs;
- human approvals and resumes;
- retries/timeouts/errors;
- final handoff or finish state;
- receipt facts that can be checked against the original plan.

* Design Rule

If a feature requires timers, sockets, long-running tasks, durable stores,
stream multiplexing, or process supervision, it belongs in Marlin/runtime.

If a feature defines policy reuse, profile override, graph shape, lifecycle
preconditions, proof facts, or runtime handoff contracts, it belongs in
Scheme/POO Flow.

This is the line that keeps the Scheme layer elegant and keeps the production
runtime in the language that is built to run for a long time.
