* 10.06.33 POO C3 Graph Algorithm Layer
** 10.06.33.10 Decision
=poo-flow= should implement a graph algorithm layer, but it must be placed at
the right boundary.

The layer is not "C3 schedules workflows". C3 remains inheritance
linearization:

#+begin_src text
POO objects express roles, policies, descriptors, and relation facts.
C3 gives deterministic role/slot precedence.
Validators check whether the composed object graph is legal.
Graph algorithms analyze relations among composed objects.
Marlin or another runtime executes validated runtime handoff descriptors.
#+end_src

The missing layer is a module-owned graph analysis library over composed POO
facts. It is directly useful for loop engines, governors, workflow planning,
sandbox capability reasoning, human-audit placement, and agent-engine
diagnostics.

** 10.06.33.20 Why This Fits POO Better Than A Raw DSL
Gerbil POO gives us an object graph naturally:

- module objects expose categories, features, profiles, cases, validators, and
  projections;
- workflow objects expose tasks, dependencies, receipts, runtime manifests, and
  result projections;
- sandbox objects expose capabilities, mounts, resources, and backend intent;
- loop objects expose patterns, autonomy level, budgets, maker/checker split,
  governors, and human gates;
- session objects expose lineage, placement, memory intent, transform receipts,
  and runtime handoff facts.

The graph layer reads these as typed nodes and edges. It must not require users
to write an extra graph DSL. Users continue writing module/profile/case POO
declarations. Module projections produce graph facts for analysis.

C3 helps before graph analysis. A node can be a composed POO object whose final
slots are C3-linearized. The graph algorithm should analyze the final validated
facts, not pre-composition source fragments.

** 10.06.33.30 Boundary Rule
Do not weaken the existing C3 boundary.

Existing C3 notes correctly say that C3 does not decide which task is ready to
execute, which loop wakes next, or which external action should be retried.
That remains true.

The graph layer answers different questions:

- Is there a cycle in a declared dependency graph?
- Which nodes are reachable from the selected entry profile or loop case?
- Which human gates dominate write-capable actions?
- Which sandbox capabilities can flow into a task or loop?
- Which tasks are blocked by missing checkpoint/session durability policy?
- Which profiles or modules are unused, shadowed, or disconnected?
- Which agent reviewer/governor path covers a maker action?
- Which graph cut separates read-only analysis from write-capable runtime
  handoff?

These are analysis and validation questions. Runtime scheduling can consume the
result later, but Scheme should still emit receipts and manifests with
=runtime-executed = #f=.

** 10.06.33.40 Module Ownership
The graph layer is a first-class internal directory:

#+begin_src text
src/graph/types.ss
src/graph/algorithms.ss
#+end_src

=src/graph/= owns reusable pure graph facts and algorithms: POO graph nodes,
POO graph edges, graph objects, analysis receipts, adjacency, reachability,
dependency cones, cycle paths, and topological order.

Module ownership still follows the Doom Emacs style:

- =workflow= / =funflow= owns workflow dependency graph projections and
  topological readiness facts.
- =loop-engine= owns recurring loop graph facts, pattern dependency analysis,
  governor path analysis, and human gate placement.
- =sandbox= owns capability graph and resource/mount flow analysis.
- =session= owns lineage graph, transform graph, checkpoint/session-durability
  reachability, and memory-intent graph facts.

Public use comes through module projections. Do not expose a top-level
=(graph ...)= user DSL. The user interface remains:

#+begin_src scheme
(use-module workflow +loop-engine +human-audit)
(use-module sandbox +nono +doctor)
#+end_src

Graph analysis is a projection from module-owned objects into the shared
=src/graph/= substrate.

** 10.06.33.50 First Algorithm Set
The first implementation is pure, small, and reusable.

Core representation:

- graph node id: symbol, string, number, or stable structured id;
- graph node: POO object with =id=, optional payload, metadata, and
  =runtime-executed = #f=;
- graph edge: POO object with =from=, =to=, =edge-kind=, metadata, and
  =runtime-executed = #f=;
- graph analysis receipt: POO object with node count, edge count, roots,
  terminals, reachability, dependency cone, topological order, cycle path,
  diagnostics, and =runtime-executed = #f=.

Implemented first:

1. adjacency projection: outgoing and incoming maps;
2. root and terminal detection;
3. reachability from one or more roots;
4. dependency cone for "why is this blocked?";
5. cycle detection with cycle path receipts;
6. topological ordering for acyclic graphs.

Next after data model stabilization:

7. dominator-style gate coverage for "does every write path pass human-audit
   or governor?";
8. connected component / strongly connected component diagnostics for loop and
   module fact graphs;
9. cut analysis for read-only side versus write-capable side;
10. path explanation receipts for agents.

** 10.06.33.60 Agent Engine Uses
Graph algorithms are useful to agent engines because agents need explainable
control-plane facts, not only runnable commands.

Concrete uses:

- loop plan selection: choose a loop pattern whose dependency cone is satisfied;
- governor coverage: prove a maker action is judged by a reviewer/governor
  path;
- human-audit placement: prove a human gate dominates write-capable actions;
- sandbox diagnosis: show which mount or capability allows or blocks an action;
- retry diagnosis: find the failed node, affected dependents, and safe re-run
  frontier;
- multi-agent coordination: detect ownership collision paths before two loops
  touch the same branch/resource;
- session continuity: show transform lineage and checkpoint reachability;
- workflow UI: project a compact graph snapshot for agent dashboards.

The important result is not a pretty graph. The important result is a typed,
replayable explanation that an agent can inspect and repair.

** 10.06.33.65 Algorithm Families
C3 belongs to the sequence-producing family of graph algorithms. It starts from
a precedence graph and returns a legal linearization. That is exactly why it
fits Gerbil POO inheritance: the runtime needs one deterministic slot/role
order.

The engine needs more than sequence linearization. It needs several graph
families, each with a different control-plane purpose:

- precedence linearization :: C3/C4-style algorithms for inheritance, role
  precedence, and composed slot visibility.
- acyclic dependency planning :: topological order, cycle rejection,
  root/terminal detection, and ready frontier facts for finite workflows.
- reachability and dependency cones :: explain why a task, loop action, session
  transform, or runtime handoff is reachable, blocked, or affected by a
  failure.
- strongly connected components :: identify recurring loop cycles, invalid
  workflow cycles, and mutually dependent policy fragments.
- dominator and gate coverage :: prove that write-capable paths pass through
  governor, checker, budget, or human-audit gates.
- cut and boundary analysis :: separate read-only analysis subgraphs from
  write-capable runtime handoff subgraphs.
- capability-flow analysis :: explain how sandbox permissions, mounts,
  resources, network rules, and store/session capabilities reach an action.
- conflict and collision graphs :: detect multi-agent ownership collisions over
  branches, files, sessions, tickets, connectors, or budgets.
- assignment and matching :: later-stage optimization for assigning reviewer,
  verifier, governor, or sandbox profiles to actions.
- weighted path and risk scoring :: later-stage diagnostics for choosing safer
  loop paths under cost, budget, latency, and confidence weights.

The first Gerbil implementation focuses on unweighted deterministic algorithms
that feed validators and receipts. Weighted optimization, matching, and
scheduling can be layered later or delegated to Marlin when execution state
matters.

** 10.06.33.70 POO Edge Objects
Edges should be POO-native where they carry policy.

Example shape:

#+begin_src scheme
(poo-flow-graph-edge 'build 'test 'dependency
                     '((composition-style . kleisli)))
#+end_src

Future policy-bearing edges may be defined with POO inheritance so that a
human-audit edge, governor edge, or sandbox-capability edge can inherit
metadata and monotonic validation requirements through POO/C3. The graph
algorithm sees the final composed edge facts.

This keeps policy extension POO-native and avoids raw alist patch languages.

** 10.06.33.80 Safety Rules
Graph algorithms can support safety, but they are not themselves the safety
boundary.

Safety still belongs to validators:

- denylist edges accumulate by union;
- permission edges may narrow but not expand;
- budget edges may tighten but not loosen;
- human gate edges may be added or escalated but not removed by ordinary task
  policy;
- a write-capable path without a governor/human gate is invalid;
- a resumable human-audit node without checkpoint/session durability is invalid;
- cycles are allowed only in declared loop-pattern graphs, never in finite
  workflow dependency DAGs.

The graph layer should produce diagnostics that make these validator failures
explainable.

** 10.06.33.90 Implementation Milestones
Milestone 1: =src/graph/= shared pure graph utilities.

- Add =src/graph/types.ss= with POO-native graph node, edge, graph, and
  analysis receipt objects.
- Add =src/graph/algorithms.ss= with deterministic pure algorithms: adjacency
  maps, root/terminal detection, reachability, dependency cones, cycle paths,
  and topological order.
- Add a focused test that proves the graph layer emits report-only POO analysis
  with =runtime-executed = #f=.

Milestone 2: workflow and funflow projection.

- Project existing funflow DAG/check-map facts into =src/graph/= nodes and
  edges.
- Replace duplicated root/terminal/topological facts in module-local code with
  calls into =src/graph/algorithms.ss= where that does not weaken ownership.

Milestone 3: loop/session/sandbox projection.

- Add loop governor and human-audit graph facts.
- Add session lineage and durable-policy reachability graph facts.
- Add sandbox capability and mount-flow graph facts.

Milestone 4: safety algorithms.

- Add gate coverage, cut analysis, and SCC diagnostics.
- Keep validators as the hard safety boundary.
