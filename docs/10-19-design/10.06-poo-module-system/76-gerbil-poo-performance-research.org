#+TITLE: 10.06.76 Gerbil POO Performance Research

* Purpose

This note records the research baseline before changing the POO Flow
performance scenarios again. The goal is not to copy the upstream project
harness scenarios, but to use them as a floor and then add scenarios that are
closer to the actual Gerbil POO implementation model.

The local research checkout is =.data/gerbil-poo=. This document uses the
local checkout and ASP owner/symbol evidence as the source of truth. The owner
map exposes the files that matter for performance work: =object.ss=,
=proto.ss=, =mop.ss=, =fun.ss=, =polynomial.ss=, =number.ss=, =fq.ss=, and
small support owners such as =brace.ss= and =io.ss=.

* Performance Interpretation

** Scheme and Gerbil Features

Gerbil POO uses advanced Scheme features for performance boundaries:

- Phase-aware macro code through =begin-syntax=.
- Slot projection macros such as =%with-slots=, =with-slots=, =def-slots=,
  and prefixed variants.
- A meta-object layer with =Class=, =Slot=, =Object=, and
  =MonomorphicObject= identities.
- Prototype composition through =compose-proto= and =compose-proto*=.
- Function-level wrappers through =Functor= and =ParametricFunctor=.

For POO Flow, this means the public API can stay POO-native while the hot path
is compiled into stable helpers at module or scenario setup time. The user
should configure object families and strategy combinators; the runtime should
not rediscover slot structure or validate raw alists inside the loop.

** Algorithmic Lessons

The Gerbil POO owners point at several concrete algorithmic rules:

- Compute slot functions once with =compute-slot-funs!= and reuse them.
- Merge inherited slots once with =merge-super-slots= before entering the hot
  path.
- Apply slot specs at the object-family boundary, not per iteration.
- Compose prototypes in a bounded phase with =compose-proto*= and instantiate
  the result once.
- Prefer monomorphic object shape when the scenario has stable slots.
- Use =slot-lens= and generated accessors for repeated projection/update.
- Treat conversion helpers such as =object<-alist=, =object<-hash=,
  =object<-fun=, and =hash<-object= as boundary adapters only.
- Model algebraic object families with normalized operations, as suggested by
  =Polynomial.=, =.normalize=, =define-type=, and finite-field support.

This is the path to beating the harness scenarios: do fewer semantic decisions
in the loop, and make the loop operate on already projected, already composed,
already checked object families.

* Local Scenario Plan

The next code work should add or refine local POO performance scenarios in this
order.

1. Monomorphic slot projection family
   - Build a fixed object family once.
   - Generate or bind slot accessors once.
   - Measure only stable projection and scalar accumulation.

2. Prototype composition cache family
   - Compose a deeper prototype chain once.
   - Instantiate the composed object once.
   - Measure repeated strategy calls that reuse the composed result.

3. Slot lens update family
   - Bind lens-like readers/writers at setup.
   - Compare repeated read/update without dynamic slot lookup.
   - Keep mutation explicit and local to the benchmarked owner.

4. Slot spec mutation family
   - Exercise =.putslot!= and =.setslot!= through a stable object shape.
   - Keep spec application outside the loop.
   - Track allocation and elapsed time under benchmark gates.

5. Algebraic object family
   - Model polynomial, integer range, or finite-field style objects.
   - Measure normalize/compose-style operations.
   - Keep POO type descriptors stable across iterations.

6. Generated receipt boundary family
   - Use a generated POO receipt object for benchmark results.
   - Materialize to alist/hash only at the outer boundary.
   - Verify receipt contract and benchmark timing in one scenario.

7. Debug instrumentation family
   - Provide a debug-enabled object family and a hot-path family.
   - Ensure debug metadata is available without taxing the hot path.

8. Marlin-style large profile family
   - Use a large config/profile object with fixed projection.
   - Exercise realistic runtime handoff fields.
   - Avoid ad hoc JSON/alist validation in the loop.

* Code Style Rules For The Follow-Up

- Keep public surfaces POO-native and functional-combinator based.
- Do not introduce raw alist DSLs or record DSLs as the normal interface.
- Treat alist/hash/function conversion as boundary adapters.
- Keep scenario support modules small; avoid one giant test support owner that
  causes compile-time churn.
- Use bounded declarative or procedural macros only when repeated object
  families justify generation.
- Prefer fixed slot projection, local scalar loop state, and precomputed helper
  functions in benchmark hot paths.
- Keep benchmark gates strict: total, parse, phase, file, RSS, hot/integration
  classification, and regression budget should all remain visible.
- Do not use =BUILD_CORE= tuning to hide compile-time problems.

* Current Gaps

- ASP can discover =gerbil-poo= owners and symbols, and it can mark symbols
  such as =compute-slot-funs!=, =merge-super-slots=, =$slot-spec=,
  =apply-slot-spec=, =%with-slots=, =def-slots=, =.putslot!=, and
  =.setslot!= as exact identities. However, the recommended query path still
  degrades to file-range selectors, while the current Codex lifecycle requires
  code queries to be routed through the =asp-explore= resident that may only
  return compact receipts. That is a selector/query protocol gap and should not
  be treated as source-body evidence.
- The upstream harness scenarios are broad, but local POO Flow still needs
  stronger direct coverage for =MonomorphicObject=, =slot-lens=,
  =compose-proto*=, and algebraic normalization.
- Existing local scenarios should be audited against compile-time cost. Adding
  scenario coverage is not enough if the build target count or stale-module
  behavior grows unexpectedly.

* Decision

The next implementation should not start from generic list or alist loops. It
should implement POO-native scenario families that compile object shape,
prototype composition, slot projection, lens access, and receipt materialization
into stable helpers, then benchmark the actual hot path.

That gives POO Flow a stronger position than the upstream harness baseline:
the harness proves the policy and benchmarking model, while POO Flow should
prove a reusable strategy engine whose configured object families run through
precomputed POO mechanisms instead of runtime rediscovery.
