* Purpose

This RFC records the proof bridge direction for user-interface policy,
strategy, workflow, and runtime-handoff configuration.

The proof source is a typed projection path:

1. Gerbil/POO builds typed policy and workflow objects.
2. Gerbil projects those objects into a small proof-facing ABI.
3. C exposes that ABI with stable tags, masks, and handles.
4. Lean imports the ABI through =@[extern]= wrappers.
5. Lean decodes ABI facts into typed =inductive= and =structure= values.
6. Lean proves invariants over those typed values.

* Decision

Use a typed C FFI bridge as the integration boundary between Gerbil Scheme and
Lean.

The proof package under =packages/proof= owns the first scaffold:

| Package path | Responsibility |
|--------------+----------------|
| =packages/proof/lean= | Lean typed model, C FFI wrapper, proof theorems |
| =packages/proof/lean/native= | C header and C ABI smoke surface for proof facts |
| =packages/proof/python= | =uv= based Lean artifact generator |

The Python package is a generator and bridge, not a proof kernel. The Lean
package is the proof owner.

* Non-Goals

- Do not prove a full Scheme interpreter.
- Do not let runtime execution receipts substitute for proof receipts.
- Do not expose raw alist patch languages as the normal user interface.
- Do not make Lean reconstruct POO inheritance, macro expansion, or module
  loading.

* Proof Boundary

The first proof boundary is the user-interface loop-engine proof manifest:

| Concern | Typed owner |
|---------+-------------|
| Manifest kind | =ManifestKind= |
| Source | =ManifestSource= |
| Proof owner/checker | =ProofOwner=, =ProofChecker= |
| Runtime owner | =RuntimeOwner= |
| Scheme projection | =SchemeProjection= |
| Proof scope | =ProofScope= |
| Object families | =ObjectFamily= |
| Receipt contracts | =ReceiptContract= |
| Runtime packet contracts | =RuntimePacketContract= |
| Obligations | =ObligationName=, =ObligationClaim=, =ObligationSource= |
| Runtime execution state | =RuntimeExecution= |

=RuntimeExecution= is intentionally typed as an inert-only boundary in the first
model. There is no constructor that represents "Scheme already executed runtime
work".

* First Obligations

The first required obligations are:

| Obligation | Meaning |
|------------+---------|
| =uiConfigWellFormed= | Runtime handoff references are present. |
| =runtimeCommandInert= | Scheme emitted handoff data without runtime execution. |
| =policyStrategyDeterministic= | Policy and strategy projection has stable precedence. |
| =workflowAgreementLinked= | Workflow agreement is carried into the runtime envelope. |
| =sandboxBoundaryLinked= | Sandbox handoff agreement is carried into proof scope. |

These obligations are intentionally small. Scheme normalizes the facts; Lean
checks that the facts satisfy the proof model.

* C FFI Shape

The initial C surface has an explicit header:

#+begin_src c
#include "poo_flow_proof_ffi.h"
#+end_src

That header is the C side of the stable interface. It defines ABI version,
obligation tags, obligation count, tag width, and the required-obligation mask.
The C implementation exports raw accessors for each field and Lean IO wrappers
for the same values:

#+begin_src c
uint32_t poo_flow_proof_abi_version_raw(void);
uint32_t poo_flow_proof_tag_width_raw(void);
uint32_t poo_flow_proof_obligation_count_raw(void);
uint32_t poo_flow_proof_required_obligation_mask_raw(void);
uint32_t poo_flow_proof_ui_config_well_formed_bit_raw(void);
#+end_src

Lean imports those wrappers and decodes the raw ABI payload into typed
=ProofAbi= and =ObligationName= values. This validates the integration mechanism
before binding richer Gerbil-owned C structures.

This follows the Lean 4 FFI pattern:

#+begin_src lean
@[extern "poo_flow_proof_required_obligation_mask"]
opaque cRequiredObligationMask : IO UInt32
#+end_src

* Gerbil Direction

Gerbil should expose a proof-facing projection helper beside existing runtime
handoff projection helpers. The helper should produce a stable C ABI payload.

The Scheme side should expose a source-level interface module, not rely on a
compiler-generated =.ssi= file as the public contract. The current interface
module is:

#+begin_src text
src/module-system/loop-engine-proof-abi.ss
#+end_src

Compiled =.ssi= artifacts may exist, but they are build products. The public
boundary is the explicit =.ss= module and its exported constants/functions.

The intended progression is:

1. Keep existing alist presentation only as an internal projection artifact.
2. Use typed symbolic constants and constructors in a dedicated Scheme module.
3. Add a C ABI header that maps proof tags to stable numeric tags and
   accessors.
4. Bind those accessors from Lean.
5. Decode the ABI payload into Lean typed constructors.
6. Prove obligations over the decoded typed value.

* Python Direction

The =uv= package exists to generate Lean declarations and to coordinate AXLE
checks. It should not validate semantics by itself.

Acceptable Python responsibilities:

- Generate Lean artifacts from typed Scheme/C metadata.
- Run local =lake build=.
- Call AXLE =check= or =verify_proof= when configured.
- Emit diagnostics for missing generated declarations.

Unacceptable Python responsibilities:

- Decide whether a policy strategy is correct.
- Replace Lean theorem checks with schema validation.

* Decision Notes

** 2026-07-02-01: Typed Lean Model

The proof model uses Lean =inductive= and =structure= definitions for manifest
kind, owners, scopes, object families, receipt contracts, runtime packet
contracts, obligation names, obligation claims, and runtime execution state.

** 2026-07-02-02: Structured C FFI Boundary

The first C ABI exports version, tag width, obligation count, the
required-obligation bitmask, and individual obligation bit accessors. Lean
imports that ABI, decodes the raw payload into typed constructors, and checks it
against the required obligation list.

** 2026-07-02-03: Python Is Only A Generator

The =uv= package remains useful for generating Lean files and coordinating AXLE
checks, but it is not part of the trusted proof base.

** 2026-07-02-04: Proof-Case Vector IR
The first proof ABI was too coarse for user-interface cases because a flat
obligation tag list can only prove that names are present. The Scheme boundary
now emits a proof-case vector IR from =src/module-system/loop-engine-proof-abi.ss=:
each obligation has =name=, =claim=, =source=, =domain=, =case-family=,
=evidence-fields=, and =runtime-executed=. Lean should prove policy/strategy
properties over this vector, while C keeps exposing ABI version, tag width,
obligation count, and required-obligation mask. Python remains a generator and
runner only.

* Open Questions

- Which Gerbil-owned C payload should follow the accessor surface: tagged
  struct, stable pointer handle plus accessor functions, or generated dual
  artifacts?
- Should AXLE be used only in CI, or also in local developer commands?
- Which user-interface package should be the first full proof case after
  loop-engine: CI/CD, session policy, sandbox, or durable runtime?

* Acceptance Criteria

- Lean build succeeds on typed proof declarations.
- C FFI smoke checks ABI version, tag width, obligation count, required mask,
  and obligation bit accessors through Lake.
- Gerbil proof projection is built through the source-level helper and remains
  inert.
- User-interface policy obligations are represented by typed Lean values.
- Decision Notes are updated whenever the proof boundary changes.
