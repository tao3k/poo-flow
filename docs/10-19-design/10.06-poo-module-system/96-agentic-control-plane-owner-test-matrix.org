* Purpose

This document assigns semantic, contract, implementation, validation, artifact,
and downstream ownership for the implementation plan in
[[file:95-agentic-control-plane-execution-plan.org][95-agentic-control-plane-execution-plan.org]].  It is a planning baseline, not
a claim that the proposed files already exist.  AC-00 must confirm every current
owner through the repository's parser-owned ASP routes before any new owner is
created.

The matrix keeps three boundaries explicit:

1. POO Flow owns the canonical Gerbil control plane and the stable C ABI.
2. The Python package is the complete production-grade validation runtime for
   that ABI; it does not become a second semantic owner.
3. Rust, TypeScript, and other downstream runtimes are adapters maintained
   outside this implementation tranche.

* Ownership vocabulary

| Role | Responsibility | Non-responsibility |
|-
| Semantic owner | Defines the canonical meaning and invariants of a concept | Foreign-language convenience APIs |
| Contract owner | Defines normalized schemas, projections, compatibility, and validation | Runtime scheduling policy |
| Implementation owner | Executes a contract without redefining it | Silent semantic extensions |
| Validation owner | Proves behavior through scenarios, conformance, fuzzing, or benchmarks | Replacing the semantic owner |
| Artifact owner | Publishes headers, wheels, manifests, evidence, and version metadata | Inventing new semantics during packaging |
| Downstream owner | Adapts the stable ABI into another language or host | Changing POO Flow's canonical contract |

* Scenario and test matrix

** Behavioral scenarios

| Scenario | First required package | Gerbil/control-plane assertion | C ABI assertion | Python assertion | Evidence/proof assertion |
|-
| Parent-child delegation | AC-01 | Capability and context only narrow across the edge | Stable identifiers and explicit result status | Delegated execution matches canonical receipt | Delegation edge is attributable and replayable |
| Context filtering | AC-03 | Child sees only projected context | No implicit host pointer or ambient state crosses ABI | Filtered context is preserved through CFFI | Projection digest binds the observed context |
| Allow and deny effect | AC-03 | Policy result is deterministic for normalized input | Decision and failure are explicit, not sentinel-overloaded | Python cannot bypass the decision lane | Policy input, result, and authority token are linked |
| Cancel, replay, and epoch | AC-05 | State transition rules are canonical | Cancellation and epoch values have versioned representations | Runtime handles retries without duplicating committed effects | Replay identity distinguishes valid replay from evidence fork |
| Checkpoint and restore | AC-05 | Checkpoint semantics preserve bundle identity | Buffers and handles have explicit lifetime rules | Existing checkpoint/store behavior conforms to ABI | Restore receipt binds checkpoint, bundle, and epoch |
| Batch and payload transport | AC-06 | Batching does not change semantics or ordering guarantees | Borrowed/owned buffers, vector calls, and error lanes are measurable | CFFI avoids per-field conversion on hot paths | Batch receipt retains item-level attribution |
| Evidence fork detection | AC-08 | Conflicting successors cannot both extend one canonical head | Evidence identifiers survive boundary transport | Python surfaces the fork as a typed failure | Chain verification rejects the conflicting branch |
| Proof mismatch | AC-09 | Proof status never substitutes for runtime policy | Proof reference and digest are unambiguous | Python reports mismatch without reinterpretation | Lean theorem identity, assumptions, and artifact digest must agree |

* Matrix maintenance rules

1. Update the matrix in the same commit that changes an owner, public contract,
   ABI lane, or required validation scenario.
2. One concept has one semantic owner.  Adapters reference that owner through a
   versioned projection; they do not mirror it as an independent model.
3. A public ABI symbol requires a contract owner, a lifetime rule, a failure
   rule, a C-harness assertion, and a Python conformance assertion.
4. A performance claim requires a reproducible fixture, environment receipt,
   payload distribution, warm/cold distinction, and copy/allocation accounting.
5. A proof claim identifies the theorem, assumptions, checked artifact, toolchain
   version, semantic digest, and failure behavior when any link differs.
6. Removed public routes receive absence tests.  Compatibility aliases are added
   only by an explicit RFC amendment.
7. Downstream Rust, TypeScript, and other adapters remain consumers of the C ABI;
   their design cannot delay correction of a false or inefficient ABI boundary.

* Acceptance criteria for this planning artifact

- Every implementation work package has a named semantic or contract owner.
- Every public boundary has both native C and Python validation responsibility.
- Functional correctness, performance, evidence integrity, and proof linkage
  have separate gates.
- The current ctypes runtime is treated as migration input; production validation
  targets out-of-line CFFI API mode after ABI freeze.
- No dual-runtime semantic ownership is introduced.
- AC-00 remains the only ready implementation tranche.
