#+TITLE: 10.06.35 Next Loop Engine Delivery Plan
#+DATE: 2026-06-22

* 10.06.35 Next Loop Engine Delivery Plan

This document audits the current =docs/10-19-design/10.06-poo-module-system=
loop-engine documents and defines the next implementation plan.

It is a delivery plan, not a new design branch. The direction remains:

- Scheme owns POO-native declarations, validation, projection, report-only
  receipts, presentation, and Marlin handoff manifests.
- Marlin owns execution, scheduling, durable state, memory mutation, sandbox
  realization, connector calls, and conflict enforcement.
- Public loop-engine configuration remains 100% POO-native under
  =(use-module loop-engine :config ...)=.

* 10.06.35.10 Document Audit

The current documentation says three important things.

1. =32-loop-engine-poo-native-config.org= establishes the public contract:
   loop-engine no longer accepts row DSL or compatibility alists. The user
   writes POO objects, Scheme validates slots, and runtime projection stays
   report-only.

2. =33-loop-engine-memory-profile-audit.org= closes the memory-profile gap:
   =memory-policies= is now the root profile slot, each
   =loop-engine-memory-policy= belongs to one declared use case, and
   =memory-receipt= carries the selected policy plus all policy rows for
   Marlin.

3. =34-loop-operations-coordination-audit.org= closes the first operational
   extension step: =policy-extensions= now carries report-only receipts for
   coordination, observability, and safety policy families without hard-coding
   those families into =loop-engine-runtime.ss=.

The old =27-next-implementation-closure-plan.org= remains useful as a closure
baseline, but it predates the loop-engine POO-native work. Its "next" language
should no longer drive implementation priority by itself.

* 10.06.35.20 Current Baseline

Treat the following surfaces as the baseline for the next push:

- POO-native loop-engine config is the only public loop-engine config shape.
- Slot validation happens before intent projection.
- Result-contract diagnostics remain report-only diagnostics when the object
  shape is structurally valid.
- Runtime command manifests expose =object-families= and =receipt-contracts= so
  Marlin can discover inert handoff payloads.
- Memory, compression, capability, lineage, selector, resource, sandbox handoff,
  and generic policy-extension receipts are report-only.
- User-interface examples cover a guarded handoff case and a multi-use-case loop
  profile.
- Focused tests cover capability, memory policy, profile projection, result
  contract diagnostics, runtime manifest projection, slot contracts, and policy
  extension receipts.

* 10.06.35.30 Remaining Gaps

** 10.06.35.31 G1: Delivery-Oriented Loop Scenario

The docs now describe real loop families, but the next implementation slice
needs one delivery scenario that ties them together as an agent-readable
profile.

The target scenario should be:

- CI Sweeper owns red CI investigation.
- Dependency Sweeper pauses while main is red.
- Daily Triage remains L1 report-only.
- Post-Merge Cleanup records cleanup intent without mutating the repository.
- Human Audit remains the explicit escalation boundary.

The Scheme side should project receipts only. It must not read CI, update state
files, open PRs, schedule agents, or call GitHub.

** 10.06.35.32 G2: Marlin ABI Consumer Fixture

The docs say Marlin should not guess entrypoints. The next push should therefore
add a minimal downstream-consumer fixture, either in this repo as inert expected
payload data or in =marlin-agent-core= as the real consumer.

The fixture should prove that one manifest exposes:

- =poo-flow.loop-governor.runtime-command-manifest.v1=;
- =loop-engine-discovery=;
- =object-families=;
- =receipt-contracts=;
- =policy-extension-receipts=;
- =sandbox-handoff-agreement=;
- no runtime execution flag set to true.

** 10.06.35.33 G3: Operational Policy Receipt Coverage

The generic policy-extension path is in place. The next gap is coverage quality:
each concrete extension family should have a user-facing scenario and a receipt
assertion that validates its family-specific slots.

This should cover:

- coordination priority, state files, =acting_on= key, conflict action, and
  human inbox;
- observability run-log schema, budget path, metric keys, and slow/pause/kill
  signals;
- safety denylist paths, allowlist paths, connector scopes, max attempts, human
  gates, and auto-merge policy.

** 10.06.35.34 G4: Documentation Status Drift

The docs now contain implementation updates, but they are distributed across 32,
33, and 34. The next docs pass should keep one status table that says which
surfaces are:

- implemented in Scheme;
- represented only in docs;
- expected from Marlin;
- explicitly out of scope.

* 10.06.35.40 Next Implementation Order

1. Add a delivery scenario fixture for the CI Sweeper / Dependency Sweeper /
   Daily Triage interaction. Start in =user-interface/custom/my-module= so the
   public POO-native surface remains the source of truth.

2. Strengthen runtime manifest tests around =loop-engine-discovery= and
   =receipt-contracts=. The test should prove Marlin can find the request shape
   without scanning private profile objects.

3. Split concrete operational policy-extension assertions by family:
   coordination, observability, and safety. Keep the generic extension receipt
   path intact.

4. Add a downstream ABI handoff note for =marlin-agent-core=: expected command,
   request shape, receipt families, and non-execution guarantees.

5. Update the document status table after code lands. The docs should not imply
   that Scheme executes locks, run logs, state writes, connector calls, memory
   mutation, or scheduling.

* 10.06.35.45 Implementation Status

The first delivery slice is now in progress in the Scheme control plane.

- =user-interface/custom/my-module/profiles/loops.ss= carries delivery-scenario
  facts through concrete coordination, observability, and safety policy
  extension slots that already project into receipts. The profile now states
  that CI Sweeper has first priority, Dependency Sweeper is covered by the
  =main-red= pause signal, Daily Triage remains last-priority L1 reporting,
  Post-Merge Cleanup stays state-file/report oriented, and red CI handoff is a
  human-gated safety fact.
- =t/loop-engine-policy-extension-test.ss= now has separate receipt assertions
  for coordination, observability, safety, and generic propagation. This keeps
  concrete policy family behavior visible without hard-coding those families
  into the runtime core.
- =t/user-interface-custom-loop-engine-runtime-manifest-test.ss= now checks
  that the legacy =govern-loop= loop-engine discovery table and the loop-engine
  runtime handoff manifest agree on command contract, object families, receipt
  contracts, control owner, execution owner, and non-execution status.

Still out of scope for this slice:

- no Scheme scheduler;
- no state-file reads or writes;
- no GitHub or connector calls;
- no memory mutation;
- no Marlin consumer implementation.

* 10.06.35.60 Non-Goals

- Do not add a scheduler in Scheme.
- Do not read or write =STATE.md=, loop state files, or run logs in Scheme.
- Do not make =marlin-agent-core= a sandbox backend name.
- Do not add row DSL compatibility for loop-engine.
- Do not duplicate receipt contracts in a second discovery table.
