#+TITLE: 10.06.57 User Interface Profile Library
#+OPTIONS: toc:2

* 10.06.57.10 Purpose

The =user-interface/= tree is not only a set of readable instance examples.  It
is also the public contribution surface for reusable POO Flow profiles,
strategies, policies, and scenario cases.

The design intent is:

1. users and agents can read =user-interface/= to learn how to write good
   configuration;
2. community contributors can add reusable profiles instead of copying local
   one-off configuration;
3. downstream users can import those profiles through the POO Flow library;
4. downstream users can extend the imported profiles through POO objects and
   functional composition;
5. every public profile passes testing and Lean proof obligations before it is
   treated as reusable library surface;
6. the same profile library becomes the scenario source for validation,
   benchmark, and proof obligations.

This makes =user-interface/= both documentation and product surface.  It should
be treated as a reusable library boundary, not as disposable sample code.

* 10.06.57.20 Core Thesis

The highest-value contribution unit is not a full application.  It is a reusable
profile that captures a policy or strategy slice clearly enough for another
project to import, inspect, extend, and prove.

#+begin_src typst
$
op("Contribution")(c)
  =>
    op("Profile")(c)
    or op("Strategy")(c)
    or op("Policy")(c)
    or op("Scenario")(c)

op("Reusable")(c)
  =>
    op("Named")(c)
    and op("Composable")(c)
    and op("Documented")(c)
    and op("ProofAddressable")(c)
$
#+end_src

The library should push contributors toward small, well-scoped objects that can
be shared across many workflows.  A downstream project should not need to copy a
large policy file to change one capability.  It should import a profile and
extend it through POO.

* 10.06.57.30 User Interface As A Library

The =user-interface/= layer has four simultaneous roles.

| Role | Meaning | Required property |
|------+---------+-------------------|
| Example surface | Shows users and agents how to write valid configuration. | The example must be readable and idiomatic. |
| Contribution surface | Lets community members contribute reusable profiles, strategies, and policies. | The contribution must be named, scoped, and documented. |
| Library surface | Lets downstream users import and reuse profiles. | The profile must be stable enough to call from other code. |
| Proof/scenario surface | Provides real cases for scenario benchmark and Lean obligations. | The profile must expose typed facts and proof-relevant boundaries. |

This prevents the common failure mode where examples diverge from the real API.
If an example is valuable, it should be shaped so it can become a library object.
If a library object is reusable, it should have a readable example.

* 10.06.57.40 Contribution Model

The expected contribution flow is:

#+begin_src mermaid
flowchart LR
  Contributor["Contributor"]
  Profile["Reusable profile / strategy / policy"]
  Library["user-interface library"]
  Downstream["Downstream project"]
  Extend["POO object extension"]
  Proof["Validation and proof obligations"]

  Contributor --> Profile
  Profile --> Library
  Library --> Downstream
  Downstream --> Extend
  Extend --> Proof
  Proof --> Library
#+end_src

A contribution should answer these questions:

| Question | Why it matters |
|----------+----------------|
| What policy or strategy does this profile capture? | Avoids vague examples that cannot be reused. |
| Which capabilities does it require? | Feeds sandbox and tool-scope obligations. |
| Which dependencies must be satisfied before use? | Feeds workflow and module start obligations. |
| Which slots are intended extension points? | Makes POO extension explicit instead of accidental. |
| Which slots must not be broadened by children? | Feeds no-policy-broadening obligations. |
| Which scenario demonstrates the profile? | Gives users and agents a concrete instance. |
| Which facts should Scheme emit for proof? | Connects the contribution to Lean work. |

* 10.06.57.50 Profile Shape

A reusable profile should be small enough to compose and explicit enough to
validate.

| Field | Purpose |
|-------+---------|
| profile id | stable public name |
| owner/module | code owner and documentation owner |
| policy object | what is allowed |
| strategy object | how the next step is selected |
| sandbox contract | required tools, files, secrets, runtime placement |
| session contract | parent/child/handoff assumptions |
| loop contract | transition, retry, terminal, and progress assumptions |
| dependencies | required modules, artifacts, or facts |
| extension points | slots downstream users may override |
| forbidden broadening | slots downstream users must not broaden |
| proof facts | facts emitted by Scheme for Lean obligations |
| scenario benchmark | representative case for validation |

This is intentionally POO-native.  The ordinary contribution path should not be
an untyped alist, ad hoc patch language, or raw JSON manifest.  Those formats may
exist at import/export boundaries, but the public authoring surface should be
Gerbil POO objects and functional composition.

* 10.06.57.60 Downstream Reuse

Downstream users should be able to consume a contributed profile in three ways:

1. import the profile as-is;
2. compose it with other profiles;
3. extend it through POO object inheritance or object extension helpers.

The important invariant is that extension must preserve the parent contract.

#+begin_src typst
$
op("Uses")(d,p)
  =>
    op("Profile")(p)
    and op("Compatible")(d,p)

op("Extends")(q,p)
  =>
    op("Lin")_(upright("POO/C3"))(q,p)
    and op("PreservesContract")(q,p)

op("PreservesContract")(q,p)
  =>
    op("Scope")(q) subset.eq op("Scope")(p)
    or op("HasRefinementProof")(q,p)
$
#+end_src

This is the reason =user-interface/= matters for the proof architecture.  The
library is where reusable policy meets real downstream extension.  If a profile
cannot survive POO extension without losing its contract, it is not a good
public contribution.

* 10.06.57.70 Agent Authoring Contract

Agents should treat =user-interface/= as a source of reusable public objects,
not as a folder of snippets to copy.

When an agent writes a new user-interface profile, it should:

1. reuse an existing profile if one matches the policy shape;
2. extend through POO when only a small behavior changes;
3. create a new profile only when the policy shape is genuinely new;
4. keep extension points explicit;
5. emit proof-addressable facts;
6. add or update a scenario benchmark;
7. run the required tests and Lean proof checks;
8. document the intended downstream reuse path and proof status.

Anti-patterns:

| Anti-pattern | Why it is wrong |
|--------------+-----------------|
| Copying a whole example to change one tool. | Loses shared policy reuse and creates drift. |
| Adding raw data without POO object ownership. | Makes extension and proof ownership unclear. |
| Hiding policy changes in local overrides. | Makes downstream behavior hard to audit. |
| Treating a scenario as only documentation. | Loses benchmark and proof value. |
| Broadening sandbox scope silently. | Breaks the parent profile contract. |

* 10.06.57.75 Acceptance Gate

Every =user-interface/= contribution, whether maintained by the core project or
contributed by the community, must pass the same acceptance gate before it is
treated as public reusable library surface.

| Gate | Required evidence |
|------+-------------------|
| POO-native shape | profile, strategy, policy, and extension points are represented through POO objects or functional helpers |
| Scenario test | at least one concrete scenario exercises the profile |
| Scheme fact projection | the profile emits proof-addressable facts for its session, sandbox, loop, profile, workflow, and runtime boundaries |
| Lean obligation mapping | emitted facts map to named Lean obligations |
| Lean proof status | required obligations are discharged, or the profile is explicitly marked experimental/rejected |
| Regression test | changes keep existing scenarios and profile reuse behavior stable |
| Documentation | contribution documents reuse intent, extension points, and proof/test status |

#+begin_src typst
$
op("PublicUserInterfaceProfile")(p)
  =>
    op("Tested")(p)
    and op("HasFactProjection")(p)
    and op("MappedToLeanObligations")(p)
    and op("ProofStatusKnown")(p)

op("ReusableLibrarySurface")(p)
  =>
    op("PublicUserInterfaceProfile")(p)
    and op("DischargedRequiredObligations")(p)
$
#+end_src

Experimental profiles may exist only when their status is explicit. They should
not be presented as stable reusable policy. A profile that cannot produce
proof-addressable facts is not ready for the public library, even if it is useful
as a local sketch.

* 10.06.57.80 Proof And Benchmark Role

Every reusable profile must be connected to scenario benchmark and Lean
obligation work before it is promoted as public library surface.

| User-interface artifact | Benchmark/proof role |
|-------------------------+----------------------|
| profile | establishes reusable policy surface |
| strategy | defines branch/selection behavior |
| policy | defines allowed action scope |
| sandbox contract | creates capability subset/refinement obligation |
| session contract | creates lineage and handoff obligation |
| loop contract | creates progress and invariant obligation |
| scenario | gives a concrete validation case |
| extension example | tests contract preservation under POO extension |

#+begin_src typst
$
op("PublicProfile")(p)
  =>
    op("HasScenario")(p)
    and op("HasFactProjection")(p)
    and op("HasProofObligations")(p)
    and op("HasTestReceipt")(p)
$
#+end_src

The goal is not to make every contribution heavyweight.  The goal is to prevent
public examples from becoming unverified folklore.  A small profile can start
with a small scenario and a small obligation set, but the public status must be
honest: core profile, community contribution, experimental sketch, rejected
case, or proof-backed reusable profile.

* 10.06.57.85 Receipt-Backed Gate Semantics
The implementation gate is intentionally POO-native. A public reusable profile is not accepted by writing a JSON manifest or hand-authored fact table. The authoring object is a profile gate, and the acceptance evidence is a profile gate receipt.
#+begin_src typst
$
op("AcceptedGate")(p)
  =>
    op("PublicProfile")(p)
    and op("InProfileSet")(p)
    and op("Tested")(p)
    and op("HasScenario")(p)
    and op("HasBenchmark")(p)

op("AcceptedGate")(p)
  =>
    op("MappedToLeanObligations")(p)
    and op("HasProofReceipt")(p)
    and op("ProofDischarged")(p)

op("ReusableLibrarySurface")(p)
  =>
    op("DependencyClosed")(p)
    and op("ScopeContained")(p)
    and op("ObservabilityClean")(p)
    and op("CounterexampleRejected")(p)
$
#+end_src
The receipt object records scenario benchmark, proof receipt, observability receipt, dependency closure, scope containment, and counterexample rejection. Scheme projects still consume POO objects and functional composition. The alist appears only as the final Scheme-to-Lean fact projection boundary.

This matters for agent-written policy. A profile can look locally correct while still being globally unsafe: it may depend on a profile outside the set, leak sandbox scope, pass a happy-path test without a benchmark receipt, or publish with an open counterexample. The gate therefore rejects a discharged proof status when the scope or counterexample facts do not support publication.
* 10.06.57.90 Documentation Rule

Any user-interface contribution should document:

1. what it is reusable for;
2. which existing profile or policy it reuses;
3. which extension points it exposes;
4. which downstream user would import it;
5. which scenario proves it is not only illustrative;
6. which Scheme facts and Lean obligations it should produce;
7. which tests and Lean checks establish its current status.

This makes the repository friendlier to humans and agents while also making the
community contribution path practical.  The same artifact teaches, composes,
validates, and proves.
