#+TITLE: RFC 93 — Assurance, Proof, And Evidence Chain
#+SUBTITLE: Digest-Linked Claims Across Bundle, Cedar, Lean, Runtime, And Effects
#+OPTIONS: toc:3 num:t
#+LANGUAGE: en

* Status
Status: *Design-approved; written-review pending*

This RFC defines claim boundaries and assurance semantics. It does not claim that every listed proof, token, receipt, mediator, signer, or external anchor is implemented.

* Purpose
POO Flow must not use “provable” as a blanket claim.

A Lean theorem, Cedar allow decision, runtime receipt, or valid hash chain proves a different proposition under a different trusted boundary. Treating any one artifact as proof of total system safety creates a circular and untestable assurance story.

This RFC defines a digest-linked chain from the Canonical Organization Bundle to accepted runtime transitions, and gives every claim:

- a typed artifact;
- a precise statement;
- an explicit trusted computing base;
- stated assumptions;
- a verification rule;
- a cumulative assurance level.

* Decision
The accepted chain is:

#+begin_src text
Canonical Bundle
  -> Cedar projection and decision
  -> one-shot AuthorizedEffectToken
  -> runtime and adapter execution receipt
  -> transition verification receipt
  -> evidence batch root
  -> signed checkpoint or external anchor

Canonical Bundle
  -> Lean statements and proof artifacts
  -> proof receipt
  -> activation or effect obligation discharge
#+end_src

The following rules are normative:

1. Every derived artifact identifies its source Bundle digest and epoch.
2. Cedar policy, entities, Lean statements, proof artifacts, runtime configuration, and evidence obligations are digest-bound projections rather than independent truth.
3. A Cedar allow result is necessary but insufficient for an authorized external effect.
4. An AuthorizedEffectToken is one-shot, short-lived, non-amplifying, and bound to the complete effect intent.
5. Runtime receipts are verifiable claims, not automatic proof that external reality matches the claim.
6. A transition is accepted only after the verifier closes every required authorization, proof, execution, and evidence obligation.
7. Trusted computing bases are defined per claim.
8. Assurance levels are cumulative; a higher level cannot conceal a missing lower-level gate.
9. Production readiness is an independent qualifier.
10. Missing, stale, replayed, mismatched, or unverifiable safety artifacts fail closed.

* Root Digests
** Semantic Root
The =semantic_root= binds the semantic meaning activated for an organization epoch.

It covers or transitively identifies:

- Bundle digest and epoch;
- parent Bundle digest;
- Bundle schema and canonicalization profile;
- compiler identity and source-input provenance;
- facet digests;
- Cedar policy, entity, and projector digests;
- Lean theorem-statement and proof-manifest digests;
- Lean import-environment identity;
- evidence schema and required-obligation digest.

The semantic root proves identity and binding. It does not prove that compilation preserved author intent unless the corresponding compiler claim is separately checked.

** Execution Root
The =execution_root= binds one runtime deployment to a semantic root.

It covers or transitively identifies:

- semantic root;
- C ABI major/minor version and negotiated capabilities;
- runtime identity, version, and measurement where available;
- adapter identities and versions;
- runtime configuration digest;
- sandbox, isolation, or attestation identity used by the assurance claim;
- evidence durability profile;
- assurance profile.

Tokens and runtime receipts MUST bind the execution root.

Changing any semantic or execution component that affects a claim produces a different appropriate root. An implementation MUST NOT retain an old root while substituting new policy, proof, runtime, adapter, or configuration material.

* Typed Artifact Kinds
The chain distinguishes at least:

- *Bundle validation receipt* — reports compiler and Bundle validation.
- *Proof receipt* — reports a named theorem checked for a bound statement and environment.
- *Authorization decision receipt* — reports Cedar input identity and allow or deny output.
- *AuthorizedEffectToken* — grants one bounded execution attempt after required gates close.
- *Runtime execution receipt* — reports what a runtime claims it attempted and observed.
- *Adapter receipt* — reports adapter-specific execution identity and outcome.
- *Transition verification receipt* — reports whether the submitted evidence justifies a modeled state transition.
- *Checkpoint receipt* — binds durable state and evidence history.
- *Audit or external-observation receipt* — provides evidence independent of the executing runtime where available.

One artifact kind MUST NOT be accepted in place of another.

A runtime success receipt cannot discharge a Lean theorem. A Lean theorem cannot establish that a network effect occurred. A Cedar allow cannot substitute for an execution receipt. A proof receipt cannot substitute for runtime conformance.

* Cedar Boundary
Cedar evaluates a bound authorization request.

The request identifies:

- principal;
- action;
- resource;
- context digest;
- effect-intent digest;
- session and transition identity;
- Bundle digest and epoch;
- policy and entity digests.

The decision receipt identifies the Cedar engine and projection versions, request digest, policy and entity digests, decision, and any typed reason information required by the evidence profile.

Cedar can guarantee the decision produced for those inputs under Cedar semantics.

Cedar cannot by itself guarantee:

- that the policy represents the author's real intent;
- that Bundle-to-Cedar projection is correct;
- that a runtime uses the decision;
- that an adapter executes the same parameters that were checked;
- that no out-of-band effect path exists;
- that the external effect succeeds.

A deny decision produces decision evidence and MUST NOT produce an effect token.

* Lean Boundary
Lean checks named formal statements over typed inputs and explicit assumptions.

Initial proof owners SHOULD remain bounded to claims such as:

- shared-identity and Bundle well-formedness properties;
- authority attenuation and delegation bounds;
- organization hierarchy not implying authority;
- facet composition properties;
- protocol safety and lifecycle transition constraints;
- cancellation preventing later modeled execution;
- evidence obligations not being silently removed;
- projection and digest refinement relationships.

Lean does not automatically prove:

- arbitrary Gerbil evaluation correctness;
- C ABI memory safety;
- Python or downstream runtime correctness;
- absence of an out-of-band effect path;
- external system truth;
- unmodeled concurrency, failure, scheduler, or environment behavior;
- model output quality.

The proof receipt binds:

- theorem identity and exact statement digest;
- Bundle and semantic root;
- typed input digest;
- proof artifact digest;
- Lean version and locked import environment;
- checker result.

The Lean kernel, theorem statement, and locked import environment belong to the proof TCB. Tactics and proof generators do not enter the final proof TCB merely because they produced an artifact accepted by the kernel.

Lean SHOULD remain off the per-effect hot path. Required proofs are checked before Bundle activation or cached as bound proof receipts.

* AuthorizedEffectToken
An AuthorizedEffectToken is a bounded capability for one effect attempt.

Its logical content binds:

- execution root;
- Bundle digest and epoch;
- session and transition identity;
- attempt identity and monotonic sequence;
- runtime instance;
- principal, action, and resource;
- complete effect-intent digest;
- authorization request and decision digests;
- required proof and evidence obligations;
- issue and expiry information;
- revocation epoch;
- unique nonce;
- parameter-buffer lease identity where applicable.

The token MUST be:

- one-shot;
- non-transferable unless transfer semantics are explicitly modeled;
- non-amplifying;
- unforgeable under the deployment's token model;
- rejected after expiry, revocation, epoch replacement, or prior consumption.

An in-process deployment MAY represent the token with an opaque handle. A cross-process deployment MAY require a cryptographically protected packet. Both representations preserve the same logical fields and verification semantics.

If authorization hashes bytes stored in a borrowed or zero-copy buffer, those bytes MUST be immutable for the authorization-to-execution lease. Mutation after decision and before effect execution invalidates the token.

* Runtime And Adapter Receipts
A runtime receipt identifies:

- execution root;
- token and effect-attempt digests;
- before-state digest;
- effect parameters or their bound digest;
- adapter identity;
- runtime identity and measurement required by the profile;
- outcome kind;
- returned payload digest;
- failure classification;
- after-state claim;
- checkpoint relationship where applicable;
- prior receipt or batch relationship.

The adapter receipt identifies the actual adapter operation and parameter digest. An adapter MUST NOT report resource B for a token that authorized resource A.

Receipt signatures or MACs establish origin under the corresponding key. They do not prove that the signer is honest. A runtime signing its own receipt remains a self-assertion unless the assurance profile adds mediation, isolation, attestation, or independent observation.

* Transition Verification
The transition verifier accepts a proposed state transition only when:

1. Bundle, epoch, semantic root, and execution root match.
2. The transition exists in the Protocol facet.
3. Required Authority and Context conditions are satisfied.
4. The authorization decision and token are valid.
5. The token has not expired, been revoked, or been consumed.
6. Effect parameters match the authorized intent digest.
7. Runtime and adapter receipts have the required types and identities.
8. Required proof obligations are discharged by matching proof receipts.
9. Required evidence is complete under the declared durability profile.
10. Before-state and after-state digests satisfy the modeled transition.

The verifier emits an accepted, rejected, or indeterminate transition-verification receipt.

A rejected transition does not rewrite evidence as though the attempt never occurred.

* Evidence History
Evidence history is append-only at the logical level.

For performance, implementations MAY accumulate a batch of canonical event digests and seal a Merkle root or equivalent batch root. Each sealed root links to the prior accepted root and identifies its range of session sequence numbers.

A valid internal hash chain does not by itself prevent truncation or forking. Profiles that claim resistance to those attacks require one or more of:

- signed checkpoints;
- externally witnessed roots;
- transparency-log publication;
- replicated independent observers;
- hardware or operating-system attestation.

Evidence storage need not enter the integrity TCB when signatures and external roots detect mutation, but availability, truncation, and fork assumptions MUST remain explicit.

* Durability Profiles
The evidence contract defines at least:

** Strict
- authorization reservation is durable before external dispatch;
- completion or indeterminate evidence is durable before the next dependent transition;
- suitable for high-value or non-idempotent effects;
- higher latency is expected and measured.

** Batched
- authorization and completion records may be buffered and sealed as a bounded batch;
- the maximum unsealed event and time window is declared;
- crash recovery may expose a bounded indeterminate suffix;
- suitable only when the protocol defines observation, reconciliation, idempotency, or compensation.

** Diagnostic
- evidence may be best-effort;
- cannot support L2 Evidenced or higher claims;
- MUST NOT be silently selected for a profile requiring durable evidence.

* Trusted Computing Bases By Claim
There is no single useful infinite global TCB.

** Bundle Semantic Claim
Trusts:

- canonical schema and encoder;
- Bundle validator;
- digest implementation.

If the claim begins at POO source rather than at canonical Bundle bytes, it also trusts the compiler and projector unless compiler-correctness evidence discharges that trust.

** Authorization Claim
Adds:

- Bundle-to-Cedar projector;
- Cedar evaluator;
- token issuer;
- digest and key implementation;
- revocation and expiry state.

** Lean Claim
Trusts:

- Lean kernel;
- theorem statement;
- typed decoder;
- locked import environment;
- digest binding.

The proof generator and tactics are outside the final proof TCB when the kernel checks the complete proof artifact.

** Actual-Effect Claim
Adds:

- C ABI transition engine;
- selected runtime;
- selected adapter;
- credential and secret boundary;
- sandbox or isolation mechanism;
- measurement, attestation, or independent observer used by the claim.

Without a mechanism preventing out-of-band effects, the runtime is cooperative and cannot claim complete mediation.

** Evidence-History Claim
Adds:

- canonical event hashing;
- sequence and replay protection;
- batch-root construction;
- checkpoint signer;
- external anchor or fork-detection mechanism where claimed.

* Assurance Ladder
Production readiness is expressed independently, for example =Production + L2 Evidenced=.

** L0 Interface
- ABI loads and negotiates;
- schema and version compatibility are known;
- no semantic or security conformance is implied.

** L1 Conformant
- state-machine vectors pass;
- lifecycle, ownership, canonical packet, and fault contracts pass;
- traces agree under declared nondeterminism;
- no claim that the runtime cannot bypass the interface.

** L2 Evidenced
- tokens, decisions, receipts, replay, revocation, failure, checkpoint, and evidence-chain rules are verifiable;
- evidence proves internal binding and declared origin;
- runtime honesty and absence of out-of-band effects remain unproved.

** L3 Mediated
- isolation, independent observation, measurement, or attestation supports the claim that all in-scope effects pass through the authorization boundary;
- the exact scope and excluded effects are declared.

** L4 Formally Linked
- named refinement and invariant claims connecting POO/Bundle/Cedar/transition semantics are checked by Lean;
- only named theorems receive this label;
- L1–L3 requirements remain applicable to runtime claims.

Higher levels are cumulative. A Lean artifact does not compensate for missing conformance or evidence. A production runtime does not gain a formal level from deployment maturity alone.

* Failure Semantics
- Bundle, policy, proof, semantic-root, execution-root, or epoch mismatch fails closed.
- Missing required proof prevents Bundle activation; it does not trigger an implicit unproved mode.
- Cedar unavailability defaults to deny unless a pre-approved bounded offline policy exists in the active assurance profile.
- Expired, revoked, stale, replayed, or already consumed tokens fail closed.
- Parameter mutation after authorization rejects the transition and records the attempt.
- Missing execution evidence after an effect that may have occurred produces =indeterminate=.
- Evidence-store failure before dispatch prevents dispatch under Strict durability.
- Evidence-store failure after dispatch requires observation, reconciliation, or compensation.
- A forked or truncated history invalidates claims beyond the last externally anchored root.
- A verifier failure is recorded as a verification failure; it does not erase the submitted runtime evidence.

* Required Counterexamples
The conformance and assurance suite MUST include:

1. organization parenthood incorrectly projected as an authority grant;
2. a new Bundle combined with an old Cedar policy;
3. an old token replayed after revocation or epoch switch;
4. a token for resource A used to execute resource B;
5. a borrowed buffer mutated after digest computation;
6. a timeout and retry executing a one-shot effect twice;
7. runtime success with no external effect;
8. an external effect with no runtime receipt;
9. a valid Lean proof for a different Bundle or theorem statement;
10. a correct formal model paired with an incorrect compiler or projector;
11. an internally valid evidence history truncated at a prior root;
12. two separately valid forks after one checkpoint;
13. one runtime self-testing, self-signing, and self-verifying an unsupported high-assurance claim;
14. a proof receipt substituted for an execution receipt;
15. a runtime receipt substituted for a proof receipt.

* Test Matrix
** Digest Binding
- every artifact with the correct roots succeeds;
- substitution of Bundle, epoch, policy, projector, proof, runtime, adapter, or config changes the root and fails matching;
- canonical ordering is deterministic;
- unknown major schema fails closed.

** Cedar
- allow and deny;
- wrong principal, action, resource, context, or payload;
- stale policy and entities;
- projector mismatch;
- unavailable evaluator;
- decision receipt tamper.

** Token
- valid one-shot use;
- expiry;
- revocation;
- replay;
- wrong runtime instance;
- wrong session or sequence;
- intent substitution;
- buffer TOCTOU;
- duplicate batch submission.

** Lean
- valid named proof;
- false statement;
- wrong Bundle;
- wrong theorem statement;
- wrong import environment;
- generated artifact corruption;
- missing required proof;
- tactic failure without a claimed proof receipt.

** Runtime And Transition
- success, denial, cancellation, timeout, failure, and indeterminate;
- before/after state mismatch;
- adapter parameter mismatch;
- receipt type substitution;
- missing external observation for an L3 claim;
- checkpoint and restore across epochs.

** Evidence History
- append and batch seal;
- sequence gap;
- duplicate event;
- tampered event;
- truncated suffix;
- forked suffix;
- invalid signer;
- external anchor mismatch;
- strict and batched crash windows.

* Security Considerations
** Circular Assurance
A component cannot establish a strong claim solely by signing and verifying its own assertions. Higher assurance requires an independently trusted boundary or a formally checked refinement whose assumptions are explicit.

** Projection Drift
All Cedar and Lean artifacts bind projector identity and Bundle digest. A valid downstream artifact derived from another semantic root is invalid for the current execution.

** Token Theft And Transfer
Tokens bind runtime instance, session, sequence, intent, expiry, and epoch. Deployment profiles add process, key, or attestation binding as required.

** TOCTOU
Authorization and execution use the same immutable intent bytes. Borrowed and arena buffers remain frozen under a lease until completion or cancellation.

** Out-Of-Band Effects
L1 and L2 do not prove absence of bypass. L3 requires a named isolation, mediation, or observation boundary.

** Formal-Model Gaps
Every L4 theorem states its assumptions and modeled domain. Unmodeled concurrency, external failures, and environment behavior remain outside the claim.

** Evidence Forks
Hash links alone do not establish one global history. Strong history claims require external roots or independent witnesses.

* Non-Goals
This RFC does not:

- prove arbitrary model output quality;
- prove external services honest;
- make Cedar a global organization prover;
- make Lean a runtime executor;
- make runtime receipts proof of external truth;
- require Lean on each hot-path effect;
- require one storage engine or signature algorithm;
- claim L3 mediation without a concrete isolation or observation mechanism;
- claim L4 for unnamed or unmodeled properties;
- replace runtime conformance with proof artifacts.

* Decision Notes
** One Global TCB Rejected
A global TCB containing every compiler, runtime, adapter, library, storage engine, tool, and external service is too broad to explain any claim. TCBs are defined per claim.

** Lean Artifact Equals Safe Deployment Rejected
Lean checks formal statements. Runtime conformance, effect mediation, and external-world evidence remain independent.

** Cedar Allow Equals Authorized Execution Rejected
A decision must be bound to an immutable request and converted to a one-shot token. The executed parameters and receipt must close the same intent.

** Runtime Receipt Equals Truth Rejected
A receipt establishes an attributable claim. Stronger truth claims require independent observation, isolation, or attestation.

** Hash Chain Alone Prevents Forks Rejected
Two forked histories may each have valid internal hashes. Strong history requires checkpoints and external anchoring.

** Formal Level As Marketing Badge Rejected
Assurance applies to named claims and cumulative gates. It is not a single project-wide label.

* Acceptance Criteria
This RFC is implemented when:

1. Every Bundle-derived Cedar, Lean, runtime, and evidence artifact binds the correct semantic root.
2. Every runtime token and receipt binds the correct execution root.
3. Proof, authorization, execution, transition, checkpoint, and audit receipts are distinct typed artifacts.
4. Cedar allow decisions produce one-shot tokens only after required obligations close.
5. Tokens bind complete immutable effect intent, runtime instance, session, sequence, epoch, expiry, and revocation state.
6. Transition verification checks all required artifacts before accepting state.
7. Missing or mismatched safety evidence fails closed or produces =indeterminate= where the effect may already have occurred.
8. Evidence histories use canonical digests, sequence protection, batch roots, and the declared fork-detection profile.
9. Strict, Batched, and Diagnostic durability profiles behave as defined.
10. Claim-specific TCBs are published with every assurance profile.
11. L0–L4 levels are cumulative and reported independently from production readiness.
12. L3 claims name a concrete mediation, isolation, attestation, or independent-observation boundary.
13. L4 claims name exact Lean-checked statements and assumptions.
14. Every required counterexample fails in the expected way.
15. The full digest, Cedar, token, Lean, runtime, transition, and evidence test matrix passes.
16. Existing proof and receipt artifact owners are linked through roots and typed artifacts rather than replaced by an untyped umbrella claim.
