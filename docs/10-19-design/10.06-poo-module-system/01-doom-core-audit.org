#+title: 10.06.01 Doom Core Audit

* 10.06.01 Doom Core Audit
** 10.06.01.10 Reference Scope
Primary references:
- [[https://github.com/doomemacs/core][doomemacs/core]]
- [[https://docs.doomemacs.org/latest/][Doom Emacs documentation]]

This audit treats Doom as a user-interface and operations reference. It does
not treat Doom as an implementation source for =poo-flow=.

** 10.06.01.20 Useful Ideas To Borrow
*** Modular Organization
Doom presents modules as opt-in units that bundle packages, configuration, and
commands. The useful idea for =poo-flow= is not the exact directory layout; it
is the user-facing promise that a module is one named unit with clear effects.

For =poo-flow=, a module should mean:
- a named Gerbil POO descriptor bundle;
- explicit imports;
- task-family descriptors;
- flow-declaration descriptors;
- stable options that can be inspected before runtime execution.

*** Command-Line Diagnostics
Doom emphasizes commands such as =doom sync= and =doom doctor=. The useful
lesson is that module systems need an operations surface, not only a loader.

For =poo-flow=, the equivalent should eventually be:
- list activated modules;
- explain which descriptors each module contributes;
- report missing imports;
- report runtime capability gaps before execution;
- show which config options came from modules.

*** Declarative Package Management
Doom's package layer is declarative and reproducible. =poo-flow= should borrow
the reproducibility principle, not the Emacs package machinery.

For =poo-flow=, module activation should stay data-first:
- module metadata is inspectable;
- activation returns an object;
- the object can lower into =run-config=;
- execution remains owned by runner and adapter layers.

*** Rule-Based UI Concepts
Doom's popup manager is a rule system for temporary windows. The useful pattern
is a rule table with named policies, not the window-management details.

For =poo-flow=, later user modules can expose rule-like descriptors for:
- adapter routing;
- output capture;
- retry policy;
- artifact projection;
- sandbox capability declarations.

** 10.06.01.30 What Not To Borrow
*** Macro Density
Doom uses many macros to make Emacs Lisp configuration concise. That is useful
for expert users but risky as the first implementation layer.

=poo-flow= should delay macro sugar until the plain descriptor model has:
- activation tests;
- failure diagnostics;
- import validation;
- documentation examples;
- a stable data projection for agents.

*** Hidden Runtime Effects
Doom can mutate editor state as part of module loading. =poo-flow= should not
do this during activation. Activation installs descriptor data only. Runtime
effects belong to runner, strategy, and adapter modules.

*** Direct Package Loading Semantics
Doom modules load files and packages. =poo-flow= modules in this first slice do
not load files. A future loader may discover module descriptors, but the core
activation API should remain independent of filesystem discovery.

** 10.06.01.40 Design Consequence
The first =poo-flow= module API is intentionally narrower than Doom:
- no module filesystem scanner;
- no macro DSL;
- no dynamic package install;
- no implicit imports.

The borrowed surface is user-facing clarity: named modules, explicit imports,
diagnostics, and a single activation object that can be inspected.
** 10.06.01.50 Current Research Landing
The local research copy is =.data/doomemacs-core=, cloned shallow from =doomemacs/core=.

The useful Doom interfaces observed in the checkout are:
- module tables keyed by group and name;
- enabled flags and features carried as metadata;
- init, config, and package phase files;
- before and after hooks around module phases;
- depth ordering that can differ between init and config phases;
- fast =modulep!= style predicates over cached module context.

=poo-flow= absorbs these ideas as explicit descriptor data, not as hidden macro context:
- =group= and =name= are descriptor slots;
- =flags= and =features= are inspectable values;
- =depth= drives phase-order projections;
- =phase-files= records candidate entrypoint names without loading files;
- =hooks= records runtime-owned payloads without running them;
- =poo-module-active?= and =poo-flow-module-value-catalog-active?= replace Doom's implicit =modulep!= macro.
