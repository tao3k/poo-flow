* Status

Status: *Accepted; implementation planning*

ABI freeze readiness: *blocked* until the gerbil-mysql owner-index research gate and every conformance, memory, fault, and performance gate in this RFC pass.

Design approval fixes the architecture and invariants. It does not freeze the final ABI v1 binary layout.

* Decision
This RFC defines the normative C Runtime Interface exported by POO Flow and the role of the first-party Python runtime.

The approved decisions are:

1. POO Flow is the contract owner for a versioned, language-neutral C runtime boundary derived from the Canonical Organization Bundle.
2. This layer does not implement first-party Rust, TypeScript, or other downstream language runtimes.
3. Python is a complete, production-grade runtime implemented exclusively through the public C ABI.
4. Python proves interface completeness under production workloads but is not the semantic owner.
5. The ABI has a control lane, batched hot event lane, and bulk buffer lane.
6. The production hot path is pull-based and batch-oriented.
7. Public types never expose =___SCMOBJ=, Scheme heap addresses, Gerbil object layout, or private runtime pointers.
8. Every crossing declares buffer ownership and lifetime.
9. Zero-copy may be claimed only when buffer stability, immutability, and lifetime are demonstrated.
10. Canonical evidence packets and native fast views derive from one schema and are differentially validated.
11. ABI v1 remains unfrozen until the Gerbil and =gerbil-mysql= research, C harness, Python runtime, memory, fault, and performance gates pass.

* Purpose

POO Flow needs a runtime boundary that is language-independent, stable, complete, explicit about memory and failure, compatible with authorization and evidence, and efficient under high event rates and large context payloads.

A function-per-object binding would cause excessive FFI crossings, implicit conversions, unclear ownership, and dependency on Scheme internals. This RFC instead specifies a semantic state-machine protocol with separate control, hot, and bulk lanes.

* Scope and Non-Goals

This RFC is the contract owner for the public C ABI boundary, version and capability negotiation, opaque handles, ownership and buffer lifetimes, the three lanes, batch representation, lifecycle/effect exchange, typed errors, Python's contract, downstream conformance, performance and fault matrices, and the ABI-freeze gate.

The Canonical Organization Bundle remains the semantic owner. RFC 93 is the contract owner for authorization, Lean, and evidence claims; this interface transports their identities, tokens, obligations, and receipts without redefining them.

This RFC does not:

- make C structure layout or Python objects canonical semantics;
- expose Gerbil or Gambit implementation layout;
- define first-party Rust or TypeScript runtimes;
- require one internal scheduler or process model;
- claim ABI conformance alone proves a runtime trustworthy;
- serialize the entire Bundle for each transition;
- claim zero-copy where conversion, allocation, pinning, or lifetime extension occurs;
- freeze ABI v1 before the acceptance gates pass.

* Normative Public Boundary

The public interface consists only of installed versioned headers, installed declared libraries, version/capability metadata, normative lifecycle/data-plane documentation, conformance vectors, and canonical schemas required by consumers.

A conforming runtime MUST be implementable without importing private Gerbil modules, calling private Scheme procedures, depending on generated internal symbols, inspecting the Scheme heap, linking source-tree-only helpers, or reproducing Python behavior.

If the production Python runtime requires a private bypass for required behavior, the ABI is incomplete. No Python exception is permitted.

The Bundle is the semantic owner, RFC 92 and its generated ABI are the runtime-contract owner, and each runtime project is its implementation owner. No runtime may introduce private semantic states, effects, authority rules, or receipt kinds and present them as POO Flow semantics.

The public ABI MUST NOT expose:

- =___SCMOBJ= or equivalent Scheme tagged values;
- movable or unpinned Scheme heap addresses;
- Gerbil class, slot, or prototype layout;
- Gambit cleanup-frame internals or Scheme closures;
- Python object pointers;
- C++ classes, exceptions, or mangled symbols;
- NUL-terminated strings as the only byte/text representation;
- structures whose semantics depend on compiler padding.

* Versioning and Capability Negotiation

The interface MUST publish ABI major/minor versions, schema version, canonicalization profile, capability identifiers, ownership/buffer profiles, concurrency profile, and batch/payload limits.

A breaking binary or semantic change requires a new ABI major. A compatible addition may increment the minor version only when explicit size and capability negotiation lets older consumers interpret or reject it safely. Incompatible surfaces MUST NOT share unversioned aliases.

Before opening a Bundle or session, a consumer supplies its supported version range, schema/profile support, requested lanes/capabilities, batch/payload limits, ownership modes, concurrency needs, runtime identity, and assurance metadata. The library returns one negotiated profile or a typed incompatibility error. Library loading alone is not negotiation.

Every extensible descriptor MUST carry a size or version discriminator. Reserved fields follow declared initialization rules and gain no semantics without a compatible version change. Semantic fields use fixed-width integer types, not implementation-dependent =int=, =long=, enum width, or pointer-sized values.

* Three-Lane Architecture

** Control Lane

The control lane handles negotiation, Bundle and session open/close, runtime registration, capability exchange, cancellation, checkpoint, restore, recovery inspection, and orderly shutdown.

Control operations return typed results and explicit handles. They MAY validate, allocate, and canonically decode. They MUST NOT hide an unbounded per-effect callback loop.

** Hot Event Lane

The hot lane transports high-frequency state-machine work through logical =poll_batch= and =submit_batch= operations.

=poll_batch= obtains ready transitions, authorized effects, denials, or control events. =submit_batch= returns effect outcomes, runtime failures, receipts, and completion information.

Single-item operations MAY support debugging and minimum conformance but are not the production target. The production profile MUST batch work to amortize boundary crossings. Its main loop is pull-based and MUST NOT require one C-to-runtime callback per effect.

** Bulk Buffer Lane

The bulk lane transports context, artifacts, model/tool payloads, checkpoints, receipts, witnesses, and other large values through explicit byte views, caller buffers, owned buffers, or negotiated arenas.

Large payloads MUST NOT be forced through repeated per-field string conversion. A hot header MAY identify slices in a bulk arena.

* Opaque Handles

Long-lived Bundle, session, agent binding, effect attempt, checkpoint, arena, and lease resources use opaque handles.

Every handle has one resource kind, belongs to its creating ABI instance, has explicit create/release operations, cannot be reinterpreted, exposes no underlying Scheme pointer, and resolves to Bundle digest/epoch where required.

Null is invalid unless explicitly documented as optional. Wrong-kind, stale, double-released, or cross-instance use MUST return a typed error or be detected by conformance instrumentation. Thread safety is negotiated; it MUST NOT be assumed.

* Batch Layout

A batch logically contains:

1. a contiguous array of fixed-layout event headers;
2. contiguous payload regions or one negotiated arena;
3. explicit item count, lengths, and capacities;
4. batch identity and session sequencing context.

A header contains fixed-width fields sufficient for event kind, flags, sequence, compact identities, payload offset/length, alignment or encoding tag, effect/result correlation, and Bundle epoch or an unambiguous session binding.

Headers use offsets and lengths rather than one allocated pointer per field. Every offset, length, stride, and capacity calculation MUST be overflow-safe and bounds-checked. Unknown tags/flags fail unless negotiated extensibility permits them.

The native fast view is process-local. It MAY use a precisely defined host-native alignment profile but MUST NOT be persisted as canonical evidence, hashed as canonical bytes, transported without explicit encoding, or used after its lease expires.

Canonical packets and native views derive from one schema. Differential tests MUST prove native-to-canonical and canonical-to-typed equivalence and that malformed native views cannot normalize into different valid events.

Static Bundle digest, policy digest, epoch, and identity tables SHOULD be cached behind validated session handles. Per-event traffic carries compact identities and changing data rather than reserializing the Bundle.

* Ownership, Buffers, and Zero-Copy

Every crossing classifies a buffer as exactly one of:

- *owned*: receiver releases it through the declared operation;
- *borrowed*: provider retains ownership and grants an explicit bounded lease;
- *caller-provided*: caller owns storage and grants declared access;
- *negotiated arena*: both sides follow negotiated allocation, generation, reclamation, and synchronization.

Ownership MUST NOT be inferred from =const=, pointer type, allocation origin, or naming.

An owned result identifies its release operation; unrelated allocators MUST NOT free it. Custom release functions MAY be used internally, but the public contract exposes documented release semantics.

A borrowed view has explicit invalidation conditions and MUST remain valid until lease expiry. A consumer MUST NOT retain it afterward.

A caller-provided buffer reports capacity plus produced/required length; insufficient capacity is typed, never silent truncation.

An arena profile defines allocator ownership, alignment, region/generation identity, publication/reclamation, capacity, concurrency, exhaustion, lease invalidation, and reuse. An in-flight region MUST NOT be recycled before release or terminal acknowledgement.

Zero-copy may be claimed only when a C-owned, caller-owned, or arena buffer stays address-stable, no hidden semantic copy occurs, checks observe immutable bytes, and lifetime/reclamation are enforced.

A Scheme heap pointer MUST NOT be exposed as zero-copy without a verified pinning mechanism compatible with GC. When authorization hashes a borrowed/arena payload, those bytes MUST remain frozen through execution so checked parameters cannot be mutated.

Hot-path data uses pointer-or-offset plus explicit length. It MUST NOT require implicit =strdup=, NUL scanning, per-field =char-string= conversion, or hidden transcoding. Text carries an encoding tag; arbitrary payloads remain bytes.

* Lifecycle Protocol

The logical sequence is negotiate, open validated Bundle, open session, poll batches, execute authorized effects or non-effect transitions, submit typed outcomes/receipts, checkpoint/restore/cancel as needed, close session, and release Bundle/ABI resources.

A session binds to one Bundle digest and epoch and MUST NOT switch silently. Restore verifies checkpoint, Bundle lineage, schema, capabilities, and required evidence. Close gives an explicit disposition for outstanding work and MUST NOT report success while effects remain indeterminate.

* Effect Exchange

A polled effect item contains or resolves to session, transition/attempt, Bundle digest/epoch, principal/action/resource, complete effect-intent digest, authorization decision, one-shot authorized-effect token when allowed, required evidence kinds, payload descriptors, and deadline/cancellation/lease data.

Denial is a typed protocol outcome, not ABI malfunction. An effect requiring authorization MUST NOT execute without a valid token.

Submission identifies the exact transition and attempt, intent digest, disposition, returned payload, runtime receipt, and runtime/adapter identity required by its assurance profile. Substituting parameters after authorization is invalid. One-shot tokens MUST NOT cross attempts, epochs, or payload digests.

Every attempt has stable correlation identity. Retry is an explicit new attempt or defined idempotent resubmission, never an untracked repeat. Partial batch acceptance reports accepted/rejected items or queryable sequence state. Resubmission of accepted work returns duplicate/previously-accepted rather than executing again.

If an external effect may have occurred but its result or receipt cannot be established durably, its disposition is =indeterminate=, not success or failure. Recovery uses observation, reconciliation, idempotency, or compensation according to Protocol and assurance profile.

* Error Semantics

The ABI MUST NOT depend on global last-error state, ambient =errno= alone, Scheme/Python/C++ exceptions, or non-local jumps across the boundary. Every operation returns typed status; diagnostics use explicit owned or borrowed buffers.

A batch has call-level status, per-item status, accepted/produced/required counts, and sequence information for partial recovery. Successful exchange does not mean every effect succeeded.

The error model distinguishes invalid argument, stale/wrong handle, incompatible version/schema, unsupported capability, malformed descriptor, buffer too small, arena exhaustion, allocation failure, Bundle/epoch/policy/token mismatch, denial, expiry/replay, cancellation, timeout, runtime/adapter failure, checkpoint incompatibility, receipt failure, duplicate submission, partial acceptance, indeterminate effect, and internal invariant violation.

Denial, runtime failure, and ABI misuse MUST NOT collapse into one result. Bundle, epoch, policy, token, bounds, and digest mismatches fail closed.

* Concurrency and Reentrancy

Negotiation declares whether sessions may run concurrently, one session may be polled concurrently, submission may overlap polling, handles are thread-affine, arenas require synchronization, and control calls are reentrant.

Absent an explicit capability, concurrent handle use is forbidden. Exceptional callbacks MAY exist only with declared thread, lifetime, error, and reentrancy rules; they do not replace batch pull.

* Python Production-Grade Validation Runtime

Python is first-party, complete, production-deployable, and a conformance/completeness validator. It is neither a reduced stub nor a privileged bypass or semantic owner.

It MUST cover scheduling, event polling, tool/adapter execution, context/memory transfer, authorization, tokens, receipts/evidence, cancellation, timeout, retry, checkpoint, restore, crash recovery, typed failures, and indeterminate outcomes exclusively through installed public artifacts.

The production binding uses CFFI out-of-line API mode, which generates compiled C wrappers that call declared functions directly. Dynamic ABI-only =dlopen=/=libffi= access may support diagnostics but is not the production baseline.

Compatible buffers SHOULD use the Python buffer protocol and =ffi.from_buffer= with explicit lease retention. An out-of-line =extern "Python"= callback MAY handle bounded exceptional integration but MUST NOT become the per-effect loop.

A Python object retaining a C view MUST retain its handle, lease, or arena generation. GC MUST NOT release native resources while in use. Deterministic resources use explicit close; finalizers are defensive fallback only.

Passing Python validates ABI completeness, trace equivalence, batching, ownership, recovery, production viability, and measured CFFI overhead. It does not override Bundle or ABI semantics.

* Downstream Runtime Adapters

Rust, TypeScript, and other integrations are downstream-owned. They MAY use different schedulers and memory models but MUST negotiate the same ABI, obey lifecycle/ownership rules, preserve semantic event and result kinds, bind receipts to Bundle/epoch, pass conformance vectors, declare capabilities/assurance, and preserve authorization/evidence identities.

POO Flow does not require downstream sources in this repository. Conformance does not imply equal assurance; each runtime publishes its own profile and evidence.

* Gerbil and Gambit Evidence Boundary

The reviewed =.data/gerbil/doc/guide/ffi.md= material establishes that Gerbil FFI supports distinctions including owned values, borrowed values, shallow pointers, C structure arrays, and custom release functions. It also shows that string-bearing members can involve copying and release behavior.

This supports the feasibility of explicit ownership profiles and demonstrates why ownership cannot remain implicit.

The Gambit C interface generates Scheme/C conversion and cleanup behavior around foreign calls. A boundary crossing therefore includes conversion, temporary allocation, cleanup, and GC interaction, not merely the called C function. The design consequently batches crossings, avoids hot-path string conversion, and measures Scheme allocation and GC time.

Reference: [[https://gambitscheme.org/latest/manual/][Gambit Scheme Manual]].

* =gerbil-mysql= Evidence and Owner-Index Blocker

The reviewed =.data/gerbil-mysql= checkout confirms separation among low-level native binding ownership in =db/_mysql.scm=, binding/public API ownership in =db/mysql.ss=, and test ownership in =db/mysql-test.ss=.

The parser owner index did not provide a valid parser-owned item selector for the relevant internal implementation during review. Therefore the review did not establish exact internal result ownership, row materialization, byte/text conversion, prepared-buffer reuse, or batch amortization.

This is an ABI-freeze blocker, not evidence that those optimizations exist or do not exist.

Before v1 freeze, focused owner-level evidence and benchmarks MUST establish those behaviors. Findings MUST inform ownership profiles, batch descriptors, release functions, Python lifetime rules, thresholds, and conformance fixtures. The ABI MUST extract only measured reusable principles; it MUST NOT copy =gerbil-mysql= internals by assumption.

* Performance Model and Matrix

The interface minimizes FFI crossings, Scheme/C/Python conversions, allocations, pointer chasing, copied bytes, repeated digest work, GC pressure, callbacks, and whole-Bundle serialization.

No unmeasured absolute latency target is normative. Acceptance uses reproducible baselines and reviewed regression thresholds.

Required batch sizes are =1 / 8 / 32 / 128 / 1024=. Required payloads are empty, 1 KiB, 64 KiB, and 1 MiB. Required paths are allow, deny, runtime error, timeout, cancel, and checkpoint/restore.

Every case compares Gerbil in-process, a pure C harness, and the Python CFFI production path. It records FFI crossings, Scheme/C/Python allocations where measurable, copied and referenced bytes, GC time, p50, p99, throughput, RSS, and error/retry counts.

Each receipt identifies revision, build profile, ABI/schema, hardware/OS, capabilities, complete input datum, and machine-checkable result. A reduced datum MUST NOT be reported as full-path performance.

Freeze evidence MUST show batching amortizes crossings, native/canonical views agree, large payloads avoid hidden per-field copying, Python is production-viable, ownership does not retain unbounded memory, and reviewed GC/allocation/copy/latency/throughput/RSS thresholds pass. Faster but semantically weaker paths are rejected.

* Conformance and Fault Matrix

Conformance tests MUST cover:

- compatible/incompatible negotiation and missing capabilities;
- ownership, concurrency, batch, and payload profiles;
- open/poll/submit/checkpoint/restore/cancel/close;
- wrong, stale, double-released, and cross-instance handles;
- batch-one equivalence, correlation, mixed outcomes, partial acceptance, duplicate submission, malformed bounds/alignment, and unknown tags;
- owned release, borrowed expiry, caller capacity, arena reclamation, Python retention, allocator mismatch, leaks, and use-after-free;
- native/canonical equivalence and C/Python normalized trace equivalence;
- Bundle/epoch and intent-digest preservation;
- allow/deny transport, expiry, replay, payload substitution, receipts, and old-epoch rejection.

Fault injection MUST cover allocation failure, arena exhaustion, malformed/truncated descriptors, integer overflow, stale handles, runtime/adapter crash, timeout before/after possible execution, cancellation before/during dispatch, duplicate retry, partial acceptance, checkpoint corruption, incompatible restore, Bundle/policy mismatch, token expiry/replay, post-authorization mutation, receipt persistence failure, unknown external outcome, and shutdown with outstanding work.

Every fault produces a typed machine-checkable result. Possible effect without durable outcome MUST be =indeterminate=.

* Security Considerations

- Prohibiting public Scheme objects isolates consumers from GC movement and tagged layouts.
- Overflow-safe bounds validation precedes payload interpretation or execution.
- Frozen authorization leases prevent check-then-mutate attacks.
- Stable attempts, one-shot tokens, partial-acceptance state, and duplicate detection prevent silent repeated effects.
- Session, effect, checkpoint, token, and receipt binding prevent epoch confusion.
- Pull batches avoid normal-path callback reentrancy and lifetime hazards.
- The ABI mediates conforming execution but cannot prevent a malicious runtime's unrelated out-of-band effects; stronger claims require isolation or independent evidence.

* Decision Notes

- *Three lanes accepted:* control, hot events, and bulk payloads have different allocation and lifecycle needs.
- *Per-effect calls rejected:* batching amortizes Gerbil, Gambit, C, and Python costs.
- *Callback-driven hot path rejected:* pull avoids reentrancy, threading, exception, and lifetime hazards.
- *Header array plus payload arena accepted:* it reduces allocation and pointer chasing.
- *Canonical-only hot path rejected:* native views provide speed while schema-derived canonical packets provide evidence and interoperability.
- *Implicit ownership rejected:* Gerbil evidence shows real owned, borrowed, shallow, array, and custom-release modes.
- *Public =___SCMOBJ= rejected:* it would bind downstream runtimes to private GC and representation details.
- *Unproven Scheme-heap zero-copy rejected:* stability, immutability, and reclamation must be demonstrated.
- *CFFI API mode accepted:* compiled out-of-line wrappers are the production path; callbacks are exceptional only.
- *Python semantics rejected as normative:* Python is a production completeness proof; the Bundle remains the semantic owner, and RFC 92 plus its generated C ABI remain the runtime-contract owner.
- *Immediate ABI freeze rejected:* unresolved owner-level binding evidence and performance measurements could otherwise make avoidable copying permanent.

* ABI v1 Freeze Gate

ABI v1 may freeze only when:

1. Installed headers/libraries expose the complete interface without source-tree dependencies.
2. Public types contain no Scheme/Python object or private generated symbol.
3. Version, capability, ownership, concurrency, batch, and payload negotiation pass.
4. All three lanes are implemented and documented.
5. A pure C harness covers every lifecycle and effect state.
6. Python implements all required behavior only through installed public artifacts and production CFFI API mode.
7. Native/canonical differential validation passes.
8. All ownership modes pass lifetime, leak, allocator, bounds, overflow, use-after-free, and double-release tests.
9. Retry, replay, partial acceptance, checkpoint, cancellation, crash, timeout, and indeterminate tests pass.
10. The complete performance matrix has machine-checkable receipts and enforced reviewed thresholds.
11. Owner-level =gerbil-mysql= research establishes result ownership, row conversion, prepared-buffer reuse, and batch amortization.
12. Gerbil/Gambit/=gerbil-mysql= findings are reflected in ownership and batching contracts.
13. Symbol versioning and breaking-change policy are reviewed.
14. No required behavior depends on private Python or Gerbil bypasses.
15. No unversioned obsolete alias competes with the canonical interface.

* Acceptance Criteria

This RFC is implemented when:

1. POO Flow publishes a versioned C interface derived from the Canonical Organization Bundle.
2. Control, hot event, and bulk buffer lanes are distinct.
3. Production exchange supports batched =poll_batch= and =submit_batch=.
4. Batches use contiguous fixed-width descriptors and bounded payload slices or negotiated arenas.
5. Handles are opaque, typed by contract, and explicitly released.
6. Every buffer has one explicit ownership/lifetime class.
7. Zero-copy paths prove stability, immutability, and reclamation.
8. No public type exposes Scheme or Python implementation objects.
9. Lifecycle, effect, cancellation, retry, checkpoint, restore, and indeterminate outcomes are typed.
10. Batch results expose call-level/item-level status and partial acceptance.
11. Sessions, effects, checkpoints, tokens, and receipts bind to Bundle digest and epoch.
12. Python is production-grade, uses installed public artifacts only, and runs its production path through out-of-line CFFI API mode.
13. Python preserves every buffer lease and deterministic resource lifecycle.
14. Downstream runtimes can implement the protocol without Gerbil/Python internals.
15. C and Python traces agree after declared nondeterminism normalization.
16. Performance, conformance, memory, and fault matrices pass.
17. The =gerbil-mysql= owner-index blocker closes with owner-level evidence and benchmark receipts.
18. Every freeze condition passes before ABI v1 compatibility is declared.
