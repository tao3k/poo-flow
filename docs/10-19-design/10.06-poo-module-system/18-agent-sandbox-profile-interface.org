* 10.06.18 Agent Sandbox Profile Interface
** 10.06.18.10 Bass-Derived User Interface Rule
Bass keeps user work visible as small, inspectable recipes before a runtime executes them. The important lesson for =poo-flow= sandbox work is not the Bass language implementation. The lesson is the user-facing shape:

- users write short scenario rows;
- rows are ordinary data before execution;
- runtime images, cache, mounts, and host access remain explicit;
- backends do the heavy execution outside the small user interface.

For =poo-flow=, agent sandbox profiles should follow the same rule. A profile is an inert POO recipe that names the backend, network policy, capabilities, resource policy, and metadata. Projection to backend descriptors is a later step. Runtime execution remains owned by =marlin-agent-core= or the selected backend adapter.

** 10.06.18.20 User Surface
Root =init.ss= keeps module selection separate from sandbox profile recipes. The public user surface is concrete module loading; the facade builds the canonical =users= profile from those rows. Built-in sandbox profile recipes live upstream in =src/modules/agent-sandbox/config.ss=, and downstream user files load those defaults instead of spelling out backend policy rows.

#+begin_src scheme
(use-module nono-sandbox +nono +doctor)
(use-module cubeSandbox +cube +doctor)

poo-flow-default-sandbox-profiles
(poo-flow-default-sandbox-profile-presentation)
#+end_src

The public user row is module-level: =(use-module nono-sandbox ...)=. The full profile construction layer remains upstream so agents or higher profiles can generate one =users= profile without making the user understand descriptor catalogs, C3 slot resolution internals, or Marlin runtime manifests.

** 10.06.18.30 Boundary
The boundary is:

- concrete =(use-module ...)= rows select sandbox modules such as =nono-sandbox= and =cubeSandbox=;
- =poo-flow-default-sandbox-profiles= declares upstream profile recipes consumed by those modules or their runtime bridges;
- downstream user config may select or override profile names, but it should not own built-in backend policy rows;
- =pooFlowSandboxProfilesPresentation= gives tools a shallow receipt with names, backend refs, runtime handoff intent, runtime summaries, and handoff summaries;
- =poo-flow-sandbox-profile->profile= projects one recipe into the existing validated sandbox profile contract;
- =agent-sandbox-profile-runtime-summary= and =agent-sandbox-profile-handoff-summary= live under =src/modules/agent-sandbox/profile.ss= so workflow and loop code consume sandbox-owned projection data instead of reparsing resource policy rows;
- no package installation, descriptor realization, or runtime command execution happens in the user interface layer.

** 10.06.18.35 Runtime Summary Ownership
Sandbox profile projection is owned by the sandbox modules, not by =loop-engine= or the module-system base presentation layer. The concrete split is:
- =src/modules/sandbox-core/profile.ss= owns generic POO profile row resolution, inheritance, and merge operators;
- =src/modules/agent-sandbox/config.ss= owns user-facing POO profile recipes and projects those recipes into validated agent sandbox profiles;
- =src/modules/agent-sandbox/profile.ss= owns the contract-level runtime summary and handoff summary, including filesystem structure, diagnostics, validation status, and =runtime-executed #f= evidence.

Workflow, CI, and loop-engine features should depend on these summaries as data. They should not infer filesystem mounts, backend refs, or validation state from raw user rows.
=loop-engine= consumes sandbox summaries through a config-level catalog. If a loop declares =(profile . ci/build)= and the same config contributes a matching POO sandbox profile, the runtime manifest request carries =sandbox-runtime-summaries= and =sandbox-handoff-summaries=. If the profile ref is not visible in the config catalog, the manifest keeps =sandbox-unresolved-profile-refs= so an agent or doctor can report the missing profile without inventing a runtime policy.

** 10.06.18.40 POO Rule
The public profile value stays a POO object. Alists are presentation products only. This follows the gerbil-poo policy surface: macros are thin syntax bridges, while object/protocol slots keep semantic state. Future nono and cubeSandbox specialization should therefore extend the profile object family or runtime bridge projection instead of making user init rows parse backend-specific strings.

** 10.06.18.50 Delivery State
Delivered in this slice:

- =src/modules/agent-sandbox/config.ss= owns the profile DSL and presentation;
- =src/modules/agent-sandbox/config.ss= owns the built-in =agent/nono= and =agent/cube= defaults;
- =src/module-system/profile-config.ss= exposes the upstream DSL and defaults to downstream init files;
- =user-interface/init.ss= remains a module-loading surface only;
- =src/module-system/profile-config.ss= builds the canonical =users= profile projection from root declarations;
- =src/module-system/base.ss= projects loop-engine sandbox profile refs into sandbox-owned runtime/handoff summaries when the selected config exposes a matching profile catalog;
- focused tests cover inert recipes, upstream runtime intent presentation, runtime summary projection, handoff summary projection, and root config loading.
