* 10.06.56.10 Purpose

This document records the support evidence behind the POO Flow proof direction.
It is not a plan and it is not a replacement for Lean code.  It explains why AI
strategy/policy needs mathematical checking, what external research and
frameworks tell us, and how those signals map back to Scheme facts and Lean
obligations in this repository.

The core claim is:

#+begin_src latex
\[
\operatorname{AgentGeneratedPolicy}(p)
  \not\Rightarrow \operatorname{SafePolicy}(p)
\]
\[
\operatorname{AgentGeneratedPolicy}(p)
  \land \operatorname{SchemeFacts}(p)
  \land \operatorname{LeanDischarge}(p)
  \Rightarrow \operatorname{ProofBackedPolicy}(p)
\]
#+end_src

This distinction matters because POO Flow is designed for complex composition:
shared profiles, POO/C3 linearized policy objects, session lifecycle links,
sandbox capability scopes, loop policies, subagent dispatch, workflow
dependencies, and runtime handoff.  That is exactly the area where generated
configuration can look plausible while violating a hidden global condition.

* 10.06.56.20 Evidence Levels

External research and production frameworks are useful, but they do not own POO
Flow truth.  We need a strict evidence ladder.

| Level | Name | Owner | Can Accept Runtime Action? |
|-------+------+-------+----------------------------|
| E0 | External claim | Paper, framework, website, benchmark | No |
| E1 | Reference observation | POO Flow docs/research note | No |
| E2 | Candidate case | POO Flow scenario or test design | No |
| E3 | Scheme fact | POO Flow Scheme module/control plane | No |
| E4 | Lean discharge | POO Flow proof package | Yes, for the proven obligation |
| E5 | Runtime gate | Runtime projection or Marlin handoff checker | Yes, if tied to E4 evidence |

#+begin_src latex
\[
\operatorname{ExternalReference}(r)
  \Rightarrow \operatorname{CandidateCase}(r)
\]
\[
\operatorname{ExternalReference}(r)
  \not\Rightarrow \operatorname{DischargedObligation}(r)
\]
\[
\operatorname{AcceptAction}(a)
  \Rightarrow
    \exists o,\;
      \operatorname{Required}(a,o)
      \land \operatorname{Discharged}(o)
\]
#+end_src

The practical rule is simple: OpenRATH, Flue, ReAct, Reflexion, ToolEmu,
AgentDojo, Proof-Carrying Code, and Proof-Carrying Authentication can shape the
questions we ask.  They cannot answer a POO Flow proof obligation unless the
obligation is represented by local Scheme facts and discharged in Lean.

* 10.06.56.30 Why General Agents Are Not Enough

General coding agents and agent harnesses are strong at generation, execution,
tool use, and iteration.  They do not, by default, solve the POO Flow
strategy/policy rigor problem.

| System class | Strength | Missing default guarantee for POO Flow |
|--------------+----------+----------------------------------------|
| Codex/Claude-style coding agents | Generate code, inspect files, run tools, iterate with feedback. | They can produce policy text, but the text is not automatically a typed proof about POO/C3 order, session lineage, sandbox scope, or loop invariants. |
| ReAct-style agents | Interleave reasoning traces and actions. | A trace is inspectable, but it is not a machine-checked theorem. |
| Reflexion-style agents | Use verbal feedback and memory to improve later attempts. | Reflection improves search behavior, but it does not certify that a strategy preserves invariants. |
| Pi/Flue-style harnesses | Provide durable sessions, tools, skills, sandbox runtime, and deployment harness. | Durability and harness structure do not prove that a generated policy is semantically safe. |
| OpenRATH-style frameworks | Make Session, Sandbox, Memory, Tool, Agent, Workflow, and Selector explicit. | Runtime vocabulary helps us model cases, but POO Flow still owns its Scheme facts and Lean obligations. |

The proof problem is therefore not "can an agent write code?"  The problem is
"can a generated strategy be rejected before it violates the control-plane
contract?"

#+begin_src latex
\[
\operatorname{CanWrite}(agent,p)
  \not\Rightarrow \operatorname{CanProve}(agent,p)
\]
\[
\operatorname{CanRun}(harness,p)
  \not\Rightarrow \operatorname{CanProveSafe}(p,\Sigma)
\]
#+end_src

* 10.06.56.40 Research Evidence

** 10.06.56.41 ReAct

[[https://arxiv.org/abs/2210.03629][ReAct]] shows that interleaving reasoning traces with actions helps LLMs update
plans, handle exceptions, and interact with external environments.  This is
important evidence for observability: a strategy should expose why an action was
chosen and which observations shaped it.

POO Flow should adopt the observability lesson, not mistake the trace for a
proof.

| ReAct signal | POO Flow implication |
|--------------+----------------------|
| Reasoning and acting are interleaved. | A loop policy should preserve action rationale and observation facts. |
| External observations can update plans. | Runtime observations must become typed facts before they affect policy. |
| Traces improve human interpretability. | Reviewability is useful, but Lean discharge is still required for critical obligations. |

#+begin_src latex
\[
\operatorname{Trace}(a)
  \Rightarrow \operatorname{Inspectable}(a)
\]
\[
\operatorname{Trace}(a)
  \not\Rightarrow \operatorname{Safe}(a)
\]
#+end_src

** 10.06.56.42 Reflexion

[[https://arxiv.org/abs/2303.11366][Reflexion]] shows that agents can improve by reflecting on feedback and storing
verbal memory.  That supports a useful design point: POO Flow should let agents
learn from rejected obligations and carry rejection receipts into later
attempts.

The boundary is still strict.  Reflection memory is evidence for search and
repair.  It is not evidence that a policy is correct.

| Reflexion signal | POO Flow implication |
|------------------+----------------------|
| Feedback can improve later decisions. | Rejected obligations should produce structured repair hints. |
| Memory affects future attempts. | Reuse of memory must be scoped to session and sandbox policy. |
| Learning happens without changing model weights. | The control plane can improve agent behavior without trusting the model more. |

#+begin_src latex
\[
\operatorname{Rejected}(o) \land \operatorname{RepairHint}(o,h)
  \Rightarrow \operatorname{CandidateRevision}(h)
\]
\[
\operatorname{CandidateRevision}(h)
  \not\Rightarrow \operatorname{Discharged}(o)
\]
#+end_src

** 10.06.56.43 ToolEmu

[[https://arxiv.org/abs/2309.15817][ToolEmu]] is evidence that tool-using language agents have realistic failure
modes with serious consequences.  The paper reports that a large fraction of
identified failures were judged valid real-world failures, and even the safest
tested agent still exhibited failures under the evaluator.

For POO Flow, the lesson is that tool access cannot be treated as a normal
configuration field.  It is a proof-relevant capability.

| ToolEmu signal | POO Flow implication |
|----------------+----------------------|
| Tool use amplifies risks such as data leakage and financial loss. | Tool capabilities belong in sandbox facts and proof obligations. |
| Long-tail risky cases are hard to enumerate manually. | Scenario benchmarks should generate obligations, not only examples. |
| Evaluation can discover failures after the fact. | Runtime gates should reject missing proof before the tool call. |

#+begin_src latex
\[
\operatorname{ToolCall}(a,t)
  \Rightarrow
    \operatorname{AllowedTool}(\operatorname{sandbox}(a),t)
    \land \operatorname{Discharged}(\operatorname{ToolObligation}(a,t))
\]
#+end_src

** 10.06.56.44 AgentDojo

[[https://arxiv.org/abs/2406.13352][AgentDojo]] shows that AI agents using tools over untrusted data are vulnerable
to prompt injection and that realistic task suites can reveal both utility
failures and security failures.

For POO Flow, untrusted tool data must not be allowed to rewrite policy.  It can
be an observation, but it cannot become an authorization fact unless a local
owner module projects it into a typed fact with an explicit trust boundary.

| AgentDojo signal | POO Flow implication |
|------------------+----------------------|
| Tool-returned data can hijack agent behavior. | Observations and policy instructions must be separated. |
| Realistic tasks expose failures even without attacks. | Proof should cover ordinary lifecycle and dependency mistakes, not only adversarial cases. |
| Security tests are extensible and adaptive. | POO Flow scenario benchmarks should evolve into obligation generators. |

#+begin_src latex
\[
\operatorname{UntrustedObservation}(x)
  \not\Rightarrow \operatorname{PolicyInstruction}(x)
\]
\[
\operatorname{PolicyInstruction}(x)
  \Rightarrow \operatorname{OwnedByScheme}(x)
       \land \operatorname{TrustBoundary}(x)
\]
#+end_src

** 10.06.56.45 Proof-Carrying Code

[[https://dl.acm.org/doi/10.1145/263699.263712][Proof-Carrying Code]] gives the architectural precedent: untrusted code can be
accepted by a host only when it carries a proof that the host can check against
its safety policy.

POO Flow does not need to prove a full Scheme interpreter to use this idea.  We
need policy-carrying strategy: the generated policy action carries facts and a
proof certificate for the specific control-plane obligation being exercised.

#+begin_src latex
\[
\operatorname{PolicyAction}(a)
  \land \operatorname{CarriesProof}(a,\pi)
  \land \operatorname{Check}(\pi,\operatorname{SafetyPolicy})
  \Rightarrow \operatorname{AcceptableAction}(a)
\]
#+end_src

** 10.06.56.46 Proof-Carrying Authentication

[[https://www.cs.princeton.edu/~appel/papers/says.pdf][Proof-Carrying Authentication]] is especially close to POO Flow because it
places the burden of proof on the requester and lets the receiver check the
proof before authorizing access.

That maps directly to session, sandbox, and subagent actions:

| Request | Proof-carrying form |
|---------+---------------------|
| Start session | request carries lineage and parent policy evidence |
| Launch subagent | request carries child scope, allowed tools, and handoff evidence |
| Open sandbox | request carries capability subset or refinement proof |
| Advance loop | request carries transition precondition and invariant preservation proof |
| Runtime handoff | request carries boundary proof that Scheme emitted a descriptor, not a claim of execution |

#+begin_src latex
\[
\operatorname{Requester}(r) \land \operatorname{Requests}(r,a)
  \Rightarrow \operatorname{MustSupplyProof}(r,a)
\]
\[
\operatorname{Receiver}(v) \land \operatorname{Receives}(v,a,\pi)
  \Rightarrow \operatorname{CheckBeforeAuthorize}(v,a,\pi)
\]
#+end_src

* 10.06.56.50 Framework Evidence

** 10.06.56.51 OpenRATH

[[https://github.com/Rath-Team/OpenRath][OpenRATH]] is useful because it makes agent runtime state explicit.  Its
vocabulary includes Session, Sandbox, Memory, Tool, Agent, Workflow, and
Selector.  It also emphasizes that large agent systems need many agents, many
branchable sessions, durable memory, sandboxed execution, and traceable lineage.

This is strong reference evidence for POO Flow's proof axes:

| OpenRATH runtime object | POO Flow proof axis |
|-------------------------+---------------------|
| Session | session lineage, branch ownership, handoff graph |
| Sandbox / Backend | capability scope, file/tool/secret boundary |
| Memory | session-scoped recall, write ownership, replay boundary |
| Tool | schema, allowed call, sandbox placement |
| Agent | transformation from one session state to another |
| Workflow | dependency graph, composition, nested flow |
| Selector | dynamic branch choice, loop transition, policy guard |

The boundary is equally important:

#+begin_src latex
\[
\operatorname{OpenRATHObject}(x)
  \Rightarrow \operatorname{ReferenceAxis}(x)
\]
\[
\operatorname{OpenRATHObject}(x)
  \not\Rightarrow \operatorname{POOFlowProofFact}(x)
\]
#+end_src

OpenRATH should influence scenario selection and vocabulary.  It should not be
used as a proof oracle.  POO Flow must still emit local facts from Scheme and
prove them in Lean.

** 10.06.56.52 Flue And Pi-Style Harnesses

[[https://flueframework.com/][Flue]] is useful because it shows the harness side of modern agents: durable
sessions, tools, skills, filesystem access, sandbox runtime, database,
deployment harness, and recovery after interruption.

This gives POO Flow a concrete checklist for runtime proof pressure:

| Harness feature | POO Flow obligation |
|-----------------+---------------------|
| Durable session | resume facts are still valid under the current policy version |
| Tool list | every tool call is allowed by sandbox scope |
| Skill list | skills cannot silently broaden capability |
| Filesystem access | file paths are inside the declared scope |
| Sandbox runtime | execution placement matches policy |
| Recovery | resumed loop state preserves invariant |
| Deployment harness | runtime handoff does not claim local execution |

#+begin_src latex
\[
\operatorname{Resume}(s)
  \Rightarrow
    \operatorname{SamePolicyVersion}(s)
    \lor \operatorname{MigrationProof}(s)
\]
\[
\operatorname{SkillEnabled}(s,k)
  \Rightarrow
    \operatorname{Capabilities}(k)
      \subseteq \operatorname{SandboxScope}(s)
\]
#+end_src

Again, the framework is not the proof.  It tells us which runtime failures are
realistic enough to deserve proof obligations.

* 10.06.56.60 Failure Modes To Prove Against

The support evidence above points to concrete failure modes.  These are not
abstract safety slogans; they are proof targets.

| Failure mode | Typical source | Required POO Flow obligation |
|--------------+----------------+------------------------------|
| Session lineage collapse | OpenRATH-style multi-session branching | child session has exactly one declared parent or an explicit merge proof |
| Sandbox scope widening | ToolEmu, Flue sandbox/tool surfaces | child sandbox scope is a subset or justified refinement of parent scope |
| Tool observation becomes instruction | AgentDojo prompt injection | untrusted observation cannot become policy instruction without Scheme owner projection |
| Loop starts without readiness | POO Flow workflow/session cases | transition preconditions are satisfied before start |
| Retry loses invariant | Reflexion-style feedback loops | retry transition preserves invariant or emits rejected obligation |
| Profile override broadens policy | POO/C3 composition | linearized override cannot expand forbidden parent capability |
| Selector chooses unsafe branch | OpenRATH selector/runtime routing | selected branch must satisfy policy guard and dependency facts |
| Runtime handoff overclaims execution | Flue/Pi-style harness boundary | Scheme may emit descriptor, but execution claim belongs to runtime receipt |
| External framework treated as truth | OpenRATH/Flue reference misuse | external observation can only become candidate case, never proof fact |

* 10.06.56.70 Scheme Fact Requirements

To turn support evidence into proof work, Scheme modules must emit typed facts
at the right boundary.  A JSON manifest is not enough because it preserves shape
but loses type intent, owner, and proof target.

Required fact families:

| Fact family | Example fields | Proof use |
|-------------+----------------+-----------|
| Session facts | session id, parent id, branch id, handoff id, owner module | lineage, acyclicity, handoff authorization |
| Sandbox facts | tool set, file roots, secret refs, backend, runtime placement | subset/refinement, tool authorization |
| Loop facts | state, transition, readiness, retry budget, terminal condition | progress, invariant preservation |
| Profile facts | profile id, parent profiles, linearized order, overridden slots | deterministic precedence, no unsafe broadening |
| Workflow facts | module id, dependencies, required artifacts, readiness facts | start preconditions, dependency closure |
| Runtime facts | handoff descriptor, receipt expectation, runtime owner | no false execution claim, durable boundary |
| Evidence facts | source owner, observation class, adopted fact id | reference evidence cannot bypass local proof |

#+begin_src latex
\[
\operatorname{Fact}(f) \Rightarrow
  \operatorname{Owner}(f)
  \land \operatorname{Kind}(f)
  \land \operatorname{TargetObligation}(f)
\]
\[
\operatorname{RawShape}(x)
  \not\Rightarrow \operatorname{Fact}(x)
\]
#+end_src

This is why support evidence must eventually be reflected in Scheme object
contracts and Lean types.  The document layer explains the risk.  The Scheme
layer emits the fact.  The Lean layer checks the proposition.

* 10.06.56.80 Lean Obligation Families

The first useful Lean layer should focus on obligations that expose real design
bugs.  The goal is not to prove every interpreter rule.  The goal is to make the
control-plane claims precise enough that bad strategies fail to typecheck or
fail theorem discharge.

| Obligation family | Informal theorem |
|-------------------+------------------|
| SessionLineage | child session inheritance is explicit, acyclic, and scoped |
| SandboxSubset | child sandbox cannot exceed parent capabilities without refinement proof |
| ToolCallAllowed | every tool call has a sandbox fact and an allowed tool proof |
| LoopProgress | every nonterminal transition either advances a measure or emits a rejection |
| LoopInvariant | retry and resume preserve required session/sandbox facts |
| ProfileLinearization | POO/C3 order is deterministic and override provenance is visible |
| NoPolicyBroadening | profile or selector override cannot broaden forbidden capability |
| DependencyStart | module start requires all dependency facts |
| RuntimeHandoffTruth | Scheme handoff descriptor is not runtime execution evidence |
| ReferenceEvidenceBoundary | external observations cannot directly satisfy local proof facts |

#+begin_src lean
structure SessionFact where
  session : SessionId
  parent  : Option SessionId
  scope   : Scope

structure SandboxFact where
  ownerSession : SessionId
  tools        : Finset ToolId
  files        : Scope

def ChildScopeAllowed (parent child : SandboxFact) : Prop :=
  child.tools \subseteq parent.tools /\ child.files \subseteq parent.files

theorem subagent_scope_safe
    (parent child : SandboxFact)
    (h : ChildScopeAllowed parent child) :
    child.tools \subseteq parent.tools := h.left
#+end_src

That Lean sketch is deliberately small, but it points to the correct direction:
prove properties over typed facts, not over an untyped JSON blob.

* 10.06.56.90 Evidence-To-Implementation Rule

When a paper or framework exposes a useful idea, the implementation path is:

1. Record the external observation as support evidence.
2. Convert it into a POO Flow candidate case.
3. Identify the Scheme owner that can emit the required facts.
4. Add or refine the Lean obligation.
5. Gate runtime projection on the discharged obligation.

#+begin_src latex
\[
\operatorname{UsefulReference}(r)
  \Rightarrow
    \operatorname{CandidateCase}(r)
    \land \operatorname{NeedsSchemeOwner}(r)
    \land \operatorname{NeedsLeanObligation}(r)
\]
\[
\operatorname{AdoptedFact}(r,f)
  \Rightarrow
    \operatorname{OwnedByPOOFlow}(f)
    \land \operatorname{MappedToObligation}(f)
\]
#+end_src

This keeps the project honest.  We can learn from OpenRATH and Flue without
outsourcing correctness to them.  We can learn from ReAct and Reflexion without
mistaking agent traces for proof.  We can learn from ToolEmu and AgentDojo
without accepting post-hoc testing as a replacement for runtime proof gates.  We
can reuse the proof-carrying architecture without proving the full Scheme
interpreter.
