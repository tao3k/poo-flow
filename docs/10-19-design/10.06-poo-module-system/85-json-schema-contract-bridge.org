* 10.06.85 JSON Schema Contract Bridge

This RFC defines the P0 bridge from JSON Schema documents to Gerbil Scheme
contracts in =poo-flow=. The design is inspired by
=.data/json-schema-to-nickel=, but the target is not a Nickel-like syntax
emitter. The target is the existing POO Flow contract spine:

#+begin_src text
JSON-like schema value
  -> normalized schema IR
  -> PooFlowObjectTypeContract / PooFlowSlotContract
  -> type facts / Lean fact contracts / bounded receipt
#+end_src

** 10.06.85.10 Design Constraints

- JSON Schema is an import boundary, not a second user-facing configuration
  language.
- Internal conversion is pure Scheme data. Runtime execution, external schema
  fetching, and large generator pipelines stay outside this owner.
- Generated contract surfaces reuse =src/utilities/contracts.ss= and
  =src/type-facts/objects.ss=.
- Raw external schemas that affect shipped contracts are pinned under
  =schemas/json/=. =.data= remains only a research/cache area.
- Unsupported keywords and unsupported references are visible diagnostics.
  They must not silently pretend to be fully validated.
- Final manifest, Marlin, test, and doctor surfaces use bounded alist
  projection from fixed records.

** 10.06.85.20 Upstream Evidence

=json-schema-to-nickel= uses a three-stage conversion:

1. parse JSON Schema into an explicit IR instead of emitting target code
   directly;
2. repeatedly apply semantics-preserving transforms, such as flattening
   logical operators, merging =required= with property optionality, inlining
   selected references, and expanding small =patternProperties=;
3. emit inspectable lazy record contracts when possible and fall back to eager
   predicate checks only when structure is not representable.

POO Flow should adopt the same structure-first rule. The emitted Gerbil object
contract must remain inspectable by the harness, type-fact projection, doctor
reports, and future proof tooling.

** 10.06.85.30 P0 Scope

P0 supports a bounded draft-07 subset:

- =type= for =object=, =array=, =string=, =number=, =integer=, =boolean=, and
  =null=;
- homogeneous array =items= as a child schema predicate;
- =properties= and =required= for object fields;
- =additionalProperties= as a recursive map-value schema, including boolean
  =#t= and =#f= forms;
- =enum= and =const= as generated predicates;
- nullable =anyOf= when one branch is =null=;
- =allOf= as conjunction and =oneOf= as exactly-one predicate composition;
- =minLength=, =maxLength=, =minimum=, =maximum=,
  =exclusiveMinimum=, =exclusiveMaximum=, =minItems=, =maxItems=,
  =minProperties=, =maxProperties=, and =pattern= as metadata-owned generated
  predicate constraints;
- =patternProperties= as recursive map-value schemas for dynamic object keys;
- local references of the form =#/definitions/<name>= and =#/$defs/<name>=.

P0 does not fetch remote references, resolve =$id= anchors, or claim full
draft-07 coverage. Those become diagnostics and a safe fallback predicate.

** 10.06.85.40 Source Ownership

The source files are:

#+begin_src text
src/contract/json-schema-ir.ss
src/contract/json-schema-source.ss
src/contract/functional.ss
src/contract/json-schema-constraints.ss
src/contract/json-schema-normalize.ss
src/contract/json-schema-emit.ss
src/contract/json-schema-receipt.ss
src/contract/json-schema-validate.ss
src/modules/funflow/github-ci-contract.ss
t/support/json-schema-contract-performance.ss
t/json-schema-contract-performance-test.ss
packages/python-runtime/src/poo_flow_runtime/json_schema_contract.py
schemas/json/github-workflow.json
#+end_src

=json-schema-ir.ss= owns fixed IR records. =json-schema-source.ss= decodes
pinned JSON files into symbol-key walists without owning schema semantics.
=json-schema-normalize.ss= lowers
JSON-like alists into normalized IR and diagnostic rows. =json-schema-emit.ss=
converts normalized IR into existing POO Flow object/slot contract records.
=json-schema-receipt.ss= packages diagnostics, type facts, Lean fact contracts,
and final alist projection. =json-schema-validate.ss= applies generated object
contracts to POO objects or bounded alists and returns a fixed validation
receipt. The validator is schema-guided: object-shaped alists and array-shaped
lists can both use Scheme lists, so the IR node kind decides whether to treat a
list as an object row set or an array value. =functional.ss= owns
contract-boundary functional plumbing such as key normalization, JSON-like
alist lookup, SRFI-backed sequence folds, and diagnostic pair mapping.
=json-schema-constraints.ss=
owns validation keyword metadata and generated constraint predicate
composition.

=t/support/json-schema-contract-performance.ss= builds deterministic recursive
contract workloads. The paired =t/json-schema-contract-performance-test.ss=
gate measures both the boolean hot path and the receipt/projection path in
microseconds. Large GitHub CI fixtures remain as a scale check, but =jobs= is
only one fixture: the same validator recurses through any normalized
=properties=, =additionalProperties=, =patternProperties=, array =items=, and
logical child schema. =json_schema_contract.py= is the Python-side source
generator for users that want to bring their own JSON Schema and emit a Scheme
contract artifact for a custom Funflow stage or profile.

=schemas/json/github-workflow.json= pins the raw SchemaStore GitHub Workflow
schema used as evidence for the =funflow/github-ci= profile contract. The
Scheme contract intentionally projects a maintainable top-level profile shape
instead of copying the whole schema into code. The audit metadata records the
upstream URL, repo path, byte size, and sha256.

Generic helpers remain in =src/utilities/functional.ss=. That owner should use
Gerbil SRFI libraries such as =:std/srfi/1= and =:std/srfi/13= for list and
string primitives, then expose project-owned names for business modules.
The contract-local helper layer follows the same rule, but narrows the names to
JSON-like contract boundaries: key normalization, object lookup, strict boolean
membership scans, SRFI-13 prefix handling, and ordered diagnostic folds.
Schema-specific projection code may still use direct SRFI-1 call heads such as
=map= and =filter-map= when that keeps parser-owned expression-level evidence
visible; reusable plumbing should move back into one of the functional helper
owners.

** 10.06.85.50 User Boundary

The first public shape is intentionally small:

#+begin_src scheme
(poo-flow-json-schema->contract-artifact
 schema
 '((source-ref . github-workflow-schema)
   (owner . workflow)
   (object-kind . PooFlowGithubWorkflow)))
#+end_src

The artifact contains the generated object contract, normalized schema,
diagnostics, type facts, and Lean fact contract rows. Later CLI or runtime
adapters may parse JSON files before calling this function, but JSON parsing is
not the semantic owner.

The first module-level profile integration is:

#+begin_src scheme
(poo-flow-funflow-github-ci-validate-workflow->alist
 '((on . ("push" "pull_request"))
   (jobs . ((build . ((runs-on . "macos-latest")
                      (steps . (((run . "gxpkg build -g"))))))))))
#+end_src

This checks the generated =funflow/github-ci/workflow= artifact recursively.
Dynamic =jobs.<job_id>= entries are matched by =patternProperties=, each job is
validated against the job schema, and nested arrays such as =steps= are checked
through their item schemas. Missing or invalid deep paths are reported as
stable slash-addressed slots such as =jobs/build/runs-on= and
=jobs/build/steps/0/id=.

The bridge also validates non-GitHub recursive maps such as
=stages.<stage_id>.tasks.<task_id>.command= through =patternProperties= and
=profiles.<profile>.limits.<resource>.value= through =additionalProperties=.
This is the important contract: users can provide a JSON Schema for a custom
Funflow stage or profile, and POO Flow normalizes that schema into one
recursive contract artifact rather than adding business-specific validation
code.

Python users can generate a Scheme contract artifact from a JSON Schema file:

#+begin_src sh
uv run --project packages/python-runtime poo-flow-jsonschema-contract \
  schemas/json/custom-stage.json \
  --name custom-stage \
  --object-key funflow/custom-stage \
  --output user-interface/custom/my-module/contracts/custom-stage.ss
#+end_src

The generated Scheme still calls =poo-flow-json-schema->contract-artifact=.
Python does not own contract semantics; it only projects user-provided JSON
Schema into Scheme source that the normal POO Flow contract bridge can
normalize, validate, and inspect.

** 10.06.85.60 Verification

The P0 gate is:

- compile =src/contract/functional.ss= and the
  =src/contract/json-schema-*.ss= files through =gxpkg env=;
- run a small Scheme smoke that converts an object schema with =required=,
  =enum=, =const=, nullable =anyOf=, =oneOf=, array =items=, and local
  =#/definitions= and =#/$defs= plus scalar, array, object, and pattern bounds;
- confirm emitted slot contracts project to type facts and Lean fact contracts;
- confirm =funflow/github-ci= validates both POO objects and bounded alists
  through the same generated object contract;
- confirm dynamic map values recurse through both GitHub CI jobs and generic
  user stage/task schemas plus additional-property profile/limit schemas;
- confirm recursive map-value validation stays inside the benchmark budget in
  =t/json-schema-contract-performance-test.ss=, including microsecond average
  gates for the boolean hot path and receipt path. The current gate keeps the
  small recursive hot path under =50us= and the receipt/projection path under
  =90us= on the best-of-five focused run;
- confirm the Python =poo-flow-jsonschema-contract= CLI emits Scheme source
  that =gxc= can compile;
- confirm unsupported keywords and external references appear as diagnostics.

Future gates can import selected JSON Schema Test Suite examples once a JSON
parser/adapter owner exists.
