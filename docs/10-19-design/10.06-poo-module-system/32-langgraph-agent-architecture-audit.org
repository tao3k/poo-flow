* 10.06.32 LangGraph Agent Architecture Audit
** 10.06.32.10 Source Snapshot
LangGraph was cloned for local research under =.data/langgraph=.
- Repository: =https://github.com/langchain-ai/langgraph=
- Local path: =.data/langgraph=
- Clone command: =gh repo clone langchain-ai/langgraph .data/langgraph -- --depth 1=
- Snapshot: =5931a5f=, 2026-06-29, =release(langgraph): 1.2.7 (#8223)=
- Audit date: 2026-06-30

This audit uses LangGraph as engineering comparison material for =poo-flow=. It does not treat LangGraph as a design target to copy. The useful evidence is how LangGraph separates graph declarations, runtime injection, durable state, human interrupts, prebuilt agent loops, and deployment tooling.
** 10.06.32.20 Project Shape
LangGraph is a monorepo with library boundaries that are explicit in its own agent instructions:
- =libs/langgraph= owns the core stateful graph framework.
- =libs/checkpoint= owns base checkpoint interfaces.
- =libs/checkpoint-postgres= and =libs/checkpoint-sqlite= own concrete checkpoint persistence.
- =libs/prebuilt= owns high-level agent and tool helpers.
- =libs/cli= owns project creation, local development server, Docker build, and config-driven graph discovery.
- =libs/sdk-py= and =libs/sdk-js= own client APIs for the LangGraph Server API.

The dependency shape is only comparison evidence. It is not the =poo-flow= engineering boundary.

The =poo-flow= engineering boundary is the Doom Emacs-style module boundary:
- categories group module families;
- modules are enabled through the compact user interface;
- features are optional module facets;
- profiles and cases are user-owned declarations under modules;
- module owners maintain their own =config.ss=, =types.ss=, validators, projections, and runtime handoff helpers;
- =init.ss= stays small, like a Doom =init.el=: it selects modules and tunes exposed options, but it does not define upstream module internals.

LangGraph's package split is useful because it shows which concerns should not collapse into one runtime object. In =poo-flow= those concerns should be absorbed through module ownership, not copied as a repository/package split. Persistence, SDK, CLI, deployment, checkpoint policy, human audit, routing, and prebuilt agent loops should enter the system as module-owned POO objects and projections. Scheme remains the module control plane; runtime execution, durable storage, process isolation, and deployment stay with Marlin or other backend owners.
** 10.06.32.25 Module Boundary Translation
Translate LangGraph terms into =poo-flow= module terms before designing implementation:
- =libs/langgraph= core graph :: =workflow= / =funflow= module-owned POO objects.
- =libs/checkpoint= saver contract :: =session-durability= or =checkpoint-policy= feature under the session/workflow module.
- =libs/prebuilt= agent helpers :: optional module profiles or cases, not root abstractions.
- =libs/cli= project config and dev server :: backend/deployment projection outside =init.ss=.
- =Runtime= injected into nodes :: module-owned runtime handoff projection.
- =interrupt= and =Command(resume ...)= :: =human-audit= and resumable route features with hard validation.

This keeps the public surface aligned with the existing =use-module= direction:
#+begin_src scheme
(use-module workflow +loop-engine +human-audit)
(use-module sandbox +nono +doctor)
#+end_src

The module owns what those features mean. User init only selects and configures exposed extension points. If a capability is required for the module to exist, it should be a kernel/default profile under the module, not another =+= feature exposed to the user.
** 10.06.32.30 User Interface
The core LangGraph user interface is builder-oriented:
#+begin_src python
graph = (
    StateGraph(state_schema=State, context_schema=Context)
    .add_node("node", node)
    .add_edge(START, "node")
    .compile(checkpointer=checkpointer, store=store)
)
#+end_src

The important properties are:
- =StateGraph= is only a builder and is not executable until =compile=.
- node functions read a shared typed state and return partial state updates.
- reducers on state keys aggregate writes from multiple nodes.
- =compile= creates a runnable graph and accepts =checkpointer=, =store=, =interrupt_before=, =interrupt_after=, =debug=, and =name=.
- checkpointer plus =thread_id= is the durable execution key for pause, resume, replay, and time-travel debugging.

For =poo-flow=, this confirms that the user surface should stay compact and declarative, but our surface should be module-owned and POO-native rather than Python builder-native. A =poo-flow= user enables modules and writes profile/case POO objects inside those module surfaces. Module projections then play the role that LangGraph's =compile= plays: they turn declarations into validated descriptors and runtime handoff receipts without executing the runtime.
** 10.06.32.40 Runtime Injection
LangGraph has a run-scoped =Runtime= object injected into graph nodes. It bundles:
- =context= for static run dependencies such as user id or database handles;
- =store= for cross-thread memory;
- =stream_writer= for custom streaming output;
- =heartbeat= as a cooperative progress signal for idle timeout policies;
- =previous= for functional API previous result access;
- =execution_info= with checkpoint id, checkpoint namespace, task id, thread id, run id, node attempt, and first-attempt timestamp;
- =server_info= for deployment metadata;
- =control= for cooperative drain requests.

This is a useful comparison point for =poo-flow=. We should keep Scheme objects inert, but the projected runtime handoff should name the same classes of facts: context, store or memory owner, stream sink, heartbeat or progress semantics, execution identity, server/deployment metadata, and cooperative control state. Those should be validated descriptors, not hidden dynamic parameters.
** 10.06.32.50 Checkpoint And Durable Execution
LangGraph makes checkpointing a first-class contract. =BaseCheckpointSaver= exposes synchronous and asynchronous methods for:
- fetching a checkpoint or checkpoint tuple;
- listing checkpoints with filters and limits;
- storing checkpoints;
- storing intermediate writes linked to task ids;
- deleting, copying, and pruning threads;
- walking parent chains for delta-channel reconstruction.

The base contract says =thread_id= is the primary key. Without it the checkpointer cannot save state, resume from interrupts, or support time-travel debugging. =InMemorySaver= stores data under thread id, checkpoint namespace, checkpoint id, task id, writes, and blobs. It is explicitly debug/test-only; production use is delegated to persistent savers such as Postgres.

For =poo-flow=, the matching design is not to implement a Scheme checkpoint database. The matching design is to model checkpoint intent and session identity as POO objects:
- =session-id= or =thread-id=;
- checkpoint namespace;
- checkpoint policy;
- memory/store owner;
- write retention and pruning policy;
- runtime handoff owner.

The validator should reject resumable human loops or durable sessions that lack a checkpoint-capable runtime policy.
** 10.06.32.60 Human Loop And Resume
LangGraph has two human-loop layers.

The low-level layer is =interrupt(value)=. It pauses execution by raising a graph interrupt and surfaces =value= to the client. Resume is done with =Command(resume=...)=. The docs in the function are explicit: interrupts require a checkpointer, and resume restarts the node, re-executing its logic. Multiple interrupts in one node are matched by call order within that node task.

The prebuilt layer adds typed human review shapes:
- =ActionRequest= names an action and its args.
- =HumanInterrupt= combines an action request, a config describing allowed human actions, and an optional description.
- =HumanResponse= can be =accept=, =ignore=, =response=, or =edit=.

This validates the current =poo-flow= distinction between =loop-governor= and =human-audit=. A governor can be an agent or multi-agent control role. A human audit is a specific human intervention node derived from governance but requiring a real human response. In POO terms, =human-audit= should inherit governance slots but add stricter gates: checkpoint required, allowed responses declared, resume policy declared, and ordinary task overrides unable to remove the gate.
** 10.06.32.70 Routing And Fanout
LangGraph's =Command= can update graph state, resume interrupts, or route to another node or graph. =Send= sends a custom state packet to a specific node and is used for fanout patterns such as map-reduce. =create_react_agent(..., version="v2")= uses =Send= to distribute tool calls across multiple tool node instances.

The =poo-flow= analogue should not be raw =goto= commands in Scheme user config. It should be typed POO routing descriptors:
- deterministic node transition;
- delegated operation;
- fanout over inputs;
- parent-graph handoff;
- resumable command with required checkpoint policy.

These descriptors can project to Marlin runtime commands later. Scheme should validate and present them, not own concurrent execution.
** 10.06.32.80 Prebuilt Agent Loop
LangGraph's prebuilt =create_react_agent= creates a graph that cycles between an =agent= node and a =tools= node until no tool calls remain. It also exposes:
- dynamic model selection from state and runtime;
- prompt, structured response, pre-model hook, post-model hook;
- state schema and context schema;
- checkpointer and store;
- interrupt before or after =agent= or =tools=;
- debug mode;
- =v1= and =v2= tool execution modes.

This is useful as a warning. A prebuilt agent loop is ergonomic, but it hard-codes a particular LLM/tool loop. =poo-flow= should expose prebuilt cases only as module profiles or cases, not as the root abstraction. The root abstraction remains a POO object graph with profile, workflow, sandbox, session, route, governor, and human-audit roles.
** 10.06.32.90 CLI And Deployment Surface
LangGraph CLI uses a =langgraph.json= file:
#+begin_src json
{
  "dependencies": ["langchain_openai", "./your_package"],
  "graphs": {
    "my_graph": "./your_package/file.py:graph"
  },
  "env": "./.env",
  "python_version": "3.11",
  "pip_config_file": "./pip.conf",
  "dockerfile_lines": []
}
#+end_src

The CLI owns:
- =langgraph new= for templates;
- =langgraph dev= for local API server with hot reload;
- =langgraph up= for Docker server launch;
- =langgraph build= for Docker image build;
- =langgraph dockerfile= for generated deployment artifacts.

For =poo-flow= this reinforces that user project config and runtime/deployment config should remain separate. =init.ss= should stay clean and POO-native. Backend/runtime descriptors can be projected into Marlin-owned config or future deployment artifacts without putting Docker/server concerns into the Scheme user interface.
** 10.06.32.100 POO Flow Design Takeaways
LangGraph is useful precisely where it exposes clean engineering boundaries:
- Builder declarations are inert until compiled.
- Runtime dependencies are injected as a structured runtime object.
- Durable execution is gated by a checkpointer and a thread id.
- Human interruption is explicit and resumable through typed command data.
- Agent loops can be prebuilt, but they compile to ordinary graph nodes and edges.
- CLI/deployment is a separate package layer.

The =poo-flow= translation should be:
- module-owned POO objects express strategy, policy fragments, workflow roles, session identity, route intent, and human gates.
- Doom Emacs-style module selection remains the public engineering boundary; package or backend splits are implementation evidence, not the user-facing architecture.
- C3 determines deterministic composition order, but validators decide whether the final object is legal.
- safety gates, denylist union, permission narrowing, budget tightening, and human-gate monotonicity remain hard validator rules.
- runtime projection produces Marlin handoff descriptors with =runtime-executed = #f= in Scheme receipts.
- checkpoint and resume requirements are validated before any human-audit or resumable loop is accepted.
** 10.06.32.110 Next Implementation Targets
1. Add =checkpoint-policy= or =session-durability= as a module-owned POO object family, probably under the session/workflow boundary, attachable to sessions, loop engines, and human-audit gates.
2. Extend =human-audit= as a workflow/loop feature whose validation requires a checkpoint-capable policy and allowed response modes.
3. Add routing descriptors as module-owned POO objects for deterministic transition, delegated operation, fanout, parent handoff, and resume command.
4. Add a runtime injection projection owned by the relevant module, presenting context, memory store, stream sink, progress heartbeat, execution identity, and control/drain state as validated handoff facts.
5. Add a module-owned graph analysis layer over POO facts so workflow, loop, session, sandbox, and human-audit modules can share reachability, cycle, dependency-cone, and gate-coverage diagnostics.
6. Add a prebuilt agent-loop as a module profile or case, not as the root workflow abstraction.
7. Keep CLI/deployment descriptors out of =init.ss=; project init should load modules and tune exposed features, while Marlin-owned outputs receive compiled descriptors.
