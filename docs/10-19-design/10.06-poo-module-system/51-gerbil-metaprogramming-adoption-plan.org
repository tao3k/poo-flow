* 10.06.51 Gerbil Metaprogramming Adoption Plan
This document is an adoption plan for the rules in
=44-current-project-programming-style.org=. It does not define a second
programming style.

The project direction is:
#+begin_src text
POO and functional code are the public model.
Gerbil macros remove repeated Scheme structure.
Generated code expands to ordinary POO-native or functional code.
Alists appear only at final projection, receipt, snapshot, or ABI boundaries.
Heavy execution remains outside Scheme.
#+end_src

** 10.06.51.10 Style Contract From 10.06.44
Every metaprogramming change must satisfy the current programming style
contract before implementation:

1. Public extension surfaces stay POO-native: =.def=, =.o=, =.ref=,
   =.mix=, slot operators, prototype composition, and named functional
   helpers.
2. Macros are allowed only for mechanical structure whose expansion target is
   ordinary POO or functional code.
3. Runtime manifests, receipts, snapshots, and Marlin handoff payloads may use
   alists only as final projection payloads.
4. Scheme owns strategy, composition, validation, presentation, and report-only
   handoff data. Runtime execution, scheduling, subprocesses, persistence, and
   concurrency stay behind runtime or Marlin boundaries.

The practical test is simple: after macro expansion, the generated code should
look like boring hand-written project code. If understanding the model requires
memorizing a new DSL, the macro is wrong for this project.

** 10.06.51.20 Adoption Gate
Before adding or broadening a macro, the owner must answer yes to all of these:

- Is the repeated code structural rather than business logic?
- Does the expansion produce ordinary POO objects, functional constructors,
  validators, accessors, or final projection functions?
- Does the call site still name the domain object, slots, contracts, ownership,
  and runtime boundary clearly?
- Does the macro remove duplicated constructor, projection, validator, or
  manifest boilerplate without hiding extension slots from users?
- Is the macro owned by the same layer whose structure it generates?

Reject the macro when a normal function, higher-order helper, or small named
projection helper is clearer. Runtime filtering, folding, data-dependent
selection, and flow composition should usually remain functional code.

** 10.06.51.30 Macro Tier Ladder
Use the smallest Gerbil metaprogramming tool that carries the shape.

1. Declarative macros with =defrules= are the default for fixed repeated
   frames: constructor families, binding forms, direct syntax-to-POO lowering,
   and simple final projection declarations.
2. Procedural macros with =defsyntax=, =syntax-case=, =with-syntax=,
   =syntax->datum=, and =datum->syntax= are appropriate when the form must
   validate clauses, normalize options, generate related identifiers, or emit
   different expansions from the same public syntax.
3. Bounded compile-time generators are appropriate only when one declaration
   must create several related artifacts, such as a constructor, predicate,
   accessor set, contract table, projection function, and focused test hooks.

The macro tier is not a prestige ladder. A declarative macro is preferred when
it keeps the generated model obvious. A procedural macro is justified only when
it provides real shape validation or identifier generation that would otherwise
be duplicated by hand.

** 10.06.51.40 Ownership Map
Macro ownership follows the layer whose code is generated:

- =src/module-system/sandbox-backend-object-syntax.ss= owns sandbox backend
  object, profile, field-contract, capability, and backend registry families.
- =src/modules/session/policy-syntax.ss= owns session policy constructor,
  accessor, predicate, validation, and projection family generation.
- =src/modules/session/receipt-syntax.ss= owns record-backed session receipt
  accessor wrappers and record/object-backed final receipt projection
  generation outside policy families.
- =src/core/object-syntax.ss= owns repeated core role-backed POO object frames
  for descriptor, registry, category, and loop policy prototypes; constructor
  semantics stay in the local descriptor owner.
- =src/core/projection-syntax.ss= owns fixed core final projections for agent
  profile, harness, session, operation, workflow-run, dispatch receipt, runtime
  snapshot, execution policy, config preflight, runtime response, adapter
  result, and control-plane failure rows, plus fixed flow-strand descriptor and
  registry snapshots.
- =src/module-system/projection-syntax.ss= owns fixed module-system final
  projections for diagnostics, doctor receipts, source refs, resolver catalog
  entries, catalog snapshots, durable policy diagnostics, and durable policy
  receipts, strict observability rows, POO slot-authoring report rows, and
  user-module selection presentation rows, plus durable runtime store contract,
  backend, negotiation, and operation receipt rows, loader receipts, and user
  profile summary rows, durable recovery scenario rows, and concrete
  tool-core/memory-core object, catalog, manifest, validation, and durable job
  projection rows, plus graph topology, Funflow functional-DAG, sandbox profile,
  sandbox resource contract validation, module object-validation, and
  module field/type validation projection rows.
- =src/module-system/config-prototype-syntax.ss= owns repeated module config
  POO prototype declarations, kind-slot predicates, and mechanical
  prototype-to-constructor converters for public config prototype objects. It
  must not generate catalog fallback, manifest projection, graph derivation, or
  runtime handoff logic.
- =src/module-system/runtime-projection-syntax.ss= owns runtime manifest,
  receipt, and Marlin handoff projection generation when the output is a pure
  final boundary projection.
- =src/module-system/init-syntax.ss= owns public =use-module= syntax and any
  private branch-generation helpers.
- =src/module-system/load-syntax.ss= owns source-aware =load!= fragment
  resolution, compile-time fragment reading, object-fragment wrapping, and
  generated custom-module binding names. It must not own runtime module
  selection, descriptor realization, benchmark payloads, or public scenario
  behavior.
- Module-local =src/modules/*/config.ss= files may expose narrow macros only
  when they lower directly to POO objects or named functional constructors for
  that module.

Generated code must preserve the local owner boundary. A module-specific
contract remains in that module. A final ABI row remains at the projection
boundary. A public user scenario remains a scenario, not a macro demo.

** 10.06.51.50 Evidence From Current Owners
The current codebase already contains the approved pattern:

1. Sandbox backend object families use a declarative syntax owner to generate
   ordinary module objects and field contracts. This is the reference for fixed
   POO object families.
2. Session policy families use a syntax owner for repeated constructor,
   accessor, alist-helper, POO receipt accessor, and projection structure while
   keeping exported functions ordinary and POO-native.
3. Runtime projection syntax owns repeated final receipt and handoff projection
   shapes. It must not become a user authoring model.
4. =use-module= remains the public authoring syntax. Repeated internal branches
   may be generated only after the lower-level object and projection families
   are stable and tested.

The next reusable family should be a POO policy or projection declaration that
generates from one named spec:
- constructor frame;
- optional predicate and slot accessors;
- expansion-time shape checks for static clauses;
- runtime validations that call named project validators;
- ordered final =->alist= projection rows;
- focused owner-boundary tests.

** 10.06.51.70 Design Rules
- POO-native public extension is non-negotiable.
- Functional composition remains the normal route for flows and graph analysis.
- Macros remove repeated structure; they do not hide the model.
- Expansion-time validation is preferred for static syntax shape errors.
- Runtime validation remains for value checks known only after composition.
- Generated projections must name the ABI, report, receipt, or manifest
  contract they satisfy.
- A generated family has exactly one owner module and focused tests at that
  boundary.
- Advanced examples must show real POO relationships, including prototypes,
  inherited slots, extension, and slot operators when those are the point.

** 10.06.51.80 Verification Gates
Macro or compile-time generator changes must be verified at the owner boundary
before broad package verification.

Focused owner-boundary examples:
#+begin_src sh
gxtest t/sandbox-backend-object-family-test.ss
gxtest t/session-policy-contract-test.ss
gxtest t/user-interface-custom-loop-engine-runtime-manifest-test.ss
#+end_src

For a new macro family, add tests that prove:
- accepted syntax expands to ordinary POO-native or functional code;
- invalid static shape fails at expansion time when feasible;
- runtime validators still reject invalid composed values;
- final alist payloads appear only at projection boundaries;
- user-facing scenarios keep the compact =use-module= or POO object style.

The broad verification gate remains package-level:
#+begin_src sh
gxpkg build -R -O
#+end_src

Do not run broad verification as a substitute for owner-boundary tests. The
focused tests are what prove the macro did not move the model boundary.
