#+title: 10.06.04 poo-flow Module System Design

* 10.06.04 poo-flow Module System Design
** 10.06.04.05 Current Implementation Delta
The module system now includes the Marlin-style facade in =src/module-system.ss= while preserving the existing descriptor/catalog/activation API.

Implemented facade pieces:
- =poo-module-interface= with schema and metadata slots;
- =poo-string-required=, =poo-string-constant=, =poo-string-default=, and =poo-string-optional= schema helpers;
- =pooModules= and =defpoo-module= for interface/config construction;
- =poo-import=, =poo-imports=, =poo-local-source=, and =poo-source-ref= for structured imports;
- inline import closure before doctor and activation;
- =poo-module-option-configs=, =poo-module-option-schemas=, and validation receipts;
- =poo-option-append=, =poo-option-override=, =poo-option-conflict=, and merge receipts;
- =poo-module-apply=, =poo-module-evaluate=, =poo-module-workflow=, =pooEvalModules=, and =pooModuleSystemPresentation= for inspectable runtime-facing projection.

Direct symbol/string imports remain closed-world dependencies. Source refs and import specs remain metadata and profile values; they do not load Scheme files or manufacture runtime handlers.
** 10.06.04.10 Architecture
The module system is a thin layer over existing =poo-flow= registries.

#+begin_src text
source ref
  -> metadata for local/package/registry/generated origin

catalog
  -> binds source refs to already-built module descriptors
  -> resolves selected sources before doctor or activation

poo-flow module descriptor
  -> task-family registry contribution
  -> flow-declaration registry contribution
  -> module options
  -> explicit imports

doctor
  -> reports missing imports
  -> reports duplicate descriptor and option keys
  -> reports empty descriptor bundles

activation
  -> validates imports
  -> combines descriptor registries
  -> creates inspectable activation object
  -> lowers to run-config
#+end_src

The implementation lives under =src/module-system/*= and is re-exported by
=src/module-system.ss=. It imports core descriptor and config APIs, but core
does not import or re-export it.

** 10.06.04.15 Module Kernel, Factor, Plugin, And Fixture Boundary
Every module is made from four layers:

1. Kernel factors.
2. Module factors.
3. Plugin factors.
4. Fixtures and scenarios.

Doom Emacs is the right reference shape here.  Its strength is not that users
can override the core.  Its strength is that core implementation, feature
implementation, and macro implementation have clear jobs:

- core implementation stays maintained, stable, and mostly hidden;
- feature implementation adds named capabilities through declared module
  surfaces;
- macro implementation compresses authoring syntax without becoming the
  semantic owner;
- package/plugin behavior is incremental and goes through the module boundary;
- doctor/sync-style diagnostics keep the system robust when modules are added.

The detailed evidence and mapping live in
[[file:86-doom-emacs-core-module-boundary-research.org][86 Doom Emacs Core Module Boundary Research]].  This section records only the
design consequence.  Agents should read the dedicated research note before
changing module kernel, public factor, plugin, macro, fixture, or diagnostic
semantics.

The lesson from Doom is not to copy its dynamic Emacs runtime.  The lesson is
the module boundary:

- keep core semantics owned by the module;
- let users select named modules and public feature factors;
- use macros only to compress authoring into inspectable module data;
- keep sync/doctor-style operations as diagnostics, not as hidden semantic
  owners.

POO Flow should follow the same separation in Gerbil POO terms:

- kernel factors are the core implementation;
- public module factors are feature implementation;
- plugins are external upward extensions through declared slots;
- macros are authoring helpers that expand to inspectable POO descriptors,
  profiles, factors, receipts, or projections;
- fixtures and scenarios are proof, not configuration.

Kernel factors are internal, inherited, and required.  They include the pieces
without which the module is not a coherent feature.  For a session/flow module,
that means session value, composition strategy, graph strategy, lifecycle/event
defaults, resolver projection, runtime handoff projection, base validation, and
diagnostics.

Kernel factors are module-owned.  They are enabled by default when the module is
selected, and normal users should not see or compose them.  They can be
inspected by maintainers, tested by scenario fixtures, and used by advanced POO
authors only through explicitly marked internal APIs.

Kernel factors must not be user-overridable through normal =use-module=,
=compose=, or plugin routes.  If users can replace the kernel, then the module
no longer has a stable contract, and every downstream framework will pressure
the module into becoming a pile of ad hoc hooks.

Module factors are selectable or configurable parts of a complete module.  They
can be enabled, disabled, narrowed, or parameterized only when the module owner
declares them as public factors.  A public factor is not a kernel override.  It
is a stable extension point maintained by the module.

Examples:

- enable a memory factor;
- disable a telemetry factor;
- select a durable backend factor;
- narrow a network capability factor;
- select a result-validation factor.

The module owner decides which factors are public.  Public factors must have
contracts, defaults, validation phases, and projection behavior.

Macro implementation is a separate concern from both kernel and features.  A
macro may make a module pleasant to use, but it must expand into the same
module-owned data model.  It must not:

- run module activation;
- bypass kernel validation;
- expose private kernel hooks as user options;
- create a second alist or record DSL;
- silently change plugin attachment rules.

Plugins are upward extensions:

- a plugin contributes a new profile, transform, backend, validator, role
  policy, or projection;
- it attaches only through declared extension slots;
- it cannot remove kernel safety or required lifecycle behavior by default;
- it must publish capability and projection contracts;
- it is selected explicitly by the user or by a higher-level profile.

Fixtures and scenarios are not extension mechanisms.  They prove the kernel,
module factors, and plugin factors behave correctly.  They may expose internal
receipts and diagnostic projections, but they do not make internal hooks public.

The boundary is:

#+begin_src text
use-module funflow
  => loads module kernel defaults
  => exposes stable session/composition concepts
  => hides internal hooks

use-module audit-roles
  => adds plugin profiles
  => extends declared slots
  => stays outside the FunFlow kernel ontology
#+end_src

This keeps the public surface small.  Users select modules, compose stable
concepts, and select public extensions.  They do not compose kernel hooks.

Normal users can:

- select modules;
- select public module factors;
- configure declared public slots;
- add plugin profiles through declared extension slots;
- use high-level POO object syntax where a module exposes it.

Normal users cannot:

- replace kernel factors;
- bypass kernel validation;
- delete required lifecycle events;
- weaken required safety policy;
- swap resolver semantics;
- mutate internal handoff schemas;
- attach plugins to undeclared internal hooks.

Advanced POO authors may use lower-level object abstractions only when the
module explicitly marks an abstraction as advanced and stable.  Even then, the
object should extend a public factor or declared slot.  It should not override
the private kernel.

In short:

#+begin_src text
kernel = closed by default
public factor = configurable under contract
plugin = upward extension through declared slots
fixture = proof and diagnostics, not API
#+end_src

This extends the root user-interface contract: upstream modules own kernel
profiles; users should not repeat them just to get a working profile.
** 10.06.04.20 Core Types
*** PooModuleSourceRef
Source refs describe where a module descriptor came from without making
activation responsible for loading that source. The implemented source ref kinds
are:
- =local=
- =package=
- =registry=
- =generated=

Source refs are metadata values and can be projected with
=poo-module-source-ref->alist=.

*** PooModuleCatalog
Catalogs bind =PooModuleSourceRef= values to already-built descriptors. They
are immutable indexes:
- =poo-module-catalog-add= returns a new catalog;
- =resolve-poo-module-source= raises typed =missing-module-source= failures;
- =resolve-poo-module-sources= produces descriptor lists for doctor or activation.

*** PooModuleDescriptor
The descriptor is a Gerbil POO object with module-owned slots:
- =name=
- =imports=
- =task-registry=
- =flow-registry=
- =options=

*** PooModuleActivation
The activation is a transparent struct containing:
- activated module descriptors;
- combined task-family registry;
- combined flow-declaration registry;
- combined options.

Activation does not execute user code and does not load files.

*** PooModuleDoctorReport
=poo-module-doctor= produces an inspectable report containing module names,
status, and diagnostics. =poo-module-resolve-doctor= applies the same report to
catalog-selected source refs.

** 10.06.04.30 Source Resolution Rules
Source resolution is value-based. A catalog entry already contains a constructed
descriptor. The resolver only matches source refs by kind and value. It does
not read files, query package managers, or evaluate Scheme code.

This keeps the boundary explicit:
- loading belongs to a later source-loader layer;
- resolving belongs to =src/module-system/resolver.ss=;
- activation receives descriptor values only.

** 10.06.04.40 Activation Rules
Imports are explicit. If a module imports =foundation=, a module named
=foundation= must be present in the activation list.

Activation appends descriptor contributions after the base registries. Duplicate
contributions do not mutate activation behavior. They are reported by
=poo-module-doctor=.

** 10.06.04.50 User Interface Direction
The implemented macro is intentionally thin:

#+begin_src scheme
(poo-module remote-runtime
  (imports foundation)
  (tasks remote-task-descriptor)
  (flows remote-flow-descriptor)
  (options (cons 'source 'user-config)))
#+end_src

The macro performs no loading, no validation, and no runtime execution. It only
builds a =PooModuleDescriptor=.

** 10.06.04.60 Diagnostics Direction
The diagnostic surface is implemented as =poo-module-doctor=. It reports:
- missing imports;
- duplicate task-family names;
- duplicate flow-declaration kinds;
- duplicate option keys;
- modules with no descriptor contributions.

Later diagnostics should include unused imports, runtime capability gaps after
activation, and source-ref drift when filesystem or package loading is added.
** 10.06.04.70 Current Modular Owner Layout
The implementation is now split under =src/module-system/= instead of being held in core or mixed with concrete modules:
- =interface.ss= owns Marlin-style interfaces, schema helpers, and object helpers;
- =source.ss= owns source refs and structured imports;
- =descriptor.ss= owns C3 role data, descriptors, imports, metadata slots, and closed-world import validation;
- =context.ss= owns Doom-style flags, hook lookup, active checks, and phase-order projections;
- =diagnostics.ss= owns doctor reports;
- =resolver.ss= owns source catalog lookup, activation, and run-config lowering;
- =loader.ss= owns pure source-ref loader backends and load receipts;
- =projection.ss= owns schemas, validation receipts, runtime projections, value catalogs, presentation receipts, and Nix-style option merge receipts;
- =doctor.ss= owns user-facing module doctor and source doctor presentations;
- =descriptor-syntax.ss= owns user-facing macro sugar that expands to descriptor/catalog data;
- =src/module-system.ss= remains a thin facade that re-exports the leaf owners.

The Doom/C3 interface additions are data-only:
- descriptor slots now include =group=, =flags=, =features=, =depth=, =phase-files=, and =hooks=;
- =poo-module-active?= and =poo-flow-module-value-catalog-active?= provide explicit flag checks;
- =poo-module-option-merge-receipts= keeps append, override, conflict, constant, required, and missing-schema findings inspectable before runtime;
- =poo-module-phase-order= projects init/config ordering from depth metadata;
- runtime projections expose phase ids, hooks, and active metadata without loading files.

This keeps the boundary stable: Scheme owns descriptor data and validation, while loader backends and runtime execution remain separate owners.
** 10.06.04.80 User Interface Contract
The user-facing import is =:poo-flow/src/module-system=. Users should not import leaf owners such as =:poo-flow/src/module-system/context= or =:poo-flow/src/module-system/projection= directly for normal module authoring.

The public interface contract is:
- declare profiles with =poo-module-interface= and string schema helpers;
- construct modules with =pooModules= or =defpoo-module=;
- express imports with =poo-import= and =poo-imports=;
- inspect descriptor metadata with =poo-module-name=, =poo-module-group=, =poo-module-flags=, =poo-module-features=, =poo-module-phase-file=, and =poo-module-hook-values=;
- ask active/flag questions with =poo-module-active?= and =poo-flow-module-value-catalog-active?=;
- inspect final option composition with =poo-module-option-merge-receipts= and =poo-module-merged-option-alist=;
- load source refs into catalogs with =poo-flow-module-static-loader=, =poo-flow-module-load-source-receipts=, and =poo-flow-module-load-catalog=;
- group modules with =pooModuleCatalog=;
- project runtime-facing receipts with =poo-module-workflow=, =pooEvalModules=, and =pooModuleSystemPresentation=;
- project pre-runtime doctor summaries with =pooModuleDoctorPresentation= and =pooModuleSourceDoctorPresentation=;
- inspect user config mistakes through validation receipts rather than runtime exceptions.

The regression owner for this contract is =t/module-system-user-interface-test.ss=. That test intentionally imports only =:poo-flow/src/module-system= plus generic POO object accessors.
** 10.06.04.85 Maintained Downstream User Interface
=user-interface/= is the root-level maintained user-interface directory. It is not a test fixture and it is not package runtime source.
The directory models the thin handoff that downstream users see:
- =init.ss= chooses hot-plug module groups, module ids, flags, and custom module entrypoints;
- =custom/*/config.ss= is the downstream-owned custom module authoring space;
- =src/module-system/user-profile.ss= projects those root declarations into the canonical profile/config objects.
This borrows Doom's small user-facing feel without copying its package-management shape. POO Flow exposes modules, not packages: descriptor schemas, imports, loaders, catalogs, doctors, merge semantics, and sandbox realization stay in =src/module-system= or concrete =src/modules/<domain>= owners. Tests may verify this directory, but examples and user-facing configuration belong in root-level =user-interface/=.
** 10.06.04.90 Doom And Nix Design Boundary
The detailed design principles are in [[file:06-doom-nix-interface-principles.org][06 Doom And Nix Interface Principles]].

The short version:
- learn Doom's authoring experience, not its hidden runtime effects;
- learn Nix's option and merge semantics, not its syntax;
- keep Gerbil POO and C3 as the implementation substrate;
- add macro sugar only after the data API and user-interface tests exist;
- require user-facing tests to import =:poo-flow/src/module-system= only.
