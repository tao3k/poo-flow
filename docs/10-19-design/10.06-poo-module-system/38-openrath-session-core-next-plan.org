* 10.06.38 OpenRath Session Core Next Plan
** 10.06.38.10 Purpose
This plan uses OpenRath's session-first user ladder as design evidence for the
next POO Flow delivery slices after =session-core=.

The goal is not API alignment and not copying OpenRath's Python interface. The
goal is to identify what OpenRath gets right, identify where its model remains
too runtime-coupled or weakly constrained, and build a better agent-flow control
plane with Gerbil POO, functional composition, typed validation, and explicit
runtime handoff.

- =.data/OpenRath/example/README.md:1-70= defines a numbered learning ladder:
  hello agent, session lineage, sandbox placement, built-in tools, custom tools,

  =src/rath/flow/selector.py=, =src/rath/flow/tool/*=,

Current POO Flow evidence:

** 10.06.38.30 What We Should Learn From OpenRath
OpenRath's strongest contribution is conceptual ordering:
- =Session= is the flowing value, not a side effect hidden inside an agent loop.
- Sandbox, memory, tools, agents, workflows, and selectors are layers around
  that flowing value.
- The user manual introduces one concept at a time, so users can understand the
  system by running a small ladder instead of reading the whole framework.
- Branching and lineage are first-class enough to inspect, export, and reason
  about.
- Sandbox placement is visible in the user-facing model instead of being an
  implementation detail below tool calls.

These are worth adopting because they make agent systems debuggable. They give
users an answer to the questions: what is flowing, where did it come from, where
will it run, what tools may it use, what memory can it touch, and which strategy
selects the next step.
** 10.06.38.40 What POO Flow Should Improve
OpenRath is useful evidence, but it is not the endpoint for POO Flow.

Problems we should avoid:
- Python method chaining can blur declaration, mutation, and runtime behavior.
  A call such as =session.to(...)= is concise, but the boundary between a value
  transform and a runtime placement request must remain auditable.
- Runtime objects and user-facing objects can become the same object graph. That
  makes examples easy, but it weakens static validation and makes dry-run
  planning harder.
- The example ladder is clear, but many steps depend on live providers. POO Flow
  should make the no-runtime receipt path the ordinary first-class path.
- Tool, memory, provider, and selector concepts can drift into independent API
  islands. POO Flow should keep them as module-system objects that merge through
  one POO extension model.
- OpenRath does not have Gerbil POO's C3 inheritance, slot operators, fixed
  point object extension, or harness-backed object contracts. We should use
  those strengths instead of emulating Python classes.

The result should be stricter than OpenRath: every agent-flow layer should be
inspectable before runtime, validated before handoff, and extensible through POO
objects rather than ad hoc runtime hooks.
** 10.06.38.50 POO Flow Agent-Flow Model
POO Flow should model an agent flow as a typed, inspectable value pipeline:

#+begin_src text
Session
  -> SessionTransform
  -> ToolIntent / MemoryIntent / PlacementIntent
  -> RuntimeHandoff
  -> Receipt
  -> DerivedSession
#+end_src

The normal path is:
- user enables capability modules with =use-module=;
- user defines POO objects or loads object fragments;
- Scheme validates object structure and extension merges;
- presentation exposes intent rows and handoff manifests;
- Marlin or another backend executes only after an explicit handoff boundary;
- receipts derive the next session value.

This is the core improvement over OpenRath: the agent flow is not just ergonomic
runtime Python. It is a staged control plane with separate declaration,
validation, handoff, execution receipt, and derived session layers.
** 10.06.38.60 Design Rule
Every next slice should pass this rule:

#+begin_quote
The user sees one compact concept; Scheme produces POO objects and functional
receipts; runtime work stays behind an explicit Marlin or backend handoff.
#+end_quote

Do not introduce raw alist DSLs, record-only profile systems, or a parallel
session model. New module rows should be POO-native and should surface through
=use-module=, profile presentation, doctor presentation, and user-interface
cases.
** 10.06.38.70 Approach Choice
There are three possible ways to use OpenRath:

1. Copy the surface ladder and make Scheme equivalents. This is rejected because
   it preserves OpenRath's runtime-coupled shape.
2. Treat OpenRath only as inspiration and continue building unrelated modules.
   This is rejected because it loses the strongest session-first insight.
3. Use OpenRath as a design test suite: learn the session-first ordering, then
   implement a stricter POO-native agent-flow model with validation and handoff
   boundaries. This is the selected approach.

The selected approach keeps the user interface compact while making the system
more suitable for reliable multi-agent work than a method-chain runtime API.
** 10.06.38.80 Milestone 1: Session Transform Core
OpenRath rung: =01_hello_agent.py=.

POO Flow target:
- Introduce a pure =session-transform= object family.
- Model =Agent= as a functional arrow: =Session -> SessionReceipt=.
- Keep provider/model execution out of Scheme and express it as an explicit
  handoff intent.
- Expose a compact user case that declares a transform over a session value.

Candidate files:
- =src/modules/session/transform.ss=
- =src/modules/session/objects.ss=, if shared object contracts need splitting
- =user-interface/custom/my-module/cases/session-transform.ss=
- =t/session-transform-test.ss=

Acceptance:
- A test can create a session, apply a report-only transform, and inspect the
  derived session id, parent session id, transform name, runtime owner, and
  =runtime-executed: #f=.
** 10.06.38.85 Milestone 1A: Memory Intent Receipts
OpenRath rung: memory-aware agent/session examples.
POO Flow target:
- Keep memory as a POO intent attached to a =session-transform=.
- Do not let Scheme recall, mutate, or commit a memory backend.
- Emit runtime-facing memory handoff rows as plain data inside the transform handoff intent.
- Emit POO memory receipts so the derived session can be audited without runtime execution.
Implemented surface:
- =poo-flow-session-memory-intent= declares store, scope, recall keys, and commit policy.
- =poo-flow-session-transform= accepts a list of memory intent objects as an extension slot.
- =poo-flow-session-transform-apply= records =memory-receipts= and =memory-receipt-count= on the transform receipt.
- =session-memory-intent= and =session-transform ... (memory-intents ...)= expose the compact user interface.
Acceptance:
- A test can create a memory intent, attach it to a transform, derive a session, inspect the memory receipt, and verify every runtime flag remains =#f=.
- A user-interface case can load the same shape through =custom/my-module/cases/session-transform.ss=.

POO Flow target:
- Treat =session.to(backend)= as POO placement inheritance plus sandbox profile
  catalog resolution.
- Produce a placement handoff ABI row, not a running sandbox.
- Cover nono, docker, and cube through existing sandbox profile objects.

Candidate files:
- =src/modules/session/placement.ss=
- =src/modules/sandbox-core/profile.ss=

  backend kind/ref, and Marlin handoff owner.
- Missing or unsafe filesystem profile data is a structured validation failure.
** 10.06.38.110 Milestone 4: Tool Core
OpenRath rungs: =04_tools_builtin.py=, =05_custom_tool.py=, =06_mcp_tool.py=.

POO Flow target:
- Add a =tool-core= module row under the workflow/session family.
- Model tools as POO object specs with schema, runtime owner, sandbox
  requirements, and handoff operation.
- Built-in filesystem/shell tools are specs only; runtime execution belongs to
  Marlin, nono, MCP, or another backend.

Candidate files:
- =src/modules/tool-core/config.ss=
- =src/modules/tool-core/objects.ss=
- =src/modules/tool-core/spec.ss=

Acceptance:
- A custom calculator-like tool can be declared as a POO object and projected to
  a handoff manifest.
- Built-in filesystem/shell tool specs require a sandbox placement profile.
- MCP tool rows can be represented without starting an MCP server.

POO Flow target:
- Add =selector-core= after session transform, tool, and memory specs exist.
- Model selection as a POO strategy object over named session transforms or
  workflows.
- Keep dynamic routing as a receipt until Marlin owns execution.

Candidate files:

- Presentation proves no model call or runtime dispatch happened in Scheme.
** 10.06.38.140 Delivery Order
1. Finish =session-transform= first. It is the missing bridge between session
   values and workflow/agent layers.
2. Expand =session-lineage= examples and export receipts next, because this is
   key-free and gives stable tests.
3. Strengthen sandbox placement handoff, reusing nono/docker/cube profile
   objects and filesystem validation.
4. Add =tool-core= only after placement is solid.
5. Add =memory-core= after tool-core, because memory has similar backend
   boundaries but different lifecycle semantics.
6. Add =selector-core= last, when named transforms and receipts are stable.

** 10.06.38.160 Current Implementation Status
Milestone 1 is now delivered as a report-only control-plane slice:
- =src/modules/session/transform.ss= defines the POO transform and receipt object family.
- =src/modules/session/config.ss= exposes =session-transform= and =transform-session= for compact user syntax.
- =src/modules/session-core/config.ss= and =src/module-system/session-core-config.ss= surface =+transform= as a default session-core feature.
- =user-interface/custom/my-module/cases/session-transform.ss= loads a downstream transform/root/receipt object list through the existing =load!= expression contract.
- =t/session-transform-test.ss= covers transform declaration, derived session lineage, placement inheritance, handoff intent, and =runtime-executed: #f=.
- Object catalog debug is contract-owned: =src/module-system/object-validation.ss= now projects object identities, direct/resolved field identities, field origins, inheritance chains, validation phases, field counts, inheritance counts, and runtime flags through validation receipts instead of adding debug slots or per-file debug variables to =objects.ss=.
Follow-up: if focused =gxtest= startup continues spending most time in =session/config=, split placement catalog resolution away from the core session object loading path so transform tests can exercise =objects + transform= without pulling the full sandbox profile catalog.
