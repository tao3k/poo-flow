#+title: Programming Style Optimization Tranche
#+subtitle: Object Families, Projection Boundaries, and Measurable Performance Gates
#+date: 2026-07-09
#+options: toc:3 num:nil

* Purpose

This document turns [[file:44-current-project-programming-style.org][the current programming-style contract]] into the next executable
optimization tranche.  It is a design and acceptance contract, not a
replacement for the style document and not an implementation plan for a
runtime executor.

The unit of optimization is one semantic object family at a time.  A family
owns one domain fact, validates it once, supports POO-native extension, and
projects it only at a receipt or runtime boundary.  File length is evidence to
investigate, not the definition of a module boundary.

* Fixed Direction

The control-plane shape remains:

#+begin_example
POO semantic owner -> pure policy/combinator -> receipt -> projection -> runtime adapter
#+end_example

- A POO object owns the semantic fact and its validated extension surface.
- Pure functions select, merge, validate, or route those facts.
- Receipts explain a decision without becoming the decision's owner.
- Alists, JSON, ABI rows, and foreign payloads are projections only.
- A runtime adapter executes a projection; it does not recover private Scheme
  state or redefine policy.

This is the shared direction for model, tool, memory, session, artifact, graph,
and future community compositions.

* Evidence Baseline

The current style document has fourteen headings that already define the
contract: POO-native public surfaces, functional composition, runtime
projection, closed kernels with declared extension slots, bounded macros,
object-family generation, contracts, hot-path discipline, and focused
scenarios.

The present source map shows a useful asymmetry:

- =model-core= has already adopted =defpoo-object-family= for three stable
  families.  It is the correct proving ground for a production-grade family
  declaration.
- =object-family-syntax= is intentionally small: one declarative macro is the
  whole mechanism today.  The next improvement belongs there, rather than
  duplicating constructor, predicate, accessor, and projection conventions in
  every module.
- =tool-core= still mixes object recognition and alist/object conversion at a
  high-traffic boundary.  It is the first consumer of the stronger family and
  projection rules, but it should not receive a blind line-count split.
- The scenario alignment, scenario/test owner matrix, observability performance
  gates, and package test project are existing places to register the resulting
  evidence rather than inventing a parallel benchmark framework.

* P0: Complete the Object-Family Declaration

Strengthen =defpoo-object-family= as a bounded compile-time declaration for a
stable POO family.  Its expansion must remain ordinary Gerbil POO and pure
Scheme code with useful source locations.

The declaration should be able to generate only the repeated, structural
surface:

- kind and schema constants;
- checked public constructors;
- named predicates and slot accessors implemented through =.ref=;
- a named projection for receipt/runtime boundaries;
- optional catalog/index helpers when the family has a real repeated lookup
  path.

It must not create a new runtime DSL, activate a provider, perform I/O, hide
inheritance conflicts, or emit an unvalidated foreign payload.  Declarations
with genuinely different behavior remain normal POO objects plus pure helper
functions.

The first P0 implementation slice is one existing =model-core= family.  It
must preserve its public POO API while removing duplicated structural code from
that family.  The macro contract is accepted only with expansion, constructor,
projection, and extension tests.

The shared predicate contract was completed on 2026-07-09 before introducing a
new consumer: a generated family predicate returns =#f= for both a non-object
and an object without a =kind= slot.  It never leaks the underlying =.ref=
exception into a policy decision.  The existing syntax test and 1,000-object
scenario both pass under that contract.

* P1: Repair the Tool Projection Hot Path

Use the completed family mechanism to make =tool-core= follow the same owner
model.  Each public tool fact must have one POO owner; alist recognition and
conversion are allowed only at a named receipt or runtime-projection boundary.

The target P1 owner graph is concrete:

- =objects-spec= owns =poo-flow-tool-spec=, its checked construction,
  predicates, accessors, and projection;
- =objects-handoff= owns the handoff manifest and its runtime-boundary
  projection;
- =objects-catalog= owns catalog summaries, indexes, and lookup helpers;
- =objects-policy-refs= owns grant references and policy-to-tool resolution
  inputs;
- =objects-validation= owns diagnostics, validation summary, and validation
  steps;
- =objects-receipt= owns the validation receipt, its validity predicate, and
  receipt projection.

=objects.ss= remains the public facade over these semantic owners and the
builtin catalog.  It is not a second semantic implementation.  The former
=objects-core= owner is deleted after all six owners are imported by the facade
and its public exports, tests, and projections are unchanged.

P1 implementation completed on 2026-07-09.  The facade imports the six
semantic owners directly; =objects-builtin= imports only the specification and
catalog owners; and =objects-core.ss= no longer exists.  Both the handoff
manifest and validation receipt are ordinary POO objects, not hidden
=defstruct= records.  Their only alist output is the named final projection at
the handoff or receipt boundary.

The acceptance evidence is intentionally narrow and reproducible:

#+begin_src sh
gxpkg env gxtest -q t/tool-core-test.ss
gxpkg env gxtest -q t/tool-core-performance-test.ss
#+end_src

Both gates passed after the split.  Bare =gxc= under the inherited Nix
=SDKROOT= is not an accepted substitute for these package tests on this
workstation: it selects a Homebrew Gerbil C runtime against Nix SDK headers and
fails before Scheme module validation.  For a narrow cache refresh, use
=env -u SDKROOT gxpkg env gxc <owner>.ss=; this preserves the project build
configuration while selecting a compatible SDK for the installed compiler.  A
full native package compile separately stalls at =init-syntax~1.scm= and must
be repaired as a build-stage issue; it must not be hidden by reducing test
input, loop iterations, or build concurrency.

Factor files only where the semantic owners differ, for example tool
specification, catalog/index, policy grant, validation receipt, and runtime
handoff.  A facade may re-export the public family.  Do not split a coherent
family merely to reduce a line count, and do not retain a legacy public API
when the POO-native public surface replaces it.

Repeated symbolic lookups must use a family catalog or a specialized cache
object.  A loop must not repeatedly construct cons-heavy temporary POO objects
or rescan the same list when an index is available.

* P2: Make Performance Evidence First-Class

Every shared family change receives a scenario-owned =benchmark.ss= and a
performance gate.  The scenario must measure the real construction, selection,
projection, and extension path; string-append microbenchmarks are not
acceptable substitutes.

The initial scale is at least 1,000 profiles or family objects, with a larger
case where the scenario models a realistic composition library.  Report:

- elapsed time with stable units and useful precision;
- peak RSS and the change over repeated runs;
- object construction, catalog selection, projection, and macro-expansion
  phases separately;
- the baseline, threshold, machine-independent assertion, and regression
  interpretation.

The gate may not pass by lowering iterations, relaxing concurrency, or hiding
allocation behind a foreign runtime.  When RSS or time is unstable, treat it as
a correctness signal first and isolate the allocation or expansion path before
changing a threshold.

The first P2 scenario upgrade is complete: =poo-object-family-syntax= now
constructs and projects 1,000 POO objects with integer references.  This
removes string/symbol allocation from the measured path.  Three independent
runs on 2026-07-09 measured 2.326-5.595 ms, with RSS between 136,527,872 and
136,626,176 bytes.  The time variation is expected at this sub-6 ms scale and
the RSS spread is 96 KiB; neither approaches the 25 ms or 512 MiB gates.

The harness receipt upgrade now measures runtime phases without a new DSL:
callers pass the phase symbol through the existing testing wrapper details
boundary and supply an ordinary higher-order thunk.  The object-family scenario
produces separate dynamic receipts for =object-construction=,
=boundary-projection=, and =object-selection=; each is asserted to carry its
own phase symbol and numeric =elapsedMs=.  Legacy fixture =observed*= fields
remain static acceptance inputs.  Compile-time expansion is still a distinct
build-stage metric and must not be fabricated as a runtime phase.

* P3: Keep Macro Power Bounded and Native

Use =defrules= for fixed declarative shape, =defsyntax= with =syntax-case= for
checked declarations, and =begin-syntax= metadata only when it removes real
runtime work.  A macro may collect static slots, validate duplicate or missing
facts, and generate POO-native constructors and functional helpers.

It may not parse a fixed declaration language at runtime, compose private
kernel hooks, launch worktrees, invoke runtime execution, or create a second
public configuration language.  Native POO remains available after expansion:
users can extend a profile with =:extends=, apply named functional extensions,
or contribute an existing POO object to a composition.

* P4: Introduce Dispatch Only at Real Extension Boundaries

Interface or generic dispatch is justified only when independent downstream
modules need behavior extension without a central symbolic branch.  Current
candidates are model providers, durable stores, tool adapters, and runtime
adapters.

A single local helper, one fixed list of profiles, or a projection helper does
not justify an interface.  In those cases, use a named pure function or a
family catalog.  This keeps dispatch costs and public surface area proportional
to actual extension needs.

* P5: Join Style, Harness, and Observability Evidence

The repair table in the style document, the harness policy, package builder,
scenario/test owner matrix, and observability performance gates must agree on
the same facts:

- which POO family owns the semantic fact;
- which macro or constructor validates the public declaration;
- which projection is handed to a runtime;
- which scenario measures the shared hot path;
- which receipt explains a rejected composition or policy decision.

Add a focused harness diagnostic whenever a new anti-pattern becomes
mechanically detectable.  A warning is not considered resolved until the
corresponding source shape, scenario, and gate agree.

* Execution Order and Acceptance

1. P0: extend the object-family declaration through one =model-core= family,
   retaining POO-native API compatibility and adding focused tests.
2. P2 foundation: register a scenario-owned benchmark and RSS/time gate for
   that family before applying the mechanism elsewhere.
3. P1: move the selected =tool-core= hot path to the same owner/projection
   shape; factor by semantic owner only where the code demonstrates separate
   responsibilities.
4. P3: add checked declarative forms only after the generated surface has a
   proven need in a second family.
5. P4 and P5: add interface dispatch and new harness diagnostics only where
   independent extension or recurring evidence shows they are necessary.

The verification gate for a code tranche is the focused family test, its
=benchmark.ss= scenario, the scoped harness-policy check, package tests, and
the normal compile path.  The build-cache and package-builder gates are part
of the same evidence chain.  A passing compile alone is not acceptance.

* Explicit Non-Goals

- No Scheme runtime executor, provider call, or durable database implementation
  is introduced by this tranche.
- No raw alist, JSON, or record DSL becomes a public composition API.
- No raw =lambda (self super)= hook becomes the normal user extension path.
- No performance regression is masked by fewer iterations or lower build
  concurrency.
- No compatibility layer is retained for a superseded public interface unless
  migration is explicitly required by a downstream user.

* Completion Condition

This tranche is complete when one family proves the generated POO-native shape
under a real scenario gate, the adjacent tool projection boundary follows the
same ownership rule, and the harness can reject the recurring anti-patterns
that the change removed.
