#+TITLE: RFC 91 — Canonical Organization Bundle
#+SUBTITLE: Typed Semantic Core for POO Flow Organizations
#+OPTIONS: toc:3 num:t
#+LANGUAGE: en

* Status

Status: *Accepted; implementation planning*

Implementation readiness depends on written suite review and the P0 gate in RFC 94.

* Decision
This RFC defines the Canonical Organization Bundle as the normative, runtime-independent semantic artifact produced by POO Flow.

The accepted decision is:

1. Gerbil POO objects and functional combinators are the public authoring surface.
2. Compilation produces one immutable Canonical Organization Bundle.
3. The Bundle contains five typed semantic facets sharing one identity universe, manifest, version lineage, and provenance chain.
4. The facets are Organization, Authority, Context, Protocol, and Evidence.
5. Each facet has its own composition algebra and validation rules.
6. C3 linearization resolves definition lookup order only; it grants no authority, relaxes no context restriction, changes no protocol obligation, and removes no evidence obligation.
7. Cross-facet validation is mandatory before activation or projection to the C Runtime Interface, Cedar, Lean, or evidence contracts.
8. Dynamic reorganization creates an immutable epoch linked to its parent; it never mutates an active Bundle in place.
9. Ambiguous or unsafe composition is a typed compilation error. Last-writer-wins behavior is forbidden.
10. The Bundle, not a Python object model, C structure, Cedar policy, Lean model, or runtime implementation, is POO Flow's cross-language semantic center.

* Purpose

The Bundle is the durable semantic boundary between organization authoring and runtime execution. It gives every conforming projection and runtime one answer to:

- which agents, roles, principals, resources, and artifacts exist;
- how responsibilities and collaboration are organized;
- which capabilities may be granted, delegated, attenuated, revoked, or leased;
- which information may flow, with what provenance and declassification authority;
- which lifecycle states, messages, transitions, effects, cancellation paths, recovery paths, and checkpoints are valid;
- which obligations, receipts, witnesses, causal links, verifiers, and proof artifacts are required.

The Bundle MUST be deterministic, typed, immutable, digest-addressed, and runtime-language independent. It is a compiled semantic artifact, not an alternative public authoring DSL or a serialization of Gerbil implementation objects.

* Normative Language and Scope

The terms MUST, MUST NOT, REQUIRED, SHOULD, SHOULD NOT, and MAY are normative.

A /definition/ is an authoring-time POO declaration or functional value. An /entity/ is a typed stable identity. A /facet/ is a typed semantic projection over the shared identity universe. A /Bundle/ is its manifest and all five validated facets as one indivisible artifact. An /epoch/ is one immutable version in a Bundle lineage. A /projection/ is a derived artifact for a specific consumer.

This RFC is the contract owner for Bundle identity, manifest and lineage, facet boundaries and composition, compilation, validation, reorganization, and conformance. The Bundle is input to later runtime, authorization, proof, and evidence stages.

* Non-Goals

This RFC does not:

- define a Rust, TypeScript, Python, or other runtime implementation;
- define the C ABI layout, Cedar syntax, or Lean proof implementation;
- require a graph database or define a generic graph overlay;
- expose raw alists, records, patch languages, or raw compute hooks as the ordinary public interface;
- make organization hierarchy imply authority;
- make a runtime object model canonical;
- place mutable runtime session state in the Bundle;
- prove a runtime correct merely because it consumes a valid Bundle;
- define an OS scheduler, container system, or distributed consensus protocol;
- preserve obsolete public authoring routes after a replacement is canonical.

* Core Semantic Model

The five facets are typed views of one organization, not independently versioned graphs. They share entity identities, schema, epoch, parent lineage, compiler provenance, Bundle digest, and one validation result. A facet MAY use graph-shaped storage internally, but its normative meaning is its typed entities, relations, constraints, and invariants.

The Bundle MUST be accepted or rejected as a whole. A consumer MUST NOT combine facets from different Bundles.

A valid Bundle is immutable. Any semantic change produces a new epoch. Caches and indexes are derived state and MUST NOT alter meaning or digest.

The Bundle MUST NOT contain Gerbil heap addresses, Scheme object layouts, Python objects, runtime-owned pointers, C ABI handles, process-local identities, or host callbacks. Optimized runtime views remain projections bound to a Bundle digest and epoch.

* Shared Identity Universe

Every cross-reference MUST use a typed stable identity. The minimum shared kinds are =Agent=, =Role=, =Principal=, =Resource=, and =Artifact=. Facet-local identities for grants, transitions, messages, obligations, receipts, or witnesses MUST also declare their entity kind.

Identity kind is semantic. Textually equal identifiers of different kinds are not interchangeable. Display names MUST NOT be used as identity.

An entity retaining its semantics across epochs MUST retain its identity. A semantic replacement MUST receive a new identity even when its display name is unchanged. Deletion in a later epoch does not invalidate historical evidence from an earlier epoch.

Every reference MUST resolve to one entity of the expected kind in the same Bundle and in a relation permitted by the referring facet. Unresolved, mistyped, multiply defined, or contextually invalid references are errors.

An agent, its role, and its principal remain distinct identities connected by explicit typed relations. The compiler MUST NOT infer authority because such identities describe the same participant.

* Bundle Manifest

Every Bundle manifest MUST contain:

- schema identifier and major/minor version;
- compiler identity, version, and provenance;
- normalized authoring-input and dependency/import provenance;
- epoch;
- parent Bundle digest except for a genesis Bundle;
- Bundle digest;
- identity and digest of every facet;
- canonicalization profile.

A genesis Bundle has no parent. Every non-genesis Bundle MUST name exactly one parent digest and have an epoch greater than its parent. Parentage records lineage but does not authorize activation.

The Bundle digest MUST cover the semantic manifest fields, identity universe, all facets, provenance required by the manifest, epoch, and parent. The digest field itself is excluded from its preimage. Digest computation MUST use the declared canonicalization profile.

The same normalized input, dependency closure, compiler version, and options MUST produce byte-identical canonical content and the same digest. Time, process IDs, random values, absolute checkout paths, map iteration order, and ambient state MUST NOT affect the digest unless declared semantic input.

* Five Typed Facets

** Organization

The Organization facet defines agents, roles, role occupancy, responsibilities, creation relationships, explicit ownership, collaboration topology, and organizational parent/child relationships.

Organizational seniority, parenthood, ownership, or proximity MUST NOT imply authority. Such rights require explicit Authority declarations and, where applicable, Context or Protocol declarations.

Organization composition MAY combine compatible responsibilities and topology. It MUST reject incompatible unique owners, contradictory role occupancy, forbidden cycles, unresolved responsibility conflicts, and implicit choices among competing definitions. Multiplicity MUST be explicit. C3 lookup MUST NOT hide a semantic ownership conflict.

** Authority

The Authority facet defines principals, capabilities, grants, issuers, delegation, attenuation, revocation, leases, and principal/action/resource dimensions.

Within a delegation chain, effective authority MUST be no broader than the delegator's authority after attenuation, lease, and revocation constraints. Independent grants MAY contribute authority only while each retains a valid issuer and grant chain. Any expansion requires a new explicit authorized issuer grant.

Composition MUST reject unauthorized widening, excess delegation, invalid leases or revocations, unresolved grant conflicts, and capabilities lacking typed principal/action/resource meaning. C3 MUST NOT act as permit-overrides, deny-overrides, or policy precedence.

** Context

The Context facet defines visibility, information labels, provenance, context leases, redaction, permitted flows, declassification, and declassification authority.

Composition follows an information-flow lattice or equivalent restriction model. When constraints combine, the result MUST be at least as restrictive as every input; for permitted readers or flows this is their intersection. Provenance MUST survive composition.

Declassification requires an explicit operation, principal, authority, source and target label or flow class, and Evidence obligation. Composition MUST reject unauthorized widening, missing labels or provenance, invalid leases, unresolved redaction conflicts, and declassification lacking Authority and Evidence links. C3 MUST NOT replace a stricter inherited constraint with a weaker one.

** Protocol

The Protocol facet defines lifecycle states, messages, transitions, effect intents, preconditions, cancellation, recovery, checkpoint/restore, failures, and required links to Authority, Context, and Evidence.

Protocol composition is refinement, not graph overlay. A refinement MAY restrict behavior, add internal structure, strengthen preconditions, make outcomes more precise, add obligations, or implement an abstract transition.

A refinement MUST NOT silently remove a required lifecycle or failure path, weaken a safety precondition, introduce an external effect without a typed intent and Authority/Evidence links, make cancellation/recovery/checkpoint semantics ambiguous, or incompatibly reinterpret a stable state or message. Incompatible transitions require explicit refinement or disambiguation; C3 order cannot make them valid.

** Evidence

The Evidence facet defines obligations, receipts, witnesses, causal relations, verifier references, proof-artifact references, and evidence required by authorization, context transfer, protocol transition, effect execution, and activation.

Evidence obligations accumulate monotonically. Composition takes the union of applicable obligations while retaining identity and provenance. Discharge MUST explicitly identify the obligation, typed receipt/witness/verifier/proof artifact, Bundle digest and epoch, and verification rule.

A discharge does not erase the obligation or history. Composition MUST reject silent removal, unmatched discharge, wrong evidence kind, missing verifier, invalid causal endpoint, or unmarked cross-epoch evidence. C3 MUST NOT override obligations.

* Facet-Specific Composition

There is no universal merge function:

- Organization resolves responsibilities and topology.
- Authority preserves grant provenance and attenuates delegation.
- Context selects the stricter information-flow result.
- Protocol checks refinement.
- Evidence accumulates obligations.

A generic map merge, edge union, last-writer-wins overlay, or untyped patch is non-conforming. When more than one semantic resolution remains possible, the author MUST supply a named POO-native or functional resolution. That resolution enters normalized provenance and affects the digest.

C3 answers only which inherited definition is examined first and how compatible POO definitions are collected. Each facet algebra decides semantic compatibility afterward.

* Compilation Pipeline

The normative pipeline is:

1. Collect Gerbil POO definitions, prototype composition, slot declarations, and functional combinator values from the declared closure.
2. Apply C3 where deterministic inherited-definition lookup is required.
3. Elaborate declarations into typed entities and facet declarations.
4. Build and validate the shared identity universe.
5. Normalize each facet with its own algebra.
6. Resolve all intra-facet and cross-facet references.
7. Validate each facet locally.
8. Validate all cross-facet invariants.
9. Canonically order and encode all semantic content.
10. Construct the manifest and compute facet and Bundle digests.
11. Mark the Bundle projection-eligible only after every validation succeeds.

The compiler MUST NOT emit an activatable partial Bundle. Derived projections MUST NOT mutate it or invent missing semantics; a discovered deficiency requires a corrected Bundle.

* Cross-Facet Invariants

A valid Bundle MUST satisfy all of the following:

1. Every reference resolves to a typed stable identity.
2. Every shared entity has one consistent identity and kind.
3. Organization hierarchy creates no implicit grant.
4. Every external Protocol effect references a typed intent, requesting principal, Authority requirement, and Evidence obligations.
5. Every context transfer declares label or flow class, provenance, lease/retention constraints, redaction, and any declassification authority.
6. Every declassification links Context, Authority, and Evidence.
7. A child agent's delegated authority is no broader than its parent's delegable authority unless an independent authorized issuer grants it.
8. Every protocol transition consuming or producing context is Context-compatible.
9. Cancellation, recovery, checkpoint, and restore have explicit lifecycle meaning and evidence requirements.
10. All referenced obligations exist and have the correct kind.
11. Discharge never erases obligation provenance.
12. Every transition, projection, and later receipt can bind to Bundle digest and epoch.
13. Manifest facet digests match the facets in that Bundle.
14. No projection mixes content from different Bundle digests.
15. Canonical encoding is deterministic for identical semantic input.

Violation of any invariant invalidates the whole Bundle.

* Dynamic Reorganization and Epochs

Reorganization is immutable Bundle succession:

#+begin_src mermaid
stateDiagram-v2
    [*] --> ActiveN: activate epoch N
    ActiveN --> Candidate: compile N+1 with parent digest
    Candidate --> Rejected: validation failure
    Candidate --> Ready: complete validation succeeds
    Ready --> ActiveNext: atomic switch at safe point
    ActiveN --> Historical: switch completes
#+end_src

The system MUST compile a candidate against the active parent, validate all facets and projections, reach a runtime safe point, atomically activate the entire new Bundle, and retain the old Bundle as immutable history.

A transition MUST NOT use mixed epochs. It either completes under the epoch that admitted it or reaches explicit quiescence, cancellation, recovery, or checkpoint before the new epoch applies. Activation MUST NOT relabel old work.

Rollback is a new epoch whose parent is the current Bundle; it is not in-place mutation. Historical receipts remain bound to their original epochs.

* Validation and Error Model

Compilation returns either one validated immutable Bundle or typed errors. Safety-relevant errors MUST NOT be downgraded to warnings.

Each error MUST include a stable code, phase, affected facet or facets, relevant typed identities, violated rule, source provenance, and all competing definitions for composition conflicts. Suggestions MAY be reported but MUST NOT be applied implicitly.

Required error classes include duplicate identity, identity-kind mismatch, unresolved reference, invalid lineage, ambiguous Organization composition, unauthorized widening, invalid delegation/revocation, Context widening, unauthorized declassification, invalid Protocol refinement, missing effect authorization/evidence, missing provenance, invalid Evidence discharge, cross-epoch misuse, canonicalization nondeterminism, and unsupported schema version.

Unknown semantic fields, unsupported major versions, invalid digests, and incomplete facets fail closed. Unknown descriptive metadata MAY be preserved only when the schema marks it non-semantic.

* Versioning and Projections

The manifest carries schema major and minor versions. A major change is REQUIRED when an existing consumer could assign different semantics to the same content. A minor version may add compatible declarations only when consumers can interpret or explicitly reject them without silent omission.

Canonicalization rules are versioned. A change to ordering, normalization, or digest preimage requires an unambiguous schema or profile change. Epoch answers which organization version; schema version answers under which semantic format.

Every projection MUST identify source Bundle digest and epoch, consume only validated content, preserve stable identities, fail rather than invent semantics, be deterministic for the same Bundle/projector version, and carry projector provenance where required.

Cedar derives authorization material; Lean derives models and obligations; the C Runtime Interface derives runtime state/effect/evidence views; evidence projection derives typed verification relationships. None becomes a second semantic owner.

* Security Considerations

- Typed identity prevents display-name and cross-kind substitution.
- Explicit grants prevent prototype or hierarchy composition from escalating privilege.
- Stricter Context composition prevents precedence-based disclosure.
- Protocol reachability alone never authorizes an external effect.
- Monotonic Evidence composition prevents inherited obligations from disappearing.
- Digest and epoch binding prevent old facets from being presented as current.
- Canonical serialization, complete semantic coverage, and version rejection prevent digest ambiguity.
- Compiler and input provenance expose the trust boundary; a digest proves content identity, not compiler correctness.
- Projectors remain bound to source digest and provenance to prevent projection drift.
- A valid Bundle constrains conforming execution but does not prove a runtime cannot bypass the interface.

* Test Matrix

A conforming implementation MUST test:

- byte-identical determinism across declaration/map order and ambient host variation;
- digest change for every semantic mutation;
- typed identity resolution, kind mismatch, replacement, and historical identity;
- Organization compatibility, multiplicity, ownership conflict, and no hierarchy-derived grant;
- Authority attenuation, issuer provenance, excess delegation, leases, and revocation;
- Context restriction, provenance preservation, widening rejection, redaction, and authorized declassification;
- Protocol refinement, incompatible transitions, effects, failures, cancellation, recovery, checkpoint, and restore;
- Evidence accumulation, typed discharge, causal validity, and deletion rejection;
- every Organization–Authority–Context–Protocol–Evidence closure required by the invariants;
- parent/epoch validation, candidate rejection, atomic activation, no mixed epochs, and rollback as a new epoch;
- mutation cases that replace an identity kind, widen context, add an unmediated effect, delete an obligation, substitute a foreign facet, alter a parent digest, or mutate canonical bytes.

Each mutation MUST either change the expected digest or fail validation. No case may succeed through an unrelated fallback.

* Decision Notes

- *Five independent graphs rejected:* they permit identity, policy, and epoch drift; the accepted design is one indivisible Bundle.
- *One generic graph rejected:* responsibility conflict, delegation attenuation, information-flow restriction, protocol refinement, and evidence monotonicity require distinct algebras.
- *Runtime object model rejected as canonical:* Python, Rust, TypeScript, C, and Gerbil layouts serve different implementation needs.
- *C3 security precedence rejected:* refactoring order cannot become a security boundary.
- *Last-writer-wins rejected:* it hides conflicts and can silently widen authority or information flow.
- *Mutable reorganization rejected:* immutable parent-linked epochs provide stable referents, atomic activation, history, and explicit rollback.
- *Projection-owned semantics rejected:* Cedar, Lean, C ABI, and evidence consumers derive meaning from the Bundle rather than inventing it.

* Acceptance Criteria

This RFC is implemented when:

1. Public Gerbil POO and functional authoring compiles to one Canonical Organization Bundle.
2. The Bundle contains one manifest, one typed identity universe, and all five facets.
3. Each facet uses and validates its distinct composition algebra.
4. C3 remains definition resolution only.
5. Every cross-facet invariant is enforced.
6. Invalid or ambiguous input produces typed errors and no activatable partial Bundle.
7. Identical normalized input and compiler closure produce identical bytes and digest.
8. Every non-genesis Bundle records an increasing epoch and parent digest.
9. Reorganization activates a fully validated Bundle atomically at a safe point.
10. No transition or projection mixes Bundle digests or epochs.
11. Cedar, Lean, C ABI, and evidence outputs are deterministic source-bound projections.
12. Organization hierarchy grants no authority.
13. Delegation cannot widen without a valid issuer grant.
14. Context cannot widen without explicit declassification authority and evidence.
15. Protocol refinement cannot introduce an untyped or unauthorized external effect.
16. Evidence obligations accumulate and discharge only explicitly.
17. The complete positive, negative, mutation, determinism, and epoch test matrix passes.
18. The Bundle is enforced as POO Flow's sole cross-language semantic center.
