* 10.06.39 Gerbil Build Optimization Audit
:PROPERTIES:
** 10.06.39.10 Finding: =std/make= Parallelism Starts Late
:PROPERTIES:
Gerbil =std/make= computes =settings-parallelize= through =gerbil-build-cores=. When =parallelize:= is left as =#t=, Gerbil reads =GERBIL_BUILD_CORES= and falls back to =0=, which becomes one worker at build time. Passing =GERBIL_BUILD_CORES= is therefore the normal package-level way to opt into local parallelism.

The worker model still has an important limit: =make-build= first normalizes the build spec, creates completion/timestamp tables, imports spec modules under a mutex, checks dependency timestamps, then pushes buildable specs to a work channel. The worker count only helps after that import/dependency phase.

For =poo-flow=, the expensive section is not only =gsc= compilation. The heavy module-system and POO macro modules make the pre-worker expander/import phase large enough to look like a single-threaded stall.
** 10.06.39.20 Finding: Thin Package Builds Depend On Clean Package Roots
:PROPERTIES:
=.data/gerbil-poo/build.ss= is the right package-level reference for a normal Gerbil library. It imports =:clan/building=, defines a small =spec= around =all-gerbil-modules=, applies one special build option to =io=, conses one test helper, and delegates the rest to =init-build-environment!=.

=.data/gerbil-utils/build.ss= is slightly larger only because it bootstraps =:clan/building= itself and defines extra multicall commands for Nix/Docker publishing. Its package surface is still discovered through =clan/building#all-gerbil-modules= with a small conventional boundary.

The lesson for =poo-flow= is not that every project needs a giant explicit stage list. The lesson is that a Gerbil package can keep =build.ss= simple when the package root contains package modules, tests that are meant to compile, and a small number of explicit special entries. If the repository root also contains research checkouts, generated state, user examples, docs, and fixtures, then =build.ss= becomes a boundary-repair layer. That is a project structure smell.
** 10.06.39.25 Measurement: Reference Simplicity Is Not The Same As Always-Fast Builds
:PROPERTIES:
With =GERBIL_BUILD_CORES=8=, the local reference checkouts behaved as follows on 2026-06-22:
- =.data/gerbil-poo=: first measured =gxpkg build= finished in 324s; a second run was stopped after 247s with no useful additional evidence.
- =.data/gerbil-utils=: first measured =gxpkg build= finished in 377s; a second run finished in 58s.

This means build-file simplicity and build speed must be analyzed separately. =clan/building= gives a small, conventional package entry point and lets =std/make= use =GERBIL_BUILD_CORES=. It does not make a cold or timestamp-invalid Gerbil compilation free. The durable engineering improvement for =poo-flow= is therefore:
1. keep =build.ss= thin and package-manager-owned;
2. keep package discovery positive by scanning the intended include dirs;
3. reduce expensive POO/macro expansion in hot modules;
4. use =GERBIL_BUILD_CORES= for parallel compiler work instead of custom thread orchestration in =build.ss=.
** 10.06.39.30 Finding: Gerbil Bootstrap Uses Direct Compiler Groups
:PROPERTIES:
Gerbil bootstrap is a different path. =src/build/build1.ss= imports =:gerbil/compiler=, defines explicit module groups, calls =compile-module= for each module with =parallel: #t=, then flushes =gsc= work once with =execute-pending-compile-jobs!=. =src/build/build-bach.ss= and =src/build/build-libgerbil.ss= go further by creating explicit Scheme/C/object tasks and using wait groups or pending compile jobs.

That strategy avoids =std/make='s package dependency tables, but it recompiles the selected modules. It is suitable for bootstrap or a deliberate full rebuild mode, not as the default =gxpkg build= path for normal package iteration.
** 10.06.39.40 POO Flow Build Direction
:PROPERTIES:
The correct next implementation is:
1. Keep default =gxpkg build= on =std/make= so package clean/build semantics and incremental checks stay intact.
2. Use =:clan/building= as the normal package build facade, matching =gerbil-poo= and =gerbil-utils=.
3. Keep =build.ss= responsible only for package include dirs and true special targets: the nono =gsc:= + =ssi:= FFI shim, the CLI executable, and the small user-interface entry surface that must be package-managed.
4. Treat =src= and root =t= as the ordinary package surface. =.data=, docs, generated state, user examples, and fixtures stay out because they are not included, not because the build file carries a long negative directory list.
5. Keep direct =compile-module= as an opt-in full-rebuild diagnostic strategy only, useful for measuring Gerbil bootstrap-style behavior or diagnosing =std/make= graph overhead.
6. Continue reducing heavy POO macro expansion in module-system files. The earlier =object<-alist= fixes are aligned with this because they reduce compile-time expansion pressure before any build runner optimization starts.

The core mistake to avoid is treating batch size or a hand-written giant stage list as the architecture. Build scripts should expose package boundaries; they should not become a second source tree model.
** 10.06.39.50 Finding: POO Construction Can Be A Compile-Time Hot Path
:PROPERTIES:
The build slowdown is not only a runner problem. Large POO object literals can become expensive macro-expansion work when a module constructs broad nested objects with =.o= at compile time. The problem showed up most clearly in presentation/profile modules: the compiler spent substantial time in expander, hash, table, and GC code before useful =gsc= parallel work began.

For package modules that produce large mostly-data POO values, prefer a smaller macro-expansion surface:
1. Use =object<-alist= for broad, data-shaped object construction.
2. Keep =.o= for compact, idiomatic user-facing objects and small prototypes.
3. Split large presentation/profile objects into named POO fragments when the fragments have independent meaning.
4. Avoid hot-path repeated =.ref= traversal through nested child POO objects; derive rows or summaries once, then pass stable values forward.
5. Keep POO extension semantics intact. The optimization target is macro expansion and construction shape, not replacing POO with raw alists in public APIs.

This fits the existing POO rules: public extension surfaces remain POO-native, but implementation modules should choose the POO construction form with the smallest compile-time footprint that still preserves object/prototype semantics.
** 10.06.39.60 Implementation Step: Thin =poo-flow= Build Entry
:PROPERTIES:
The current build implementation has been corrected toward the =gerbil-poo= / =gerbil-utils= pattern:
- =build.ss= imports the clan filesystem/building helpers, but ordinary source discovery is project-owned and positive: scan only the include dirs that define the package surface;
- package discovery is positive: runtime modules come from =src=, and package-managed tests come from =t= while =fixtures= and =scenarios= stay outside broad discovery;
- integration-only fixture wrappers such as =t/fixtures/object-load-valid/objects.ss= are explicit test build-spec entries, so fixture source remains outside package scanning while package-local imports have real compiler artifacts;
- the nono =gsc:= + =ssi:= FFI shim, the CLI executable, and the small user-interface entry surface remain explicit build-spec entries;
- =GERBIL_BUILD_CORES= remains the normal parallelism knob, matching the reference packages and avoiding custom thread logic in =build.ss=.
- clean optimized package builds now run a small uncached =runtime-bootstrap= stage for durable runtime store dependencies before the main package stage. This is a dependency-order repair for modules such as =durable-runtime-store=, =durable-runtime-store-backend=, =durable-runtime-store-operation=, and =durable-recovery-scenario=; it is not a fixed worker-count policy. The main package stage still self-schedules through =GERBIL_BUILD_CORES= or the host CPU count.
- focused CLI compiler/test gates use the project package-local =.gerbil/lib= loadpath and never depend on a user's =~/.gerbil/lib= cache. This is fixed as benchmark scenario =GERBIL-SCHEME-AGENT-R039= under =t/scenarios/performance/focused-build-loadpath/benchmark.ss=: the optimization is not raw speed alone, but moving the focused build surface toward Gerbil/gxpkg's package-local build model while preventing stale user-cache artifacts from entering the compiler path.
- package-local compilation stays owned by =gxpkg env=. A bare =direnv exec . printenv GERBIL_PATH= returns no package path, while =direnv exec . gxpkg env printenv GERBIL_PATH= returns the project-local =.gerbil= directory. =build.ss= therefore must not compute its own local cache path; it rejects package compile/clean when =GERBIL_PATH= is missing and lets =gxpkg env= provide the package-local environment. The old direct path was measured at =103.54s= and emitted artifacts under =~/.gerbil/lib=; the testing path is now =gxpkg env gxi build.ss test= through the harness testing framework, while package receipt refresh remains =gxpkg env ./build.ss compile= with package-local output under =.gerbil/lib=.

The next refinement is structural: keep future package-owned modules under explicit include surfaces. If a new tree needs package compilation, it should be added as an include surface with a clear reason instead of expanding a negative exclude list.

This keeps the ownership boundary clear. =std/make= and =gxpkg env= still own package compilation; the compiled CLI owns cheap status checks for agents deciding whether that package compile is necessary.

** 10.06.39.70 Field Report: Durable POO Performance Work
:PROPERTIES:
The most useful optimization evidence from this pass was not a single faster build command. It was the separation of four different failure modes that had been collapsed into one "Gerbil is slow" symptom:
1. package build overhead belongs to =gxpkg build= / =std/make=;
2. fast compile gates should compile the selected roots directly;
3. broad POO literals and repeated object traversal can become compile-time or runtime hot paths;
4. aggregate test roots can hide memory growth unless the roots are semantic boundaries.
This separation is now the engineering rule for future POO optimization work. A fix is only durable when it lands at the owner boundary that created the cost. Shell wrappers, fallback behavior, giant negative file lists, or arbitrary test group sizes can reduce one receipt while leaving the next agent with the same root problem.

The performance scenarios under =t/scenarios/performance= are the place to preserve this evidence. Each scenario should state the optimization focus in its =benchmark.ss= metadata, keep a local =maxRssMb= budget, and exercise one concrete object boundary. Current examples cover object/list merge, extension list merge, inheritance chains, validation summaries, POO construction, catalog validation, clone override, cross-contribution targeting, flow-strand registry merge, and loop contract projection. These scenarios are intentionally narrow: they protect the exact shape that regressed instead of relying on one large end-to-end test to explain every cost.

The benchmark policy should stay strict but honest:
1. add or update a scenario before optimizing a large POO object path;
2. keep RSS budgets close to the scenario that owns the allocation risk;
3. record wall-clock receipts for representative local runs, but do not pretend one host-specific time is a portable contract until the runner records a calibrated =timeBudgetMs= or equivalent field;
4. treat a missing benchmark field as a gate gap, not as permission to hand-wave the result.

The compile/test gate has the same rule. The fast gate for tests is direct root compilation, not a full package build through =gxpkg env=. The current fast path compiles only the semantic test roots, which reduced the old package-sized test run from about =76.10s= and =1.98GB= RSS to repeated local receipts in the =6.71s= to =9.41s= range with about =229MB= RSS. Full package builds still belong to =gxpkg build= or =gxpkg build -R -O=; they should not be used as the default focused performance gate.

For runtime tests, semantic roots are the guardrail. Splitting a high-RSS aggregate is correct only when the split names a real owner boundary, such as contract, integration, performance, catalog fixture, load fixture, or kernel/profile behavior. Splitting merely to lower a process maximum is not a structural fix. If a root still shows high RSS after the obvious fixture moves, the next step is to find the retained module/object owner and add a scenario that proves the specific allocation pattern.

The durable pattern is therefore:
1. reproduce with a focused command and record time/RSS;
2. identify whether the owner is build runner, macro construction, object operation, validation traversal, fixture load, or test-root aggregation;
3. add or reuse a scenario with =optimizationFocus= and =maxRssMb=;
4. change the implementation at that owner boundary;
5. rerun the scenario and the relevant semantic root;
6. document the receipt here when it changes the engineering rule.
** 10.06.39.75 Field Receipt: Harness Native POO Policy Hot Path
:PROPERTIES:
The next optimization pass used the Gerbil Scheme harness native POO scenarios as the pattern source. The useful rules were:
1. keep public extension APIs as POO objects and prototypes;
2. avoid rebuilding adapter objects or child POO objects inside validation loops;
3. replace repeated list membership and duplicate scans with local hash indexes;
4. replace per-entry =filter= plus =append= update loops with one extension index, one base preservation pass, and one materialization pass;
5. keep diagnostics indexed so a duplicate key is reported once without walking accumulated diagnostics.

The owner boundary for this pass was =src/modules/sandbox-core/profile-support/policy.ss=. The code now uses indexed helpers for:
1. registry entry and alias flattening;
2. duplicate registry-key diagnostics;
3. alias target validation;
4. distinct capability append;
5. registry extension overwrite materialization.

Local package-gate receipts on 2026-06-29 with =GERBIL_BUILD_CORES=8=:
- pre-change package receipt recorded earlier in this pass: =gxpkg build -R -O=, =real 40.97s=;
- first build after the policy hot-path rewrite: =gxpkg build -R -O=, =real 26.00s=;
- second package build after that still compiled remaining downstream targets and measured =real 41.77s=;
- third package build still compiled a broad downstream group but returned to =real 24.82s=.

The conclusion is deliberately split. The policy dependency chain improved enough to justify the indexed POO implementation change, but the full package aggregate gate is not a stable no-op: repeated receipts in this pass ranged from =24.82s= to =41.77s= while still recompiling broad downstream groups. The next speedup owner is therefore not more sandbox-policy micro-optimization. It is either the focused =poo-flow= CLI gate returning to the package workspace, or the build-spec/cache owner that decides why the aggregate package gate keeps recompiling broad downstream groups.

This is the boundary that keeps POO-native design and performance work aligned. Public APIs remain object/prototype oriented, while implementation hot paths choose the construction and traversal form that keeps macro expansion, allocation, and validation costs bounded.
** 10.06.39.80 Field Receipt: Generated Receipt Struct Boundary
The next optimization owner was the loop-engine capability receipt. The earlier bounded-alist stopgap fixed the serialization boundary, but it still blurred internal representation and ABI projection. The better Gerbil-native shape is now the project rule:
