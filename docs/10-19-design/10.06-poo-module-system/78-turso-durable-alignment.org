#+title: 10.06.78 Turso Durable Runtime Alignment
#+startup: overview

* Scope
This document records how POO Flow should align Scheme durable policies with the Python runtime Turso backend.

The Scheme side owns durable policy intent. The Python runtime language owns database execution, checkpoint persistence, store IO, and sync driver binding. The contract between them is a fail-closed capability check: a Scheme manifest may require backend capabilities, and the Python adapter must prove those requirements through a runtime receipt before executing durable workflows.

Turso is only a durable storage/backend profile. It must not become the root model for artifact management. Artifact policy is backend-agnostic and should classify artifact kind, scope, lifecycle, permissions, analysis, and publishing requirements before selecting a backend profile such as KV, file store, SQLite/libSQL, object storage, vector index, or Turso.

* Turso Research Summary
Official Turso Python documentation says new Python projects should use =pyturso=. The installed package name is =pyturso=, while the import name is =turso=. The documented connection surface is:

#+begin_src python
import turso

conn = turso.connect("app.db")
#+end_src

The relevant durable capabilities are:

- =pyturso= is the recommended local/embedded Python package for new Turso projects.
- It is built on the Turso Database engine, described by Turso as a rewrite of SQLite with MVCC concurrent writes and async IO.
- =turso.sync= is the local-first sync path for Python when cloud synchronization is needed; reads and writes happen locally, then =push()= sends changes and =pull()= receives remote changes.
- Embedded replicas provide local reads with writes delegated to the cloud primary, but Turso recommends Turso Sync for new sync designs.
- Turso/libSQL vector search supports native vector similarity search without an extension. The documented index/query pair is =libsql_vector_idx(column)= and =vector_top_k(idx_name, q_vector, k)=.
- Vector indexing uses DiskANN-style approximate nearest neighbor search, trading some accuracy for speed on large tables.
- Turso positions this as useful for AI agents, on-device RAG, local vector search, and offline-ready workflows.

* Durable Capability Model
The backend capability exposed by the Python runtime is:

#+begin_src text
Backend =
  { driver
  , driver-package
  , driver-version
  , connection-module
  , concurrent-writes
  , sync-model
  , ai-vector-search
  , vector-index
  , vector-query
  }
#+end_src

For the current Turso backend:

#+begin_src text
driver              = turso
driver-package      = pyturso
connection-module   = turso
concurrent-writes   = true
sync-model          = local-first-push-pull
ai-vector-search    = true
vector-index        = libsql_vector_idx
vector-query        = vector_top_k
#+end_src

The database path or URL is deliberately not written into the adapter receipt. A local path or remote URL may carry deployment-sensitive information. The receipt records capability facts, not connection secrets.

* Scheme Policy Alignment
The Scheme durable policy manifest may declare backend requirements with =backend-*= fields:

#+begin_src text
backend-driver=turso
backend-concurrent-writes=true
backend-sync-model=local-first-push-pull
backend-ai-vector-search=true
backend-vector-index=libsql_vector_idx
backend-vector-query=vector_top_k
#+end_src

Python parses these into the manifest =backend= requirement map. A Turso store or checkpointer must validate the requirement map before opening the driver connection.

Core invariant:

#+begin_src typst
$
forall r in "Requirements"(P),quad "Capability"("TursoRuntime") models r
$
#+end_src

Operationally:

- If Scheme requires =backend-driver=turso=, the Python adapter must prove =driver=turso=.
- If Scheme requires =backend-concurrent-writes=true=, the Python adapter must prove =concurrent-writes=true=.
- If Scheme requires =backend-ai-vector-search=true=, the Python adapter must prove =ai-vector-search=true=.
- If Scheme requires vector functions, the adapter must prove the expected =vector-index= and =vector-query= names.
- If any requirement is not satisfied, adapter construction fails before a store or checkpointer is used.

* AI Runtime Implications
POO Flow should not put vector storage, embedding generation, or retrieval execution into the Scheme control plane. Scheme should describe the policy shape:

- whether a workflow requires vector retrieval,
- which checkpoint and replay policy applies,
- which tool/sandbox scope may read or write durable memory,
- whether local-first/offline semantics are required,
- whether a human or subagent handoff can resume from a durable checkpoint.

Python, Marlin, or another runtime language should execute those IO-heavy parts. The durable adapter receipt is the bridge that lets proof gates and agents check that the runtime backend satisfies the Scheme policy.

* Concurrency Implications
Turso's MVCC concurrent-write claim is useful for agent workloads where many sessions may update independent checkpoint/store namespaces. POO Flow should still model concurrency at the policy level:

- Use per-session/thread checkpoint IDs.
- Keep namespace prefixes explicit, such as =(tenant agent session)=.
- Do not use global mutable keys for agent loop state.
- Treat sync as explicit runtime lifecycle: local write, durable receipt, optional push, optional pull.
- Never claim cross-device convergence unless the selected sync backend and conflict policy are declared.

* Current Python Runtime Contract
The current Python implementation exposes:

- =TursoRuntimeGraphBackend= for capability metadata.
- =turso_runtime_graph_backend= for runtime capability discovery.
- =TursoRuntimeGraphStore= for namespace/key durable state.
- =TursoRuntimeGraphCheckpointer= for thread checkpoints and history.
- =RuntimeDurableAdapter.turso= for Scheme manifest to Turso runtime binding.

The adapter receipt includes the capability fields listed above. This receipt is the durable runtime fact consumed by tests, proof gates, and downstream runtime integration.

#+begin_src typst

#+end_src

and:

#+begin_src typst
$
"RequiresConcurrentCheckpoint"(P) =>
  "ReceiptField"(R, "concurrent"_"writes", "true")
$
#+end_src

This keeps the formal model small: Lean does not need to prove Turso's database engine. Lean only proves that our Scheme policy cannot be handed to a runtime backend whose declared receipt contradicts the required durable capability.
