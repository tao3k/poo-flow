#+TITLE: 10.06.09 M0-M5 Delivery Plan

* 10.06.09.01 Scope

This milestone plan turns the module-system work into a maintainable delivery
track.  It is intentionally split from the design narrative so implementation
status, verification gates, and remaining work can change without rewriting the
architecture documents.

The invariant is:

- =src/modules= owns the upstream module system implementation.
- =user-interface= owns the downstream practice template.
- =poo-flow= emits Scheme-side contracts, projections, receipts, and manifests.
- =marlin-agent-core= owns backend execution, LLM/API integration, and runtime
  handler lifecycle.

* 10.06.09.02 M0: Stabilize The Current Slice

Status: in progress.

Current landed work:

- =build.ss= uses the package build shape and supports =gxi build.ss compile=.
- =src/modules/module-system.ss= is the public module-system facade.
- =src/modules/user-config.ss= owns the thin downstream config projection.
- =src/extensions/agent-sandbox-nono-c-binding.ss= has been split into
  descriptor and runtime owners.

Exit criteria:

- No untracked source files remain from intentional splits.
- =asp gerbil-scheme check --workspace . --summary= passes.
- =gxi build.ss compile= passes without project-local load path hacks.
- Targeted user-interface and module-system tests pass.
- Full unit test pass is recorded before commit.

* 10.06.09.03 M1: Documentation Closure

Status: in progress.

The 10.06 document set remains the source of truth for module-system design.
The code has moved faster than the roadmap, so this milestone updates docs with
actual implementation surfaces instead of aspirational descriptions.

Required doc updates:

- Implementation roadmap records the concrete M0-M5 status.
- Downstream onboarding points to =user-interface/config.ss= and
  =pooUserConfigPresentation=.
- Doom/Nix principles stay as interface guidance, not a reason to copy Doom's
  package-management surface.
- Runtime boundary docs state that Marlin owns backend execution.

* 10.06.09.04 M2: User Interface Contract

Status: in progress.

The user interface is a small declarative layer.  It should feel closer to
Doom's module selection ergonomics while preserving Nix-style explicit data
receipts.

Public downstream entrypoints:

- =poo-user-module-selection=
- =poo-settings=
- =pooUserConfig=
- =pooUserConfigPresentation=

Non-goals:

- No descriptor realization in root user files.
- No loader backend registration in root user files.
- No package manager clone of Doom's =packages.el=.
- No runtime execution from presentation helpers.

* 10.06.09.05 M3: Doctor And Resolver Completeness

Status: in progress.

Doctor output should answer what happened without changing activation behavior.
Loader receipts are data, resolver errors are typed failures, and source doctor
presentation must make missing loaders obvious.

Required presentation signals:

- =load-status= for =ok=, =error=, or =empty= source selections.
- =missing-count= for source refs not loaded by any backend.
- =runtime-executed = #f= on every doctor/presentation edge.
- =runtime-owner = "marlin-agent-core"= wherever backend lifecycle ownership is
  exposed.

* 10.06.09.06 M4: Marlin Runtime Boundary

Status: in progress.

The module system lowers Scheme-side module data into replayable evidence.  It
does not create backend handlers, parse Scheme source in Marlin, or call LLM/API
surfaces.

Handoff shape:

- POO Flow emits module selections, option receipts, loader receipts, merged
  options, diagnostics, and runtime manifests.
- Marlin consumes those data products at the runtime boundary.
- Any C binding execution proof remains Marlin-owned even if the descriptor is
  authored here.

* 10.06.09.07 M5: Harness Sync

Status: pending.

Once M0-M4 are green, the Gerbil harness should learn only the stable surface:
root-level downstream config, presentation receipts, and package/build shape.
Do not publish transient implementation internals as harness guidance.

Sync artifacts:

- A downstream onboarding example using =user-interface/init.ss= plus =src/modules/user-interface/config.ss= projection helpers.
- Verification commands for =gxi build.ss compile= and =gxtest=.
- A note that =.data= research checkouts remain reference material.
