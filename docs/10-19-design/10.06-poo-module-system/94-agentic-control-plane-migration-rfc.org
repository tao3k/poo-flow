#+TITLE: RFC 94 — Agentic Control Plane Migration
#+SUBTITLE: Compiler-First Vertical Slices, Evidence Gates, And Canonical Cutover
#+OPTIONS: toc:3 num:t
#+LANGUAGE: en

* Status
Status: *Accepted; implementation planning*

This RFC defines migration order and acceptance gates. It does not authorize implementation before the written RFC suite is reviewed.

* Purpose
The architecture in RFC 90–93 spans organization semantics, composition, runtime ABI, authorization, proof, and evidence. Implementing those layers as one replacement would create an untestable flag day and would obscure semantic drift.

The accepted migration is compiler-first and vertical-slice driven:

- retain the existing POO-native authoring direction;
- compile current scenarios into a shadow Canonical Organization Bundle;
- validate semantics before changing runtime behavior;
- introduce the public C ABI and first-party Python runtime only after the Bundle contract is stable enough to measure;
- connect Cedar, evidence, and Lean through bound artifacts;
- cut over once and remove obsolete public routes.

There is no isolated public “POO Flow v2” namespace and no permanent dual semantic center.

* Decision
Migration proceeds through P0–P6:

* First Vertical Slice
The first slice is deliberately small but crosses every accepted semantic boundary.

** Scenario
A parent agent delegates one restricted tool capability to a child agent.

The child receives a filtered context projection and executes a single external tool effect through a protocol supporting:

- session start;
- child creation or activation;
- capability delegation;
- context projection;
- one effect intent;
- Cedar allow and deny outcomes;
- cancellation;
- checkpoint and restore;
- typed execution result;
- evidence closure.

The first-party Python runtime executes the effect through the installed public C ABI.

** Required Entities
The Bundle contains at least:

- one parent =Agent=;
- one child =Agent=;
- one parent and one child =Principal=;
- one parent and one child =Role=;
- one delegated =Capability=;
- one =Resource= representing the tool boundary;
- one context =Artifact= with provenance and visibility restrictions;
- one protocol and its lifecycle states;
- one effect intent;
- authorization, execution, transition, and proof obligations.

** Required Invariants
The slice validates:

1. the child authority is a subset of the parent's delegable authority;
2. organization parenthood does not grant authority by itself;
3. the child context excludes data outside its declared visibility;
4. the effect is reachable only from the declared protocol state;
5. the effect cannot be dispatched without a matching Cedar decision and token;
6. cancellation prevents later execution transitions;
7. checkpoint and restore retain Bundle digest and epoch identity;
8. every allow, deny, execution result, failure, cancellation, and restore produces the required typed evidence;
9. a stale-epoch or replayed token fails closed;
10. the accepted state transition binds the exact effect intent and runtime receipt.

** Slice Success
The slice is complete only when the same scenario passes:

- Bundle compilation and negative mutation tests;
- a pure C conformance harness;
- the Python production runtime through the public installed ABI;
- Cedar allow and deny cases;
- evidence replay and tamper cases;
- focused Lean obligations;
- failure injection for timeout, cancellation, duplicate submission, checkpoint, and restore;
- the defined hot-path performance matrix.

* P0 — RFC And Vocabulary Closure
** Entry
The approved architecture discussion exists and the current code remains unchanged by this RFC phase.

** Work
- Review RFC 90–94 as one coherent suite.
- Resolve conflicts with existing proof, session, policy, receipt, composition, runtime-boundary, and durability documents.
- Adopt exact meanings for Bundle, facet, effect intent, token, proof receipt, runtime receipt, transition receipt, semantic root, execution root, conformance, mediation, and assurance.
- Record explicit non-goals and downstream boundaries.

** Exit Gate
- All five RFCs contain no placeholders or contradictory ownership claims.
- Every normative concept has exactly one normative contract owner.
- Existing design documents are linked as evidence rather than copied into a new parallel architecture.
- Written repository review approves the suite.

* P1 — Shadow Canonical Organization Bundle
** Entry
P0 is accepted.

** Work
- Add an internal compiler path from one existing POO/session scenario to the Bundle manifest and shared identity universe.
- Produce Organization, Authority, Context, Protocol, and Evidence facets without changing runtime behavior.
- Bind schema version, compiler provenance, epoch, parent digest, facet digests, and Bundle digest.
- Keep the new artifact internal and read-only.

** Exit Gate
- Identical normalized input produces byte-identical Bundle output and digest.
- Every typed identity resolves with the correct kind.
- Existing authoring produces no runtime behavior change.
- Invalid identity, missing reference, duplicate owner, authority widening, context widening, invalid protocol, and evidence deletion fixtures fail with typed errors.
- The shadow Bundle is captured as a focused receipt artifact.

* P2 — Five-Facet Validation And Differential Projection
** Entry
P1 determinism and identity gates pass.

** Work
- Implement each facet's distinct composition algebra.
- Implement cross-facet validation.
- Project the vertical slice into the existing control-plane representation in shadow mode.
- Compare current and Bundle-derived policy, lifecycle, context, composition, and receipt results.

** Exit Gate
- C3 affects only definition lookup and cannot widen authority or information flow.
- Every external effect closes Protocol–Authority–Evidence relationships.
- Every context transfer closes Context–Protocol–Evidence relationships.
- Differential tests explain every difference as either an accepted correction or a defect.
- No unexplained difference is accepted as a compatibility quirk.
- The Bundle is still not a public runtime contract.

* P3 — High-Performance C ABI And Python Runtime
** Entry
P2 establishes stable enough semantics for runtime projection.

** Work
- Generate the C header and library boundary from the Bundle/runtime schema.
- Implement control, hot-event, and bulk-buffer lanes.
- Add the pure C conformance harness.
- Implement the first-party Python production runtime with CFFI out-of-line API mode.
- Exercise the full vertical slice with no private Gerbil access.
- Complete the Gerbil/Gambit FFI and gerbil-mysql binding research gates defined by RFC 92.

** Exit Gate
- C and Python consume only installed headers and the shared library.
- No public ABI exposes =___SCMOBJ=, Scheme GC pointers, Python objects, or Gerbil object layout.
- Handle ownership, buffer ownership, lifetime, release, thread affinity, and error semantics pass sanitizer and fault tests.
- Scalar compatibility, batch, and negotiated zero-copy profiles are distinguished explicitly.
- Python can implement scheduling, effect execution, context/memory handling, cancellation, timeout, retry, checkpoint, restore, and receipts through the public ABI.
- Batch and buffer performance receipts pass for the agreed matrix.
- ABI v1 remains unfrozen until every P3 gate passes.

* P4 — Cedar Authorization, Tokens, And Evidence Chain
** Entry
P3 proves the runtime interface is functionally complete and measurable.

** Work
- Project Cedar policy and entity material from the Bundle.
- Bind policy, entities, projector, Bundle, and epoch digests.
- Issue one-shot AuthorizedEffectToken values after allow decisions.
- Implement decision, execution, transition, and checkpoint receipt types.
- Implement evidence batch roots, replay protection, fork detection, and declared durability profiles.

** Exit Gate
- Every external effect requires a valid, unexpired, unreplayed token.
- Token intent digest and adapter parameters match.
- Deny produces decision evidence and no token.
- New Bundle plus old policy, old token after revocation, stale epoch, parameter substitution, mutable-buffer TOCTOU, duplicate retry, missing receipt, and evidence fork fixtures fail as specified.
- An effect with unknown durable outcome becomes =indeterminate=.
- L2 Evidenced claims are available; L3 mediation is not claimed without isolation or independent observation.

* P5 — Lean Refinement And Formal Link
** Entry
P4 establishes bound authorization and evidence artifacts.

** Work
- Project typed Lean values and theorem statements from the validated Bundle.
- Bind proof manifests and proof receipts to the semantic root.
- Prove the first slice's bounded invariants.
- Connect accepted proof receipts to Bundle activation and token prerequisites where the assurance profile requires them.

** Minimum Theorems
- child delegated authority does not exceed the parent's delegable authority;
- organization hierarchy does not imply an Authority grant;
- cancellation prevents a later execute transition in the modeled lifecycle;
- Bundle, Cedar projection, and transition model digests refer to the same semantic root;
- evidence obligations cannot be silently removed by composition.

** Exit Gate
- Lean checks the typed statements under a locked import environment.
- Invalid or mismatched theorem, Bundle, projection, or import-environment digests are rejected.
- Proof generators and tactics are not treated as proof kernels.
- Runtime conformance and mediation remain separate gates.
- L4 Formally Linked applies only to named proved claims.

* P6 — Canonical Cutover And Removal
** Entry
P1–P5 gates pass for the first vertical slice and the release decision identifies the new canonical routes.

** Work
- Make the Bundle compiler and public runtime ABI the canonical internal/runtime route.
- Update public docs, examples, tests, and package metadata.
- Remove obsolete public authoring, projection, and receipt paths.
- Remove shadow dual-read or dual-write machinery.
- Publish conformance vectors, ABI version, Python runtime, proof/evidence artifacts, and acceptance receipts.

** Exit Gate
- Exactly one public semantic route remains for each replaced capability.
- Old alist, =.def=, raw record, or other retired authoring routes fail fast or no longer exist.
- No public documentation recommends a retired path.
- No test silently exercises the retired path.
- Runtime ABI and artifact versions are explicit.
- The release receipt identifies source revision, Bundle/schema version, ABI version, proof environment, performance results, and known assurance level.

* Shadow Mode Rules
Shadow mode exists only to compare semantics before cutover.

It MUST:

- be internal;
- be read-only with respect to real external effects;
- record both current and Bundle-derived results;
- classify every difference;
- avoid becoming a public compatibility contract;
- have a removal gate and owner.

Shadow mode MUST NOT:

- execute an external effect twice;
- authorize through one path and execute through another;
- write two competing durable truths;
- hide a difference by normalizing it away after comparison;
- remain after P6.

* Test Matrix
** Compiler And Bundle
- golden Bundle fixtures;
- deterministic canonicalization;
- identity-kind and reference failures;
- facet-local composition positives and negatives;
- cross-facet invariant negatives;
- epoch, parent, and activation tests;
- mutation tests for every digest-bound field.

** C ABI
- version and capability negotiation;
- handle ownership and release;
- borrowed-buffer lease safety;
- caller-buffer and arena behavior;
- invalid pointer, length, alignment, and capacity errors;
- thread-affinity and concurrency contracts;
- scalar and batch equivalence;
- cancellation, close, and restore races;
- fuzzing of public packet and state-machine inputs.

** Python Runtime
- built only against installed public headers and library;
- CFFI out-of-line API path;
- no private Gerbil import or representation access;
- C harness and Python trace equivalence under declared nondeterminism;
- production workload, long-run, memory-pressure, and restart scenarios;
- buffer protocol lifetime tests;
- exception and error translation;
- runtime receipt completeness.

** Authorization And Evidence
- allow and deny;
- stale policy, Bundle, and epoch combinations;
- replayed, expired, revoked, or reused token;
- intent/parameter mismatch;
- mutable-buffer TOCTOU;
- retry and duplicate effect;
- effect success with receipt failure;
- hash-chain truncation and fork;
- signer, runtime, or adapter identity mismatch.

** Proof
- successful named theorem set;
- false obligation;
- wrong Bundle digest;
- wrong policy or projector digest;
- wrong import environment;
- valid proof for the wrong epoch;
- proof receipt substituted for an execution receipt;
- execution receipt substituted for a proof receipt.

** Fault Injection
- Cedar unavailable;
- evidence store unavailable before dispatch;
- evidence store unavailable after effect;
- process crash before and after submission;
- timeout concurrent with completion;
- cancellation concurrent with dispatch;
- checkpoint corruption;
- restore under a different Bundle;
- partial batch acceptance;
- runtime restart with outstanding tokens.

* Performance Gate
Performance claims use focused, machine-checkable receipts.

The matrix covers:

- batch sizes =1, 8, 32, 128, 1024=;
- payloads empty, 1 KiB, 64 KiB, and 1 MiB;
- allow, deny, success, runtime error, timeout, cancellation, and restore paths;
- Gerbil in-process, pure C harness, and Python CFFI production paths.

Receipts report:

- FFI crossings;
- allocations by runtime domain;
- bytes copied;
- GC time;
- p50 and p99 latency;
- throughput;
- resident memory;
- batch utilization;
- evidence sealing overhead.

No absolute latency requirement is frozen before a reproducible hardware and build baseline exists. ABI v1 requires a recorded budget and no unexplained regression on the actual hot path.

* Rollback And Recovery
Rollback does not restore mutable in-memory semantics silently.

For Bundle semantics:

- every change creates a new epoch;
- rollback creates another new epoch whose content may correspond to an earlier state;
- receipts remain bound to the epoch under which they were created.

For migration:

- before P6, the current canonical runtime path remains available while the new path is shadowed or explicitly gated;
- cutover requires a release decision and receipt;
- after P6, recovery reactivates a validated Bundle/ABI release or creates a new epoch; it does not restore obsolete public semantics.

For effects:

- an effect known not to have dispatched may be retried under policy;
- an effect known to have completed follows its recorded transition;
- an effect with unknown outcome becomes =indeterminate= and requires observation, reconciliation, or compensation;
- a one-shot effect is never blindly retried from an ambiguous state.

* Compatibility Policy
The migration distinguishes authoring compatibility from external ABI compatibility.

** Authoring
When a canonical POO-native authoring surface replaces an older public form, the old route is removed after the cutover gate. Permanent aliases, duplicate semantics, and stale examples are rejected.

** Bundle Schema
Breaking semantic changes increase the Bundle schema major version. A consumer never ignores unknown semantic content silently.

** C ABI
Breaking ABI changes increase the ABI major version. Struct-size fields, explicit versions, and capability negotiation support compatible extension. Unversioned parallel entry points are not retained as informal compatibility.

** Evidence And Proof
Receipts and proof artifacts identify their schema, Bundle, epoch, projector, and verification environment. A new verifier may validate old artifacts only under an explicit supported schema contract.

* Downstream Runtime Boundary
This migration implements no Rust, TypeScript, or other downstream runtime.

Downstream work may begin after a sufficiently stable public C ABI and conformance pack exist, but it is accepted independently. A downstream runtime claim requires:

- declared ABI and schema versions;
- conformance results;
- capability profile;
- runtime and adapter identity;
- evidence support;
- assurance level;
- any sandbox, isolation, or attestation used for mediation claims.

POO Flow does not grant assurance merely because a downstream adapter links successfully.

* Risks And Controls
| Risk | Control |
|------+---------|
| A monolithic rewrite hides semantic drift | Shadow Bundle and phase-specific differential receipts |
| C ABI freezes implementation accidents | Bundle-first ordering and ABI freeze gate |
| Python becomes a private semantic owner | Public installed-ABI-only build and tests |
| C3 precedence widens security semantics | Facet-specific algebra and negative fixtures |
| High-performance views create second semantics | One generated schema and native/canonical differential validation |
| Zero-copy introduces lifetime or TOCTOU defects | Explicit leases, freeze rules, ownership tags, and mutation tests |
| Cedar allow is treated as total safety | Claim discipline and evidence closure |
| Lean artifact is used as runtime proof | Separate conformance, mediation, and proof gates |
| Shadow mode becomes permanent | P6 removal owner and gate |
| Old public routes survive indefinitely | Canonical cutover and stale-route deletion |
| Evidence chain is truncated or forked | Signed checkpoints and external anchors where required |
| Retry duplicates an external effect | One-shot tokens, idempotency contract, and indeterminate state |

* Security Considerations
- Shadow mode MUST remain read-only for external effects so comparison cannot execute an effect twice.
- Canonical cutover MUST reject mixed Bundle epochs, policies, tokens, and receipt schemas.
- High-performance native views MUST pass the same authorization, mutation, and evidence tests as the portable path.
- A Python production runtime remains an implementation owner, not the semantic owner.
- P6 MUST remove retired public routes so a caller cannot select a weaker legacy semantic path.
- Rollback creates a new validated epoch; it never rewrites historical evidence or silently revives revoked authority.

* Decision Notes
** Compiler-First Accepted
Runtime contracts derived before organization semantics would fossilize accidental representations. The Bundle becomes observable in shadow mode before runtime replacement.

** Public V2 Namespace Rejected
A public v2 would encourage indefinite dual semantics. Migration uses internal shadow comparison followed by one canonical cutover.

** Full Agent Society As First Slice Rejected
The first slice must exercise every boundary without requiring markets, debates, dynamic swarms, distributed scheduling, or multiple runtime languages.

** Downstream Runtime Implementation Deferred
Rust and TypeScript adapters are valuable consumers of the C ABI, not prerequisites for proving the POO Flow interface complete. Python and the C harness close the first-party implementation gate.

** Numeric Performance Target Before Baseline Rejected
An arbitrary nanosecond target is not portable evidence. The RFC requires a reproducible matrix, receipts, and a release budget established on the actual hot path.

** Permanent Compatibility Layer Rejected
Compatibility is bounded migration machinery. A canonical replacement retires its old public route.

* Acceptance Criteria
This migration RFC is satisfied when:

1. RFC 90–94 pass written review.
2. The first vertical slice is represented by one deterministic Bundle.
3. P1–P6 each produce focused exit receipts.
4. Shadow mode is read-only for external effects and removed at cutover.
5. The C and Python paths use only the installed public C ABI.
6. The performance, conformance, fault, authorization, evidence, and proof matrices pass for the first slice.
7. Every assurance claim names its satisfied level and trusted boundary.
8. ABI v1 freezes only after the Gerbil/Gambit/CFFI/gerbil-mysql research and performance gates close.
9. P6 leaves one canonical public semantic route for each migrated capability.
10. Obsolete public routes, documentation, examples, and tests are removed.
11. No Rust, TypeScript, or other downstream runtime is implemented as part of this migration tranche.
12. The release receipt binds source revision, Bundle/schema version, ABI version, runtime identity, proof environment, performance evidence, and known limitations.
