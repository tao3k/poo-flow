* 10.06.86 Doom Emacs Core Module Boundary Research
** 10.06.86.10 Purpose
This note is the current Doom Emacs calibration document for =poo-flow='s module
system.  It is intentionally more concrete than the short module-system design
summary.  Agents should read this document before changing module kernel,
factor, plugin, macro, or fixture semantics.

The goal is not to port Doom Emacs.  The goal is to extract a durable boundary
model:

- what Doom keeps as core mechanics;
- what Doom exposes as user-facing module and feature selection;
- how macros make authoring concise without being the semantic owner;
- how diagnostics and sync commands keep the module graph operable;
- how =poo-flow= should map those lessons into Gerbil POO values and
  functional composition.

** 10.06.86.30 Doom's Boundary Model
Doom's useful design is a boundary model, not a file layout.

It has five separable planes:

1. Declaration plane.
   The user declares selected modules and feature flags through compact
   authoring syntax.  The declaration is optimized for user ergonomics, not for
   being the internal semantic model.

2. Registry and context plane.
   Core module code owns module identity, enabled state, flags, feature
   metadata, path/context lookup, and phase membership.  This plane is what
   makes a module coherent across many files and macros.

3. Phase and contribution plane.
   Module contributions are split by phase names such as init, config, and
   packages.  The important idea is not those exact names; the important idea is
   that contribution phases are explicit enough for ordering, diagnostics, and
   selective loading.

4. Authoring macro plane.
   Macros such as package helpers, deferred configuration helpers, and hook
   helpers make common authoring paths concise.  A macro is useful because it
   owns one narrow authoring problem.  It is dangerous if it becomes a hidden
   second module system.

5. Operations plane.
   Sync, doctor, env, and CLI commands reconcile and diagnose the system.  They
   are not just convenience commands.  They make the module graph inspectable
   and maintainable after users add modules, packages, flags, and profiles.

This separation is why the core is not merely a set of names.  It gives users a
small declaration surface while preserving a maintained internal substrate.

** 10.06.86.35 Mechanism Walk-Through
The useful Doom flow can be described without copying its implementation:

#+begin_src text
user declaration
  -> module registry/context normalization
  -> enabled module and feature set
  -> phase contribution discovery
  -> package/config/hook authoring helpers
  -> sync/doctor/env operational reconciliation
  -> loaded editor behavior
#+end_src

The important part is where ownership changes:

- the user declaration owns intent;
- the registry/context layer owns identity and validity;
- phase discovery owns contribution ordering;
- macros own authoring compression only;
- operations own reconciliation and diagnostics;
- runtime/editor behavior is the final consumer, not the module definition.

The equivalent =poo-flow= shape should be:

#+begin_src text
use-module / use-composition declaration
  -> module descriptor and kernel-factor materialization
  -> public factor/profile selection
  -> plugin slot validation
  -> doctor/projection receipts
  -> runtime-facing projection values
  -> Python, Rust, Marlin, or other runtime execution
#+end_src

This is the reason kernel factors should be inherited automatically.  The user
should not compose graph core, resolver core, lifecycle core, runtime handoff
core, and diagnostics core by hand.  Those are the maintained substrate.  The
user should select public factors and plugins that the module owner has made
stable.

When an Agent implements a new user-facing module form, the proof obligation is
not "does the syntax look compact".  The proof obligation is:

- what internal kernel was inherited;
- which public factors were selected;
- which plugin slots were used;
- which receipts prove the projection;
- what runtime-facing value was produced;
- which private hooks remained private.

** 10.06.86.40 What "Core" Means Here
Core means the maintained substrate that users should not replace in normal
configuration.

For Doom, this includes:

- module identity and grouping;
- enabled/disabled state and feature flags;
- module context and path lookup;
- contribution phase membership;
- package declaration and sync integration;
- profile/context ownership;
- doctor/sync/env diagnostics.

The key design lesson is that users are not expected to override the core to get
a working system.  They select modules and flags.  They may extend through
documented surfaces.  They do not rebuild the loader, registry, package sync, or
diagnostic model per configuration.

For =poo-flow=, this means every serious module must have a closed kernel.  The
kernel is not a convenience profile.  It is the contract that makes a module
remain one thing across user profiles, plugin extensions, runtime projections,
and test fixtures.

** 10.06.86.50 What "Feature" Means Here
Doom's feature surface is not the same thing as arbitrary internal hooks.

Useful feature properties:

- a feature is named;
- it is selected or implied through a module boundary;
- it can affect module contribution behavior;
- it is known to diagnostics;
- it does not require users to override core mechanics.

For =poo-flow= this maps to public module factors:

- public factors are declared by the module owner;
- public factors have defaults and validation;
- public factors project into receipts and runtime-facing values;
- public factors can be enabled, disabled, narrowed, or parameterized only when
  the module declares that behavior.

A public factor is not a private kernel hook with a nicer name.  If changing it
can destroy the module's basic contract, it belongs in the kernel and should not
be public.

** 10.06.86.60 What "Plugin" Means Here
Plugins are upward extensions.  They should extend declared surfaces, not
replace the kernel.

The Doom lesson is that package and module extension behavior becomes robust
only when the system knows how to sync, diagnose, and contextualize it.  Merely
allowing users to run arbitrary code is not a module system.

For =poo-flow= a plugin should:

- attach to a declared extension slot;
- publish capability and projection contracts;
- produce receipts that doctor/projection lanes can inspect;
- preserve required lifecycle, resolver, and safety behavior by default;
- be selected explicitly by a user profile or by a higher-level module profile.

The plugin should not:

- delete required kernel factors;
- attach to undeclared hooks;
- mutate internal handoff schema;
- override resolver semantics;
- weaken required safety policy;
- replace diagnostics with hidden side effects.

** 10.06.86.70 What Macros Should Learn From Doom
Doom uses macros because a large Lisp-family system needs a compact authoring
surface.  =poo-flow= should copy the discipline, not the exact syntax.

Good macro properties for =poo-flow=:

- a macro has one owner and one authoring purpose;
- expansion produces ordinary Gerbil bindings, POO descriptors, profile values,
  factors, receipts, projections, or functional combinator expressions;
- the expansion can be inspected through the same accessors as hand-written
  values;
- diagnostics can explain what the macro contributed;
- macro syntax does not become a second alist or record DSL.

Bad macro properties:

- expansion runs activation;
- expansion wakes runtimes, allocates worktrees, calls connectors, or starts
  agents;
- expansion bypasses kernel validation;
- expansion hides option merge conflicts;
- expansion exposes private kernel hooks as normal user knobs;
- expansion changes plugin attachment semantics silently.

The correct Gerbil shape is: macro syntax is authoring sugar over POO-native and
functional values.  It is not the owner of the runtime.

** 10.06.86.80 POO Flow Mapping Table
| Doom concept | POO Flow concept | Design rule |
|--------------+------------------+-------------|
| Compact declaration syntax | =use-module= and module-specific authoring macros | Keep user syntax small, but expand into inspectable POO values. |
| Module registry/context | Module kernel factors | Core module semantics are owned by the module and closed by default. |
| Enabled state and flags | Public module factors and profile selections | Users select declared factors; they do not override kernel internals. |
| Init/config/packages phases | Contribution phases and projections | Keep phase ownership explicit enough for ordering and diagnostics. |
| =modulep!= style checks | Explicit data predicates and receipts | Prefer inspectable predicate inputs over hidden ambient source context. |
| Package declaration/sync | Runtime/package/backend projection lanes | Reconcile external effects through explicit operations and receipts. |
| =doom doctor= / =doom sync= | =profile-doctor=, projection receipts, scenario diagnostics | Diagnostics are first-class operations, not afterthoughts. |
| Profiles | Higher-level composition profiles | Profiles select modules/factors/plugins; they do not replace module kernels. |
| Built-in modules | =src/modules/<domain>= module owners | Concrete modules live under domain owners, not under the core module-system mechanics. |
| Macro helpers | Gerbil declarative/procedural macros | Macros compress authoring and expand into normal POO/functional values. |

** 10.06.86.90 Agent Decision Procedure
When an Agent changes =poo-flow= module semantics, use this procedure:

1. Identify the module owner.
   If the change is generic mechanics, the owner is =src/module-system/*=.  If
   the change is a concrete domain feature, the owner is =src/modules/<domain>/*=.

2. Classify the change.
   It must be one of:
   - kernel factor;
   - public module factor;
   - plugin factor;
   - fixture or scenario;
   - macro authoring helper;
   - operation/diagnostic lane.

3. Apply the boundary rule.
   - Kernel factor: closed by default, not user-overridable through normal
     =use-module= or =compose=.
   - Public factor: configurable under module-owned contract.
   - Plugin factor: attaches through declared slots and cannot delete kernel
     safety.
   - Fixture/scenario: proves behavior, not an extension API.
   - Macro helper: expands to inspectable POO or functional values.
   - Diagnostic lane: reports and reconciles; it does not silently own semantics.

4. Check user-interface pressure.
   If a user would need to compose internal hooks to get a working module, the
   kernel boundary is wrong.  Selecting the module should bring the maintained
   kernel with it.

5. Check runtime pressure.
   If a runtime language must understand private Scheme internals to run a
   profile, the projection boundary is wrong.  Runtimes should receive stable
   projection values and ABI-facing receipts.

6. Check diagnostics.
   Every new public factor or plugin slot needs enough receipt data for doctor,
   scenario, or projection lanes to explain it.

** 10.06.86.100 What POO Flow Should Not Copy
Do not copy these Doom traits directly:

- dynamic editor-state mutation during module loading;
- implicit file loading as the semantic core;
- path-derived identity as the only module identity;
- macro context that cannot be inspected as data;
- package manager behavior mixed directly into module semantics;
- user configuration that can replace core registry mechanics;
- terminology that exposes internal implementation pieces as user choices.

=poo-flow= is a Scheme control plane for policy, composition, and runtime
projection.  It should absorb Doom's boundary discipline while staying
POO-native, data-first, receipt-driven, and runtime-neutral.

** 10.06.86.110 Design Consequences For Current Module Work
The module-system design should stay small at the public surface:

#+begin_src scheme
(use-module funflow)
#+end_src

Selecting the module should bring its kernel factors.  A user should not have
to spell out graph core, router core, lifecycle core, resolver core, handoff
core, or diagnostics core just to get a usable flow module.

Public factors and plugins should be explicitly declared:

#+begin_src scheme
(use-module funflow as ff
  (profiles github-ci))

(use-composition repo-ci
  (compose
    (profile ff github-ci)))
#+end_src

The user sees =github-ci= because it is a public factor/profile.  The user does
not see the internal graph, resolver, lifecycle, or projection factors unless
the module intentionally exposes advanced POO abstractions for maintainers.

The implementation implication is:

- module activation must have kernel defaults;
- profile composition must distinguish kernel-owned factors from public factors;
- plugin attachment must validate declared slots;
- doctor/projection receipts must expose what was selected, what was inherited,
  and what remains internal;
- macro expansion should produce the same values as explicit POO construction.

** 10.06.86.120 Relationship To Agent Framework Research
This Doom research is not an agent-framework audit.  Its value for AgentTeams,
AutoGen, CrewAI, LangGraph, OpenRath, and Codex-style role kernels is the
module boundary:

- keep the execution substrate generic;
- define built-in role profiles as public module factors;
- keep lifecycle and resolver kernels closed by default;
- allow multi-agent modes to attach through declared plugin slots;
- project runtime handoff data through stable values and receipts.

This prevents every downstream framework from forcing a new namespace and a new
set of hand-written hooks.  The framework-specific vocabulary becomes a profile
or composition surface over the same kernel/factor/plugin model.

** 10.06.86.130 Open Gaps To Track
The current =poo-flow= design still needs implementation-level checks for:

- kernel-factor visibility: normal =use-module= should inherit kernels without
  exposing them as user-composed profiles;
- public-factor declaration: factors need contracts, defaults, and projection
  behavior;
- plugin-slot validation: plugin contributions must attach only where declared;
- macro expansion receipts: user macros should be explainable by doctor and
  scenarios;
- profile-doctor output: diagnostics should show selected modules, public
  factors, plugin factors, inherited kernels, and rejected overrides;
- fixture discipline: scenarios can inspect internal receipts but must not turn
  internal hooks into public API.

These are the concrete checks that keep the Doom lesson from remaining a shallow
analogy.
