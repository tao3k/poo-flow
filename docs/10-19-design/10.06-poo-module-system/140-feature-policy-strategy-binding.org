#+title: RFC 140: Feature Policy and Strategy Algebra Binding
#+status: implemented

* Decision

Phase 3E introduces POO-native policy and strategy contributions plus an
immutable =FeaturePolicyStrategyBinding= receipt.

The binder consumes an accepted =FeatureDomainCaseAssembly=.  It does not
modify the closed Domain Case, create a second Case representation, construct a
runtime execution policy, evaluate Cedar, or call an adapter.

Three existing owners remain authoritative:

1. =FeatureCompositionPlan= owns dependency-qualified contribution order;
2. the closed Domain Case owns the single policy-algebra and strategy-algebra
   identities declared by its Case components;
3. POO =role-compose= owns prototype composition and override semantics.

The new layer only proves that ordered Feature extensions can bind to those
owners without collapsing object inheritance, contracts, policy, and strategy
into one mechanism.

* POO contribution contract

A policy contribution and a strategy contribution are immutable POO objects.
Each carries:

- a semantic =contribution-id=;
- the Domain Case =algebra-id= it extends;
- a POO =prototype= containing its declarative extension slots.

The ordinary declaration surface is:

#+begin_src scheme
(defpoo-feature-policy-contribution agent-policy
  (contribution-id 'agent-policy)
  (algebra-id 'agent-policy-algebra)
  (prototype agent-policy-role))

(defpoo-feature-strategy-contribution agent-strategy
  (contribution-id 'agent-strategy)
  (algebra-id 'agent-strategy-algebra)
  (prototype agent-strategy-role))
#+end_src

These objects are passed through the existing =policy-contributions= and
=strategy-contributions= Feature clauses.  Raw symbols, alists, records, and
unnamed =(lambda (self super) ...)= hooks are not valid binding inputs.

The constructors remain available for functional generation of large Feature
families, but their result is the same POO object accepted by the declarative
macros.

* Binding algorithm

=feature-policy-strategy-binding= performs the following pure control-plane
transition:

1. require a valid, accepted Feature Domain Case assembly;
2. read the closed Domain Case's policy and strategy algebra identities;
3. validate every ordered contribution as the correct POO contribution kind;
4. reject duplicate contribution identities using an ephemeral hash index;
5. require each contribution's algebra identity to equal the Domain Case owner;
6. compose contribution prototypes in dependency order using =role-compose=;
7. return separate immutable policy and strategy binding receipts.

The validation/index pass is linear in the number of contributions.  Prototype
composition retains the cost and semantics of the existing POO role owner.  No
quadratic duplicate scan or hidden global registry is introduced.

The declarative binding spelling is:

#+begin_src scheme
(defpoo-feature-policy-strategy-binding agent-binding
  (from-assembly agent-domain-case-assembly))
#+end_src

=require-feature-policy-strategy-binding= converts a rejected receipt into an
explicit build failure carrying the structured diagnostics.

* Ordering and override semantics

Feature dependencies are the only ordering authority.  A parent Feature's
contribution appears before a dependant Feature's contribution.  The binder
reverses that ordered prototype list when calling =role-compose=, matching the
existing Domain Case component composition rule: the more specific dependant
can override a shared slot while unrelated parent and child slots are both
preserved.

The binder never invents missing dependency edges and never sorts by
contribution id.  Therefore the resulting prototype is traceable to the same
resolver receipt used by component and contract assembly.

* Rejection semantics

Binding is rejected with POO diagnostic objects when any of these conditions is
observed:

- the input is not a Feature Domain Case assembly;
- the assembly was rejected by the existing Domain Case owner;
- a contribution is not the required POO kind or its prototype is not a POO
  object;
- a contribution id is repeated within its channel;
- contributions exist but the Domain Case declares no matching algebra;
- a contribution names a different algebra identity;
- POO role composition reports a real prototype conflict.

Policy and strategy diagnostics remain separate by channel.  A failed policy
binding cannot be hidden by a successful strategy binding, and the top-level
receipt is ready only when both channels are accepted.

* Layer boundary

This binding is a declarative prototype result, not an execution decision.
=src/core/strategy.ss= and =src/core/policy.ss= remain runner-facing planning
and execution snapshot owners.  Phase 3E does not manufacture those structs
from Feature metadata.

Likewise, Cedar authorization, C ABI calls, Python runtime work, adapter I/O,
and Lean/Evidence emission stay outside this pure Scheme control-plane layer.
Later boundaries may consume the binding receipt explicitly; they may not
reach backward and reinterpret raw Feature declarations.

* Qualification gates

The implementation proves:

- the public contribution and binding surface is POO-native and declarative;
- a real accepted Domain Case assembly supplies both algebra identities;
- parent and child prototype slots survive while child overrides take
  precedence;
- raw contributions, mismatched identities, duplicate ids, non-POO prototypes,
  and missing Domain Case algebra declarations are rejected;
- a rejected Domain Case assembly never reaches prototype composition;
- 256 dependency-ordered Feature contributions bind under the Scheme memory
  guard;
- isolated AOT execution does not depend on the repository source tree.

* Next mainline stage

Phase 3F should bind adapter requirements to an explicit module-owned adapter
capability catalog and relate Feature projection requests to the already closed
Domain Case projection catalog.  That stage must remain a pure POO receipt;
adapter loading, C calls, Cedar evaluation, runtime execution, and evidence
emission remain later handoff boundaries.
