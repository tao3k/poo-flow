#+TITLE: 10.06.25 Funflow CI/CD Check Map Gap Audit

* 10.06.25 Funflow CI/CD Check Map Gap Audit
** 10.06.25.10 Question
This audit checks where the Bass/Funflow research should drive the next code
slice. The goal is not to add adapter execution. The goal is to find the next
control-plane object that can run as Scheme data, project receipts, and hand off
runtime work to Rust/Marlin later.

** 10.06.25.20 Evidence
The Bass audit already made the boundary decision:
- CI/CD is a =(+cicd ...)= payload on the =funflow= workflow module, not a
  top-level category beside =:workflow=.
- Adapter names such as =+github=, =+docker=, and =+nix= are metadata or later
  modules, not replacements for the base CI/CD contract.
- Milestone 2 is config-only Funflow payload.
- Milestone 3 is check-map receipts.
- Milestone 4 is runtime-command manifest handoff.

Current source state:
- =src/modules/funflow/config.ss= already exports
  =poo-flow-funflow-cicd-default-payload= with =checks=, =artifacts=,
  =release=, =webhook=, and =runtime= rows.
- =user-interface/custom/my-module/profiles/cicd.ss= already declares concrete
  sandbox profiles for =ci/check=, =ci/build=, =cd/release=, and
  =cicd/artifact-promote=.
- =src/modules/workflow/flows.ss= already owns the Funflow tutorial alignment
  report and tracks runtime-heavy tutorial gaps as deferred or runtime-owned.
- =t/runtime-manifest-test.ss= proves generic runtime command descriptors can
  project CLI manifests and execute the stdout protocol through the runtime
  adapter boundary.
- =t/funflow-tutorial-alignment-report-test.ss= proves the tutorial alignment
  report exposes CI receipt manifest metadata, result gates, command lists, and
  =runtime-executed = #f=.

** 10.06.25.30 Gap
Milestone 2 is complete enough to stop treating =+cicd= as only a design note.
Milestone 4 has a generic runtime manifest substrate, but it is not yet tied to
CI/CD check objects.

The missing object is Milestone 3:
- a named CI/CD check map;
- per-check sandbox profile reference;
- command intent data;
- artifact/cache/secret declarations;
- typed receipt projection;
- runtime manifest readiness projection;
- no execution in Scheme.

Without this object, the user can declare CI/CD profiles and the system can
produce generic runtime manifests, but there is no stable POO-level bridge from
=funflow (+cicd ...)= to concrete CI check receipts.

** 10.06.25.40 Recommended Shape
Add a workflow-owned CI/CD data object, not an adapter:

#+begin_src scheme
(poo-flow-cicd-check-map
 'default
 (checks
  (check 'build
         profile: 'ci/build
         command: '("gxpkg" "build")
         artifacts: '(build-log)
         cache: '(gerbil-build-cache)
         secrets: '()
         runtime: 'manifest-handoff)
  (check 'test
         profile: 'ci/check
         command: '("gxpkg" "env" "gxtest" "t/unit-tests.ss")
         artifacts: '(test-receipt)
         cache: '()
         secrets: '()
         runtime: 'manifest-handoff)))
#+end_src

This is intentionally a POO object/projection surface, not a shell runner. The
command vector is runtime intent data. The sandbox profile symbol points at the
existing sandbox profile layer. Artifacts, caches, and secrets stay declarative
until the runtime handoff layer consumes them.

** 10.06.25.50 Owner Boundary
Recommended ownership:
- =src/modules/workflow/cicd.ss= or =src/modules/funflow/cicd.ss= should own the
  check-map object and receipt projection.
- =src/modules/funflow/config.ss= should continue to own only default feature
  payload rows.
- =src/modules/sandbox-core/profile.ss= and backend sandbox modules should keep
  profile validation.
- =src/core/runtime-adapter.ss= and runtime manifest helpers should remain
  generic. They should not learn GitHub, GitLab, Nix, or Docker semantics for
  this milestone.
- =marlin-agent-core= remains the heavy runtime owner.

The better first owner is =src/modules/workflow/cicd.ss= if the object is meant
to be engine-independent workflow data. Use =src/modules/funflow/cicd.ss= only
if the first implementation intentionally depends on Funflow-specific graph
projection.

** 10.06.25.60 Acceptance Criteria
The next code slice is complete when:
- a Scheme test constructs a named check map with at least =build= and =test=
  checks;
- the check map projects to an inspectable receipt object/list;
- the receipt includes check name, profile, command vector, artifact outputs,
  cache intents, secret requirements, runtime handoff mode, and

- a runtime-manifest readiness projection exists but does not execute;
- no GitHub/GitLab/Nix/Docker adapter is added in the same slice;

** 10.06.25.65 Implementation Update
The check-map slice now exists in =src/modules/workflow/cicd.ss=:
- checks and check maps remain inert POO control-plane objects;
- receipts and runtime-manifest readiness carry =profile-refs=, =sandbox-runtime-summaries=, =sandbox-handoff-summaries=, and =sandbox-unresolved-profile-refs=;
- sandbox summaries are projected by =src/modules/agent-sandbox/config.ss= from a provided profile catalog, so CI/CD code does not parse filesystem/resource policy rows;
- =src/modules/funflow/config.ss= lowers compact Funflow pipeline rows into the workflow-owned check-map object;
- focused tests cover both unresolved refs and catalog-resolved summaries.

The next implementation contract should connect this readiness data to the broader runtime-command manifest handoff and configured registry path, without adding provider-specific CI adapters.
** 10.06.25.66 Runtime Command Manifest Bridge Update
The workflow-owned bridge now projects CI/CD readiness into the generic runtime
command manifest substrate:
- =poo-flow-cicd-check->runtime-command-manifest= builds one command manifest
  from a check and embeds the readiness row as the runtime request payload.
- =poo-flow-cicd-check-map->runtime-command-manifests= preserves check-map order
  and returns one manifest per check.
- The bridge uses =runtime-command-descriptor->manifest= from
  =src/core/runtime-adapter.ss=, so workflow code does not copy runtime schema
  shape or learn provider-specific adapter behavior.
- Manifest projection remains report-only: =runtime-executed= stays =#f= and no
  CLI process is launched in Scheme.

The remaining follow-up is configured registry discovery: user/module config
should be able to obtain the check-map and project these manifests without
directly calling fixture-specific test helpers.
** 10.06.25.67 Configured Registry Presentation Bridge Update
The configured registry path now exposes the same runtime-command manifest map:
- =poo-flow-user-config-workflow-cicd-runtime-command-manifests= discovers
  check maps from selected modules and uses the configured sandbox profile
  catalog, including project overrides and upstream defaults.
- =pooFlowUserConfigPresentation= adds
  =workflow-cicd-runtime-command-manifest-map-count= and
  =workflow-cicd-runtime-command-manifests= so downstream tools can inspect
  CLI handoff data from the user-facing config surface.
- =pooFlowUserProfilePresentation= and
  =pooFlowUserProfileDoctorPresentation= carry the same fields, keeping the
  profile and doctor surfaces aligned with the base user-config presentation.
- =poo-flow-funflow-pipeline-runtime-command-manifests= gives the Funflow
  module owner a thin, named public projection without duplicating the
  workflow-owned CI/CD manifest implementation.
- Runtime-command manifest summaries expose =status= and =handoff-ready= from
  sandbox profile resolution, so unresolved inherited profile refs are visible
  as blocked handoff rows before any runtime adapter executes.

This closes the fixture-only gap for Milestone 4: a downstream
=use-module funflow :config (pipeline ...)= declaration can now be projected
through module config into report-only runtime command manifests, while Scheme
still performs no runtime execution.

** 10.06.25.80 Reference Research Result
Bass gives us a compact user-facing vocabulary, but not a mature pipeline model to copy:
- A thunk is the stable primitive: command intent plus controlled environment.
- =run= raises on failure, =succeeds?= returns a boolean, and =read= projects stdout through a protocol such as =:json=, =:lines=, =:unix-table=, or =:raw=.
- =cache-dir=, =mask=, and =with-mount= show that cache and secret requirements should stay declarative until runtime.
- Thunk paths are artifact references. They can be serialized as provenance and exported later.
- Bass's own pipeline CI/CD guide is explicitly future work, so copying a Bass pipeline DSL would import unfinished surface area.

Funflow tutorials give the better result ladder:
- =Tutorial1.ipynb= starts with a single =PureTask= projected to a =Flow= and executed by =runFlow=.
- =WordCount.ipynb= names small task flows, composes them with arrow syntax, and runs a pipeline with tuple input.
- =ExternalConfig.ipynb= introduces =DockerTaskConfig= with =image=, =command=, and =args=, then runs through =RunFlowConfig= with =configFile= and =storePath=, returning =CS.Item=.
- =TensorflowDocker.ipynb= chains =putDirFlow=, =dockerFlow= training, =dockerFlow= inference, =getDirFlow= artifact lookup, and final typed output rendering.
- =ExternalConfig/flow.yaml= is only external config data, not a workflow shape; it proves config injection, not pipeline structure.

The smallest concrete example to copy is therefore not "CI pipeline syntax". It is a named workflow check object that carries command intent, sandbox/runtime profile, input bindings, config source, cache intents, secret intents, artifact outputs, and a typed result projection.
** 10.06.25.90 Next Implementation Contract
The next code slice should introduce a workflow-owned check-map object with this narrow shape:
#+begin_src scheme
(check 'build
       profile: 'ci/build
       command: '("gxpkg" "build")
       inputs: '()
       config: '()
       artifacts: '(build-log)
       cache: '(gerbil-build-cache)
       secrets: '()
       result: '(read :lines)
       runtime: 'manifest-handoff)
#+end_src
This maps directly to the reference evidence:
- =profile= is our POO/sandbox equivalent of a controlled environment.
- =command= and =inputs= are Bass thunk intent and Funflow =DockerTaskConfig= intent, not execution.
- =config= captures Funflow =RunFlowConfig= style external configuration.
- =artifacts= map to Funflow =CS.Item= and Bass thunk paths.
- =cache= and =secrets= stay declarative like Bass =cache-dir= and =mask=.
- =result= is the typed receipt projection, analogous to Bass =read= protocols.
- =runtime= remains =manifest-handoff= so Scheme stays a control plane.
