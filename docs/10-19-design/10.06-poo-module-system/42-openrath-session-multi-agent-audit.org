* 10.06.42 OpenRath Session Multi-Agent Audit
** 10.06.42.10 Purpose
This audit continues the OpenRath research with a narrower focus: session as
the flowing value and multi-agent coordination as session graph composition.

The goal is not to copy OpenRath's Python API. The goal is to use OpenRath as
evidence for a stronger POO Flow design:
1. user declarations stay POO-native;
2. functional arrows transform session values;
3. generated runtime receipts use fixed internal structures where appropriate;
4. Marlin or another runtime owns execution through explicit handoff payloads.

** 10.06.42.20 OpenRath Evidence
OpenRath's README states the core model directly:
- =Session= carries conversation state and inter-agent collaboration lineage.
- =Agent= is a reusable session transformation layer.
- =Workflow= composes agents and workflows.
- =Selector= chooses between workflows at runtime while Python keeps ordinary
  =if= / =while= control flow.

The important implementation owners confirm that the README is not only
marketing language:
- =src/rath/session/session.py= defines =Session= with chunk table, id,
  sandbox placement, lineage attributes, token usage, pending lazy work, and
  fork/detach/merge operations.
- =src/rath/session/loop.py= constructs a lazy output session, stamps lineage,
  registers user, agent, and output sessions, binds sandbox ownership, and
  submits the actual loop to the background runtime.
- =src/rath/session/manager.py= keeps a weak session registry plus an active
  session id, avoiding a long-running process retaining every transcript.
- =src/rath/session/graph/recording.py= stamps flat lineage attributes and
  optionally records session ids in a lineage journal.
- =src/rath/flow/agent.py= models an agent as a =Workflow= that injects memory,
  calls =run_session_loop=, and returns a new =Session=.
- =src/rath/flow/workflow.py= makes =forward(session) -> session= the normal
  composition contract and tracks attached =AgentParam= values.
- =src/rath/flow/selector.py= deliberately breaks the normal transform contract:
  it returns a chosen =Workflow= or =EmptyWorkflow=, not a =Session=.
- =src/rath/_async/runtime.py= owns one background asyncio loop and tracks
  in-flight futures weakly.

The example ladder reinforces the same model:
1. hello agent;
2. session lineage;
3. sandbox backend;
4. built-in tools;
5. custom tools;
6. MCP tools;
7. streaming;
8. compression;
9. memory;
10. provider variation;
11. dynamic selector.

** 10.06.42.30 What OpenRath Gets Right
OpenRath's strongest design choice is that the agent loop is not the root
concept. The root concept is =Session=.

That gives the framework a single value that can answer:
1. what chunks exist now;
2. which sandbox placement is active;
3. which parent sessions produced this session;
4. which agent/system session contributed instructions;
5. which token usage and pending work belong to the session;
6. which workflow or selector should consume it next.

This is especially important for multi-agent systems. If every agent owns its
own hidden message history, then branch, merge, compression, and provenance
become post-hoc log reconstruction. If sessions are first-class, then
multi-agent coordination becomes graph composition.

** 10.06.42.35 Project Session Space
The real user shape is broader than one root session. A project can have many
root sessions, and each root session can create many sub-agent sessions. Every
session, including child sessions owned by sub-agents, needs a stable id.

The identity model should be:
1. =project-id=: the workspace or long-lived project boundary;
2. =root-session-id=: one top-level user/task session inside that project;
3. =child-session-id=: one sub-agent session created under a root session or
   another child session;
4. =agent-id=: the logical agent/persona/tool-runner bound to a child session;
5. =conversation-id= or =channel-id=: an optional communication edge between
   sessions.

This makes parent/child and peer communication explicit:
- parent-child relationship is a lineage edge:
  =(parent-session-id . child-session-id)=;
- a sub-agent can address its parent by =parent-session-id=;
- sibling sub-agents can communicate by targeting another =child-session-id=
  through a declared channel or message receipt;
- cross-root communication must include both =project-id= and target
  =root-session-id= so routing never guesses from a bare child id.

POO Flow should not treat child agents as anonymous rows under one loop. Child
agents should be session-addressable participants. Their session id is both a
lineage fact and a communication address.

** 10.06.42.37 Fine-Grained Session Policy
Session ids are also policy anchors. A project may run four or more sub-agents
at the same time; by default they should be isolated from each other even when
they share one root session. Any shared context, shared sandbox, history grant,
or communication channel must be declared as policy.

The policy model should be explicit:
1. =isolation-policy=: whether a child session is isolated from siblings, can
   read parent context, can write back to parent, or can communicate with
   peers;
2. =sandbox-policy=: which sandbox profile or sandbox instance is bound to the
   session id, and whether children inherit, clone, or replace parent sandbox
   placement;
3. =context-policy=: what context is visible to the child: none, parent summary,
   selected parent chunks, selected sibling outputs, or full inherited context;
4. =history-policy=: which historical records can be sent into the child
   session: last turn, last failure, selected receipts, memory recall, or a
   bounded transcript slice;
5. =communication-policy=: which session ids can send messages to which target
   session ids, through which channel, with which payload shape;
6. =resource-policy=: budgets and capabilities keyed by session id, so one
   sub-agent cannot silently consume another sub-agent's resources;
7. =sharing-policy=: explicit grants for shared memory, shared artifacts, shared
   tool results, or shared workspace paths.

The default must be conservative:
- child sessions do not read sibling context unless a channel or context grant
  says so;
- child sessions do not share sandbox handles unless the sandbox policy says
  =shared=;
- child sessions can inherit a parent summary without inheriting full parent
  transcript;
- write-back to parent is a receipt or message, not direct mutation;
- cross-root communication requires =project-id=, =source-root-session-id=,
  and =target-root-session-id=.

This gives the policy system the precision needed for real agent projects:
session id is the address, but policy decides what that address can see, touch,
share, and mutate.

** 10.06.42.37.50 Agent Execution And Tool Permission Plane
OpenRath is useful evidence for session-first runtime structure, but its examples do not define a fine-grained authorization plane for tools, hooks, sub-agents, prompts, or resource sharing. POO Flow should make that plane explicit and composable.
The policy unit is not "the whole process may call tool X." The policy unit is:
#+begin_src text
project-id
  -> root-session-id
  -> child-session-id
  -> agent-id
  -> trigger-id
  -> tool/action/resource grant
#+end_src
This matches real agent projects:
- the main agent may read and summarize project state, but a build verifier sub-agent may run build commands in a constrained sandbox;
- a hook may read staged files or receipts, but it should not inherit the sub-agent's shell/build permissions;
- a reviewer sub-agent may use one model and prompt session, while a governor sub-agent uses another model and a stricter result contract;
- sibling sub-agents may exchange a receipt through a channel without sharing full transcript, sandbox handles, tool result cache, or writable filesystem grants.
The POO layer should expose these as object families, not raw permission maps:
1. =agent-model-policy= for provider/model/capability/budget;
2. =agent-prompt-policy= for prompt session/chunk refs and context mode;
3. =agent-tool-permission-policy= for tool/action/resource grants keyed by agent/session id;
4. =hook-tool-permission-policy= for trigger-specific tool grants and escalation rules;
5. =resource-sharing-policy= for workspace paths, artifacts, tool results, memory refs, and accounting ownership;
6. =agent-execution-policy= for the composed policy refs bound to one agent/session.
These objects should compose through POO/C3 like the rest of the module system. A project can define shared defaults, a root session can narrow them, a role policy can add capabilities, and a child agent policy can override or deny selected slots. Validation happens after composition so the effective policy is inspectable before runtime.
The target invariant is stricter than coarse tool permission:
- tool grants include =tool-ref=, =action=, =resource-ref=, and =trigger-ref=;
- every grant is anchored to =project-id=, =root-session-id=, =child-session-id=, or =agent-id= through the policy scope;
- hook policy is independent unless a visible policy object explicitly composes it with an agent policy;
- resource sharing is denied unless the grant names both scope and accounting owner;
- context and history sharing are separate from tool permission, so "can read files" never implies "can read sibling transcript."
** 10.06.42.38 POO Flow Strengthening Thesis
OpenRath's weak points should become POO Flow's strongest layer. OpenRath shows
that session-first is the right runtime idea; POO Flow should make the missing
policy plane explicit, typed, composable, and inspectable before runtime.

The strengthening direction is:
1. =project-session-space= is the top-level object. It owns root sessions,
   child sessions, address allocation, communication topology, and policy
   summaries.
2. =session-id= is a capability boundary, not just a label. Every sandbox,
   context, history, memory, artifact, and communication grant is keyed through
   session identity.
3. POO/C3 composes policy defaults. Parent project policy, root-session policy,
   role policy, and child-session policy can inherit and override each other.
4. Safety is decided after composition. C3 chooses value precedence; validators
   decide whether the composed policy is safe, monotonic, and allowed.
5. Runtime work is never implied by policy. Policies lower into handoff
   receipts and Marlin-owned execution contracts.
6. Generated policy decisions can use fixed Gerbil =defstruct= receipts
   internally, with bounded alist projection at manifest, JSON, test-summary,
   and Marlin handoff boundaries.

This makes POO Flow stricter than OpenRath in the exact area large agent
systems need most: a sub-agent is not just a function call or a hidden message
history. It is a session-addressed participant with an explicit isolation,
sandbox, context, history, communication, and resource policy.

The target invariant is deny-by-default:
- no sibling context unless granted;
- no shared sandbox unless granted;
- no parent write-back except through a receipt or message;
- no history replay except through bounded grants;
- no cross-root communication without explicit project/root/session ids;
- no resource sharing unless the policy says how accounting and ownership work.

** 10.06.42.40 What POO Flow Should Not Copy
OpenRath is useful, but several parts should remain runtime-side evidence
rather than Scheme-side architecture:
1. =session.to(...)= mutates placement on the session object. POO Flow should
   keep placement as an explicit POO prototype/mixin or generated receipt so a
   dry-run presentation can validate it before runtime.
2. The process-wide async runtime is convenient in Python. POO Flow should not
   create a Scheme execution loop for agent turns; it should emit runtime
   handoff facts for Marlin.
3. =AgentParam= is compact but weakly constrained. POO Flow should model the
   same concept as typed POO objects plus structured validation: agent system
   session, provider ref, memory refs, tool refs, and sandbox requirements.
4. =Selector= breaks the normal =Workflow.forward(session) -> session= contract.
   POO Flow should represent selector output as a selection receipt with a
   contract, candidate set, fallback behavior, and runtime owner.
5. Lazy materialization is a runtime behavior. Scheme should describe
   =materialization-state= and =pending-runtime-ref= as receipts, not block on
   provider execution.

** 10.06.42.50 Current POO Flow Status
POO Flow already has the foundation, but not the full multi-agent session graph.

Delivered:
- =src/modules/session/objects.ss= models session values, chunks, lineage,
  placement, handoff rows, graph edges, and graph presentation. Chunks are
  cheap rows; lineage and placement are POO mixins composed into the final
  session value.
- =src/modules/session/transform.ss= models report-only session transforms and
  memory intents. A transform derives a session receipt and emits handoff
  intent without provider, tool, memory, selector, or sandbox execution.
- =src/modules/session-core/config.ss= and
  =src/module-system/session-core-config.ss= expose the upstream session-core
  module row and feature flags.
- =src/module-system/loop-engine-runtime-agent.ss= projects agent profiles,
  harnesses, sessions, workflow runs, dispatch receipts, and operations for
  loop-engine handoff.

Gap:
The session layer and loop-engine multi-agent rows are still adjacent rather
than unified. Loop-engine can name agent sessions, but the session module does
not yet own a reusable multi-agent session topology that other workflows,
selectors, or future runtime owners can consume.

** 10.06.42.60 Design Direction
POO Flow should treat multi-agent orchestration as a session graph problem:

#+begin_src text
ProjectSessionSpace
  -> RootSessionValue
  -> ChildAgentSessionValue
  -> SessionTransform
  -> AgentSessionNode / CommunicationEdge
  -> MultiAgentSessionGraph
  -> RuntimeHandoff
  -> RuntimeReceipt
  -> DerivedSessionValue
#+end_src

The public authoring path should stay compact:
#+begin_src scheme
(use-module session-core
  :config
  (.def (ci-session @ session/value ...)
    ...)
  (.def (audit-agent @ session/agent ...)
    ...)
  (.def (ci-agent-graph @ session/agent-graph ...)
    ...))
#+end_src

The exact syntax can evolve, but the invariants should not:
1. agent prompt/instruction is a session chunk, not pasted prompt text;
2. agent identity is a POO object with provider/tool/memory/sandbox refs;
3. multi-agent collaboration is graph data with parent/edge semantics;
4. every root and child session has an addressable id;
5. child sessions record both parent lineage and communication channels;
6. isolation, sandbox, context, history, communication, sharing, and resource
   policy are first-class objects;
7. selector output is a receipt, not Scheme-side dynamic dispatch;
8. runtime execution is a handoff boundary.

  ref, parent session refs, =materialization-state=, pending runtime ref,
  sandbox handle ref, token usage summary, error summary, runtime owner, and
  =runtime-executed: #f=.
- Supported states are =pending=, =materialized=, =failed=, and =dropped=.
  This maps OpenRath's =_pending= future, =synchronize= result, and dropped
  error warning path into explicit receipt data without blocking in Scheme.
- =user-interface/custom/my-module/cases/session-materialization.ss= exposes
  pending and failed materialization receipts as downstream custom config data.
- =t/session-materialization-receipt-test.ss= covers pending, materialized, and
  failed receipts.
- =t/scenarios/performance/session-materialization-receipt/benchmark.ss= and
  =t/session-materialization-receipt-performance-test.ss= make defstruct
  materialization projection a hot benchmark gate over 300 receipts.

** 10.06.42.90 Conclusion
The key lesson from OpenRath is not "make a Python-like API." It is:

#+begin_quote
Multi-agent systems become understandable when Session is the flowing value and
agents are transformations over session graphs.
#+end_quote

POO Flow can make that stronger than OpenRath by moving the declaration,
validation, graph topology, and handoff contracts into Gerbil POO while leaving
execution, lazy materialization, tool calls, and memory mutation to Marlin or
other runtime owners.
