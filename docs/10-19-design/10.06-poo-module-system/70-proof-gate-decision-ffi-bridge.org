#+title: Proof Gate Decision and FFI Bridge
#+startup: overview

* Problem Boundary

POO Flow does not execute LangGraph, LangChain, durable checkpoint runtime, tool
runtime, or sandbox runtime in Scheme. Those runtime systems belong to Marlin or
another runtime owner. The Scheme control plane is responsible for producing a
precise strategy/policy artifact that a runtime can consume without guessing.

This document records the concrete bridge added after the proof-gate bundle:
receipts are normalized into proof facts, proof facts are wrapped into FFI-ready
wires, wires are grouped into a gate bundle, and the bundle is collapsed into a
decision payload for runtime handoff.

The important point is that the final payload is not a JSON proof and not Lean
source generated by Scheme. It is a typed Scheme-side wire contract whose fields
are aligned with Lean facts. Lean proves the invariant shape; Scheme emits the
runtime-facing evidence.

* Artifact Pipeline

#+begin_src mermaid
flowchart LR
  A["Composition / scenario / handoff receipts"] --> B["Proof facts"]
  B --> C["FFI wire facts"]
  C --> D["Proof gate bundle"]
  D --> E["Decision"]
  E --> F["FFI bridge payload"]
  F --> G["Runtime owner boundary"]

  L["Lean fact modules"] -. align .-> B
  L -. prove .-> D
  L -. check .-> E
#+end_src

The runtime handoff receives only the final bridge payload. Earlier artifacts
remain useful because they make failures auditable:

- Receipt: the user-interface or scenario-level declaration.
- Proof fact: normalized fields with stable schema identity.
- FFI wire: schema/version/id/accepted/rejection-rule fields for foreign owners.
- Bundle: one composition decision plus scenario-gap and handoff evidence.
- Decision: accepted or rejected, with explicit reasons.
- FFI bridge payload: the runtime-facing contract.

* Decision Semantics

#+begin_src latex
\[
  \mathrm{accept}(b) \iff
    \mathrm{valid}(b) \land
    \mathrm{accepted}_{composition}(b) \land
    \mathrm{accepted}_{scenario}(b) \land
    \mathrm{accepted}_{handoff}(b) \land
    \mathrm{runtimeBoundaryOk}(b)
\]

\[
  \mathrm{reasons}(b) =
    R_{bundle}(b) \cup
    R_{composition}(b) \cup
    R_{scenario}(b) \cup
    R_{handoff}(b) \cup
    R_{runtime}(b)
\]
#+end_src

This is intentionally small but strict. It lets the runtime owner reject a
payload without reconstructing POO Flow internals. It also gives Lean a stable
surface: each rejection reason corresponds to a missing or invalid invariant,
not to a textual diagnostic.

* Runtime Boundary

The bridge keeps runtime ownership explicit:

- Scheme may say that a durable/checkpoint/handoff strategy is required.
- Scheme may say which capability, profile, scope, or artifact is part of the
  contract.
- Scheme must not pretend that runtime execution already happened.
- Scheme must not encode runtime implementation details as proof facts.
- Runtime owners consume the payload and execute with their own language/runtime.

This prevents the old drift where a UI scenario tried to imitate a runtime
executor. For LangGraph-like scenarios, POO Flow proves that its Policy, Durable,
Checkpoint, Graph, Loop, and Composition contracts are sufficient to express the
scenario. It does not embed LangGraph as a Scheme runtime module.

* Lean Alignment

Lean is the mathematical owner of the invariant vocabulary. Scheme owns the
control-plane artifact. The bridge is correct only when the same facts appear on
both sides:

#+begin_src latex
\[
  \forall f \in \mathrm{facts}(receipt),
  \mathrm{schema}(f) \in \Sigma
  \land
  \mathrm{requiredFields}(\mathrm{schema}(f)) \subseteq \mathrm{fields}(f)
\]

\[
  \mathrm{bundleAccepted}(b) \Rightarrow
  \mathrm{compositionConsistent}(b)
  \land
  \mathrm{scenarioGapClosed}(b)
  \land
  \mathrm{handoffContractClosed}(b)
\]
#+end_src

Generated Lean modules are still hand-aligned in this stage. The next stronger
step is receipt-to-Lean fact generation, but that generator must emit Lean facts,
not JSON manifests.

* Implementation Surface

The concrete Scheme API added by this step is:

- =poo-flow-proof-gate-bundle->decision=
- =poo-flow-proof-gate-receipts->decision=
- =poo-flow-proof-gate-decision-accepted?=
- =poo-flow-proof-gate-decision-rejection-reasons=
- =poo-flow-langgraph-user-interface-proof-gate-decision=
- =poo-flow-proof-gate-decision->ffi-payload=
- =poo-flow-proof-gate-receipts->ffi-payload=
- =poo-flow-langgraph-user-interface-proof-ffi-payload=

The bridge payload deliberately preserves the bundle and the decision. This is
more verbose than a boolean gate, but it is the minimum useful audit artifact for
agent/human review: a runtime owner can see why a payload was accepted and which
part of the policy graph caused rejection.
