* 10.06.57 User Graph P0 Thesis
This document defines the P0 product thesis for =poo-flow= after the BAML comparison.

The core correction is simple:
#+begin_src text
Users can describe an agentic workflow graph in natural language.
Agents understand that graph through a guide system and generate Scheme/POO.
Validators give immediate feedback after each agent edit.
#+end_src
** 10.06.57.10 P0 Problem
The current project has strong mechanisms: workflow, sandbox, session, loop, governor, human audit, runtime manifests, type facts, graph algorithms, and Lean-facing proof facts.

The weakness is not theoretical capability. The weakness is that the user-facing thesis and agent feedback loop are not yet single enough.

BAML is easier to understand because its product thesis is short:
#+begin_src text
Write LLM functions and get typed outputs.
#+end_src

=poo-flow= needs an equally clear thesis:
#+begin_src text
Describe an agentic workflow graph in natural language and get Scheme/POO
policy code, immediate validation feedback, rejection reasons, and an
explainable runtime handoff.
#+end_src

This does not mean users cannot write Scheme. Scheme/POO is the programmable authoring surface. The correction is that ordinary users should not need Lean, type facts, or manifest internals as their first concepts.
** 10.06.57.20 User Graph Vocabulary
The default ordinary user vocabulary is:
- workflow graph;
- loop;
- step;
- model;
- module;
- gate;
- human approval;
- runtime handoff;
- result path.

The programmable user and module-author vocabulary may include:
- Scheme;
- POO object;
- profile;
- module;
- validator;
- projection;
- type fact;
- Lean fact contract.

The default ordinary user vocabulary is not:
- schema;
- Lean;
- type facts;
- contract receipts;
- ABI rows;
- manifest internals;
- C3 linearization internals.

Those concepts are still important. They belong to agent repair, validation, proof, and runtime integration surfaces, and they remain available to advanced users who deliberately work below the graph layer.

** 10.06.57.35 Guide System
The guide system is the bridge between natural-language graph intent and Scheme/POO code.

It should teach agents how to map:
- user-described steps into workflow step declarations;
- loops into graph patterns with exit conditions;
- model mentions into model bindings;
- module mentions into capability providers;
- gates into validation and human-audit points;
- failure paths into retry, escalation, or stop edges;
- graph diagnostics back into user-readable repair feedback.

This is where =poo-flow= differs from a DSL-first product. BAML asks users to write DSL functions. =poo-flow= should let users describe an agentic graph, then let agents generate and repair the Scheme/POO policy code.
** 10.06.57.40 P0 Feedback Loop
The P0 feedback loop is an agent editing loop, not only a user-facing graph display.

BAML's editor/playground matters because it gives immediate feedback after each authoring change: prompt, type, output, and error are visible without waiting for a full runtime integration. =poo-flow= needs the same strength for agent-authored configuration. After an agent changes modules, graph declarations, Scheme/POO profiles, validators, or runtime handoff settings, it must immediately know:
- did the graph become valid?
- what changed in the graph?
- where was the change rejected?
- which validator rejected it?
- why was it rejected in user-facing graph terms?
- what declaration should the agent repair next?

This loop is critical for agents because agents do not have stable human intuition about the whole composed policy graph. Without fast feedback, an agent can keep editing Scheme/POO in the wrong layer, accidentally widen permissions, remove gates, break loop exits, or generate manifests that no longer correspond to the user-visible graph.

Priority order:
1. Authoring delta: show what the agent changed in graph intent or Scheme/POO declarations.
2. Immediate graph preview: show the affected loops, steps, models, modules, gates, and runtime handoff.
3. Immediate validation result: show pass/fail, validator owner, rejected node or edge, and reason.
4. Agent repair hint: explain which declaration or module choice should change next.
5. Advanced drill-down: Scheme/POO diff, manifest preview, type facts, and Lean facts when needed.

This is the local equivalent of BAML's playground. BAML previews prompt, type, output, and error. =poo-flow= should preview graph, validation result, rejection reason, and repair target after every agent edit.
** 10.06.57.45 Productized Agent Feedback Loop
The productized feedback loop should feel like an agent repair playground, not a manifest inspector.

The loop has six visible stages:
1. Intent capture. The user describes a desired workflow in natural language or edits a graph/module intent file.
2. Agent generation. The agent lowers the graph intent into Scheme/POO declarations.
3. Immediate preview. The system shows the graph produced by the agent's current declarations.
4. Immediate diagnostics. Invalid graph areas are marked at the node or edge level: missing model, missing module capability, unsafe gate, missing loop exit, blocked sandbox/resource policy, or incomplete runtime handoff.
5. Agent repair. The system explains what the agent should change next in Scheme/POO declarations. The repair plan should use user vocabulary first and only then offer a Scheme/POO diff for advanced review.
6. Readiness confirmation. Validators confirm the graph is valid and ready for manifest projection or return graph-positioned errors. Type facts, Lean facts, and manifest rows are available as drill-down artifacts.
** 10.06.57.50 Observability As The Feedback Backbone
Diagnostic records should not be treated as a standalone feature. They are one projection of the =poo-flow= observability framework.

The design rule is:
#+begin_src text
Observability owns the middle evidence.
Diagnostics explain rejected evidence.
Receipts package evidence for agents.
Projections hand validated evidence to runtime, proof, or reports.
#+end_src

This matters because the agent feedback loop needs more than string errors. After each edit, the agent must receive a structured observation of what changed, which semantic graph facts were rebuilt, which validators fired, which node or edge was rejected, and which declaration layer is legal to repair next.

The observability framework should be implemented on the Scheme side as POO-native object families:
- =observability/event=: the atomic fact that something was built, changed, validated, rejected, projected, or handed off;
- =observability/span=: a bounded operation such as graph rebuild, C3 composition, contract validation, manifest projection, or doctor preview;
- =observability/diagnostic=: a validator-owned explanation of invalid or risky evidence;
- =observability/receipt=: the agent-facing packet that summarizes delta, diagnostics, repair plan, readiness, and drill-down artifacts;
- =observability/projection=: a stable view emitted to graph preview, doctor reports, Lean facts, or runtime manifests.

These are not raw alists. They should be POO objects with typed slots, source provenance, graph coordinates, validator ownership, and projection contracts.
** 10.06.57.55 POO Extension Shape
Each module should be able to extend observability without editing a central diagnostic switch.

The normal extension route is:
#+begin_src scheme
(defobservability-class sandbox-resource-diagnostic
  (:inherits (observability/diagnostic graph-positioned-diagnostic))
  (:slots
   (validator-id symbol?)
   (node graph-node-ref?)
   (edge maybe-graph-edge-ref?)
   (reason symbol?)
   (safety-class symbol?)
   (repair-target repair-target?)))
#+end_src

The exact macro names can change during implementation, but the shape is important:
- macro syntax declares repeated object families and typed slots;
- POO inheritance captures common diagnostic behavior;
- module-specific diagnostics extend the shared base object;
- validators emit objects, not text-only failures;
- graph explanation and agent repair receipts consume the shared object protocol.

This is where POO is stronger than a flat DSL. A sandbox diagnostic can inherit from generic diagnostic behavior, graph-positioned behavior, safety-policy behavior, and agent-repair behavior at the same time. C3 gives deterministic method order, while validators still enforce safety legality.
** 10.06.57.60 Agent Observability Flow
#+begin_src mermaid
flowchart LR
  Edit["Agent edit\nScheme/POO or graph intent"]
  Span["Observability span\ncapture edit + source provenance"]
  Build["Semantic graph rebuild\nPOO + C3 + functional builders"]
  Events["Observability events\nnodes, edges, policies, projections"]
  Validate["Validators\nslot contracts, hard gates, graph rules"]
  Diag["Diagnostics\nvalidator-owned rejected evidence"]
  Receipt["Agent receipt\ndelta, diagnostics, repair, readiness"]
  Repair["Agent repair\nhighest legal declaration layer"]
  Projection["Validated projections\nmanifest, Lean facts, reports"]
  Edit --> Span --> Build --> Events --> Validate
  Validate -->|"invalid"| Diag --> Receipt --> Repair --> Edit
  Validate -->|"valid"| Receipt --> Projection
  Events --> Receipt
#+end_src

The receipt is the agent's default interface. It should answer:
1. what did my edit change?
2. what graph nodes or edges changed?
3. which observability span produced the evidence?
4. which validators accepted or rejected it?
5. where is the rejected evidence on the graph?
6. what is the highest legal layer I should repair next?

The receipt should be derived from observability objects, not assembled by scraping terminal output. This lets the same evidence feed doctor output, graph explanation, scenario tests, proof facts, and runtime previews.
** 10.06.57.65 Receipt Contract
The agent-facing receipt should stay small and stable:
#+begin_src scheme
(feedback-receipt
  id: feedback-id
  source: source-provenance
  span: observability-span-ref
  delta: graph-delta
  diagnostics: diagnostic-list
  repair: repair-plan
  readiness: readiness-state
  artifacts: drill-down-artifacts)
#+end_src

Diagnostics inside the receipt should be graph-positioned:
#+begin_src scheme
(diagnostic
  severity: 'error
  node: 'release
  edge: '(test -> release)
  validator: 'sandbox-resource-validator
  reason: 'permission-widening-denied
  safety-class: 'hard-gate
  message: "release cannot widen filesystem write scope"
  repair-target: '(user-interface/init.ss :workflow +cicd release))
#+end_src

The repair target must point to the highest legal authoring layer. For ordinary agent repair that means graph intent, =init.ss= module selection, feature configuration, or Scheme/POO declarations. It should not tell the agent to patch manifest rows, Lean facts, or runtime payloads unless the request is explicitly a low-level drill-down task.
** 10.06.57.70 Implementation Priority
The first implementation should not try to build a full tracing platform. It should establish the object protocol and one visible loop:
1. define shared observability object families and typed slots;
2. convert validator failures into graph-positioned diagnostics;
3. build a feedback receipt from graph delta, diagnostics, repair, and readiness;
4. expose a doctor or scenario command that prints the receipt in agent-readable form;
5. let sandbox and workflow modules extend diagnostics through POO inheritance instead of a central case table.

Acceptance criteria:
- no validation failure is string-only;
- every diagnostic has source provenance, graph coordinates, validator owner, reason, and repair target;
- every agent receipt states whether the graph is valid and what to repair next;
- module-specific observability objects can be added without changing the core receipt consumer;
- valid receipts can project to manifest/Lean/report artifacts without making those artifacts semantic owners.

The feedback contract:
#+begin_src text
Natural-language intent -> agent-generated Scheme/POO -> graph preview
-> validation result -> rejection reason -> agent repair.
#+end_src

The first feedback response after an agent edit should answer:
- What did the agent change?
- What did that change do to the graph?
- Is the graph valid now?
- If invalid, which node or edge was rejected?
- Why was it rejected?
- What should the agent change next?

The drill-down screens can answer:
- What Scheme/POO was generated?
- Which validator rejected the graph?
- Which manifest row will be emitted?
- Which type or Lean fact supports this decision?

*** 10.06.57.70.20 Next Slice: Validator Integration
The next implementation slice should connect existing validators to the shared receipt builder:
1. sandbox resource validation: permission widening, filesystem resource shape, and runtime-executed = #f;
2. workflow graph validation: missing step/model/module/gate and invalid edge shape;
3. loop/governor validation: missing exit, budget widening, and human gate removal;
4. doctor/scenario projection: print the same receipt shape rather than terminal-only strings.

Each integration should prove:
- diagnostics are observability objects before projection;
- =graph= names the affected graph/module concept;
- =repair= points to the highest legal repair layer;
- =readiness= is =ready= only when diagnostics are empty;
- manifest/type/Lean artifacts remain drill-down data.
** 10.06.57.75 Feedback Object Model
Every product feedback response should have four top-level objects:
- =graph=: the current agentic workflow graph, including loops, steps, models, modules, gates, and handoff nodes;
- =diagnostics=: graph-positioned findings with severity, node or edge reference, user-facing message, validator owner, and agent repair hint;
- =repair=: proposed Scheme/POO declaration changes, grouped by module or user-interface file;
- =readiness=: whether the graph is ready for validation, proof drill-down, manifest projection, and runtime handoff.

Advanced artifacts may hang under those objects:
- generated Scheme/POO;
- validation receipts;
- manifest preview;
- type facts;
- Lean fact contracts;
- runtime handoff rows.

The important rule is that advanced artifacts attach to graph concepts and agent repair decisions. They should not become the root of the feedback model.
** 10.06.57.80 Strength Positioning
Do not describe =poo-flow= as weaker than BAML in type capability.

BAML has a strong productized type path for LLM function output. =poo-flow= has a stronger theoretical substrate for whole-system agent policy:
- Scheme is a real host language.
- POO and C3 can compose multiple policy owners.
- Structured validation can check object graphs.
- Graph algorithms can explain system behavior.
- Lean-facing facts can support proof obligations.

The missing work is to productize these strengths behind the graph surface and agent feedback loop.
** 10.06.57.85 Acceptance Criteria
New user-facing docs, examples, diagnostics, and agent receipts should pass these checks:
- Can a user understand the workflow from the graph without reading manifest rows?
- Can an agent immediately see what its edit changed?
- Can an agent map graph errors back to declarations?
- Are Schema, Lean, and type facts absent from the required first user view while still available for advanced users and agents?
- Does the graph show loop, step, model, module, gate, and runtime handoff?
- Does the feedback loop explain invalid graph points before showing low-level artifacts?
- Can natural-language graph intent be lowered by an agent into Scheme/POO authoring code?
- Does every failed validation produce a graph-positioned rejection reason and next repair target?
