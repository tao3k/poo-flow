* 10.06.17 Module Extension Fixed Point
** 10.06.17.10 Decision
POO is the core extension model for =poo-flow=. Workflow, CI/CD, sandbox, and user modules should contribute POO object operations into the module system instead of each feature owning a private merge engine.

The module system owns fixed-point convergence over object graphs:
- scalar slots use override;
- list slots use prepend, append, and remove-elements;
- child nodes use extend and remove by stable object identity;
- repeated application is idempotent and converges to a stable graph.

** 10.06.17.20 Object Graph
The generic graph shape is:
#+begin_src scheme
(poo-flow-module-extension-node
 'workflow/pipeline/default
 '((needs . (test)))
 (list build-node legacy-node))
#+end_src

Feature modules should translate their domain syntax into extension contributions:
#+begin_src scheme
(poo-flow-module-extension-contribution
 'workflow/pipeline/default
 (list
  (poo-flow-module-extension-slot-append 'needs '(lint))
  (poo-flow-module-extension-node-remove
   'workflow/pipeline/default/task/legacy)))
#+end_src

** 10.06.17.30 Workflow Boundary
=workflow-config= should become an adapter:
- parse =(pipeline ...)= and =(task ...)=;
- emit module extension nodes and operations;
- avoid workflow-specific merge/remove logic;
- read the stable graph after the module-system fixed point has converged.

That keeps CI/CD and workflow declarative while preserving POO as the single extension engine.

** 10.06.17.40 Current Stage
The first code stage introduces =:modules/extension= with:
- generic extension node values;
- slot override, append, prepend, and remove operations;
- child node extend and remove operations;
- fixed-point result receipts with root, iteration count, and stable flag.

The next stage should remove workflow-private merge code and emit these generic operations from workflow syntax.

** 10.06.17.50 POO Shared Objects
NixOS is useful as an interface reference because many modules can contribute
to one final =config= graph. Its =options= layer is also partly a workaround
for a DSL without native Scheme objects, contracts, and POO inheritance.

=poo-flow= should keep the good part and avoid copying the hack:
- modules export shared POO module objects, not ad-hoc option alists;
- module objects carry identity, inherited objects, field contracts, and
  metadata;
- field contracts are object-internal slot contracts: slot identity, value
  kind, merge strategy, default value, and metadata;
- config contributions are small POO objects that target a module object and
  carry a field contract plus a value;
- =poo-flow-module-config-mk-merge= normalizes those contributions into the
  generic extension operations and then runs the module-system fixed point.

The user-facing shape can still feel like a NixOS/Doom-style module system:
many files declare small pieces of configuration, and the system produces one
resolved object graph. The implementation stays Scheme-native: typed contracts,
POO objects, reusable merge semantics, and extension by stable object identity.

Example:
#+begin_src scheme
(def shared-sandbox
  (poo-flow-module-object
   'sandbox/shared
   '()
   (list
    (poo-flow-module-field-contract
     'backend 'Symbol 'override 'sandbox '())
    (poo-flow-module-field-contract
     'runtime-args 'List 'append '() '()))
   '((domain . sandbox))))

(def nono-sandbox
  (poo-flow-module-object
   'sandbox/nono
   (list shared-sandbox)
   (list
    (poo-flow-module-field-contract
     'backend 'Symbol 'override 'nono '()))
   '((backend . nono))))

(poo-flow-module-config-mk-merge
 root-node
 (poo-flow-module-object-contributions
  nono-sandbox
  '((runtime-args . ("--trace-nono")))))
#+end_src

That is the intended reading of "options become objects": the shared namespace
contains reusable module objects, and each object owns the contracts for its
slots. Nono, CubeSandbox, workflow pipelines, cacheable tasks, and similar
domains should extend objects instead of duplicating option maps.

** 10.06.17.60 Objects File Layout
Module files use a Doom/NixOS-like shape, but the meaning is POO-native:
- =config.ss= declares module activation and user-facing module rows;
- =objects.ss= declares reusable POO module objects for that namespace;
- top-level =src/modules/objects.ss= owns =objects.shared.*=;
- module-local object files own =objects.<module>.*=;
- all object files contribute to the final =objects= attribute graph.

Concrete sandbox shape:
#+begin_src text
src/modules/objects.ss
  objects.shared.sandbox

src/modules/nono-sandbox/objects.ss
  objects.nono-sandbox.sandbox inherits objects.shared.sandbox

src/modules/cubeSandbox/objects.ss
  objects.cubeSandbox.sandbox inherits objects.shared.sandbox
#+end_src

Directory placement is only authoring organization. The merge key is the object
identity under the =objects= attribute. Any module can contribute an object, and
any object can inherit another object as long as its stable identity is known.

** 10.06.17.70 Loader Tree Projection
=modules/loader.ss= is the bridge from file-tree authoring to module-system data.
It supports two related layouts.

Upstream module roots use the compact module layout:
- =<module-root>/config.ss= for activation and module rows;
- =<module-root>/objects.ss= for POO object prototypes and object contributions.

User roots use the profile layout:
- =init.ss= for the Doom-style enabled-module declaration;
- =objects.ss= for user-owned shared POO objects;
- =config.ss= for the tool-facing user facade;
- =modules/config.ss= for user-facing configuration macros and helpers.

User-facing object contracts live in =objects.ss=. That file is intentionally
narrow: only POO module objects, field contracts, object inheritance, and object
extension belong there. Sandbox profile recipes and other non-object config
belong in =config.ss= or another role-specific entrypoint, not =objects.ss=.
A separate =modules/types.ss= layer would duplicate the same responsibility and
should not be part of the user-root contract.

The loader attaches role policies to each user-root entrypoint. =init.ss= is
=init-switches-only=: it may select a profile and toggle modules/features, but
it may not define objects, sandbox profile recipes, settings, package sync,
descriptor realization, or runtime execution. Those responsibilities belong to
=objects.ss=, =config.ss=, or upstream implementation owners.

The loader still does not evaluate files or activate modules. It emits source
refs and lazy load plans, so core code and user projects can share tree
contracts while preserving the existing resolver boundary.

** 10.06.17.80 Auto Import Graph
Directory entrypoints can be projected into an =auto-imports= POO graph:
- each =config.ss=, =objects.ss=, or helper entrypoint is a child node;
- the child identity is the stable source-ref value;
- node slots keep the source ref, entrypoint role, tree kind, and enabled flag;
- =module-system.ss= can stay small by reexporting loader/extension APIs instead
  of naming every concrete module object file.

Disabling an automatically discovered entrypoint is also POO-native:
#+begin_src scheme
(poo-flow-module-extension-contribution
 'auto-imports
 (list
  (poo-flow-module-extension-node-remove
   "user-interface/custom/my-module/config.ss")))
#+end_src

This keeps discovery, enablement, and removal in the same extension model.
Gerbil compile-time imports remain explicit at build boundaries, but module
selection and user-facing auto imports are data driven through loader source
refs and POO graph edits.
