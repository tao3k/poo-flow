* 10.06.53 LoopX Pressure Test And Control-Plane Closure
** 10.06.53.10 Purpose
This document records the reflection from studying the downstream LoopX
project under =.data/loopx=. LoopX is useful pressure evidence because it is a
real community-style product that hand-builds a local control plane for long
running agents: goals, todos, gates, quota, monitors, run history, projections,
and public/private boundaries.

The local conclusion is direct: =poo-flow= should not copy LoopX as an
application. =poo-flow= should absorb the invariant underneath it and expose a
better control-plane layer:

#+begin_src text
POO-native policy families
  + functional strategy combinators
  + validated receipt/projection surfaces
  + high-performance runtime handoff ABI
  -> Marlin, Rust, or another runtime executes heavy work
#+end_src

If a downstream project must re-invent a LoopX-shaped control plane with large
CLI condition trees, ad hoc state patches, or source-specific display parsers,
then the gap is in our user interface, strategy engine, or runtime handoff
contract.

** 10.06.53.20 Pressure Evidence From LoopX
LoopX organizes its product around seven layers:
- registry;
- goal state;
- adapter pre-tick;
- run log;
- run history;
- status / attention queue;
- compute quota.

It also names the product actors: goal, executor, user, and dashboard. Its
agent-facing commands and docs expose concrete concepts such as:
- =todo claim=, =todo update=, =continuous_monitor=, =user_gate=, and
  =claimed_by=;
- =quota should-run=, =scheduler-ack=, =monitor-poll=, =spend-slot=, and
  =void-slot=;
- =interaction_contract=, =protocol_action_packet=, =operator_gate=,
  =work_lane_contract=, and =handoff_packet=;
- append-only run history and compact public-safe projection surfaces;
- projection sink rules that prevent dashboards, chat summaries, or reports
  from parsing private project files directly.

These are not strange requirements. They are normal long-running agent
requirements. The lesson is that users need a control plane, but they should
not need to invent one per product.

** 10.06.53.30 Dimensional Advantage
=poo-flow= is lower-level than LoopX because it owns the language and
composition model:
- task families are POO descriptors, not hard-coded runner branches;
- registries are immutable POO policy bundles;
- strategies choose planning, frontier, cache, and failure policy;
- runners validate plans and emit receipt trees;
- runtime adapters normalize request/response boundaries;
- durable operation receipts project append, checkpoint, lease, repair,
  artifact, communication, and sandbox intents into Marlin handoff data.

=poo-flow= is also higher-level than LoopX because it can make those concepts
domain-neutral. A downstream "monitor todo" is just a policy family plus a
strategy rule. A downstream "quota should-run" decision is just a composed
strategy result with gates, capability checks, outcome floors, and fallback
policy. A downstream "frontstage row" is just a final projection of validated
receipts and policy state.

The dimensional advantage only matters if the public interface is good enough.
The user should author POO objects, profile fragments, and functional
combinators. They should not author raw alists, patch languages, giant command
switches, or one-off projection parsers.

** 10.06.53.40 Runtime Boundary Decision
This repository should build the best Scheme-side interface to Marlin, Rust,
or another runtime. It should not own the runtime implementation.

Scheme owns:
- POO object graph construction;
- policy family declaration and validation;
- functional flow and strategy composition;
- graph/frontier explanation;
- receipt and projection schemas;
- runtime request envelopes;
- capability negotiation metadata;
- handoff manifests;
- replay, lease, durability, and compensation intent.

Runtime owners such as Marlin or Rust own:
- shared thread pools and async schedulers;
- process, filesystem, network, and connector IO;
- durable append, fsync, checkpoints, derived indexes, and repair jobs;
- lease conflict resolution and compare-and-swap writes;
- sandbox operations and artifact retention;
- telemetry ingestion and high-volume persistence.

The interface must be optimized for that split. Scheme must not pass closures,
large raw payloads, or private transcript blobs across the runtime boundary.
It should pass stable ids, refs, operation rows, capability flags, watermarks,
policy snapshots, and request ids. Rust can then optimize the hot path without
reverse-engineering Scheme internals.

** 10.06.53.50 Current Strengths
The current codebase already has the correct direction:
- =src/core/task.ss= makes task family descriptors POO objects and keeps
  adapter-routed task requests symbolic.
- =src/core/strategy.ss= keeps planning and frontier selection in strategy
  policy rather than runtime execution.
- =src/core/runner.ss= validates a plan before execution and records policy,
  frontier, adapter decision, request id, status, and child receipts.
- =src/core/runtime-adapter.ss= defines versioned request, response, and
  runtime command descriptor schemas.
- =src/core/receipt.ss= turns receipt trees into compact audit events without
  storing raw input/output values in the audit stream.
- =src/module-system/durable-runtime-store-operation.ss= defines durable
  operation receipts and Marlin handoff manifests.
- =src/module-system/durable-runtime-store-operation-bridge.ss= maps rows into
  append-fact, append-communication-event, claim-job-lease, retain-artifact,
  and attach-sandbox-handle operation receipts while keeping Scheme side-effect
  free.

This means the architecture is not missing its spine. The missing work is to
make that spine the ordinary user surface and the ordinary runtime interface.

** 10.06.53.60 Gaps Exposed By LoopX
*** 10.06.53.60.10 Policy Family Gap
LoopX exposes many concepts as CLI flags and runtime records:
=task_class=, =action_kind=, =required_write_scope=, =required_capability=,
=decision_scope=, =claimed_by=, =blocks_agent=, =monitor_target_key=,
=cadence=, and =next_due_at=.

In =poo-flow= these should become named POO policy families and profile
fragments. A user should declare them as policy objects or functional helpers,
not by assembling command-line state.

Closure requirement:
- define first-class policy families for work lanes, user gates, monitors,
  capability gates, write scopes, claim ownership, and successor handoff;
- make them composable through C3 role inheritance and slot precedence;
- provide final alist/JSON-like rows only at projection and runtime handoff
  boundaries.

*** 10.06.53.60.20 Strategy Engine Gap
LoopX's =quota should-run= logic is valuable because it encodes real agent
decisions: run, observe, ask owner, quiet-skip, safe bypass, self-repair,
capability bridge repair, workspace guard, and scheduler backoff.

If =poo-flow= cannot express those as strategy combinators, users will rebuild
large condition trees downstream.

Closure requirement:
- introduce functional strategy combinators for eligibility, gate blocking,
  capability matching, monitor cadence, fallback routing, outcome floors,
  self-repair, and no-op backoff;
- keep strategy decisions inspectable as receipts;
- make conflict reasons first-class, not prose-only explanations;
- keep runtime execution outside the strategy engine.

*** 10.06.53.60.30 User Interface Gap
The desired user interface is not a raw workflow DSL. It is a POO-native
authoring surface where a user can say:
- this goal has these lanes;
- this lane has these gates and capabilities;
- this monitor observes this evidence at this cadence;
- this fallback may run when this gate blocks;
- this runtime may consume these operation families;
- this projection is public-safe.

Closure requirement:
- expose small macros or helpers that lower to ordinary POO objects and
  functional values;
- use compile-time validation for static declaration shapes;
- preserve object names, slots, policy kind, owner, and projection schema at
  the call site;
- avoid a parallel alist DSL.

*** 10.06.53.60.40 Projection And Frontstage Gap
LoopX is careful that dashboards and reports consume projections, not private
files. =poo-flow= needs the same discipline as a reusable project invariant.

Closure requirement:
- define projection families for status, attention queue, handoff packet,
  operator gate, run summary, runtime manifest, and public/private warning;
- keep lineage fields such as =source-ref=, =supersedes=, =causal-refs=,
  =watermark=, and =runtime-owner= as data;
- make public/private projection checks part of the receipt/projection layer,
  not a dashboard-specific convention.

*** 10.06.53.60.50 Runtime ABI Performance Gap
The runtime interface is already shaped correctly, but it must become
performance-grade. Rust should be able to attach shared threads, queues,
durable logs, indexes, and sandbox workers without parsing Scheme source or
guessing policy semantics.

Closure requirement:
- keep runtime envelopes versioned and minimal;
- prefer ids, refs, handles, policy snapshots, and operation batches over raw
  payload transfer;
- make idempotency keys, watermarks, lease refs, and conflict policies explicit;
- support batched manifests for high-volume durable operations;
- make response normalization cheap and monotonic;
- keep every Scheme-produced runtime manifest marked =runtime-executed = #f=;
- let Marlin/Rust return compact receipt updates, telemetry refs, and failure
  diagnostics instead of large runtime payloads.

*** 10.06.53.60.55 Marlin ABI Pressure Relief
Marlin is the best real runtime-language integration case for =poo-flow=. The lesson is not that =poo-flow= should duplicate Marlin's runtime ABI. The lesson is that =poo-flow= must reduce pressure on that ABI by sending already-normalized strategy facts instead of forcing Marlin to reconstruct user intent from rich Scheme objects.

Evidence from the current Marlin Rust side:
- =GraphLoopStrategyRuntime= already distinguishes =StaticPolicy=, =NativeScheme=, and =NativeGerbil=. The native variants are typed C ABI adapters, not a license to expose raw Scheme internals.
- =GraphPolicyProposal= carries schema, strategy identity, optional native ABI requirement, graph, input/output digests, and diagnostics. Rust validates this proposal before it compiles or executes a graph.
- =GraphNativeAbiRequirement= and =GraphNativeAbiReadinessReceipt= make ABI id, version, required symbols, available symbols, missing symbols, and readiness status explicit.
- kernel tests require readiness to pass before producing an execution request; missing symbols produce diagnostics instead of late runtime failure.
- =GraphLoopRunRequest= combines the executable graph request with policy profile, governance, stop policy, iteration budget, and evidence policy.
- =LoopPolicyProfile= already names queue, continuation, tool-batch, model-route, evidence, failure, memory, human-gate, and self-evolution policy families.
- =SessionRuntimeSnapshot= and =TokioRuntimePolicyReceipt= keep runtime concurrency details Rust-owned: worker threads, blocking threads, fanout join, blocking bridge, diagnostics, and shutdown timeout.
- =marlin-gerbil-scheme= uses ABI id/version/status codes, borrowed UTF-8 descriptors, request/projection structs, one-time Scheme runtime initialization, typed projection receipts, and a safe Rust selector wrapper.
- the native request path tracks conversion cost and supports epoch-owned backing for stable match keys, so repeated native calls can reuse graph/profile/root scalar backing and marshal only per-decision payload.

Design implication:
- =poo-flow= should project POO policy families into compact, Marlin-aligned policy packets: profile id, policy epoch, graph/profile/root ids, action kind, gates, capability/readiness requirements, evidence refs, watermarks, idempotency keys, lease refs, and conflict policy.
- =poo-flow= should resolve rich POO inheritance, functional combinators, strategy eligibility, and user-facing diagnostics before handoff. Marlin should not need to know why a policy object inherited a slot or which downstream UI concept produced it.
- ABI readiness must become a normal strategy gate. If readiness fails, =poo-flow= should emit =ask-owner=, =quiet-skip=, =self-repair=, or diagnostic receipts before runtime execution.
- The C ABI remains the compiled native adapter and SDK target. The user interface remains POO-native policy families and functional strategy combinators.
*** 10.06.53.60.60 Validation Gap
LoopX keeps many smoke tests because regressions strand automation. =poo-flow=
should reduce the need for downstream smoke sprawl by validating shared
contracts once.

Closure requirement:
- each policy family needs constructor, validation, projection, and scenario
  tests;
- each strategy combinator needs a focused acceptance scenario;
- each runtime envelope needs one request, one response, one failure, and one
  batch/handoff test;
- public user-interface scenarios should prove that downstream users can
  configure the behavior without writing a new control-plane engine.

** 10.06.53.70 Closure Definition
The perfect closure for this layer is not "POO Flow executes everything."
The closure is:

1. A downstream agent-control product can express its goal/todo/gate/quota/
   monitor/handoff model as POO policies and functional strategy composition.
2. The Scheme side validates, explains, and projects the model without owning
   heavy side effects.
3. The runtime handoff is stable, compact, versioned, and fast enough for Rust
   or Marlin to optimize real execution independently.
4. Dashboards and operator surfaces read projections, not private source files.
5. Every runtime-facing row says what it is, who owns execution, what ids and
   refs it touches, what policy produced it, and whether runtime execution has
   happened.

When those conditions hold, LoopX-style products become configuration and
projection packages over =poo-flow=. When they do not hold, they become
parallel engines.

** 10.06.53.80 Next Design Work
The next concrete design work should be ordered this way:

1. Define =work-lane-policy=, =monitor-policy=, =user-gate-policy=,
   =capability-gate-policy=, =claim-policy=, and =handoff-policy= as
   POO-native families.
2. Define strategy combinators that consume those families and emit an
   inspectable decision receipt.
3. Define a compact runtime action packet that can represent run, observe,
   quiet-skip, ask-owner, self-repair, fallback, and handoff actions.
4. Extend durable runtime operation handoff with idempotency, watermark, lease,
   and conflict-policy fields where they are not already explicit.
5. Add one user-interface scenario that recreates the LoopX-shaped lifecycle
   without introducing a LoopX-shaped control-plane engine.

This should be documented before implementation and then advanced through
focused Gerbil tests. The implementation goal is not compatibility with LoopX.
The goal is to make a LoopX-like downstream product unnecessary as a separate
control-plane layer.
