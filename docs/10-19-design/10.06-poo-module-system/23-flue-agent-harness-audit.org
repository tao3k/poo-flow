* 10.06.23 Flue Agent Harness Audit
** 10.06.23.10 Status
This audit records the first pass over =.data/flue=, cloned from =withastro/flue= on 2026-06-20.

Flue is useful reference material for =poo-flow= because it treats agents and workflows as project-owned declarations that are compiled into deployable runtime artifacts. Its public surface is intentionally small: users define agents, workflows, profiles, tools, skills, and sandbox adapters, while the runtime owns admission, sessions, dispatch, events, persistence, and deployment targets.

** 10.06.23.20 Project Shape
Flue's own =AGENTS.md= describes it as a framework where projects containing agents and workflows are compiled into deployable server artifacts.

The core package split is:

- =packages/runtime/= owns sessions, harnesses, tools, skills, sandbox plumbing, dispatch, event streams, and the public HTTP API.
- =packages/cli/= owns build/dev tooling, discovery, target integration, and generated entries.
- =packages/sdk/= owns client-side consumption of deployed agents and workflows.
- =packages/react/= projects workflow streams into UI-friendly state.
- =examples/*= are small integration fixtures, especially agent, workflow, Cloudflare, and channel examples.

Flue also has a strong source layout rule: agent and workflow sources use either =<root>/.flue/= or =<root>/=. When =.flue/= exists, bare =agents/= and =workflows/= are ignored. This is a useful precedent for avoiding ambiguous user-project roots.

** 10.06.23.30 User Interface
The user-facing agent API is compact.

An addressable agent is usually a default export from =agents/<name>.ts=:

#+begin_src typescript
import { createAgent, defineAgentProfile } from '@flue/runtime';

const assistant = defineAgentProfile({
  instructions: 'You complete task requests submitted directly to this agent.',
});

export default createAgent(() => ({ profile: assistant }));
#+end_src

A workflow is a file under =src/workflows/= that exports =run(...)=:

#+begin_src typescript
import { createAgent, type FlueContext } from '@flue/runtime';

const summarizer = createAgent(() => ({
  model: 'anthropic/claude-haiku-4-5',
  instructions: 'Summarize the supplied document clearly and concisely.',
}));

export async function run({ init, payload }: FlueContext<{ text: string }>) {
  const harness = await init(summarizer);
  const session = await harness.session();
  const response = await session.prompt(payload.text);
  return { summary: response.text };
}
#+end_src

This shape is important for =poo-flow=: users should not configure the entire runtime graph directly. They should declare compact project-owned objects and let the framework compile or project those declarations into runtime descriptors.

** 10.06.23.40 Runtime Boundary
Flue draws a clear distinction between workflows and continuing agents.

- Workflows create workflow runs. A run has a =runId=, events, logs, result or error, and inspection surfaces such as =flue logs <runId>= and =GET /runs/<runId>=.
- Direct agent prompts and =dispatch(...)= operate inside persistent agent sessions. They are not workflow runs and do not appear in run history.
- =dispatch(...)= returns a =dispatchId= receipt after admission; it does not wait for model execution, tool calls, or a reply.

For =poo-flow= this maps well to three separate concepts that should not collapse into one object:

- workflow run: finite unit with receipt and event stream;
- agent session: continuing stateful interaction;
- dispatch/admission receipt: accepted asynchronous input, not a run result.

** 10.06.23.50 Harness Model
Flue's conceptual stack is:

#+begin_src text

Agent module
Agent instance
Harness
Session
Operation
Turn

Workflow
Workflow run
#+end_src

The useful part for =poo-flow= is the =Harness= boundary. A harness is the initialized environment for a created agent. It exposes:

- =session(name?)= for conversation state;
- =shell(...)= for command execution in the sandbox;
- =fs= for filesystem operations outside the conversation transcript.

This is close to the boundary we want for Scheme-side POO objects. =poo-flow= can express profile, harness, session, operation, sandbox, and receipt as composable POO objects, while Rust or backend runtimes own durable execution, filesystem isolation, process control, and event storage.

** 10.06.23.60 Delegation And Structured Results
Flue's =session.task(...)= delegates work to a detached child session. It can select a named subagent profile and require a validated structured result. This is directly relevant to =loop-governor= and =human-audit=.

Design lessons for =poo-flow=:

- A delegated task should be an operation under a parent session, not a hidden workflow.
- A governor or reviewer profile should be selectable by name.
- Structured results should be part of the operation contract, not parsed from prose after the fact.
- Child-session history should have an explicit ownership and retention rule.

This supports the current direction where =loop-governor= and =human-audit= are specializations over the same POO governor lineage rather than unrelated ad-hoc features.

** 10.06.23.70 Observability Projection
Flue's React package projects workflow streams into a small state machine:

- =idle=
- =connecting=
- =running=
- =completed=
- =errored=
- =disconnected=

It stores events, logs, result, and error in a =WorkflowSnapshot=. This is a useful UI-facing projection for =poo-flow=: keep the runtime/event receipt rich, but expose a small stable status model for agents, CLI tools, and dashboards.

** 10.06.23.80 POO Flow Design Takeaways
The main lesson is not to copy Flue's TypeScript API. The lesson is to keep the user's interface small while preserving strong runtime boundaries.

Recommended =poo-flow= direction:

- Keep =poo-flow!= and =(use-module ...)= as the project interface.
- Let profiles and cases declare compact POO objects.
- Compile user declarations into internal module descriptors, run configs, sandbox descriptors, loop-engine descriptors, and runtime receipts.
- Treat workflow runs, agent sessions, dispatch receipts, and event streams as separate object families.
- Expose optional UI/state projections as derived views, not as the canonical runtime state.
- Keep heavy execution, durable event storage, sandbox process control, and deployment integration outside Scheme.

** 10.06.23.90 Next Implementation Targets
1. Add a =workflow-run= object family with fields for =run-id=, =status=, =result=, =error=, =events=, and =logs=.
2. Add an =agent-session= object family that can own operations without becoming a workflow run.
3. Add a =dispatch-receipt= object family separate from workflow receipts.
4. Extend =loop-engine= profile examples to show named governor, reviewer, and human-audit profiles as delegated operations.
5. Add a small stable projection function that turns rich receipts into a UI/CLI snapshot similar to Flue's workflow snapshot states.
** 10.06.23.95 Scheme Closure Status
The Flue learning loop is closed for the current Scheme-side scope. =poo-flow= now keeps the compact user interface while projecting the richer control-plane object graph:
- =agent-profile=, =agent-harness=, =agent-session=, =workflow-run=, =dispatch-receipt=, =agent-operation=, and =runtime-snapshot= are separate inert object families.
- =loop-engine= user declarations project runtime handoff facts, command manifests, and snapshot rows without realizing descriptors.
- named governor, reviewer, auditor, and =human-audit= gates are visible as =agent-profile=, =agent-harness=, =agent-session=, and =delegated-operation= projections over the control-plane object graph.
- =loop-engine-result= POO objects project structured reviewer and human-audit result contracts into profiles, operations, delegated operations, manifests, and presentation rows, including report-only validation diagnostics for malformed declarations.
- presentation exposes =loop-engine-delegated-operations= so agents can inspect the Flue-style delegation boundary from user config.

Runtime execution, durable events, sandbox process control, and deployment integration remain outside this milestone. They are runtime/backend responsibilities and are intentionally not implemented in Scheme here.
