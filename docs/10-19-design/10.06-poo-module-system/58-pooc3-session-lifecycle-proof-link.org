* 10.06.58.10 Intent

This document records the implementation contract for proving the session
lifecycle boundary used by the Scheme control plane.

The proof target is not the Scheme interpreter. The proof target is the typed
projection emitted from POO-native session, policy, strategy, placement, and
handoff objects into Lean lifecycle declarations.

* 10.06.58.20 Scheme Boundary

The current implementation evidence maps to these Scheme-side surfaces:

| Surface | Proof Atom |
| src/modules/session/objects.ss | public session object facade |
| src/modules/session/objects-core.ss | chunk, lineage, placement, value |
| src/modules/session/objects-handoff.ss | handoff receipt ABI |

The handoff boundary is intentionally report-only:

| Receipt Field | Proof Fact |
| runtime owner marlin-agent-core | runtimeOwnerMarlin |
| runtime-executed #f | runtimeExecutedFalse |
| handoff-required #t | handoffRequiredTrue |
| runtime parses Scheme source #f | runtimeParsesSchemeSourceFalse |
| Scheme manufactures runtime handlers #f | schemeManufacturesRuntimeHandlersFalse |

* 10.06.58.60 Projection Contract

The Scheme-to-Lean link should project POO session objects into the exact
`SessionState` fields. The projection is valid only when the Scheme source
preserves these facts:

| session chunk | chunkPresent |
| session lineage | lineagePresent |
| session placement | placementPresent |
| placement diagnostics without missing-profile | not placementMissingProfile |
| placement resolved | placementResolved |
| session value | valuePresent |
| registered agent | agentRegistered |
| registered subagent | subagentRegistered |
| approved communication edge | channelAuthorized |
| dispatched message | messageDispatched |
| runtime summary | runtimeSummaryPresent |
| handoff receipt | handoffReceiptPresent |

Session channel receipt and strict agent graph projection are owned by
=SessionLifecycle=. =ScenarioProof= keeps forwarding theorem names during the
migration, but it should not define session endpoint authorization semantics
directly.

UI strategy, runtime manifest, handoff, and scenario benchmark blockers are
owned by =UserInterface=. =ScenarioProof= should compose those blockers into
scenario packets and failure matrices instead of re-proving UI dependency
semantics locally.

UI scenario facts projected from Scheme are also owned by =UserInterface=.
=ScenarioProof= keeps compatibility aliases for the old names, but
=UiScenarioFact=, =uiScenarioStateOfFacts=, and
=UiScenarioProjectionFailureMatrix= are now part of the UI model surface.

* 10.06.58.69 Proof Reflection And Failure Obligations

The first proof pass was not strong enough to discover defects. It mostly
translated already-aligned positive facts into Lean fields:

#+begin_src latex
\[
\begin{aligned}
GraphProjection(g)
  = \langle AgentIdsPresent(g),
           LineageEdgesPresent(g),
           ChannelReceiptsAuthorized(g) \rangle
\end{aligned}
\]
#+end_src

That shape is useful as a bridge check, but it cannot find an invalid channel
receipt. The risk is hidden inside a caller-supplied proposition. The stricter
model splits receipt presence from receipt authorization:

#+begin_src latex
\[
\begin{aligned}
Authorized(r) \equiv&
  (SourceInGraph(r) \lor SourceIsLoopEngine(r)) \\
  &\land (TargetInGraph(r) \lor TargetIsLoopEngine(r)) \\
  &\land (SourceInGraph(r) \lor TargetInGraph(r)) \\
AuthorizedReceipts(R) \equiv&
  \forall r \in R.\ Authorized(r)
\end{aligned}
\]
#+end_src

This creates a real failure obligation:

#+begin_src latex
\[
\begin{aligned}
\exists R.\ ReceiptsPresent(R)
  \land \neg AuthorizedReceipts(R)
\end{aligned}
\]
#+end_src

The reflection rule for future proof work is: every positive bridge theorem
needs a matching failure theorem. A scenario proof is incomplete if it cannot
state a counterexample for the common bad configuration that the Scheme/UI
surface is supposed to reject.

The first complete failure matrix covers the six problems exposed by this
reflection pass:

| Exposed issue | Lean obligation |
| weak graph model | receipt presence does not imply authorization |
| positive-only proof | message dispatch is blocked without authorized receipts |
| Scheme/Lean mismatch | strict graph facts derive channel authorization from endpoint authorization |
| missing scope model | channel and message start are blocked when active scope is too small |
| missing strategy/policy model | policy, strategy ownership, and local validation denial block authorized dispatch witnesses |
| missing branch/runtime blockers | sibling branch conflict blocks dual start; handoff receipt requires runtime summary, Marlin owner, and required handoff |

#+begin_src latex
\[
\begin{aligned}
\neg(ChannelScope \le ActiveScope(s))
  &\Longrightarrow \neg CanStart(s, ChannelAuthorization) \\
\neg(MessageScope \le ActiveScope(s))
  &\Longrightarrow \neg CanStart(s, MessageDispatch) \\
(\forall m.\neg PolicyAllows(m))
  &\Longrightarrow \neg AuthorizedDispatch \\
(\forall a,m.\neg StrategyOwnsStep(a,m))
  &\Longrightarrow \neg AuthorizedDispatch \\
BranchSibling(A,B) \land Exclusive(s)
  &\Longrightarrow \neg(CanStart(s,A) \land CanStart(s,B)) \\
\neg RuntimeSummary(s)
  &\Longrightarrow \neg CanStart(s, HandoffReceipt)
\end{aligned}
\]
#+end_src
