#+title: Composition Proof Gate

* Boundary

Composition is a control-plane object. It combines profiles, local overrides,
module order, and scenario gates into one reusable policy surface. It must not
execute the runtime.

The runtime boundary is part of the proof:

#+begin_src latex
CompositionAccepted(c) \Rightarrow NoRuntimeExecution(c)
#+end_src

* Receipt Facts

The Lean proof gate receives normalized facts:

#+begin_src lean
structure CompositionReceiptFacts where
  profileRefsOk : Bool
  overridesScopedOk : Bool
  modulesOrderedOk : Bool
  scenarioGateOk : Bool
  noRuntimeExecution : Bool
#+end_src

The acceptance rule is:

#+begin_src latex
Accepted(c) \equiv
  ProfileRefsOk(c)
  \land OverridesScopedOk(c)
  \land ModulesOrderedOk(c)
  \land ScenarioGateOk(c)
  \land NoRuntimeExecution(c)
#+end_src

* Why This Exists

LangChain and LangGraph scenarios are not runtime implementations in this
project. They are composition cases that prove POO Flow can express the same
control-plane intent with reusable profiles, scoped overrides, graph policies,
and formal acceptance gates.

* Rejection Rules

Every failed receipt must have a proof-facing reason:

#+begin_src latex
\neg Accepted(c) \Leftarrow
  \neg ProfileRefsOk(c)
  \lor \neg OverridesScopedOk(c)
  \lor \neg ModulesOrderedOk(c)
  \lor \neg ScenarioGateOk(c)
  \lor \neg NoRuntimeExecution(c)
#+end_src

The current Lean module defines one rejection theorem per failed contract bit.
Generated artifacts should call those theorems instead of constructing ad hoc
proof branches.

Current theorem coverage:

- =compositionReceiptRejectedByProfileRefs=
- =compositionReceiptRejectedByOverrideScope=
- =compositionReceiptRejectedByModuleOrder=
- =compositionReceiptRejectedByScenarioGate=
- =compositionReceiptRejectedByRuntimeExecution=

* Scheme Facts

=src/module-system/composition-proof-facts.ss= converts normalized Scheme
composition contract bits into structured proof facts. It does not inspect
runtime state, does not execute a graph, and does not concatenate Lean source
strings.

#+begin_src scheme
(poo-flow-composition-contract->proof-facts
 'generated-langgraph-composition-receipt
 #t #t #t #t #t #f)
#+end_src

These facts are FFI-ready data for a proof bridge. Lean modules stay on the Lean
side.

This gives the user-interface layer a concrete path:

1. Build or reuse a composition.
2. Produce a normalized composition receipt.
3. Export typed proof facts through the bridge.
4. Require the Lean artifact or generated Lean bridge module to compile before
   accepting the scenario.
