* Scope

This research separates two mechanisms that must not share a name, owner, or
performance model:

- POO native laziness is demand-driven evaluation of Gerbil POO object slots.
- =lazy-load= is a project macro for deferred module resolution built on
  =std/lazy=.

The first is the normal evaluation model for large POO semantic objects. The
second is module-loading syntax. Neither replaces the other.

* Evidence

** Gerbil POO Object Evaluation

=instantiate-object!= creates an object instance, computes prototype
precedence, and compiles slot functions. It does not materialize every slot
value. =.ref= is the demand point: it obtains a requested slot through the
object instance hash, while =.ref/cached= reads only an already materialized
slot. =force-object= explicitly walks every slot, so it is a full
materialization operation rather than an ordinary accessor.

Slot specifications establish a delayed value graph:

- constant slots are already known values;
- thunk slots defer an external computation;
- self slots compute from the receiving object;
- computed slots receive both the receiving object and the inherited slot
  function.

This is slot-granular call-by-need. A large POO object should be treated like a
demand graph, not like an eager record that happens to have accessors.

** Extension And Cache Lifetime

=.extend= creates a new object with the old object as a super. It reuses the
prototype and slot-definition graph, but the derived object has its own
instance lifetime. A derived object must not assume that a value forced on its
parent is automatically reused from the parent instance cache.

There are therefore three distinct reuse levels:

1. Repeated reads of one slot on one object reuse that object's materialized
   slot value.
2. Related objects reuse prototype precedence and slot definitions through
   extension or composition.
3. Derived objects that need the same expensive fact must reference a shared
   fact object. They must not expect prototype inheritance alone to share a
   parent object's materialized values.

The object tests confirm that instance-local mutation does not become a
prototype-wide semantic update. This is the correct boundary for persistent
policy objects: evolve by extension or composition, not by treating a forced
instance cache as shared application state.

** Prototype Composition

=compose-proto= and =compose-proto*= build prototype closures. They do not
provide a request-wide composition cache. Compose a stable family once at its
composition root, instantiate the resulting object once, and reuse that object
for repeated strategy or projection reads. Recreating a =.mix= or composition
chain in a per-request loop discards this benefit.

** Materialization Boundaries

=force-object=, =object-values=, =.alist=, and =MonomorphicObject= validation
walk all slots. They are correct at explicit boundaries:

- final ABI, JSON, or receipt serialization;
- complete contract validation;
- an intentionally complete report;
- a Scenario assertion that proves full materialization.

They are incorrect in a partial presentation, route selection, profile lookup,
or any hot path that needs only a small field subset.

** Slot Hygiene

The downstream =poo-dynamic-slot-binding-leak= Scenario records a concrete
failure mode: a form such as =(.o (slot slot))= can allow POO slot binding to
capture the right-hand lexical identifier and recursively force the same slot.
The repair is to give the lexical value a distinct binding before it enters the
slot declaration. Public syntax must preserve hygiene and generate distinct
value bindings; a macro must never manufacture same-name slot/value pairs.

** Standard Lazy Support

=std/lazy= exposes =delay-lazy= and =force-lazy=. A lazy object moves through
=lazy=, =eager=, and =resolved= state. Successful forcing resolves and reuses
the result. A lazy result can share its inner lazy state with the outer value.
The implementation has explicit reentrance-state checks. A failed thunk is not
written as a resolved value, so a later force can retry.

The source does not establish a cross-thread safety guarantee. Native lazy is
therefore a Scheme control-plane mechanism. Runtime singleflight, cancellation,
durable retries, and resource scheduling remain runtime responsibilities.

** Current Module Syntax

=load!= is a compile-time macro that expands to source inclusion. Wrapping it
in a runtime promise cannot make the inclusion lazy: macro expansion happens
before the promise exists. The current module-system lazy-loader test models
deferred loading with an explicit plan structure, but it is not the target
public architecture.

* Conclusions

** Composition Is The Performance Unit

The performance unit is a large POO composition, not an isolated policy or a
flat registry entry. A user composition joins real profile, agent, model,
sandbox, strategy, task, durable, graph, and handoff objects into one namespace
and prototype topology. The composition root is the runtime demand root.

For a composition graph with object nodes =V=, object-reference or prototype
edges =E=, and a runtime request that reaches subgraph =R=, the required shape
is:

- compose namespace and prototype topology once, outside the request loop;
- force only =R= on the first demand, never all of =V= by default;
- reuse object-local slot values on repeated demand;
- share high-fanout facts by direct reference to one fact object;
- use a narrow extension overlay for changed objects while retaining references
  to unchanged fact objects;
- project only the demanded runtime root at the ABI boundary.

The objective is therefore a first demand proportional to the reachable object
subgraph, =O(|R|)= plus ordinary slot lookup, rather than =O(|V|)= full
materialization. A repeated demand should perform only the necessary cached slot
lookups. This is the practical POO equivalent of a demand-driven lazy object
graph.

Composition is also a namespace. Static component names and public object slots
must be established by the composition syntax and POO object graph, not found
by string scans, raw alist traversal, or a second descriptor language. A policy
is simply one possible object node; it is not the model around which the system
is optimized.

** POO Is The Primary Lazy Substrate

Large policy, strategy, profile, graph, and presentation objects must use POO
slot demand as their default evaluation model. Do not place a parallel
=vector=, alist, or global memo cache around computed slots. Such caches repeat
the object-instance responsibility and make cache lifetime invisible to POO
extension.

An expensive, shared semantic fact belongs in a named POO fact object. Multiple
profiles or compositions reference that fact object. Each consumer then forces
only the needed fact slots, while the fact object owns its own materialized
values.

** Lazy Evolution Is Persistent Object Evolution

Lazy evolution does not mean mutable invalidation of a generic cache. It means
constructing a derived POO object with only the changed slots and preserving
shared fact-object references for unchanged semantics. The new object receives
its own demand graph. This makes the recomputation boundary visible in the
prototype and object graph.

** Lazy Load Is A Separate Macro

The replacement for the old lazy-load-plan model is a project =lazy-load=
macro. It accepts a runtime source reference or resolver expression and lowers
that expression to =std/lazy= delay semantics. Forcing occurs through the
standard lazy interface at an explicit module-resolution boundary.

=lazy-load= must not accept or wrap =load!= source inclusion. Compile-time
source inclusion and runtime deferred resolution are different phases with
different owners.

* Required Scenario Evidence

The eventual Scenario suite must use real input and expected artifacts, not
synthetic path strings alone. It must demonstrate:

- a large composition of real profile, agent, model, sandbox, strategy, task,
  graph, and handoff objects has a stable namespace and precomposed topology;
- demanding one runtime root forces only its reachable object subgraph and
  leaves unrelated composition branches unmaterialized;
- a partial POO read does not materialize unrelated slots;
- repeated reads of one computed slot reuse the object-local result;
- multiple demanded roots reuse one shared fact object rather than recomputing
  its high-fanout slots;
- a derived composition reuses shared fact objects while changed object slots
  receive a new demand graph;
- full ABI projection intentionally materializes the required complete shape;
- same-name dynamic slot/value declarations are rejected or rewritten
  hygienically;
- =lazy-load= delays a runtime resolver expression, while =load!= remains
  compile-time inclusion.
