The current gap is not a missing row operator. The gap is a first-class
profile-policy layer that explains why a profile is legal for a backend, which
capabilities it requires, which safety rules are monotonic, and how validation
failures are projected.

** 10.06.41.40 Design Direction

Model profile policy as POO objects, not as a new raw alist DSL.

Profile policy should be a composable object family with slots for:

- =required-capabilities=: command, filesystem, code interpreter, network,
  persistence, or backend-specific capabilities.
- =backend-intent=: backend kind, optional image, entrypoint, environment,
  timeout, working directory, and placement hints.
- =resource-policy=: filesystem, network, env, mounts, budget, and isolation
  requirements.
- =safety-policy=: deny entries, human gates, write capability, production
  access, and irreversible-action gates.
- =failure-policy=: structured failure kind, severity, recoverability, and
  report-only versus hard-error behavior.
- =projection-policy=: how a valid profile lowers into backend handoff data
  without executing it.

Composition must use POO/C3-style precedence for role and slot ordering, but
safety is not decided by precedence. Safety is decided by validation phases
after composition.

** 10.06.41.50 Validation Phases

Adopt a multi-phase profile-policy validator:

1. =authoring=: row syntax, operator validity, known slots, and backend row
   ownership.
2. =composition=: POO supers, row contribution merge, derivation metadata, and
   profile lineage.
3. =capability=: compare required capabilities against backend capability
   objects.
4. =resource=: filesystem, network, env, and resource contract validation,
   including path existence where the policy declares a concrete workspace
   path.
5. =safety=: monotonic rules for permission narrowing, denylist union, budget
   tightening, and human-gate escalation.
6. =projection=: backend handoff shape is complete enough for Marlin/Rust
   without running the backend.

Validation output should be structured diagnostics, not ad hoc error strings:

#+begin_src text
(kind phase severity profile slot source detail recoverable?)
#+end_src

This mirrors OpenRath's structured =ToolExecutionFailure= idea while keeping
=poo-flow= on the Scheme control-plane side.

** 10.06.41.60 Capability Registry Plan

Add a Scheme-side backend capability registry. It should be report-only and
static by default, like OpenRath's =Capabilities=:

- backend kind.
- isolation level.
- supports command.
- supports filesystem.
- supports code interpreter.
- supports network.
- supports persistence.
- max sandboxes or concurrency hint.
- cold-start hint.
- availability check receipt.

Backend modules such as nono, cube, docker, and future opensandbox-like
backends should publish capability objects. Profiles then require capabilities
rather than naming backend behavior implicitly.

The registry must not perform runtime work. Availability checks should remain
cheap and diagnostic. Real execution still belongs outside =poo-flow=.

** 10.06.41.80 P1-P4 Delivery Plan
This is the current staged implementation plan. It is feature-led: each phase
has a user-visible result, a Scheme object boundary, and a receipt or diagnostic
shape that can be checked without copying OpenRath internals.

P1: Profile Policy Validation Receipts
- User-visible result: sandbox profiles produce structured profile-policy
  diagnostics that explain concrete resource, safety, and projection failures.
- Scheme boundary: =profile-policy= remains a POO object family. Validation reads
  composed POO slots and emits diagnostic POO objects or serializable receipts.
- Main owners:
  - =src/modules/sandbox-core/profile-support/policy.ss=
  - =src/modules/sandbox-core/profile-support/resolve.ss=
  - =src/modules/sandbox-core/profile-support/derivation.ss=
  - =src/modules/agent-sandbox/config.ss=
- Required behavior:
  - represent diagnostics as =profile-policy.diagnostic.v1= receipts.
  - validate concrete filesystem/workspace paths when a profile declares them.
  - report missing filesystem policy when a sandbox profile enables resource
    isolation that cannot be meaningful without filesystem scope.
  - preserve profile lineage, source object, phase, slot, severity, and
    recoverability in diagnostics.
  - keep row operators as authoring helpers only. Do not add a second patch DSL.
- Out of scope in P1:
  - runtime provisioning.
  - backend process startup.
  - current testing/CLI refactor files.

P2: Backend Capability Registry
- User-visible result: every sandbox backend can explain what it supports before
  a profile is handed to runtime code.
- Scheme boundary: backend capabilities are POO objects that profile policy can
  inherit, extend, and validate against.
- Main owners:
  - =src/modules/sandbox-core/profile-support/policy.ss=
  - =src/modules/sandbox-core/profile-support/resolve.ss=
  - =src/modules/agent-sandbox/config.ss=
  - backend modules for nono, docker, cube, and future sandbox providers.
- Required behavior:
  - publish static capability objects for command, filesystem, network,
    persistence, interpreter, isolation, and concurrency hints.
  - make capability lookup extension-friendly. Backend modules should register
    POO objects, not string-only cases.
  - emit capability mismatch diagnostics through the same P1 receipt channel.
  - keep availability checks cheap and diagnostic. Runtime work belongs outside
    this registry.
- Exit criteria:
  - a profile can declare required capabilities and get a deterministic
    validation result against a selected backend capability object.

P3: Session/Profile Handoff Policy
- User-visible result: session and agent-flow objects can request a named sandbox
  profile and receive a validated placement receipt before runtime handoff.
- Scheme boundary: session placement is functional data over POO objects. The
  Scheme layer builds receipts and intent objects; Marlin/Rust or backend code
  owns execution.
- Main owners:
  - =src/modules/session-core/config.ss=
  - =src/module-system/session-core-config.ss=
  - =src/modules/session/objects.ss=
  - =src/modules/session/transform.ss=
  - =src/modules/sandbox-core/profile.ss=
  - =src/modules/agent-sandbox/config.ss=
- Required behavior:
  - attach profile-policy projection receipts to session placement results.
  - preserve session lineage, requested profile, backend intent, and validation
    diagnostics.
  - make invalid placement a structured failure value, not an implicit runtime
    exception.
  - avoid runtime handles, refcounts, or backend lifecycle state in Scheme
    profile objects.
- Exit criteria:
  - session configuration can show which validated profile would be used for
    each runtime placement path.

P4: Funflow and CI/CD Profile Policy Integration
- User-visible result: workflow and CI/CD pipelines can request sandbox profiles
  for checks, build tasks, artifact tasks, and release tasks with the same
  profile-policy validation path.
- Scheme boundary: flow composition stays functional. Task and pipeline objects
  reference profile-policy objects by name or object identity, then lower to
  projection receipts.
- Main owners:
  - =src/modules/funflow/config.ss=
  - =src/modules/workflow/config.ss=
  - workflow CI/CD feature modules.
  - user configuration examples under =user-interface/custom/my-module/=.
- Required behavior:
  - let a pipeline task request a profile-policy object without embedding backend
    runtime details in the task.
  - include profile, capability set, resource scope, safety policy, and
    projection status in the pipeline receipt.
  - support multiple pipelines and task inheritance through POO supers, with
    validation after composition.
  - make invalid CI/CD profile use fail at configuration validation time.
- Exit criteria:
  - a CI/CD example can explain the selected sandbox profile for each task and
    show deterministic diagnostics for illegal profile inheritance or missing
    resources.
** 10.06.41.90 Harness Scenario Lessons
The upstream Gerbil harness scenarios show a better optimization loop than
one-off implementation fixes. The pattern to copy is the evidence shape, not the
specific test harness internals:
- keep the scenario compact and user-shaped.
- preserve mixed dependency context instead of reducing everything to the POO
  dependency alone.
- emit selectors, minimal forms, failure cases, and quality signals as
  structured evidence.
- keep local fixture facts source-loadable so stale installed packages cannot
  hide scenario drift.
- prefer named POO constructors, mixins, and functional helpers over anonymous
  raw compute hooks.
- split hot-path object reads into stable helpers. Avoid repeatedly reading
  expensive nested child POO objects in validation loops.
- write negative scenarios for similar-but-wrong dependencies, inactive
  extensions, illegal profile inheritance, missing concrete paths, and forbidden
  runtime provisioning.

For =poo-flow= this means P1-P4 should be driven by scenario fixtures that prove
the object boundary:

#+end_src

Testing files are intentionally excluded from this slice because the project
testing layer is being refactored. The next code slice should add implementation
owners first, then wire focused scenario gates once the testing boundary is
stable again.

** 10.06.41.90 Non-Goals

- Do not make OpenRath a dependency of =poo-flow=.
- Do not move backend execution into Scheme.
- Do not invent a second raw patch DSL beside POO object extension and existing
  row operators.
- Do not make availability checks perform expensive runtime provisioning.
- Do not let profile inheritance silently widen safety-critical access.
