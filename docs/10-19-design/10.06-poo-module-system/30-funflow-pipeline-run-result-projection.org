* 10.06.30 Funflow Pipeline Run Result Projection
** 10.06.30.10 Purpose
This document closes the next Funflow alignment slice after dependency graph projection.

The previous graph work made =dependency-refs:= inspectable as control-plane topology data. This slice projects that topology into a user-visible pipeline run and result receipt without adding a scheduler, backend adapter, or command execution path in Scheme.
** 10.06.30.20 Boundary
Scheme owns:
- POO-native Funflow pipeline declarations;
- workflow-owned CI/CD check maps;
- dependency graph diagnostics;
- pipeline-run admission receipts;
- blocked/ready result projections;
- sandbox profile reference visibility.

Runtime owners still own:
- process execution;
- parallel scheduling;
- Docker, GitHub, GitLab, Nix, or Marlin provider behavior;
- durable artifact and event storage.
** 10.06.30.30 Implemented Projection
The current projection path is:
#+begin_src text
legacy row surface [removed]
  -> POO-native funflow-check and funflow-pipeline declarations
  -> workflow CI/CD check map
  -> dependency graph
  -> pipeline run
  -> pipeline result
  -> user-interface presentation row
#+end_src

The important behavior is admission, not execution:
- valid topology can produce a ready report run;
- duplicate check names block admission;
- unresolved dependency refs block admission;
- dependency cycles block admission;
- unresolved sandbox profile refs block runtime handoff readiness;
- all blocked paths preserve diagnostics for agents and doctor-style reports.
** 10.06.30.40 POO Shape
The POO boundary stays object-first:
- Funflow config owns the public syntax.
- Workflow CI/CD owns check maps, dependency graphs, pipeline runs, and results.
- User-interface presentation owns compact rows for downstream agents.
- Sandbox modules own profile summaries and unresolved-profile visibility.

This keeps extension local to the object family that understands the slot. A sandbox profile problem should not become a Funflow parser rule, and a topology problem should not become sandbox policy.
** 10.06.30.50 User-Facing Example
The expected downstream surface is POO-native:
#+begin_src scheme
(use-module funflow
  :config
  (.def (funflow/build @ funflow-check
                       check-name profile-ref command-vector runtime-mode)
    check-name: 'build
    profile-ref: 'ci/build
    command-vector: '("gxpkg" "build")
    runtime-mode: 'manifest-handoff)
  (.def (funflow/test @ funflow-check
                      check-name profile-ref command-vector
                      dependency-refs runtime-mode)
    check-name: 'test
    profile-ref: 'ci/check
    command-vector: '("gxpkg" "env" "gxtest" "t/unit-tests.ss")
    dependency-refs: '(build)
    runtime-mode: 'manifest-handoff)
  (.def (funflow/default @ funflow-pipeline pipeline-name checks)
    pipeline-name: 'default
    checks: (list funflow/build funflow/test)))
#+end_src

The visible result should answer the agent's first operational questions:
- what checks exist;
- which checks depend on which checks;
- whether the pipeline is handoff-ready;
- why it is blocked when it is not ready;
- whether Scheme executed anything.
