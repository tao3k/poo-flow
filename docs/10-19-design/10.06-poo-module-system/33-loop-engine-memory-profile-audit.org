#+TITLE: 10.06.33 Loop Engine Memory Profile Audit

* Decision
Loop memory must be modeled from real loop cases, not from one global memory
slot. The next loop-engine step is to promote memory into per-use-case POO
objects under the root loop profile.

Do not introduce a second public memory contract named
=loop-engine-memory-profile=. The existing =loop-engine-memory-policy= should
become the canonical POO object for one loop case's memory behavior. The root
=loop-engine-profile= should expose:

#+begin_src scheme
(.def (repo-loop-profile @ loop-engine-profile
                         use-cases memory-policies)
  use-cases: (list daily-triage-loop
                   ci-sweeper-loop
                   issue-triage-loop)
  memory-policies: (list daily-triage-memory
                         ci-sweeper-memory
                         issue-triage-memory))
#+end_src

This keeps the user interface 100% POO-native. The user declares extensible
objects and composes them with ordinary Scheme values. There is no public alist
DSL, record DSL, or compatibility row syntax.

* Evidence From loop-engineering
The local reference clone is =.data/loop-engineering=. Its examples make memory
case-specific:

- =patterns/daily-triage.md=: reads previous =STATE.md=, CI, issues, and recent
  commits; writes last run, high priority items, watch list, recent noise, and
  human overrides. It starts L1 report-only for one to two weeks.
- =patterns/ci-sweeper.md=: tracks commit SHA, failing job, failure class,
  attempt count, worktree or PR link, flake watch entries, and resolved items.
  It requires early exit when CI is green and escalates after repeated attempts.
- =patterns/issue-triage.md=: tracks backlog health, top five issues, proposed
  labels, possible duplicates, ignored noise, and allowlisted labels. L1 must
  not auto-label or close issues.
- =starters/dependency-sweeper=: tracks watched manifests, in-flight updates,
  denylist packages, and human overrides.
- =starters/post-merge-cleanup=: tracks pending cleanup, completed cleanup,
  deferred human decisions, and denylist paths.

The shared abstraction is not "one memory store". It is a state spine plus a
different recall, commit, retention, and human-gate shape per loop use case.

* Current Gap
The current =loop-engine-profile= carries a singular =memory-policy= slot. The
runtime emits one =memory-receipt= with one store, one scope, one recall list,
one commit list, one ranking, and one retention value.

That is sufficient for a generic report-only demo, but it cannot faithfully
represent a repo profile that contains several loops:

- daily triage should recall =state-spine=, =ci-summary=, =issue-summary=,
  =prior-high-priority=, and =human-overrides=;
- CI sweeper should recall =ci-status=, =failing-job=, =flake-history=,
  =attempt-count=, and =branch-allowlist=;
- issue triage should recall =open-issues=, =prior-top-five=, =label-policy=,
  =possible-duplicates=, and =human-overrides=;
- dependency sweeper should recall =watched-manifests=, =lockfile-drift=,
  =denylist-packages=, and =human-overrides=;
- post-merge cleanup should recall =recent-merges=, =pending-cleanup=,
  =denylist-paths=, and =deferred-decisions=.

Trying to compress those into one global =memory-policy= erases the loop case
boundary and makes Marlin guess how to bind memory facts to a selected loop.

* Target POO Shape
Keep =loop-engine-memory-policy= as the public object family, but make each
object name the loop case it belongs to:

#+begin_src scheme
(.def (daily-triage-memory @ loop-engine-memory-policy
                           use-case store state-path scope recall commit
                           ranking retention)
  use-case: 'daily-triage
  store: 'state-file
  state-path: "STATE.md"
  scope: 'project-day
  recall: '(state-spine ci-summary issue-summary prior-high-priority
            human-overrides)
  commit: '(last-run high-priority watch-list recent-noise human-overrides)
  ranking: 'priority
  retention: 'rolling-report)

(.def (ci-sweeper-memory @ loop-engine-memory-policy
                         use-case store state-path scope recall commit
                         ranking retention)
  use-case: 'ci-sweeper
  store: 'state-file
  state-path: "ci-sweeper-state.md"
  scope: 'branch-failure
  recall: '(ci-status failing-job flake-history attempt-count branch-allowlist)
  commit: '(failure-classification attempt-count worktree-ref verifier-result
            escalation-reason)
  ranking: 'risk-first
  retention: 'last-seven-days)
#+end_src

Validation should enforce:

- =use-case= is a symbol and matches a declared =loop-engine-use-case= name;
- =store= and =scope= are symbols when present;
- =state-path= is a string or absent;
- =recall= and =commit= are symbol lists;
- =ranking= and =retention= are symbols when present;
- duplicate memory policies for the same use case are structural errors.

Scheme still does not read, write, rank, or prune memory. Scheme only lowers
these POO objects into report-only receipts. Marlin owns durable stores,
recall execution, commit execution, ranking, retention, and mutation.

* Receipt Direction
The next =memory-receipt= should expose the case boundary explicitly:

#+begin_src scheme
((kind . memory-receipt)
 (contract . poo-flow.loop-engine.memory-receipt.v1)
 (selected-use-case . ci-sweeper)
 (policies
  (((use-case . daily-triage) ...)
   ((use-case . ci-sweeper) ...)
   ((use-case . issue-triage) ...)))
 (runtime-owner . "marlin-agent-core")
 (runtime-executed . #f))
#+end_src

This receipt alist is an internal projection format, not a public user DSL.
The public user surface remains POO objects inside =(use-module loop-engine
:config ...)=.

* Implementation Plan
1. Extend =loop-engine-memory-policy= with =use-case= and =state-path= slots.
2. Replace the public root =memory-policy= profile slot with
   =memory-policies= as a list of =loop-engine-memory-policy= objects.
3. Update object-to-intent lowering to validate use-case membership and reject
   duplicate memory policies.
4. Update =memory-receipt= to emit all memory policies and the selected
   use-case's matching policy without executing memory operations.
5. Update =user-interface/custom/my-module/profiles/loops.ss= with real cases:
   daily triage, CI sweeper, issue triage, dependency sweeper, and post-merge
   cleanup. These cases should demonstrate different recall/commit/retention
   shapes.
6. Add focused tests for valid multiple memory policies, duplicate use-case
   rejection, invalid slot types, and receipt projection.

* Implementation Status
This slice is implemented in the module-system loop-engine path. The public
profile slot is now =memory-policies=, each =loop-engine-memory-policy= requires
a declared =use-case=, and duplicate policies for one use case are rejected
before intent projection. The =memory-receipt= remains report-only and carries
the selected policy plus the complete policy list for Marlin handoff.

The user-interface profile now shows the five real loop-engineering cases:
daily triage, CI sweeper, issue triage, dependency sweeper, and post-merge
cleanup. The concrete guarded handoff case also uses the new
=memory-policies= list shape.

* Non-Goals
- Do not build a scheduler in Scheme.
- Do not auto-run skills or subagents in Scheme.
- Do not write =STATE.md=, issue labels, CI comments, or dependency updates
  from Scheme.
- Do not model Marlin as a sandbox backend. =marlin-agent-core= remains the
  runtime owner; =nono-sandbox= and =cube-sandbox= remain sandbox profiles.
