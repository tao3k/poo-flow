* 10.06.29 Gerbil POO Programming Guidelines

** 10.06.29.10 Source Evidence

These guidelines are based on =.data/gerbil-poo=:

- =object.ss:120-170= shows that =.ref= instantiates the object and resolves
  the requested slot through =hash-ensure-ref=.
- =object.ss:160-161= shows =.mix= as the primitive for composing prototypes
  through supers plus local slots.
- =object.ss:221-229= and =object.ss:391-400= show =.o= as syntax for object
  slot definitions and prototype construction.
- =doc/poo.md:2-80= distinguishes prototypes from instances: prototypes are
  open recursive specifications; instances are fixed-point slot values.
- =doc/poo.md:130-224= describes multiple inheritance and C3 linearization as
  the scalable way to compose mixins.
- =t/object-test.ss:1-220= demonstrates the idioms this project should follow:
  ={(:: @ super) ...}=, ={(:: @ [a b]) ...}=, ==>= slot extension, ==>.+=
  nested prototype extension, =?= defaults, and =.mix=.

=doc/poo.md:150-177= explains multiple inheritance as a DAG that must be linearized into a precedence list with C3.
=doc/prototypes.md= frames Prototype Object Orientation as incremental composition of code and data fragments with late binding, implemented as fixed-points of higher-order functions.
=fun.ss:1-80= models =Category.=, =Functor.=, =ParametricFunctor.=, and =Identity= as POO type objects. This is the source-level reason POO Flow treats functional programming and POO as one design strand.
=doc/TODO.md= places compositional MOP, method-combination, and storage strategies in future-work territory. POO Flow should not expose those as user-level requirements before the upstream model stabilizes.
=mop.ss= demonstrates meta-level object operations around =.ref=, =.putslot=, slots, methods, and object/type descriptors.
=brace.ss= keeps object authoring syntax small by lowering brace forms to =.o=.
=type.ss= and =fun.ss= show type descriptors and functional dictionaries as POO objects rather than a parallel class system.
=t/object-test.ss= covers =with-slots=, =def-slots=, prefixed slot binding, pattern matching, defaults, ==>=, ==>.+=, and multiple inheritance.

** 10.06.29.20 Project Invariant

This document is the programming contract for code written after studying =.data/gerbil-poo=. It is more specific than a style guide:

1. POO Flow code must be POO-native at the abstraction boundary.
2. User-facing extension APIs must compose prototypes, slots, and named functional helpers.
3. Functional programming is the preferred Scheme control-plane style.
4. Raw alists, record DSLs, direct merge scripts, and naked =(lambda (self super) ...)= compute hooks are not the primary interface.

The practical rule is:

#+begin_src scheme
;; Good: named prototype extension plus a functional constructor.
(.def (nono-sandbox-resources @ base-sandbox-resources)
  resources: =>.+ {cpu: 2 memory: "4Gi"})

;; Bad: a user-owned mini language that manually walks old/new records.
(resources :compute (lambda (self super) ...))
#+end_src

The bad form may appear inside a small, named helper when there is no better POO operator, but it must not become the syntax users learn first.

** 10.06.29.30 Mental Model

Think in this order:

1. Prototype: an open recursive specification of slots.
2. Composition: combine prototypes with =.mix=, supers, and C3 precedence.
3. Slot contract: expose stable names that downstream code can extend.
4. Instance: read slots only after the object has been composed.
5. Projection: convert to plain data only at ABI, receipt, or runtime handoff boundaries.

Do not start from an alist and then retrofit POO on top. If a concept can be extended, it should first be a prototype slot. Alists are acceptable for final receipts, JSON-like ABI payloads, and report-only facts that do not participate in inheritance.

** 10.06.29.40 Slot Operator Policy

Use the highest-level POO operator that expresses the intent:

| Intent | Preferred POO form | Notes |
| Direct value | =(slot value)= or =slot: value= | Whole-slot override. |
| Extend inherited value | =(slot => function extra ...)= | Always invokes the inherited computation. |
| Override nested prototype | =(slot =>.+ prototype)= | Preferred for sandbox/resource/profile fragments. |
| Default base | =(slot ? default)= | Defines the fallback without forcing the user to repeat inherited values. |
| Multiple supers | ={(:: @ [a b c]) ...}= | Lets C3 own precedence. |
| Raw computed slot | =$computed-slot-spec= | Internal escape hatch only. |

The public user interface should prefer ==>=, ==>.+=, =?=, =.mix=, and named constructors. Raw =$computed-slot-spec= and direct =(lambda (self superfun) ...)= code should stay inside implementation helpers with tests.

** 10.06.29.50 Super And Override Discipline

In Gerbil POO, earlier and leftmost child prototypes override later/rightmost super prototypes. The =super= concept is not a reified super-instance. The normal way to use it is to invoke the inherited computation for the same slot.

Guidelines:

1. Use ==>= when the slot value should be derived from the inherited slot value.
2. Use ==>.+= when the inherited value is itself a prototype and the user wants a nested override.
3. Do not assume a helper can inspect arbitrary slots from a notional =super= object.
4. Do not reverse the precedence order to mimic Nix extension functions.
5. Keep override helpers named after the domain operation, not the merge mechanism.

Example:

#+begin_src scheme
(.def base-sandbox
  resources: {cpu: 1 memory: "1Gi"})

(.def (large-sandbox @ base-sandbox)
  resources: =>.+ {cpu: 4 memory: "8Gi"})
#+end_src

This keeps the domain slot stable and delegates nested object merging to POO.

** 10.06.29.60 Functional Kernel Policy

=fun.ss= shows that Category, Functor, and Identity can be modeled as POO type objects. POO Flow should use that pattern for flows:

1. Pure arrows are values.
2. Composition is a function over flow values.
3. Metadata and strategy selection are POO slots.
4. Runtime work is represented as a request or receipt, not performed in Scheme.
5. Hygienic Gerbil macros may improve authoring, but they must expand to POO objects and functional combinators.

This is why =src/core/flow.ss= owns the functional kernel and why workflow module macros must produce ordinary =flow= values. The Scheme layer is a control plane, not a scheduler.

** 10.06.29.70 User Interface Policy

Users should author small declarations that extend upstream prototypes. They should not maintain implementation facades, duplicate contracts, or write custom merge algorithms.

Preferred user-facing shapes:

1. Select an upstream module/profile.
2. Extend a named prototype.
3. Override a stable slot.
4. Add a custom module body under =user-interface/custom=.
5. Inspect presentations and doctor receipts.

Non-preferred shapes:

1. Root-level =objects.ss= or =config.ss= facades under =user-interface=.
2. Alist DSLs that encode inheritance, merge, and validation rules.
3. Row operators as the primary happy path when a named prototype would be clearer.
4. Runtime adapter construction in user config.
5. Direct Scheme execution of Docker, CAS, subprocess, or scheduler behavior.

** 10.06.29.80 Performance And Instantiation Policy

=.ref= instantiates the object and resolves the slot through cached slot functions. This is correct and useful, but it means object shape matters.

Use these rules:

1. Compose mixins first; then read direct slots from the final object.
2. Avoid nested child POO objects when validation or presentation loops will repeatedly inspect them.
3. Keep runtime policy and profile values as native POO objects while they are still extension surfaces.
4. Use Gerbil =defstruct= for generated runtime receipts that are fixed internal facts rather than extension points.
5. Materialize receipt structs to bounded alists only once at ABI, manifest, JSON, test-summary, snapshot, or Marlin handoff boundaries.
6. Keep hot presentation paths over direct slots, fixed struct accessors, cached primitive vectors, or boundary alists, not repeated child =.ref= walks.
7. Do not build POO objects inside tight loops. Accumulate scalar/list/hash state first and construct one final =.o= object at the extension boundary.

The receipt layering rule is:
1. User authoring and module config stay POO: =.o=, =.def=, mixins, slot operators, and =object<-alist= at named adapter edges.
2. Generated runtime receipts use =defstruct= with fixed fields and accessors.
3. External payloads use explicit =receipt->alist= projection and never leak internal struct layout as a public serialization ABI.

Example:
#+begin_src scheme
(defstruct loop-engine-capability-receipt
  (backend backend-kind backend-capabilities supported-backends valid?
   diagnostics isolation required optional unsupported-behavior
   sandbox-ref session-ref)
  transparent: #t)

(def (loop-engine-capability-receipt->alist receipt)
  (list
   (cons 'kind 'capability-receipt)
   (cons 'backend (loop-engine-capability-receipt-backend receipt))
   (cons 'valid? (loop-engine-capability-receipt-valid? receipt))
   (cons 'diagnostic-count
         (length (loop-engine-capability-receipt-diagnostics receipt)))))
#+end_src
** 10.06.29.90 Native POO Scenario Lessons
The Gerbil Scheme harness scenarios make the performance boundary more concrete:
1. Public APIs stay native POO: prototypes, mixins, slot operators, and functional helpers remain the user-visible model.
2. Adapter construction such as =object<-alist= belongs at named boundaries. Do not rebuild adapter objects inside validation loops or merge loops.
3. Repeated membership tests over slot lists should use a local hash index when the operation is part of registry, profile, or policy validation.
4. Repeated =append=, =filter=, or duplicate scans in extension code are a warning sign. Prefer one pass to index extension values, one pass to preserve untouched base values, and one pass to materialize final values in order.
5. Duplicate diagnostics should keep a diagnostic index so the validator reports each duplicate key once without repeatedly walking accumulated diagnostics.
6. Object reads should be hoisted to the owner boundary. A loop should receive the scalar slot values it needs instead of repeatedly traversing child POO objects.
7. POO composition such as =.mix=, =.extend=, clone overrides, or lens-style mutation should not appear inside tight loops unless the loop's purpose is explicitly to construct those objects.

This is not a license to replace POO with raw data. The intended shape is POO-native at the extension boundary, functional and indexed in the hot implementation path, then POO receipts or projections at the next boundary.
5. If a test hangs, inspect for slot self-reference and repeated nested object instantiation before changing the runtime.

** 10.06.29.90 Test Policy

Every new POO-native surface needs tests at three levels:

1. Prototype construction: the object can be built directly.
2. Composition behavior: supers, defaults, ==>=, and ==>.+= resolve as expected.
3. Projection boundary: receipts, presentations, or ABI payloads stay report-only and do not execute runtime work.

Tests should include at least one invalid or cyclic case when the surface accepts user extension. Prefer package-context =gxtest= runs for module-system changes because load path and policy harness behavior are part of the contract.

** 10.06.29.100 MOP Boundary Policy

=doc/TODO.md= is clear that a compositional MOP and method-combination are future design directions. POO Flow should benefit from the existing POO object model without requiring downstream users to understand or implement a MOP.

Rules:

1. Do not expose MOP terms such as method-combination, primary/before/after/around methods, storage strategy, or custom linearizer as normal user configuration.
2. Do not implement a local method-combination layer to solve ordinary resource, sandbox, profile, or workflow composition.
3. Use stable POO slots and named helper constructors until upstream POO provides a settled MOP abstraction.
4. If a future feature truly needs MOP behavior, write it as an internal module boundary with tests and a report-only presentation first.
5. Keep user documentation phrased in prototype, slot, and functional-composition terms.

This prevents POO Flow from inventing a second object system while still leaving room to adopt upstream MOP work later.

** 10.06.29.110 Syntax And Macro Policy

=brace.ss= shows that syntax should lower to the same object core, not define a new semantic language. POO Flow macros should therefore be hygienic authoring conveniences over POO objects and functional combinators.

Allowed:

1. A macro that binds a named flow to =flow-arr=, composition, fanout, or Kleisli helpers.
2. A macro that constructs a module row and lowers it to upstream POO object constructors.
3. A macro that gives a short form for a stable domain object.

Not allowed:

1. A macro that loads files, executes runtime work, or mutates global registries as part of expansion.
2. A macro that hides an alist/record DSL behind Scheme syntax.
3. A macro that bypasses POO validation gates.

When in doubt, expand to an ordinary binding whose value can be inspected and tested as a POO object or flow value.

** 10.06.29.120 Type Object And Interface-Passing Policy

=doc/prototypes.md= describes typeclasses as prototypes for type descriptors and names Interface-Passing Style as a core use. =type.ss= and =fun.ss= make that concrete: type descriptors, Category, Functor, Identity, wrapper behavior, and binding/mapping behavior are objects.

Rules for POO Flow:

1. Model strategy dictionaries, flow categories, sandbox capability sets, and adapter contracts as objects or functional values, not as class hierarchies.
2. Pass behavior explicitly through slots or constructor arguments; do not rely on ambient global dispatch.
3. Prefer small dictionaries with named slots over large records with private interpretation rules.
4. Keep parametric behavior visible: if a helper maps or binds over a flow, name the category/functor object that supplies the operation.
5. Use plain alists only after the object has been projected to a receipt, manifest, or ABI payload.

This keeps polymorphism inspectable and fits Gerbil Scheme's strengths: hygienic macros, lexical binding, and explicit object composition.

** 10.06.29.130 Mutation And Cache Policy

POO supports mutation helpers such as =.put!= and =.putslot!=, but the implementation caches instantiated slot values. Mutating a prototype after consumers have instantiated it creates stale-instance risks and makes validation order matter.

Project rules:

1. Treat upstream profile, sandbox, workflow, and strategy prototypes as immutable after export.
2. Prefer constructing a new prototype with =.mix=, =.extend=, =.+=, ==>=, or ==>.+= over mutating an existing prototype.
3. Reserve =.put!= and =.putslot!= for tests, REPL exploration, or tightly-scoped internal builders that do not leak partially-instantiated objects.
4. Do not use mutation to implement downstream customization.
5. If mutation is unavoidable, add a validation receipt that proves no exported object was already instantiated.

For normal code review, a new =.putslot!= call should be treated as a design smell until proven otherwise.

** 10.06.29.140 Diagnostics And Presentation Policy

Because POO makes recursive extension easy, every user-facing POO surface needs an equally explicit presentation surface.

Presentations should show:

1. object identity;
2. direct supers and resolved precedence chain;
3. direct slots owned by the object;
4. inherited slots used by the object;
5. slot operators used: direct, ==>=, ==>.+=, =?=, raw computed;
6. whether any mutable operation was used;
7. projection boundary: POO object, plain receipt, runtime manifest, or ABI payload;
8. =runtime-executed = #f= unless the test explicitly runs a local validation runner.

Doctor output should prefer these facts over dumping nested objects. The goal is to make C3 and slot resolution inspectable without forcing users to learn the implementation layout.

** 10.06.29.142 Observability Contract Policy
Object observability is a validation contract, not a POO slot. Do not add debug
slots, debug mixins, or per-=objects.ss= debug variables just to inspect module
object catalogs.

The normal path is:
1. =objects.ss= declares POO module objects only.
2. =object-validation.ss= reads the object through the existing validation
   boundary.
3. The validation receipt exposes direct fields, resolved fields, field origins,
   inheritance chains, validation phases, runtime flags, and diagnostics.
4. Doctor and presentation code consume the plain receipt projection.

This keeps observability outside POO lazy slot resolution while still making C3
composition inspectable. If a debug feature needs to dereference objects, place
it behind a validation or doctor contract and return plain receipt data.
** 10.06.29.145 Executable Guard Landing

The first executable guard for these rules is now in =src/modules/sandbox-core/profile.ss=:

1. =poo-flow-sandbox-profile-object-row-authoring-diagnostics= reports row-level authoring facts.
2. =poo-flow-sandbox-profile-object-authoring-diagnostics= batches those facts for profile forms.
3. =advanced-row-operator= marks =:append=, =:prepend=, =:remove=, and =:override= as advanced escape hatches.
4. =raw-compute-hook= marks =:compute=, =$computed-slot-spec=, or literal =lambda= in profile rows as non-primary user interface shapes.
5. =runtime-executed-marker= catches user metadata that tries to mark Scheme profile authoring as runtime execution.

The proof lives in =t/sandbox-core-profile-authoring-diagnostics-test.ss=. It keeps clean POO-native rows diagnostic-free and reports the advanced/non-native shapes above. The guard is report-only: it does not change merge behavior or execute runtime work.

** 10.06.29.150 Anti-Pattern Matrix

| Anti-pattern | Why it is wrong | Preferred shape |
| Raw alist/record DSL | Reimplements inheritance outside POO | Prototype slots plus projection at the edge |
| User-authored =(lambda (self super) ...)= | Exposes low-level slot machinery | Named helper using ==>= or ==>.+= |
| Repeated nested child =.ref= reads | Re-instantiates and hides hot paths | Compose mixins first, read direct slots |
| Root =user-interface/objects.ss= facade | Duplicates upstream contracts | Keep implementations under =src/modules= |
| Custom method-combination layer | Premature MOP fork | Stable slots and helper constructors |
| Runtime work in Scheme config | Breaks control-plane/runtime boundary | Report-only request or Marlin handoff |
| Mutation-based customization | Cache and stale-instance risk | Build a new prototype with supers |

** 10.06.29.160 Rule: Model Extensions as Prototypes

Prefer POO prototypes and mixins over nested record trees.

Good:

#+begin_src scheme
(.o (:: @ [lineage placement])
    kind: 'poo-flow.session.value
    session-id: 'root
    chunks: chunks)
#+end_src

Bad:

#+begin_src scheme
(.o kind: 'poo-flow.session.value
    lineage: lineage-object
    placement: placement-object)
#+end_src

The bad shape makes callers repeatedly =.ref= child objects. Each =.ref= may
instantiate that child object's fixed point. The good shape lets the composed
session expose inherited slots directly.

** 10.06.29.170 Rule: Use Slots as the User Contract

Public object APIs should expose stable slot names. If a concept is meant to be
extended, make it a slot on a prototype:

- =parent-session-ids= instead of a hidden lineage child object;
- =placement-profile-ref= instead of a nested placement object;
- =runtime-executed= as a direct receipt slot;
- =metadata= or scoped metadata slots for report-only facts.

Accessor helpers are allowed, but they should be thin wrappers around direct
slot reads on the composed object.

** 10.06.29.180 Rule: Avoid Hot-Path Child =.ref=

Do not build presentation or validation loops that repeatedly call =.ref= on
nested POO subobjects. That pattern magnifies fixed-point costs and can look
like a hang when the object graph grows.

Preferred patterns:

- compose mixins first, then read direct slots on the final object;
- compute lists of primitive receipt rows before inserting them as slots;
- use report-only alists for external ABI receipts when they are not intended
  to participate in POO inheritance;
- keep expensive runtime data outside Scheme and expose only handoff summaries.

** 10.06.29.190 Rule: Use Multiple Inheritance Explicitly

When a value combines independent policy surfaces, use C3 composition rather
than copying fields:

#+begin_src scheme
(.o (:: @ [session-lineage
          sandbox-placement
          governance-policy])
    kind: 'poo-flow.session.value)
#+end_src

This is the core advantage over a plain DSL: a downstream user can extend,
override, or remove behavior by composing prototypes instead of rewriting
configuration records.

** 10.06.29.200 Rule: Keep Runtime Receipts Report-Only

Scheme-side POO objects may describe runtime intent, but they must not imply
execution. Use explicit slots:

- =runtime-owner=;
- =handoff-required=;
- =runtime-executed=;
- =runtime-parses-scheme-source=;
- =scheme-manufactures-runtime-handlers=.

If a runtime backend needs heavy data, put the heavy data behind Marlin or
another runtime-owned handoff. Scheme should publish compact receipts and
agreement diagnostics.

** 10.06.29.210 Rule: Tests Must Catch Expensive POO Shapes

Focused tests should include:

- direct construction of each mixin prototype;
- construction of the final composed object;
- presentation reads of direct slots;
- one cyclic or invalid graph case as structured validation data;
- a package-context =gxtest= run for each new module.

If a focused test hangs while reading a POO object, first check whether the code
is repeatedly =.ref= reading child POO objects instead of composing their slots.

** 10.06.29.220 Rule: Avoid Slot Self-Reference

Inside an =.o= slot expression, a bare identifier may resolve as a slot
reference in the fixed-point environment. Do not bind constructor parameters
with the same name as the slot being defined.

Bad:

#+begin_src scheme
(def (make-session session-id chunks)
  (.o session-id: session-id
      chunks: chunks))
#+end_src

The slot =chunks= can become self-referential and hang when read.

Good:

#+begin_src scheme
(def (make-session session-id chunks)
  (let ((session-id-value session-id)
        (chunk-values chunks))
    (.o session-id: session-id-value
        chunks: chunk-values)))
#+end_src

Use suffixed local names such as =-value=, =-values=, or =-prototype= when
feeding data into slots.

** 10.06.29.230 Session Object Guideline

For OpenRath-inspired session work, =lineage= and =placement= should be POO
mixins. The session object should inherit them:

#+begin_src scheme
(def lineage
  (.o lineage-kind: 'poo-flow.session.lineage
      parent-session-ids: '(root)
      branch-kind: 'fork))

(def placement
  (.o placement-kind: 'poo-flow.session.placement
      placement-profile-ref: 'agent/session))

(def session
  (.o (:: @ [lineage placement])
      kind: 'poo-flow.session.value
      session-id: 'child
      chunks: chunks
      runtime-executed: #f))
#+end_src

Do not store =lineage= and =placement= as opaque child objects unless the caller
will never inspect them in validation or presentation loops.
