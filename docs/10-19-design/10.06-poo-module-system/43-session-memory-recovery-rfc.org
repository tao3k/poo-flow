* 10.06.43 Session Memory Durable Policy RFC
** 10.06.43.10 Status
Draft for discussion.

This RFC pauses implementation and defines the architecture we should agree on
before adding more Scheme code. The target is not "copy Codex" or "copy
OpenRath". The target is a POO Flow design that uses their good evidence while
correcting their weak spots for a stronger agent-flow system.

** 10.06.43.20 Evidence
*** 10.06.43.20.10 OpenRath Evidence
OpenRath's useful idea is session-first composition:
- A session is the flowing value.
- An agent transforms one session into another session.
- Workflow composition is session graph composition.
- Parent and child agent history is represented as lineage, not as anonymous
  hidden message arrays.

The useful part for POO Flow is not OpenRath's Python API. The useful part is
the shape:
#+begin_src text
Session -> Agent/Workflow -> Session
#+end_src

This aligns with our functional + POO direction. A flow can be Arrow/Kleisli-
like at the composition level while each session, policy, and profile remains a
POO object with C3 inheritance and native extension.

OpenRath's weak point is that it does not go far enough on policy. A child
agent can be session-addressable, but the authorization surface needs to be
more precise than "the process has a tool". POO Flow should treat session ids
as policy anchors.

*** 10.06.43.20.20 Codex Evidence
Codex's useful idea is persistent recovery through layered state:
- rollout JSONL is an append-only session fact log;
- SQLite state is a derived index and job state;
- startup backfill reconstructs state DB rows from rollout files;
- DB corruption recovery backs up only the broken runtime DB and sidecars;
- memory is an independent job pipeline with leases, retries, watermarks, and
  usage tracking;
- thread spawn edges persist parent/child thread relationships;
- trace reduction models root and child threads as typed runtime objects with
  execution windows and terminal status.

Useful evidence paths in =.data/codex=:
- =codex-rs/rollout/src/recorder.rs= for append-only rollout recording,
  resume, persist, flush, and history loading.
- =codex-rs/state/migrations/0001_threads.sql= for thread metadata index.
- =codex-rs/state/migrations/0021_thread_spawn_edges.sql= for parent/child
  thread edges.
- =codex-rs/state/src/runtime/recovery.rs= for DB backup and fresh-start
  recovery.
- =codex-rs/state/src/runtime/memories.rs= for memory job leases, stage-1
  extraction, and global consolidation.
- =codex-rs/rollout-trace/src/model/session.rs= for typed thread/session
  runtime objects.

The lesson is not "Scheme should implement SQLite". The lesson is that durable
agent systems need a fact log, derived indexes, resumable jobs, and explicit
recovery policy.

** 10.06.43.30 Design Thesis
POO Flow should treat durable recovery as a shared checkpoint capability, not
as a standalone user-facing recovery object.

The user-facing object is a =durable-policy= or =checkpoint-policy=. Session,
memory, loop, and workflow policies can inherit or reference it. Recovery is the
runtime procedure that consumes durable facts, checkpoints, journals, and
derived indexes.

POO Flow should split this durable capability into three layers.

1. User authoring layer:
   Native POO objects declare session, memory, lineage, durable checkpoint,
   context, communication, sandbox, and resource policy. Users should write
   small =use-module= and =.def= forms, not ad hoc alists.

2. Scheme runtime projection layer:
   Scheme lowers POO objects into bounded receipts and manifests. For hot
   generated receipts, Gerbil =defstruct= is preferred internally, with one
   explicit =->alist= projection at ABI boundaries.

3. Rust/Marlin execution layer:
   Rust/Marlin owns JSONL/SQLite/storage, checkpoint materialization, leases,
   recovery, backfill, memory extraction, compaction, sandbox handles,
   subprocesses, and cross-session message delivery.

This keeps Gerbil doing what it is good at: high-level strategy, POO-native
composition, contracts, and handoff shapes. It keeps Rust doing what it is good
at: durable storage, concurrent runtime, IO, and recovery machinery.

** 10.06.43.40 Core Objects
The durable architecture should be object-centric, not option-centric.

Objects are the configurable units:
- =session-object=: stable id, project id, parent ids, role, placement, and
  lifecycle state.
- =session-lineage-object=: parent/child/session graph edges.
- =session-context-policy-object=: which chunks, summaries, and receipts are
  visible to a child session.
- =session-history-policy-object=: bounded historical input rules.
- =session-communication-policy-object=: which session ids can address which
  targets and through which channel.
- =session-sandbox-policy-object=: inherit, clone, replace, or isolate sandbox
  placement.
- =session-memory-policy-object=: recall, commit, retention, ranking, and
  memory backend contract.
- =durable-policy-object=: shared capability that declares fact journal,
  checkpoint cadence, derived index, watermarks, repair mode, and runtime
  owner.
- =checkpoint-policy-object=: bounded durable-point policy used by the durable
  policy. It says what is captured, when, and under what resume modes.

The important constraint: these are POO objects. They can use C3 multiple
inheritance, local override, and extension. They should not become a Nix-style
option tree because Scheme + POO already gives us a stronger object kernel.

** 10.06.43.45 Durable Policy As Shared Capability
The durable policy should be a reusable POO prototype. It is not owned only by
session and not owned only by memory.

Conceptually:
#+begin_src scheme
(.def (durable/default @ durable-policy
                       fact-log derived-index checkpoint
                       resume repair bounds runtime-owner)
  ...)

(.def (session/project @ session-policy durable/default
                       context history communication sandbox)
  ...)

(.def (memory/project @ memory-policy durable/default
                      recall commit ranking retention)
  ...)
#+end_src

The exact syntax is not decided, but the inheritance direction is important:
- =session-policy= can inherit durable behavior because sessions need journal,
  checkpoint, and resume semantics.
- =memory-policy= can inherit the same durable behavior because memory recall,
  extraction, and consolidation need watermarks and retryable job state.
- =loop-engine= can consume the resulting policy through receipt projection.
- Runtime recovery is not a separate user object. It is derived from durable
  policy plus checkpoint facts.

This is the POO-native advantage over a Nix-style option tree. A profile can
define =durable/default= once, then specialize session and memory policies with
C3 inheritance, local overrides, and extension.

** 10.06.43.50 Identity Model
Session ids are routing and policy identities, not display labels.

Required identity axes:
- =project-id=: workspace or long-lived project boundary;
- =root-session-id=: one top-level user/task session inside the project;
- =child-session-id=: one sub-agent session under a root or another child;
- =agent-id=: logical agent/persona/tool-runner bound to a session;
- =plan-id=: plan/checklist identity for resumable work;
- =loop-run-id=: one concrete runtime loop activation;
- =checkpoint-id=: one durable recovery point;
- =channel-id=: optional communication edge between sessions.

Parent/child and peer routing rules:
- a child session must know its parent session id;
- siblings communicate only through declared channels;
- cross-root communication must include project id and target root session id;
- display names and nicknames are never routing identities;
- write-back to parent is a receipt/message, not direct parent mutation.

** 10.06.43.60 Persistence Model
POO Flow should adopt Codex's layered persistence shape, but as a policy and
handoff architecture.

Proposed durable layers:
1. Fact journal:
   Append-only JSONL or equivalent. This is the source of truth for session
   events, loop receipts, sub-agent starts/stops, checkpoints, and handoff
   payloads.

2. Derived index:
   Query-optimized state DB. This can be rebuilt from the fact journal. It
   stores session metadata, lineage edges, active/archived state, preview rows,
   current checkpoints, and job status.

3. Memory jobs:
   Independent job tables or backend equivalents. Memory extraction and
   consolidation are not inline session mutation. They are claimable jobs with
   lease, retry, owner, watermark, and stale-source checks.

4. Runtime artifacts:
   Sandbox handles, worktree copies, logs, build outputs, and external runtime
   state. These are referenced by receipt ids and artifact handles, not stored
   as Scheme object graphs.

5. Recovery diagnostics:
   Startup and doctor checks that compare journal facts, derived indexes,
   checkpoints, and artifact references.

** 10.06.43.70 Checkpoint Contract
A checkpoint is the concrete durable-point fact emitted under a durable policy.
It should not serialize the entire POO object graph.

Minimum checkpoint fields:
- =checkpoint-id=
- =project-id=
- =root-session-id=
- =session-id=
- =parent-session-ids=
- =plan-id=
- =loop-run-id=
- =sequence=
- =created-at=
- =journal-ref=
- =state-watermark=
- =memory-watermark=
- =artifact-refs=
- =sandbox-ref=
- =resume-mode=
- =resume-owner=
- =diagnostics=

Resume modes should be explicit:
- =resume-from-last-checkpoint=
- =resume-from-journal-watermark=
- =rebuild-derived-state=
- =fork-from-checkpoint=
- =manual-review-required=

Scheme declares these policies and emits receipts. Rust/Marlin decides how to
load bytes, rebuild indexes, restart loops, or ask for review.

Recovery policy should therefore be a slot or facet of durable policy:
- =resume-mode= says how a checkpoint can be used;
- =repair-mode= says how derived state can be rebuilt;
- =corruption-mode= says how broken state is isolated or backed up;
- =runtime-owner= says which runtime implements the procedure.

The runtime executes recovery. The Scheme layer declares durable capability and
projects checkpoint/recovery receipts.

** 10.06.43.80 Memory Policy
Current loop-engine memory policy is too shallow if it only says recall/commit.
It should declare enough durable behavior for the runtime to recover correctly.

Additional policy concepts to discuss:
- =durable=: durable policy object or ref inherited by memory policy;
- =fact-log=: source journal family for memory-relevant events;
- =derived-index=: query/index backend to rebuild;
- =checkpoint=: checkpoint policy facet inherited from durable policy;
- =repair-mode=: derived-state rebuild behavior inherited from durable policy;
- =job-kind=: memory job family, such as stage-1 extraction or consolidation;
- =lease-seconds=: runtime lease contract;
- =retry-limit=: bounded retry policy;
- =source-watermark=: what input watermark makes memory stale;
- =selection-window=: bounded history window for recall;
- =max-context-items=: hard cap for injected memory;
- =max-context-tokens=: hard cap for model-visible memory.

This follows Codex's good practice: no unbounded context injection, no inline
memory mutation, and no hidden global memory job without ownership and leases.

The key design correction is that memory does not own its own unrelated
recovery model. Memory participates in the same durable policy system as
session. Its job-specific slots add extraction and consolidation semantics.

** 10.06.43.90 Session Policy
Session policy should be fine-grained and default conservative.

Policy families:
- isolation: sibling isolation, parent visibility, child write-back;
- sandbox: inherit, clone, replace, shared, or isolated sandbox placement;
- context: none, parent summary, selected chunks, selected sibling outputs,
  selected receipts, or full inherited context;
- history: last turn, last failure, selected rollout slice, selected receipts,
  or bounded transcript;
- communication: source session ids, target session ids, channels, payload
  shape, and delivery policy;
- resources: budgets, tool grants, file scopes, network scopes, and collision
  classes;
- sharing: shared memory, shared artifacts, shared tool results, shared
  workspace paths.

Default policy:
- child sessions do not read sibling context;
- child sessions do not share sandbox handles;
- child sessions may inherit a parent summary, not full transcript;
- cross-root communication is denied unless declared;
- all model-visible context is bounded.

** 10.06.43.100 Module-System Question
There is an unresolved module-system question:

Should =session-core= become a first-class init category row, for example:
#+begin_src scheme
(poo-flow!
 :session
 (session-core +lineage +placement +handoff +graph +transform +doctor
               +persistence +checkpoint +resume +memory-policy)
 :workflow
 (funflow ...)
 (loop-engine)
 :sandbox
 (nono-sandbox +nono +doctor))
#+end_src

Arguments for =:session=:
- session is not workflow, sandbox, loop, or custom;
- session is the flowing value across workflows and agents;
- persistent memory and recovery are core context concerns;
- user-facing init becomes closer to Doom's category/module shape.

Arguments against adding it immediately:
- current init categories do not include =:session=;
- =session-core= may be a module selected through profile/kernel bundles rather
  than a root user category;
- adding a category is a public interface change and needs tests/docs first.

RFC recommendation:
- Do not land =:session= until this RFC is accepted.
- Treat =:session= as likely correct, but confirm naming and migration path.
- If accepted, add it with contract tests and update user-interface examples in
  the same milestone.

** 10.06.43.110 Runtime Boundary
Scheme must not:
- open SQLite databases;
- own JSONL appenders;
- run background recovery jobs;
- summarize transcripts;
- mutate durable memory;
- restart sandbox processes;
- infer sibling communication from ids alone.

Scheme should:
- declare policy as POO objects;
- validate object shape and contract boundaries;
- project bounded receipts;
- expose doctor/presentation rows;
- carry runtime-owner fields;
- make unsupported runtime behavior explicit.

Rust/Marlin should:
- persist journals;
- maintain derived indexes;
- run backfill and rebuild;
- claim and retry jobs;
- enforce leases;
- execute resume/fork/recovery;
- enforce communication and sandbox policy at runtime.

** 10.06.43.120 Implementation Ladder
Milestone 1: RFC and contracts.
- Finalize session identity vocabulary.
- Decide whether =:session= is a root init category.
- Define shared =durable-policy= and =checkpoint-policy= POO objects.
- Define receipt contracts without backend execution.

Milestone 2: Scheme declaration layer.
- Add POO prototypes for durable/checkpoint/session policy.
- Extend session-core and loop-engine policy rows.
- Project durable/checkpoint receipts into presentation and runtime handoff.
- Add negative contract tests for unbounded context and invalid ids.

Milestone 3: User interface.
- Update =user-interface/init.ss= only after the category decision.
- Add user profile examples that inherit recovery policy objects.
- Keep examples small and Doom-like.

Milestone 4: Runtime handoff.
- Add Marlin handoff ABI fields for durable policy, journal, state index,
  checkpoint, repair mode, resume mode, and memory job policy.
- Keep Scheme receipts bounded.

Milestone 5: Rust/Marlin implementation.
- Implement fact journal, derived state, checkpoint, backfill, and recovery.
- Implement memory jobs with leases and watermarks.
- Implement doctor parity checks between journal, state DB, and checkpoint.

** 10.06.43.130 Open Questions
1. Should the public init category be =:session=, =:context=, or something
   else?
2. Should durable/checkpoint policy live under =session-core=, or under a
   shared module-system object family consumed by session, memory, loop-engine,
   and workflow modules?
3. Should loop-engine memory policy reference durable policy by object ref, or
   inherit it directly through POO supers?
4. What is the minimum runtime ABI that Marlin needs for first recovery support?
5. Which context bounds are hard policy: item count, token count, byte count,
   or all three?
6. Should cross-session communication be a first milestone feature or only
   represented as receipts until runtime support exists?

** 10.06.43.140 Proposed Direction
Use POO objects as the policy kernel, use durable/checkpoint policy as the
shared capability, and use Codex-style persistence as the runtime model.

That means:
- session is the flowing value;
- durable policy is the shared checkpoint/recovery capability;
- POO objects describe policy, inheritance, and extension;
- functional flow composes session transforms;
- receipts are bounded ABI projections;
- Rust/Marlin owns durable execution;
- recovery is runtime behavior derived from journal + derived index +
  checkpoint + jobs;
- every child agent is a session-addressable participant;
- policy decides what each session id can see, touch, share, and resume.
