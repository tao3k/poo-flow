* 10.06.10.01 Purpose

The user interface is the downstream-facing module selection surface. It is not
a second implementation of the module system. Root-level user files declare:

- selected module group/name pairs;
- flags that describe hot-plug intent;
- custom module entrypoints owned by the downstream repository.

Upstream-owned modules then project those declarations into profile, config,
doctor, and presentation data. Root user files do not own settings projection,
object contracts, descriptor realization, package sync, loaders, or runtime
execution.

** 10.06.10.05 Upstream Kernel Profile Layout

The user-interface layer mirrors Doom's module-selection block:

- =src/module-system/profiles/kernel.ss= composes the built-in kernel profile.
- =src/modules/funflow/config.ss= owns the Funflow module row.
- =src/modules/loop-governor/config.ss= owns the loop governor row.
- =src/modules/nono-sandbox/config.ss= owns the nono sandbox row.
- =src/modules/cubeSandbox/config.ss= owns the CubeSandbox row.
- =src/module-system/root-objects.ss= owns user-interface object contracts.
- =src/module-system/root-profile.ss= owns root declaration projection helpers.

=user-interface/init.ss= exports category/module/feature bundle rows. Upstream
=src/module-system/root-profile.ss= extends =poo-flow-kernel-profile= with
those rows. Core workflow, loop, and sandbox rows stay upstream-owned in profile
config files; users should not have to repeat them just to get a working
profile.

Downstream extension happens in =user-interface/init.ss= for optional feature
rows and under =user-interface/custom= for custom module bodies. Direct
=(use-module ...)= helper rows belong to config/helper code, not the root init
block.

** 10.06.10.06 Doom-Style Profile Set Contract

POO Flow borrows Doom's profile registry shape without copying package
management:

- =poo-flow-profile-set= declares a named registry, a default profile, and profile values.
- =poo-flow-init-module-bundles= is the normal root init entrypoint and creates inspectable bundle rows from category/module/feature declarations.
- =pooFlowRootProfile= turns root bundle rows into inspectable profile data.
- =pooFlowRootProfileSet= builds the canonical root profile registry.
- =pooFlowRootConfig= projects root bundle rows into config data.
- =pooFlowRootProfileSetPresentation= exposes selected/default profile metadata for inspection.
- =pooFlowRootProfileSetDoctorPresentation= reports duplicate profile names, empty registries, and missing defaults before activation.

=src/module-system/profile-config.ss= owns profile/config data APIs.
=src/module-system/root-profile.ss= owns the root user-interface projection
helpers. =src/module-system/profiles/kernel.ss= owns the kernel profile set.

Root =user-interface/init.ss= should stay close to Doom's =init.el= ergonomics:
the visible block groups only optional patches under category markers. Required
module capabilities live in profiles, not in the user file. It must not contain
package declarations, runtime sync commands, loader internals, descriptor
constructors, settings projection, object contracts, or manual profile
constructors.

* 10.06.10.02 Root Template

The maintained practice template lives at:

- =user-interface/init.ss=
- =user-interface/custom/my-module/config.ss=

The root template must stay small. Users list only optional
category/module/feature patches in =init.ss= and place custom module bodies
under =custom=. Root =user-interface= must not contain implementation facade
files such as =objects.ss=, =config.ss=, or =modules/config.ss=.

Its normal form is a Doom-like =poo-flow-init-module-bundles= declaration using
=:category= markers and =+feature= flags for things that can be enabled,
disabled, or customized. Presentations such as =pooFlowUserConfigPresentation=
and =pooFlowUserProfileSetDoctorPresentation= are projected by upstream helpers.
Ordinary users should not have to call =pooFlowUserConfig=,
=pooFlowUserProfile=, descriptor constructors, loader backend constructors,
merge internals, or runtime adapters.

* 10.06.10.03 Presentation Contract

=pooFlowUserConfigPresentation= exposes:

- selected module keys and flags;
- selected setting keys and values;
- public entrypoints;
- ownership boundaries;
- =descriptor-realized? = #f=;
- =runtime-executed = #f=;
- =runtime-owner = "marlin-agent-core"=.

This mirrors Doom's inspectable user ergonomics without importing Doom's package
manager surface. The implementation remains Gerbil-native and data-first.

* 10.06.10.04 Test Gate

The user contract is guarded by:

- =t/user-interface-config-test.ss=
- =t/declaration-case-test.ss=
- =t/module-system-user-interface-test.ss=

The root-facing tests import =:poo-flow/user-interface/init= for declarations
and project them through =:poo-flow/src/module-system=. They must not depend
on a root =:poo-flow/user-interface/config= facade.
