#+title: Proof Gate Receipt Integration

* Boundary

The proof gate no longer starts from hand-written boolean fixtures. Scheme now
accepts structured receipts from composition, scenario, and handoff producers,
then converts them into proof facts, FFI wires, and a gate bundle.

Runtime execution remains outside this project.

* Receipt Entry

The executable entrypoint is:

#+begin_src scheme
(poo-flow-proof-gate-receipts->bundle receipts)
#+end_src

The receipt shape is:

#+begin_src scheme
((schema . poo-flow.user-interface.langgraph.proof-receipts)
 (composition . ...)
 (scenario . ...)
 (handoff . ...))
#+end_src

Each sub-receipt is converted through the existing proof facts layer:

- =poo-flow-composition-receipt->proof-facts=
- =poo-flow-scenario-gap-receipt->proof-facts=
- =poo-flow-control-plane-handoff-receipt->proof-facts=

* LangGraph-Like UI Gate

=poo-flow-langgraph-user-interface-proof-gate-bundle= is the first
receipt-driven gate for a LangGraph-like scenario. It covers:

- accepted profile composition;
- accepted graph/scenario capability row;
- external runtime ownership;
- deferred execution;
- declared artifacts.

The rejection matrix includes:

- bad composition override;
- missing scenario capability;
- local runtime ownership.

* FFI Contract

Receipts do not cross the FFI boundary directly. They are normalized into proof
facts, wrapped by =poo-flow-proof-facts->ffi-wire=, and then grouped by
=poo-flow-proof-gate-bundle=.

This keeps the bridge contract stable:

#+begin_src latex
Receipt \to ProofFacts \to FFIWire \to GateBundle
#+end_src

* P0 External Blocker

The ASP lifecycle audit issue is outside POO Flow. The root rollout exists, but
=asp agent session lifecycle audit= reports =scannedRolloutCount = 0=. That
means ASP lifecycle is not resolving the root rollout from =CODEX_THREAD_ID=.

POO Flow should not work around this by reading raw source. The correct fix
belongs in =agent-semantic-protocols= lifecycle audit.
