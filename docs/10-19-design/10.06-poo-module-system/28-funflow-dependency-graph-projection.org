#+TITLE: 10.06.28 Funflow Dependency Graph Projection

** 10.06.28.10 Question
The current Funflow alignment gap is dependency shape, not execution. A user
can write a compact pipeline with =:needs=, but the workflow layer must expose
that dependency as inspectable POO control-plane data before any runtime owner
or provider adapter is allowed to schedule work.

** 10.06.28.20 Implemented Shape
Funflow =:needs= lowers into workflow-owned CI/CD dependency facts:

- =src/modules/funflow/config.ss= accepts a single symbol or a list of symbols
  for =:needs=.
- Non-symbol dependency refs are rejected at the Funflow config boundary.
- =src/modules/workflow/cicd.ss= exposes
  =poo-flow-cicd-check-dependency-refs= and
  =poo-flow-cicd-check-map->dependency-graph=.
- The dependency graph projects =nodes=, =duplicate-nodes=, =edges=,
  =unresolved-dependency-refs=, =cycle-nodes=, =order-policy=,
  =ready-order=, =unordered-nodes=, =blocked-order?=, =diagnostics=,
  =valid?=, and =runtime-executed = #f=.
- Receipts, runtime-manifest readiness rows, runtime command request payloads,
  manifest metadata, and user-facing manifest summaries preserve
  =dependency-refs=.

** 10.06.28.30 Boundary
This is still Scheme control-plane work. The graph is declarative data:

- no process is launched;
- no scheduler is introduced;
- no GitHub/GitLab/Nix/Docker provider semantics are added;
- heavy execution remains owned by Rust/Marlin or later backend modules.

** 10.06.28.40 User-Facing Result
The compact shape now has a concrete projection:

#+begin_src scheme
(use-module funflow
  :config
  (.def (funflow/build @ funflow-check
                       check-name profile-ref command-vector
                       result-protocol runtime-mode)
    check-name: 'build
    profile-ref: 'ci/build
    command-vector: '("gxpkg" "build")
    result-protocol: '(read :lines)
    runtime-mode: 'manifest-handoff)
  (.def (funflow/test @ funflow-check
                      check-name profile-ref command-vector
                      dependency-refs result-protocol runtime-mode)
    check-name: 'test
    profile-ref: 'ci/check
    command-vector: '("gxtest" "t/unit-tests.ss")
    dependency-refs: '(build)
    result-protocol: '(read :lines)
    runtime-mode: 'manifest-handoff)
  (.def (funflow/default @ funflow-pipeline pipeline-name checks)
    pipeline-name: 'default
    checks: (list funflow/build funflow/test)))
#+end_src

The resulting dependency graph contains:

#+begin_src scheme
((nodes . (build test))
 (duplicate-nodes . ())
 (edges . (((from . build) (to . test))))
 (unresolved-dependency-refs . ())
 (cycle-nodes . ())
 (order-policy . declaration-topological-report)
 (ready-order . (build test))
 (unordered-nodes . ())
 (blocked-order? . #f)
 (diagnostics . ())
 (valid? . #t)
 (runtime-executed . #f))
#+end_src

** 10.06.28.50 Topology Validation Update
The graph now validates topology before runtime handoff:

- duplicate check names are reported as =duplicate-nodes=;
- dependency refs that remain unresolved are reported as
  =unresolved-dependency-refs=;
- cycles are reported as =cycle-nodes=;
- =diagnostics= records the failing topology classes;
- =valid?= is true only when all topology diagnostics are empty.

** 10.06.28.60 Ordering Policy Update
The graph now reports ordering facts without scheduling:

- =order-policy= is =declaration-topological-report=.
- =ready-order= is the dependency-ready node order found by fixed-point scan.
- =unordered-nodes= lists nodes that could not enter the report order.
- =blocked-order?= is true when topology diagnostics exist or not all nodes can
  be ordered.

This keeps Scheme on the control-plane side. Backends can consume the report,
but task execution and parallel scheduling remain runtime-owned.
