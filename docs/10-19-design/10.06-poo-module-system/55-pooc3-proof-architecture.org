#+TITLE: 10.06.55 PooC3 Proof Architecture
#+OPTIONS: toc:2

* 10.06.55.10 Purpose
This document is the reader-facing architecture for the POO/C3 linearization proof layer. It is not only a list of Lean files. It explains the theory, mathematical model, design intent, and expected use by humans and agents.

PooC3 means the proof-facing view of POO objects after C3 linearization. In other words, it is POO/C3 linearization plus the control-plane proof obligations that must hold for the linearized result. It starts from the practical problem in POO flow: policy objects, strategy objects, shared profiles, sessions, sandboxes, subagents, and runtime handoff can be freely composed. That freedom is the value of POO, but it also creates hidden correctness risks:

- a module may start before its real dependencies are complete;
- a selector override may broaden a policy outside its parent scope;
- a shared profile may be reused in a context where its sandbox is too wide;
- a subagent may receive a message without lifecycle, channel, or policy evidence;
- a production reference such as OpenRATH may be mistaken for a POO-owned proof fact.

The proof layer is the answer to those risks. It makes the implicit control-plane rules explicit, gives them mathematical names, and checks them in Lean. The goal is not to prove a full Scheme interpreter. The goal is to prove that policy, strategy, workflow, session, and runtime-handoff decisions emitted by this project satisfy the POO flow rules they claim to satisfy.

* 10.06.55.15 Why Scheme Modules Need Proof
Scheme is the right language for this control plane because it gives us macros, symbolic data, functional composition, and Gerbil POO prototype objects. Those strengths are exactly why the proof boundary is needed. The same mechanisms that make policy code compact and extensible also make it easy to create a configuration that looks reasonable but is semantically too broad.

The Scheme implementation owns several high-impact module families:

| Scheme Area | Control-Plane Power | Proof Risk |
| =src/core/= | Builds flow, policy, receipt, role, runtime protocol, and runner concepts. | A generated flow may claim a step can run without enough dependency or runtime-boundary evidence. |
| =src/module-system/= | Builds module activation, profiles, object validation, presentation, and projection. | A POO/C3-linearized object may resolve a slot but still violate scope or parent policy. |
| =src/modules/session/= | Builds session and handoff objects. | A session may dispatch or hand off without lifecycle facts. |
| =src/modules/agent-sandbox/= | Builds sandbox/profile policy. | Tool, file, or capability permissions may be reused outside the allowed scope. |
| =src/modules/workflow/= | Builds workflow and CI/CD runtime projections. | Runtime metadata may be mistaken for proof that a workflow is executable. |
| =user-interface/= | Builds realistic downstream configuration surfaces. | User-facing config may compose several policies whose combined behavior is not obvious locally. |

The proof layer does not replace these Scheme modules. It gives their outputs a mathematical contract. Scheme can remain expressive, macro-driven, and POO-native, while Lean checks the claims that must not be left to convention.

#+begin_src typst
$
op("SchemeModule")(x)
  & => op("EmitsFacts")(x) \
op("EmitsFacts")(x)
  & arrow.r.double.not op("Safe")(x) \
op("EmitsFacts")(x) and op("DischargedObligations")(x)
  & => op("ProofBacked")(x)
$
#+end_src

This distinction is important for implementation work. A Scheme module can emit a fact, a receipt, or a runtime metadata field. That emission is only data. It becomes proof-backed only when the corresponding Lean obligation is discharged.

* 10.06.55.20 Core Thesis
POO/C3 linearization decides local object precedence. It does not by itself prove safety.

#+begin_src typst
$
op("Lin")_(upright("POO/C3"))(M) & != op("Safety")(M,Sigma) \
op("Lin")_(upright("POO/C3"))(M) & => op("ResolvedSlots")(M) \
op("Safety")(M,Sigma)
  & =>
    op("DepsCompleted")(M,Sigma)
    and op("ScopeAllowed")(M,Sigma)
    and op("PolicyHolds")(M,Sigma)
$
#+end_src

This distinction matters. POO/C3 linearization can tell us which object slot wins. It cannot automatically tell us whether the resulting flow is allowed to start, whether the selected strategy preserves parent policy, or whether a shared profile is valid for a session. The PooC3 proof layer reasons over the linearized POO/C3 result and asks whether that result is safe in the current state.

The central contract is =CanStart=. A module can start only when dependency, scope, policy, precondition, and guard conditions all hold.

#+begin_src typst
$
op("CanStart")(s,m) equiv &
  op("DepsCompleted")(s,m) \
  & and op("ScopeAllowed")(s,m) \
  & and P_m(s) \
  & and Q_m(s) \
  & and G_m(s)
$
#+end_src

Here:

- =s= is the current proof state;
- =m= is the module, policy, strategy step, or lifecycle step being considered;
- =P_m= is the policy predicate;
- =Q_m= is the structural precondition;
- =G_m= is the guard, usually the narrow condition that blocks unsafe starts.

* 10.06.55.30 Mathematical Vocabulary
The proof layer uses a small vocabulary repeatedly. The names below are intentionally close to the Lean API so a reader can move from the article-level explanation to the proof code.

| Concept | Mathematical Role | Lean Surface |
| State | A snapshot of facts known about the flow | =FlowState=, =SessionState=, fact environments |
| Module | A startable unit in the control graph | =FlowModule=, =SessionModule= |
| Scope | A partial order limiting where a module can start | =ScopeOrder=, =activeScope= |
| Dependency | A directed edge that must be completed first | =dependsOn= |
| Policy | A predicate that must hold before selection or start | =policyAllows= |
| Strategy | A selector that chooses a permitted next step | =StartFrame=, policy bundles, trace steps |
| Trace | A proof-carrying event history | =SessionTrace=, =LifecycleTrace= |
| Obligation | A claim that must be discharged before it becomes evidence | =CandidateObligation= |
| Rejection | A proof that an obligation cannot be discharged | =RejectionCertificate= |

The system should be read as a typed transition system:

#+begin_src typst
$
Sigma & = "typed state facts" \
M & = "modules, policies, strategy steps, lifecycle steps" \
E & = "runtime and control-plane events" \
Sigma "overset"(m, arrow.r) Sigma'
  & => op("CanStart")(Sigma,m) \
Sigma "overset"(e, arrow.r) Sigma'
  & => op("AllowedEvent")(Sigma,e)
$
#+end_src

The design deliberately avoids a raw JSON proof manifest. JSON can carry data, but it cannot by itself express or check the dependent relationship between facts, predicates, traces, and proofs. Lean is used because its types and propositions can encode those relationships directly.

* 10.06.55.35 Why This Matters For AI Agents
AI agents are useful in this project because they can assemble policy objects, extend profiles, compose flows, and generate scenario configurations quickly. That speed creates a different kind of risk from ordinary hand-written software. An agent can produce a configuration that is syntactically valid, locally plausible, and still globally unsafe.

The proof layer gives agents a hard contract:

- do not infer safety from naming, shape, or nearby examples;
- do not treat a production reference as a proof fact;
- do not broaden a parent policy through an override;
- do not dispatch to subagents without lifecycle evidence;
- do not claim runtime execution when Scheme only emitted a handoff descriptor;
- when evidence is missing, emit an obligation instead of a stronger claim.

For AI work, proof is not just a correctness luxury. It is a control mechanism for generated behavior.

#+begin_src typst
$
op("AgentGenerated")(c)
  & => op("CandidateObligations")(c) \
op("AllDischarged")(c)
  & => op("AcceptableProjection")(c) \
exists o,op("Rejected")(o)
  & => not op("AcceptableProjection")(c)
$
#+end_src

This changes how agents should work. The agent should not only write Scheme configuration. It should also preserve the path from configuration to proof facts:

1. which POO object or profile was composed;
2. which C3 linearization result was used;
3. which facts were emitted by Scheme;
4. which Lean obligation those facts target;
5. whether the obligation was discharged, rejected, or left open.

That path makes generated policy reviewable. A human reviewer can inspect not only what the agent wrote, but why the agent believes it is safe.

* 10.06.55.37 AI Strategy/Policy Boundary
The proof target is not a worker-execution claim. A worker executes a unit of work after a policy has selected it. The hard object is the AI strategy/policy system: the logic that decides when a session may start, continue, hand off, launch a subagent, open a sandbox scope, call a tool, retry a loop, or declare completion.

For POO Flow, that policy is not a free-form prompt. It is the composition of POO/C3-linearized profiles, Scheme facts, runtime readiness predicates, sandbox capability scopes, and loop transition rules. The verified surface must answer questions like:
| Axis | Policy question | Proof obligation |
|------+-----------------+------------------|
| Session | Which session owns this action and which child sessions can inherit it? | Lineage and handoff edges are explicit, acyclic, and authorized. |
| Sandbox | Which tools, files, secrets, and runtime resources are available? | Child capability scope is a subset or justified refinement of the parent scope. |
| Loop | Which transition starts next and when may it stop or retry? | Every transition consumes required facts and preserves the loop invariant. |
| Flow | Which module can start after which dependencies? | A successor cannot start until all declared prerequisites are satisfied. |
| Profile | Which shared policy object wins when profiles compose? | POO/C3 order is deterministic and every override is accounted for. |

#+begin_src typst
$
op("AcceptAIAction")(a) =>
  E_(upright("session"))(a) and
  E_(upright("sandbox"))(a) and
  E_(upright("loop"))(a)
$

$
op("SubagentEdge")(p,c) =>
  op("child")(c,p) and
  op("Scope")(c) subset.eq op("Scope")(p) and
  op("Tools")(c) subset.eq op("Tools")(p)
$

$
op("Start")(B) =>
  and.big_(A in op("Requires")(B)) op("Satisfied")(A)
$
#+end_src

This is why the Scheme side must expose typed facts, not loose JSON manifests. The Lean side should check propositions about the same policy objects that the Scheme control plane will execute. The runtime can then reject an agent action because a proof obligation is absent, not because a human reviewer noticed the bug after execution.
* 10.06.55.39 Relation To Agent Frameworks And Research
OpenRATH and Flue are useful reference frameworks because they make real production concerns visible: durable sessions, sandbox placement, tools, skills, workflows, memory, selectors, and runtime harnesses. They are not verification authorities for POO Flow. A reference framework can tell us which runtime axes matter; it cannot prove that a POO/C3-linearized Scheme policy is safe unless its facts are mapped into our Lean obligations.

| Reference | Useful signal | Boundary for POO Flow |
|-----------+---------------+-----------------------|
| [[https://arxiv.org/abs/2210.03629][ReAct]] | Interleaves reasoning traces and actions, making action plans easier to inspect. | A trace improves observability, but it is not a machine-checked policy proof. |
| [[https://arxiv.org/abs/2303.11366][Reflexion]] | Uses feedback and episodic memory to improve later decisions. | Reflection can improve search, but it does not certify session, sandbox, or loop invariants. |
| [[https://arxiv.org/abs/2309.15817][ToolEmu]] | Shows tool-using LM agents can produce realistic high-impact failures. | It is an evaluation and risk-discovery framework, not the proof contract for our policies. |
| [[https://arxiv.org/abs/2406.13352][AgentDojo]] | Shows prompt-injection and untrusted-tool-data risks in realistic agent tasks. | It gives adversarial scenarios; POO Flow still needs typed local proof obligations. |
| [[https://dl.acm.org/doi/10.1145/263699.263712][Proof-Carrying Code]] | Establishes the pattern that code can carry a proof checked by the host. | We adapt the pattern to policy actions, not to proving a full Scheme interpreter. |
| [[https://www.cs.princeton.edu/~appel/papers/says.pdf][Proof-Carrying Authentication]] | Shows a requester can supply a proof that the receiver checks before authorization. | We use the same shape for agent actions: construct evidence, then check before accept. |
| [[https://github.com/Rath-Team/OpenRath][OpenRATH]] | Makes Session, Sandbox, Memory, Tool, Agent, Workflow, and Selector first-class runtime objects. | It is a production reference for axes and vocabulary, not a source of POO Flow truth. |
| [[https://flueframework.com/][Flue]] | Treats sessions, tools, skills, filesystem access, and sandbox runtime as a programmable harness. | It helps frame harness requirements; it does not discharge POO/C3 or Lean obligations. |
| Codex/Claude-style coding agents | Strong at generating code, running tools, and iterating with human feedback. | They need an external typed proof contract before generated policy can be trusted as rigorous logic. |
| Pi/Flue-style harnesses | Useful for durable execution and runtime packaging. | Harness durability is not the same as verified strategy/policy correctness. |

POO Flow facts still come from this repository: Scheme modules, POO/C3 profile composition, declared dependencies, sandbox contracts, session links, and loop strategy rules.

* 10.06.55.40 Architecture
The architecture has three layers: POO/C3 linearization, proof semantics, and runtime evidence.

#+begin_src mermaid
flowchart TB
  UI["UI configuration"]
  POO["POO objects and shared profiles"]
  C3["C3 linearization"]
  Graph["Linearized POO/C3 graph"]
  Policy["Policy predicates"]
  Strategy["Strategy selector"]
  Trace["Proof-carrying trace"]
  Lean["Lean semantics"]
  Runtime["Runtime handoff"]
  Reference["Production reference evidence"]
  Audit["Reference audit"]

  UI --> POO --> C3 --> Graph
  Graph --> Policy
  Graph --> Strategy
  Policy --> Trace
  Strategy --> Trace
  Trace --> Lean
  Lean --> Runtime
  Reference --> Audit
  Audit --> Lean
#+end_src

The important boundary is that runtime evidence and production reference evidence are not automatically proof evidence. They become useful only after the proof layer can type them as owned facts, adopted facts, or rejected obligations.

#+begin_src typst
$
op("PooFlowOwnedFact")(C) & => C \
op("ReferenceObservation")(R,C) & arrow.r.double.not C \
op("VerifiedFromReference")(r)
  & => op("AdoptionCertificate")(r) \
op("AdoptionCertificate")(r) & => C
$
#+end_src

This is why OpenRATH can be a production reference but not a verification authority. It can suggest cases. It cannot close a POO flow proof unless the POO proof boundary adopts the observation through a certificate.

* 10.06.55.45 Scheme To Lean Proof Chain
The proof chain is the bridge between implementation and mathematics. It is also the chain agents should preserve when they modify the system.

#+begin_src mermaid
flowchart LR
  Scheme["Scheme module implementation"]
  POO["POO object / profile / policy"]
  C3["POO/C3 linearization"]
  Facts["Typed emitted facts"]
  Obligation["Lean obligation"]
  Theorem["Lean theorem"]
  Cert["Discharge or rejection certificate"]
  Runtime["Runtime projection / handoff"]

  Scheme --> POO
  POO --> C3
  C3 --> Facts
  Facts --> Obligation
  Obligation --> Theorem
  Theorem --> Cert
  Cert --> Runtime
#+end_src

The chain can be summarized mathematically:

#+begin_src typst
$
op("Implement")_("Scheme")(m)
  & => op("Facts")(m) \
op("Facts")(m)
  & => op("Obligations")(m) \
op("Proof")(m)
  & equiv
    forall o in op("Obligations")(m),
      op("Discharged")(o) or op("Rejected")(o) \
op("Projectable")(m)
  & =>
    forall o in op("RequiredObligations")(m),
      op("Discharged")(o)
$
#+end_src

The chain is intentionally asymmetric. Scheme emits facts and candidate obligations. Lean checks whether those obligations can be discharged. Runtime projection should consume only the discharged part of the proof surface, while rejected obligations explain why a flow must not proceed.

This gives us a practical rule for code review:

| Scheme Change | Required Proof Question |
| Adds a policy object | Does it refine the parent policy or declare a new boundary? |
| Reuses a profile | Are tool, file, and capability permissions still within scope? |
| Adds a session step | Does =CanStart= imply the required lifecycle facts? |
| Adds subagent messaging | Is there active-subagent, channel, and policy evidence? |
| Adds runtime metadata | Is this data only a handoff descriptor, or a proof-backed runtime claim? |
| Imports production reference behavior | Is it an observation, an adopted certificate, or a rejection? |

* 10.06.55.50 Policy And Strategy Semantics
Policy and strategy are separate. A policy states what is allowed. A strategy chooses a next step. The proof layer checks that the selected step is allowed by the policy and that overrides do not broaden the parent contract.

#+begin_src typst
$
op("select")(S,Sigma) = m
  & => op("CanStart")(Sigma,m) \
op("override")("child","parent")
  & => "child" subset.eq "parent" \
"child" subset.eq "parent"
  & equiv
    forall x,  "child"(x) => "parent"(x)
$
#+end_src

The refinement relation is the key mathematical idea for reusable POO policy objects. Reuse is safe only when the child policy is a subset of the parent policy. If a shared profile is reused in a narrower context, the proof must show that every permission it grants is still allowed by the parent scope.

For agents, this means policy generation should produce proof obligations, not only configuration text. A generated policy is incomplete until it can answer:

- Which parent policy does it refine?
- Which scope bounds it?
- Which dependencies must already be complete?
- Which guard blocks unsafe starts?
- Which trace facts prove the selected strategy step?

* 10.06.55.60 Session And Subagent Semantics
Session lifecycle proof is where the abstract policy model meets real agent behavior. A session must establish enough facts before it can dispatch work to agents or subagents.

#+begin_src typst
$
op("CanStart")(Sigma,op("messageDispatch"))
  & => op("channelAuthorized")(Sigma) \
op("channelAuthorized")(Sigma)
  & =>
    op("agentRegistered")(Sigma)
    and op("subagentRegistered")(Sigma)
$
#+end_src

The latest control-link proof also exposes a real gap in the base control-plane event model:

#+begin_src typst
$
op("BaseTrace")(op("sendToSubagent")(a))
  <= top
$
#+end_src

That formula says the base event alone is too weak: it can construct a send trace without lifecycle evidence. The strengthened lifecycle trace requires active subagent, channel authorization, and policy permission:

#+begin_src typst
$
op("LifecycleTrace")(op("sendToSubagent")(a))
  & => op("activeSubagent")(a) \
op("LifecycleTrace")(op("sendToSubagent")(a))
  & => op("channelAllowed")(a) \
op("LifecycleTrace")(op("sendToSubagent")(a))
  & => op("policyAllowsSend")(a)
$
#+end_src

This is exactly the kind of issue the Lean layer should reveal. The proof does not merely mirror the implementation. It identifies where the implementation surface is too permissive and defines the stronger contract that real flow execution should satisfy.

* 10.06.55.90 Agent Usage Contract
Agents should treat the proof layer as a contract, not a passive appendix. When an agent writes or edits policy, strategy, workflow, session, or UI configuration, it should produce or preserve the following evidence:

| Agent Question | Required Evidence |
| Which module starts next? | =CanStart= or a theorem implying it. |
| Which policy object was reused? | A refinement/subset proof against its parent policy. |
| Which profile grants tool or file access? | Sandbox trace theorem for tool/file scope. |
| Which subagent receives a message? | Lifecycle trace with active subagent, channel, and policy facts. |
| Which production reference was used? | Reference observation plus adoption certificate or rejection. |
| Which runtime boundary is crossed? | Handoff fact proving Scheme does not manufacture runtime handlers. |

If the agent cannot supply this evidence, it should not silently emit a stronger claim. It should emit an obligation that remains open, or a rejection certificate when a policy, scope, or lifecycle fact is denied.

#+begin_src typst
$
op("GeneratedPolicy")(p)
  & =>
    exists o,  op("CandidateObligation")(o) \
op("Discharged")(o)
  & => op("ProofBacked")(p) \
op("Rejected")(o)
  & => not op("ProofBacked")(p)
$
#+end_src

* 10.06.55.95 Problems Solved
The proof layer exists because the project is explicitly designed for high-composition policy systems. The problems below are not hypothetical; they are the normal failure modes of an AI-assisted POO control plane.

| Problem | Without Proof | With Proof |
| Precedence mistaken for safety | A C3 winner is treated as executable because the slot resolved. | Linearization and safety are separated; =CanStart= must still be proven. |
| Profile reuse too broad | A shared profile grants a permission that is invalid in the new scope. | Refinement and sandbox theorems force permission subset checks. |
| Agent override broadens policy | An agent replaces a selector policy and accidentally expands allowed starts. | Policy refinement rejects overrides that are not subsets of parent behavior. |
| Missing lifecycle facts | A subagent receives a message because the config shape looks complete. | =LifecycleTrace= requires active subagent, channel, and policy evidence. |
| Runtime confusion | Scheme metadata is interpreted as if runtime execution already happened. | Runtime handoff facts distinguish descriptor ownership from runtime ownership. |
| Reference confusion | OpenRATH or another production system is treated as proof. | Reference observations become obligations; only adoption certificates discharge them. |
| Review opacity | A reviewer sees generated Scheme but not why the agent thinks it is safe. | The proof chain records emitted facts, obligations, discharges, and rejections. |

This is the practical value in AI. The system does not need to trust the agent's explanation. It can require the agent to produce typed evidence. If the evidence is missing, the output remains an obligation rather than becoming a hidden assumption.

#+begin_src typst
$
op("LLMClaim")(C) & arrow.r.double.not C \
op("LLMClaim")(C) and op("LeanEvidence")(C)
  & => C \
op("LLMClaim")(C) and not op("LeanEvidence")(C)
  & => op("OpenObligation")(C)
$
#+end_src

* 10.06.55.100 Future Expectations
The proof architecture should move in four directions.

First, Scheme should emit typed proof facts instead of loose metadata. The control plane can stay in Scheme, but the facts it exports should have a stable Lean-facing contract: module id, scope, dependency refs, policy refs, sandbox permissions, lifecycle facts, and runtime handoff facts.

Second, user-interface cases should become scenario proofs. A UI config is not only a sample. It is a concrete composition of profiles, policies, strategies, and sessions. The proof layer should be able to say which scenario starts, which step is blocked, and which policy fact explains the decision.

Third, production references should remain reference evidence. OpenRATH and similar systems are useful because they reveal real control-plane shapes. They should generate candidate obligations, not verified facts. POO flow remains the authority for POO flow correctness.

Fourth, runtime and FFI boundaries should become explicit certificates. When Gerbil, C FFI, or Marlin runtime handoff is involved, the proof should record what the Scheme side owns, what the runtime side owns, and which side is forbidden from manufacturing the other side's proof facts.

  Facts --> Lean["Lean proof obligations"]
  UI["UI scenarios"] --> Lean
  Reference["Production references"] --> Obligations["Candidate obligations"]
  Obligations --> Lean
  Lean --> Certificates["Discharge or rejection certificates"]
  Certificates --> Runtime["Runtime handoff"]

* 10.06.55.110 Acceptance Rule
A flow is proof-backed only when the emitted strategy step carries Lean evidence for the relevant start, trace, policy, lifecycle, and runtime boundary claims.

Manifest completeness is only a bridge. It can tell us that data exists. It cannot prove that the data is safe. The proof-backed acceptance rule is:

#+begin_src typst
$
op("ProofBackedFlow")(F)
  equiv &
    op("CanStartEvidence")(F) \
    & and op("PolicyRefinementEvidence")(F) \
    & and op("TraceEvidence")(F) \
    & and op("LifecycleEvidence")(F) \
    & and op("RuntimeBoundaryEvidence")(F)
$
#+end_src

That is the standard future implementation should target: not more prose claims, not raw JSON manifests, and not informal production analogies, but typed evidence that can be checked, rejected, and explained.
