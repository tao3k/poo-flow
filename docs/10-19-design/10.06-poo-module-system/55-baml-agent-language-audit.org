* 10.06.55 BAML Agent Language Audit
This document records what =poo-flow= should learn from BoundaryML's BAML project. The goal is not to copy BAML syntax or create a second language beside Gerbil Scheme. The useful lesson is language-engineering discipline for agent-facing control surfaces.

** 10.06.55.20 What BAML Is Building
BAML presents itself as a programming language for agents and structured LLM output. Its public user story is intentionally simple:
1. write =.baml= functions, classes, enums, prompts, clients, and tests;
2. generate language clients;
3. call the generated client from Python, TypeScript, Go, Ruby, or another runtime;
4. receive typed outputs, retries, robust parsing, and editor feedback.

The deeper engineering lesson is the compiler boundary. The architecture note describes a staged pipeline:
#+begin_src mermaid
flowchart LR
  Source["BAML source text"]
  Lexer["Lexer"]
  CST["Parser / CST"]
  AST["AST\nsemantic desugaring"]
  HIR["HIR / PPIR / TIR\nquery layers"]
  MIR["MIR\nexecution-ready representation"]
  Bytecode["BexVM bytecode"]
  Clients["Generated clients\nPython / TS / Go / Ruby"]
  Runtime["Runtime call\nLLM request + parse + typed result"]
  Source --> Lexer --> CST --> AST --> HIR --> MIR --> Bytecode
  AST --> Clients --> Runtime
  HIR --> Clients
#+end_src

The architecture rule that matters for us is "upstream over downstream": place meaning in the earliest layer that can own it. The parser should parse, the AST should own semantic desugaring, later layers should not rediscover high-level language facts by inspecting lowered payloads.
** 10.06.55.30 Similarities To poo-flow
BAML and =poo-flow= share several pressures even though the implementations are different:
- Both expose a compact agent-facing authoring surface.
- Both need typed declarations that survive into runtime-visible projections.
- Both must make invalid agent-authored changes fail early.
- Both need editor/doctor/playground-style feedback because agents and humans need fast inspection loops.
- Both separate author declarations from runtime execution.
- Both treat generated or projected artifacts as boundary products, not as the source of semantic truth.

The closest local analogy is:
#+begin_src mermaid
flowchart LR
  Author["Author surface\ninit.ss, use-module, POO profiles"]
  POO["Semantic object graph\nPOO objects + C3 composition"]
  Contracts["Contract layer\nslot types, harness checks, hard gates"]
  Graph["Graph layer\nDAG, loop, session, sandbox diagnostics"]
  Manifest["Runtime handoff manifest\nfinal projection only"]
  Marlin["Marlin / Rust runtime\nheavy execution"]
  Receipts["Receipts and telemetry"]
  Author --> POO --> Contracts --> Graph --> Manifest --> Marlin --> Receipts
  Receipts --> Graph
#+end_src

For BAML, generated clients are the practical bridge from language declarations to application code. For =poo-flow=, runtime manifests and typed projections are the bridge from Scheme/POO policy declarations to Marlin/Rust execution.
** 10.06.55.35 User-Facing Thesis Correction
The ordinary user-facing thesis for =poo-flow= is not schema, Lean, manifest, contract, or type facts. Advanced users may still program Scheme/POO directly and inspect proof or runtime artifacts deliberately.

The user-facing thesis is:
#+begin_src text
Describe an agentic workflow graph in natural language.
Understand the loops, steps, models, modules, gates, runtime handoff, and
result paths on that graph.
Let agents use a guide system to generate Scheme/POO and let validators handle
the internal type, proof, and manifest details.
#+end_src

The internal thesis remains stronger:
#+begin_src text
The graph is backed by a composed policy graph.
The policy graph is built from Scheme and POO objects.
C3 determines composition order.
Validators prove legal composition.
Graph explanation makes the result inspectable.
Manifest projection hands a validated policy to runtime.
Lean-facing facts are proof products, not user concepts.
#+end_src

This gives four distinct views:
- user view: =Agentic Workflow Graph=;
- agent/repair view: declarations, modules, steps, gates, and diagnostics;
- runtime view: validated manifest and receipts;
- proof view: type facts and Lean fact contracts.

These views must not be collapsed. Exposing Lean, schema, or manifest details as the required first concept for ordinary workflow authors is a product error even when those mechanisms are technically strong. Keeping them available for advanced users, module authors, agents, proof tooling, and runtime integrators is a strength.
** 10.06.55.40 Key Difference
BAML is a new DSL and compiler for LLM functions, prompts, structured output, and generated clients.

=poo-flow= should not become a new standalone language. The project advantage is different:
- Gerbil Scheme already gives us S-expressions, macros, modules, and functional composition.
- Gerbil POO gives us objects, prototypes, C3 linearization, extension, and multiple inheritance.
- The project layer can use these directly instead of inventing a separate parser and syntax family.

=poo-flow= also has a proof-facing route through Lean. That makes its type and validation ceiling higher than a normal agent DSL, as long as the proof machinery stays behind the user-facing graph.
Therefore the adoption decision is:
#+begin_src text
Do learn BAML's layer discipline, type persistence, generated-boundary
discipline, and fast feedback model.

Do not copy BAML syntax, compiler architecture, or client SDK strategy as the
main Scheme-side interface.
#+end_src
** 10.06.55.42 Product Surface Difference
The product comparison is:
#+begin_src text
BAML hides runtime complexity behind a DSL.
poo-flow should hide system complexity behind a graph-oriented Scheme/POO
interface and an agent guide system.
#+end_src

BAML asks users to learn function, prompt, class, enum, client, and test declarations in a dedicated language. Even if those declarations lower to objects, ordinary users still need to learn the DSL's conceptual surface. =poo-flow= should keep the ordinary surface thinner: workflow category, modules, features, loops, steps, gates, models, and runtime handoff. Scheme/POO remains available as the programmable layer, but ordinary users should be able to start from graph intent.
** 10.06.55.45 Strength Comparison
BAML's type story is productized better today. It gives users a direct path from function declarations to typed outputs, generated clients, editor feedback, and runtime parsing.

=poo-flow= is not theoretically weaker. Its type and proof potential is stronger because it is not limited to a narrow DSL:
- Scheme is the host language, not a made-up configuration syntax.
- POO and C3 can describe multi-module policy composition, not only function signatures.
- The project harness can validate structured POO objects.
- Lean-facing facts can turn selected validation results into proof obligations.
- Graph algorithms can explain whole-system policy behavior rather than a single LLM call.

The current weakness is expression and product shape, not theoretical ceiling. BAML has a clearer user path. =poo-flow= must make its stronger internals visible through an understandable graph surface, not by asking users to learn schema, Lean, manifests, or internal validators.
** 10.06.55.50 Lessons To Adopt
*** 10.06.55.50.10 Upstream Semantic Ownership
Feature meaning must live as high as possible.

In =poo-flow= this means:
- module selection and feature selection belong in the module/profile layer;
- workflow semantics belong in workflow POO objects and functional builders;
- sandbox semantics belong in sandbox module objects and resource contracts;
- loop semantics belong in loop-engine, governor, and human-audit objects;
- runtime manifests only project already-validated facts.

If a feature can only be understood by looking at a final alist or ABI payload, it is probably in the wrong layer.
*** 10.06.55.50.20 Runtime Type Persistence
BAML's type-system note emphasizes that declared types remain meaningful at runtime. The local equivalent is stronger than a DSL type annotation when it is routed through Scheme, POO, harness validation, and Lean-facing facts:
#+begin_src text
A runtime handoff row may be an alist only because it is the final boundary
encoding. Its schema, object kind, source, contract owner, and provenance must
come from validated POO objects.
#+end_src

This aligns with the existing safety rule:
- POO expresses strategy fragments.
- C3 determines composition order.
- validators check legality.
- Marlin executes the final policy.

C3 is not safety. C3 only answers precedence. Safety comes from contracts, hard gates, denylist union, monotonic permission narrowing, budget tightening, and human-gate monotonicity.
*** 10.06.55.50.30 No Untyped Public Blobs
BAML is valuable because it refuses a loose =any= culture. =poo-flow= should apply the same pressure to public control-plane surfaces:
- no raw alist DSL as the normal user interface;
- no anonymous manifest payload without schema and producer;
- no task policy that can overwrite safety policy by ordinary override;
- no sandbox profile row whose resource, filesystem, capability, or permission shape is unvalidated;
- no human-audit decision without owner, evidence, decision kind, and allowed mutation target.
*** 10.06.55.50.40 Generated Boundary Products
BAML generates clients from source declarations. =poo-flow= should generate or project:
- typed Scheme projection helpers;
- runtime manifest rows;
- doctor/preview reports;
- graph explanation receipts;
- Marlin handoff payloads.

These outputs are not semantic owners. They are products of the POO graph and contract layer.
*** 10.06.55.50.50 Fast Feedback Surface
BAML's editor/playground story matters. Agent-facing configuration needs a fast preview loop:
- show selected modules and features;
- show composed POO ancestry and C3 precedence;
- show which validators fired;
- show sandbox/resource/security deltas;
- show graph reachability, cycles, gates, and denied expansion;
- show the final manifest without executing heavy runtime work.

This maps directly to =poo-flow doctor=, scenario tests, profile previews, and manifest projection tests.
*** 10.06.55.50.55 Agent Editing Feedback Correction
The fast feedback lesson should be read as an agent editing loop, not only a product preview loop.

BAML's editor/playground story matters because the author immediately sees prompt, type, output, and error. The equivalent =poo-flow= weakness is the absence of an equally fast loop after an agent changes configuration.

After an agent changes modules, graph declarations, Scheme/POO profiles, validators, or runtime handoff settings, it must immediately know:
- did the graph become valid?
- what graph nodes or edges changed?
- where was the change rejected?
- which validator rejected it?
- why was it rejected in graph terms?
- what declaration should the agent repair next?

This is critical because agents do not have stable human intuition about the whole composed policy graph. Without fast feedback, an agent can keep editing Scheme/POO in the wrong layer, accidentally widen permissions, remove gates, break loop exits, or generate manifests that no longer correspond to the user-visible graph.

Therefore the feedback surface must show:
- the agent's authoring delta;
- the affected graph nodes and edges;
- selected modules and features;
- fired validators and rejection reasons;
- graph reachability, cycles, gates, and denied expansion;
- sandbox/resource/security deltas;
- the next repair target.

Manifest rows, type facts, and Lean facts are drill-down artifacts. The default feedback must answer whether the agent's edit worked, where it failed, and what to repair next.
*** 10.06.55.50.60 Observability Framework Correction
The diagnostic surface should be understood as part of a Scheme-side observability framework.

BAML's product feedback is strong because authoring evidence is visible immediately. =poo-flow= should not copy that as a text-only doctor command. It should build a POO-native observability object graph whose projections include diagnostics, receipts, doctor reports, graph previews, proof facts, and runtime handoff previews.

The corrected layer is:
#+begin_src mermaid
flowchart LR
  Edit["Agent edit"]
  Obs["POO observability objects\nevent, span, diagnostic, receipt"]
  Diag["Diagnostic projection\nvalidator rejection"]
  Receipt["Agent receipt\nwhat changed, why rejected, next repair"]
  Doctor["Doctor/report projection"]
  Proof["Proof/type-fact projection"]
  Runtime["Runtime manifest preview"]
  Edit --> Obs
  Obs --> Diag --> Receipt
  Obs --> Doctor
  Obs --> Proof
  Obs --> Runtime
#+end_src

This keeps diagnostic behavior extensible. A workflow module, sandbox module, loop-engine module, or human-audit module can add its own diagnostic class by extending the shared observability protocol. Core receipt consumers should depend on the shared object slots and methods, not on a central list of every possible module-specific error.

The design lesson from BAML is therefore not "make a playground." It is:
#+begin_src text
Make every authoring edit produce structured, immediate, inspectable evidence.
For poo-flow, that evidence is a POO observability graph.
The agent-facing receipt is one projection of that graph.
#+end_src
** 10.06.55.60 Proposed poo-flow Layer Names
Canonical P0 boundary and namespace names are defined in =58 P0 Boundary Namespace Design=. The names below are the BAML comparison vocabulary and should be mapped to the canonical boundary names before implementation.

Use the following layer names when designing new work:

1. Author surface.
   =init.ss=, =use-module=, profiles, feature flags, and POO-native declarations.
2. Semantic object graph.
   POO objects, prototype extension, C3 composition, inherited policy fragments, and functional workflow builders.
3. Contract and validator layer.
   Slot contracts, structural harness validation, hard safety gates, budget gates, denylist union, permission narrowing, and human-gate monotonicity.
4. Graph explanation layer.
   DAG, loop graph, session graph, sandbox graph, dependency cones, cycle checks, reachability, and gate coverage.
5. Boundary projection layer.
   Final alists, receipts, runtime manifests, ABI rows, and report-only payloads.
6. Runtime owner.
   Marlin/Rust, or another external runtime, owns heavy execution.

The layer model:
#+begin_src mermaid
flowchart TD
  A["Author surface"]
  B["Semantic object graph"]
  C["Contract and validator layer"]
  D["Graph explanation layer"]
  E["Boundary projection layer"]
  F["Runtime owner"]
  A --> B --> C --> D --> E --> F
  C -->|"reject invalid expansion"| A
  D -->|"doctor and preview feedback"| A
  F -->|"receipts"| D
#+end_src
** 10.06.55.70 Agent Configuration Implication
Agents should modify the highest legal layer, not generated payloads.

Allowed:
- add or remove module features through =use-module=;
- extend a profile through POO-native objects;
- tighten sandbox resources;
- add human gates;
- add loop-governor checks;
- add workflow graph checks;
- propose spec/profile/eval/policy mutation through Human Audit.

Not allowed as ordinary behavior:
- edit final runtime manifests directly;
- widen permissions after validation;
- erase denylist facts;
- remove human gates through task-local override;
- bypass contract checks by providing raw alists;
- encode feature semantics only inside Marlin ABI rows.
** 10.06.55.80 Implementation Plan
Phase 1: document and style closure.
- Add this audit as the BAML comparison document.
- Add an "upstream semantic owner" rule to the programming style document.
- Cross-link this document from the module-system index.

Phase 2: manifest contract tightening.
- Ensure each runtime manifest family carries schema, producer, object kind, source, and provenance.
- Add focused validation tests that invalid POO object slots fail before projection.
- Treat anonymous untyped payload rows as implementation bugs except at private test fixtures.

Phase 3: doctor and preview surface.
- Add a profile preview path that shows author surface, C3 order, validator decisions, graph facts, and final report-only manifest.
- Keep the preview non-executing.
- Add scenario cases for sandbox, workflow, loop-engine, governor, and human-audit interaction.

Phase 4: generated projection helpers.
- Use Gerbil macros only for mechanical constructor/projection/validator frames.
- Keep expansion output ordinary POO-native or functional code.
- Do not generate a second public DSL.

Phase 5: Marlin-facing stability.
- Treat Scheme projections as typed contracts for Marlin consumers.
- Keep runtime execution outside Scheme.
- Add receipts that make contract version, producer, consumer, and request id visible.
** 10.06.55.90 Non-Goals
- Do not port BAML.
- Do not create a new parser for the main =poo-flow= authoring surface.
- Do not replace Gerbil POO with a BAML-like type model.
- Do not move LLM prompt authoring into the workflow core as a first-class runtime dependency.
- Do not let generated manifests become the source of truth.
