* 10.06.17 POO Best Practices
** 10.06.17.10 Purpose
This document records the practical engineering rules for writing POO-style
module code in =poo-flow=.

Detailed programming rules derived from =.data/gerbil-poo= live in
[[file:29-gerbil-poo-programming-guidelines.org][29 Gerbil POO Programming Guidelines]].
This file is the module-system entrypoint; =29= is the deeper language-style
contract.

The goal is to keep =poo-flow= native to Gerbil POO while preserving a simple
declarative user interface.

** 10.06.17.20 Reference Signals
The useful external patterns are:
- Doom Emacs: small user-facing declarations with a richer upstream
  implementation.
- Haumea/POP practice: separate loader/importer/layout concerns from final
  object composition.
- grafonnix: constructors return POP objects, extension stays on objects, and
  final values project into plain data only at the boundary.

The =poo-flow= interpretation is POO-native: user files extend prototype
objects directly rather than calling a chain API or writing row/alist DSLs.

** 10.06.17.30 Local Owner Rules
The current =poo-flow= rule is:

#+begin_src text
extension graph primitives live in src/modules/extension.ss.
object contracts and object-aware wrappers live in src/modules/object-core.ss.
shared object catalogs live in src/modules/objects.ss.
sandbox profile authoring prototypes live in src/modules/*/profile-interface.ss.
user-interface files declare data only.
#+end_src

=src/modules/extension.ss= owns generic graph facts only:
- extension nodes;
- extension operations;
- extension contributions;
- fixed-point convergence;
- structural snapshots.

=src/modules/object-core.ss= owns object semantics:
- module object kind ids;
- field contracts;
- field contributions;
- contribution-to-extension projection;
- config merge receipts;
- object nodes;
- object catalogs over the extension graph.

Profile interface modules own public POO prototypes and helpers. They are the
surface users and agents should import through =init-syntax= or the facade.

** 10.06.17.35 Harness API Rule
=poo-flow= must not copy the Gerbil harness proof vocabulary.

The upstream dependency =gerbil-scheme-language-project-harness= owns
structured POO evidence APIs such as:

#+begin_src scheme
poo-pattern-structural-validation
#+end_src

=src/modules/object-validation.ss= may build source references and local
diagnostics, but it should call the upstream API for structural harness
validation. =load!= follows the same rule for object entrypoints.

Validation receipts must have a stable user-facing projection. Do not
reimplement local variants of =type-proof=, =relation-proof=, proof trees, or
obligation engines in =poo-flow=. If a stronger formal validation surface is
needed, extend the harness dependency first and then consume the exported API
here.

** 10.06.17.40 Object Shape
Every module object should have:
- an identity under the =objects.*= namespace;
- a narrow domain, such as =sandbox=, =profile=, or =workflow=;
- explicit supers when it inherits behavior;
- direct field contracts only for fields it owns;
- metadata that is an association list;
- no backend execution side effects.

Use Gerbil POO supers for inheritance:

#+begin_src scheme
(poo-flow-module-object
 'objects.nono-sandbox.profile
 (list poo-flow-sandbox-core-profile-object
       poo-flow-nono-sandbox-object)
 fields
 metadata)
#+end_src

Do not implement a second recursive inheritance merge. Gerbil POO/C3 owns
precedence. Local code may project resolved fields into module-specific
receipts.

** 10.06.17.50 Transformer Rule
Transformers should be standard-library-shaped functions wrapped by POO object
contracts.

Good names are intentionally boring:
- =list.append=
- =list.remove=
- =list.prepend=
- =map.set=
- =map.unset=
- =entity.remove=
- =entity.insert-after=

The user should not need to learn a private merge language. The system should
lower each transformer into a typed object-owned contribution:

#+begin_src text
field contract + transformer args
=> field contribution
=> extension contribution
=> fixed-point graph result
#+end_src

This is intentionally not a second extension graph operation set. Transformer
contracts validate agent/user repair payloads and then produce ordinary field
contributions. The fixed-point engine still only sees extension primitives from
=src/modules/extension.ss=.

** 10.06.17.60 User Interface Rule
The downstream user interface should stay declarative, but declarative now
means Gerbil POO declarations rather than a profile row DSL.

Preferred user shape:

#+begin_src scheme
(use-module nono-sandbox
  (.def (ci/build @ ci/check
                  capabilities resources metadata)
    capabilities: => (lambda (super-capabilities)
                       (append super-capabilities
                               '(filesystem-write cache-mount)))
    resources: =>.+ readwrite-project-workspace-resources
    metadata: => (lambda (super-metadata)
                   (append super-metadata build-metadata))))
#+end_src

Rules:
- Use =(.def (name @ super slot ...) ...)= for profile authoring.
- Use =slot:= for direct values.
- Use ==>= with one value expression for inherited-slot transforms.
- Use lambda transforms when the transform needs captured data.
- Use ==>.+= only when extending an inherited child POO object.
- Keep raw alists at projection, ABI, manifest, or negative-test boundaries.
- Do not expose =:append=, =:remove=, =:override=, =:compute=, or
  =(lambda (self super) ...)= as normal user syntax.

Custom object catalogs may be split into small files, but the entry remains
declarative:

#+begin_src scheme
;;; objects.ss
(load! "objects/sandbox-profile")
#+end_src

The loaded fragment returns a list of =poo-flow-module-object= values. It does
not define doctor commands, runtime adapters, or backend execution. The =load!=
macro applies the same validation gate as first-party object catalogs.

** 10.06.17.70 Diagnostics And Observability
POO makes recursive extension easy, so diagnostics must be explicit.

Every object-aware merge path should be able to report:
- source object identity;
- supers / prototype chain;
- direct fields;
- resolved fields;
- field contribution list;
- extension contribution list;
- fixed-point iteration count;
- stable? flag;
- validation diagnostics.

This is the object-system analogue of Doom's doctor commands, but it remains
upstream-owned. User-interface files should call or receive diagnostics, not
define them. Runtime backend projections should preserve the same repairability
principle.

** 10.06.17.80 Executable Guard
The code-side guard for this document is:

#+begin_src text
t/module-object-practice-test.ss
t/module-object-validation-test.ss
t/sandbox-core-resource-contract-test.ss
t/sandbox-core-profile-authoring-diagnostics-test.ss
#+end_src

Those tests verify the intended layering:
- field contracts are constructed from =object-core=;
- object-owned rows become field contributions;
- config merge receipts preserve original contributions;
- Gerbil POO resource objects go through the harness-backed contract;
- non-primary authoring shapes are diagnostic facts, not recommended syntax.

** 10.06.17.90 Anti-Patterns
Do not:
- put field contracts, profile prototypes, or transformer contracts in
  =extension.ss=;
- let user-interface files define validation or doctor logic;
- load object fragments without the =poo-flow-require-module-objects-validation!=
  gate;
- create a second merge language when a POO slot transform is enough;
- hide object inheritance behind ad hoc alist recursion;
- make backend runtime execution part of object construction;
- add compatibility aliases for old object namespaces after a canonical owner
  exists.

** 10.06.17.100 Landed Implementation Slice
The POO-native profile interface slice is now implemented through:
- =src/modules/sandbox-core/profile-interface.ss=
- =src/modules/nono-sandbox/profile-interface.ss=
- =src/modules/cubeSandbox/profile-interface.ss=
- =src/modules/docker-sandbox/profile-interface.ss=
- =src/module-system/init-syntax.ss=

The important design rule remains fixed:

#+begin_src text
public profile authoring is POO .def, not a row/alist DSL.
#+end_src

Future repair operators should become named POO-native helpers or functional
helpers rather than another private merge language.
