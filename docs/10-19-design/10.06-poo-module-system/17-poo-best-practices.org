* 10.06.17 POO Best Practices
** 10.06.17.10 Purpose
This document records the engineering rules for writing POO-style module code in =poo-flow=.

It is intentionally practical. The goal is not to copy Doom Emacs, Haumea, or grafonnix. The goal is to keep the =poo-flow= module system native to Gerbil POO while preserving a simple declarative user interface.

** 10.06.17.20 Reference Signals
The useful external patterns are:

- Doom Emacs: small user-facing declarations with a richer upstream implementation.
- Haumea/POP practice: separate loader/importer/layout concerns from final object composition.
- grafonnix: one domain object per file, each =new= constructor returns a POP object, methods are extension functions, internal state is hidden, and final values are projected into plain data through =__unpop__=.

The local grafonnix research checkout is:

#+begin_src text
.data/grafonnix
#+end_src

Relevant grafonnix files:

- =flake.nix= aggregates the public =lib= surface.
- =grafonnix/dashboard.nix= defines a dashboard object with methods such as =addPanel=, =addTemplate=, and =addLink=.
- =grafonnix/row.nix= defines a row object with =addPanel= and =addPanels=.
- =grafonnix/graph_panel.nix= and =grafonnix/stat_panel.nix= show panel-specific object methods and hidden counters such as =_nextTarget=.
- =internal/default.nix= defines recursive projection from POP objects to plain data.

** 10.06.17.30 Local Owner Rules
The current =poo-flow= rule is:

#+begin_src text
extension graph primitives live in src/modules/extension.ss.
object contracts and object-aware wrappers live in src/modules/object-core.ss.
shared object catalogs live in src/modules/objects.ss.
object validation receipts live in src/modules/object-validation.ss.
user-interface files declare data only.
#+end_src

=src/modules/extension.ss= owns generic graph facts only:

- extension nodes;
- extension operations;
- extension contributions;
- fixed-point convergence;
- structural snapshots.

=src/modules/object-core.ss= owns object semantics:

- module object kind ids;
- field contracts;
- field contributions;
- contribution-to-extension projection;
- config merge receipts;
- object nodes;
- object catalogs over the extension graph.

=src/modules/objects.ss= is a catalog/facade. It may aggregate shared objects and re-export object-core for convenience, but implementation modules that need object primitives should import =object-core= directly. This mirrors grafonnix: =flake.nix= aggregates =lib=, while object files receive the full =lib= directly.

** 10.06.17.35 Harness API Rule
=poo-flow= must not copy the Gerbil harness proof vocabulary.

The upstream dependency =gerbil-scheme-language-project-harness= owns structured POO evidence APIs such as:

#+begin_src scheme
poo-pattern-structural-validation
#+end_src

=src/modules/object-validation.ss= may build a source reference for the module object and may add local diagnostics for =poo-flow= object contracts, but it should call the upstream API for structural harness validation.

Do not reimplement local variants of =type-proof=, =relation-proof=, proof trees, or obligation engines in =poo-flow=. If a stronger formal validation surface is needed, extend the harness dependency first and then consume the exported API here.

** 10.06.17.40 Object Shape
Every module object should have:

- an identity under the =objects.*= namespace;
- a narrow domain, such as =sandbox=, =profile=, or =workflow=;
- explicit supers when it inherits behavior;
- direct field contracts only for fields it owns;
- metadata that is an association list;
- no backend execution side effects.

Use Gerbil POO supers for inheritance:

#+begin_src scheme
(poo-flow-module-object
 'objects.nono-sandbox.profile
 (list poo-flow-sandbox-core-profile-object
       poo-flow-nono-sandbox-object)
 fields
 metadata)
#+end_src

Do not implement a second recursive inheritance merge. Gerbil POO/C3 owns precedence. Local code may project resolved fields into module-specific receipts.

** 10.06.17.50 Transformer Rule
Transformers should be standard-library-shaped functions wrapped by POO object contracts.

Good names are intentionally boring:

- =list.append=
- =list.remove=
- =list.prepend=
- =map.set=
- =map.unset=
- =entity.remove=
- =entity.insert-after=

The user should not need to learn a private merge language. The system should lower each transformer into a typed object-owned contribution:

#+begin_src scheme
field contract + transformer args
=> field contribution
=> extension contribution
=> fixed-point graph result
#+end_src

Each transformer contract must state:

- input kind;
- argument kind;
- output kind;
- idempotency when true;
- identity key for entity lists;
- diagnostic payload when validation fails.

** 10.06.17.60 User Interface Rule
The downstream user interface should stay declarative.

Preferred user shape:

#+begin_src scheme
(use-module nono-sandbox
  :config
  (profiles
   (ci/build
    (:extends cicd/build)
    (metadata (stage . build)))))
#+end_src

Row operators such as =:append= and =:remove= are implementation escape hatches. They are useful for tests and advanced repair cases, but they should not be the primary story.

** 10.06.17.70 Diagnostics And Observability
POO makes recursive extension easy, so diagnostics must be explicit.

Every object-aware merge path should be able to report:

- source object identity;
- supers / prototype chain;
- direct fields;
- resolved fields;
- field contribution list;
- extension contribution list;
- fixed-point iteration count;
- stable? flag;
- validation diagnostics.

This is the object-system analogue of Doom's doctor commands, but it remains upstream-owned. User-interface files should call or receive diagnostics, not define them.

** 10.06.17.80 Executable Guard
The code-side guard for this document is:

#+begin_src text
t/module-object-practice-test.ss
#+end_src

That test verifies the intended layering:

- field contracts are constructed from =object-core=;
- object-owned rows become field contributions;
- field contributions project into extension contributions;
- config merge receipts preserve original contributions;
- the final graph is produced by the generic fixed-point extension engine.

This keeps the best practice executable. If future code moves contract wrappers back into =extension.ss=, this test and the import graph should fail loudly during review.

** 10.06.17.90 Anti-Patterns
Do not:

- put field contracts, profile prototypes, or transformer contracts in =extension.ss=;
- let user-interface files define validation or doctor logic;
- create a second merge language when a standard list/map/entity transform name is enough;
- hide object inheritance behind ad hoc alist recursion;
- make backend runtime execution part of object construction;
- add compatibility aliases for old object namespaces after a canonical owner exists.
