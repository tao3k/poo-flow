#+TITLE: 10.06.27 Next Implementation Closure Plan

* 10.06.27 Next Implementation Closure Plan

** 10.06.27.10 Purpose

This document audits the next implementation push after the workflow CI/CD
user-interface bridge landed. It is a closure plan, not a new design branch.

The current direction remains:
- Scheme owns POO declarations, object graphs, projections, policy snapshots,
  traces, receipts, and report-only runtime handoff data.
- Rust/Marlin owns heavy execution, process control, durable state, real
  sandbox execution, and provider-specific adapters.

The next work should therefore close the control-plane surface that lets agents
inspect and validate a workflow handoff before any runtime executes.

** 10.06.27.20 Current Closure Evidence

The following surfaces are now implemented enough to treat as the current
baseline:

- =src/modules/workflow/cicd.ss= owns inert workflow CI/CD check and check-map
  objects.
- =src/modules/funflow/config.ss= lowers compact
  =(use-module funflow :config (pipeline ...))= declarations into workflow
  check maps.
- =pooFlowUserConfigPresentation= exposes workflow CI/CD pipeline count,
  readiness rows, receipt rows, runtime command manifest maps, compact manifest
  summaries, sandbox runtime summaries, sandbox handoff summaries, unresolved
  profile refs, and presentation trace stages.
- =pooFlowUserProfilePresentation= and
  =pooFlowUserProfileDoctorPresentation= forward the same workflow CI/CD
  projection surface.
- =src/module-system/declaration-case.ss= treats the expanded workflow CI/CD
  trace stages as part of the maintained downstream declaration case.
- Focused tests now cover Funflow pipeline lowering, workflow check maps,
  user-interface CI/CD presentation, profile presentation, declaration cases,
  unit aggregation, and project policy checks.

This closes the earlier fixture-only gap: downstream user declarations can now
reach report-only runtime command manifest data through the public module
configuration surface.

*** 10.06.27.33 P1: Configured Registry Bridge

Risk:
- =10.02= and =10.03= still identify configured registries as the next Funflow
  alignment gap.
- CI/CD check maps now reach user presentation, but configured runner paths
  still need to consume task-family and flow-declaration registries as explicit
  run-config data.

Closure:
- Let run configs carry task-family registries and flow-declaration registries.
- Thread configured registries through strategy planning and runner validation.
- Emit typed control-plane failures when a configured workflow references an
  unsupported task family or flow declaration.
- Keep heavy scheduling, Docker/CAS, and durable persistence outside Scheme.

*** 10.06.27.34 P1: Runtime Handoff Contract

Risk:
- Runtime command manifest maps are inspectable, but Marlin-facing handoff
  semantics are still only a report-only shape.
- Without a stable compact handoff contract, provider adapters may accidentally
  learn Scheme internals or CI/CD-specific details.

Closure:
- Treat compact manifest summaries as the user/agent audit surface.
- Treat full manifests as the runtime handoff payload.
- Define the fields that Marlin must consume first: operation, request id,
  argv, request payload, policy, runtime owner, handoff-required flag, and
  runtime-executed flag.
- Add validation that manifests and summaries agree on check identity, argv,
  runtime owner, unresolved sandbox refs, and non-execution status.

Status 2026-06-20:
- M4 now exposes a stable Marlin-facing ABI projection from workflow CI/CD check-map runtime command manifests.
- Scheme still only manufactures validated handoff data; it does not parse Scheme source inside Marlin or execute runtime adapters.
*** 10.06.27.35 P2: Tutorial Result Ladder Resume Point

Risk:
- CI/CD bridge work can become self-contained and drift away from Funflow
  tutorial-result parity.

Closure:
- Resume =10.04= at the runtime-heavy result gates only after P1 bridge
  validation is stable.
- The next tutorial-aligned implementation should be external config preflight
  and runtime command handoff, not Docker execution in Scheme.
- Docker/CAS visible results remain Rust/Marlin runtime targets.
