#+title: LangGraph Scenario Gap Audit for POO Flow
#+date: 2026-07-05

* Purpose

This document treats LangGraph as a scenario, not as the semantic owner of POO
Flow.  The goal is to use a LangGraph-like agent graph to test whether POO
Flow's own =src= capabilities are complete enough:

- Graph and loop topology;
- policy and strategy composition;
- durable and checkpoint intent;
- sandbox, tool, and subagent scope;
- human approval and resume gates;
- runtime handoff contracts;
- Lean proof projection.

If the scenario is hard to express, that is evidence of a missing POO Flow
platform primitive.  The fix belongs under =src=, then the scenario should be
rewritten as ordinary =user-interface= composition.

* Direction

The direction of dependency is:

#+begin_src typst
$
op("POOFlowSrc")
  =>
  op("Composition")
  =>
  op("Scenario")
  =>
  op("RuntimeHandoff")
$
#+end_src

not:

#+begin_src typst
$
op("LangGraph")
  =>
  op("POOFlowPolicy")
$
#+end_src

LangGraph-like behavior should be an instance of POO Flow's Graph, Loop,
Durable, Checkpoint, Policy, and Composition system.

* Reference Pressure Points

The scenario pressures are taken from LangGraph's public documentation and the
checked-out reference repository under =.data/langgraph=.  The important point
is not to copy those APIs.  The important point is to ensure POO Flow's own
platform primitives can express the same class of agent/workflow problems.

- Graph API pressure: sequences, branches, loops, =Send= for map-reduce style
  fanout, and =Command= for state update plus routing.
- Runtime pressure: actor/channel/superstep execution creates requirements for
  graph visibility boundaries, merge contracts, and checkpoint cadence.
- Persistence pressure: checkpointers and stores create requirements for
  checkpoint identity, thread scope, short-term state, long-term store scope,
  replay, fork, and fault tolerance.
- Human-in-loop pressure: interrupt/resume requires checkpoint-before-human,
  resume identity, and receipt validation.
- Streaming pressure: runtime events require Scheme-declared stream intent and
  receipt event labels, but not a Scheme streaming runtime.
- Subgraph and multi-agent pressure: nested graphs and parent/child navigation
  require subgraph/subagent-as-node contracts and explicit state boundary facts.
- Fault-tolerance pressure: retry, timeout, and error handler policies must be
  declared in Scheme and accounted for by runtime receipts.

* Capability Matrix

Status terms:

- =supported= means the primitive exists as a general POO Flow capability.
- =partial= means the scenario can be approximated, but the contract is not yet
  first-class enough for reuse and proof.
- =missing= means the capability must be added under =src= before the scenario
  should be considered ordinary composition.
- =runtime-owned / scheme-declared= means Scheme must declare and prove the
  intent, but Marlin/runtime performs the actual execution.

* Gap Classification

** P0: Platform gaps blocking a clean scenario

These must be implemented under =src= before a LangGraph-like composition can be
treated as a strong community example:

1. Reducer and merge intent for concurrent branch writes.
2. Dynamic command/goto intent as a graph control primitive.
3. Fanout/fanin/barrier intent for map-reduce and parallel branches.
4. Checkpoint-before and checkpoint-cadence gates.
5. Subgraph/subagent-as-node handoff contracts.
6. Receipt-to-Lean fact generation for the above contracts.

** P1: Runtime-owned but Scheme-declared gaps

These should not become Scheme runtime implementations.  They must be declared
in Scheme, proven as policy intent, and checked against runtime receipts:

1. Streaming/event projection declarations.
2. Retry, timeout, and error-handler declarations.
3. Replay/fork/time-travel policy intent.
4. Persistent store scope requirements.
5. Observability event labels.

** P2: Later hardening

These make production evolution safer but can follow the first proof loop:

1. Backward-compatible state/profile version gates.
2. Migration and drain policies for parked sessions.
3. Cross-version receipt compatibility checks.

* Implementation Rule

Scenario code must stay thin:

- reusable primitives go in =src=;
- reusable community profiles go in =user-interface/profiles=;
- scenario compositions go in =user-interface/cases=;
- proof models and generated facts go in =packages/proof/lean=;
- runtime execution remains outside Scheme, behind handoff receipts.

* First Gaps To Close

The first implementation pass should close the gaps that make a LangGraph-like
composition less than ordinary composition:

1. Add first-class graph merge/reducer intent.
2. Add first-class checkpoint-before gate intent.
3. Add subgraph/subagent-as-node handoff intent.
4. Add runtime stream/retry/timeout declarations as handoff intent, not runtime.
5. Extend generated Lean facts so the scenario proves these contracts.

These are not LangGraph-specific features.  They are POO Flow platform
capabilities that make many future scenarios easier to express.

* Verification Target

A LangGraph-like scenario is acceptable when it can be expressed as:

#+begin_src scheme
(use-composition langgraph-like-agent
  ...)
#+end_src

with no scenario-specific runtime code in =user-interface= and no raw ad hoc
alist DSL.  The composition must produce:

- a Scheme plan;
- generated Lean facts;
- a runtime handoff contract;
- a receipt-checking contract for Marlin/runtime.

The scenario is complete only when the same primitives can be reused for other
community workflows without changing the LangGraph-specific case.

* 2026-07-05 Implementation Pass

The first P0-P2 pass added a general =src= intent surface:

- =src/graph/scenario-gap-intent.ss= defines POO-native intent constructors for
  reducer, command/goto, fanout, barrier, checkpoint cadence,
  checkpoint-before, subgraph node, subagent node, stream, retry, timeout,
  error route, replay/fork, store scope, observability labels, and state version
  hardening.
- =t/scenarios/langgraph-scenario-gap-intent-test.ss= proves these primitives
  can be collected into one ordinary scenario gap plan without adding
  LangGraph-specific runtime code to =user-interface=.
- =packages/proof/lean/PooFlowProof/PooC3/ScenarioGap.lean= gives the Lean-side
  fact shape and boundary theorems:
  - P0 belongs to Scheme =src= platform capabilities.
  - P1 is runtime-owned but Scheme-declared.
  - P2 is production hardening.

This pass closes the first control-plane layer.  It does not implement runtime
execution, checkpoint storage, streaming, retry loops, or timeout handling in
Scheme.  Those remain Marlin/runtime responsibilities and must be checked by
runtime receipts.

The next pass started the receipt-checking layer by adding a stable
missing-kind diagnostic:

- =poo-flow-scenario-gap-receipt-missing-kinds= compares a scenario plan against
  runtime-accepted intent kinds.
- The scenario gate now proves that a complete accepted-kind set has no missing
  contract and that removing =runtime.error-route= reports exactly that missing
  kind.

Full receipt fact-row matching remains the next step.  It should be introduced
as a small, stable contract over runtime rows, not as a Scheme runtime
implementation.

The follow-up pass fixed the plan identity boundary and made the runtime row
check stable:

- scenario plans now use =plan-id= instead of a generic =name= slot.  The
  generic slot can interact badly with POO object lookup and should not be used
  for plan identity.
- runtime rows are checked with three small predicates:
  - the row's =plan-id= matches the Scheme plan;
  - =rejected-kinds= is empty;
  - all required intent kinds are present in =accepted-kinds=.
- Lean models this as =ScenarioRuntimeRowFacts= and proves that
  =planOk ∧ rejectionsOk ∧ acceptedOk= implies the row matches the plan.
- =packages/proof/lean/PooFlowProof/Generated/ScenarioGapRuntimeRow.lean= is a
  checked-in generated module for the valid row case, so the receipt boundary is
  part of the regular Lean build instead of only a Scheme string rendering
  smoke test.
- Invalid rows do not receive a generated =runtimeRowMatchesPlan= theorem.  The
  renderer emits =RuntimeRowComplete := false= for a bad row, and Lean has
  explicit negative examples for missing accepted kinds, wrong plan identity,
  and runtime rejections.
- The scenario gate now checks the exact generated Lean source shape for a valid
  runtime row.  This prevents the Scheme renderer and the checked-in generated
  proof module from drifting silently.
