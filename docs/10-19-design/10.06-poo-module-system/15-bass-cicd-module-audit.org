The audit used Bass documentation first, then sampled the dedicated Bass entry
files named by the documentation. The relevant local evidence is:

- =.data/bass/README.md= describes Bass as glue-code Lisp for project commands,
  CI, Dockerfile replacement, Bash replacement, thunk recipes, cached outputs,
  hermetic leaning, and a modular runtime currently backed by BuildKit.
- =.data/bass/CONTRIBUTING.md= names the project-facing command files:
  =bass/build=, =bass/docs=, =bass/test=, and =bass/shipit=.
- =.data/bass/docs/lit/guide.md= documents common tasks: running thunks,
  reading output, secrets, cache directories, Git checkout helpers, exporting
  files/images, =bass.lock= memoization, code sharing, and server mode for
  webhook-style CI/CD endpoints.
- =.data/bass/docs/lit/bassics.md= defines modules as scopes, thunks as
  serializable commands, thunk paths as first-class artifacts, and runtime
  dispatch as image/runtime-selected execution.
- =.data/bass/docs/lit/stdlib.md= keeps the standard library small: a ground
  module inherited by all modules, script bindings, and a few focused modules.

** 10.06.15.20 Interface Lesson
Bass keeps the public language interface short by making scenario scripts thin.
The visible command files have a consistent shape:

- load a shared module with =use=;
- read structured input from =*stdin*=;
- call named domain functions;
- emit, write, or start checks;
- keep runtime work behind thunks and the runtime backend.

Examples from the sampled files:

- =bass/build= calls =bass:build= and =bass:smoke-test=.
- =bass/test= calls =bass:coverage= and writes =cover.html=.
- =bass/docs= emits =bass:docs=.
- =bass/shipit= composes release asset creation and GitHub release publication.
- =bass/github-hook= exposes CI check maps such as =build-linux=,
  =build-darwin=, =build-windows=, =test=, =nix=, and =docs=, then hands them
  to a loop runner.

This is not a monolithic core. It is a domain module plus thin scenario entry
points.

** 10.06.15.25 User Manual Lesson
The stronger lesson is in the user manual, not the implementation. Bass docs do
not ask users to understand the runtime architecture first. The guide opens as a
quick reference for common tasks, then shows runnable forms:

- install one =bass= binary;
- run one demo;
- create a thunk;
- call =run= or =succeeds?=;
- read output;
- pass secrets;
- use cache directories;
- build files and images.

The manual keeps module language small:

- a module is a scope in its own file;
- the ground module is inherited by all modules;
- modules can run as commands or as thunks;
- focused modules such as =.strings= and =.git= are introduced by the task they
  solve, not by their internal hierarchy.

This is the pattern =poo-flow= should copy. The user manual should not expose
our C3 graph, descriptor registry, loader backend, runtime-command manifest
schema, or Rust handoff internals first. Those belong in design docs and
diagnostics. The user-facing manual should show the smallest useful action and
let the module system remain mostly invisible.

For CI/CD the first manual page should look closer to this:

#+begin_src scheme
(poo-flow! poo-flow-user-profile poo-flow-user-profile-set
  (profile developer)
  :workflow
  (funflow
   +functional
   +dag
   (+cicd
    (checks +parallel +typed-receipts)
    (artifacts +export)
    (release +manual-gate))))


#+end_src

The above is intentionally not the final implementation syntax. It is the
manual-facing target: small verbs, obvious nouns, no registry terminology.

** 10.06.15.26 User Documentation Shape Audit
The Bass user documentation separates two tracks that =poo-flow= should also separate:
- =docs/lit/guide.md= is task-first. It is a quick reference for common work: run thunks, read output, provide secrets, cache directories, pass thunk paths, export files/images, pin dependencies, share code, and serve webhook-style scripts.
- =docs/lit/bassics.md= is concept-first. It defines the small language vocabulary: scope, keyword, operative, applicative/function, source, sink, path, thunk, thunk path, module, hermetic, and runtime.
The ordering matters. A new user does not start with runtime architecture. They first see:
- one installable binary;
- one demo command;
- one runnable thunk;
- one way to run/check success;
- one way to read output;
- one way to pass secrets or caches;
- one way to export artifacts.
Only after that does the documentation explain the internal language terms.

The Bass docs also make runtime boundaries feel natural:
- a =thunk= is a serializable command object, not a runtime implementation;
- a =thunk path= is a first-class artifact reference, not a filesystem copy recipe;
- an =image= chooses the runtime, but users still write ordinary =from= or =with-image= forms;
- a =module= is just a scope in a file, introduced by the task that needs it;
- server mode is presented as a small command for HTTP/webhook endpoints, with limitations stated plainly.

The =poo-flow= user manual should therefore have two matching tracks:
- =guide=: task-first recipes such as define a workflow, run/check it, inspect a receipt, pass a secret, mount/cache an artifact, export a runtime manifest, define CI checks, and hand off to Marlin.
- =poo-flow bassics=: concept glossary for profile, category, module, =+feature=, POO object, role/C3 composition, descriptor, receipt, runtime manifest, and adapter.

The =init.ss= surface should now be documented as a category tree, not as a constructor API:
#+begin_src scheme
(poo-flow! poo-flow-user-profile poo-flow-user-profile-set
  (profile developer)
  :workflow
  (funflow +functional +dag +typed-receipts)
  :sandbox
  (nono-sandbox +nono +doctor)
  (cubeSandbox +cube +doctor)
  :custom
  (my-module "./custom/my-module" +private +doctor))
#+end_src
The visible hierarchy is:
- category marker: =:workflow=, =:sandbox=, =:custom=;
- module row: =funflow=, =nono-sandbox=, =cubeSandbox=, =my-module=;
- feature flags: =+functional=, =+dag=, =+doctor=, =+private=.
This is the same lesson as Bass: users manipulate small language objects, while runtime mechanics and adapter details stay behind inspection and handoff APIs.
** 10.06.15.27 Funflow/CI Boundary Decision
Bass aligns strongly with our Funflow route because both designs treat executable work as inspectable objects before runtime execution. The lesson is not to copy Bass as a language DSL. The useful part is the user-facing layer: short task verbs, clear nouns, first-class work objects, and runtime mechanics hidden behind receipts and handoff APIs.

Therefore CI/CD should be modeled as a primary =funflow= feature payload at the workflow-engine layer. Running CI/CD still means declaring and executing a workflow graph: checks, jobs, artifacts, releases, secrets, caches, webhook events, policy gates, and runtime-command manifests are all workflow-node, policy, or projection concerns.

The boundary is nested rather than category-sibling. =:workflow= selects the workflow family, =funflow= selects the engine module, and =(+cicd ...)= gives that engine the CI/CD vocabulary:
- checks;
- jobs;
- artifacts;
- caches;
- secrets;
- releases;
- webhooks;
- runtime manifests.
Those nouns should be visible inside the =+cicd= payload:
#+begin_src scheme
(poo-flow! poo-flow-user-profile poo-flow-user-profile-set
  (profile developer)
  :workflow
  (funflow
   +functional
   +dag
   +typed-receipts
   (+cicd
    (checks +parallel +typed-receipts)
    (artifacts +export)
    (release +manual-gate))
   +runtime-manifest)
  :sandbox
  (nono-sandbox +nono +doctor)
  (cubeSandbox +cube +doctor))
#+end_src

Funflow remains the implementation substrate for CI/CD graph composition:
- CI checks are workflow nodes;
- dependency order is a DAG;
- artifacts are typed outputs;
- cache/secret/runtime constraints are policy metadata;
- Marlin/Rust receives runtime-command manifests for heavy execution.

The =+cicd= feature should be a first-class =funflow= capability, not a minor bridge. It means this workflow can project into CI/CD receipts, runtime manifests, and policy-controlled handoff requests. The public manual should teach =(+cicd ...)= as the user-facing domain vocabulary, with Funflow explained as the engine underneath it.
** 10.06.15.30 poo-flow Decision
CI/CD should be implemented as a =funflow= feature payload plus optional adapters, not as logic inside =src/core= and not as a competing top-level category marker.

The Scheme side should own:

- module descriptors and user-facing scenario names;
- POO/C3 role composition for CI/CD policies, sources, artifacts, checks,
  releases, secrets, cache intents, and runtime handoff receipts;
- static and dynamic validation receipts;
- declarative workflow/check maps that can be inspected before execution;
- runtime-command manifests that Rust or Marlin-owned runtime code can execute.

The Scheme side should not own:

- container execution;
- GitHub/GitLab API clients;
- Docker/BuildKit/Nix process management;
- long-running webhook servers;
- artifact upload side effects.

Those belong to Rust, Marlin runtime, or downstream adapters. =poo-flow= should
describe and validate the control plane.

** 10.06.15.40 Proposed Module Shape
Use a Funflow-owned feature surface rather than a core path:

- =src/modules/funflow/config.ss= owns the default =(+cicd ...)= payload.
- =src/modules/funflow/interface.ss= may own the public projection once the surface outgrows config-only data.

The first public shape should stay small:

#+begin_src scheme
(poo-flow! poo-flow-user-profile poo-flow-user-profile-set
  (profile developer)
  :workflow
  (funflow
   +functional
   (+cicd
    (checks +parallel)
    (artifacts +export)
    (release +manual-gate)
    (webhook +server))))
#+end_src

The descriptor surface should project to ordinary module-system data:

#+begin_src scheme
(poo-flow-module
  name: 'cicd/checks
  group: 'cicd
  provides: '(checks artifacts runtime-manifest)
  policies: '(non-executing inspectable hermetic-preferred)
  runtime-owner: 'marlin-agent-core)
#+end_src

The important rule is that =+github=, =+docker=, =+nix=, and similar names are adapter metadata, not replacements for the base =+cicd= contract.

** 10.06.15.50 POO/C3 Use
Gerbil POO and C3 multiple inheritance are a strength here. CI/CD descriptors
should be role-composed instead of flattened into one record type.

Recommended role families:

- =poo-flow-case-module= for all scenario modules;
- =ci-check-provider= for named check maps;
- =ci-artifact-producer= for files, images, and coverage outputs;
- =ci-release-producer= for release bundles and publication intents;
- =ci-secret-consumer= for declared secret requirements;
- =ci-cache-consumer= for cache intents and invalidation policy;
- =runtime-handoff-producer= for Rust/Marlin execution manifests.

C3 lets a GitHub release module combine check, artifact, release, secret, and
runtime-handoff roles without making core know about GitHub.

- Project check maps into runtime-command manifests.

- Keep execution out of Scheme.

- Add optional =github=, =gitlab=, =nix=, and =docker= modules only after the base
  CI/CD vocabulary is stable.
