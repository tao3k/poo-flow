#+title: Gerbil Bazel Downstream Convergence
#+startup: overview

* Status

Implemented for the shared capability and the POO Flow consumer.  The final
cold-build timing gate remains evidence-driven: it must run only when the
Scheme admission guard reports host pressure below the CPU-adaptive runnable
limit.

* Decision

=gerbil-bazel= is the sole owner of generic Gerbil Bazel capabilities:

- host discovery and Darwin/Linux normalization;
- toolchain and native ABI providers;
- normalized =gxi=, =gxc=, =gxpkg=, and =gxtest= executables;
- project-declared Gerbil library views;
- writable staged project actions driven by the consumer =build.ss=;
- reusable =GerbilProjectInfo=, complete logs, and JSON receipts;
- the workspace-aware =install_dependencies= capability.

Downstream repositories retain only domain rules and build truth.  POO Flow
retains =build.ss=, its Scheme-owned semantic receipt, Scheme project
projection, runtime-C rules, and POO-specific policy.  =gerbil-scheme-rust=
retains Rust/FFI and Cargo domain rules.  Neither downstream owns a second
generic Gerbil repository, toolchain, project compile, project test, or
project-development implementation.

* Public capability contract

Every host or prebuilt provider repository publishes:

- =registered_toolchain= and =toolchain_impl=;
- =native_scheme_env= and the four normalized Gerbil tools;
- =dependency_libraries= and =dependency_library_root=;
- =toolchain.receipt.json=;
- =gerbil_project_compile=, =gerbil_project_dev=,
  =gerbil_project_test=, and =GerbilProjectInfo=;
- =install_dependencies=.

The dependency installer enters =BUILD_WORKSPACE_DIRECTORY=, defaults
=GERBIL_PATH= to the workspace-local =.gerbil= tree, invokes the standard
=gxpkg deps --install= path through the normalized native environment, and
finishes with =gxpkg list=.

* Removed POO Flow legacy surface

The POO Flow consumer no longer carries compatibility implementations for:

- local Gerbil repository discovery;
- a local Gerbil toolchain provider or toolchain type;
- host-system normalization;
- native ABI fingerprinting;
- native tool and environment templates;
- dependency-install templates;
- bootstrap/finalize shell targets and their tests;
- local Scheme project compile/test rules and shell runners.

Runtime-C contract and test rules consume the same public
=GERBIL_TOOLCHAIN_TYPE= and unified toolchain =runfiles= depset as the Scheme
project rules.  The only remaining local Scheme Bazel rule is the POO-specific
projection from shared =GerbilProjectInfo=.

* Performance gate

Cold timing is invalid under runnable saturation.  The latest observed host had
12 logical CPUs and 127 runnable processes.  The Scheme guard uses the
machine-adaptive threshold =logical-cpu-count * 2=, emitted
=blocked-host-pressure= with =runnable-saturation=, and reported
=compiled-stage-count=0=.  No downstream migration may bypass that guard or
replace it with a hard-coded CPU count.

The cold acceptance gate is:

1. admission outcome is =ready=;
2. default Bazel sandbox execution completes;
3. the total compile receipt is compared with a receipt carrying the same
   cache state, strategy, CPU count, worker count, and host-pressure fields;
4. a second run proves the incremental cache hit.
