#+TITLE: 10.06.34 Loop Operations Coordination Audit

* Decision
The next loop-engine research result is that =memory-policies= are necessary
but not sufficient. They describe how each loop use case remembers state, but
real production loops also need three separate policy surfaces:

- coordination: which loop owns a branch, PR, or resource when multiple loops
  overlap;
- observability: how every loop run is logged, budgeted, measured, slowed,
  paused, or killed;
- safety: which paths, connector writes, merge actions, and human gates are
  denied or escalated.

These must not be folded back into =loop-engine-memory-policy=. Memory is the
state spine contract. Coordination, observability, and safety are operational
contracts around that state.

* Evidence From loop-engineering
The local reference clone is =.data/loop-engineering=.

** GitHub Actions examples
=examples/github-actions/README.md= describes event-driven and scheduled loops
without a TUI session. The flow is:

1. gather context, such as CI, merges, failures, issues, or releases;
2. update state files;
3. delegate to the agent harness in the "Invoke agent" step.

The important boundary is explicit: the state schema and skills are
tool-agnostic, and only the invocation step differs. This supports our current
Scheme/Rust split. Scheme should declare a report-only handoff surface; Marlin
or external automation owns scheduling, event handling, and agent invocation.

** Multi-loop coordination
=docs/multi-loop.md= gives the missing production rule set:

- at most one loop may mutate a branch per hour;
- daily triage reports, while action loops execute;
- each pattern gets its own state file;
- action loops write =acting_on= to state;
- before spawning a fix, an action loop reads other pattern state files and
  skips if another loop is already acting on the same branch or PR;
- ambiguous cross-loop conflicts go to a shared human inbox in =STATE.md=.

The recommended state layout includes =STATE.md=, =ci-sweeper-state.md=,
=dependency-sweeper-state.md=, =post-merge-state.md=, and an append-only
=loop-run-log.md=.

** Run log and budget
=docs/operating-loops.md= makes observability a production gate. It requires
budget estimates before scheduling, early exit when there is no actionable
work, and an append-only run log. The minimal run-log schema records:

- =run_id=;
- =pattern=;
- =duration_s=;
- =items_found=;
- =actions_taken=;
- =escalations=;
- =tokens_estimate=;
- =outcome=.

It also defines slow, pause, and kill criteria: budget over 80%, repeated false
positives, repeated escalations, incidents, reviewer absence, consistent
failure modes, or cost exceeding value for multiple weeks.

** Safety and failure evidence
=docs/safety.md= defines path denylists, no-auto-merge by default, least
privilege connector scopes, required human gates, secrets redaction, and flake
handling. =stories/multi-loop-collision.md= gives a concrete failure: CI
Sweeper and PR Babysitter both fixed the same PR branch, producing two
conflicting commits and roughly five times the expected token spend. The root
cause was no =acting_on= collision check.

* Implication For POO Flow
We should keep the current =memory-policies= implementation as the state shape
for each use case, then add a POO-native policy extension surface around it.
The runtime core should not grow one bespoke function per policy family.

Operational policy families should be regular POO objects that can project
report-only receipts through a shared extension collection:

- =loop-engine-coordination-policy=: priority order, state files,
  =acting-on= key, conflict action, branch or PR lock scope, human inbox path;
- =loop-engine-observability-policy=: run-log path, run-log schema, budget
  path, metric keys, retention window, slow/pause/kill signals;
- =loop-engine-safety-policy=: denylist paths, allowlist paths, human gate
  categories, max attempts, connector scopes, auto-merge policy.

All three should follow the existing POO-native pattern:

#+begin_src scheme
(.def (repo-loop-coordination @ loop-engine-coordination-policy
                              priority state-files acting-on-key
                              conflict-action human-inbox)
  priority: '(ci-sweeper pr-babysitter dependency-sweeper
              post-merge-cleanup daily-triage)
  state-files: '((daily-triage . "STATE.md")
                 (ci-sweeper . "ci-sweeper-state.md")
                 (dependency-sweeper . "dependency-sweeper-state.md")
                 (post-merge-cleanup . "post-merge-state.md"))
  acting-on-key: 'acting_on
  conflict-action: 'skip-and-log
  human-inbox: "STATE.md#Human Inbox")
#+end_src

The receipt projection should stay report-only and be collected generically:

- =coordination-receipt= states the intended ownership and conflict behavior;
- =observability-receipt= states run-log/budget schema and pause signals;
- =safety-receipt= states denylist, connector, and human-gate policy.

The first implementation step is now in place as a shared =loop-engine-policy-extension= prototype plus concrete POO-native sub-prototypes: =loop-engine-coordination-policy-extension=, =loop-engine-observability-policy-extension=, and =loop-engine-safety-policy-extension=. Profiles attach concrete extension objects through =policy-extensions=, and lowering projects a =policy-extension-receipts= collection. The runtime handoff, manifest request, and presentation surfaces pass that collection through without hard-coding =coordination=, =observability=, or =safety= in =loop-engine-runtime.ss=.

The initial user-facing scenario is now a three-object operational policy set in =user-interface/custom/my-module/profiles/loops.ss=: =repo-loop-coordination=, =repo-loop-observability=, and =repo-loop-safety=. They are normal POO objects that extend the concrete policy-extension prototypes and declare priority, state files, =acting_on=, run-log schema, lifecycle signals, path policy, connector scopes, human gates, and auto-merge policy. Scheme still does not acquire locks, schedule loops, mutate state files, pause or kill loops, or resolve conflicts; those remain Marlin/runtime responsibilities.

Scheme must not read other state files, acquire locks, pause schedulers, call
GitHub Actions, or write run logs. Those are Marlin or external runtime
responsibilities.

* Next Implementation Order
1. Add a POO-native policy-extension protocol so =loop-engine-runtime= can
   collect extension receipts without knowing each concrete policy family.
2. Re-express existing policy receipts through that collection path where
   practical, while keeping the current manifest contract stable.
3. Add =loop-engine-coordination-policy= as the first new extension. The
   multi-loop collision story shows this is the highest-value missing
   operational contract.
4. Add =loop-engine-observability-policy= with run-log and budget schema.
5. Add =loop-engine-safety-policy= with denylist/human gate facts.
6. Update the user-interface profile to demonstrate CI Sweeper owning red CI,
   Dependency Sweeper pausing while main is red, and Daily Triage remaining L1
   report-only.

* Non-Goals
- Do not implement a scheduler in Scheme.
- Do not read or write =STATE.md=, =loop-run-log.md=, or pattern state files in
  Scheme.
- Do not build GitHub Actions integration in this package.
- Do not make Scheme own token accounting. It only declares budget schema and
  pause criteria.
