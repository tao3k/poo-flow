** 10.06.21.20 Zerolang Agent Contract
Zerolang states its agent boundary directly:
- the graph is the program database;
- agents inspect and patch the graph;
- humans review readable =.0= projections;
- projection import/export is not the normal agent edit path;
- checked graph patches validate shape before the package store changes.

This is the important lesson for =poo-flow=. The strongest design idea is not a
new syntax. The design idea is ownership:
- semantic state has one authority;
- human-readable text is a projection;
- agents use query and checked patch operations;
- validation happens before the runtime input is mutated;
- text and JSON outputs are separate reading modes.

** 10.06.21.30 Translation Boundary
Zerolang is not a template for =poo-flow=. Its CLI, projection format, compiler
model, and language ownership are different from ours. The useful reference is
the agent-facing engineering boundary: semantic facts have an authority,
human review is a first-class loop concern, and runtime effects are gated by
explicit policy.

The Zerolang guide exposes a short daily loop:

#+begin_src sh
zero query
zero patch --op help
zero patch --op 'addMain'
zero check
zero test
zero run -- <args>
#+end_src

It also makes projection review explicit:

#+begin_src sh
zero export
zero verify-projection
zero import
zero check
#+end_src

The rule is precise in Zerolang: export projections when a human asks for
source-like review, or when CI needs a drift gate. Do not export as an
automatic agent step.

=poo-flow= should not copy this command shape. It should translate the same
engineering boundary into POO terms. The equivalent user/agent surface is a
configured loop strategy over a POO object graph fact layer:

#+begin_src sh
poo-flow loop inspect
poo-flow loop doctor
poo-flow loop plan
poo-flow loop handoff
#+end_src

The exact CLI names can still change. The semantic contract should not:
- =inspect= reads the current POO profile/module/loop object graph;
- =doctor= returns loop and profile diagnostics as repair contracts;
- =plan= resolves the configured loop strategy, selected patterns, gates,
  budgets, validation mode, and runtime boundary;
- =handoff= projects runtime command/resource data for Marlin or another
  backend without Scheme owning timers, worktrees, connector IO, or process
  execution.

** 10.06.21.40 Mapping To POO Flow
Zerolang's graph only partially maps to =poo-flow='s POO object layer. The
important mapping is about authority and review boundaries, not surface syntax.

| Zerolang | POO Flow |
| graph is the program database | POO objects are the profile/module/workflow fact layer |
| =.0= projection | human audit loop and review receipts |
| =zero query= | profile/config/presentation inspection |
| checked graph patch | checked POO object/profile patch |
| =zero verify-projection= | config/profile/loop contract validation |
| diagnostics as repair contracts | doctor diagnostics and fix plans |
| build/run owned by compiler/runtime | heavy workflow runtime owned by =marlin-agent-core= |

The current =poo-flow= direction is mostly aligned:
- =user-interface/init.ss= is already a small validated configuration surface.
- =poo-flow!= short form keeps profile construction out of root init.
- =src/module-system/user-profile.ss= owns the maintained =users= profile.
- profile presentations are shallow, replayable receipts with
  =runtime-executed = #f=.
- doctor reports are pure declaration receipts and do not load packages,
  realize descriptors, or execute runtime hooks.

The gap is the agent operation surface. Today the object graph can be declared
and projected, and =src/loops/strategy.ss= already has loop strategy plans,
level ceilings, validation policy, pattern selection, human-gated projection,
and Marlin handoff metadata. The next design step is to make that loop strategy
object the primary configurable agent engineering surface.

** 10.06.21.45 POO Object Graph Fact Layer
For =poo-flow=, the fact layer is not a Zero-style compiler graph. It is a
Gerbil POO object graph:
- root/user config declares validated configuration rows that agents may modify
  through checked interfaces;
- profile objects declare user/module state;
- module objects declare extension and policy boundaries;
- workflow objects declare task composition and receipt semantics;
- sandbox objects declare capabilities, resources, mounts, and backend intent;
- loop strategy objects declare recurring agent control policy;
- presentation objects and receipts are review artifacts, not authorities;
- loop governor objects declare agent and multi-agent governance facts;
- human audit loops own real human review, approval, escalation, and sign-off.

This matters because Gerbil POO/C3 gives us deterministic role composition.
The agent should not reason over scattered Scheme text or ad hoc alists. It
should reason over composed object facts:
- which profile is selected;
- which modules and features are active;
- which loop patterns are within the configured level ceiling;
- which human gates are required;
- which agent auditor, verifier, or governor roles judge the loop;
- which sandbox profile and mounts are available;
- which local validation mode is allowed;
- which runtime handoff owner receives the manifest.

The loop layer should therefore become a configurable engineering object, not
just a runner option. A concrete agent loop should be a POO object composed from
roles such as:
- purpose;
- autonomy level;
- schedule;
- durable state;
- sandbox profile;
- budget;
- maker/checker split;
- verifier requirements;
- human gates;
- runtime handoff;
- observability and receipts.

=loop-governor= and =human-audit= are separate loop concepts:
- =loop-governor= owns agent governance. In a multi-agent setup one agent can
  act as auditor, another as verifier, and another as governor; they can judge
  each other before a handoff leaves the control plane.
- =human-audit= owns human intervention as a governor-derived human node. It
  inherits the governor policy shape through POO/C3, consumes governor facts
  and manifests, and records decisions as human decisions rather than agent
  judge decisions.

The implementation should expose this as a POO node graph, not only as policy
alist fields:
- =loop-governor-node-role= is the common node layer;
- =loop-governor-agent-node-role= models machine-side auditor/verifier/governor
  participants;
- =loop-governor-human-node-role= models human authority while retaining
  governor lineage;
- =agent-judge-nodes= projects machine governance nodes for Marlin and doctors;
- =audit-node= projects the human audit object through the same node contract.

This is where =poo-flow= differs from Zero. Zero uses graph patches to make
program edits safer. =poo-flow= uses POO/C3-composed loop and policy objects to
make agent operation safer and more reusable.

** 10.06.21.46 Config Layer And Human Loop
=init.ss= is not the human projection layer. It is a configuration layer:
- compact enough for humans to read;
- structured enough for agents to edit;
- validated by syntax, POO object contracts, profile doctor checks, and policy
  warnings;
- projected upstream into the canonical =users= profile and module facts.

The human-facing layer is a loop on top of that configuration and fact graph.
A human loop should receive review artifacts such as:
- proposed config/profile/module changes;
- type and contract validation results;
- loop strategy resolution;
- sandbox mount/capability deltas;
- runtime handoff manifest;
- execution receipts and replay evidence;
- approval, rejection, escalation, or requested-change decisions.

This keeps agent configurability and human accountability separate. Agents may
modify the configuration through safe checked paths. Humans audit and approve
through a human loop that observes the composed facts and receipts.

The same separation applies inside the loop layer. Agent judges belong to
=loop-governor= because they are still machine governance. True human
intervention belongs to the =human-audit= human node. It is still derived from
the governor chain, but it marks =governance-node-kind = human= and
=human-intervention = #t=.

** 10.06.21.47 Three-Layer Zerolang Re-Audit
The three-layer audit shows why =poo-flow= should not copy Zero directly.

Layer 1, configuration:
- Zero uses project/package/compiler commands and graph patch operations around
  a new language.
- =poo-flow= uses Scheme files and POO macros as a validated configuration
  surface.
- The =poo-flow= config layer is intentionally agent-editable through checked
  interfaces; it is not just a human-only projection.

Layer 2, fact graph:
- Zero's compiler graph is the program authority for a new DSL/language.
- =poo-flow='s authority is a Gerbil POO object graph composed from config,
  profile, module, workflow, sandbox, strategy, governor, and audit objects.
- This keeps us inside Scheme+POO instead of inventing a separate language.

Layer 3, governance and human loop:
- Zero exposes human review through projection review, diagnostics, and docs
  assistant guardrails.
- =poo-flow= should expose agent governance as =loop-governor= facts, including
  multi-agent auditor/verifier/governor judgement.
- =poo-flow= should expose human review as a separate human audit loop over
  composed facts, manifests, validation results, and receipts.
- Human decisions belong in loop state and audit receipts, not as comments or
  hidden edits in =init.ss=.

This audit reframes Zero as contrast evidence. Zero proves the value of
agent-oriented fact authority and review boundaries. =poo-flow= implements that
value with Gerbil POO/C3 roles, validated Scheme configuration, and explicit
human audit loop objects.

** 10.06.21.48 Code Landing
The first code slice for this three-layer boundary is now:
- =src/loops/human-audit.ss= defines a POO human audit review object.
- =src/loops/agent.ss= re-exports the human audit surface.
- =t/loop-human-audit-test.ss= covers review items, decisions, and invalid
  decisions.

This slice does not add a scheduler, source-text patcher, approval UI, or
runtime executor. It turns governor facts into review items and records human
decisions as typed loop data.

The boundary is intentionally strict:
- governor facts may include agent auditor/verifier/governor judgement;
- governor contracts project =agent-judge-nodes= as POO node contracts;
- human audit consumes those facts and also projects its own =audit-node=;
- human audit is a governor-derived human node, not an agent-governor judge
  pretending to make human decisions.

** 10.06.21.50 Documentation Interface Lesson
Zerolang docs do not start with compiler architecture. They start with:
- install;
- ask an agent for behavior;
- inspect current graph state;
- patch one fact;
- run only the validation needed;
- review projections only when useful.

=poo-flow= documentation should keep the same split:
- a task-first guide for users and agents;
- a concept reference for POO, C3 roles, profile sets, descriptors, receipts,
  runtime manifests, and adapters;
- a diagnostics guide where every warning is a repair contract.

The task-first guide should show =init.ss= first, but it should not stop there.
It should immediately show the inspectable result:

#+begin_src scheme
(poo-flow!
 :workflow
 (funflow
  (+cicd
   (checks +parallel +typed-receipts)
   (artifacts +export)
   (release +manual-gate)
   (runtime +manifest-handoff)))
 (loop-engine)
 :sandbox
 (nono-sandbox +nono +doctor)
 (cubeSandbox +cube +doctor)
 :custom
 (my-module "./custom/my-module" +private +doctor))
#+end_src

Detailed loop configuration is not written as init-row flags. It belongs in a
POO-native module config:

#+begin_src scheme
(use-module loop-engine
  :config
  (.def (repo-loop @ loop-engine-use-case name)
    name: 'repo-doctor)
  (.def (repo-profile @ loop-engine-profile use-case)
    use-case: repo-loop))
#+end_src

Then the guide should show the resolved =users= profile, doctor status,
feature facts, CI/CD intents, sandbox mounts, loop strategy facts, human audit
items, and runtime handoff manifest.

** 10.06.21.60 Docs Chat Lesson
Zerolang's docs chat route is intentionally constrained:
- it says Zero is pre-1 and experimental;
- it tells the agent to use documentation files as ground truth;
- it exposes read-only =bash= and =readFile= tools over a virtual docs workspace;
- it forbids write/create/delete shell operations;
- it uses tool steps with a fixed maximum step count;
- it rate-limits the public chat surface.

=poo-flow= can translate this boundary for an agent configuration assistant:
- mount only project docs, =user-interface=, and selected profile receipts;
- give the agent read-only inspect tools by default;
- expose write operations only as checked POO patch plans;
- make sandbox/resource policy explicit in the profile receipt;
- keep runtime execution behind Marlin handoff, not in the docs assistant.

This connects directly to the current =object-extension= sandbox profile work.
The realistic mount example should become an agent-facing receipt, not only a
Scheme snippet:
- project workspace, read-write, marker =gerbil.pkg=;
- =.data= research checkouts, read-only;
- semantic cache, read-write;
- user module config, read-only;
- secrets from environment, read-only.

** 10.06.21.70 Implementation Gaps
The next implementation should focus on configurable loop engineering, not more
syntax and not a source-text patch system.

- Loop graph inspection
   - Input: root user-interface declaration, selected =users= profile, and loop
     strategy plan objects.
   - Output: concise text by default, structured POO/alist/JSON on request.
   - Data: selected modules, feature facts, CI/CD intents, sandbox profile
     names, mounts, loop patterns, level ceiling, human gates, validation mode,
     and runtime handoff owner.

- Loop doctor
   - Reuse existing profile/profile-set doctor objects and loop strategy
     validation.
   - Present diagnostics as repair contracts with location, problem, evidence,
     fix intent, and constraints.
   - Keep warning output visible without making every warning fatal.

- Loop strategy resolver
   - Resolve a concrete loop strategy object from profile/module facts and
     configured loop descriptors.
   - Select actionable patterns through level ceiling, priority, gate, budget,
     sandbox, and validation policy.
   - Return the next recommended pattern as data, not as a scheduler side
     effect.

- Loop manifest projection
   - Project the selected loop strategy into a runtime handoff manifest.
   - Keep Scheme as control plane and policy owner.
   - Emit command/resource/state requirements for Marlin or another backend.
   - Never turn the Scheme user-interface layer into a timer, worktree manager,
     connector client, or process runner.

- Config contract check
   - Verify that =user-interface/init.ss= remains a valid configuration surface
     for the maintained =users= profile and expected loop facts.
   - Catch stale snippets in documentation and generated examples.
   - Treat the root init as a safe configuration surface that agents may edit
     through checked interfaces.

- Human audit loop
   - Project config/profile/loop/runtime facts into human review items.
   - Record approval, rejection, escalation, requested changes, and rationale.
   - Keep human decisions as loop state, not as ad hoc comments in config.

** 10.06.21.80 Near-Term Milestone
The next concrete milestone should be:

#+begin_src sh
poo-flow loop inspect user-interface/init.ss
poo-flow loop doctor user-interface/init.ss
poo-flow loop resolve user-interface/init.ss --profile users
poo-flow loop manifest user-interface/init.ss --profile users
#+end_src

If the CLI is not ready, the same milestone can be delivered first as Scheme
entrypoints and tests:
- =pooFlowUserInterfaceLoopInspect=
- =pooFlowUserInterfaceLoopDoctor=
- =pooFlowUserInterfaceLoopResolve=
- =pooFlowUserInterfaceLoopRuntimeManifest=

Success criteria:
- agents can inspect without dumping raw source;
- agents can resolve a configured loop without mutating files;
- humans can review through a human audit loop;
- diagnostics explain the safest next operation shape;
- runtime work remains outside Scheme.

** 10.06.21.90 Decision
Do not copy Zerolang's language, CLI, or compiler graph. Extract the
agent-facing engineering boundary and implement it in =poo-flow='s own design:

The POO object graph is the semantic authority. =init.ss= is the validated
configuration surface, not the human projection layer. Loop strategy objects
are the configurable agent engineering surface. Agents inspect, doctor, resolve,
and project loop handoff manifests from composed POO facts. Humans participate
through a human audit loop that reviews facts, diagnostics, plans, manifests,
and receipts. Runtime execution remains an explicit handoff to the backend.
