#+title: Bazel Runtime-C External Build Design
#+author: poo-flow contributors
#+date: 2026-07-12
#+startup: overview

* Status

Accepted RFC and design specification.

This document defines an optional Bazel/Starlark build surface that completely
replaces =bindings/runtime-c/Makefile= while preserving =build.ss= as the
authoritative native Scheme build system.

* Objective

Provide a reproducible external build choice for the Scheme-to-C-to-Python
runtime boundary.  The Bazel path MUST build the runtime contract, C library,
C tests, benchmarks, and Python runtime integration without invoking Make.

Bazel MUST NOT replace or reproduce the native Scheme build semantics owned by
=build.ss=.

* Ownership Boundary

** Scheme build authority

=build.ss= remains the sole owner of Gerbil/Scheme build semantics, including:

- module and package stages;
- build specifications;
- Scheme compilation;
- Scheme-side cleaning;
- worker configuration;
- Scheme-side build receipts.

Bazel MUST NOT interpret or duplicate the Scheme module topology.  The Scheme
build MUST remain usable without Bazel.

** Contract authority

The existing Scheme contract generator remains the sole owner of generated
contract content.  Bazel invokes that generator as a declared tool and consumes
its declared outputs.  Bazel does not contain a second contract generator.

** External action-graph authority

Bazel owns the external action graph formerly expressed by
=bindings/runtime-c/Makefile=:

1. read the authoritative runtime schema;
2. invoke the Scheme contract generator;
3. compile and link the runtime C library;
4. build and run the C tests;
5. build and smoke-test the C benchmarks;
6. provide runtime artifacts and metadata to Python runtime tests.

There is no Make fallback after migration.  Once the exit criteria pass, the
Makefile and all public references to its commands are removed.

* Architecture

#+begin_src text
build.ss (Scheme build authority)
              |
              v
Scheme contract generator
              |
              v
generate_runtime_v0_contract
              |
              v
runtime_v0_generated_sources
              |
              v
runtime_c_library
       /             \
      v               v
runtime_c_tests   runtime_c_benchmarks
      |
      v
python_runtime_ffi_tests
#+end_src

The arrow from =build.ss= describes the Scheme-owned semantic boundary, not a
requirement that Bazel redirect ordinary Scheme compilation through itself.

* Bazel Target Contract

Target labels may be adjusted to match repository conventions during
implementation.  Their ownership and dependency boundaries are normative.

** runtime_v0_schema

A source-only target containing the authoritative Org/schema inputs.  It
performs no generation and exposes no files derived from an earlier local
build.

** generate_runtime_v0_contract

A Starlark rule or macro-backed action that invokes the existing Scheme
contract generator.

Its declared inputs include:

- the generator executable and Scheme modules;
- the authoritative schema;
- generation parameters;
- the hermetic Gerbil execution environment.

It declares every generated file and writes only to the Bazel output tree.
Generator failure, a missing declared file, an undeclared required file, or a
contract-version mismatch is a hard action failure.

** runtime_v0_generated_sources

A provider boundary that aggregates generated C sources, headers, contract
metadata, and version information.  Consumers depend on this provider instead
of a private output-directory layout.

The provider MUST remain limited to runtime-contract artifacts.  It MUST NOT
model Scheme build stages.

** runtime_c_library

A native Bazel C/C++ library target consuming
=runtime_v0_generated_sources=.  It explicitly declares sources, public
headers, include paths, compilation definitions, and link dependencies.

When both forms are required, static and shared libraries are exposed as
separate targets.  C compilation uses Bazel toolchains rather than custom shell
actions that reproduce Makefile compiler commands.

** runtime_c_tests

Native =cc_test= targets cover the C ABI, generated contract, and runtime
behavior.  Fixtures and runtime files are declared through target dependencies
and =data=.  Tests MUST NOT discover artifacts through working-directory
assumptions.

** runtime_c_benchmarks

Explicit benchmark binaries and smoke-test targets.  Benchmark construction is
fully migrated from Make, but performance runs are not part of the default unit
test gate.  The initial exit gate requires successful builds and a bounded
smoke run, not portable performance thresholds.

** python_runtime_ffi_tests

Python runtime integration tests explicitly depend on the runtime library,
contract metadata, and required dynamic-library runfiles.  They validate the
Scheme projection and contract boundary without locating products through
Makefile paths or undeclared local installations.

* Starlark Design Rules

Starlark defines only:

- the contract-generation action;
- the generated-contract provider;
- small macros that remove repeated target declarations;
- bounded platform selection not covered by standard toolchains.

Native Bazel rules own ordinary C and Python compilation and testing.  Starlark
MUST NOT implement Scheme compilation semantics or replace a native rule with
an opaque shell action when a standard rule suffices.

* Hermeticity and Reproducibility

Every generator input, executable, environment setting, and output is declared.
The generator writes only within the Bazel sandbox and output tree.

The build MUST NOT depend on:

- an ambient user Gerbil package cache;
- implicit =PATH= lookup for the generator or compiler;
- previously generated files in the source tree;
- a previously installed runtime library;
- Make-created directories or environment variables.

The rule constructs the required Gerbil load path from declared inputs.  C
compilation and platform behavior are delegated to Bazel toolchains.  Linux and
macOS differences use toolchain constraints or bounded =select()= expressions,
not scattered shell conditionals.

Changes to the schema, generator, generation parameters, C sources, or relevant
toolchain inputs MUST invalidate the corresponding actions and downstream
dependents.

* Failure Model and Diagnostics

A Scheme generator failure is a hard Bazel action failure, and its diagnostic
output is preserved.  Missing or unexpected required artifacts and contract
version mismatches are also hard failures.

Python integration failures report the selected contract metadata, runtime
library or runfile, and the relevant ABI or version mismatch.

No rule silently falls back to Make, a source-tree artifact, or a locally
installed library.

Cleaning is not reimplemented as an application-specific target.  Bazel owns
its output tree and standard clean behavior.  =build.ss= independently retains
Scheme-side cleaning semantics.

* Delivery Slices

** Slice 1: Bazel foundation

- establish Bazel workspace or module metadata;
- declare the minimum supported Bazel version;
- declare the runtime schema and Scheme generator inputs.

** Slice 2: Contract generation

- implement the contract-generation rule;
- implement the generated-contract provider;
- prove clean-checkout generation and cache invalidation.

** Slice 3: Native C graph

- add native C library targets;
- migrate C contract and runtime tests;
- migrate benchmark construction and bounded smoke tests.

** Slice 4: Python runtime integration

- connect Python integration tests to declared runtime artifacts;
- remove assumptions about Make output paths and local installations;
- validate the Scheme projection, contract metadata, and ABI boundary.

** Slice 5: Make removal

- replace CI, documentation, test, and script references to Make commands;
- delete =bindings/runtime-c/Makefile=;
- rerun every exit criterion after deletion.

No slice redirects Scheme module compilation away from =build.ss=.

* Exit Criteria

The Makefile is removed only when all criteria pass:

1. A clean checkout generates the runtime-v0 contract through Bazel.
2. Required static and shared runtime libraries build with native Bazel
   toolchains.
3. C contract and runtime tests pass.
4. C benchmarks build and complete a bounded smoke run.
5. Python runtime FFI and projection integration tests pass against
   Bazel-declared artifacts.
6. A schema change invalidates and rebuilds generation, the C library, and
   affected Python tests.
7. Generated contract and ABI behavior are equivalent to the former Makefile
   path.
8. Repository CI, documentation, tests, and scripts no longer reference public
   Makefile commands.
9. Deleting =bindings/runtime-c/Makefile= leaves the complete Bazel acceptance
   path green.
10. The independent =build.ss= Scheme compile and test path remains green and
    has no Bazel dependency.

* Non-Goals

- replacing =build.ss= or the Gerbil package build system;
- expressing Scheme module topology in Starlark;
- making Bazel mandatory for Scheme-only development;
- preserving Make as a compatibility wrapper after acceptance;
- treating benchmark performance numbers as portable pass/fail thresholds in
  the first migration.
