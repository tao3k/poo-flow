#+title: AC-04 Read-Only Shadow Projection Design

* Status

Approved in conversation on 2026-07-11.  This document is the written design
gate; implementation and qualification remain pending.

* Objective

AC-04 compares the current control-plane projection with a validated Canonical
Organization Bundle without changing runtime execution.  Both sides project
into stable typed semantic facts.  A pure comparator produces one deterministic
internal receipt that records equivalence and precise differences.

This is migration evidence, not a second runtime, an activation path, or a
public compatibility API.  No external effect is executed for comparison.

* Version policy

The Bundle and facet schemas remain pre-release until the complete agentic
control-plane chain is stable.  AC-04 performs a direct schema cutover:

- =poo-flow.organization-bundle.v2= becomes
  =poo-flow.organization-bundle.draft.3=;
- =poo-flow.organization-facet.v1= becomes
  =poo-flow.organization-facet.draft.3=;
- each incompatible pre-release change increments =draft.N=;
- =poo-flow.organization-bundle.v1= is reserved for the final suite freeze.

The final freeze requires stable Bundle and Kernel semantics, shadow
qualification, C ABI control/hot/bulk lanes, the production-capable Python CFFI
validation runtime, Cedar projection, Lean/Evidence linkage, performance gates,
conformance vectors, and canonical cutover.  Shadow equivalence alone never
asserts v1 conformance.

* Architecture

#+begin_src mermaid
flowchart LR
  Current["Current control-plane projection"] --> CurrentFacts["Current typed semantic facts"]
  Bundle["Validated Bundle draft.3"] --> BundleFacts["Bundle typed semantic facts"]
  Profile["Explicit comparison profile"] --> Comparator["Pure read-only semantic comparator"]
  CurrentFacts --> Comparator
  BundleFacts --> Comparator
  Comparator --> Receipt["Internal deterministic shadow receipt"]
  Receipt --> Gate["Migration qualification only"]
#+end_src

The implementation owner is
=src/semantic/organization-bundle-shadow.ss=.  The module is internal and is not
re-exported through the canonical public facade.  Existing projection owners
may supply typed facts, but do not own comparison semantics.

* Typed semantic fact contract

A shadow fact is a POO-native internal value with the following stable slots:

| Slot | Meaning |
|------+---------|
| =facet= | one of Organization, Authority, Context, Protocol, Evidence |
| =path= | canonical semantic path within that facet |
| =key= | stable property or relation key |
| =value= | canonical semantic value |

Facts describe semantic meaning only.  Runtime addresses, timestamps, cache
state, process identity, incidental allocation, source presentation order, and
other operational metadata are excluded.  Construction order cannot influence
the normalized fact sequence or receipt.

The current-path producer and Bundle-path producer must satisfy the same fact
contract.  Raw POO source text and whole-object digests are not accepted as
substitutes for semantic facts: text is presentation-sensitive, while a digest
cannot identify the location or class of a difference.

* Inputs and comparison profile

The comparator accepts exactly:

1. a validated immutable Bundle Kernel state;
2. current-path typed semantic facts;
3. an explicit comparison profile naming the facets and semantic paths in
   scope.

The profile prevents a partial producer from appearing equivalent by omission.
An unknown facet, malformed path, duplicate =facet/path/key= identity, or
unstable/non-canonical value fails closed.  Inputs are normalized by facet
order, then path, then key before comparison.

* Shadow receipt

The receipt is an internal POO-native value containing:

| Slot | Meaning |
|------+---------|
| =schema= | shadow-receipt draft schema |
| =bundle-identity= | compared Bundle identity |
| =bundle-epoch= | compared immutable epoch |
| =compared-facets= | normalized profile coverage |
| =matched-paths= | paths whose scoped facts are equal |
| =missing-current= | facts required by Bundle but absent on current path |
| =missing-bundle= | current facts absent from Bundle projection |
| =mismatched-values= | same identity with unequal canonical values |
| =accepted?= | input and coverage validation succeeded |
| =equivalent?= | accepted and all three difference sets are empty |
| =v1-conformant?= | always =#f= during draft qualification |
| =diagnostics= | deterministic typed rejection details |

Difference entries preserve =facet/path/key= identity plus expected and
observed semantic categories.  They are ordered by canonical facet order,
path, key, and diagnostic code.  Missing-current, missing-bundle, and value
mismatch remain distinct; one category is never collapsed into another.

* Purity and mutation boundary

Shadow projection and comparison are pure functions.  They cannot:

- mutate or advance the Bundle Kernel;
- activate a Bundle or change the canonical path;
- call a runtime adapter or cross the future C ABI;
- execute, replay, or suppress an external effect;
- write caches, receipts, files, or process-global state;
- expose a new public authoring or compatibility surface.

The receipt may be consumed later by an explicit qualification harness.  Such a
harness owns persistence; the semantic comparator does not.

* Failure semantics

Input validation precedes equality comparison.  Invalid Bundle state, unknown
facet, uncovered profile path, duplicate fact identity, malformed path, or
unstable value returns =accepted? #f=, =equivalent? #f=, and deterministic
diagnostics.  It is not represented as an ordinary semantic difference.

Valid unequal inputs return =accepted? #t= and =equivalent? #f= with precise
difference sets.  Valid equal inputs return both flags true.  In every AC-04
case =v1-conformant?= remains =#f=.

* Qualification

The feature is complete only when focused tests prove:

1. Bundle and facet schemas cut over to =draft.3= with no stale public version;
2. equal facts produce an accepted, equivalent receipt;
3. input construction order does not affect normalized output or receipt;
4. missing-current, missing-bundle, and value mismatch are reported precisely;
5. duplicate identity, unstable value, unknown facet, and incomplete profile
   fail closed;
6. Bundle Kernel identity, epoch, status, and digest remain unchanged;
7. AC-01, AC-02, and AC-03 focused suites remain green;
8. every receipt explicitly reports =v1-conformant? #f=;
9. the shadow owner is absent from the public facade;
10. Org parsing and =git diff --check= pass.

* Out of scope

AC-04 does not implement runtime execution, C ABI layouts, Python CFFI,
downstream Rust or TypeScript adapters, Cedar evaluation, Lean proof generation,
evidence persistence, dual writes, dual effects, public compatibility routes,
or the final Bundle v1 freeze.

* Removal

Shadow machinery is intentionally temporary.  AC-11 removes the comparator,
fact producers used only for dual projection, and migration-only receipts after
the Bundle path becomes canonical and the final qualification pack is accepted.
Removal does not affect the public API because AC-04 exports none.

* Delivery boundary

The complete AC-04 feature commit will include the approved design, an
implementation plan, the =draft.3= schema cutover, internal semantic owner,
focused scenarios, corrected owner/status documentation, and a qualification
receipt.  Pre-existing build and CLI-support changes remain outside that
path-scoped commit.
