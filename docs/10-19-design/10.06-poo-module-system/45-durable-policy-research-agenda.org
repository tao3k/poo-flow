* 10.06.45 Durable Policy Research Agenda
** 10.06.45.10 Status
Draft for discussion.

This document strengthens the durable-policy thread started in [[file:43-session-memory-recovery-rfc.org][43 Session Memory Durable Policy RFC]].
The goal is not to copy Codex, Claude Code, OpenRath, Temporal, Azure Durable Functions, or any agent-memory paper.
The goal is to extract the durable invariants that fit =poo-flow=: functional flow values, POO-native policy objects, explicit session identity, and Rust/Marlin-owned durable runtime machinery.

** 10.06.45.20 Main Conclusion
Durable should be a first-class capability, not a helper called "recovery".

The correct shape is:

#+begin_src text
POO durable policy
  -> bounded Scheme receipt
  -> runtime fact log
  -> derived indexes
  -> replay/backfill/repair jobs
  -> resumable session, memory, workflow, sandbox, and artifact state
#+end_src

In user-facing Scheme, durable remains a POO object that can be inherited, specialized, and validated.
In the runtime boundary, durable becomes a set of explicit receipts and manifests.
In Rust/Marlin, durable becomes append-only facts, checkpoint materialization, leases, watermarks, indexes, and repair behavior.

This means =durable-policy= is shared by =session= and =memory= instead of being owned by one module.
Recovery is a runtime procedure derived from durable policy plus stored facts.

*** 10.06.45.30.10 Codex Evidence From =.data/codex=
The local Codex checkout is at =.data/codex=, revision =cfead68e5d= when this audit was written.

Relevant owners:
- =codex-rs/rollout/src/recorder.rs=
- =codex-rs/state/src/runtime/backfill.rs=
- =codex-rs/state/src/runtime/recovery.rs=
- =codex-rs/state/src/runtime/memories.rs=
- =codex-rs/state/src/runtime/threads.rs=
- =codex-rs/rollout-trace/src/model/session.rs=
- =codex-rs/core/src/agent/control/spawn.rs=

Key evidence:
- Rollout JSONL is the canonical session fact log.
- Runtime SQLite is a derived state/index layer, not the only source of truth.
- Backfill persists progress through watermarks; =checkpoint_backfill= records a running status and last watermark.
- Runtime DB corruption repair backs up one broken database and its sidecars instead of destroying unrelated runtime state.
- Memory is a separate durable job pipeline with startup claims, leases, retry bounds, stale-source detection, and stage/global consolidation.
- Parent/child sessions are persisted as directional spawn edges; =upsert_thread_spawn_edge= makes child id the stable edge key.
- Trace reduction models sessions/threads as typed runtime objects with origin, execution window, and terminal state.

Design lesson:
Codex shows that durable agent systems need both a fact log and repairable indexes.
It is not enough to persist "latest state"; derived state must be rebuildable, claimable, and bounded.

What =poo-flow= should do better:
- Make durable policy explicit at authoring time, not hidden in runtime conventions.
- Attach durable behavior to session, memory, workflow, sandbox, and artifact policies through POO inheritance/composition.
- Validate resume/replay constraints before projection.
- Keep Scheme objects high-level and POO-native, but project hot receipts into fixed =defstruct= values before ABI alists.

*** 10.06.45.30.20 OpenRath Evidence From =.data/OpenRath=
The local OpenRath checkout is at =.data/OpenRath=, revision =ab884ab= when this audit was written.

Relevant owners:
- =src/rath/session/session.py=
- =src/rath/session/persistence/writer.py=
- =src/rath/flow/workflow.py=
- =src/rath/flow/agent.py=
- =src/rath/memory/adapters/local.py=
- =tests/session/test_lineage.py=

Key evidence:
- =Session= is the flowing value.
- =Agent.forward= transforms a =Session= into a new =Session= through =run_session_loop=.
- =Session.fork= duplicates transcript, preserves lineage, and shares sandbox reference.
- =Session.merge= concatenates rows, sums usage, stamps both parent sessions, and keeps the first session sandbox.
- =SessionWriter= is an append-only JSONL writer with a partial-file marker and atomic rename on close.

Design lesson:
OpenRath gets the functional shape right:

#+begin_src text
Session -> Agent/Workflow -> Session
#+end_src

This is close to our Funflow direction: flows compose values, while objects carry policy and metadata.

What =poo-flow= should do better:
- Fork and merge behavior must be policy-governed, not hard-coded.
- Sandbox sharing must be declared through session/sandbox policy.
- Persistence must include explicit checkpoint/replay semantics, not only append-only session files.
- Memory and sub-agent policy must be more precise than "inject and commit around forward".

*** 10.06.45.30.30 Claude / Anthropic Evidence
Official Claude Code documentation gives three useful durable signals:

1. Checkpointing is a user-visible safety feature.
   Claude Code creates prompt/edit checkpoints, persists them across resumed conversations, supports restore of code, conversation, or both, and supports targeted summarization.
   It also documents the limits: shell-command file changes and unrelated external changes are not fully tracked.
   Source: [[https://code.claude.com/docs/en/checkpointing][Claude Code Checkpointing]].

2. Subagents are identity-bearing contexts.
   Claude Code subagents run in separate context windows with separate tool access and permissions.
   A resumed subagent keeps its prior conversation, tool calls, results, and reasoning; the parent can address it by agent id through =SendMessage=.
   Subagent transcripts are stored separately under the project/session subagent directory.
   Source: [[https://code.claude.com/docs/en/sub-agents][Claude Code Subagents]].

3. Memory is context, not enforcement.
   Claude Code distinguishes user/project/org instructions from auto memory, and says these are loaded as context rather than hard enforcement.
   Enforcement belongs in hooks such as =PreToolUse=.
   The Claude API memory tool is explicitly client-side: Claude requests file operations, while the application owns storage, path policy, and returned tool results.
   Sources: [[https://code.claude.com/docs/en/memory][Claude Code Memory]] and [[https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool][Claude Platform Memory Tool]].

Anthropic's "Building effective agents" gives the control-plane principle:
- workflows are predefined code paths;
- agents dynamically direct tool use;
- successful production systems prefer simple composable patterns;
- agents need environment feedback, checkpoints, sandbox testing, and guardrails.
Source: [[https://www.anthropic.com/engineering/building-effective-agents][Building effective agents]].

Design lesson:
Claude's surface is strong at user ergonomics, but its documented checkpointing is explicitly not a full durable runtime.
=poo-flow= should learn the user affordance, not stop at it.
Our durable policy must cover tool-side effects, shell/runtime work, sandbox placement, memory writes, and sub-agent communication, because those are exactly the places a simple local checkpoint cannot fully recover.

*** 10.06.45.30.40 Durable Execution Systems
Temporal and Azure Durable Functions provide the durable execution baseline.

Temporal:
- a workflow execution is durable and recoverable;
- each execution has local state and communicates through signals/activities;
- replay checks generated commands against event history;
- after failure, execution resumes from the latest recorded event.
Sources: [[https://docs.temporal.io/workflow-execution][Temporal Workflow Execution]] and [[https://docs.temporal.io/encyclopedia/event-history][Temporal Event History]].

Azure Durable Functions / Durable Task:
- orchestration instances have stable instance ids;
- progress is checkpointed at =await= / =yield= boundaries;
- event sourcing records actions instead of dumping full runtime state;
- orchestrators must be deterministic because they replay.
Sources: [[https://learn.microsoft.com/en-us/azure/durable-task/common/durable-task-orchestrations][Durable Orchestrations]] and [[https://learn.microsoft.com/en-us/azure/durable-task/common/durable-task-code-constraints][Orchestrator Code Constraints]].

Design lesson:
The strong durable pattern is not heap snapshotting.
It is event history plus deterministic replay plus explicit side-effect boundaries.

For =poo-flow=:
- Scheme flow composition should be replay-safe by construction where possible.
- Runtime effects should sit behind named activities/actions/handoffs.
- Session ids, loop-run ids, checkpoint ids, artifact ids, and sandbox handles must be durable addresses.
- Durable policy must decide when a flow can replay, resume, compensate, or require human intervention.

*** 10.06.45.30.50 Recovery And Long-Lived Transaction Papers
ARIES provides the database-recovery baseline.
It uses write-ahead logging, log sequence numbers, restart recovery, redo/undo, and partial rollback.
Source: [[https://dl.acm.org/doi/10.1145/128765.128770][ARIES ACM record]], with accessible PDF mirrors such as [[https://web.stanford.edu/class/cs345d-01/rl/aries.pdf][Stanford ARIES PDF]].

Sagas provide the long-lived-transaction baseline.
A saga decomposes a long transaction into smaller transactions with compensating actions.
Source: [[https://dl.acm.org/doi/10.1145/38713.38742][Sagas ACM record]], with accessible PDF mirrors such as [[https://www.cs.cornell.edu/andru/cs711/2002fa/reading/sagas.pdf][Cornell Sagas PDF]].

Design lesson:
Agent workflows are long-lived, side-effect-heavy computations.
They need compensation and repair semantics, not only "retry".

For =poo-flow=:
- Each durable action should declare whether it is replayable, idempotent, compensatable, or terminal.
- A workflow task should be able to project a compensation receipt.
- A failed child session should produce structured repair facts for the parent instead of mutating parent context directly.

*** 10.06.45.30.60 Workflow Pattern Literature
The workflow-patterns literature separates control-flow, data, resource, and exception concerns.
It is useful because =poo-flow= is not only an agent-loop system; it is also a workflow, CI/CD, deploy, sandbox, and policy system.

Sources:
- [[https://www.workflowpatterns.com/documentation/][Workflow Patterns documentation]]
- [[https://link.springer.com/article/10.1023/A:1022883727209][Workflow Patterns, Distributed and Parallel Databases]]
- [[https://pure.tue.nl/ws/files/2053121/613310.pdf][Workflow Patterns publisher PDF]]

Design lesson:
Durable cannot be only "save session state".
It must cover:
- control flow state;
- data flow state;
- resource ownership;
- exception and compensation state;
- runtime participant identity.

For =poo-flow=, this argues for a durable policy that can be inherited by workflow, session, memory, sandbox, and CI/CD modules.

*** 10.06.45.30.70 Agent Memory Papers
Generative Agents uses observation, memory stream, reflection, and planning to sustain believable long-horizon behavior.
Source: [[https://arxiv.org/abs/2304.03442][Generative Agents]].

MemGPT frames LLM memory as virtual context management with tiers inspired by operating systems.
Source: [[https://arxiv.org/abs/2310.08560][MemGPT]].

Reflexion stores verbal feedback in episodic memory to improve later trials without model weight updates.
Source: [[https://arxiv.org/abs/2303.11366][Reflexion]].

Voyager builds an executable skill library and uses feedback/self-verification to improve long-running exploration.
Source: [[https://arxiv.org/abs/2305.16291][Voyager]].

Design lesson:
Memory durability is not one blob of transcript.
It is a policy-governed pipeline:
- observation/fact capture;
- recall selection;
- reflection/consolidation;
- usage accounting;
- skill/artifact promotion;
- retention and deletion.

This matches Codex's local memory job pipeline and should become explicit in =poo-flow= as =memory-policy= inheriting or referencing =durable-policy=.

** 10.06.45.40 Design Approaches
*** 10.06.45.40.10 Approach A: Git/Checkpoint Wrapper Only
This copies the visible Claude Code checkpoint affordance.

Pros:
- simple user mental model;
- useful for code-edit rollback;
- easy to explain.

Cons:
- does not cover shell side effects, sandbox state, sub-agent transcripts, memory writes, external tools, or runtime jobs;
- cannot express replay/compensation semantics;
- not enough for project/root/child session graphs.

Verdict: useful as one backend capability, but too weak as the durable architecture.

*** 10.06.45.40.20 Approach B: Session Transcript Only
This copies the simplest OpenRath-style append-only session persistence.

Pros:
- matches functional =Session -> Flow -> Session= composition;
- easy to inspect;
- supports lineage and replay input.

Cons:
- derived indexes and repair jobs are missing;
- memory consolidation is bolted on;
- workflow/resource state must be reconstructed ad hoc;
- partial failures need stronger semantics.

Verdict: necessary, but incomplete.

*** 10.06.45.40.30 Approach C: Event Log + Checkpoint + Derived Index
This combines:
- OpenRath's session-as-value model;
- Codex's fact-log/index/job split;
- Temporal/Azure durable replay discipline;
- ARIES/Saga repair and compensation lessons;
- agent-memory paper pipelines.

Pros:
- strong enough for long-lived agent projects;
- supports parent/child/session graph recovery;
- handles memory as a durable pipeline;
- lets Scheme stay POO-native and high level;
- lets Rust/Marlin own IO, concurrency, storage, and repair.

Cons:
- requires careful receipt contracts;
- must define deterministic/replay-safe boundaries;
- needs performance gates for projection and validation.

Verdict: recommended.

** 10.06.45.50 POO Flow Capability Model
Durable should appear in Scheme as a composable capability object family.

Conceptual object families:
- =durable-policy=: fact log, checkpoint cadence, replay mode, repair mode, retention, backend owner.
- =checkpoint-policy=: what to capture at turn/task/session boundaries.
- =journal-policy=: append-only fact categories and redaction rules.
- =index-policy=: which derived indexes are required and rebuildable.
- =resume-policy=: when to auto-resume, ask user, fork, or fail closed.
- =repair-policy=: backup, rebuild, retry, compensate, quarantine, or stop.
- =memory-policy=: recall, extraction, consolidation, ranking, retention, usage accounting.
- =session-policy=: root/child identity, lineage, context grants, communication grants, sandbox grants.
- =workflow-policy=: task state, retry, compensation, parallel branches, join semantics.
- =artifact-policy=: stable handles, content-addressed storage, provenance, retention.
- =sandbox-policy=: inherit, clone, share, isolate, or replace placement.

These should be POO objects, not option trees.
C3/C4-style composition decides inherited slot precedence.
Validation decides whether the resulting policy is legal.
Projection produces bounded runtime receipts.

** 10.06.45.60 Runtime Receipt Shape
The runtime should receive fixed, bounded shapes.
The Scheme side can use native POO while authoring and validating, but hot receipts should use =defstruct= internally before ABI projection.

Representative receipt fields:

#+begin_src scheme
(defstruct durable-policy-receipt
  (policy-id
   project-id
   root-session-id
   session-id
   parent-session-id
   loop-run-id
   checkpoint-policy
   journal-policy
   index-policy
   resume-policy
   repair-policy
   memory-policy-ref
   sandbox-policy-ref
   artifact-policy-ref
   diagnostics)
  transparent: #t)
#+end_src

Bounded projection:

#+begin_src scheme
(def (durable-policy-receipt->alist r)
  ...)
#+end_src

This follows the project direction:
1. user authoring uses POO;
2. generated runtime receipt uses =defstruct=;
3. manifest / JSON / Marlin handoff uses bounded alist serialization.

** 10.06.45.70 Durable State Categories
The durable runtime should separate these categories.

1. Fact log:
   append-only events, session turns, tool requests/results, memory facts, spawn edges, sandbox events, artifact events.

2. Checkpoint:
   bounded materialized state for resume, not a whole heap dump.

3. Derived index:
   queryable views over sessions, children, artifacts, memory summaries, jobs, and latest status.

4. Job state:
   leases, watermarks, retry counters, ownership token, last error, last success watermark.

5. Memory store:
   durable notes, summaries, reflections, skill artifacts, usage counters, retention metadata.

6. Repair journal:
   corruption backup, rebuild attempt, compensation execution, quarantine reason, human-decision receipt.

7. Communication ledger:
   parent/child messages, sibling channels, request/response ids, permission decisions.

8. Sandbox ledger:
   placement, handle id, shared/clone/isolate mode, active resources, close/reopen facts.

** 10.06.45.80 Milestone Ladder
*** 10.06.45.80.10 P0: Evidence And Contract Closure
- Keep 43 RFC as the architecture proposal.
- Keep this 45 document as the research evidence map.
- Add any missing local evidence from `.data/codex`, `.data/OpenRath`, Claude docs, and durable papers before implementation.
- Decide the final public names: =durable-policy=, =checkpoint-policy=, =journal-policy=, =resume-policy=, =repair-policy=.

*** 10.06.45.80.20 P1: POO Object Layer
- Add POO-native durable objects under the module-system object layer.
- Let session, memory, workflow, sandbox, and artifact policies inherit/reference durable prototypes.
- Validation must reject fake durable profiles: no checkpoint store, no journal owner, no resume identity, no repair mode.

*** 10.06.45.80.30 P2: Receipt Projection
- Add =defstruct= receipts for durable policy, checkpoint policy, session lineage, memory job, and communication channel.
- Provide explicit =->alist= projection only at manifest/ABI boundaries.
- Add performance gates for user-interface projection.

*** 10.06.45.80.40 P3: Runtime Store Contract
- Define the Rust/Marlin-owned store contract:
  fact log, checkpoint store, derived index, job store, memory store, repair journal.
- Scheme should only validate and project the contract.
- Runtime owns filesystem, SQLite or other storage, locks, leases, corruption handling, and backfill.

*** 10.06.45.80.50 P4: Recovery And Replay Tests
- Simulate process crash after fact append but before derived index update.
- Simulate partial checkpoint write.
- Simulate memory job lease expiry.
- Simulate child-session failure and parent resume.
- Simulate sandbox handle loss and repair policy.
- Simulate non-deterministic workflow branch rejection.

*** 10.06.45.80.60 P5: Multi-Agent Session Graph
- Make project/root/child session ids first-class in receipts.
- Persist parent-child and sibling communication edges.
- Add policy validation for context grants, history grants, sandbox sharing, and channel permissions.
- Support resumable child sessions by id.

*** 10.06.45.80.70 P6: Memory Pipeline
- Treat memory extraction/consolidation as durable jobs.
- Add watermarks, usage counts, stale-source checks, and retention policy.
- Separate user-authored memory, auto memory, reflections, summaries, and promoted skills/artifacts.

*** 10.06.45.80.80 P7: Workflow And CI/CD Integration
- Apply durable policy to workflow/funflow tasks.
- Each task should declare replay/compensation/idempotency class.
- CI/CD pipelines should produce durable task receipts, artifact receipts, and compensation/cleanup receipts.

** 10.06.45.90 Research Questions
1. Which parts of durable policy should be built into =session-core= and which should be shared by the module-system object layer?
2. Should =memory-policy= inherit =durable/default= directly, or reference it through a =durable-ref= slot?
3. What is the smallest replay-safe Scheme surface we can validate without over-constraining user workflows?
4. What are the durable action classes we need first: =replayable=, =idempotent=, =compensatable=, =terminal=, =manual=?
5. How much checkpoint state should be stored in Scheme receipts versus runtime-owned manifests?
6. Should sibling communication be synchronous message passing, append-only mailbox facts, or both?
7. What storage backend should Rust/Marlin use first for a local durable prototype: JSONL + SQLite, only SQLite, or pluggable store traits?
8. Which observability hooks must fire for every durable event: fact append, checkpoint write, replay start/end, repair start/end, job claim, job finish, compensation?

** 10.06.45.100 Proposed Direction
Use Approach C.

Treat durable as a shared POO capability with runtime-backed fact logs, checkpoints, derived indexes, repair jobs, and memory pipelines.

The user should see concise POO-native policy and module forms.
The Scheme implementation should validate object composition and project bounded receipts.
The runtime should own storage, replay, backfill, leases, corruption repair, and cross-session delivery.

This makes durable a real differentiator for =poo-flow=:
not just "resume a chat", but a policy-governed substrate for long-running agent projects, multi-agent session graphs, workflow execution, memory consolidation, sandbox isolation, CI/CD, deploy, and recovery.
