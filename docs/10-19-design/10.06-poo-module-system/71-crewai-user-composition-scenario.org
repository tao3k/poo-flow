#+title: CrewAI-Style User Composition Scenario
#+startup: overview

* Purpose

This document records the CrewAI-style user composition scenario. CrewAI is a reference framework for collaborative agents, crews, tasks, and flows; it is not the source of truth for POO Flow. The POO Flow fact surface remains profile objects, strategy composition, graph/loop facts, runtime handoff receipts, and Lean proof obligations.

The goal is to show that a CrewAI-like crew/flow can be expressed as a small reusable profile library plus one composition case:

#+begin_src scheme
(use-composition crewai-research-flow-composition
  (modules
    (use-profile crewai as crew))
  (stage production
    (compose
      (profile crew agent)
      (profile crew task)
      (profile crew crew)
      (profile crew sequential-process)
      (profile crew flow-state)
      (profile crew flow-router)
      (profile crew flow-persist)
      (profile crew guardrail)
      (profile crew human-input)
      (profile crew observability)
      (profile crew runtime-handoff))
    (graph crewai-research-flow-graph)
    (loop #:fuel 6 #:exit final-output)
    (prove agent-tool-scope-contained
           task-dependencies-closed
           crew-members-declared
           task-order-respects-dependencies
           router-targets-declared
           checkpoint-before-resume
           guardrail-before-downstream-task
           human-review-before-final-output
           trace-covers-agent-task-flow
           handoff-after-proof-gate)
    (handoff marlin-control-plane)))
#+end_src

The public CrewAI docs describe the same high-level shape:

- Agents compose tools, memory, knowledge, and structured outputs.
- Flows orchestrate start/listen/router steps, manage state, persist execution, and resume long-running workflows.
- Tasks and processes cover sequential, hierarchical, and hybrid execution with guardrails, callbacks, and human-in-the-loop triggers.

CrewAI expresses those capabilities as Python agents/tasks/crews/flows. POO Flow expresses them as reusable POO profiles and a composition case. Runtime execution remains outside Scheme; the Scheme control plane produces policy facts and handoff receipts.

* Runtime Language Handoff

The CrewAI-style case must lower into a runtime language rather than becoming a Scheme executor. Python is the current runtime language target because the Python runtime already has graph execution, long graph execution, stream/event consumption, interrupt/resume, handoff/reducer behavior, and C ABI graph validation.

The intended lowering is:

#+begin_src text
user-interface/cases/crewai-research-flow.ss
  -> POO Flow composition object
  -> Scheme graph/loop/proof receipt
  -> Python RuntimeGraphProgram / C ABI
  -> runtime execution receipt
  -> Lean checked facts
#+end_src

This is also how Pi Agent core-style capabilities should be represented. POO Flow can describe session state, tool calling, agent loop progress, attachment/message state, and provider/tool bindings as policy/profile facts. Python then implements the runtime loop and returns receipts. The Scheme layer should prove that the selected plan is internally sound and that the returned runtime receipt matches the plan.

#+begin_src typst
$
op("CrewAIStyleComposition")(c)
  and
op("RuntimeLanguage")(upright("python"))
  and
op("RuntimeAccepts")(upright("python"), c)

op("LowCostRuntimeProjection")(c, upright("python"))

#+end_src

The benchmark implication is direct: once the Python runtime can execute long graphs and graph receipts, implementing a CrewAI-style or Pi Agent core-style scenario should be a matter of composition lowering and proof/receipt alignment, not a new Scheme runtime clone.
* Reuse Rule

The scenario intentionally uses =use-profile= instead of hardcoding module binding names. This keeps the user-facing fragment reusable:

- stable capabilities go into =user-interface/profiles/crewai.ss=;
- the reusable workflow goes into =user-interface/cases/crewai-research-flow.ss=;
- downstream users can override or extend the profiles through normal POO object extension;
- a one-off local override should stay local to the composition until it becomes a stable public profile.

The important distinction is that POO Flow does not merely reproduce CrewAI's runtime surface. It makes the strategy composition explicit and proof-facing, so unsafe task dependencies, undeclared routers, unchecked guardrails, or missing human review can be rejected before a runtime language executes the flow.
