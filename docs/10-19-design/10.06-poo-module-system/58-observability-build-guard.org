* 10.06.58 Observability Build Guard Reflection
This note records the failure mode exposed while adding the user-interface
profile proof gate. It is implementation documentation, not an implementation
plan.

** 10.06.58.10 What Went Wrong
The existing observability surface could describe module-system observations,
but it did not prevent the build from continuing through a dangerous Gerbil
compile path. The profile proof gate was first placed in =profile-core.ss= as a
large set of public helpers. That violated the POO owner boundary: core profile
objects became responsible for proof-gate construction, proof status checks,
Lean fact projection, and fact-key validation.

The concrete symptoms were:
- =profile-core.ss= became a high fan-out rebuild target for proof-gate edits;
- repeated public accessors and predicates expanded the owner surface instead
  of keeping the POO object compact;
- the build printed elapsed time after =std/make= returned, but did not turn a
  slow single target into a warning or failure;
- runtime-oriented observability did not observe build-stage budget violations.

#+begin_src latex
\operatorname{Observation}(e) \nRightarrow \operatorname{Prevention}(e)

\operatorname{Prevention}(e)
  \equiv \operatorname{Observation}(e)
       \land \operatorname{Budget}(e)
       \land \operatorname{Gate}(e)
#+end_src

The failure was not caused by build parallelism. The mistake was a wrong owner
boundary and an observability path that reported facts without enforcing a
budget.

** 10.06.58.20 Corrected Boundary
The profile proof gate now lives in a leaf owner:

#+begin_src mermaid
flowchart LR
  ProfileCore["profile-core.ss\nprofile object owner"]
  ProfileGate["profile-gate.ss\nproof gate owner"]
  ProfileConfig["profile-config.ss\npublic facade"]
  Tests["profile library gate tests"]
  Lean["Lean UI profile library proofs"]

  ProfileCore --> ProfileGate
  ProfileGate --> ProfileConfig
  ProfileGate --> Tests
  Tests --> Lean
#+end_src

The public Scheme surface is intentionally small:
- construct a POO-native gate object;
- ask whether the gate is accepted;
- project final Scheme-to-Lean facts;
- validate the fact-key contract.

Intermediate accessors and predicates stay internal. They should not become a
parallel public DSL. The Lean fact projection is the alist boundary; the gate
itself remains a POO object.

** 10.06.58.30 Why The Old Observability Did Not Prevent It
The module-system observability owner is intentionally strict and report-only.
It avoids POO objects so it can inspect recursive presentation paths safely.
That is correct for runtime and module presentation debugging, but insufficient
for build safety.

#+begin_src mermaid
flowchart TB
  Build["Gerbil build target"]
  DebugLine["compile debug line"]
  Human["human notices slow target"]
  Gate["build budget gate"]

  Build --> DebugLine
  DebugLine --> Human
  DebugLine -. missing before .-> Gate
#+end_src

The missing part was the gate. A debug line can tell us that a target took 60
seconds, but if nothing consumes that fact then the build still proceeds. That
is why the leak risk was visible only after the user noticed resource pressure.

** 10.06.58.40 Build Guard Semantics
Build observability now has a separate guard in the build owner. This avoids a
bootstrap cycle: =package-build.ss= must not depend on the runtime
=module-system/observability.ss= while it is compiling runtime modules.

Default budgets:
- =POO_FLOW_BUILD_WARN_SINGLE_TARGET_SECONDS= defaults to =30=;
- =POO_FLOW_BUILD_FAIL_SINGLE_TARGET_SECONDS= defaults to =120=;
- =POO_FLOW_BUILD_WARN_STAGE_SECONDS= defaults to =60=;
- =POO_FLOW_BUILD_FAIL_STAGE_SECONDS= defaults to =300=.

When a budget is crossed, the build emits:

#+begin_src scheme
|poo-flow-build-observability
#+end_src

Warning budgets produce a diagnostic line. Failure budgets raise an error and
stop the build after the target returns.

The build owner also wraps single-target =std/make= calls in a live watchdog
thread. The watchdog does not replace =std/make=, does not change Gerbil build
parallelism, and does not run raw =gxc= directly. It only watches the elapsed
time of the current single-target compile. If the target is still running after
=POO_FLOW_BUILD_FAIL_SINGLE_TARGET_SECONDS=, it emits:

#+begin_src scheme
|poo-flow-build-observability
#+end_src

with =budget-status= =single-target-live-time-budget-exceeded=, then exits the
current build process with status 124. This is the minimal prevention layer that
was missing in the earlier failure: an abnormal single-target compile no longer
requires a human to notice and manually kill Gerbil.

** 10.06.58.42 P0 Compiler-Heap Containment
The live watchdog above was not sufficient for a full =build.ss compile=.  A
compiler invoked repeatedly through =std/make= can retain its compiler heap in
the long-lived control-plane =gxi=.  Adding more watcher work to that process
does not contain the heap; it can increase pressure while the compiler is still
live.  Runs that grow to gigabyte scale must therefore never be reported as a
successful build.

The corrected boundary is process isolation, while build planning and policy
remain Scheme-native:

#+begin_src mermaid
flowchart TB
  BuildAPI["build.ss compile\nScheme build API"]
  Plan["POO compile plan\ntopology layers + currentness"]
  ControlGuard["Scheme control-plane guard\nRSS + total timeout"]
  Unit["one stale compile unit"]
  GXC["short-lived native gxc\ncompiler heap dies on exit"]
  Structured["atomic structured spec\nminimal std/make child"]
  Artifact["verified artifacts"]
  Receipt["POO compile receipt\ncompleted or failed"]

  BuildAPI --> Plan
  Plan --> ControlGuard
  ControlGuard --> Unit
  Unit -->|plain Scheme source| GXC
  Unit -->|gsc/ssi group| Structured
  GXC --> Artifact
  Structured --> Artifact
  Artifact --> Receipt
  ControlGuard --> Receipt
#+end_src

The implementation invariants are:
- =build.ss compile= enters the guarded project-compile API; it must not call
  the unguarded whole-project compiler directly;
- Scheme owns topology order, stale-target selection, budgets, and the final
  POO-native receipt;
- a plain stale =.ss= target is compiled by a short-lived native =gxc=, so one
  target's compiler heap cannot accumulate across the complete project;
- structured =gsc:/ssi:= specs remain atomic and execute in a minimal child
  =gxi= that imports only =std/make=;
- the long-lived control-plane =gxi= has a Scheme-side RSS guard.  The default
  limit is =768 MiB= and a violation exits with a non-zero guard receipt;
- =POO_FLOW_BUILD_TOTAL_TIMEOUT_SECONDS= owns the whole compile timeout and
  defaults to =1800= seconds.  It is intentionally distinct from historical
  single-stage timing diagnostics;
- native commands are resolved from the scrubbed project environment (or an
  explicit =POO_FLOW_GXC=/=POO_FLOW_GXI= override); the build must not silently
  select a Nix Gerbil toolchain;
- TERM of the supervising build is not sufficient evidence of cleanup.  Final
  verification must show no descendant or orphan =gxi=/=gxc= compiler process.

This containment changes the safety property from observation after an
unbounded compile to a bounded lifetime for every compiler heap:

#+begin_src latex
\operatorname{Compile}(P)
  = \operatorname{TopoFold}
      (\lambda u.\operatorname{SpawnAndReap}(\operatorname{Compiler},u), P)

\forall u \in P:\quad
  \operatorname{HeapLifetime}(u)
  \subseteq \operatorname{ProcessLifetime}(u)
#+end_src

** 10.06.58.43 P0 Validation Receipt (2026-07-13)
The first complete cold run after introducing compiler-heap containment returned
status 0 with the following Scheme guard facts:
- elapsed: =1,733,255 ms=;
- control-plane peak: =709,100,728 bytes=;
- control-plane limit: =805,306,368 bytes=;
- total timeout: =1,800,000 ms=;
- remaining compiler processes after completion: zero.

One macro-heavy unit, =module-system/init-syntax.ss=, transiently used a child
=gxc= plus =gsc= tree of roughly 0.50 GiB plus 1.18 GiB.  That cost was confined
to one compiler tree and the complete tree was reaped before the next unit.  It
must not be confused with permission to run multiple such trees in parallel.
Future child-RSS policy must aggregate the whole descendant tree, not only the
immediate =gxc= process.

The immediate warm rerun returned status 0 with:
- Scheme guard elapsed: =1,373 ms=;
- Scheme guard peak: =406,802,840 bytes=;
- operating-system maximum resident set size: =225,034,240 bytes=;
- remaining compiler processes after completion: zero.

These receipts are the P0 acceptance shape: a cold compile must finish inside
the hard control-plane budgets, and a warm compile must prove that current
artifacts are skipped without reviving compiler children.

** 10.06.58.45 Scenario Test Placement Rule
The profile library gate exposed a second build-shape issue. A scenario test placed directly under =t/= became part of the package =tests= compile stage, even though it was meant to run through the gxtest scenario harness. On a cold compile this single target took more than a minute and emitted:
#+begin_src scheme
|poo-flow-build-observability
#+end_src
with =budget-status= =single-target-time-budget-warning=.

The fix was not to change build parallelism. The fix was to restore the owner boundary:
- package compile owns reusable runtime modules and top-level compiled tests;
- scenario harness owns scenario tests under =t/scenarios/=;
- user-interface proof gates should run through scenario receipts, not through package compile side effects.

The practical rule is:
#+begin_src latex
\operatorname{ScenarioTest}(t)
  \Rightarrow
    t \in \texttt{t/scenarios}
    \land \operatorname{HarnessReceipt}(t)
    \land \neg \operatorname{PackageCompileTarget}(t)
#+end_src
This keeps proof and benchmark coverage visible without turning every new user-interface scenario into a package build hotspot.

** 10.06.58.47 Single Stale Target Profiling
The composition macro work exposed a third build-shape issue. After the
=use-profile= syntax was split into its own owner, touching only
=profile-composition-profile-syntax.ss= still made =gxpkg build -g= report a
single-target warning around the runtime stage. The source target itself was not
the whole problem: the build runner was mixing three costs into one line:

- stale target discovery across the full runtime stage;
- the actual =std/make= call for the one stale source;
- full stage receipt refresh after the target returned.

The corrected shape is:
- debug/runtime and debug/tests stages use the profiled path even when there is
  only one stale target;
- =package-stage-spec= lines own the actual single-target budget;
- profiled =package-stage= aggregate lines remain diagnostic but do not emit a
  duplicate single-target budget warning;
- when a stage stamp already exists, successful compile refresh retouches the
  stamp instead of rewriting the full receipt for every target in the stage;
- when the stage is stale because sources are newer than the stamp, stale target
  discovery uses source-vs-stamp checks plus lightweight output-presence checks
  instead of full output mtime scans for every target.

#+begin_src latex
\operatorname{SingleStale}(s)
  \Rightarrow
    \operatorname{ProfiledSpec}(s)
    \land \operatorname{BudgetOwner}(s) = \texttt{package-stage-spec}
    \land \operatorname{Refresh}(Stage) = \operatorname{RetouchStamp}
#+end_src

This preserves the useful warning for genuinely slow source owners while
preventing wrapper and cache-maintenance time from being misreported as source
compile time.
** 10.06.58.50 Future Observability Shape
The long-term design should split observability into three layers:

#+begin_src mermaid
flowchart TB
  Kernel["strict observability kernel\nno POO, no runtime adapters"]
  Runtime["runtime/module observations"]
  Build["build watchdog observations"]
  Policy["policy/test gates"]

  Kernel --> Runtime
  Kernel --> Build
  Runtime --> Policy
  Build --> Policy
#+end_src

The kernel should own strict record constructors and final alist projection.
Runtime observability can continue to inspect module presentation and POO slot
authoring. Build observability should own elapsed time, output-count deltas,
target identity, optional RSS sampling, and hard budgets. Policy gates should
then fail before an agent-authored change can create another high-risk compile
path.
