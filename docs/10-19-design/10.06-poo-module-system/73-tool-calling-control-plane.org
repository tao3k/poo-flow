#+title: Tool Calling Control Plane
#+startup: overview

* Purpose

Tool calling is a first-class control-plane concern. LangChain, LangGraph, CrewAI, Pi Agent core, and similar frameworks all need an agent to choose tools, pass arguments, receive results, update state, and continue the loop. POO Flow should not treat that as an incidental node label. It should model the policy layer explicitly, then lower the verified plan to a runtime language such as Python.

This document uses "tool calling" for the end-to-end control path. If a scenario also needs cooldown/rate-limit behavior, that remains a policy profile inside the same tool-calling control plane.

* Boundary

Scheme owns the control plan:

- tool request ownership;
- tool schema and argument contract;
- tool permission and sandbox scope;
- argument validation before runtime;
- untrusted output boundary;
- cooldown or retry policy;
- result contract before downstream steps;
- runtime binding contract;
- runtime receipt gate;
- observability coverage.

Python or another runtime language owns execution:

- invoking the actual tool;
- provider and SDK calls;
- process/network/filesystem IO;
- streaming tool events;
- retry timers and wall-clock enforcement;
- returning execution receipts.

* Formal Obligations

#+begin_src latex
\[
\operatorname{ToolCall}(a,t)
  \Rightarrow
  \operatorname{OwnerSession}(a)
  \land
  \operatorname{DeclaredTool}(t)
  \land
  \operatorname{AllowedTool}(a,t)
  \land
  \operatorname{SandboxScopeContains}(a,t)
\]

\[
\operatorname{RuntimeToolCall}(r,t,args)
  \Rightarrow
  \operatorname{ArgumentsMatchSchema}(t,args)
  \land
  \operatorname{ValidateBeforeRuntime}(args)
\]

\[
\operatorname{ToolOutput}(o)
  \Rightarrow
  \operatorname{Observation}(o)
  \land
  \neg \operatorname{AuthorizesPolicy}(o)
\]

\[
\operatorname{ReceiptMatchesToolPlan}(receipt, plan)
  \Rightarrow
  \operatorname{ToolPolicySoundRun}(receipt)
\]
#+end_src

The key security boundary is the untrusted observation rule. A tool result may inform the next step, but it cannot rewrite sandbox scope, grant permissions, or become an authorization fact unless a local owner module explicitly projects it into a typed trusted fact.

* Runtime Language Projection

The Python runtime should expose tool binding capabilities that correspond to the control-plane facts:

| Scheme fact | Python runtime responsibility |
|---+---|
| =tool-arguments-match-schema= | validate serialized arguments before invocation |
| =tool-permission-before-call= | reject missing capability before calling |
| =tool-scope-contained= | execute inside the declared sandbox/tool scope |
| =cooldown-before-retry= | enforce retry/cooldown timers in runtime |
| =tool-result-before-downstream-step= | materialize typed result before downstream node |
| =runtime-receipt-matches-tool-plan= | return a receipt that names tool, args digest, result digest, status, and trace |

This keeps POO Flow strong at composition and proof, while Python remains responsible for the operational tool call.

* Large POO Object-List Implementation Note

The Scheme control plane keeps tool calling as POO objects, not as a vector or
raw record escape hatch.  The performant shape is:

- each tool-call fact is a small POO entry object with stable slots;
- the plan and runtime receipt are larger POO objects that own an object-list of
  entries;
- the constructor builds the entry index once and stores it as a cached slot;
- proof and runtime projection use the cached index, then only lower to alist or
  ABI data at the boundary.

The important implementation rule is that a POO slot RHS must not reuse the slot
identifier as the value binding.  Forms like ~(key key)~, ~(value value)~,
~(entries entries)~, and ~(index index)~ can push Gerbil POO expansion/runtime
lookup onto a bad path.  Constructors should use names such as ~entry-key~,
~entry-value~, ~plan-entry-list~, and ~plan-index-cache~ while preserving public
slot names.

This is not an argument against large POO objects.  It is the opposite: large
POO object families are the correct public control-plane shape, but they require
object-list/index discipline so the object model remains fast and inspectable.
