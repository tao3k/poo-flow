* 10.06.24 Agent Harness, Session, Run, And Dispatch Design
** 10.06.24.10 Status
This document turns the Flue audit and current =poo-flow= design into an implementation-oriented design for agent harnesses, sessions, workflow runs, operations, dispatch receipts, and UI snapshots.

The decision is not to copy Flue's TypeScript API. The decision is to adopt the boundary discipline:

- users declare compact project intent;
- Scheme owns POO object families, profile policy, validation, projections, and receipts;
- Rust or another runtime owns durable execution, process control, filesystem isolation, event storage, and deployment integration.

** 10.06.24.20 Design Problem
The current =poo-flow= control plane already has flows, tasks, strategies, runners, runtime adapters, receipts, module descriptors, sandbox profile recipes, loop governor objects, and user-facing root declarations.

The missing layer is the agent harness boundary:

- a workflow run should be a finite run with a run id, events, logs, result, and error;
- an agent session should be continuing conversation state and should not automatically become a workflow run;
- a dispatch receipt should record accepted asynchronous input without pretending to be a completed result;
- an operation should be a typed action inside a session or workflow run;
- a UI or CLI snapshot should be a stable projection over rich receipts and event streams.

Without these separate object families, loop-engine, governor, human-audit, sandbox, and workflow receipts can collapse into one vague "run" concept. That would make policy review and runtime handoff harder for agents.

** 10.06.24.40 Object Families
*** 10.06.24.41 Agent Profile
An =agent-profile= is a reusable policy object. It owns:

- name;
- model policy or runtime model selector;
- instructions or prompt policy metadata;
- tool and skill references;
- sandbox profile reference;
- loop/governor policy references;
- compaction and budget policy;
- observability policy.

It should be a POO object, not only an alist. Alists are presentation products.

*** 10.06.24.42 Agent Harness
An =agent-harness= is an initialized agent environment boundary. It is the object that joins:

- one selected agent profile;
- one sandbox profile projection;
- runtime adapter intent;
- filesystem and shell capability facts;
- session namespace;
- observability sink intent.

The harness must remain inert in Scheme unless a specific lightweight test backend is selected. Production execution is a runtime handoff.

*** 10.06.24.43 Agent Session
An =agent-session= is named continuing state under a harness. It owns:

- session name;
- parent harness id;
- current status;
- active operation id or =#f=;
- conversation-state reference;
- retention policy;
- operation history refs.

It is not a workflow run. A session may exist before and after many operations.

*** 10.06.24.44 Agent Operation
An =agent-operation= is a typed action inside a session or workflow run. Initial operation kinds:

- =prompt=;
- =skill=;
- =task=;
- =shell=;
- =fs-read=;
- =fs-write=;
- =compact=;
- =governor-judge=;
- =human-audit=.

An operation owns request shape, result contract, runtime handoff intent, status, and receipt pointer. Delegated =task= operations create child session facts; they do not create workflow runs by default.

*** 10.06.24.45 Workflow Run
A =workflow-run= is a finite invocation. It owns:

- =run-id=;
- workflow descriptor or case reference;
- input payload reference;
- status;
- initialized harness refs;
- event stream reference;
- log refs;
- result or error;
- terminal receipt.

Only workflow invocations create workflow runs. Direct agent prompts and asynchronous dispatches do not appear in workflow run history unless a workflow explicitly creates them.

*** 10.06.24.46 Dispatch Receipt
A =dispatch-receipt= records admission of asynchronous input to an existing agent session. It owns:

- =dispatch-id=;
- target agent/profile name;
- target agent instance id or session id;
- accepted payload reference;
- accepted timestamp;
- admission status;
- runtime queue handoff intent.

It does not own a workflow =run-id= and does not imply the model has processed the input.

*** 10.06.24.47 Snapshot Projection
A =runtime-snapshot= is a stable shallow projection for CLI, UI, and agent inspection. Initial statuses:

- =idle=;
- =admitted=;
- =connecting=;
- =running=;
- =blocked=;
- =waiting-human=;
- =completed=;
- =errored=;
- =disconnected=.

The snapshot includes status, ids, last event index, result summary, error summary, and presentation trace. It must be derived from richer POO objects or receipts, not become the canonical runtime state.

** 10.06.24.50 Boundary Rules
1. User declarations stay compact. =user-interface/init.ss= should select modules and feature flags, not define harness internals.
2. Module descriptors stay inert. Activation may lower to config and runtime descriptors, but it must not execute handlers.
3. Sandbox profiles stay inert. They project backend intent and filesystem/resource facts, but do not start containers in Scheme.
4. Workflow runs, agent sessions, and dispatch receipts are separate object families.
5. Loop governor and human audit are operation/profile specializations over the same POO governor lineage.
6. Production event streams, persistent sessions, queues, process control, and durable storage are runtime-owned.

** 10.06.24.60 User Interface Direction
The downstream-facing syntax should stay close to the current root interface:

#+begin_src scheme
(poo-flow!
  (use-module workflow +funflow +runtime-manifest)
  (use-module loop-governor +strategy +policy)
  (use-module nono-sandbox +nono +doctor)
  (use-module custom-loop +reviewer +human-audit))
#+end_src

Custom modules should define compact profile and case rows, not runtime internals:

#+begin_src scheme
(profile reviewer
  :inherits (loop/governor-agent)
  :sandbox agent/nono
  :result (schema review-result))

(case cicd-review
  :workflow funflow/cicd
  :harness reviewer
  :session "default"
  :governor reviewer
  :human-audit required)
#+end_src

The exact macro names can change during implementation. The contract should not change: users select modules, profiles, and cases; upstream code builds descriptors and runtime handoff data.

** 10.06.24.70 Implementation Milestones
*** 10.06.24.71 M1: Pure Object Families
Add inert object constructors, predicates, and alist projections for:

- =agent-profile=;
- =agent-harness=;
- =agent-session=;
- =agent-operation=;
- =workflow-run=;
- =dispatch-receipt=;
- =runtime-snapshot=.

No runtime execution in this milestone.

*** 10.06.24.72 M2: Module-System Projection
Extend module-system presentations so selected modules can project:

- declared profile facts;
- declared harness facts;
- declared session defaults;
- workflow-run handoff facts;
- dispatch/admission capabilities;
- snapshot presentation traces.

*** 10.06.24.73 M3: Loop Governor Integration
Represent governor, reviewer, verifier, and human-audit as named profile/operation nodes:

- machine reviewer derives from governor agent role;
- human audit derives from governor human role;
- loop cases select them by name;
- receipts record which node judged or approved which operation.

*** 10.06.24.74 M4: Runtime Handoff Contract
Define a runtime handoff manifest for:

- start workflow run;
- admit dispatch;
- open or create session;
- execute operation;
- stream events;
- read snapshot.

This contract is the boundary for =marlin-agent-core= or another backend runtime.

The current user-config projection exposes this handoff as report-only loop-engine
data:
- =workflow-run= projects the finite workflow admission plan;
- =dispatch-receipt= projects asynchronous input admission;
- =agent-operation= projects the governor or human-audit node action;
- =runtime-snapshot= projects the inspectable UI/CLI status.
- =runtime-command-manifest= projects the concrete =runtime-adapter= CLI
  handoff envelope for =marlin-agent-core=.
The manifest request declares the object families it carries:
=workflow-run=, =dispatch-receipt=, =agent-operation=, and =runtime-snapshot=.
The presentation also exposes a =runtime-command-manifest-summary= row so
agents can validate contract, schema, operation, argv, and runtime ownership
without walking the full request payload.
These projections live inside =pooFlowUserConfigPresentation= loop-engine intent
rows, so custom user modules remain compact while downstream agents still see
the full runtime boundary.
*** 10.06.24.75 M5: User-Facing Examples
Add one live presentation case and one non-executing runtime handoff case:

- CI/CD workflow with reviewer agent and human audit gate;
- sandbox-backed agent session with nono or cubeSandbox profile;
- structured result schema for reviewer output;
- snapshot projection for CLI/agent inspection.

** 10.06.24.80 Testing Strategy
Tests should start at the public projection boundary:

- object constructors preserve POO slots and C3 inheritance intent;
- alist projections are stable and shallow;
- user root config does not execute runtime code;
- workflow run and agent session ids do not collide semantically;
- dispatch receipts do not contain workflow =run-id= unless explicitly linked by a workflow;
- human-audit operation inherits governor fields and adds human decision fields;
- snapshot projection never mutates canonical objects.

Avoid testing private helpers when a presentation or receipt contract already exercises the behavior.

** 10.06.24.90 Open Decisions
1. Whether =agent-harness= should live under =src/agent= or =src/runtime= in Scheme.
2. Whether =workflow-run= belongs beside existing =src/workflow= store code or under a new =src/runtime= boundary.
3. Whether snapshot status names should be symbols only or a validated enum object.
4. How much of result schema validation should use =gslph= structured checks in the first implementation.
5. Whether the first runtime handoff manifest should be a new object family or extend existing runtime-command manifest projection.
