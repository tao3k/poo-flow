#+title: RFC 0144: POO-native Bundle v1 Scheme Lowering
#+status: accepted
#+phase: 3I

* Conclusion

Phase 3I implements a pure Bundle v1 lowering plan in the Scheme control
plane. Its public inputs and outputs are POO objects. Alists are used only by
the internal POO slot assembly implementation; they are neither exported nor
accepted as a user DSL.

This phase does not write through C pointers and does not call Python. It
lowers the component, edge, and evidence identities required by a Domain Case
into fixed-width compact IDs, strictly sorted native rows, aligned regions, a
256-byte descriptor receipt, and a 32-byte Bundle digest. The next phase can
encode this receipt into a registered foreign arena without interpreting POO
semantics again.

The Phase 3I plan is an immutable semantic value. The C ABI never observes a
Scheme object, and the Scheme control plane does not assemble C structs
directly.

* POO-native public surface

Public constructors and operations:

- =feature-bundle-v1-component=: case, component, object, type, contract,
  role, capability, policy, strategy, adapter, projection, and composition
  order;
- =feature-bundle-v1-edge=: case, source, target, relation, and order;
- =feature-bundle-v1-evidence=: case, obligation, contract, evidence type,
  proof system, and order;
- =feature-bundle-v1-lowering=: Bundle identity, uint64 epoch, and the three
  POO value lists;
- =require-feature-bundle-v1-lowering-plan=: accepts only a plan whose
  =accepted?= slot is =#t=.

The output surface also consists entirely of POO objects:

- =feature-bundle-v1-compact-id=;
- =feature-bundle-v1-region=;
- =feature-bundle-v1-descriptor=;
- =feature-bundle-v1-diagnostic=;
- =feature-bundle-v1-lowering-plan=.

A module-local declarative macro generates the repeated object-family
constructors and expands into ordinary POO object construction. The module
uses an explicit export list, so its macro, native-row constructors,
validators, and layout helpers do not leak into the public API. There is no
raw =(lambda (self super) ...)= compute hook and no public record, alist, or
patch language.

* Scheme identity to C compact ID

Every native identity uses the following canonical input:

#+begin_example
poo-flow.bundle-v1.id/<domain>/<typed-semantic-id>
#+end_example

The typed semantic ID explicitly distinguishes symbols, strings, and exact
integers. SHA-256 is calculated over the canonical string. The first eight
bytes form =high= in big-endian order, and the following eight bytes form
=low=.

Domain separation prevents identically named roles, components, and contracts
from being treated as the same C identity. The frozen vector is:

#+begin_example
domain=obligation source=0
high=15650260190551699577
low=10580127755728368254
#+end_example

Future Rust, Python, and TypeScript adapters must reuse this vector.

* Functional lowering

The implementation reuses =poo-flow-map=, =poo-flow-fold-left=,
=poo-flow-all?=, and =poo-flow-list-of?= from
=src/utilities/functional.ss=. Gerbil =stable-sort= produces a new list and
does not mutate the input composition.

#+begin_src mermaid
flowchart TD
  INPUT["POO values"] --> VALIDATE["all? identity/order validation"]
  VALIDATE -->|"invalid"| REJECT["POO diagnostic + rejected plan"]
  VALIDATE --> MAP["map semantic IDs to compact IDs"]
  MAP --> SORT["stable-sort by frozen C key order"]
  SORT --> UNIQUE["strict adjacent-key validation"]
  UNIQUE -->|"duplicate"| REJECT
  UNIQUE --> LAYOUT["fold sizes into aligned regions"]
  LAYOUT --> DIGEST["canonical SHA-256 Bundle digest"]
  DIGEST --> READY["accepted POO lowering plan"]
#+end_src

Sort keys exactly match the C validator:

- component: =(case-id, component-id)=;
- edge: =(case-id, source-component-id, composition-order,
  target-component-id, relation-id)=;
- evidence: =(case-id, obligation-id)=.

The C validator performs one =O(n)= strict-order validation when opening a
Bundle. Runtime component lookup continues to use the =O(log n)= lower-bound
algorithm defined by RFC 0143.

* Frozen layout receipt

The Scheme plan freezes the following C layout facts:

| Region | Stride | Alignment |
|--------+--------+-----------|
| symbols | 32 B | 8 B |
| components | 200 B | 8 B |
| edges | 80 B | 8 B |
| evidence obligations | 96 B | 8 B |
| metadata bytes | 1 B | 1 B |

The descriptor size is fixed at 256 bytes. The arena is at least 64 bytes and
is rounded to a 64-byte boundary. The lowering plan fixes every C descriptor
field: =struct_size=, flags, schema major and minor, =reserved0=, Bundle ID,
digest, epoch, arena bytes, five regions, and seven reserved uint64 values.
Native component, edge, and evidence rows also carry their required zeroed
reserved fields explicitly. The arena writer therefore encodes the receipt;
it does not add or infer ABI semantics.

Phase 3I does not materialize symbol metadata, so the symbols and metadata
regions are valid empty regions. Every execution-time identity has already
been lowered into a compact ID.

* Fail-closed rules

The following inputs produce a POO plan with =status=rejected=:

- the Bundle ID is not a non-empty symbol, string, or uint64;
- the Bundle epoch is not a uint64;
- a collection is not a proper list;
- a component, edge, or evidence value is not the corresponding POO object;
- an identity slot is absent or has an invalid type;
- a composition order is outside the uint64 range;
- a component, edge, or evidence key is duplicated.

The lowering operation does not accept raw alist components.
=require-feature-bundle-v1-lowering-plan= is the final fail-closed gate before
runtime handoff.

* fold-left semantic regression

The first 1,000-agent benchmark correctly rejected a duplicate evidence key.
The audit found that the shared =poo-flow-fold-left= callback order is
=(element accumulator)=, while the first compact-ID implementation used
=(accumulator element)=. That error collapsed SHA-256 segments into small
integers.

The correction added two gates:

- compact-ID high and low segments must preserve the complete 64-bit SHA-256
  prefix;
- =obligation/0= must match the frozen vector above.

This demonstrates that fail-closed duplicate validation detects identity
lowering defects in addition to malformed input.

* 1,000-agent validation

The Scheme benchmark constructs:

- 1,000 components;
- 999 dependency edges;
- 1,000 Lean evidence obligations;
- varied types, contracts, roles, capabilities, policies, strategies,
  adapters, and projections.

The observed result is:

#+begin_example
bundle-epoch=1
arena-bytes=375936
digest-bytes=32
elapsed-seconds=.618149995803833
input-interface=poo-native
layout=typed-native-regions
json-in-hot-path=false
#+end_example

The complete =gxi= process peaked at approximately 226 MB with zero swap.
This is an engineering regression receipt for the current machine, not a
cross-hardware SLA.

* Relationship to Phase 3G

The Phase 3G =FeatureRuntimeHandoffPlan= separates Bundle, Cedar, and
Lean/Evidence handoffs. Phase 3I freezes the native identities and table
layouts ultimately consumed by those handoffs.

The next phase must add a POO-native projection from an accepted Domain Case
assembly and its adapter/projection bindings into the component, edge, and
evidence objects defined by this RFC. Users must not be required to enumerate
native rows manually. After that projection is complete, the implementation
can add the foreign arena writer and make Runtime v1 consume Bundle v1
directly.

* AOT and Scheme memory guard

This phase does not use the =gxi/gxc= process exit code as its only success
criterion. The gates also inspect error text, expected artifacts, process-tree
RSS, outer RSS, and swap:

- the standard incremental =gxpkg build= completed; the Scheme build-API
  guard recorded a process-tree peak of 397,364,416 bytes, below the
  805,306,368-byte limit;
- compiling only =bundle-v1-lowering.ss= into an isolated artifact root
  exited with code 0, produced nine artifacts, matched no errors, and peaked
  at 263,634,944 bytes RSS;
- compiling the entire tree from an empty artifact root with
  =build.ss compile= failed closed at 813,491,984 bytes, exceeding the limit
  by 8,185,616 bytes. It exited with code 70 and used no swap.

The focused AOT gate demonstrates that Phase 3I itself does not approach the
memory limit. The full-tree gate remains a release blocker and must not be
hidden by raising the RSS limit. That full tree includes unrelated uncommitted
modules from the current worktree, so a later phase must attribute the peak by
active compile unit instead of assigning the aggregate failure to this
lowering module.

* Follow-up gates

1. Domain Case to Bundle v1 POO projection;
2. Scheme-side registered foreign arena writer;
3. validation of the same arena through the C validator and Python CFFI;
4. Cedar and evidence consumers reading the same obligation rows;
5. one-step removal of the runtime-v0 header, symbols, Python module,
   environment variables, and tests;
6. green Runtime v1 symbol-surface, ASan/UBSan, AOT/RSS, and wheel-install
   gates.

No v0 wrapper and no legacy row syntax will be added.
