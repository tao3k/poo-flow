#+TITLE: POO Performance Scenario Alignment

* Research basis

This note records the current performance direction for POO Flow after comparing
three evidence streams:

- upstream gerbil-scheme-language-project-harness POO scenarios;
- local POO Flow benchmark receipts;
- Gerbil compiler and ~std/make~ behavior under incremental builds.

* Gerbil compile model implications

Gerbil ~std/make~ first resolves buildspec targets and timestamp dependencies,
then invokes ~gxc~ through ~compile-module~.  A single module can generate
runtime, loader, meta, phi, static, and optional ssxi outputs, and gsc work can be
queued as compile jobs.

The practical rule for POO Flow scenarios is:

- keep scenario-specific cases in small owner modules;
- keep shared fixture registration narrow;
- avoid growing one very large test-support module with unrelated scenario code;
- avoid adding ad hoc buildspec exceptions for each scenario.

The current build shape follows that rule by adding
~t/module-system-poo-performance-test-support~ as a tracked support owner and
keeping the tool-calling and fixed-slot projection cases in small modules.

* Upstream POO practice map

The upstream harness scenarios establish these performance rules:

- validation: validate a stable native POO shape once at the boundary, not in the
  measured loop;
- clone override: keep scalar loop state and apply one final native clone
  override boundary;
- materialization: snapshot broad object data once at the boundary;
- object iteration: avoid repeated object snapshots inside loops;
- fixed slot projection: bind fixed POO slots once and run the loop over scalar
  state;
- dynamic boundaries: adapters are only for external data; native POO remains the
  core shape.

* Local scenario coverage

Current local scenarios now track:

- construction;
- materialization;
- validation;
- catalog validation;
- object iteration;
- clone override;
- field lookup;
- extension merge and contribution targeting;
- tool-calling object-list control;
- fixed-slot projection.

The next missing upstream-aligned coverage should be:

- slot predicate loops;
- slot-spec mutation loops;
- POO type construction families;
- lens/projection composition;
- macro-family thin wrapper generation.

* Implementation rule

Do not replace POO with raw records, vectors, or alists in the public control
plane.  The optimization route is to keep POO-native source shapes and move work
to explicit boundaries: validation, materialization, clone, projection, indexing,
and generated helper families.
