#+title: POO Flow Composition Interface
#+property: header-args :mkdirp yes

* Purpose

`profile-composition` is the core composer for turning several module profiles
into one staged strategy composition. It lives under `src/module-system`; the
user-interface layer should call it rather than owning the composer. It is not a
production-only profile, and it is not an SDLC DSL. Develop, staging, and
production are stage names inside the composition, while the main concept stays
POO Flow strategy composition.

The key design rule is POO-native profile selection:

#+begin_src scheme
(profile session hardened)
#+end_src

expands to a profile reference on a module object:

#+begin_src scheme
(.ref session 'hardened)
#+end_src

That means each module is a POO object, and each exported profile is a slot on
that object. There is no JSON manifest, string lookup registry, or separate
catalog DSL in the normal user path.

* Syntax

#+begin_src scheme
(use-composition rag-agent
  (modules
    (use-module session-module #:as session)
    (use-module sandbox-module #:as sandbox)
    (use-module retriever-module #:as retriever)
    (use-module llm-module #:as llm))

  (stage develop
    (compose
      (profile session dev)
      (profile sandbox local)
      (profile retriever mock)
      (profile llm debug)))

  (stage production
    (compose
      (profile session hardened)
      (profile sandbox restricted)
      (profile retriever vector-store)
      (profile llm chat))
    (graph guarded-rag-flow)
    (loop #:fuel 8 #:exit answer-ready)
    (prove scope-contained dependency-ready graph-reachable loop-progress)))
#+end_src

The syntax is intentionally a macro layer over ordinary POO objects. The user
can always drop to direct POO when needed:

#+begin_src scheme
(compose
  (.ref session 'hardened)
  (.ref sandbox 'restricted))
#+end_src

* User-Side Load Form

The composer lives in `src/module-system`. User files should load the core
composition macro package through the normal User Module path instead of adding
a standalone `user-interface/composition.ss` facade.

`load!` fragments are included as declaration values. The loader owns the
generated binding and export, while `use-composition` returns a composition
object expression:

#+begin_src scheme
(use-composition rag-agent
  (modules
    (use-module session-module #:as session)
    (use-module sandbox-module #:as sandbox))
  (stage production
    (compose
      (profile session hardened)
      (profile sandbox restricted))
    (graph guarded-flow)
    (loop #:fuel 4 #:exit done)))
#+end_src

In a normal module body the caller may bind the value explicitly:

#+begin_src scheme
(def rag-agent
  (use-composition rag-agent ...))
#+end_src

Inside a `load!` fragment the surrounding loader generates the exported
binding, so the fragment should return the declaration value directly. The
composition interface adds no registry, JSON manifest, or alternate profile
semantics.

A reusable case fragment should not hardcode the generated binding name of a
profile fragment. `load!` names depend on the owner custom module. The reusable
shape stays declarative by using `use-profile` inside `use-composition`:

#+begin_src scheme
;; user-interface/cases/langchain-linear.ss
(use-composition langchain-linear-composition
  (modules
    (use-profile langchain #:as chain))
  (stage production
    (compose
      (profile chain memory)
      (profile chain prompt)
      (profile chain model)
      (profile chain parser)
      (profile chain no-tool))
    (graph langchain-linear-chain)
    (loop #:fuel 1 #:exit parsed-output)
    (prove chain-order prompt-before-model parser-after-model)))
#+end_src

`use-profile` is a macro-only child form. It resolves `langchain` to the
profile module loaded in the same user module, such as
`(load! "profiles/langchain")`, without exposing generated binding names in the
case fragment.

* Profile Reuse Modes

POO Flow should prefer reusable user-module profiles whenever a profile has a
stable meaning beyond one local case. This is one of the core strengths of the
project: downstream users can contribute a profile once, and later modules,
cases, and compositions can reuse it through ordinary POO object extension.

There are two valid override modes.

** Local Override For Local Problems

Use a local profile override when the specialization only exists to solve one
composition-local problem. This keeps the user surface compact and avoids
forcing every temporary edge, router, or loop policy into a shared module file.

#+begin_src scheme
(let* ((audited-chain
        (.o (:: self langchain-profile-module)
            (model
             (.o (name 'langchain-audited-chat-model)
                 (contract 'single-turn-input-output)
                 (policy 'audit-model-call-before-terminal)))))
       (composition
        (use-composition audited-langchain-linear-composition
          (modules
            (use-module audited-chain #:as chain))
          (stage production
            (compose
              (profile chain memory)
              (profile chain prompt)
              (profile chain model)
              (profile chain parser)
              (profile chain no-tool))
            (graph langchain-linear-chain)
            (loop #:fuel 1 #:exit parsed-output)))))
  composition)
#+end_src

This mode is appropriate when the override is not yet a real profile contract.
The proof projection should still record it, but its provenance is local to the
composition.

#+begin_src latex
LocalOverride \subset Composition

provenance(LocalOverride) = local-composition-profile
#+end_src

** Reusable Profile And Reusable Composition

When an override becomes a real profile, it should move into a user module
profile fragment so downstream users can load and extend it. The final
composition can then be reused by other modules as well.

#+begin_src scheme
;; config.ss
(load! "profiles/langgraph")
(load! "cases/langgraph-state")

(def langgraph-state-composition
  poo-flow-custom-module-langgraph-state-case)
#+end_src

The profile fragments own stable profile contracts:

#+begin_src scheme
;; user-interface/profiles/langgraph.ss
(.o (router (.o (name 'langgraph-conditional-router)
                (contract 'predicate-selects-next-node)
                (policy 'branch-target-is-declared)))
    (bounded-loop (.o (name 'langgraph-bounded-loop)
                      (contract 'fuel-decreases-on-step)
                      (policy 'loop-has-progress-measure))))
#+end_src

The composition case should then compose those reusable profiles rather than
redeclare the same override locally:

#+begin_src scheme
;; user-interface/cases/langgraph-state.ss
(use-composition langgraph-state-composition
  (modules
    (use-profile langgraph #:as graph))
  (stage production
    (compose
      (profile graph router)
      (profile graph bounded-loop))
    (graph langgraph-state-graph)
    (loop #:fuel 8 #:exit terminal-edge)
    (prove declared-branch-targets bounded-loop-progress)))
#+end_src

This mode is preferred whenever the profile or composition can help more than
one downstream user. It turns the User Interface into a contribution library:
users contribute stable profiles, and other users compose, override, and extend
them without duplicating policy objects.

#+begin_src latex
ReusableProfile \in UserModule

ReusableComposition =
  compose(ReusableProfile_1, \ldots, ReusableProfile_n)

provenance(ReusableProfile) = user-module-profile
#+end_src

The practical rule is:

- keep one-off strategy changes local to the composition;
- promote stable policy objects into `profiles/`;
- promote useful composed workflows into `cases/`;
- let downstream modules reuse both the profiles and the final composition.

* Graph Control Analysis

The `graph` clause in a composition is not just a symbol for display. It must
lower to graph facts owned by `src/graph`, so LangChain-like and LangGraph-like
cases can be checked by the same proof bridge that later feeds Lean.

The graph layer has one topology boundary and one control-analysis boundary:

- `src/graph/algorithms.ss` owns topology facts: node ids, edge pairs,
  reachability, dependency cones, cycle paths, and topological order.
- `src/graph/control-analysis.ss` is the stable public facade for graph control
  analysis.
- `src/graph/control-core.ss` owns the receipt projection. It computes entry
  ids, finish ids, conditional edges, explicit loop edges, undeclared branch
  targets, unreachable nodes, dead ends, and finish-total facts once, then
  projects them directly into a `PooFlowGraphAnalysis` receipt.
- `src/graph/control-utils.ss` owns pure helper functions for metadata lookup,
  edge filtering, and id-set operations.

This boundary intentionally avoids a second POO fact object family for the
graph-control path. A separate `PooFlowGraphControlFacts` module and object made
small graph edits recompile too much Gerbil code without improving the
user-facing API. The semantic boundary is still real: control facts are named
receipt metadata fields, but the compile-time shape is lighter. The public
proof surface is the receipt constructor; ad hoc wrapper accessors should not be
added unless a real downstream caller needs them. The public facade should stay
thin so changing the import surface does not force the full control proof
implementation to recompile.

Two tempting splits are intentionally rejected unless a later benchmark proves
otherwise:

- Do not split `control-core.ss` into separate query and receipt modules just
  for conceptual neatness. In the current Gerbil build, that adds compiled
  boundaries and did not lower the receipt-only compile path.
- Do not move the receipt constructor into `control-utils.ss`. It makes the
  helper layer too heavy and still leaves `control-core.ss` expensive because
  the import/export shape dominates the compile path.

The rule is empirical: graph proof code may be split only when the measured
`gxpkg build -g` path improves for the files users or agents actually edit.

LangChain-like workflows should prove as total linear control graphs:

#+begin_src latex
Entry = \{memory\}

Finish = \{parsed\_output\}

Reachable(Entry) \subseteq CanReach(Finish)

CyclePath = \varnothing
#+end_src

LangGraph-like workflows may contain explicit loop edges, but the loop must be
declared as part of the control graph rather than discovered accidentally:

#+begin_src latex
LoopEdges \neq \varnothing

ConditionalTargets \subseteq Nodes

Reachable(Entry) \subseteq CanReach(Finish)

DeadEnd(Reachable(Entry) - Finish) = \varnothing
#+end_src

The Scheme implementation follows the current programming style by keeping the
runtime graph analysis functional, bounded, and single-pass at the receipt
projection boundary. Repeated object families are only introduced when they
remove real complexity; in this hot graph path, direct functional helpers are
the higher-performance POO-compatible form.

The receipt path must also reuse proof facts that it already computed. For
example, once a receipt has established that `CyclePath = \varnothing`, it
should call the acyclic topological-order helper instead of re-entering the
public topological-order function and walking the graph again to rediscover
the same cycle fact.

#+begin_src scheme
(poo-flow-graph-control-analysis-receipt langgraph-state-graph)
#+end_src

The expression returns a `PooFlowGraphAnalysis` object whose metadata contains
proof-facing fields such as `conditional-edge-pairs`, `loop-edge-pairs`,
`branch-targets-declared?`, and `finish-total?`.

This keeps the user interface small: a reusable case can write
`(graph langgraph-state-graph)`, while the core graph layer owns the
mathematical interpretation of that graph.

* POO Object Model

The macro expands to ordinary POO objects:

#+begin_src text
PooFlowComposition
PooFlowCompositionModuleBinding
PooFlowCompositionStage
PooFlowCompositionClause
#+end_src

The composition object keeps module bindings and stages. A stage keeps clauses
such as `compose`, `graph`, `loop`, `prove`, and `handoff`. The `compose`
clause stores actual profile objects, not profile names.

#+begin_src latex
Module = \{ profile_i : PooProfile \}

ProfileRef(module, slot) = module.slot

CompositionStage =
Profiles \times Graph \times LoopPolicy \times ProofObligations
#+end_src

* Precision Rule

Profile calls must name the module alias and the profile slot:

#+begin_src scheme
(profile session hardened)
#+end_src

The default interface should not accept a naked profile name such as:

#+begin_src scheme
(profile hardened)
#+end_src

That keeps selection precise when several modules expose the same profile name.

* Stage Semantics

Stage names are open symbols. The common names are `develop`, `staging`, and
`production`, but the strategy engine does not become an SDLC engine.

#+begin_src latex
StageName \in Symbol

develop, staging, production \in StageName
#+end_src

A stage is only a context for selecting profile slots and proof obligations.
The loop engine and graph engine remain the core semantics.

* Proof Bridge

The Lean proof surface should reason about POO Flow facts after projection:

#+begin_src latex
ProfileObjects \to ProofFacts

ProofFacts \vdash
ScopeContained \land DependencyReady \land GraphReachable \land LoopProgress
#+end_src

LangChain-like and LangGraph-like workflows are examples of composition:

#+begin_src text
LangChain-like = staged composition + linear graph
LangGraph-like = staged composition + guarded graph
Loop = graph runtime semantics with exit/fuel policy
#+end_src

They should not introduce standalone Scheme core modules.
* Macro Ownership Boundary

~use-composition~ is source programming, so its implementation belongs under
~src/module-system~ with the rest of the composition compiler surface. The
~user-interface~ layer may expose a friendly import facade, but it must not own
the macro expansion semantics.

Current boundary:

- ~src/module-system/profile-composition-use-syntax.ss~ defines
  ~use-composition~ and owns only the composition grammar.
- ~src/module-system/profile-composition-profile-syntax.ss~ resolves
  ~use-profile~ profile fragment names to the binding names generated by
  ~load!~. It intentionally does not import the loader owner because that made
  the profile bridge expensive to recompile; the alignment is guarded by the
  LangChain/LangGraph load-and-compose scenario.
- ~src/module-system/profile-composition.ss~ exports the complete composition
  surface, including the user declaration macro.
- User configuration files load the core composition macro package through the
  existing user-module loading path; there is no standalone
  ~user-interface/composition.ss~ facade.
- ~user-interface/init.ss~ is not changed by the composition interface. Its
  aggregation grammar must remain aligned with the existing User Module pattern.

This preserves the direction of the system: user files declare production,
staging, or development composition values with Scheme macros, while the
compile-time ownership and reusable POO-native semantics stay in the project
core.
