* Scope
This document defines the POO Flow artifact control-plane direction.

Artifacts are not a Turso concern. Turso is one possible database/backend
profile. The artifact model must stay backend-agnostic and describe
classification, policy scope, lifecycle, permissions, analysis, indexing,
retention, and publishing gates. Runtime languages bind those policies to
concrete storage and execution backends after the Scheme control plane has
validated the policy receipt.

* Current Implementation Receipt
- Scheme now models artifact profiles, database/backend profiles, durable artifact declarations, policy validation receipts, manifest receipts, and Marlin handoff envelopes.
- Manifest receipts are fixed runtime receipts with bounded alist projection at the handoff boundary.
- The user-facing surface remains POO-native: users author artifact profiles, database profiles, and durable artifacts as POO objects/macros; runtime storage, indexing, publishing, and retention side effects remain outside Scheme.
- Focused gates:
  - =gxpkg env gxtest t/scenarios/durable-artifact-policy-test.ss=
  - =gxpkg env gxtest t/user-interface-custom-durable-artifact-test.ss=
* Framework Audit
The `.data` framework audit shows that artifact handling is fragmented across
agent platforms and frameworks.

- LangChain separates traces, file callbacks, and in-memory indexes across
  different surfaces.
- LangGraph is checkpoint-first, with backend-specific checkpoint
  implementations and conformance tests.
- CrewAI documents memory, files, checkpointing, traces, and publishing as
  adjacent features rather than one unified artifact policy.
- OpenRath has structured memory registry, backend capability tracking,
  adapters, persistence paths, and config store, but these remain scoped to
  runtime implementation details.
- AG2 exposes file APIs, trace/result stores, working and episodic memory
  policies, checkpoint clients, Redis stream storage, file-search tooling, and
  publish-via-context design.
- Pi Agent Core has session memory/storage, JSONL persistence, shell/file
  output capture, file mutation queueing, sandbox restore, publish automation,
  and checkpoint-oriented extension examples.
- Codex distributes artifact workflows, file system access, JSONL events, trace
  context, and citations across protocol and runtime layers.
- LoopX provides benchmark artifacts, event-store migration, and memory adapter
  protocols, but not a general artifact control-plane model.

The common gap is not storage capability. The gap is policy unification:
artifact scope, ownership, lifecycle, access, analysis, publishing, and
retention are usually split across callbacks, checkpoint stores, file APIs,
trace logs, memory stores, JSONL/session stores, output accumulators, file
mutation queues, Redis/event streams, memory adapters, and publish tooling.

** POO-Native Implementation Constraint
The Scheme implementation must prefer Gerbil POO-native object construction and
projection for its normal path.

- =.o= constructs profile, database, artifact, and receipt objects with stable
  slots.
- =.ref= reads stable fields after an object has been constructed.
- =.slot?= checks optional override fields before profile hooks project them.
- =.putslot!= and =.setslot!= are mutation-family operators. Mutation should
  stay behind explicit hot-path helpers and must not become the normal
  user-facing artifact policy interface.

The macro layer may parse declarative section lists at the syntax boundary, but
runtime profile, database, artifact, and receipt values must be projected back
into POO objects immediately. Profile hooks receive and return POO objects, and
local overrides are expressed as POO objects instead of raw alist patches.

* Native POO Extension Contract
The profile macro is only the ergonomic declaration layer. The real extension
surface must expose native POO objects. Macro expansion must create or collect
POO objects, then hand those objects to =use-composition=. It must not trap the
profile inside an opaque alist DSL or runtime-only patch language.

Core user-facing declaration:

#+begin_src scheme
(use-composition agent-artifacts
  (use-module artifact
    (profile research-report
      :extends report/base
      :scope (session human-handoff publish-channel)
      :storage (file object vector)
      :analysis (checksum schema provenance)
      :publish (human-approved proof-gated))))
#+end_src

The macro-expanded value must be a POO object equivalent to:

#+begin_src scheme
(def research-report
  (.o (:extends report/base)
      (kind 'report)
      (scope '(session human-handoff publish-channel))
      (storage '(file object vector))
      (analysis '(checksum schema provenance))
      (publish '(human-approved proof-gated))))
#+end_src

Users can then extend that object through native POO composition:

#+begin_src scheme
(def research-report/internal
  (.o (:extends research-report)
      (publish '(human-approved proof-gated internal-registry))
      (retention '(project-retained audit-log))))
#+end_src

Local override remains available inside a composition:

#+begin_src scheme
(use-composition agent-artifacts
  (use-module artifact
    (profile research-report
      :extends report/base
      :scope (session human-handoff publish-channel)
      :storage (file object vector)
      :analysis (checksum schema provenance)
      :publish (human-approved proof-gated))

    (profile internal-report
      :extends research-report
      :publish (human-approved proof-gated internal-registry)
      :retention (project-retained audit-log))))
#+end_src

Profile hooks must also be reusable POO extension functions:

#+begin_src scheme
(def (with-audit-retention profile)
  (.o (:extends profile)
      (retention '(project-retained audit-log))
      (analysis (append (.ref profile 'analysis)
                        '(provenance citation-trace)))))

(use-composition agent-artifacts
  (use-module artifact
    (profile audited-report
      :extends (with-audit-retention research-report))))
#+end_src

The important rule is that =:extends= can receive a symbol, a POO object, or an
expression that returns a POO object. This keeps Gerbil/Scheme higher-order
composition available at the user interface instead of sealing it behind macro
syntax.

The native extension surface has three layers:

- =:extends base-profile= is the normal Doom-style declarative path.
- =:with extension-fn= composes reusable extension functions.
- =(profile existing-poo-object)= directly reuses an already-defined POO object.

Reusable profile libraries can also be imported slot-by-slot without writing
repeated =poo-flow-profile-ref= calls:
#+begin_src scheme
(use-composition crewai
  (use-module crew
    (profiles crewai
      agent
      task
      crew
      planning
      memory
      knowledge
      guardrail
      human-input
      runtime-handoff)))
#+end_src
This form is only syntactic sugar for collecting existing POO objects from a
POO module object. It is not the old =(modules ...)= API and it does not create a
separate module DSL. The collected values are still native POO objects and can
be extended locally with the normal =(profile name :extends ...)= form.
Stages use the same group shape when composing imported profiles:
#+begin_src scheme
(use-composition crewai
  (use-module crew
    (profiles crewai
      agent
      task
      guardrail
      runtime-handoff))
  (stage production
    (compose
      (profiles crew
        agent
        task
        guardrail
        runtime-handoff))))
#+end_src
Single profile references remain available only for small local cases:
#+begin_src scheme
(compose
  (profile artifact research-report)
  (profile database turso))
#+end_src
Local native POO override is expressed by extending the imported object in the
same =use-module= block:
#+begin_src scheme
(use-composition crewai-enterprise
  (use-module crew
    (profiles crewai agent task memory)
    (profile audited-memory
      :extends (poo-flow-profile-ref crewai 'memory)
      :with (audit-retention citation-trace)
      :retention (project-retained audit-log)))
  (stage production
    (compose
      (profiles crew agent task audited-memory))))
#+end_src
Advanced native POO reuse keeps full Scheme power outside the macro boundary:
#+begin_src scheme
(def enterprise-report
  (.o (:extends report/base)
      publish: '(human-approved legal-review proof-gated)
      retention: '(seven-years audit-log)))

(use-composition agent-artifacts
  (use-module artifact
    (profile enterprise-report)))
#+end_src
In all three cases the macro only collects POO objects; policy meaning stays in
the object family and proof/test gates.
** Composition Performance Observability
The previous observability failed because it watched fixture shape and final
benchmark receipt status, but it did not observe the allocation boundary inside
the benchmark thunk. A benchmark could repeatedly construct a full composition
inside each iteration and still look structurally valid until RSS exploded.

The composition macro scenarios now have five explicit performance gates:
- inline profile declaration
- grouped =profiles= import and compose
- local native POO override
- reusable hook override
- direct native POO object reuse

Each gate must satisfy two contracts:
- the benchmark receipt passes the time and memory fixture gate
- the composition construction counter is exactly =1= for the whole gate

That second contract is the important observability repair. It prevents the
same bug from returning: the gate may iterate over a projected payload, but it
must not rebuild the macro-expanded POO composition inside the hot benchmark
loop.
The final profile API must support:

#+begin_src scheme
(profile name
  :extends base
  :with (fn1 fn2)
  :scope (...)
  :storage (...)
  :analysis (...)
  :publish (...))
#+end_src

and direct POO object reuse:

#+begin_src scheme
(profile existing-poo-object)
#+end_src

This satisfies the four design constraints:

- the user interface stays compact and declarative
- =src= object families remain reusable
- native POO extension is not sealed by macros
- downstream community profiles can contribute reusable objects directly

Macro expansion strategy:

- =profile= creates or references a POO object
- =use-module= collects POO profile objects into a module object
- =use-composition= collects module objects and stage policy
- runtime behavior happens only after control-plane receipts are validated

* User Composition Syntax
The user-facing artifact interface should allow local inline module profiles
inside a composition. This is the ergonomic macro surface we discussed for
cases where the profile is useful locally, but does not yet deserve a reusable
module-level profile file.

#+begin_src scheme
(use-composition agent-artifacts
  (use-module artifact
    (profile research-report
      :extends report/base
      :scope (session human-handoff publish-channel)
      :storage (file object vector)
      :analysis (checksum schema provenance)
      :publish (human-approved proof-gated))

    (profile internal-report
      :extends research-report
      :publish (human-approved proof-gated internal-registry)
      :retention (project-retained audit-log))))
#+end_src

The same profile can later be promoted into a reusable module-level profile and
consumed by downstream compositions. This keeps the local-problem path compact
while preserving POO Flow's main reuse model.

* Policy Scope
Artifact policy must explicitly scope every artifact.

Scope dimensions:

- `tenant`
- `project`
- `workflow`
- `session`
- `agent`
- `subagent`
- `sandbox`
- `tool-call`
- `human-handoff`
- `publish-channel`

Core invariant:

#+begin_src typst
$
"ArtifactVisible"(a, "act" or) =>
  "Scope"("act" or) subset.eq "Scope"(a) or "HasExplicitGrant"("act" or, a)
$
#+end_src

This prevents the common failure mode where a tool-generated file, checkpoint,
trace, or memory record becomes globally reachable because the framework treats
it as a raw file path or unscoped store key.

* Lifecycle Policy
The artifact lifecycle is explicit:

#+begin_src text
created
  -> stored
  -> indexed
  -> analyzed
  -> callable
  -> publish-approved
  -> published
  -> retained | expired | archived | revoked
#+end_src

The Scheme control plane validates lifecycle states and only permits documented
edges. Runtime languages execute storage, indexing, and publishing after the
control plane has produced a valid receipt.

* Scheme Implementation Surface
This document is implemented by =src/module-system/durable-artifact-policy.ss=.

- =artifact-profile= defines reusable artifact policy profiles.
- =database-profile= defines backend capability profiles. Turso belongs here,
  not in the artifact model itself.
- =durable-artifact= creates the actual policy-managed artifact object.
- =poo-flow-durable-artifact-visible?= implements the scope formula: actor
  scope must be contained by artifact scope unless an explicit artifact grant
  exists.
- =poo-flow-durable-artifact-transition= advances lifecycle state only along
  the documented lifecycle edges.
- =poo-flow-durable-artifact-validate= validates a real artifact against an
  artifact profile and a database profile, then emits a POO policy receipt.
- =poo-flow-durable-artifact-policy-receipt?= checks that the receipt is owned
  by this artifact policy surface.
- =poo-flow-durable-artifact-policy-receipt-valid?= is the scenario and runtime
  handoff gate.
- =t/scenarios/durable-artifact-policy-test.ss= is the scenario gate for this
  document.

The control-plane receipt is the handoff boundary. A runtime language should
not execute artifact storage, indexing, publishing, or retention work until the
receipt is valid.

The receipt validates:

- profile scope contains the artifact scope
- profile storage supports the artifact storage class
- profile analysis supports the artifact analysis policy
- profile publish policy is gated by human approval and proof gate where needed
- profile retention supports the artifact retention policy
- artifact lifecycle state is known
- database storage supports the artifact storage class
- database capabilities satisfy the artifact index policy

Diagnostics are first-class receipt fields. Invalid artifacts must produce a
receipt with =valid? = #f= and structured diagnostic symbols instead of
falling through to runtime execution.
