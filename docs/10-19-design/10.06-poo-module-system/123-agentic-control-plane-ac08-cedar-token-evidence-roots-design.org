#+title: AC-08 Cedar, Authorized Effect Tokens, and Evidence Roots Design
#+date: 2026-07-12

* Status

Approved in conversation on 2026-07-12.  This document is the AC-08 design
gate.  Implementation and qualification have not started.  ABI v1 remains
unfrozen.

* Objective

AC-08 binds local Cedar authorization and runtime evidence to exact semantic
and execution roots.  It introduces one-shot =AuthorizedEffectToken= values,
atomic runtime mediation, replay/fork protection, and Strict/Batched/Diagnostic
durability profiles without transferring semantic ownership away from the
Scheme POO control plane.

* Ownership

- Scheme POO control plane canonically constructs the semantic root, intent,
  token envelope, issuer identity, and signature.
- Cedar evaluates policy and entities and returns a typed decision projection;
  Cedar is not a semantic or token owner.
- The runtime kernel atomically validates, reserves, binds, consumes, executes,
  observes, and commits execution/batch roots.
- Runtime adapters perform effects and return observations.  They do not issue
  tokens or decide organization semantics.
- Lean/Evidence proves declared bindings and inclusion properties.  It cannot
  upgrade an unobserved effect to L3 Verified.

Python, Rust, TypeScript, and other runtimes consume the same C ABI mediation
contract.  Python remains one production adapter, not a privileged owner.

The semantic root is stable across high-frequency runtime events.  The
execution root does not become an organization-semantic owner.  An immutable
mediation receipt is the only bridge between the two roots.

A single global root is rejected because it couples semantic and execution
mutation, increases contention, and obscures ownership.  A receipt-only chain
is rejected because batch inclusion, fork detection, and checkpoint
reconciliation become unnecessarily expensive.

* AuthorizedEffectToken contract

Every token binds:

- schema and version;
- token identity and unique nonce;
- semantic root;
- Bundle digest and epoch;
- Cedar policy, entity, and decision digests;
- intent digest, attempt identity, and effect kind;
- runtime and session identities;
- batch sequence range;
- arena identity, generation, payload offset/length, and payload digest;
- lease identity;
- issued-at, not-before, and expiry timestamps;
- revocation epoch;
- durability profile;
- required evidence bits;
- issuer identity and signature.

Tokens are one-shot capabilities.  Retry requires a new attempt identity,
nonce, and token.  Token substitution across Bundle, policy, runtime, session,
lease, buffer generation, sequence, or payload fails closed.

* Atomic mediation

    K-->>S: "mediation receipt"

Nonce reservation, lease binding, and token consumption form one kernel atomic
transaction.  Mutable-buffer TOCTOU is prevented by binding arena identity,
generation, range, and payload digest before entering execution.

=Indeterminate= is not deny, success, or safe-to-retry.  Once an effect may
have occurred, failure to obtain observation or durable root cannot be
collapsed into a deterministic outcome.

* Evidence roots

** Semantic root

The Scheme-owned root covers Bundle identity, Cedar policy/entities, intent,
and authorization decision.  Equivalent semantic inputs produce the same
root regardless of construction order.

** Execution root

Each session owns an incremental root over token consumption, runtime
transition, adapter observation, checkpoint, and external-effect evidence.
Every commit binds the previous root to expose forks.

** Batch root

The Batched profile aggregates execution leaves with a Merkle root.  Each leaf
still binds nonce, sequence, payload digest, and semantic root.  A batch root
cannot hide an omitted required receipt or replace the semantic root.

* Mediation receipt

The immutable receipt includes:

- before/after execution roots and semantic root;
- token, intent, Bundle, policy/entity, and decision digests;
- runtime/session/epoch/sequence/lease/generation bindings;
- committed, rejected, revoked, or indeterminate outcome;
- adapter observation and checkpoint/evidence references;
- durability profile and required evidence bits;
- replay and fork counters;
- kernel identity and signature.

* Durability profiles

** Strict

Strict is the default.  Every effect synchronously commits token consumption,
execution evidence, and the next execution root.  Missing required evidence
prevents an L2 claim.  A commit failure after a possibly executed effect yields
=Indeterminate=.

** Batched

Batched requires an explicitly negotiated capability.  It aggregates leaves
in =O(k)= and records the crash/recovery window, inclusion proof, and batch
root.  It is never selected implicitly.

** Diagnostic

Diagnostic validates and records prospective decisions but cannot transition
to =Executing= or authorize a production effect.

* Error policy

- Bundle, policy, decision, or semantic-root mismatch: pre-execution rejection.
- Stale epoch, expiry, not-before, revocation, nonce reuse, replay, token
  substitution, or fork: pre-execution negative receipt.
- Arena generation/range/payload-digest mismatch: TOCTOU rejection.
- Explicit adapter failure: durable failure observation; token remains spent.
- Possible effect with unknown observation or failed root commit:
  =Indeterminate=.
- Missing required receipt: no L2 Evidenced claim.
- L3 Verified requires concrete mediation and observation proof.

* Qualification matrix

** Determinism

- Semantic-root construction-order invariance.
- Deterministic execution leaf/root vectors.
- Stable batch inclusion proofs.

** Security

- Replay, fork, nonce reuse, and token substitution.
- Bundle/policy/decision/epoch/revocation mismatch.
- Expiry and not-before boundaries.
- Cross-runtime/session/lease/batch substitution.
- Mutable-buffer TOCTOU and omitted receipts.

** Durability

- Strict synchronous root commit and crash boundaries.
- Batched leaf recovery and inclusion proof after restart.
- Diagnostic proof that effect execution is unreachable.
- Checkpoint/root reconciliation and fork detection.

** Differential

- Scheme projection, C kernel, and Python CFFI agree on shared token vectors.
- Cedar decision receipt agrees with the semantic root.
- Lean/Evidence fixtures verify declared bindings without overstating L3.

** Performance

- Token validation, nonce reservation, and leaf hashing are =O(1)= per item.
- Batch Merkle construction is =O(k)=.
- Root-history lookup uses an ordered/indexed route rather than linear history
  replay.
- Strict and Batched have separate real-path gates.

* Exit criteria

1. AC-08 owner/test matrix assigns Cedar projection, token issuance, kernel
   mediation, root persistence, and negative fixtures.
2. All mismatch, replay, stale, reuse, substitution, TOCTOU, omission, and fork
   tests pass.
3. L2 Evidenced chains replay and verify from immutable receipts.
4. L3 is claimed only with concrete mediation/observation proof.
5. Strict is the default; Batched is negotiated; Diagnostic cannot execute.
6. Cross-language vectors and performance gates pass.
7. Qualification reports =abi-v1-frozen? #f=.

* Delivery boundary

The implementation plan may change only Scheme POO token/root projections,
Cedar adapter boundaries, runtime-v0 mediation/evidence APIs, Python CFFI
projections, focused proof/evidence fixtures, owner/status documents, and
qualification receipts.  Existing unrelated CLI/build/CI changes remain
outside the feature commit.
