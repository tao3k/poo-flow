#+title: AC-02 Immutable Organization Bundle Kernel Design

* Status

Approved in conversation on 2026-07-11.  Implementation and focused
qualification are complete.

* Objective

AC-02 adds a pure functional lifecycle kernel over the AC-01 Canonical
Organization Bundle.  The kernel turns immutable Bundle values into replayable
state transitions with explicit identity, epoch, validation, conflict, and
receipt semantics.

The public surface remains POO-native.  The kernel does not mutate Bundle
objects, introduce a parallel canonical representation, or embed runtime
language state.

* Scope

This feature owns:

- immutable Bundle kernel state;
- =open=, =validate=, and compare-and-advance operations;
- monotonic kernel epochs;
- identity and epoch conflict detection;
- deterministic POO-native kernel receipts;
- focused positive, conflict, rejection, no-op, replay, and immutability tests.

It does not own persistence, locks, threads, C handles, C ABI layout, Python
execution, Cedar evaluation, Lean proof generation, or downstream adapters.

* Owner boundary

=src/semantic/organization-bundle.ss= remains the owner of Bundle construction,
canonical normalization, SHA-256 identity, and semantic validation.

=src/semantic/organization-bundle-kernel.ss= becomes the owner of lifecycle
state and transitions.  It calls AC-01 operations through their public functions
and never copies normalization or validation rules.

=src/contract/organization-bundle.ss= remains the read-only Bundle contract
projection.  AC-02 does not add mutation through the contract layer.

* Kernel state

A kernel state is a POO object with these slots:

| Slot | Meaning |
|------+---------|
| =kind= | =poo-flow.organization-bundle-kernel-state= |
| =schema= | versioned kernel state schema |
| =phase= | =candidate=, =validated=, or =advanced= |
| =bundle= | immutable Bundle value associated with this state |
| =canonical-payload= | AC-01 normalized payload |
| =identity= | AC-01 Bundle identity |
| =epoch= | non-negative kernel evolution epoch |
| =previous-identity= | prior state identity or =#f= for the first state |
| =operation= | operation that produced the state |
| =validation-receipt= | AC-01 receipt or =#f= before validation |

The kernel state is a new immutable value after every successful semantic
transition.  An earlier state remains valid for replay and comparison.

* Operations

** Open

=poo-flow-organization-bundle-kernel-open= accepts a Bundle and optional digest
provider selection.  It canonicalizes the Bundle and computes its identity.  It
returns a =candidate= state and an accepted =kernel-opened= receipt.

Open does not claim semantic validity.  Canonicalization or identity failure
returns =kernel-canonicalization-failed= without constructing a partial state.

** Validate

=poo-flow-organization-bundle-kernel-validate= accepts a candidate or already
validated state.  It invokes the AC-01 validator.

An accepted validation returns a new =validated= state at the same epoch.  A
rejected validation returns =kernel-bundle-rejected= and embeds the complete
AC-01 validation receipt.  It does not construct a rejected state that could be
mistaken for an activatable Bundle.

Validating an already validated state is deterministic and idempotent.

** Compare and advance

=poo-flow-organization-bundle-kernel-advance= accepts:

- a validated or advanced current state;
- the caller's expected current identity;
- the caller's expected current epoch; and
- a next Bundle.

It checks identity and epoch before canonicalizing the next Bundle.  This order
prevents expensive or invalid next input from obscuring a stale caller.

If the next Bundle is semantically identical, the operation returns
=kernel-noop=, preserves the state and epoch, and remains accepted.

If the next Bundle changes canonical semantics and passes AC-01 validation, the
operation returns a new =advanced= state whose epoch is exactly current epoch
plus one and whose =previous-identity= is the current identity.

Rejected operations return receipts and leave the supplied current state
unchanged.  The diagram does not represent rejected values as new kernel states.

* Identity and epoch invariants

1. Identity is derived only from the AC-01 canonical payload and digest provider.
2. Kernel epoch expresses successful semantic evolution order only.
3. A no-op does not increment epoch.
4. A successful semantic advance increments epoch by exactly one.
5. A rejected operation does not produce a successor state.
6. The same current state and operation input produce equal receipts.
7. An old state remains unchanged after every successor is constructed.
8. Identity conflicts and epoch conflicts fail closed before next-Bundle work.

* Receipt contract

A kernel receipt is a POO object with:

| Slot | Meaning |
|------+---------|
| =kind= | =poo-flow.organization-bundle-kernel-receipt= |
| =schema= | versioned receipt schema |
| =operation= | =open=, =validate=, or =advance= |
| =accepted?= | operation acceptance |
| =code= | stable result code |
| =before-identity= | current identity or =#f= |
| =after-identity= | successor identity or =#f= |
| =before-epoch= | current epoch or =#f= |
| =after-epoch= | successor epoch or =#f= |
| =validation-receipt= | embedded AC-01 receipt when validation ran |
| =diagnostics= | deterministic ordered diagnostic POO values |

The receipt contains no timestamps, process identifiers, foreign pointers,
runtime handles, or mutable tables.

* Stable result codes

| Code | Accepted | Meaning |
|------+----------+---------|
| =kernel-opened= | yes | candidate state constructed |
| =kernel-validated= | yes | Bundle validation accepted |
| =kernel-noop= | yes | next canonical payload equals current payload |
| =kernel-advanced= | yes | valid semantic successor constructed |
| =kernel-canonicalization-failed= | no | Bundle cannot enter canonical domain |
| =kernel-bundle-rejected= | no | current Bundle validation rejected |
| =kernel-identity-conflict= | no | expected identity differs from current |
| =kernel-stale-epoch= | no | expected epoch differs from current |
| =kernel-next-bundle-rejected= | no | next Bundle canonicalizes but fails validation |
| =kernel-invalid-phase= | no | operation is illegal for the current phase |

There are no fallback transitions or compatibility result codes.

* Qualification

The focused AC-02 scenario must prove:

1. open then validate succeeds for the frozen AC-01 Bundle;
2. successful semantic advance changes identity and increments epoch by one;
3. the previous state remains unchanged;
4. a semantic no-op preserves both state identity and epoch;
5. stale epoch fails before next-Bundle processing;
6. identity conflict fails before next-Bundle processing;
7. invalid next Bundle returns its AC-01 validation evidence;
8. invalid current Bundle cannot become validated;
9. repeated equal inputs yield equal normalized receipts;
10. illegal phase transitions fail closed;
11. AC-01's eight cases and twenty checks remain green;
12. module compilation, Org parsing, and =git diff --check= pass.

* Delivery

Design, implementation plan, semantic owner, focused tests, owner matrix update,
qualification receipt, and AC-01 regression evidence form one complete feature
commit.  Pre-existing build and CLI-support changes remain outside that commit.
