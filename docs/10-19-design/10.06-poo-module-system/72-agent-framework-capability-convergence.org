#+title: Agent Framework Capability Convergence
#+startup: overview

* Purpose

LangChain, LangGraph, CrewAI, Pi Agent core, and similar agent frameworks expose many overlapping capabilities:

- model and tool calling loops;
- agent/session state;
- graph or flow routing;
- task and process dependencies;
- checkpoint, resume, and human-in-the-loop;
- memory, attachments, and context management;
- guardrails, permissions, and sandbox/tool scopes;
- streaming events, traces, and observability.

POO Flow should not clone each framework one by one. POO Flow should own the common control-planner layer that those frameworks repeatedly need: reusable policy objects, strategy composition, graph/loop/durable intent, proof-facing facts, and runtime handoff receipts.

The important point is direction. POO Flow does not map itself into LangGraph, CrewAI, LangChain, or Pi Agent core. It lifts their shared policy/control layer into reusable POO profiles and composition, then lowers the verified plan into a runtime language such as Python.

* POO Flow Advantage

The advantage is not only that POO Flow can express these patterns. The advantage is that POO Flow can express them with maximal reuse and a proof boundary.

- A shared profile can be reused by many downstream compositions.
- A local override can solve a one-off scenario without polluting the reusable library.
- A stable override can be promoted to =user-interface/profiles= and reused by other users.
- A reusable composition can live under =user-interface/cases= and serve as both documentation and executable scenario input.
- Python or another runtime language implements the data-plane behavior.
- Lean checks the mathematical facts that Scheme generated and the runtime receipt returned.

#+begin_src latex
\[
\operatorname{SharedProfile}(p)
  \land
\operatorname{Composition}(c)
  \land
p \in c
  \Rightarrow
\operatorname{ReuseWithoutDuplication}(p,c)
\]

\[
\operatorname{Override}(o)
  \land
\operatorname{PreservesContract}(o,p)
  \Rightarrow
\operatorname{SafeSpecialization}(o,p)
\]
#+end_src

This is why POO Flow is a control planner rather than another agent framework. It pushes policy, strategy, composition, proof, and runtime receipt alignment to the strongest layer, then lets runtime languages implement the operational behavior.

* Runtime Language Boundary

Python is currently the concrete runtime language target. The Python runtime already demonstrates long graph execution, router execution, stream/events, interrupt/resume, reducer/handoff behavior, and C ABI graph validation. Therefore, implementing LangGraph-like, CrewAI-like, LangChain-like, or Pi Agent core-like behavior should be a lowering problem:

#+begin_src text
POO user composition
  -> Scheme receipt
  -> Python RuntimeGraphProgram / runtime binding
  -> execution receipt
  -> Lean generated facts
#+end_src

The Scheme layer should only grow new source capabilities when the missing feature is a control-plane primitive, such as a new graph fact, proof obligation, profile contract, or composition clause. If the missing feature is execution, IO, storage, provider integration, or event streaming, it belongs in the runtime language.

* Proof Value

Frameworks can execute plausible agent plans. POO Flow should reject unsafe plans before runtime execution.

#+begin_src latex
\[
\operatorname{CanRun}(f,p)
  \not\Rightarrow
\operatorname{PolicySound}(p)
\]

\[
\operatorname{POOComposition}(p)
  \land
\operatorname{SchemeFacts}(p)
  \land
\operatorname{LeanDischarge}(p)
  \land
\operatorname{RuntimeReceiptMatches}(r,p)
  \Rightarrow
\operatorname{PolicySoundRun}(r)
\]
#+end_src

This is the reason to invest in Scheme macros, POO profile composition, C ABI receipts, high-performance runtime projection, and Lean proofs at the same time. The user should be able to build a stronger agent than a single framework template by composing verified reusable policies instead of rewriting the same runtime and safety code again.

* Scenario Implication

Existing scenarios should be interpreted as examples of the same convergence layer:

- =profiles/langchain.ss= and =cases/langchain-linear.ss= show the linear agent loop / tool-free chain shape.
- =profiles/langgraph.ss= and =cases/langgraph-state.ss= show graph routing, bounded loop, and runtime handoff.
- =profiles/crewai.ss= and =cases/crewai-research-flow.ss= show agent/task/crew/flow composition.
- =profiles/tool-calling.ss= and =cases/tool-calling-agent-loop.ss= show the shared tool calling control-plane layer.
- Future Pi Agent core-style scenarios should focus on session state, tool loop, attachment/message state, event streaming, and runtime provider bindings.

The implementation target is not a separate module per framework. The target is a small number of strong POO Flow primitives that make these framework-style capabilities cheap to express, prove, and run.
