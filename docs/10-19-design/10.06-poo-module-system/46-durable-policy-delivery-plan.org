* 10.06.46 Durable Policy Delivery Plan
** 10.06.46.10 Status
Active staged implementation plan.

This plan turns [[file:45-durable-policy-research-agenda.org][45 Durable Policy Research Agenda]] into an executable delivery path.
It keeps the same architecture:

#+begin_src text
POO durable policy
  -> bounded Scheme receipt
  -> runtime fact log
  -> derived indexes
  -> replay/backfill/repair jobs
  -> resumable session, memory, workflow, sandbox, and artifact state
#+end_src

The plan is intentionally staged.
Durable touches session identity, memory, workflow execution, sandbox placement, artifacts, observability, and runtime handoff.
If those are implemented in isolated fragments, the result will be a weak checkpoint feature instead of a durable substrate.

Current implementation state:
- P0 contract vocabulary is closed enough to support code slices.
- P1 shared POO durable policy and session policy composition are landed.
- P2 first =defstruct= receipt and batch projection gate are landed for durable policy.
- P3 runtime store contract receipt projection, backend negotiation, operation receipt projection, and Marlin handoff ABI are landed.
- P5 first multi-agent graph durable refs and communication ledger receipt projection are landed.
- P6 first memory durable job receipt projection is landed.
- P7 first workflow/CI/CD/artifact durable projection is landed.
- P4 first crash/replay/repair scenario receipt and recovery observability gate is landed.
- P5, P6, and P7 still need runtime-backed completion slices; P3 still needs the Rust implementation behind the handoff ABI.

** 10.06.46.20 Goal
Deliver durable as a shared POO capability for long-running agent projects.

Success means:
- user-facing configuration stays POO-native;
- durable policy can be inherited by session, memory, workflow, sandbox, and artifact policies;
- Scheme validates policy shape and projects bounded receipts;
- hot receipts use Gerbil =defstruct= internally before alist/JSON ABI projection;
- Rust/Marlin owns fact logs, indexes, checkpoints, leases, recovery, backfill, and side-effect execution;
- crash/replay/recovery behavior is tested as a product feature, not only as serialization tests.

** 10.06.46.30 Approach Decision
*** 10.06.46.30.10 Rejected: Checkpoint Wrapper Only
A checkpoint wrapper is useful for user-facing restore, but it does not cover shell side effects, sandbox handles, sub-agent transcripts, memory writes, runtime jobs, or external tools.
It can be one backend capability later, not the architecture.

*** 10.06.46.30.20 Rejected: Session Transcript Only
Append-only session persistence is necessary, but incomplete.
It does not define derived indexes, job leases, repair semantics, memory consolidation, workflow compensation, or sandbox recovery.

*** 10.06.46.30.30 Chosen: Event Log + Checkpoint + Derived Index
This is the durable substrate:
- fact log is canonical;
- checkpoints are bounded materialized resume points;
- derived indexes are rebuildable;
- jobs use leases and watermarks;
- side effects declare replay/compensation behavior;
- session ids are policy anchors;
- memory is a durable pipeline, not a transcript blob.

** 10.06.46.35 Evidence-To-Delivery Mapping
The research agenda gives five implementation pressures. Each pressure becomes a concrete delivery line:

1. Codex evidence: fact logs, derived indexes, backfill, repair, memory jobs, and parent/child spawn edges.
   Delivery line: P3 runtime store contract, P4 crash/replay/repair tests, P5 durable session graph, and P6 memory jobs.

2. OpenRath evidence: =Session -> Agent/Workflow -> Session= is the right functional shape, but fork/merge and sandbox sharing are too implicit.
   Delivery line: P5 session graph policies must make lineage, communication, sandbox sharing, and write-back explicit POO policies and receipts.

3. Claude evidence: checkpointing and subagents are good user affordances, while tool/memory execution remains application-owned.
   Delivery line: Scheme exposes ergonomic durable/session policies, but runtime side effects, tool execution, memory storage, and repair remain explicit handoff contracts.

4. Durable execution systems evidence: Temporal and Durable Task use persisted event history and deterministic replay constraints rather than heap snapshots.
   Delivery line: P3/P4 must validate replay classes and runtime event history before any workflow claims resumability.

5. Recovery and workflow literature: ARIES, sagas, and workflow patterns separate logging, compensation, resources, and exception behavior.
   Delivery line: P7 workflow/task/artifact receipts must declare action class, artifact provenance, sandbox refs, and compensation refs.

This mapping is the guardrail against copying a reference project. =poo-flow= should absorb the durable invariant, then express it as POO-native policy plus functional flow values.
** 10.06.46.40 Phase Map
*** 10.06.46.40.10 P0: Contract Closure
Purpose:
Close the vocabulary and object boundaries before implementation.

Deliverables:
- final public object names:
  =durable-policy=, =checkpoint-policy=, =journal-policy=, =index-policy=, =resume-policy=, =repair-policy=;
- final identity fields:
  =project-id=, =root-session-id=, =session-id=, =parent-session-id=, =agent-id=, =plan-id=, =loop-run-id=, =checkpoint-id=, =channel-id=;
- durable action classes:
  =replayable=, =idempotent=, =compensatable=, =terminal=, =manual=;
- module placement decision:
  shared durable objects live in the module-system object layer, while concrete session/memory/workflow modules inherit or reference them;
- validation error vocabulary:
  missing journal owner, missing checkpoint store, missing resume identity, illegal repair mode, unsafe replay class, unresolved runtime owner.

Validation gate:
- Org docs parse through ASP.
- No Scheme code is required yet.
- Index links to the durable RFC, research agenda, and delivery plan.

Exit criteria:
- A maintainer can answer where durable objects live and which modules consume them.

*** 10.06.46.40.20 P1: POO Object Layer
Purpose:
Implement durable as POO-native object families.

Implementation scope:
- add shared durable object prototypes;
- add checkpoint, journal, index, resume, and repair object prototypes;
- extend session, memory, workflow, sandbox, and artifact policy objects to inherit or reference durable prototypes;
- keep user-facing forms object-native; no raw alist DSL or ad hoc patch language;
- keep validation in structured object validation, not in tests or case-specific transformers.

Expected source ownership:
- =src/module-system/= for shared durable object support;
- =src/modules/session/= for session policy integration;
- =src/modules/workflow= or the existing workflow/funflow module area for workflow/task durable integration;
- =src/modules/sandbox-core/= for sandbox durable placement policy;
- memory module area once the concrete module boundary is settled.

Validation gate:
- object validation rejects fake profiles;
- POO inheritance works for shared default durable policy and specialized session/memory policy;
- user-interface examples show concise policy activation and object extension.

Exit criteria:
- A downstream profile can define one =durable/default= object and have session/memory/workflow policies inherit it without duplicate configuration.

*** 10.06.46.40.30 P2: Receipt Projection Layer
Purpose:
Lower validated POO objects into fixed runtime receipts.

Implementation scope:
- introduce =defstruct= receipts:
  =durable-policy-receipt=,
  =checkpoint-policy-receipt=,
  =journal-policy-receipt=,
  =resume-policy-receipt=,
  =repair-policy-receipt=,
  =session-lineage-receipt=,
  =memory-job-policy-receipt=,
  =communication-channel-receipt=;
- provide explicit =->alist= projection only at ABI/manifest boundaries;
- avoid hot-path nested POO =.ref= walks;
- keep diagnostics attached to receipts.

Validation gate:
- focused unit tests compare POO input to struct receipt fields;
- manifest projection tests verify bounded alist shape;
- performance gate covers user-interface projection batch latency and allocation pressure.

Exit criteria:
- Runtime handoff can consume durable policy without reading POO object graphs.

*** 10.06.46.40.40 P3: Runtime Store Contract
Purpose:
Define the Rust/Marlin-owned durable runtime ABI.

Contract scope:
- fact log categories:
  session turn, tool request/result, spawn edge, memory fact, sandbox event, artifact event, workflow task event, repair event;
- checkpoint store:
  checkpoint id, parent checkpoint id, materialized state references, schema version, integrity data;
- derived indexes:
  sessions, session edges, artifacts, latest task state, memory summaries, job state, communication ledger;
- job store:
  job kind, job key, lease owner, ownership token, retry count, retry-at, watermark, last success watermark, last error;
- repair journal:
  corruption backup, rebuild attempt, compensation action, quarantine reason, human decision.

Scheme responsibility:
- validate and project the contract.

Runtime responsibility:
- storage backend, locks, leases, fsync/atomicity, backfill, corruption handling, concurrency, and side effects.

Validation gate:
- contract examples are executable as alist/JSON-like manifests;
- schema versioning is explicit;
- illegal missing runtime-owner or store-owner is rejected before handoff.

Exit criteria:
- Scheme and runtime teams can implement independently against one contract.

*** 10.06.46.40.50 P4: Crash, Replay, And Repair Tests
Purpose:
Make durable behavior observable and testable.

Test scenarios:
- crash after fact append but before derived index update;
- crash during checkpoint write;
- derived index missing rows and backfill rebuild;
- memory job lease expiry and retry;
- child session failure and parent resume;
- sandbox handle loss with repair policy;
- non-deterministic workflow branch rejected by replay policy;
- compensation action generated for a failed workflow task;
- corrupted runtime DB/index quarantined without destroying unrelated state.

Validation gate:
- each scenario produces a durable receipt and an observability event;
- replay either succeeds deterministically or fails with a typed repair requirement;
- tests use the canonical project harness path.

Exit criteria:
- Durable has product-level tests, not only serializer tests.

*** 10.06.46.40.60 P5: Multi-Agent Session Graph
Purpose:
Make project/root/child sessions a durable communication and policy graph.

Implementation scope:
- persist parent-child session edges;
- add sibling communication channels;
- enforce context/history/sandbox/resource grants by session id;
- ensure child write-back is a receipt/message, not direct parent mutation;
- support resumable child sessions by id;
- support project-level routing across multiple root sessions.

Validation gate:
- child sessions cannot read sibling context without a grant;
- child sessions cannot share sandbox handles without policy;
- parent can resume a child by id;
- sibling communication requires declared channel id.

Exit criteria:
- A project can run multiple root sessions and multiple child agents with explicit isolation and communication rules.

*** 10.06.46.40.70 P6: Memory Pipeline
Purpose:
Turn memory into durable jobs and policy, not context text.

Implementation scope:
- separate memory categories:
  user-authored memory, auto memory, session summary, reflection, extracted fact, skill/artifact promotion;
- add extraction and consolidation job policies;
- add watermarks, usage counters, stale-source checks, retry policy, and retention;
- make memory recall policy session-aware;
- keep memory context distinct from enforcement policy.

Validation gate:
- fake memory profiles without durable store or job policy are rejected;
- stale transcript/source triggers extraction job;
- consolidation job records usage and last success watermark;
- recall can be scoped by project/root/session/child session id.

Exit criteria:
- Memory can survive restart and be rebuilt or compacted without replaying every raw transcript into user context.

*** 10.06.46.40.80 P7: Workflow, CI/CD, And Artifact Integration
Purpose:
Extend durable from agent loops into the broader =poo-flow= domain.

Implementation scope:
- task policy declares action class:
  replayable, idempotent, compensatable, terminal, manual;
- workflow task receipts include durable task id, dependency edge, artifact refs, sandbox refs, and compensation refs;
- CI/CD examples produce durable build/test/artifact/release receipts;
- artifacts become stable handles with provenance and retention policy;
- deploy/cleanup tasks can express compensation.

Validation gate:
- workflow replay reuses completed idempotent tasks when policy permits;
- non-idempotent tasks require checkpoint or manual policy;
- artifact receipts survive session resume;
- failed deploy can produce cleanup/compensation receipt.

Exit criteria:
- Durable applies to Funflow/workflow and CI/CD, not only chat/session.

** 10.06.46.50 Cross-Cutting Requirements
*** 10.06.46.50.10 POO-Native Public Surface
All public configuration remains POO-native.
Objects, prototype inheritance, slot operators, and module macros are the normal path.
Raw alists only exist at bounded ABI/manifest projection boundaries.

*** 10.06.46.50.20 Functional Flow Discipline
Flow composition should remain value-oriented:

#+begin_src text
Session -> Workflow/Agent -> Session
TaskState -> RuntimeAction -> TaskState
MemorySource -> ExtractionJob -> MemoryFact
#+end_src

Side effects belong behind runtime handoff receipts.

*** 10.06.46.50.30 Observability
Every durable event should be observable:
- fact append;
- checkpoint write;
- replay start/end;
- repair start/end;
- job claim/finish/fail;
- compensation generated/executed;
- session edge created;
- channel message sent/received;
- sandbox handle attached/lost/repaired.

Observability should hook into object validation and receipt projection so debugging can start from =objects.ss= and user-interface profiles.

*** 10.06.46.50.40 Performance Gates
Performance gates must cover:
- POO object validation with inherited durable policy;
- struct receipt construction;
- bounded alist projection;
- user-interface projection batch;
- memory policy projection;
- session graph projection.

The target is not micro-optimizing user authoring.
The target is preventing dynamic POO object graphs from leaking into runtime hot paths.

*** 10.06.46.50.50 Package And Build Discipline
Use package-managed Gerbil flows.
Do not add ad hoc environment scripts or package-manager state cleanup.
Focused gates should use the project CLI and canonical package harness path.

** 10.06.46.60 Dependency Order
1. P0 must finish before any code lands.
2. P1 can land without runtime implementation, but must include validation.
3. P2 depends on P1 object shape.
4. P3 depends on P2 receipt names and fields.
5. P4 depends on P3 contract examples.
6. P5 can start after P2 for receipt shape, but full recovery requires P3/P4.
7. P6 depends on P1 durable policy and P3 job-store contract.
8. P7 depends on P2 receipts and P3 artifact/task contract.

** 10.06.46.66 Slice Cadence
Each implementation slice should be large enough to prove a feature boundary, not a single-field patch.

Required slice contents:
- update the POO object or policy owner;
- add or extend a =defstruct= receipt;
- add bounded =->alist= projection only at ABI/manifest boundary;
- add structured validation diagnostics;
- update at least one user-interface case when the feature affects users;
- add focused tests, plus a performance gate when the slice touches projection hot paths;
- record delivery notes in this document.

Preferred slice order from the current state:
1. M4 crash/replay/repair scenario harness.
2. M3 runtime store backend negotiation and Rust/Marlin implementation handoff.
3. M7 runtime-backed workflow replay/artifact/compensation behavior.
4. M8 observability and performance closure.

** 10.06.46.80 Risk Register
*** 10.06.46.80.10 Risk: Durable Becomes A Generic Option Tree
Mitigation:
Keep public shape as POO objects.
Use inheritance and validation, not Nix-style option merging.

*** 10.06.46.80.20 Risk: Runtime Concerns Leak Into Scheme
Mitigation:
Scheme defines policy and receipts.
Rust/Marlin owns storage, concurrency, leases, and IO.

*** 10.06.46.80.30 Risk: Recovery Is Tested Only As Serialization
Mitigation:
P4 requires crash/replay/repair scenario tests with observability events.

*** 10.06.46.80.40 Risk: POO Object Graphs Become Runtime Hot Path
Mitigation:
Use =defstruct= receipts after validation.
Only project to alists at ABI boundaries.

*** 10.06.46.80.50 Risk: Session Graph Security Is Too Coarse
Mitigation:
Use session ids as policy anchors and require explicit context/history/sandbox/channel grants.

*** 10.06.46.80.60 Risk: Memory Pollutes Enforcement Policy
Mitigation:
Treat memory as context and job output.
Keep enforcement in validation, hooks, runtime policy, and grants.

** 10.06.46.90 Completion Definition
Durable is considered delivered when:
- POO policy objects exist and are validated;
- session and memory can inherit/reference durable policy;
- runtime receipts are =defstruct= based and bounded at ABI projection;
- runtime store contract covers fact log, checkpoint, index, job, memory, repair, communication, and sandbox ledgers;
- crash/replay/repair scenario tests exist;
- multi-agent session graph uses ids and policy grants;
- workflow/CI/CD task receipts declare replay and compensation semantics;
- observability exposes durable events across object validation, projection, and runtime handoff.
