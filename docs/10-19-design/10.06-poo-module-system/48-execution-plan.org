* 10.06.48 Execution Plan
This is the executable advancement plan for =poo-flow=. It starts from the structure audit in =10.06.47= and turns the project direction into ordered work packages.

The plan is operational rather than aspirational. Each package has a purpose, ownership boundary, deliverables, validation surface, and exit criteria. We use it to decide what to start next, what must remain stable, and when a slice is complete enough to move forward.

** 10.06.48.10 Operating Principles
1. Public authoring stays POO-native. Users and agents write =.def=, POO objects, prototype extension, slot operators, and functional constructors. Internal macros may generate code, but they must not become a second user language.
2. Functional composition is the control-plane default. Flow, loop intent enrichment, sandbox agreement, policy selection, and receipt assembly should be value-in/value-out until an explicit runtime or Marlin handoff boundary.
3. Alists are boundary payloads only. Runtime manifests, ABI receipts, snapshots, and report-only handoff payloads may be alists. User configuration and internal extension surfaces should remain POO-native or functional.
4. Gerbil macros are an internal compiler for repeated POO-native and functional structures. Use declarative macros for fixed families, procedural macros for checked syntax families, and compile-time helpers only when they produce ordinary Scheme functions and POO objects.
5. Testing follows the current testing framework. Focused single-file selection is handled by native =gxtest=. This plan names test files and scenario gates rather than prescribing obsolete wrapper commands.
6. Scenarios are acceptance gates. Every structural refactor must be tied to at least one real =user-interface/*= scenario and one focused test file. User-facing scenarios must demonstrate actual sandbox, loop, funflow, session, or Marlin use cases.
7. Scheme remains the strategy/control plane. This repository owns POO-native configuration, functional composition, validation, presentation, manifest/receipt projection, and report-only handoff. Heavy execution and scheduling belong in =marlin-agent-core= or another runtime owner.
8. Marlin integration is consumer-owned. The Gerbil package under =marlin-agent-core/crates= should depend on =poo-flow= through =gerbil.pkg= and adapt the ABI there. =poo-flow= provides Scheme-side contracts, runtime manifest discovery, and receipt/request shapes; it does not become the direct Rust ABI implementation owner.

** 10.06.48.20 Start Conditions
Before starting a package:
- Identify the source owner through ASP owner/selector evidence.
- Identify the user-facing scenario that exercises the behavior.
- Identify the focused =gxtest= single-file selection surface.
- Confirm whether the change is public, internal, projection-only, or downstream ABI-facing.
- Keep generated code boring: ordinary POO values, functional helpers, fixed projections, or final boundary alists.
- Keep =AGENTS.md= focused on current project programming style; put delivery plans in =docs/10-19-design/10.06-poo-module-system/=.
