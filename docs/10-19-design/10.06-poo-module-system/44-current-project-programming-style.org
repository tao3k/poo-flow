* 10.06.44 Current Project Programming Style

This document is the current programming contract for =poo-flow=. It narrows
the broader POO guidelines into rules agents should apply while changing the
present codebase.

** 10.06.44.10 Core Rule

=poo-flow= is a Gerbil Scheme control plane built from POO-native objects and
functional composition. New code must not drift into a parallel record DSL,
alist DSL, or hand-maintained projection language.

Use Gerbil metaprogramming for mechanical Scheme structure, not for hiding the
model:

1. Public extension surfaces remain POO-native: =.def=, =.o=, =.ref=, =.mix=,
   slot operators, prototype composition, and named functional helpers.
2. Internal repeated object families should be generated by Gerbil declarative
   macros, procedural macros, or bounded compile-time metaprogramming when the
   generated code expands to ordinary POO objects.
3. Runtime, ABI, JSON-like receipts, and Marlin handoff payloads may be alists
   only at projection boundaries.
4. Heavy execution stays outside Scheme. Scheme owns strategy, composition,
   validation, presentation, and report-only handoff data.

** 10.06.44.20 Macro And Metaprogramming Policy

Prefer Gerbil metaprogramming when all of these are true:

- the repeated code is structural rather than business logic;
- the expansion target is ordinary POO or functional code;
- the call site names the domain object, slots, contracts, and ownership
  clearly;
- the macro removes duplicated constructor/projection/validator boilerplate
  without hiding extension slots from users.

Do not introduce a macro when a normal function or higher-order helper is
clearer. Filtering, folding, and bounded list transforms should usually stay
functional unless they are part of a larger repeated declaration form.

Use the smallest macro tier that carries the structure:

1. Declarative macros, such as =defrules=, are the default for repeated
   constructor families, binding forms, category walkers, and direct
   syntax-to-POO lowering.
2. Procedural macros, such as =defsyntax= with =syntax-case=, =with-syntax=,
   =syntax->datum=, and =datum->syntax=, are appropriate when the macro must
   validate shape, normalize clauses, build identifiers, or emit different
   expansions from the same public form.
3. Broader compile-time metaprogramming is appropriate for generated object
   families, projection families, and contract tables when the generator keeps
   the ownership boundary explicit and has focused expansion tests.

The preferred declarative internal shape is:

#+begin_src scheme
(defrules defpoo-domain-family ()
  ((_ binding domain-identity slot-spec ...)
   (def binding
     (poo-domain-object 'domain-identity slot-spec ...))))
#+end_src

The preferred procedural internal shape is:

#+begin_src scheme
(defsyntax (defpoo-domain-projection stx)
  (syntax-case stx ()
    ((_ binding source slot ...)
     (with-syntax (...)
       #'(def (binding source)
           ...)))))
#+end_src

The non-preferred shape is a macro that creates a new user DSL whose expansion
must be memorized before the POO object model can be understood.

** 10.06.44.22 Gerbil Macro Research Baseline

The local research checkout is =.data/gerbil=. Gerbil is not just a hygienic
macro system; its standard library uses a full expansion-time programming
model:

- =src/gerbil/core/expander.ss= exposes =syntax-rules=, =syntax-case=,
  =with-syntax=, =datum->syntax=, =syntax->datum=, =syntax-local-value=,
  generated identifiers, syntax source wrapping, local expander context,
  import/export expanders, and user expanders.
- =src/std/generic/macros.ss= uses =begin-syntax= compile-time objects plus
  procedural =defsyntax= forms to register generic metadata, inspect type
  information with =syntax-local-value=, generate implementation identifiers,
  and emit normal runtime dispatch functions.
- =src/std/protobuf/macros.ss= is the stronger model for declaration-heavy
  forms: it parses a compact declaration, checks duplicates and invalid type
  combinations at expansion time, builds generated identifiers with =format-id=,
  and emits ordinary structs, methods, readers, writers, and syntax metadata.
- =src/gerbil/runtime/syntax.ss= shows the small-tier pattern: =defrules=
  handles direct syntactic lowering, while a bounded procedural helper handles
  recursive pattern generation and syntax errors.

The project implication is direct: when the syntax is static, prefer
expansion-time validation over runtime alist parsing. When several declarations
must share compile-time facts, use a small =begin-syntax= record or class owned
by the macro module. The output still must be ordinary POO-native or functional
code.

** 10.06.44.23 Macro Tier Decision Rules

Use this list before adding another hand-written object or projection family:

- Constant frame with caller-supplied names and slots: use =defrules=
  declarative macros. Local targets are sandbox backend object families and
  runtime receipt projections.
- Static declaration that needs validation, defaulting, generated identifiers,
  or alternate expansions: use =defsyntax= with =syntax-case= and
  =with-syntax*=. Local targets are policy object declarations and
  =use-module= branch generation.
- Multiple declarations sharing compile-time metadata: use a =begin-syntax=
  record or class plus a procedural macro. Local targets are policy registries,
  contract tables, and reusable profile family metadata.
- Runtime user choices, filtering, folding, or data-dependent behavior: use a
  function or functional combinator. Local targets are flow composition,
  profile selection, and report filtering.

The elegance rule is: the call site names the domain facts, and the macro owns
only the repeated frame. A good project macro removes boilerplate around POO
object construction, slot projection, validation, and receipt emission. It must
not hide which object, policy, profile, slot, contract, or runtime boundary is
being declared.

** 10.06.44.24 Advanced Gerbil Feature Map
Use these features as engineering tools, not as surface complexity. The repair rule for agent-authored code is: if the code is a repeated structural frame, move it to the smallest Gerbil mechanism that owns that frame. If the code is business behavior, keep it as a function and use =match=, =using=, folds, and named helpers to make it direct.

- =match= extension and =match-macro=: use when a domain object has a recurring destructuring pattern that is currently expressed by nested =cond=, =and=, =object?=, =.ref=, and list checks. The match extension should make the pattern explicit and keep the body smaller. Do not use it to hide policy decisions or runtime searches.
- Syntax parameters: use when a form has context-bound names that are only valid inside that form, such as receive-envelope fields or module expansion context. This is the right answer for lexical context facts; it is not a substitute for passing normal runtime arguments.
- Compile-time metadata objects: use =begin-syntax= classes or records plus =syntax-local-value= when multiple declarations must share static facts, such as policy kind metadata, profile family metadata, contract slot metadata, or generated accessor/projection ownership. This should replace duplicated runtime rows, not create a second registry that users must know.
- Generated identifiers: use =format-id=, =stx-identifier=, =genident=, and =gentemps= when the generated API has a stable naming convention. Keep source locations with =syntax/loc= or =stx-wrap-source= so expansion errors point at the user declaration.
- Parser and lexer generators: use expansion-time parsers when the accepted language is static, such as fixed contract syntax, profile declaration syntax, policy selector syntax, or =use-module= branches. Do not parse these repeatedly at runtime.
- Generic methods and interfaces: use =defgeneric=, =defmethod=, =using=, and interface casts when the extension point is type or object behavior. This is preferable to growing a long symbolic =case= over object kind when downstream modules should add behavior.
- Specialized tables and caches: use generated table APIs or explicit cache structs when a hot path repeatedly looks up symbolic keys, object classes, or registry entries. Keep hash/equality choices visible; do not hide a performance-sensitive registry behind untyped helper names.
- Source-aware macros: use =defsyntax-call= or source-location helpers when generated code needs the declaration's file, position, or colocated resource. Avoid runtime current-directory guesses for static assets.
- FFI and runtime adapter macros: use boundary macros when the repetitive structure is an ABI adapter. The macro may be large internally, but the public project API must stay named, typed, and POO-native.
*** Agent Repair Signals
Use these signals while reviewing or repairing generated Scheme:
- Nested =cond= blocks that repeatedly check object kind, slot presence, and list shape usually want =match=, a named predicate, or a domain match macro.
- Runtime parsing of fixed declaration syntax usually wants =defsyntax= parsing and expansion-time errors.
- Repeated constructor, accessor, validator, and projection rows usually want a declaration macro backed by compile-time metadata.
- Mutable module globals used only to communicate macro context usually want syntax parameters.
- Long =case= branches over object behavior usually want generic methods, interfaces, or a generated dispatch table.
- Hot symbolic lookup paths usually want a specialized table, explicit cache object, or cached prototype rather than ad hoc hash helpers.
- Boundary marshalling code may generate repetitive adapters, but the public side must remain POO-native and typed.
** 10.06.44.25 Macro Ownership Discipline

Gerbil macros in this project are owned by the layer whose structure they
generate:

- =src/core/*-syntax.ss= owns functional flow authoring macros.
- =src/module-system/*-syntax.ss= owns module object, profile, registry,
  manifest, and use-module expansion helpers.
- =src/modules/*/config.ss= may expose narrow user-facing macros only when
  they lower to POO objects or functional constructors.

Macro implementations must keep expansion targets boring. Generated code should
look like hand-written POO-native code: =.o=, =object<-alist= at adapter edges,
=poo-flow-module-object=, =poo-flow-module-field-contract=, functional
constructors, fixed projection helpers, or final alist receipts.

Procedural macros should be used more aggressively than ad hoc runtime parsing
when the accepted syntax is static. Shape errors should fail at expansion time
when possible, before runtime validation or Marlin handoff.

** 10.06.44.27 Project Adoption Pattern

Current code already contains the preferred direction:

- =src/modules/session/policy-syntax.ss= defines
  =defpoo-session-policy-family=, and =src/modules/session/policy.ss= uses it
  to declare many policy constructors while keeping policy kind, schema, scope,
  slots, and validation explicit at the call site.
- =src/module-system/sandbox-backend-object-syntax.ss= defines
  =defpoo-sandbox-backend-object-family=, a declarative macro that generates
  ordinary =poo-flow-module-object= and =poo-flow-module-field-contract= values
  for backend object families.
- =src/module-system/runtime-projection-syntax.ss= defines
  =defpoo-runtime-receipt-projection= for fixed final alist receipt boundaries.

Deepening the style means extending this pattern, not inventing a parallel DSL.
The next reusable family should be a POO policy declaration that can generate,
from one named spec:

1. the constructor frame;
2. slot accessors and optional predicates;
3. focused expansion-time shape checks;
4. runtime validations that still call named project validators;
5. final =->alist= projection rows at the boundary;
6. focused scenario tests that import the public module.

Use procedural macros only after the declarative form cannot express the static
checks or identifier generation cleanly. When a procedural macro is used, keep
helper functions small and named after the generated artifact:
=parse-policy-slots=, =emit-policy-constructor=, =emit-policy-projection=,
=emit-policy-validation=.
** 10.06.44.30 Current Refactoring Targets

These targets are ordered by value and risk:

1. Session policy families. The constructor family already uses
   =defpoo-session-policy-family=; the next win is to pull accessors, common
   predicates, validators, and =->alist= rows from one POO policy spec instead
   of hand-maintaining slot reads and projection lists.
2. Sandbox backend object families. =defpoo-sandbox-backend-object-family=
   already covers the backend module-object frame. The next closure is to
   ensure =nono-sandbox=, =cubeSandbox=, and =docker-sandbox= keep public
   profile helpers explicit while sharing internal backend object, capability
   registry, profile object, and catalog generation.
3. Loop runtime manifests and Marlin discovery receipts.
   =defpoo-runtime-receipt-projection= is the base pattern; extend it only for
   stable contract/projection shapes whose fields are final boundary alists.
4. =use-module= syntax branches. Keep the public syntax stable, but generate
   repetitive module-specific branches from procedural macro helpers when the
   branches share the same parse/contract/config pattern.
5. Contract/profile registries. If multiple modules need the same policy-object
   metadata, move the shared expansion-time facts into a tiny =begin-syntax=
   table owned by the macro module rather than duplicating runtime alist rows.

** 10.06.44.40 User Interface Rule

User-facing files should demonstrate real scenarios, not introspection toys.
Examples must show how a user or agent extends an actual profile, session,
sandbox, workflow, or loop policy.

Acceptable user-facing forms:

#+begin_src scheme
(use-module loop-engine
  :config
  (.def (current-system-build-profile @ loop-engine-profile)
    sandbox: current-system-build-sandbox
    memory-policies: (list current-system-build-memory-policy)))
#+end_src

Avoid documenting lower-level macro-generated helpers as the normal user path.
Those helpers are implementation infrastructure.

** 10.06.44.50 Projection Rule

Projection code must have an owner and a boundary:

- POO objects are the extension surface.
- Functional combinators are the flow composition surface.
- =defstruct= or fixed helper functions are acceptable for generated internal
  facts.
- Alists are final payloads for receipts, manifests, tests, snapshots, and
  Marlin ABI handoff.

If the same projection list appears in more than one owner, define a named
projection helper, a declarative macro family, or a procedural macro-generated
projection family.

** 10.06.44.60 Verification Rule

Refactors that introduce or change macros or compile-time generators must run
focused tests at the owner boundary first. For sandbox backend object family
changes, run:

#+begin_src sh
gxtest t/agent-sandbox-profile-test.ss
gxtest t/agent-sandbox-cube-interface-test.ss
gxtest t/docker-descriptor-test.ss
gxtest t/nono-sandbox-c-binding-test.ss
#+end_src

For session policy, loop runtime, use-module, and procedural macro work, add or
update tests at the public scenario boundary before broad aggregate tests.
