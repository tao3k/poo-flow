* 10.06.44 Current Project Programming Style
This document is the current programming contract for =poo-flow=.  It is not a
general Gerbil style guide.  It tells agents how to write and repair code in
this repository.

The upstream reference is:

- =/Users/guangtao/ghq/github.com/tao3k/agent-semantic-protocols/languages/gerbil-scheme-language-project-harness/docs/10-19-research/11.13-gerbil-advanced-feature-exploration.org=

The key lesson from that research is simple: advanced Gerbil features are not
for clever syntax.  They are for replacing agent failure modes with owned
language mechanisms.

** 10.06.44.10 Core Contract
=poo-flow= is a Gerbil Scheme control plane built from POO-native objects,
functional strategy combinators, contract/projection receipts, and runtime
handoff values.

New code must follow these rules:

1. Public extension surfaces are POO-native first.
   Use =object<-alist=, =.o=, =.ref=, slot helpers, prototype composition,
   named POO object constructors, and functional helpers.  Do not expose a raw
   alist DSL, record DSL, JSON row language, or provider-specific option map as
   the public model.

2. Functional composition owns behavior selection.
   Use pure helpers for selection, filtering, routing, validation summaries, and
   projection.  Side effects belong behind explicit runtime handoff boundaries.

3. Runtime values are projections, not semantic owners.
   ABI rows, JSON-like receipts, Python/Rust/Marlin payloads, and runtime
   adapter options must project already-validated POO facts.  They must carry
   schema, producer, source, object kind, and provenance when crossing a runtime
   boundary.

4. Heavy execution stays outside Scheme.
   Scheme owns policy, composition, model/catalog selection, diagnostics,
   validation, presentation, and report-only handoff data.  Runtime languages
   execute.

5. A module has a closed kernel and declared extension surfaces.
   Users should select modules and public factors.  They should not compose
   kernel hooks by hand to make a module work.

** 10.06.44.20 Upstream Gerbil Feature Baseline
The harness research identifies stable high-yield Gerbil owners:

- runtime MOP for class descriptors, slots, methods, method lookup,
  specialization, sealing, and next-method support;
- runtime and std interfaces for interface descriptors, per-class interface
  tables, prototype method dispatch, and syntax-time interface metadata;
- C3/C4 linearization and slot computation;
- SSXI optimizer declarations for type, class, method, primitive predicate,
  primitive procedure, and inline metadata;
- core contracts for checked boundaries over runtime type descriptors, class
  slots, signatures, checked methods, checked macros, and checked contracts;
- extensible match macros for recurring domain destructuring;
- macro objects, =syntax-local-value=, compile-time metadata, and generated
  identifiers;
- std iteration forms such as =for=, =for/collect=, and =for/fold=;
- explicit parser hot paths with typed constructors, front-door checks, state
  transitions, and fixnum arithmetic.

The project rule is: use the smallest Gerbil feature that owns the repeated
structure.  If the feature is business behavior, use a named function.  If the
feature is a repeated declaration frame, use a macro.  If the feature is object
behavior, use POO objects, classes, interfaces, or generic dispatch.  If the
feature is a public boundary, use contracts and receipts.

** 10.06.44.30 Repair Bad Agent Code
Use this table while reviewing generated Scheme:

| Bad shape | Required repair |
|-----------+-----------------|
| Stable hash or alist field model | POO object constructor plus slot accessors. |
| Repeated =.ref= / slot checks across a domain | Domain predicate, match macro, or accessor family. |
| Long symbolic =case= over object behavior | Interface, generic method, prototype dispatch, or generated dispatch table. |
| Comments that say "expects X" | Contract, validation receipt, or checked boundary. |
| Runtime parsing of fixed syntax | =defsyntax= parser with expansion-time errors. |
| Repeated constructor/projection/validator rows | Declaration macro backed by compile-time metadata. |
| Mutable module global used only for macro context | Syntax parameter or explicit compile-time context object. |
| Broad source collector for module facts | Module/context API or owner-indexed catalog. |
| Hot symbolic lookup in a loop | Specialized table, cache object, or indexed family. |
| Runtime adapter option owns semantics | POO object plus projection into adapter payload. |

Do not "simplify" these repairs back into alists.  Alists are acceptable only at
projection boundaries, metadata payloads, or small diagnostics where the owning
POO object is already known.

** 10.06.44.40 Macro And Metaprogramming Policy
Use Gerbil macros when all of these are true:

- the repeated code is structural, not business logic;
- the call site names domain facts clearly;
- the expansion target is ordinary POO-native or functional code;
- expansion errors point at the user declaration;
- the macro does not hide extension slots, merge policy, or runtime handoff.

Use the smallest macro tier:

1. =defrules= for direct constructor families, field-row helpers, simple binding
   forms, and syntax-to-POO lowering.
2. =defsyntax= with =syntax-case= and =with-syntax= when the form needs shape
   validation, generated identifiers, defaults, alternate expansions, or
   expansion-time error messages.
3. =begin-syntax= metadata records or classes when several declarations share
   static facts such as policy kinds, profile families, contract slots, or
   generated projection ownership.
4. Macro objects only when the macro itself has classed behavior and the owner
   module benefits from explicit expander state.

** 10.06.44.42 Phase, Hygiene, And Module Context
Macros that parse a project language must preserve Gerbil syntax context.  The
parser owns syntax objects, source locations, grammar validation, and a bounded
compile-time plan.  Runtime modules own POO construction and policy behavior.

Use these ownership rules:

- put reusable grammar parsing and immutable expansion plans in a dedicated
  syntax-support module;
- import syntax support through =for-syntax= rather than loading or evaluating
  its source;
- keep the public macro owner thin and export only the user-facing macro;
- expose selected POO/runtime extension helpers through a stable facade, not a
  wildcard re-export from the syntax owner;
- rely on Gerbil module registration for one compiled module context instead of
  mutable global macro registries or bespoke source collectors;
- keep runtime and macro artifacts in their native phases and never project a
  syntax plan into JSON, an alist DSL, or a runtime row.

Grammar identifiers must be compared with binding-aware syntax operations.
Do not identify fixed grammar with =symbol->string=, printed text, or unscoped
string tables.  Keep user expressions as syntax objects until a deliberate
quoted payload boundary.  Generated identifiers must inherit a template syntax
context, and expansion errors must retain the offending source syntax.

The compile-time hot path follows the same allocation discipline as runtime
code.  Parse each declaration once, accumulate with bounded reverse collectors
or an equivalent linear builder, and lower the completed plan once.  Nested
=map= plus =apply append=, repeated rescans of the same forms, temporary runtime
POO objects during expansion, and source loading during expansion are forbidden.
For =n= declarations and =m= clauses, target both work and generated form count
at =O(n + m)=.

The reference architecture is
[[file:88-phase-aware-composition-macro-architecture.org][Phase-Aware Composition Macro Architecture]].

** 10.06.44.45 Landed Object Family Rule
Stable large POO object families must not hand-write the same predicate,
accessor, and boundary projection pattern for every object.

Use =src/module-system/object-family-syntax.ss= when the family has:

- one stable =kind= slot;
- fixed public slot accessors;
- one or more alist projections used only at runtime, receipt, or diagnostics
  boundaries.

The reference scenario is
=t/scenarios/performance/poo-object-family-syntax/=. Its =input.ss= shows the
old repeated accessor/projection shape; its =expected.ss= shows the required
=defpoo-object-family= repair. The pattern is already exercised by
=src/modules/model-core/objects.ss= and =t/model-core-test.ss=.

Good project macro shape:

#+begin_src scheme
(defrules defpoo-domain-family ()
  ((_ binding domain-kind (slot value) ...)
   (def binding
     (domain-object 'domain-kind slot value ...))))
#+end_src

Bad project macro shape:

#+begin_src scheme
(defsyntax (magic-runtime-form stx)
  ;; Expands into hidden activation, hidden IO, and uninspectable adapter rows.
  ...)
#+end_src

Macros must not:

- run module activation;
- execute runtime work;
- allocate worktrees or call connectors;
- parse source files at runtime;
- bypass kernel validation;
- hide option conflicts;
- expose private kernel hooks as normal user knobs;
- create a second alist or record DSL.

** 10.06.44.50 POO Object Families
Object-family code should look like current =tool-core=, =memory-core=, and
=model-core= owners:

- constants name object kinds;
- constructors validate public inputs at the boundary;
- objects carry =kind=, =schema=, domain refs, runtime owner, runtime-executed
  state, and metadata;
- accessors use =.ref= behind named functions;
- catalogs summarize refs and counts once;
- find/select helpers are pure functions;
- receipts explain validation, projection, or selection decisions;
- =->alist= functions exist only as projection boundaries.

Prefer a macro such as a field-row helper when many projections repeat the same
frame.  Prefer a normal function when the logic is data-dependent.

** 10.06.44.60 Interfaces, Methods, And Dispatch
Use interfaces, generic methods, or prototype dispatch when downstream modules
need to add behavior without editing a central symbolic branch.

Good candidates:

- multiple model providers implement the same selection/projection behavior;
- multiple stores implement recall/commit/durable-job behavior;
- multiple tools implement render/validate/emit behavior;
- multiple runtime adapters implement the same ABI surface.

Do not introduce an interface for a single local helper.  The trigger is a real
extension point with repeated operation names across object families.

** 10.06.44.70 Contracts And Checked Boundaries
Public APIs, macro-generated APIs, runtime handoff boundaries, and module
extension slots need executable evidence.  Use:

- constructor validation for POO object inputs;
- contract or checked macro boundaries for public forms;
- validation receipts for module/profile/plugin legality;
- projection receipts for runtime-facing values;
- private unchecked helpers only after a public checked boundary.

Do not scatter defensive checks inside business logic when one named boundary
can own the rule.

** 10.06.44.80 Match, Iteration, And Hot Paths
Use =match= or a domain match macro when a recurring pattern is currently
expressed with nested =cond=, =and=, =pair?=, =null?=, object-kind checks, slot
checks, and list checks.

Use =for=, =for/collect=, and =for/fold= when they make a traversal clearer and
do not hide allocation behavior.  Keep explicit recursion when the code is a
hot path and the accumulator/order/performance is clearer by hand.

Use specialized tables or cache objects for repeated symbolic lookup.  Keep the
hash/equality choice visible.  Do not hide performance-sensitive registry logic
behind vague helper names.

Use parser/lexer generators or expansion-time parsing only for fixed languages:
contract syntax, profile declaration syntax, policy selector syntax,
=use-module= branches, and static runtime handoff schemas.  Do not repeatedly
parse fixed declaration syntax at runtime.

** 10.06.44.90 Module, Factor, Plugin, And Fixture Boundary
Each module is made from:

1. Kernel factors.
2. Public module factors.
3. Plugin factors.
4. Fixtures and scenarios.

Kernel factors are module-owned and closed by default.  Public factors are
configurable under module-owned contracts.  Plugins attach through declared
slots.  Fixtures prove behavior and may inspect internal receipts, but they do
not make private hooks public.

This boundary follows the Doom calibration in
[[file:86-doom-emacs-core-module-boundary-research.org][86 Doom Emacs Core Module Boundary Research]].

Agents must classify every module change as one of:

- kernel factor;
- public factor;
- plugin factor;
- fixture or scenario;
- macro authoring helper;
- diagnostic or operation lane.

If a normal user must compose internal hooks to get a working module, the module
boundary is wrong.

** 10.06.44.100 Model Module Rule
Model selection is a control-plane policy domain, not a runtime-provider option
map.

The =model-core= module demonstrates the required shape:

- =poo-flow-model-spec= is a POO object for provider, model id, capabilities,
  modalities, token limits, runtime owner, handoff operation, and backend;
- =poo-flow-model-catalog= is the owner-indexed model set;
- =poo-flow-model-selection-policy= is a pure policy object over candidates,
  fallback, required capabilities, routing strategy, and budget policy;
- =poo-flow-model-select= produces a receipt before runtime execution;
- =poo-flow-model-selection-receipt->alist= is a projection boundary only.

Do not put model semantics directly into Python, Rust, Marlin ABI rows, or raw
provider config.  Runtime languages receive the selected projection.  Scheme
owns the policy and diagnostics.

** 10.06.44.110 Runtime Projection Rule
Runtime-facing code must distinguish these layers:

- POO object: semantic owner.
- Policy/combinator: pure selection or composition behavior.
- Receipt: explainable validation/selection/projection result.
- Projection alist/JSON/ABI row: final payload for runtime handoff.
- Runtime adapter: execution owner outside Scheme.

If a runtime adapter needs to understand private Scheme internals, the
projection is wrong.  If Scheme executes provider calls directly, the runtime
boundary is wrong.

** 10.06.44.120 Performance Discipline
Performance is part of style.

- Avoid repeated list scans on hot paths when an indexed family or cache object
  is available.
- Avoid cons-heavy POO construction in loops; construct prototypes and reuse
  objects where the domain permits it.
- Keep compile-time generation bounded and measurable.
- Prefer expansion-time validation for static declarations.
- Preserve source locations in generated code so compile errors point at the
  user declaration.
- Do not lower test/build concurrency to hide performance regressions.
- Add focused performance scenarios for object-family, macro-expansion,
  projection, and selection hot paths when a change touches shared behavior.

** 10.06.44.125 Build-Stage Receipts And Warm-Cache Measurement

Build and control-plane outcomes are part of the POO boundary. A caller that
needs to make a decision about a stage must receive a stable POO-native receipt,
not parse diagnostic output. Human-readable =stdout= remains useful for
operators, but it is never the input to a runtime, policy, test, or cache
decision.

For a repeated result family, define one static prototype for its invariant
identity, then construct each result with =.mix= and dynamic slot rows. This
keeps the result shape explicit, permits prototype reuse, and avoids treating a
fresh interpreted =.o= allocation as a hot-path construction technique. The
public receipt exposes the stage label, specs, outcome, reason, cache-status,
elapsed time, and a named contract projection. Runtime adapters consume that
projection; they do not consume build logs.

The performance boundary is defined by the work that actually happened:

- A changed-cache run that compiles one or more targets is a repair or cold-path
  observation. Report it separately from the warm baseline.
- A warm run is valid only when every stage returns =skip= and the compiled
  target count is zero. Do not lower configured parallelism to make that
  observation look better.
- Measure package AOT build execution. A direct interpreted =gxi= probe includes
  module loading and is not a substitute for the package warm-build benchmark.
- Keep the measurement in a project-owned Harness scenario with explicit input
  and expected values. The live stage receipt supplies observed elapsed
  microseconds; its bound is derived from the scenario target in milliseconds.
  This joins type, cache state, outcome, and timing in one check rather than
  separate synthetic benchmarks.

When a performance gate fails, first identify whether it is a changed-cache
compile, a cache-validity miss, or an actual warm-path regression. Repair the
owner of that boundary and retain the receipt evidence; do not disable the gate,
parse stdout, or reduce concurrency as a workaround.

** 10.06.44.126 Native Lazy Evaluation And Lazy Load Boundary

Gerbil POO is the normal lazy substrate for large semantic objects. Object
construction may compile precedence and slot functions, but ordinary values are
materialized only when a consumer reads the corresponding slot. Write large
policy, profile, graph, and presentation objects as demand graphs.

The evidence and semantic rationale live in
[[file:89-poo-native-lazy-evaluation-research.org][POO Native Lazy Evaluation Research]].

The optimization unit is a large POO composition. A composition is both the
namespace that names profile, agent, model, sandbox, strategy, task, graph, and
handoff objects, and the object/prototype graph that connects them. A runtime
demand starts from one composition slot and must force only its reachable object
subgraph. Do not replace this graph with a policy descriptor table, raw registry
rows, string lookup, or a second data-model language.

Compose the stable namespace and prototype topology once. Reuse the resulting
composition object for repeated demands. Share high-fanout semantic facts by
direct POO object reference. An extension overlay changes only its object slots
and retains unchanged fact-object references. The normal first-demand cost is
proportional to the reachable object subgraph; repeated demand reuses the
object-local slot values already materialized by =.ref=.

1. Use constant, thunk, self, and computed POO slots for semantic values. Do
   not add a parallel vector, alist, or global memo cache around normal POO
   computed slots.
2. Treat =.ref= as demand, and reserve =force-object=, =.alist=,
   =object-values=, complete =MonomorphicObject= validation, and complete
   serialization for named materialization or ABI boundaries.
3. Compose a stable prototype family once at its composition root and reuse the
   composed object. Do not recreate =.mix= or a prototype chain in a request or
   per-slot loop.
4. =.extend= reuses slot definitions but creates a new object demand graph.
   When derived objects need the same expensive fact, reference one shared POO
   fact object; do not assume that a forced parent instance cache is inherited.
5. Evolve semantic policy objects through prototype extension or composition.
   Do not use instance mutation as a policy-evolution mechanism.
6. Keep slot names distinct from lexical value bindings in dynamic POO forms.
   Public macros must preserve hygiene and generate distinct value bindings so
   a slot cannot recursively force itself through a same-name capture.
7. Use =std/lazy= only for a pure deferred calculation that is not naturally a
   POO slot. It is a Scheme control-plane mechanism, not a runtime concurrency
   primitive.

=lazy-load= is a separate project macro. It lowers a runtime source-ref or
resolver expression to =std/lazy= delay semantics. It must not wrap =load!=:
=load!= is compile-time inclusion, while =lazy-load= is runtime deferred module
resolution. The public module surface has no compatibility lazy-load-plan
record. Runtime singleflight, cancellation, retries, and scheduling remain
outside Scheme.

Every lazy change needs a real Scenario input and expected result. The Scenario
must distinguish partial slot demand, repeated demand of the same slot, derived
object reuse through shared fact objects, full materialization at a named
boundary, and hygienic rejection of same-name dynamic slot/value forms.

** 10.06.44.130 Agent Checklist
Before committing Scheme changes, check:

- Is the semantic owner a POO object, contract, validator, graph fact, or
  functional combinator rather than a final payload row?
- Did any new macro expand to ordinary POO-native or functional code?
- Are grammar literals binding-aware and are user expressions retained as
  syntax objects until an intentional payload boundary?
- Are compile-time parsing, public syntax, and runtime construction owned by
  separate modules with explicit phase imports?
- Does a large declaration path parse and lower once with linear allocation?
- Are public inputs validated once at the boundary?
- Are runtime handoff values projections with receipts?
- Are kernel factors hidden from ordinary users?
- Are public factors and plugin slots declared and validated?
- Does the code avoid broad alist/hash object models?
- Is repeated dispatch represented through interfaces/generics/tables when the
  extension point is real?
- Did focused tests or scenarios prove the behavior?

If the answer is unclear, stop and write the owner boundary first.
