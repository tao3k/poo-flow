#+title: AC-09 Lean / Evidence Proof Bridge Implementation Plan
#+date: 2026-07-12
#+status: approved-for-implementation

* Delivery

Implement the approved design in
=127-agentic-control-plane-ac09-lean-evidence-proof-bridge-design.org= as seven
bounded features. Each step must leave its focused gates green and is committed
only when the feature is complete.

The plan does not freeze ABI v1 and does not introduce a graph ABI compatibility
surface.

* Step 1 — Canonical Proof Manifest

Create one machine-readable ProofCaseVector v1 manifest under =packages/proof=
that owns field order, widths, tags, obligation bits, endianness, alignment,
reserved space, and schema fingerprint inputs.

Extend the existing Python proof generator rather than creating a second
generator. Generate and freshness-check:

- a public C layout/tag projection;
- Lean layout/tag declarations;
- Python layout/tag constants;
- deterministic positive and malformed test vectors.

Focused gates:

- generator unit tests;
- generated artifact byte-for-byte freshness;
- schema fingerprint stability;
- duplicate tag/bit and layout overlap rejection;
- Python package suite;
- Lean build and current FFI smoke.

Commit boundary: canonical manifest, generators, generated projections, and
freshness tests form one feature.

* Step 2 — Scheme POO Proof Projection

Add POO-native proof-case, obligation, evidence-root, and AuthorizedEffectToken
projection objects. Scheme normalizes proof facts but does not assert Lean proof
success.

Reject raw alist DSLs, record DSLs, ad hoc patch languages, and raw compute hooks
as the ordinary public API.

Focused gates:

- object construction and prototype-extension tests;
- construction-order invariance;
- missing/duplicate obligation rejection;
- AC-08 token/evidence regression suite.

* Step 3 — Kernel And Bulk C ABI

Implement =poo_flow_proof_case_measure= and =poo_flow_proof_case_write= over a
validated handle and caller-owned buffer. Add stable transport errors, exact
sizing, no partial publication, and bounded scalar diagnostic accessors.

Focused gates:

- generated header freshness;
- pure-C layout and vector conformance;
- buffer-too-small, invalid-handle, stale-handle, schema-mismatch, and malformed
  evidence faults;
- installed consumer;
- sanitizer/leak gates on supported hosts;
- no per-field FFI reconstruction on the production path.

* Step 4 — Lean Typed AuthorizedEffectToken Proof
The typed vector covers every field in the 424-byte manifest. Receipt digests
are derived internally with manifest-owned SHA-256 domains and are not
constructor arguments. Lean treats SHA-256 as a trusted primitive; native
differential tests verify its executable realization.

Decode ProofCaseVector v1 into typed Lean values. Implement the first theorem
set for policy/effect binding, semantic/execution roots, obligation completeness,
replay protection, Diagnostic non-execution, and L3 completeness.

Emit a ProofReceipt that binds vector digest, theorem-set digest, semantic root,
execution root, and any claimed committed batch root.

Focused gates:

- Lake build and native FFI smoke;
- positive AuthorizedEffectToken theorem fixtures;
- refuted, malformed, and unsupported-obligation fixtures;
- no constructor that treats Scheme runtime execution as already proved.

* Step 5 — Python CFFI Differential Runtime
The installed-consumer gate builds a non-editable wheel, installs it in an
isolated environment without checkout =PYTHONPATH=, and consumes the public C
shared library with the canonical vector and digest.
The differential harness executes Scheme rather than substituting a Python
fixture, compares against the generator-owned Lean bytes, and round-trips the
same buffer through CFFI. Failures identify the first divergent owner, field,
and offset.
Scheme first projects the POO proof case into the exact caller-owned 424-byte
native vector and computes the manifest-owned domain-separated SHA-256. The
encoder rejects malformed digest widths, invalid hex, unsupported tags, and
incorrect target sizes before the C/Python differential comparison.

Expose the bulk proof-case ABI through the retained production Python runtime.
Compare Scheme/kernel vectors at byte, field, and semantic levels. Python remains
a generator and differential oracle, never a proof owner.

Focused gates:

- CFFI layout and lifetime tests;
- exact buffer sizing and zero-copy view lifetime rules;
- byte/field/semantic differential vectors;
- production Python runtime suite;
- installed wheel consumer.

* Step 6 — Security, Property, And Performance Qualification
Run table-driven structural mutations through both Python and C, including
both reserved boundaries and mutable borrowed-buffer TOCTOU. Run every
token/root/binding substitution through digest identity checks and require the
first divergent owner, field, and offset.

Add replay, fork, nonce reuse, token/effect/policy/root substitution,
cross-runtime/session/lease/batch substitution, mutable-buffer TOCTOU, omitted
receipt, missing obligation, and malformed-vector gates.

Prove with real-path fixtures:

- O(1) per-item vector construction;
- O(k) batch construction;
- O(log n) root-history lookup;
- one bulk FFI projection rather than one crossing per field;
- no per-item heap allocation in caller-owned writes.

Freeze concrete time budgets only after reproducible baselines exist.

* Step 7 — AC-09 Qualification Receipt

Aggregate Scheme, kernel, C, Python, Lean, installed-consumer, security,
differential, determinism, and performance receipts. Update owner/test/status
documents from measured evidence.

Completion must state:

- AuthorizedEffectToken proof chain verified;
- AC-01 through AC-08 regressions green;
- no retired graph ABI wrapper;
- =abi-v1-frozen? #f=;
- AC-10 integrated qualification is next.

* Stop Conditions

- Stop if the canonical manifest would require JSON in the native ABI.
- Stop if a proposed field exposes Gerbil object or GC layout.
- Stop if generated language projections can drift independently.
- Stop if a transport success is treated as proof verification.
- Stop if an unsupported obligation can execute.
- Stop if a performance gate measures a synthetic path instead of the bulk ABI.
- Stop if unrelated build/CLI/downstream-runtime work enters a feature commit.

* Validation Order

Within each feature, run the narrowest owner tests first, then generated-artifact
freshness, cross-language consumers, package suites, and finally AC regression
gates. Expensive benchmark and sanitizer gates run only after functional gates
are green.
