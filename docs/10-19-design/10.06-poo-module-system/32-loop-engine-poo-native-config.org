#+TITLE: 10.06.32 Loop Engine POO-Native Config

* Decision

=loop-engine= accepts only native Gerbil POO configuration at the public
=use-module= surface:

#+begin_src scheme
(use-module loop-engine
  :config
  (.def (repo-loop @ loop-engine-use-case name level mode goal)
    name: 'repo-doctor
    level: 'l1
    mode: 'report-only
    goal: 'inspect-profile-and-policy-warnings)

  (.def (repo-profile @ loop-engine-profile use-cases runtime)
    use-cases: (list repo-loop)
    runtime: repo-runtime))
#+end_src

The =:config= token is a grouping marker only. The body is a set of POO
objects. No public alist DSL, record DSL, =+use-case= row DSL, or compatibility
reader is retained for loop-engine configuration.

POO-native does not mean untyped. Each loop-engine POO object slot is validated before object-to-intent lowering. Structural slot errors, such as a non-list =capabilities:= value or a non-symbol =backend:= value, fail before presentation or runtime manifest projection. Domain contract errors that are structurally valid, such as an empty result =required-fields:= list, remain reportable receipt diagnostics.

* Boundary

Scheme owns loop composition, profile selection, policy facts, result-contract
expectations, and Marlin handoff receipts. It does not execute loops, schedule
agents, mutate state stores, or run sandbox backends. Runtime execution remains
owned by =marlin-agent-core=.

The public loop-engine object graph is:

- =loop-engine-use-case= for named loop work.
- =loop-engine-governor= for strategy and policy capability flags.
- =loop-engine-agent-judges= for auditor, verifier, reviewer, and governor
  refs.
- =loop-engine-human-audit= for human gate actions.
- =loop-engine-schedule=, =loop-engine-state=, =loop-engine-sandbox=,
  =loop-engine-budget=, =loop-engine-result=, =loop-engine-observability=, and
  =loop-engine-runtime= for focused policy fragments.
- =loop-engine-profile= as the single root object accepted by one
  =loop-engine= config block.

OpenRath's async session loop adds one important extension point: tool dispatch
is grouped by a =resource_key(args)= concept so same-resource calls serialize
while independent resources may proceed in parallel. POO Flow should model this
as a future POO-native =loop-engine-resource-policy= object rather than a
numeric scheduling flag or row DSL clause.

* Materialization

The macro builds real POO objects with =.o= and passes those objects to the
loop-engine materializer. The materializer emits an internal
=:loop-engine-intent= flag for the existing report-only runtime projection
pipeline. This is not a user DSL; it is a private bridge between the POO object
surface and the existing loop intent projection.

Old rows such as =(+use-case ...)= and =(+result ...)= are intentionally not
accepted by =use-module loop-engine=. Root =poo-flow!= init rows may select
=loop-engine= as a module, but detailed loop configuration belongs in a POO
config object.

* User-Interface Scenarios

The user-interface tree must show real scenarios, not introspection toys:

- a single guarded handoff loop in
  =user-interface/custom/my-module/cases/loop-engine.ss=;
- a multi-use-case loop profile in
  =user-interface/custom/my-module/profiles/loops.ss=;
- invalid result and sandbox cases in tests, expressed as POO objects so the
  validation path remains covered after the old DSL is removed.

The next scenario to add after this migration is a resource/collision case that
shows two tool refs sharing one resource key and projecting a serial dispatch
receipt without Scheme executing the tool.

OpenRath's session model sharpens that next scenario into three POO-native object extensions:

- =loop-engine-lineage-policy=: projects parent session refs, lineage kind, lineage operator, and journal/export readiness as report-only receipt facts.
- =loop-engine-selector-policy=: declares candidate use cases, judge inputs, fallback behavior, and selected branch reference without running the scoring loop in Scheme.
- =loop-engine-resource-policy=: declares symbolic resource keys, collision classes, and intended serial/parallel dispatch groups on tool references.
- =loop-engine-capability-policy=: declares sandbox backend capability expectations with validated backend names.
- =loop-engine-memory-policy=: declares store/scope, recall keys, commit keys, ranking, and retention expectations without reading or writing memory in Scheme.

These policy objects stay inside =(use-module loop-engine :config (.def ...))=. They are not alist/record DSL clauses, not scheduler commands, and not compatibility branches for the removed row config.

Implementation update: the OpenRath-derived policy objects are now implemented as POO-native profile slots and report-only receipts. =loop-engine-lineage-policy= projects =lineage-receipt=, =loop-engine-selector-policy= projects =selector-receipt=, =loop-engine-resource-policy= projects =resource-dispatch-receipt=, =loop-engine-capability-policy= projects =capability-receipt=, and =loop-engine-memory-policy= projects =memory-receipt=.

* Marlin ABI Surface

The loop-engine runtime manifest exposes two different tables for Marlin:

- =object-families=: the report-only object families carried by the handoff request, including =lineage-receipt=, =selector-receipt=, =resource-dispatch-receipt=, =capability-receipt=, and =memory-receipt=.
- =receipt-contracts=: the stable schema ids for report-only receipt rows. This includes lineage, selector, resource-dispatch, capability, memory, and sandbox handoff agreement contracts.

This split is intentional. =runtime-handoff-contracts= names operations that Marlin may implement, such as =start-workflow-run= and =read-runtime-snapshot=. =receipt-contracts= names inert data schemas that Scheme projects for audit and handoff. Scheme must not use receipt contracts as executable commands.

The public presentation exposes the same =receipt-contracts= table so downstream agents can validate the manifest shape without walking every runtime request payload.

OpenRath's backend, memory, and compression layers suggest manifest extensions, but not Scheme runtime implementations. =loop-engine-capability-policy= validates backend values as sandbox backend names: =nono-sandbox= and =cube-sandbox= are accepted; =marlin-agent-core= remains the runtime owner, not a sandbox backend. =loop-engine-memory-policy= is report-only: Marlin owns durable memory stores, recall execution, commit execution, ranking, retention, and mutation. =loop-engine-compression-policy= remains the next policy object to add.
