#+TITLE: 10.06.75 POO Performance Scenario Replication

* Purpose

The current project needs to treat POO as the high-performance object system for the control plane. Large and complex objects must not push the implementation back toward raw alists, string assembly, or temporary lambda patching.
For that reason, performance benchmarks cannot cover only tiny happy paths. They must exercise real POO object construction, repeated access, projection, receipt boundaries, tool-calling control, and composition.

The upstream =gerbil-scheme-language-project-harness= scenarios are reference facts, not final POO Flow facts. We replicate them to bring already validated high-performance POO idioms into this project, then strengthen those idioms under POO Flow's policy, strategy, composition, proof, and runtime handoff semantics.

* Replicated Scenario Families

- Large object construction: native =.o= is constructed once at the boundary, and loops consume only cached scalar values or stable slots.
- Object iteration: an =.alist= or object-list snapshot is created once before the tight loop, avoiding repeated slot resolution inside the loop.
- Large profile projection: the profile remains a native POO object, projection specs are static, and descriptors are generated only at the boundary.
- Generated receipt boundary: receipts use generated struct/accessor surfaces inside the control plane, and alists are projected only at the runtime handoff boundary.
- Tool calling object-list control: plans, receipts, and facts remain stable POO object lists, and a combined validation/proof projection reuses one traversal.
- Composition: profile, policy, and strategy combinations go through POO owner families and stable contribution boundaries instead of scattered raw patches.

* Gate Contract

Each benchmark fixture must satisfy three contracts at the same time:

- benchmark contract: includes =sourcePath=, timing budgets, observed timings, input/expected comparison, and standard measurement phases.
- memory contract: declares =maxRssMb=, =memoryMetric=, and =memoryUnit=, and includes =assert-memory-gate= in =measurementPhases=.
- POO evidence contract: declares =pooFormEvidence= and =pooUsageCallEvidence= to prove that the scenario remains centered on the POO API.

These three contracts serve different purposes:

- The benchmark contract prevents a scenario directory from landing without an executable benchmark surface.
- The memory contract prevents large-object scenarios from passing with unbounded memory behavior.
- The POO evidence contract prevents optimizations from bypassing POO just to run faster, which would miss the project goal.

* Implementation Notes

The new =poo-large-profile-projection= scenario keeps the profile as a native POO object and treats projection specs as stable data:

#+begin_src scheme
;; profile remains POO-native; projection descriptors are boundary values.
#+end_src

The new =poo-generated-receipt-boundary= scenario uses typed/generated accessors for receipts inside the Scheme control plane. Alists exist only at the runtime handoff boundary:

#+begin_src scheme
;; receipt construction is typed/generated; alist exists only at the boundary.
#+end_src

Neither scenario is intended to make Scheme own runtime execution. Scheme owns the rigorous policy, control, proof, and composition structure; Python, Marlin, or another runtime language remains responsible for actual execution.

* Scenario-Level Tests

The fixture contract alone is not enough. A fixture can satisfy the benchmark metadata shape while still failing to expose a reusable POO implementation idiom. For that reason, the replicated scenarios also have support helpers and test cases in the package-local POO performance suite.

=poo-large-profile-projection= now exposes a reusable helper surface:

- =poo-performance-large-profile-projection-descriptors=
- =poo-performance-large-profile-projection-valid-count=
- =poo-performance-large-profile-projection-gate-receipt=

The test verifies that projection produces a stable descriptor vector, that the first and last projected values match the native POO profile, and that the benchmark gate receipt passes.

=poo-generated-receipt-boundary= now exposes a reusable helper surface:

- =poo-performance-generated-receipt-boundary-alist=
- =poo-performance-generated-receipt-boundary->alist=
- =poo-performance-generated-receipt-boundary-valid-count=
- =poo-performance-generated-receipt-boundary-gate-receipt=

The internal receipt is a generated struct/accessor surface. The exported test helper returns the boundary alist without exporting the internal constructor, so downstream tests can validate the runtime handoff shape without making raw alists the control-plane representation.

* Design Boundary

The long-term value of these scenarios is not to prove that a single micro benchmark is fast enough. The value is to establish a reusable set of POO object implementation idioms:

- Large objects are allowed, but they must be constructed, indexed, and projected at explicit boundaries.
- Hot paths can be fast, but they must not bypass the POO public surface.
- Receipts can target runtime handoff, but the control plane should retain typed/generated shape.
- Composition can combine profiles, policies, and strategies freely, but it must go through stable owner and contribution boundaries.

If a policy, strategy, loop, graph, or tool-calling module later shows a 10-second-level regression, the first response should be to reduce it into these scenario families instead of adding a one-off script.
