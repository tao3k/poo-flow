* 10.06.47 Current Code Structure Audit And Complete Improvement Plan
This document is the complete project advancement plan for the current =poo-flow= codebase. It is broader than a next-step list: it defines the target structure, workstreams, dependency order, application scenarios, deliverables, validation gates, and risk controls.

The plan assumes the project invariants already established in =10.06.44= and =10.06.46=:
- public extension APIs stay 100% POO-native;
- flow composition prefers functional programming;
- macros are internal generators for mechanical POO/functional/projection structure, not a new user DSL;
- Scheme owns strategy, composition, validation, presentation, and report-only handoff data;
- Rust and =marlin-agent-core= own heavy execution and scheduling.
** 10.06.47.10 Current Structure Baseline
The current codebase already has a useful layered shape:

1. Build and testing boundary.
   =build.ss= owns package build options. =src/cli-support/testing-project.ss= binds the harness project root, roots =src=, =user-interface=, and =t=, and keeps policy checks enabled.
   Single-file test selection belongs to the new testing framework and =gxtest='s native single-file selection support.

2. Core functional and POO foundations.
   =src/core/flow-declarations.ss= defines flow declaration descriptors and registries as POO-native values. =src/core/flow.ss= contains flow category access. =src/module-system/object-core-support/object.ss= owns module object construction, field resolution, inheritance, contribution indexing, and merge behavior.

3. Module system facade and syntax.
   =src/module-system/init-syntax.ss= owns public =use-module= syntax, profile construction helpers, and module bundle collection. It is the current public syntax choke point.

4. Sandbox backend modules.
   =src/module-system/sandbox-backend-object-syntax.ss= owns the internal =defpoo-sandbox-backend-object-family= macro. Concrete backends such as =src/modules/docker-sandbox/objects.ss= already call that macro instead of hand-writing every object/profile/capability family.

5. Session policy module.
   =src/modules/session/policy.ss= owns policy predicates, constructors, accessors, durable policy attachment, tool grants, policy checks, and final alist projections.

6. Loop engine runtime module.
   =src/module-system/loop-engine-runtime.ss= owns runtime envelopes, manifests, receipts, capability lookup, memory/compression policy projection, context enrichment, and loop-engine intent collection.

7. User-interface scenarios and tests.
   =user-interface/custom/my-module/cases/loop-engine.ss= shows a real POO-native loop-engine profile with sandbox, runtime, memory, compression, lineage, selector, resource, and capability policies. =user-interface/custom/my-module/cases/session-policy.ss= shows real session policy use across tool grants, model, prompt, resource, and agent execution. Tests such as =t/user-interface-custom-loop-engine-runtime-manifest-test.ss=, =t/user-interface-custom-loop-engine-memory-policy-test.ss=, and =t/session-policy-contract-test.ss= are the scenario-level acceptance surfaces.
** 10.06.47.20 Structural Diagnosis
The project direction is coherent, but the structure has predictable pressure points:

1. Large owners mix several change axes.
   =loop-engine-runtime.ss= currently mixes runtime ABI projection, receipt construction, policy enrichment, capability diagnostics, sandbox agreement, and config collection.

2. Session policy code repeats a policy family pattern.
   =session/policy.ss= repeats validation, object construction, accessor reads, policy checks, durable attachment, and projection fields.

3. Runtime receipts repeat projection family structure.
   Receipt and manifest alists are valid final ABI/report payloads, but the field lists should not be maintained by hand across source and tests.

4. =use-module= repeats branch mechanics.
   Public syntax belongs in =init-syntax.ss=, but repeated branches should be generated from private procedural helpers once the lower-level macro pattern is proven.

5. Scenarios exist but are not yet an acceptance matrix.
   We have real loop, sandbox, memory, runtime, and session policy cases. They need to be treated as product-facing acceptance gates, not incidental tests.
** 10.06.47.30 Target Architecture
The target architecture is a small set of stable owners with clear responsibility:

| Owner | Responsibility | Public surface |
|-------+----------------+----------------|
| =src/core/*= | Functional flow descriptors, flow registry, DAG/arrow-style composition kernel | Functional constructors and POO values |
| =src/module-system/object-core-support/*= | Module object contract, inheritance, field merge, contribution indexing | Internal object engine |
| =src/module-system/sandbox-backend-object-syntax.ss= | Declarative backend object/profile/capability generation | Internal macro only |
| =src/modules/session/policy-syntax.ss= | Session policy family generation | Internal macro only |
| =src/modules/session/policy.ss= | Public policy constructors/checks/projections | Stable POO-native policy functions |
| =src/module-system/runtime-projection-syntax.ss= | Runtime receipt/manifest projection generation | Internal macro only |
| =src/module-system/loop-engine-runtime-receipts.ss= | Generated and hand-owned receipt functions | Internal ABI/report helpers |
| =src/module-system/loop-engine-runtime-intent.ss= | Functional loop intent enrichment pipeline | Internal pure pipeline helpers |
| =src/module-system/loop-engine-runtime.ss= | Orchestrates runtime intent, receipts, and projections | Existing compatibility owner during migration |
| =src/module-system/init-syntax.ss= | Public =use-module= syntax | User-facing module entry |
| =user-interface/*= | Real scenario examples for users and agents | Public examples |
| =t/*= | Owner and scenario gates | Verification contract |

The migration should keep compatibility at public names while shrinking mixed owners behind generated and functional sub-owners.
** 10.06.47.40 Planning Options
Recommended: program-level staged architecture plan.

This plan coordinates macro adoption, runtime owner splitting, user scenarios, Marlin handoff, and harness verification. It is slower than a single refactor, but it keeps the system coherent and prevents each slice from inventing its own local pattern.

Option A: local cleanup only.

This would refactor whichever file looks worst first. It is fast initially, but it does not create a reusable policy/projection pattern and would likely leave =use-module= and runtime receipts drifting.

Option B: macro-first rewrite.

This would aggressively replace repetition with macros before the acceptance matrix is complete. It uses Gerbil power, but it risks hiding behavior changes and creating compile-time complexity before the user scenarios are pinned.

The recommended path is staged architecture: scenario matrix first, generated families second, owner splits third, broad =use-module= cleanup last.

*** 10.06.47.50.60 Workstream F: Funflow Functional Kernel

*** 10.06.47.50.70 Workstream G: =use-module= Procedural Branch Generation
Purpose: reduce public syntax branch repetition only after lower-level generated families are stable.

Deliverables:

    :config

Acceptance:
- Existing user-interface scenarios keep working.
- Error messages still direct users to POO-native =.def= forms.
- No public raw alist/record DSL is introduced.
*** 10.06.47.50.80 Workstream H: Marlin ABI And Downstream Boundary
Purpose: make Scheme-side discovery predictable for =marlin-agent-core= while keeping execution in Rust.

Deliverables:
- Runtime manifest/bridge discovery surface that exposes loop governor request, runtime command contract, receipt contracts, and report-only handoff facts.
- Documentation for downstream Gerbil package consumer shape.
- ABI consumer audit in =marlin-agent-core= before any breaking contract changes.

Acceptance:
- Marlin does not guess entrypoints.
- Scheme exports stable contract-bearing requests and receipts.
- No scheduler, durable state writer, or heavy runtime is added to this repository.
*** 10.06.47.50.90 Workstream I: Build, Harness, And Performance
Purpose: keep iteration fast and verification canonical.

Deliverables:
- Focused single-file gates for each owner through the new testing framework and =gxtest= native selection support.
- Documented understanding that native =gxtest= single-file selection is the canonical single-file testing model for this plan.
- Full =gxpkg build -R -O= gate after owner-level tests.
- Build-performance watch for macro expansion cost.
- Harness policy warnings fixed through the package CLI path, not separate shadow checks.

Acceptance:
- Refactors never require direct generated artifact cleanup.
- Tests run through the native CLI/harness path.
- Compile-time metaprogramming does not make normal agent iteration materially slower.

** 10.06.47.90 Definition Of Done
A phase is done only when all of these are true:

- The public surface is still POO-native or functional.
- Generated code expands to ordinary POO-native/functional/projection code.
- Final alists exist only at ABI/report/test boundaries.
- The relevant user-interface scenario exists and is realistic.
- Focused native =gxtest <file>= gates pass.
- Full package build is run when a phase changes shared owners.
- Documentation and =AGENTS.md= remain aligned with the new owner boundary.
** 10.06.47.100 Risk Register
| Risk | Impact | Mitigation |
|------+--------+------------|
| Macro overreach becomes a hidden DSL | Users and agents lose the POO model | Keep macros internal unless explicitly public and POO-lowering |
| Projection generation changes ABI shape | Marlin and tests break | Add compatibility tests before migration |
| =use-module= refactor changes public syntax | User examples break | Delay until lower-level patterns are stable |
| Large owner split creates circular imports | Build complexity increases | Split by dependency direction: syntax -> generated helpers -> runtime orchestrator |
| Macro expansion slows builds | Agent iteration suffers | Track build performance after each macro phase |
| Scenario docs drift from tests | Documentation becomes ornamental | Bind each scenario to a named test file |
** 10.06.47.110 Operating Model For Each Slice
Every implementation slice should follow this sequence:

1. Use ASP owner/selector evidence to identify the real source owner.
2. Update or confirm the relevant scenario and acceptance test.
3. Make the smallest owner-boundary code change.
4. Run the focused single-file gate through the project testing framework and native =gxtest= selection support.
5. Run broader =gxpkg build -R -O= only after focused gates pass or when shared build/package owners changed.
6. Update docs and user-interface examples in the same slice when public behavior changes.
7. Commit only the bounded slice.
