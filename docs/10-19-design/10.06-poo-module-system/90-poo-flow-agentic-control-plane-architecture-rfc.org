#+TITLE: RFC 90 — POO Flow Agentic Control Plane Architecture
#+SUBTITLE: Agent Society Compiler, Provable Control Plane, And Runtime Boundary
#+OPTIONS: toc:3 num:t
#+LANGUAGE: en

* Status
Status: *Accepted; implementation planning*

This RFC records the approved architectural boundary. It does not claim that the described compiler, Bundle, C ABI, Python runtime, Cedar projection, Lean linkage, or evidence chain is already implemented.

Implementation planning remains closed until the written RFC suite has passed repository review.

* Purpose
POO Flow needs a durable long-term identity that is broader than a policy DSL or workflow library but narrower and more testable than a claim that POO Flow alone is an operating system.

The accepted positioning is:

#+begin_src text
POO Flow = Agent Society Compiler
         + Provable Control Plane
         + C Runtime Interface

Agentic OS instance = POO Flow
                    + one conforming runtime
                    + authorization and proof artifacts
                    + effect adapters and external systems
#+end_src

POO Flow owns organization semantics, compilation, runtime contracts, authorization and proof projections, evidence requirements, and conformance rules. It does not own every runtime implementation or every external effect executor.

* Decision
The project SHALL evolve around these decisions:

1. Gerbil/POO and functional combinators remain the only ordinary public authoring route.
2. POO Flow compiles authoring values into one Canonical Organization Bundle.
3. The Bundle is the cross-language semantic center.
4. The Bundle contains typed Organization, Authority, Context, Protocol, and Evidence facets that share one identity universe and epoch.
5. POO Flow exposes a stable, high-performance C Runtime ABI rather than making a language-specific runtime object model canonical.
6. A first-party Python runtime SHALL be production-capable and SHALL use only the public C ABI. Its purpose is to prove that the interface is sufficient for a complete runtime.
7. Rust, TypeScript, and other runtime adapters are downstream implementations. They may be production runtimes but do not own POO Flow semantics.
8. Cedar is the local effect-authorization decision component. It is not the organization language or global semantic center.
9. Lean checks explicitly declared Bundle, composition, projection, and transition claims. It does not prove arbitrary runtime implementation behavior or external-world truth.
10. Typed evidence connects compilation, proof, authorization, runtime execution, and accepted state transitions.
11. Assurance claims are cumulative and explicit. Production readiness and formal assurance are separate dimensions.
12. Canonical replacements retire obsolete public routes instead of preserving permanent parallel semantics.

** Ownership Role Vocabulary
The RFC suite uses =owner= only with an explicit role:

| Role | Meaning |
|------+---------|
| Semantic owner | The artifact whose typed meaning is authoritative. The Canonical Organization Bundle is the cross-language semantic owner. |
| Contract owner | The RFC, schema, or ABI specification that defines a normative boundary. |
| Artifact owner | The module or package that produces, stores, checks, or publishes an artifact. |
| Implementation owner | The codebase responsible for one implementation of a contract. |
| Validation owner | The test, proof, conformance, or benchmark lane that decides a named gate. |
| Downstream owner | An external project responsible for its runtime or adapter implementation. |

These roles MUST NOT be collapsed. A Python implementation may be an implementation owner without becoming the semantic owner. A Lean package may be a proof-artifact owner without owning runtime truth. A test suite may be the validation owner for ABI conformance without defining ABI semantics.

** POO Flow Normative Roles
Within the vocabulary above, the POO Flow project holds these contract, artifact, implementation, and validation roles:

- public Gerbil/POO organization authoring;
- typed elaboration and facet-specific composition;
- canonical identity, schema, epochs, provenance, and Bundle digests;
- cross-facet invariants;
- C Runtime ABI semantics and conformance;
- Cedar, Lean, and evidence projections;
- assurance-profile vocabulary and acceptance gates;
- the first-party Python runtime used to validate interface completeness.

** First-Party Validation Ownership
The Python runtime is a first-party, production-capable implementation, but it is not a second semantic owner.

If Python requires a private Gerbil path or private representation to implement required runtime behavior, the C ABI is incomplete. The remedy is to repair the ABI, not to bless a Python-only escape hatch.

** Downstream Ownership
Downstream projects may implement Rust, TypeScript, or other runtime adapters. They own their implementation, packaging, scheduling policy, sandbox integration, and deployment-specific effect adapters.

They MUST consume the public C ABI and MUST declare a conformance and assurance profile before making POO Flow compatibility claims.

POO Flow does not implement these downstream runtimes in the migration defined by this RFC suite.

** External Components
Cedar and Lean are integrated components with bounded responsibilities:

- Cedar evaluates local authorization requests derived from Bundle semantics.
- Lean checks explicit formal statements over typed projections and declared assumptions.

MCP, A2A, tool APIs, model providers, storage systems, and external services are adapters or external systems. Their data and effects enter POO Flow through typed contracts; they do not own organization semantics.

* Inputs And Outputs
The organization compiler accepts typed values representing:

- a task or objective;
- organization constitution and governance constraints;
- POO profiles and composed prototypes;
- agent and model capabilities;
- tools and resources;
- environment facts;
- context and memory constraints;
- security, proof, evidence, and runtime requirements.

It produces a validated Bundle from which deterministic projections can be generated:

- organization and responsibility topology;
- identity, capability, delegation, revocation, and lease relations;
- context visibility, provenance, redaction, and declassification constraints;
- lifecycle, message, transition, effect, cancellation, checkpoint, and recovery contracts;
- evidence obligations and verifier relationships;
- C Runtime ABI views;
- Cedar policy, entity, and request material;
- Lean statements and proof obligations;
- deployment and runtime capability requirements;
- conformance and assurance manifests.

* Semantic Center
The Canonical Organization Bundle is the only cross-language semantic center.

The following are projections and MUST NOT become competing sources of truth:

- Python classes or dictionaries;
- C structs and process-local handles;
- Cedar policies and entities;
- Lean values and proof artifacts;
- runtime scheduler state;
- JSON, alist, database, or graph-store representations;
- adapter-specific configuration.

A concept required by a projection but absent from the Bundle is a schema or compiler deficiency. A projector MUST fail rather than invent private semantics.

* Public Programming Model
The project invariants remain binding:

- ordinary extension APIs are POO-native;
- composition is expressed as POO objects and functional values;
- Scheme remains the high-level control plane;
- effects remain behind explicit runtime handoff boundaries;
- raw alist DSLs, record DSLs, patch languages, and raw =(lambda (self super) ...)= hooks are not ordinary public APIs;
- Gerbil declarative macros, procedural macros, and bounded compile-time metaprogramming may generate repeated internal object families, manifests, and projections when the expansion remains ordinary POO-native or functional code.

C3 linearization resolves definition lookup. It is not an authorization, information-flow, protocol, or evidence policy.

* Top-Level Data Flow
The normative data flow is:

1. Collect public POO objects and functional values.
2. Resolve inherited definitions with C3 where applicable.
3. Elaborate typed entities and facet contributions.
4. Normalize each facet under its own composition algebra.
5. validate local and cross-facet invariants.
6. Canonicalize and digest one immutable Bundle.
7. Derive versioned C ABI, Cedar, Lean, and evidence projections.
8. Activate the Bundle only when its required proofs, policies, runtime capabilities, and evidence contracts are available.
9. Admit effects through the C ABI state machine and Cedar decision boundary.
10. Accept state transitions only after the required typed evidence closes.

The architecture is compiler-first. Runtime-specific convenience objects are downstream of compilation.

* Claim Discipline
POO Flow SHALL use narrow claims:

- a valid Bundle proves only that declared compilation and validation rules passed;
- a Cedar decision proves only the result for a bound request, policy, entities, and context;
- a Lean proof proves only its theorem under its formal model and assumptions;
- a runtime receipt is a verifiable execution claim, not automatic proof of external truth;
- an accepted evidence chain proves internal consistency and binding, not absence of truncation or bypass unless the assurance profile includes the required independent controls;
- a production runtime may have a lower assurance level than a research runtime with stronger attestation, and vice versa.

The phrase “provable Agentic OS” MUST NOT be used as an undifferentiated blanket safety claim.

* Security Considerations
- Organization hierarchy and C3 lookup MUST NOT imply authority.
- Context composition MUST NOT widen visibility through ordinary precedence.
- Projection drift is prevented by Bundle, projector, policy, proof, and execution-root binding.
- A conforming runtime is not automatically a mediated runtime; bypass claims require the L3 boundary defined by RFC 93.
- A valid Lean artifact applies only to its exact theorem, model, assumptions, Bundle, and import environment.
- Runtime and evidence claims remain vulnerable to external-world deception unless the assurance profile names independent observation or attestation.

* Architecture Validation Matrix
| Boundary | Validation owner | Required evidence |
|----------+------------------+-------------------|
| POO authoring to Bundle | RFC 91 compiler and Bundle fixtures | deterministic digest, typed errors, facet and cross-facet receipts |
| Bundle to C runtime boundary | RFC 92 C harness and Python runtime | conformance, ownership, fault, and performance receipts |
| Bundle to Cedar and tokens | RFC 93 authorization suite | request, policy, decision, token, replay, and mutation receipts |
| Bundle to Lean | RFC 93 proof packages | named theorem, statement, environment, and proof receipts |
| Runtime result to accepted transition | RFC 93 evidence verifier | execution-root binding and transition-verification receipt |
| Migration and cutover | RFC 94 phase gates | P0–P6 exit receipts and stale-route removal evidence |

* Architecture-Level Failure Rules
The architecture fails closed when:

- a Bundle, facet, projection, or digest does not match;
- an unsupported schema or ABI major version is encountered;
- required proof or policy artifacts are absent;
- a token refers to a stale epoch, expired lease, or revoked grant;
- a runtime submits a result for an unrecognized effect or mismatched intent;
- required evidence is missing, malformed, replayed, or bound to another execution root.

An effect that may have occurred but lacks a durable completion receipt enters an explicit =indeterminate= state. It MUST NOT be silently represented as either success or failure.

* Non-Goals
This RFC does not:

- make POO Flow itself a general-purpose operating system kernel;
- implement production Rust, TypeScript, or other downstream runtimes;
- implement a container runtime, distributed scheduler, or consensus protocol;
- replace Cedar with a Scheme policy engine;
- replace Lean with schema validation or runtime receipts;
- prove model output quality, external-service honesty, or arbitrary real-world outcomes;
- make all runtime internals part of one global proof;
- require a graph database;
- freeze an ABI before the Gerbil, Gambit, CFFI, and gerbil-mysql performance gates pass;
- preserve retired public authoring surfaces indefinitely.

* Compatibility And Removal
During migration, an internal shadow projection MAY compare the current path with the new Bundle path. Shadow machinery is not a new public API.

Once a new authoring, Bundle, ABI, or evidence route becomes canonical and its cutover gate passes:

- obsolete public routes SHALL fail fast or be removed;
- stale documentation and examples SHALL be updated or deleted;
- tests SHALL target the canonical route;
- no indefinite dual-write or dual-semantics mode is permitted.

Externally consumed ABI changes follow explicit major-version rules. Removing stale Scheme authoring routes and versioning an external ABI are different concerns and MUST NOT be conflated.

* Decision Notes
** Agentic OS As POO Flow Alone Rejected
Calling POO Flow alone an Agentic OS collapses the compiler, control plane, runtime, effect mediation, external adapters, and deployment environment into one unverifiable boundary.

The accepted architecture names POO Flow as the compiler, control plane, and runtime interface. An Agentic OS instance is an assembled deployment that includes a conforming runtime and adapters.

** Dual Runtime Ownership Rejected
Python and a hypothetical Rust runtime are not parallel semantic owners inside this project. Python is the first-party complete runtime used to validate the public ABI. Rust and TypeScript adapters are downstream.

** Workflow Graph As Core Model Rejected
A workflow graph cannot distinguish responsibility, authority, information flow, lifecycle, and evidence. RFC 91 defines typed facets instead.

** One Global Proof Claim Rejected
Static semantics, local authorization, runtime conformance, mediated execution, and external truth have different trusted bases. RFC 93 defines claim-specific TCBs and assurance levels.

** ABI Convenience Before Semantics Rejected
Freezing a C layout before the Bundle and effect semantics stabilize would make implementation accidents normative. The Bundle precedes ABI freeze.

* Acceptance Criteria
RFC 90 is satisfied when:

1. All RFC 90–94 documents are reviewed and internally consistent.
2. The project uses the architecture vocabulary in public design and implementation planning.
3. The Canonical Organization Bundle is the declared semantic center.
4. Gerbil/POO remains the public organization-authoring route.
5. C ABI, Cedar, Lean, evidence, Python, and downstream runtimes are treated according to their accepted ownership boundaries.
6. No document claims that a valid Bundle, Cedar allow, Lean artifact, or runtime receipt alone proves total system safety.
7. The migration plan provides a vertical slice and executable gates before canonical cutover.
8. Retired public routes are removed after their canonical replacement passes the cutover gate.
