#+title: CI Capability Cache Invariants
#+language: en

* Status

This document records the implemented CI cache boundary that follows the
adaptive Scheme build guard.  It is normative for Bazel-owned Gerbil toolchain
and dependency bootstrap behavior.

* Metric boundaries

The accepted POO Flow project cold-build reference is =161.694s=.  A first-use
Gerbil toolchain bootstrap is a separate provisioning metric and MUST NOT be
reported as project cold-build time.  A repeated CI run for the same source,
compiler capability, architecture profile, and dependency inputs is expected
to restore the toolchain cache and return to the project-level measurement.

* Capability cache invariants

1. Local and downstream builds default to the =native= architecture profile so
   the compiler may select the highest-performance instructions available on
   the current machine.
2. Hosted CI uses the =portable= architecture profile so cache identity remains
   reusable across compatible runners.
3. The ABI fingerprint hashes compiler-selected diagnostics, compiler identity,
   target tuple, and architecture profile.  Random =cc*.s= paths and probe file
   locations are execution noise and MUST NOT enter the cache key.  A missing
   compiler argument resolves through the host's =cc= capability, while a
   command name such as =gcc= resolves through =PATH= before validation.
4. The dependency installer derives its writable package root from
   =GERBIL_PATH=, falling back to the Bazel run workspace's project-local
   =.gerbil= root when =GERBIL_PATH= is absent.  The installer exports the
   selected root before invoking =gxpkg=.  Bazel owns this invariant;
   workflow-specific directory creation is not an acceptable substitute.
5. CPU parallelism is discovered from the executing system.  No workflow,
   test, or cache rule may encode a fixed logical CPU count.
6. CI requires ccache as a bootstrap capability.  The Bazel-owned bootstrap
   tool wraps the selected compiler, observes compiler activity, and records
   the result in its receipt.  Restoring a ccache directory without proving
   compiler activity is not accepted as cache integration.

The Bazel repository rules and their generated tools own the arrows in this
graph.  GitHub Actions selects the portable profile and supplies cache storage;
it does not reimplement toolchain or package-root policy.

* Failure-closed behavior

- A missing or non-executable compiler terminates fingerprint generation.
- An unknown architecture profile terminates fingerprint generation.
- A host without SHA-256 tooling terminates fingerprint generation.
- A dependency install cannot begin until the selected Gerbil package root
  exists.
- CI bootstrap terminates when ccache is absent or observes zero compiler
  activity.
- A non-deterministic fingerprint is a cache correctness failure, not an
  acceptable cache miss.

* Safe Gerbil bootstrap caches

An interrupted in-tree Make build is not a valid cache unit.  Generated Gambit
sources and target timestamps can diverge when a process is cancelled, causing
a restored tree to report completed prerequisites while files such as
=_gambit.c=, =gsi=, or =gsc= are absent.  Only ABI-bound ccache objects and a
fully validated installation prefix may cross an interrupted bootstrap.  A
source-cache save therefore requires successful completion; a failed source
bootstrap starts again from a fresh checkout while retaining ccache.

The Darwin PR lane resolves the platform capability from Homebrew Gerbil
0.18.2.  Resolution is layout-independent: the workflow discovers candidates
through both the Homebrew opt prefix and Cellar inventory, then accepts a root
only when the =gxi=, =gsc=, and library capabilities are all present.  The
matrix also declares the upstream versioned home name.  When a bottle flattens
=bin= and =lib= into the keg root, CI creates a versioned directory containing
relative links to the validated =bin=, =include=, =lib=, and =share=
capabilities.  This lets =std/make= resolve the same =gsc= and library root
without embedding a machine-specific path.  The pinned post-release source
bootstrap remains available for Linux and explicit toolchain seeding, but it
is not repeated for ordinary macOS project validation.

Homebrew Gerbil 0.18.2 predates the upstream =git.cons.io= package-provider
fix.  Darwin therefore builds only the upstream =gxpkg= module at the pinned
=add922481cec81d05431db014bcb91b30485cf21= ref.  Its compiled module set and
JSON receipt live in a separate native-ABI-keyed Actions cache; no generated
binary is committed, and the compiler/runtime capability is not rebuilt.

Provider-qualified package names are dependency identities, not interchangeable
download mirrors.  The root manifest therefore uses the canonical
=git.cons.io/mighty-gerbils/gerbil-utils= identity already declared by
=gerbil-poo=.  Mixing that identity with the historical =github.com= mirror
causes =gxpkg= to clone and compile the same package twice, defeating the
package cache and the twelve-minute CI gate.

The native CI lane performs one canonical project compilation: Bazel invokes
=build.ss= and then runs the declared Scheme and runtime-C acceptance targets.
The custom loop-engine capability root is a first-class Bazel test target.
CI must not clean those outputs and repeat the same full project compilation
through a second shell route; that duplicates work without adding coverage and
can push macOS beyond the twelve-minute gate.

Every expensive source phase atomically updates an English-keyed JSON state receipt.
The workflows upload that receipt even after cancellation, making the active
phase, selected compiler, architecture, available build cores, elapsed time,
and exit status observable without scraping an unbounded console log.

The installed Gerbil prefix remains the primary capability cache.  When that
cache is absent, CI starts from a fresh upstream checkout and restores only the
ccache directory keyed by runner OS, architecture, Gerbil source label, and the
deterministic native compiler ABI fingerprint.  An interrupted Make tree is
discarded; only compiler objects may survive the retry.  A completed build
installs into a fresh prefix and must pass the existing activity and runtime
guards before that prefix can become the primary cache.

The source tree also carries a configure signature over the Gerbil ref,
installation prefix, resolved compiler command, and architecture profile.  A
matching signature reuses =config.status= and skips unconditional configure
regeneration; a mismatch reruns configure and replaces the stamp.  Existing
capability-keyed CI trees may adopt a completed =config.status= once when
migrating from the pre-stamp cache format.

* Acceptance receipts

An interrupted source bootstrap is accepted only when its ccache is preserved,
its raw source tree is rejected, and the uploaded
=poo-flow.gerbil-bootstrap-state.v1= receipt identifies the last active phase.
A successful bootstrap must produce the validated installed-prefix cache;
subsequent source jobs must hit that cache and skip the bootstrap step entirely.

The implementation is accepted only when all of the following hold:

- Bash 3.2 parses the fingerprint and dependency installer tools.
- Repeated native fingerprints are identical on one unchanged host.
- Repeated portable fingerprints are identical on one unchanged host.
- Native and portable fingerprints are distinct.
- =bazel build //scheme:deps= rematerializes the dependency installer.
- Main CI and Python runtime CI use one OS/compiler/ref-keyed ccache namespace.
- The bootstrap receipt reports ccache as required and enabled with non-zero
  compiler activity.
- The first portable CI run saves the toolchain cache.
- A rerun of the same commit restores that cache and reports project cold-build
  time independently from any first-use bootstrap time.
