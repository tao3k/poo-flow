* Status

This document is normative for the Gerbil v0.18.2 build lanes.

- Source version: =v0.18.2=
- Source revision: =07c8481588a8b07dbf05832687817cd398902ac0=
- Source owner: =https://github.com/mighty-gerbils/gerbil.git=
- macOS formula: =tao3k/gerbil/gerbil-scheme=
- macOS runtime home: =$(brew --prefix tao3k/gerbil/gerbil-scheme)/current=
- Scheme build truth: =build.ss=
- Outer orchestration: Bazel and GitHub Actions

* Provider-era decision

Gerbil v0.18.2 belongs to the GitHub provider era.  The =git.cons=
provider transition is a v0.19 migration concern and must not leak into the
v0.18.2 dependency graph.

Therefore:

1. The default v0.18.2 source remote is GitHub.
2. v0.18.2 package identities use =github.com/...=.
3. CI must not install a =git.cons= package directory for v0.18.2.
4. CI must not manufacture provider aliases, filesystem overlays, or synthetic
   package identities.
5. A future v0.19 migration must be explicit and atomic: toolchain, dependency
   identities, caches, receipts, and documentation move together.

* Capability model

The build consumes a Gerbil capability, not an assumed filesystem layout.

** Source capability

=tools/bazel/bootstrap_gerbil_toolchain.sh= owns source acquisition,
configuration, parallel build, installation, and the bootstrap receipt.

The default source URL is the v0.18.2 GitHub owner.  =GERBIL_SOURCE_URL= may
select an equivalent mirror, but the receipt must record the selected URL and
the exact revision.  A mirror does not change the package-provider identity.

The source checkout presents its version as =Gerbil 07c8481=, while the
qualified Homebrew formula presents =Gerbil v0.18.2=.  The Bazel toolchain
accepts both strings because they are two presentations of the same pinned
revision, not two interchangeable Gerbil revisions.

The bootstrap selects available parallelism at runtime.  It must not encode a
fixed logical CPU count.  =GERBIL_BUILD_CORES= is an explicit override only.

** Homebrew capability

The macOS lane installs the fully qualified formula
=tao3k/gerbil/gerbil-scheme=.  It resolves the formula prefix declaratively and
uses the formula's =current/= capability directory as both =GERBIL_PREFIX= and
=GERBIL_HOME=.

The workflow must validate at least:

- =current/bin/gxi=
- =current/bin/gsc=
- =current/lib=
- Gerbil reports =v0.18.2=
- Gambit reports =v4.9.7=

The workflow must not probe Cellar versions, create replacement symlinks, or
derive a home from a Homebrew/core layout.

** Dependency capability

=gerbil.pkg= is the dependency identity owner.  Bazel may materialize and run
the dependency installation, but it must not rewrite provider identities.

For v0.18.2, =gerbil-poo= and =gerbil-utils= use GitHub identities.
=gerbil-poo= is pinned to =1c3225637525400341a09e958296962c10184ead=,
the direct parent of the provider-migration commit.  The migration commit
=f4b322299830acbd1cd5b91984c4107ac56bdac3= is not a valid v0.18.2 pin because
its manifest changes =gerbil-utils= to =git.cons=.  The dependency installer
invokes native =gxpkg deps --install= without adding a =git.cons= directory and
without an overlay.

* Cache identity

A cache is reusable only when every ABI-relevant capability matches.

The cache key must distinguish:

- operating system and architecture;
- exact Gerbil label and revision;
- native ABI fingerprint;
- architecture profile;
- dependency manifest digest;
- the capability generation when the source owner or formula layout changes.

The source URL and resolved prefix are also written to receipts.  They are
evidence fields, not permission to reuse a cache with a different revision.

* Performance and observability invariants

1. The source bootstrap uses all available build parallelism unless explicitly
   overridden.
2. Source lanes require =ccache= and record whether compiler activity occurred.
3. The bootstrap writes a JSON state receipt during every phase and a final JSON
   receipt after installation.
4. Cache hits skip the source build.  Cache misses remain observable rather
   than being hidden by retry loops.
5. A cold build at or above 720 seconds is a blocking regression and requires
   phase-level diagnosis.
6. Performance comparisons are made per operating system and cache state; a
   warm run is never reported as a cold baseline.

* Build ownership

=build.ss= remains the canonical Scheme project build definition.  Bazel owns
toolchain discovery, dependency materialization, cache boundaries, runtime-C
rules, and test orchestration.  GitHub Actions declares the machine matrix and
persists capability-keyed caches.

This separation prevents a CI-only shell recipe from becoming a second build
system.

** Python wheel canonical Scheme-action reuse

The Linux =native= job produces the canonical =//scheme:compile= Bazel
action.  The reusable Python wheel workflow depends on that job and consumes
the same capability-keyed =setup-bazel= disk-cache namespace.  Its distinct
BUILD generation restores through the namespace prefix and then saves the
enriched cache containing the projection action.  The consumer receipt must
distinguish its requested generation from the cache lineage it consumed.

=GerbilProjectProjection= consumes
=GerbilProjectInfo.compiled_root= and declares one projection action.  It
must not invoke =build.ss=, compile the project again, or copy the compiled
tree.  =build.ss= remains the only Scheme build owner.

The Python wheel CI receipt records the producer/consumer phase, requested
cache generation and role, cache lineage, restore mode, elapsed time, status,
disk-cache hit count, and canonical Scheme receipt availability.  A
non-dispatch consumer must complete the warm =//scheme:compile= gate within
30 seconds and observe at least one disk-cache hit.  The projection action
must emit the declared wheel projection artifact within 120 seconds.

* Acceptance gates

A v0.18.2 provider-boundary change is complete only when all of the following
hold:

- no active workflow or Building Framework script contains =add922=;
- no active v0.18.2 source or package path contains =git.cons.io=;
- no gxpkg provider overlay target or script remains;
- =gerbil-poo= is pinned before its provider-migration commit;
- the Linux and Python lanes use revision =07c8481588a8b07dbf05832687817cd398902ac0=;
- the macOS lane uses =tao3k/gerbil/gerbil-scheme= and =current/=;
- workflow YAML parses successfully;
- shell syntax checks pass;
- the Org document contains both Mermaid diagrams;
- CI validates the Bazel graph, Scheme build, runtime-C tests, Python
  projection, and Lean proof bridge.

* v0.19 migration boundary

The future v0.19 change is a new capability generation, not a compatibility
patch for v0.18.2.  Its RFC must update the source owner, package identities,
cache namespace, toolchain acceptance predicate, and receipts in one change.
Until that RFC lands, v0.18.2 remains GitHub-native.
