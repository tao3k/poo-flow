* Provider-era decision

Gerbil v0.18.2 belongs to the GitHub provider era.  The =git.cons=
provider transition is a v0.19 migration concern and must not leak into the
v0.18.2 dependency graph.

Therefore:

1. The default v0.18.2 source remote is GitHub.
2. v0.18.2 package identities use =github.com/...=.
3. CI must not install a =git.cons= package directory for v0.18.2.
4. CI must not manufacture provider aliases, filesystem overlays, or synthetic
   package identities.
5. A future v0.19 migration must be explicit and atomic: toolchain, dependency
   identities, caches, receipts, and documentation move together.

* Capability model

The build consumes a Gerbil capability, not an assumed filesystem layout.

** Source capability

=tools/bazel/bootstrap_gerbil_toolchain.sh= owns source acquisition,
configuration, parallel build, installation, and the bootstrap receipt.

The default source URL is the v0.18.2 GitHub owner.  =GERBIL_SOURCE_URL= may
select an equivalent mirror, but the receipt must record the selected URL and
the exact revision.  A mirror does not change the package-provider identity.

The source checkout presents its version as =Gerbil 07c8481=, while the
qualified Homebrew formula presents =Gerbil v0.18.2=.  The Bazel toolchain
accepts both strings because they are two presentations of the same pinned
revision, not two interchangeable Gerbil revisions.

The bootstrap selects available parallelism at runtime.  It must not encode a
fixed logical CPU count.  =GERBIL_BUILD_CORES= is an explicit override only.

** Homebrew capability

The macOS lane installs the fully qualified formula
=tao3k/gerbil/gerbil-scheme=.  It resolves the formula prefix declaratively and
uses the formula's =current/= capability directory as both =GERBIL_PREFIX= and
=GERBIL_HOME=.

The workflow must validate at least:

- =current/bin/gxi=
- =current/bin/gsc=
- =current/lib=
- Gerbil reports =v0.18.2=
- Gambit reports =v4.9.7=

The workflow must not probe Cellar versions, create replacement symlinks, or
derive a home from a Homebrew/core layout.

** Dependency capability

=gerbil.pkg= is the dependency identity owner.  Bazel may materialize and run
the dependency installation, but it must not rewrite provider identities.

For v0.18.2, =gerbil-poo= and =gerbil-utils= use GitHub identities.
=gerbil-poo= is pinned to =1c3225637525400341a09e958296962c10184ead=,
the direct parent of the provider-migration commit.  The migration commit
=f4b322299830acbd1cd5b91984c4107ac56bdac3= is not a valid v0.18.2 pin because
its manifest changes =gerbil-utils= to =git.cons=.  The dependency installer
invokes native =gxpkg deps --install= without adding a =git.cons= directory and
without an overlay.

The cache key must distinguish:

- operating system and architecture;
- exact Gerbil label and revision;
- native ABI fingerprint;
- architecture profile;
- dependency manifest digest;
- the capability generation when the source owner or formula layout changes.

The source URL and resolved prefix are also written to receipts.  They are
evidence fields, not permission to reuse a cache with a different revision.

* Performance and observability invariants

1. The source bootstrap uses all available build parallelism unless explicitly
   overridden.
2. Source lanes require =ccache= and record whether compiler activity occurred.
3. The bootstrap writes a JSON state receipt during every phase and a final JSON
   receipt after installation.
4. Cache hits skip the source build.  Cache misses remain observable rather
   than being hidden by retry loops.
5. A cold build at or above 720 seconds is a blocking regression and requires
   phase-level diagnosis.
6. Performance comparisons are made per operating system and cache state; a
   warm run is never reported as a cold baseline.
7. Development receipts identify the persistent output root and are compared
   only with receipts from the same machine and cache state.  Their observed
   latency is evidence, not a cross-machine fixed threshold.
8. Adaptive slow-window diagnostics expose a logarithmic frontier for further
   diagnosis.  A selected window is not reported as per-module timing or as
   proof that the underlying compiler work is logarithmic.
9. Native CI job and dependency-materialization timeouts are explicit matrix
   capability fields.  The outer job timeout must leave every declared phase
   budget executable; it must not cancel a Scheme compile before that compile's
   own timeout or receipt guard can decide the result.
10. A source-bootstrap measurement names its cache mode.  A normal, install
    cache-miss, and fully cold receipt are different evidence classes and may
    not be compared as if they exercised the same capability state.
11. Bootstrap state records an ordered phase ledger for fetch, configure,
    build, install, verification, and receipt production.  The final receipt
    retains the completed build phases instead of projecting only the last
    active phase.

** Controlled Linux source-bootstrap evidence lane

The manual CI input =gerbil_source_bootstrap_cache_mode= exposes three bounded
evidence modes without deleting a shared GitHub cache:

- =normal= preserves the pull-request and push behavior: restore the Gerbil
  install cache and bootstrap ccache, and build only after an install-cache
  miss;
- =install-miss= bypasses the Gerbil install cache but restores capability-keyed
  ccache, measuring a source bootstrap with compiler-product reuse;
- =cold= bypasses both the Gerbil install cache and bootstrap ccache, measuring
  the complete source bootstrap on the hosted Linux capability.

Forced evidence runs do not save the Gerbil install tree into the immutable
production key.  A completed compiler cache remains safe to publish under its
run-scoped, native-ABI-keyed ccache identity; subsequent =cold= runs bypass that
cache explicitly.  Pull-request and push events always select =normal=.

The state JSON records the current phase, its elapsed time, and the completed
phase ledger.  The terminal bootstrap receipt records cache mode, adaptive
available-core selection, compiler and ccache activity, total elapsed time, and
the same completed phase ledger.  CI uploads both receipts.

The Linux 60-minute job timeout remains a provisional liveness ceiling while
this controlled cold receipt is absent.  It is not an accepted performance
budget.  After the cold run, the measured bootstrap phase receives its own
timeout and the outer job timeout is reduced to the declared phase envelope.

* Build ownership

=build.ss= remains the canonical Scheme project build definition.  Bazel owns
toolchain discovery, dependency materialization, cache boundaries, runtime-C
rules, and test orchestration.  GitHub Actions declares the machine matrix and
persists capability-keyed caches.

This separation prevents a CI-only shell recipe from becoming a second build
system.

** Hermetic CI and incremental development lanes

The Bazel graph exposes two explicitly different execution lanes over the same
canonical =build.ss= definition and the same registered Gerbil toolchain:

- =//scheme:compile= and =//scheme:unit_tests= remain hermetic CI targets.  The
  compile action owns a declared tree artifact, and no CI consumer may depend on
  developer-local state.
- =//scheme:dev_compile= and =//scheme:dev_unit_tests= are tagged =local= and
  =manual=.  They are entered through =bazel run= and reuse the stable
  workspace-local =.gerbil= root so Building Framework can preserve module
  products across edits.  =POO_FLOW_BAZEL_DEV_ROOT= may select another
  workspace-relative root when a developer needs isolation.

The development launcher does not rediscover =gxi=, =gxc=, or =gxpkg= from the
host =PATH=.  Bazel resolves those executable capabilities through the Gerbil
toolchain and projects their runfile paths into the existing Scheme project
runner.  =gxc= therefore remains the compiler used by =build.ss=; it is not an
independent build-system entrypoint.

The local output root is an acceleration surface only.  It is not a declared
Bazel artifact, a CI cache contract, or a producer-consumer handoff.  The
hermetic lane remains the only acceptance authority, while the development lane
provides fast edit-compile-test feedback without duplicating build semantics.

** Python wheel canonical Scheme-action reuse

The Linux =native= job produces the canonical =//scheme:compile= Bazel
action and then produces =//scheme:wheel_smoke_projection_artifact= from the
declared compiled tree.  The reusable Python wheel workflow depends on the
complete =native= matrix and downloads that projection as a named workflow
artifact.  It verifies the producer receipt and artifact digest before any
runtime packaging step.

The capability-keyed =setup-bazel= disk cache remains an acceleration layer.
It is not the cross-job projection delivery contract: a prefix cache restore
can select a valid cache container without proving that a specific declared
action result is present.  The named workflow artifact is the explicit,
fail-closed handoff between the native producer and Python wheel consumer.

=GerbilProjectProjection= consumes
=GerbilProjectInfo.compiled_root= and declares one projection action.  It
must not invoke =build.ss=, compile the project again, or copy the compiled
tree.  =build.ss= remains the only Scheme build owner.

The producer receipt records the owning Bazel action, canonical compile owner,
elapsed time, status, budget, artifact availability, and SHA-256 digest.  The
consumer receipt records the named workflow artifact, transport, verified
digest, and fail-closed verification result.  The projection action must emit
the declared wheel projection artifact within 120 seconds.  The Python wheel
consumer must never invoke =//scheme:compile= or rebuild the projection.

*** Runtime-C native producer handoff

The Linux native matrix entry is the single CI producer for both the canonical
Scheme projection and the runtime-C library consumed by the Python wheel.  It
reuses the same Bazel server that owns =//scheme:compile=, builds the existing
=runtime_c_static= and =runtime_c_shared= targets, and runs the complete Linux
runtime-C acceptance matrix before publishing any runtime library.

The named =poo-flow-runtime-native-library= artifact contains the staged shared
library, its producer receipt, and the Bazel acceptance receipt.  The producer
receipt is fail-closed and records the producer job and target, platform,
current commit, Gerbil native-ABI capability, artifact path and byte count,
SHA-256 digest, exported-symbol digest and count, acceptance-receipt digest,
elapsed time, budget, status, and availability.

The reusable Python workflow contains only the wheel consumer.  It downloads
the Scheme projection and runtime-C handoffs, verifies their commit identities
and digests, runs the Python native-runtime validation, and builds and installs
the wheel.  It performs zero Gerbil setup and zero Bazel invocation.  Disk
cache state is therefore an acceleration concern of the native producer, not a
cross-job delivery protocol.

* Acceptance gates

A v0.18.2 provider-boundary change is complete only when all of the following
hold:

- no active workflow or Building Framework script contains =add922=;
- no active v0.18.2 source or package path contains =git.cons.io=;
- no gxpkg provider overlay target or script remains;
- =gerbil-poo= is pinned before its provider-migration commit;
- the Linux and Python lanes use revision =07c8481588a8b07dbf05832687817cd398902ac0=;
- the macOS lane uses =tao3k/gerbil/gerbil-scheme= and =current/=;
- workflow YAML parses successfully;
- shell syntax checks pass;
- the Org document contains both Mermaid diagrams;
- CI validates the Bazel graph, Scheme build, runtime-C tests, Python
  projection, and Lean proof bridge.

** Incremental development lane gate

- =//scheme:dev_compile= and =//scheme:dev_unit_tests= are =local= and =manual=
  targets and are absent from the CI dependency graph.
- Both targets resolve every Gerbil executable through the registered Bazel
  toolchain and invoke the canonical =build.ss= runner.  Direct host-PATH
  compiler discovery is forbidden.
- The default persistent root is =.gerbil=; any override remains
  workspace-relative and rejects parent traversal.
- =dev_unit_tests= carries the complete Scheme test source runfiles and runs the
  canonical =t/unit-tests.ss= root after incremental compilation.
- Repeated runs clean temporary tool wrappers and leave the persistent compile
  products reusable.
- A development receipt cannot replace hermetic =//scheme:compile=,
  =//scheme:unit_tests=, or a named CI producer artifact.

** Linux source-bootstrap evidence gate

- Manual =normal=, =install-miss=, and =cold= modes must preserve distinct cache
  semantics and emit their selected mode in the terminal receipt.
- The =cold= mode must skip both install-cache and ccache restoration without
  deleting or overwriting an existing install-cache key.
- The bootstrap test must verify the ordered phase ledger, cache mode, adaptive
  core count, ccache activity, terminal receipt, and fail-closed invalid-mode
  handling.
- No measured cold receipt may be converted into a new outer timeout until the
  bootstrap phase has its own explicit guard.
- The source bootstrap step must fail at 12 minutes, independently of the outer
  job ceiling.
- The Linux outer job ceiling must remain 45 minutes or lower unless a newer
  phase ledger justifies a different declared envelope.
- A forced source bootstrap must upload its hidden state receipt even when the
  build fails or times out; a missing receipt is itself an acceptance failure.
- The always-run bootstrap finalizer must close a =running= timeout receipt as
  =timed-out=, append the terminal phase, and remain idempotent when invoked on
  an already-terminal state.
- A successful source-build step with an unclosed state receipt must fail the
  finalizer with exit code 70 instead of publishing ambiguous evidence.

** Runtime-C producer-consumer gate

- Linux native must build =runtime_c_static= and =runtime_c_shared= and must run
  =runtime_c_tests=, =runtime_c_sanitizer_tests=, =runtime_c_leak_test=,
  =runtime_v0_batch_benchmark_smoke=, and
  =runtime_v0_gerbil_ffi_benchmark= before uploading the runtime handoff.
- The runtime producer receipt must report schema, producer identity and target,
  platform and native ABI capability, artifact availability, expected artifact
  path, SHA-256 and exported-symbol digests, elapsed time, budget, and status.
- The Python consumer must verify both producer receipts, both artifact digests,
  the runtime acceptance-receipt digest, and the runtime exported-symbol digest
  before executing Python tests or building the wheel.
- The consumer receipt must report the named producer artifacts, transport,
  expected and actual runtime digests, verification status,
  =gerbilSetupAbsent: true=, and =bazelInvocationAbsent: true=.
- The reusable wheel workflow must contain exactly one job and must contain no
  Gerbil activation, Gerbil package installation, setup-bazel action, or Bazel
  command.  This structural zero-Gerbil/zero-Bazel property is an acceptance
  invariant, not a timing inference.
- The runtime-C producer build has a 90-second budget and the complete Bazel
  acceptance phase retains its 150-second budget.  A topology change is not
  accepted unless the complete workflow remains below the recorded pre-change
  wall-clock baseline.

* v0.19 migration boundary

The future v0.19 change is a new capability generation, not a compatibility
patch for v0.18.2.  Its RFC must update the source owner, package identities,
cache namespace, toolchain acceptance predicate, and receipts in one change.
Until that RFC lands, v0.18.2 remains GitHub-native.
