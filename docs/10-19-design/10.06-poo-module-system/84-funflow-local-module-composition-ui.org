* 10.06.84 FunFlow Local Module Composition UI

This note fixes the public authoring boundary between =use-module funflow= and
=use-composition=. The intent is to keep FunFlow capability loading inside the
composition scope while preventing =use-composition= from becoming a backend or
runtime definition language.

** 10.06.84.10 Decision

=use-composition= owns local composition assembly. A composition may load a
module locally, bind it to an alias, select one or more module profiles, and
then compose those profiles before declaring stages, steps, edges, or routes.

#+begin_src scheme
(use-composition repo-ci
  (use-module funflow as ff
    (profiles github-ci))

  (compose
    (profile ff github-ci))

  (stage pull-request
    (step build
      (run "gxpkg" "build" "-g"))
    (step test
      (run "uv" "run" "pytest" "-q"))
    (edges
      (build -> test))))
#+end_src

The important relationship is local and explicit:

#+begin_src text
use-module funflow as ff
  -> ff/github-ci profile slot
  -> compose consumes (profile ff github-ci)
  -> stages consume generic flow/DAG clauses
#+end_src

=github-ci= is not a global symbol. It is a profile slot on the locally bound
=ff= module object. That module profile owns the GitHub Actions schema, type
facts, contracts, and projection rules. The composition only consumes the
profile.

** 10.06.84.20 Boundary

The boundary is:

- =use-module funflow= loads FunFlow-owned profiles and capability contracts.
- =as ff= creates a composition-local module binding.
- =(profiles github-ci)= selects profiles exported by the loaded module.
- =compose= combines selected profiles; it does not expose FunFlow internals.
- =stage=, =step=, =edges=, and =route= describe generic flow/DAG structure.

Do not expose =core=, =dag=, or =router= as profiles that users must compose.
Those are FunFlow or graph-engine internal capabilities. Users compose the
profile that matters to their target, such as =github-ci=, =python-anyio=, or a
project-specific durable profile.

** 10.06.84.30 Why Not Global use-module

This should not be split into a global import plus a separate composition:

#+begin_src scheme
(use-module funflow as ff
  (profiles github-ci))

(use-composition repo-ci
  (compose
    (profile ff github-ci))
  ...)
#+end_src

That shape leaks composition-local dependencies into global state. It also makes
it harder to know which module profile set was active when the flow instance was
assembled. Keeping =use-module= inside =use-composition= makes the dependency
graph local, inspectable, and reusable.

** 10.06.84.40 Profile Levels

The user interface has three levels.

The default level is module profile selection:

#+begin_src scheme
(use-module funflow as ff
  (profiles github-ci))
#+end_src

The local override level extends a selected profile only when the specialization
belongs to this composition:

#+begin_src scheme
(use-module funflow as ff
  (profiles github-ci)
  (profile strict-ci
    :extends (profile ff github-ci)
    :capabilities (runner cache artifact permissions proof-gate)))
#+end_src

The advanced POO-native level reuses an existing object directly:

#+begin_src scheme
(def enterprise-ci
  (.o (name 'enterprise-ci)
      (schema 'github-actions/workflow)
      (projection 'github-actions-yaml)
      (capabilities '(runner cache artifact permissions audit-log))))

(use-composition repo-ci
  (use-module funflow as ff
    (profile enterprise-ci))
  (compose
    (profile ff enterprise-ci))
  ...)
#+end_src

All three levels normalize to POO profile objects on the module alias. Runtime
languages receive a projection after composition; they do not own this syntax.

** 10.06.84.45 Python Runtime Load Fixture

Python runtime fixtures must exercise the same public UI that users write. The
fixture loaded by =poo_flow_runtime.load(...)= should therefore return a
=use-composition= object:

#+begin_src scheme
(use-composition python-runtime-ci
  (use-module funflow as ff
    (profiles github-ci python-anyio))
  (compose
    (profiles ff github-ci python-anyio))
  (stage default
    (step build ...)
    (edges
      (build -> test))))
#+end_src

The Scheme runtime-load projection owns normalization from this composition
object to the FunFlow plan projection consumed by Python AnyIO. The fixture must
not fall back to the old =(use-module funflow :config ...)= selection shape,
because that bypasses the composition-local module binding and top-level
=compose= contract.

** 10.06.84.46 Python Runtime Load Performance

Python =load(...)= has two distinct costs:

- cold projection: run Scheme once to normalize user =use-composition= syntax
  into a FunFlow plan projection;
- warm projection: reuse the normalized Scheme projection rows and bind fresh
  Python actions each call.

The warm path is the user-facing steady-state contract. The Python runtime keeps
an in-process projection-row cache keyed by the user Scheme file, cwd, runtime
projection owner, and the Scheme module owners that define composition and
FunFlow projection semantics. A changed file or changed owner invalidates the
cache. Users may call =clear_load_cache()=, pass =cache=False=, or set
=POO_FLOW_SCHEME_LOAD_CACHE=0= when they need a forced cold projection.

Cold projection should prefer a direct =gxi= invocation when the local Gerbil
package environment is already materialized under =.gerbil=. It falls back to
=gxpkg env gxi= when direct loading cannot be proven safe. This keeps package
resolution correct while avoiding repeated =gxpkg= startup on the hot path.

Gerbil AOT is a separate cold-start acceleration lane. Python exposes
=precompile_load(...)= to compile the generated projection runner with Gerbil's
native =gxc -exe -o= executable path. Ordinary =load(...)= will reuse an
existing fingerprint-matched AOT binary when present, but it does not compile
one synchronously by default. This is deliberate: the executable compile is
appropriate for CI, package installation, or an explicit warmup step, while a
user's first request must not unexpectedly spend minutes in native compilation.
Callers that explicitly want compile-on-load may pass =compile_aot=True= or set
=POO_FLOW_SCHEME_LOAD_AOT_COMPILE=1=.

The package exposes the explicit warmup path as a module command, without
adding another project script entry:

#+begin_src sh
uv run --project packages/python-runtime python -m poo_flow_runtime.scheme_load_aot \
  precompile packages/python-runtime/tests/fixtures/funflow_user_module.ss \
  --cwd .
#+end_src

The command emits one =poo-flow.scheme-load-aot.v1= receipt with the source,
workspace root, binary path, rebuild flag, and elapsed time. That receipt is the
CI/debug surface for proving that AOT was prepared before a fresh Python
process relies on it.

Exact benchmark receipts are owned by the Python runtime benchmark control
plane:

#+begin_src sh
uv run --project packages/python-runtime python -m poo_flow_runtime.benchmarks \
  scheme-load-aot --iterations 10 \
  --source packages/python-runtime/tests/fixtures/funflow_user_module.ss \
  --cwd .
#+end_src

This suite reports =poo-flow.scheme-load-aot-benchmark.v1= rows for
=scheme-load-cache-hit=, =scheme-load-in-process-aot-reuse=, and
=scheme-load-fresh-process-aot-reuse=. Each row carries exact =samples-us= plus
=min-us=, =median-us=, =mean-us=, =p95-us=, and =max-us=. The
=scheme-load-interpreter-no-cache= baseline is available through
=--include-interpreter= because it is intentionally seconds-scale and should not
be hidden inside default runtime checks.

After a projection artifact is exported, benchmark the no-Gerbil path with:

#+begin_src sh
uv run --project packages/python-runtime python -m poo_flow_runtime.benchmarks \
  scheme-load-aot --iterations 10 --include-artifact --preproject-artifact \
  --source packages/python-runtime/tests/fixtures/funflow_user_module.ss \
  --cwd .
#+end_src

This adds a =scheme-load-projection-artifact= row and measures Python loading
the already-normalized Scheme datum artifact without calling =gxi=, =gxc=, or
the AOT executable.

Local receipt on 2026-07-08 for =funflow_user_module.ss=:

** 10.06.84.46 No-Gerbil Runtime Dependency

AOT alone does not fully solve downstream adoption if it requires every runtime
consumer to install the Gerbil compiler. The stable boundary is:

- Gerbil is an authoring/build-time dependency for Scheme source,
  macro-expansion, normalization, and projection generation.
- Python runtime users should not need =gxi= or =gxc= when they run an already
  published flow.
- Runtime loading must prefer a preprojected artifact before trying an AOT
  executable or a development Gerbil fallback.

The artifact format stays Scheme-native. It is a Scheme datum, not JSON, with a
small manifest wrapper:

#+begin_src scheme
((schema "poo-flow.scheme-load-projection-artifact.v1")
 (source-digest "...")
 (projection-digest "...")
 (rows ((schema "poo-flow.funflow-plan-projection.v1") ...)))
#+end_src

The =rows= payload is exactly the normalized projection that Python already
parses today. Python can therefore load the artifact with its existing Scheme
datum parser and does not need a Scheme runtime for the steady path. The digest
fields prove that the artifact matches the source file and the projection owner
used to generate it. When the source or projection owner changes, the artifact
is ignored and the loader falls through to AOT/Gerbil developer paths.

The runtime load order is:

1. in-process projection cache when a matching cache key is already warm;
2. adjacent or cache-local preprojected Scheme datum artifact;
3. existing fingerprint-matched Gerbil AOT runner;
4. optional compile-on-load AOT only when explicitly requested;
5. development Gerbil interpreter fallback.

The publish-time command shape is:

#+begin_src sh
uv run --project packages/python-runtime python -m poo_flow_runtime.scheme_projection_artifact \
  export packages/python-runtime/tests/fixtures/funflow_user_module.ss \
  --cwd .
#+end_src

That command is allowed to require Gerbil because it belongs to the
Gerbil-enabled authoring/release lane. Downstream runtime loading must work
without Gerbil when the artifact is shipped next to the source file.

The Python runtime wheel workflow therefore has two explicit lanes:

1. =projection-artifact=: Gerbil-enabled authoring/release lane. It restores or
   builds Gerbil with =ccache=, routes package compilation through a
   =GERBIL_GSC= wrapper that passes Gambit =gsc -cc= to a =ccache gcc= wrapper,
   builds the POO Flow package, runs
   =poo_flow_runtime.scheme_projection_artifact export= against packaged Scheme
   projection sources, and uploads the resulting
   =*.poo-flow-projection.sexp= files as CI artifacts.
2. =wheel=: no-Gerbil runtime lane. It downloads those generated artifacts,
   builds the Python wheel, installs the wheel in a clean virtualenv, asserts
   =gxi=, =gxc=, and =gxpkg= are absent, and loads the packaged projection from
   package resources.

The no-Gerbil wheel lane command shape is:

#+begin_src sh
uv build --project packages/python-runtime --wheel --out-dir dist/python-runtime-wheel
python -m venv /tmp/poo-flow-runtime-wheel-smoke
/tmp/poo-flow-runtime-wheel-smoke/bin/python -m pip install \
  dist/python-runtime-wheel/poo_flow_runtime-0.1.0-py3-none-any.whl
/tmp/poo-flow-runtime-wheel-smoke/bin/python -m poo_flow_runtime.scheme_projection_artifact --help
/tmp/poo-flow-runtime-wheel-smoke/bin/python -m poo_flow_runtime.benchmarks scheme-load-aot --help
#+end_src

This CI lane proves that the downstream Python package can install and expose
the runtime/artifact entrypoints without =gxi= or =gxc=. It is intentionally
incapable of generating projection artifacts; it must consume the real artifacts
uploaded by the Gerbil-enabled lane.

For built-in flows, the release lane must place both the Scheme source and the
adjacent =*.poo-flow-projection.sexp= artifact under the Python package tree
before =uv build=, for example under =src/poo_flow_runtime/projections/=. The
wheel build lane verifies that these package data files are present in the
wheel and that the installed package can load them without a source checkout.
It does not generate real projection artifacts; that remains the responsibility
of the Gerbil-enabled authoring/release lane.

The intended deployment pattern is therefore: precompile during CI, install, or
runtime warmup; publish the preprojected artifact for no-Gerbil consumers; use
AOT binary reuse only as a native acceleration fallback; then let the
in-process cache own steady-state repeated calls.

** 10.06.84.50 Acceptance

The interface is acceptable when:

- =use-composition= accepts an inline =(use-module funflow as ff ...)= form;
- =compose= can consume =(profile ff github-ci)= and batch =(profiles ff ...)=;
- =stage= can carry generic =step=, =edges=, and =route= clauses without
  interpreting backend details;
- top-level composed profiles stay separate from stage clauses in the POO
  composition object;
- Python =load()= can consume the composition object and project it into a
  FunFlow plan without requiring the old =:config= fixture;
- repeated Python =load()= calls for the same unchanged Scheme composition stay
  on the warm projection path and do not restart Scheme;
- Python =load()= can consume a preprojected Scheme datum artifact without
  requiring =gxi= or =gxc= on the runtime machine;
- precompiled Gerbil AOT projection runners can be reused across Python
  processes without changing the public Scheme UI;
- existing POO-native direct object reuse remains valid.
