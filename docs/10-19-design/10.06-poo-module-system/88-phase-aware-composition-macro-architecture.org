* Status

This document is the approved architecture contract for the next
=use-composition= refactor.  It applies the Gerbil macro and module model from
[[file:/Users/guangtao/ghq/github.com/tao3k/agent-semantic-protocols/languages/gerbil-scheme-language-project-harness/docs/10-19-research/11.12-gerbil-macro-module-system-research.org][Gerbil Macro And Module System Research]] to =poo-flow=.

It extends, but does not replace,
[[file:44-current-project-programming-style.org][Current Project Programming Style]].  Runtime execution remains outside this
project.  The refactor changes composition declaration, expansion, ownership,
and evidence; it does not add a Scheme runtime executor.

* Problem

The current =profile-composition-use-syntax.ss= owner combines four different
responsibilities:

- the public =use-composition= macro;
- two large =begin-syntax= helper regions;
- compile-time parsing and intermediate-list construction;
- a re-export of the inline runtime helper module.

The public macro already lowers to POO-native builders, which is the correct
semantic direction.  The implementation still has structural defects:

- the grammar accepts a legacy alias-only =use-module= branch;
- the =as= marker is recognized through printed symbol text;
- composition forms are traversed separately for compose and stage output;
- profile lists are flattened through nested =map=, =apply append=, and
  =append= operations;
- syntax-time helpers are not materialized as parser-owned items;
- syntax-time, public syntax, and runtime exports share one module boundary;
- no 1,000-profile or 5,000-profile macro-expansion/RSS gate owns the path.

These are not cosmetic issues.  They obscure hygiene, phase ownership,
single-instantiation behavior, and compile-time allocation.

* Design Principles

1. Syntax objects remain syntax objects until an explicit payload boundary.
2. Compile-time and runtime helpers have different module owners.
3. The public macro is thin: validate, parse once, lower once.
4. Runtime semantic owners remain POO objects and pure functional helpers.
5. Fixed grammar errors are expansion-time errors with source locations.
6. A module context is imported normally and instantiated once; source loading
   and bespoke module collectors are not part of composition expansion.
7. Compile-time performance is measured on the real Gerbil expansion path.

* Target Ownership

| Owner | Responsibility | Public surface |
|-------+----------------+----------------|
| =profile-composition.ss= | Stable composition facade | =use-composition= plus explicitly selected native POO extension helpers |
| =profile-composition-use-syntax.ss= | Thin public macro and final lowering | =use-composition= only |
| =profile-composition-syntax-plan.ss= | Phase-owned grammar parser, validation, and immutable syntax plan | Compile-time import only |
| =profile-composition-builders.ss= | Ordinary POO composition, module-binding, stage, and clause builders | Selected through the facade |
| =profile-composition-inline-runtime.ss= | Runtime construction of inline/imported POO profiles | Selected through the facade; no wildcard re-export from syntax owner |

The facade is the consumer-facing module.  The syntax owner must not re-export
the complete runtime module.  Generated identifiers carry their original
module bindings, so lowering does not require exposing private runtime helpers
as user configuration.

The syntax-plan owner is compiled macro support.  It is imported through the
canonical phase-adjusted Gerbil module path for the active compiler.  It must
not be loaded with =load=, evaluated from source, or stored in a mutable module
global.

* Expansion Dataflow

#+begin_src mermaid
flowchart LR
  A["use-composition syntax object"] --> B["shape and literal validation"]
  B --> C["immutable composition syntax plan"]
  C --> D["single lowering traversal"]
  D --> E["ordinary POO-native builder forms"]
  E --> F["composition POO object"]
  F --> G["receipt or runtime projection"]
#+end_src

The syntax plan retains syntax objects for the composition name, module name,
alias, profile expressions, extension expressions, clauses, and stages.  It is
a private phase-owned class with named fields, constructors, and accessors.  It
exists only to model expander state and syntax objects; runtime semantic owners
remain POO objects.  The class is not a public record DSL and it is never
projected to JSON or runtime rows.

* Canonical Grammar

The supported user form is:

#+begin_src scheme
(use-composition composition-name
  (use-module module-name as alias
    (profiles imported-profile ...)

    (profile local-profile
      :extends base-profile
      :with (extension ...)
      :kind report
      :scope (...)
      :capabilities (...)
      :storage (...)
      :analysis (...)
      :publish (...)
      :retention (...)))

  (compose
    (profile alias profile-name)
    (profiles alias profile-name ...))

  (stage stage-name
    (graph ...)
    (loop ...)
    (prove ...)
    (handoff ...)
    (step ...)
    (edges ...)
    (route ...)))
#+end_src

The profile surface also supports:

#+begin_src scheme
(profile existing-poo-object)
#+end_src

and =:extends= may receive a POO object expression, including an expression
returned by a named profile hook.

The alias-only legacy form is removed.  There is no compatibility expansion,
warning-only path, or second grammar.  Existing project fixtures are migrated
atomically and a negative test proves that the legacy form fails at expansion.

* Hygiene And Syntax Context

The grammar literals =use-module=, =as=, =profiles=, =profile=, =compose=,
=stage=, =graph=, =loop=, =prove=, =handoff=, =step=, =edges=, and =route=
are matched as syntax literals, not through =symbol->string=.

The following rules are mandatory:

- preserve source and lexical context while parsing and planning;
- use binding-aware literal comparison for grammar identifiers;
- keep profile expressions, =:extends= expressions, and hook expressions as
  syntax objects;
- use =syntax->datum= only when intentionally producing a quoted runtime symbol
  or immutable literal payload;
- use a template syntax object when a generated identifier is required;
- never generate identifiers from bare strings;
- do not place hidden activation, I/O, connector calls, or runtime execution in
  the expansion.

* Syntax Plan Contract

One parse produces one immutable plan containing:

- composition name syntax;
- imported module and alias syntax;
- ordered profile declarations and references;
- ordered top-level compose profile expressions;
- ordered stages and their clause syntax;
- source syntax for every diagnostic boundary.

The parser validates before lowering:

- exactly one canonical =use-module ... as ...= form;
- known profile section keys;
- even section key/value arity;
- non-empty imported profile groups;
- unique local profile names;
- unique stage names;
- known top-level and stage clause kinds;
- valid =profile= and =profiles= shapes.

The lowering pass consumes the plan without reparsing source forms.

* Error Contract

Grammar and shape failures are syntax errors, not runtime errors.  Each error
must identify the offending syntax object and use a stable diagnostic category:

- =composition-invalid-module-form=;
- =composition-legacy-module-form=;
- =composition-unknown-profile-section=;
- =composition-missing-profile-section-value=;
- =composition-duplicate-profile=;
- =composition-duplicate-stage=;
- =composition-invalid-compose-clause=;
- =composition-invalid-stage-clause=.

Messages explain the canonical repair.  Expansion errors retain the user
source location.  Runtime builder validation remains responsible for POO object
and policy legality after expansion.

* Allocation And Complexity Contract

The expansion hot path must be linear in the number of declarations and emitted
forms.

- Parse the top-level form once.
- Parse each profile and stage once.
- Accumulate with explicit reverse accumulators or an equivalent bounded
  collector.
- Do not use nested =apply append= over mapped profile groups.
- Do not rescan all forms independently for compose and stage output.
- Do not construct temporary POO runtime objects during expansion.
- Reuse the compiled syntax-plan module through the Gerbil module registry.

For =n= profile references and =m= stage clauses, the target work is
=O(n + m)= and generated form count is also =O(n + m)=.

* Performance Evidence

The scenario owner is
=t/scenarios/performance/composition-macro-expansion/=.  It contains a real
declarative composition rather than a string-generation microbenchmark.

The evidence matrix contains:

| Case | Purpose |
|------+---------|
| 1,000 profiles | Required regression gate |
| 5,000 profiles | Stress and allocation gate |
| cold expansion | Full module-context, runtime/meta generation, and compiler path |
| warm expansion | Module-context reuse and no duplicate expansion |
| generated forms | Linear output-size assertion |
| repeated runs | RSS stability and absence of monotonic growth |

The pre-refactor implementation is measured first and stored as the baseline.
The replacement must not regress elapsed time or peak RSS at 1,000 profiles,
must complete the 5,000-profile case without monotonic RSS growth, and must show
module-context reuse on the warm path.  Thresholds are recorded from the real
baseline receipt; profile count, iteration count, and build concurrency are not
reduced to pass the gate.

* Harness Evidence

Harness policy follows parser facts.  The upstream Gerbil harness receives
these scenario-backed facts before new warnings are enabled:

- =macro-hygiene-context-preservation=;
- =phase-aware-helper-split=;
- =single-instantiation-composition-expansion=.

Required parser facts include syntax owner, expansion phase, syntax-object
operations, phase-adjusted imports/exports, and compiled macro boundaries.
Policy must not trigger from the downstream name =poo-flow= or from broad text
matching.

* Public API And Migration

The migration is intentionally compatibility-free:

- the canonical facade becomes the only documented import path;
- the syntax owner exports only =use-composition=;
- native POO extension helpers are selectively exported by the facade;
- project user-interface compositions migrate in the same change;
- the legacy inline module grammar is rejected by a focused negative test;
- no legacy wrapper module or alias remains after consumers migrate.

The generated composition object and its runtime projections preserve their
existing semantic contract.  Runtime languages remain responsible for
execution.

* Relationship To Tool-Core

The current =tool-core= split is accepted as the semantic-owner baseline:
specification, catalog, handoff, policy refs, validation, and receipt have
separate owners behind one facade.  This architecture does not reopen that
split.

Object-family macro adoption in =tool-core= is allowed only where a stable
predicate/accessor/projection frame is demonstrably repeated.  Data-dependent
validation and receipt logic remain ordinary functions.

* Acceptance Contract

The architecture is complete when all of the following are true:

- only the canonical grammar expands;
- generated bindings preserve syntax context and source locations;
- syntax-time parser/planner and runtime builders have distinct owners;
- the public syntax owner no longer re-exports the entire runtime module;
- the expansion parses forms once and uses bounded collectors;
- 1,000-profile and 5,000-profile receipts report expansion time and RSS;
- warm evidence proves module-context reuse;
- Harness scenarios materialize syntax/phase/module facts;
- project programming style documents the new rules;
- user-interface compositions and focused tests use the canonical facade;
- no direct consumer depends on a legacy composition or tool-core owner.

* Non-Goals

- executing agent runtimes in Scheme;
- replacing POO runtime objects with compile-time records;
- turning every helper into a macro;
- introducing a second composition language;
- hiding runtime work in syntax expansion;
- changing build concurrency or benchmark scale to mask regressions.
