#+TITLE: 10.06.77 Gerbil Scheme High Performance Implementation Report

* Executive Summary

This report treats Gerbil Scheme as the current POO Flow control-plane context.
The core conclusion is direct: the high-performance path is not to make each
downstream scenario write more objects. The path is to make objects reusable as
policy families, strategy profiles, prototype compositions, slot projections,
and runtime boundary receipts.

The project already has POO surfaces under =src=, but many reusable shapes
are still expressed as local objects per feature. That weakens the main value of
OOP: object reuse and extension. If two runtime policies, tool-call policies,
or benchmark receipts share the same slots, validation, projection, and
materialization behavior, they should share an object family instead of
duplicating object definitions.

The high-performance target is:

- Configure object families once.
- Compose prototypes and strategies once.
- Compute slot helpers once.
- Validate and materialize only at boundaries.
- Run the hot path through stable function/object helpers.
- Keep runtime handoff thin and typed enough for Rust, Python, and other
  runtimes to call without reimplementing policy semantics.

* Current Evidence

Local =.data/gerbil-poo= research exposed these important POO owners:

- =object.ss=: object instantiation, slot spec application, slot function
  computation, super-slot merging, object adapters, slot accessors, and
  =with-slots=/=def-slots= macros.
- =proto.ss=: prototype identity, prototype composition, and composed
  prototype instantiation.
- =mop.ss=: =Object=, =Class=, =Slot=, =MonomorphicObject=, method call
  routing, slot checking/definition, and =slot-lens=.
- =fun.ss=: functor and parametric functor wrapping, application, and call
  surfaces.
- =polynomial.ss=, =number.ss=, and =fq.ss=: algebraic object families,
  normalization, composition, and finite-field style type construction.

Local =src= evidence shows these current POO Flow anchors:

- =src/module-system/tool-calling-control.ss=: fact-set, fact-ref,
  runtime-validation proof facts, and index ref surfaces.
- =src/core/runtime-protocol.ss=: runtime request/response schemas,
  runtime response normalization, adapter conversion, and invalid response
  handling.
- =src/cli-support/package-build.ss=: build specs, test specs, compile targets,
  build observability budget status, live watchdog, debug tracking lines, and
  package compile entry points.

These anchors prove that POO Flow already has object-like surfaces and
performance gates. The missing piece is stronger reuse: common object families
should be promoted to shared owner families instead of being redefined in each
scenario or subsystem.

* Thinking Framework

** Layer 1: Semantic Object Families

The first layer is the semantic object family. A family is a reusable POO
object shape with stable slots, validation, projection, extension, and
materialization behavior.

Examples:

- Runtime request family.
- Runtime response family.
- Tool-call fact family.
- Policy proof fact family.
- Strategy profile family.
- Durable receipt family.
- Benchmark fixture family.
- Marlin runtime handoff profile family.

The rule is simple: if a shape appears in more than one downstream task, it
should be a family, not a local object.

** Layer 2: Prototype And Strategy Composition

The second layer is composition. Gerbil POO has explicit prototype composition
surfaces. POO Flow should use the same model for strategy composition.

The preferred shape is:

- Base object family defines invariant slots and methods.
- Prototype extension adds scenario-specific behavior.
- Functional strategy combinators compose behavior without mutating the base
  family.
- Runtime adapters only observe the composed result.

This is how reuse and extensibility become cheap. New scenarios should extend a
known family instead of copying the old object and editing slots manually.

** Layer 3: Slot Projection And Hot Path

The third layer is the hot path. Slot access cannot stay dynamic inside tight
loops. The Gerbil POO model points to a better shape:

- Compute slot functions once.
- Merge inherited slots once.
- Apply slot specs once.
- Bind projected slots once with =with-slots= or generated helpers.
- Use monomorphic object shape when the slot set is stable.
- Use lens-like accessors for repeated read/update.

After setup, the loop should operate on projected values, stable closures, and
precomputed object helpers. Boundary conversion from alist/hash/function is
allowed, but only outside the hot path.

** Layer 4: Runtime Boundary

The fourth layer is runtime boundary design. Scheme owns the policy semantics;
Rust, Python, and other runtimes own scheduling, I/O, and low-level execution.

The boundary should expose:

- Stable request/response object families.
- Minimal ABI payloads.
- Prevalidated strategy handles.
- Durable receipt handles.
- Debug metadata that can be enabled without changing the hot path.

The runtime should not need to rebuild policy semantics. It should call into a
stable strategy/control-plane surface.

** Layer 5: Performance Gate And Debug Evidence

The fifth layer is measurement. Every new object family should have a scenario
that proves both semantic behavior and performance behavior.

The gate should retain:

- Total elapsed budget.
- Parse/file/phase timing where applicable.
- Hot versus integration classification.
- RSS or allocation-oriented evidence when available.
- Regression budget.
- Debug receipt that explains which owner or object family caused cost.

Performance work is not complete if it only runs fast once. It must leave a
debuggable receipt.

* Gerbil Scheme Features To Use

** Macro And Phase Features

Use Gerbil's macro system to generate repeated object-family scaffolding:

- Declarative macros for fixed object-family definitions.
- Procedural macros for bounded generation where slot families repeat.
- =begin-syntax= for compile-time helper construction.
- Syntax-local registries for object-family metadata when the metadata must be
  known during expansion.
- Syntax parameters for controlled contextual behavior.

Macro usage should be bounded. The output should be ordinary POO-native or
functional code, not an opaque DSL.

** POO And MOP Features

Use POO/MOP-style features for object reuse:

- Prototype composition for extension.
- =MonomorphicObject= style shapes for stable slot sets.
- =Class=, =Slot=, and =Object= abstractions for reusable family metadata.
- =slot-lens= style helpers for repeated projection/update.
- Slot checkers and slot definers for boundary validation.

These features should replace repeated local object definitions.

** Functional Strategy Features

Use functional combinators for strategy reuse:

- Functor-like wrapping and unwrapping.
- Parametric strategy wrappers.
- Stable call surfaces.
- Composition before execution, not interpretation during execution.

This matches POO Flow's core direction: POO-native policy families plus
functional strategy combinators.

** Runtime And ABI Features

Use Gerbil/Scheme as the policy control plane and leave runtime scheduling to
runtime languages:

- Rust can own shared threads, scheduling, and lower-level performance.
- Python can validate runtime-language integration through CFFI/ctypes style
  access without becoming the policy engine.
- Scheme should expose stable object-family and strategy handles, not force
  every runtime to understand the entire internal POO implementation.

* High-Performance Implementation Options

** Option 1: Static Object Family Reuse

Create shared object-family owners for repeated shapes. This is the immediate
fix for the current duplication problem.

Use this for:

- Tool-call facts.
- Runtime requests and responses.
- Benchmark receipts.
- Durable policy receipts.
- Large runtime profiles.

Expected result: fewer repeated objects, less compile churn, and a clearer
extension model.

** Option 2: Generated Slot Projection

Generate projection helpers for stable slot sets. The scenario defines the
object family once; the hot path uses generated accessors.

Use this for:

- Large profile projection.
- Tool-call fact lookup.
- Runtime response normalization.
- Marlin config-interface style handoff.

Expected result: fewer dynamic lookups and less allocation in tight loops.

** Option 3: Prototype Composition Cache

Compose prototype families once and cache the composed shape. Scenario-specific
extension should become a cheap composition step, not a copy-paste object.

Use this for:

- Runtime strategy variants.
- Durable policy variants.
- Debug-enabled versus hot-path variants.
- Tool-call policy variants.

Expected result: better reuse and clearer extension points.

** Option 4: Lens-Based Update Path

Use lens-like accessors for repeated read/update. This is useful when mutation
is explicit and local, such as receipts, counters, and state snapshots.

Use this for:

- Runtime state snapshots.
- Debug counters.
- Durable receipt update.
- Benchmark attempt/best-result tracking.

Expected result: localized mutation without repeated dynamic slot discovery.

** Option 5: Algebraic Object Families

Use object families for algebraic strategy domains, not only runtime records.
The =gerbil-poo= algebra owners show normalization and composition as first
class operations.

Use this for:

- Policy algebra.
- Strategy algebra.
- Predicate families.
- Cost/budget expressions.
- Durable consistency rules.

Expected result: more semantic reuse than Python-style validation objects.

* Objects Reuse Problem

The current risk is that =src= contains object-like implementations, but reuse
is not yet the default engineering reflex. This creates four problems:

- Repeated object definitions drift over time.
- Slot validation is duplicated.
- Boundary materialization is repeated.
- Hot paths rediscover object shape instead of using precomputed helpers.

The fix is to promote shared shapes into named owner families.

Required family boundaries:

- =runtime-request-family=
- =runtime-response-family=
- =tool-call-fact-family=
- =policy-proof-fact-family=
- =strategy-profile-family=
- =durable-receipt-family=
- =benchmark-receipt-family=
- =large-runtime-profile-family=

Each family should define:

- Stable slots.
- Constructor or prototype instantiation path.
- Boundary validation.
- Projection helpers.
- Extension protocol.
- Materialization protocol.
- Performance scenario.

This is the point where POO becomes useful. Reuse is not achieved by writing
objects; reuse is achieved by making object families extensible and shared.

* Recommended Roadmap

** Step 1: Inventory And Classify Existing Objects

Create a local object-family inventory for =src=. Classify every object-like
shape as one of:

- Family candidate.
- Family extension.
- Boundary adapter.
- Hot-path helper.
- One-off local value.

Only the last category may remain local.

** Step 2: Extract Shared Families

Start with the highest reuse surfaces:

- Runtime request/response.
- Tool-call fact/proof facts.
- Benchmark receipt.
- Durable receipt.
- Strategy profile.

Do not preserve old object duplication patterns as compatibility layers unless
there is a real public API requirement.

** Step 3: Add Scenario Families

Add performance scenarios that prove reuse:

- Shared family construction.
- Prototype extension without duplication.
- Slot projection hot loop.
- Lens update hot loop.
- Boundary materialization only once.
- Debug-enabled family versus hot-path family.

** Step 4: Add Build/Compile Tracking

Reuse must not increase compile time. The package build gate should show:

- Which support owner became stale.
- Which scenario owner caused compile cost.
- Whether new generated helpers increased phase time.
- Whether target count grew unexpectedly.

** Step 5: Runtime Boundary Integration

Expose the reused families through stable runtime bindings:

- Scheme keeps policy semantics.
- Runtime languages receive handles or compact payloads.
- Rust/Python do not reimplement object-family rules.
- Debug receipts can explain strategy family, object family, and runtime
  adapter behavior.

* Final Position

Gerbil Scheme gives POO Flow a strong high-performance implementation model:
macros can remove repeated boilerplate, POO/MOP can express reusable object
families, prototype composition can make extension cheap, functional
combinators can make strategy composition explicit, and runtime boundaries can
stay thin.

The current weakness is not that the project uses POO. The weakness is that
some POO usage has not yet been lifted into reusable object families. The next
performance work should therefore focus on reusable families and generated
helpers, not more local objects.
