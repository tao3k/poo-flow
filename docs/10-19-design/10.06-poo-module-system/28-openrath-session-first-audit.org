** 10.06.28.02 Executive Finding

OpenRath's strongest idea is not a specific Python API. Its strongest idea is
that an agent system should make the flowing runtime value explicit.

OpenRath names that value =Session=. A session carries structured chunks,
lineage, backend placement, usage, and branchability. =Agent=, =Workflow=,
=Tool=, =Memory=, =Sandbox=, and =Selector= are then composable layers around
that flowing value.

POO Flow already has adjacent pieces:

- a POO-backed functional flow category with =compose=, =identity=, =arr=,
  =map=, =bind=, =fanout=, =first=, and =second=;
- sandbox profile objects with derivation-path metadata and POO-native slot
  transforms such as ==>= and ==>.+=;
- report-only user presentations that keep runtime ownership explicit and
  preserve =runtime-executed: #f=;
- CI/CD Marlin handoff bundles that package manifests, agreement reports, ABI
  summaries, and receipts without Scheme executing the runtime.

The missing layer is a first-class POO Flow session object family. That family
should make the runtime value inspectable before execution: chunks, lineage,
placement, memory bindings, tool-call receipts, and handoff manifests.

** 10.06.28.03 OpenRath Model

OpenRath presents itself as a PyTorch-like multi-agent and multi-session
framework. Its public vocabulary is intentionally small:

- =Session=: flowing runtime value, conversation state, lineage, usage;
- =Sandbox=: where tools actually run;
- =Memory=: persistent state across runs;
- =Tool=: model-visible callable operation with runtime behavior;
- =Agent=: reusable session transformation layer;
- =Workflow=: composable container of agents and nested workflows;
- =Selector=: runtime router that keeps ordinary Python control flow usable.

The important structural point is the ordering. OpenRath does not start from
"agent loop" as the root concept. It starts from session dataflow, then treats
agents as transformations over that dataflow.

Implementation evidence from AST summaries:

- =src/rath/session/session.py= defines =Session= with constructors such as
  =from_user_message= and placement/operation methods such as =to=,
  =require_sandbox=, =fork=, and =detach=.
- =src/rath/session/graph/*= owns export, recording, traversal, acyclicity,
  ancestors, descendants, and edge pairs.
- =src/rath/flow/agent.py= defines =Agent= as a subclass of =Workflow=.
- =src/rath/flow/workflow.py= makes =Workflow= callable on a session and tracks
  named agents through attribute assignment.
- =src/rath/flow/selector.py= defines =Selector= as a workflow subclass.
- =src/rath/backend/abc.py= separates =Backend=, =BackendSandbox=, and
  =BackendSandboxSpec=.
- =src/rath/memory/abc.py= separates =MemoryBackend=, =MemoryStore=, and
  =MemoryStoreSpec=.

This split is useful for POO Flow because it mirrors our intended separation:
Scheme owns object configuration, strategy, validation, receipts, and handoff;
runtime backends own execution.

** 10.06.28.04 User Interface Lessons

OpenRath's examples expose a compact user surface:

#+begin_src python
agent = flow.Agent("You are a concise assistant.", provider_from_env())
user = Session.from_user_message("...").to("local")
result = agent(user)
#+end_src

The ladder adds one concept at a time:

- =01_hello_agent.py=: smallest =flow.Agent= program;
- =02_session_lineage.py=: =fork=, =detach=, session graph, JSONL export;
- =03_sandbox_backend.py=: =Session.to(backend, spec=...)=;
- =04_tools_builtin.py=: built-in filesystem and shell tools;
- =05_custom_tool.py=: user-defined =FlowToolCall=;
- =06_mcp_tool.py=: MCP tools;
- =07_streaming.py=: event stream and usage;
- =08_compress.py=: =flow.Compressor=;
- =09_memory.py=: =flow.Agent(memory=...)=;
- =10_provider_variation.py=: provider swap;
- =11_dynamic_selector.py=: =flow.Selector(provider)= routing across agents.

The lesson for POO Flow is that user examples should be a reviewed public
surface, not ad hoc test fixtures. Our =user-interface/custom/my-module/cases=
tree should grow a similar ladder, but in POO style:

- module enablement stays in =init.ss=;
- object definitions stay in =objects.ss= or loaded object fragments;
- workflow/session/agent cases use =use-module= and =load!=;
- runtime-heavy results remain receipts and handoff manifests.

We should not copy Python method chaining directly. In Gerbil, the cleaner
surface is POO object inheritance plus small macros that build objects:

#+begin_src scheme
(use-module workflow
  :config
  (session ci/check
    :inherits (sessions/project-workspace sandbox/nono-project)
    :chunks ((user . "Run the package checks."))
    :placement sandbox/nono-project))
#+end_src

The exact syntax needs its own design pass, but the target shape should be as
small as the OpenRath examples: construct a flowing value, place it on a
backend profile, pass it through an agent/workflow layer, inspect the receipt.

** 10.06.28.05 Current POO Flow Fit

*** 10.06.28.05.01 Functional Flow Kernel

POO Flow already has the right foundation for OpenRath-like composition. The
=flow-category-prototype= records the functional operations as object slots:
=compose=, =identity=, =arr=, =map=, =bind=, =fanout=, =first=, and =second=.

This should stay below user-level workflow syntax. OpenRath's =Workflow= maps
well to our category object, but our advantage is that the category itself can
be extended with POO mixins and C3 inheritance instead of being locked to one
class hierarchy.

*** 10.06.28.05.02 Sandbox Placement
The first landed slice is a report-only session placement bridge:
- =src/modules/session/objects.ss= owns =poo-flow-session-placement-resolve=.
- The function resolves a placement ref against the existing sandbox profile
  catalog, for example =poo-flow-default-sandbox-profiles= from
  =src/modules/agent-sandbox/config.ss=.
- A resolved placement remains a POO mixin prototype. It adds direct slots such
  as =placement-resolved?=, =placement-diagnostics=,
  =placement-runtime-summary=, and =placement-handoff-summary=.
- Missing or invalid sandbox profiles become diagnostic alist receipts instead
  of runtime execution or hard failure during session graph construction.
- =pooFlowSessionGraphPresentation= exposes placement refs, resolved flags, and
  diagnostics alongside lineage edges and chunk counts.

This matches OpenRath's =Session.to(...)= lesson without copying the Python
runtime API. Scheme names the desired placement and validates the catalog
projection; Rust/Marlin or another backend still owns actual sandbox
realization.

The current user-interface case is:
#+begin_src scheme
(def custom-session-placement
  (poo-flow-session-placement-resolve
   'agent/nono
   poo-flow-default-sandbox-profiles))
#+end_src

The important design constraint is that placement is not a nested POO child
inside session. Lineage and placement are mixins composed into the session value
through C3 inheritance. Chunks and diagnostics are plain receipt rows because
they are hot presentation data, not extension surfaces.

*** 10.06.28.05.04 Next Implementation Milestones
1. Add a user-facing session macro under the module-system facade so cases can
   declare session objects without spelling constructor internals.
2. Add agent/workflow transformation receipts over session values, using the
   functional kernel for composition and POO mixins for strategy extension.
3. Add memory/tool-call receipt slots as direct session mixins, with runtime
   execution still delegated outside Scheme.
4. Add a small tutorial ladder mirroring OpenRath's examples:
   session lineage, sandbox placement, tool-call receipt, memory receipt,
   selector receipt, and runtime handoff bundle.

OpenRath's =Session.to(backend, spec=...)= is the closest match to our sandbox
profile derivation.

POO Flow already records derivation steps as audit rows, strips stale
=runtime-executed= metadata when deriving children, rejects backend rows from
user profile rows, and resolves row merge operators through field contracts.
That means backend placement should be modeled as object inheritance and
validated profile projection, not as a direct runtime call.

Future placement object shape:

- =shared sandbox objects= define common filesystem/cpu/memory/network policy;
- =nono=, =docker=, and =cube= inherit and specialize;
- session/workflow cases inherit placement objects;
- projection emits runtime handoff facts;
- runtime validates and executes outside Scheme.

*** 10.06.28.05.03 Presentation and Runtime Boundary

POO Flow's =pooFlowUserConfigPresentation= already exposes a broad report-only
surface: selected modules, feature facts, CI/CD pipelines, runtime command
manifests, Marlin handoff ABIs, loop-engine intents, sandbox summaries, user
entrypoints, API entrypoints, boundary ownership, and replayability.

This is aligned with OpenRath's session graph idea, but currently our flowing
value is scattered across presentation rows. We need a session object that
collects those facts into one value before projection.

The CI/CD Marlin handoff bundle is a good precedent: it packages manifest maps,
summaries, agreement diagnostics, ABI counts, receipt counts, and runtime owner
while keeping =runtime-executed #f=. Session receipts should follow that shape.

** 10.06.28.06 Gap Analysis

*** 10.06.28.06.01 Missing Session Object Family

OpenRath makes =Session= the first-class dataflow value. POO Flow has flow
objects, profile objects, presentation rows, loop-engine intents, and runtime
handoff bundles, but not a single inspectable session value.

Needed objects:

- =session/chunk=: typed content row with role, source, and visibility;
- =session/value=: ordered chunk table plus metadata;
- =session/lineage=: parent ids, branch kind, derivation reason;
- =session/placement=: sandbox profile reference plus projection status;
- =session/usage=: token/runtime/resource usage receipt;
- =session/handoff=: Marlin runtime manifest and result contract.

*** 10.06.28.06.02 Session Graph Operations

OpenRath exposes =fork=, =detach=, lineage recording, JSONL export, traversal,
and acyclicity checks. POO Flow should add report-only equivalents first:

- derive child session from parent;
- detach a session while preserving provenance;
- export lineage as JSONL-compatible alists;
- validate acyclicity in object-validation;
- expose graph traversal receipts in presentation.

Runtime execution can remain out of scope for the first milestone.

*** 10.06.28.06.03 Agent and Selector Surface

OpenRath's =Agent= is a =Workflow= subclass and =Selector= is also a workflow.
In POO Flow, this should not become a hidden class hierarchy. It should become
object mixins:

- =workflow/agent=: prompt, provider ref, tool refs, memory refs;
- =workflow/selector=: selection policy, candidate workflow refs, route result;
- =workflow/compressor=: session summarization intent;
- =workflow/tool=: model-visible schema plus runtime owner.

C3 multiple inheritance is an advantage here. A selector can inherit governance,
sandbox, memory, and provider policy objects without a deep class tower.

*** 10.06.28.06.04 Memory Is Not Yet a First-Class Module

OpenRath has =MemoryBackend=, =MemoryStore=, local memory adapters, BM25,
embedding ranking, resource ingestion, commit operations, and optional external
providers. POO Flow currently has enough handoff machinery to represent memory,
but not a dedicated memory object module.

Recommended future module:

- =src/modules/memory/objects.ss=
- =src/modules/memory/config.ss=
- object contracts for store spec, memory op, ranking strategy, commit receipt;
- runtime owner set to Marlin or an external memory runtime.

This should be staged after session objects, because memory attaches naturally
to session and agent objects.

** 10.06.28.07 Roadmap

*** 10.06.28.07.01 Milestone A: Report-Only Session Objects

Implement =src/modules/session/objects.ss= and =src/modules/session/config.ss=
as POO object modules. The first milestone should not execute runtime work.

Acceptance tests:

- object validation rejects invalid chunk rows and cyclic lineage;
- user-interface case builds a project workspace session by inheritance;
- presentation exposes session count, lineage graph, and placement summary;
- no runtime executed flag flips to true.

*** 10.06.28.07.02 Milestone B: Placement Projection

Connect session placement to existing sandbox profiles.

Acceptance tests:

- =nono=, =docker=, and =cube= placement refs resolve through profile objects;
- unresolved placement refs appear as diagnostics;
- projection emits Marlin handoff manifests;
- filesystem workspace constraints are validated before runtime handoff.

*** 10.06.28.07.03 Milestone C: Agent and Selector Objects

Add report-only agent/workflow/selector objects under the workflow category.
Do not make =agent= a top-level module name if it conflicts with category or
module-system naming. Prefer a workflow-scoped object namespace.

Acceptance tests:

- an agent object transforms one session intent into another receipt;
- selector object records candidate workflows and selected route as data;
- governance policy and sandbox placement are inherited through POO supers;
- tests prove no Scheme runtime execution occurs.

*** 10.06.28.07.04 Milestone D: Example Ladder

Mirror OpenRath's numbered examples as user-interface cases:

- =01-session-hello=
- =02-session-lineage=
- =03-sandbox-placement=
- =04-tool-intent=
- =05-custom-tool-object=
- =06-mcp-handoff=
- =07-streaming-receipt=
- =08-compressor-intent=
- =09-memory-intent=
- =10-provider-variation=
- =11-selector-routing=

Each case should be loaded through the existing module system, not by a separate
test-only macro.

** 10.06.28.08 Non-Goals

- Do not vendor or depend on OpenRath.
- Do not copy Python method chaining as the primary Scheme interface.
- Do not move runtime-heavy execution into Scheme.
- Do not introduce a separate module-system path for examples.
- Do not bypass object-validation or Gerbil harness policy to make examples
  pass.

** 10.06.28.09 Decision

Adopt OpenRath's session-first framing as a roadmap input.

The next POO Flow implementation slice should be a report-only session object
family that integrates with existing POO inheritance, sandbox profile
derivation, presentation traces, and Marlin handoff receipts. After that,
agent/selector/memory objects can be layered without disturbing the current
workflow and CI/CD implementation.
