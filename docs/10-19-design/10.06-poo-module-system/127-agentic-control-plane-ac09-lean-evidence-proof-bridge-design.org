#+title: AC-09 Lean / Evidence Proof Bridge Design
#+date: 2026-07-12
#+status: approved-design

* Status

Approved design. Implementation has not started.

This RFC is the formal AC-09 owner for the Agentic Control Plane roadmap. It
specializes the earlier research in =54-lean-c-ffi-proof-bridge-rfc.org= for
the AuthorizedEffectToken and evidence-root architecture delivered by AC-08.
It does not freeze ABI v1.

* Objective

Turn the AC-08 authorization and mediation evidence chain into a typed,
fail-closed Lean proof input without leaking Gerbil object layout, using JSON,
or making Python a proof kernel.

The first complete proof case is an =AuthorizedEffectToken= bound to:

- the Cedar policy decision and policy revision;
- the normalized effect and token digest;
- the semantic, execution, and optional batch roots;
- the runtime-v0 mediation outcome and durability profile;
- the required and present proof obligations;
- the nonce, epoch, and previous evidence root needed to reject replay.

* Ownership

| Owner | Responsibility |
|-------+----------------|
| Scheme POO control plane | Declare proof facts, obligation profiles, and evidence projections as POO-native objects |
| Cedar adapter | Produce and preserve the authorization decision; it does not generate Lean theorems |
| Runtime-v0 kernel | Validate mediation state and construct the canonical proof-case vector |
| C ABI | Transport the stable native representation and errors; it does not decide proof truth |
| Python runtime | Production-capable CFFI runtime, generator, and differential oracle |
| Lean package | Typed decode, theorem ownership, and proof-receipt construction |
| Downstream adapters | Adapt the public C ABI into Rust, TypeScript, or other runtime languages |

The =poo-flow= layer does not implement a second Rust or TypeScript runtime.
Downstream runtimes remain downstream-owned. Python remains a valid production
runtime and the complete retained validation runtime for this repository.

AXLE is an optional AC-09 CI qualification tool. It is not a production runtime
dependency. Local development requires only the ordinary Lean/Lake proof gates.

The architectural boundary is a generated, fixed-width Proof-Case Vector IR.
Lean never interprets a Scheme object. Scheme and Python never decide that a
Lean theorem has been proved.

* ProofCaseVector v1

** Representation

The canonical vector is versioned, deterministic, and native. Its logical
fields are:

| Group | Fields |
|-------+--------|
| Identity | ABI version, case kind, schema fingerprint |
| Authorization | token digest, policy revision, effect digest, subject/resource/action binding |
| Evidence | semantic root, execution root, optional batch root |
| Execution | mediation outcome, sequence, durability profile |
| Obligations | required mask, present mask, obligation count, obligation vector |
| Replay protection | nonce, epoch, previous evidence root |

The manifest fixes field order, width, tag values, endianness, alignment, and
reserved space. A single canonical manifest generates:

- the public C header and test vectors;
- Lean tags and typed decoding constants;
- Python CFFI constants and layout assertions;
- the schema fingerprint used by every consumer.

Generated artifacts have a freshness gate. Hand-edited language projections
are invalid.

** Digest identity derivation
Proof identity v1 uses =sha256= with two distinct, manifest-owned domains:
- =poo-flow.proof-case-vector.v1= hashes the domain, one zero separator byte,
  and the exact canonical 424-byte native vector;
- =poo-flow.authorized-effect-theorem-set.v1= hashes the domain, one zero
  separator byte, and the ordered newline-separated theorem names.

The digest algorithm, domains, and ordered theorem set are part of the
canonical manifest and generated into C, Lean, and Python projections. Lean
models SHA-256 as a trusted primitive, but =verifiedProofReceipt= applies that
primitive internally. Callers cannot supply =vectorDigest= or
=theoremSetDigest=. Executable runtimes discharge the trusted primitive
through native SHA-256 and cross-runtime differential gates.

** Rejected representations

- JSON is not part of the ABI or the Proof-Case Vector IR.
- Gerbil structs and GC-managed object layouts are not exposed.
- C-owned result allocations are not the normal return path.
- Per-field accessor calls are not the production hot path.
- The retired graph ABI is not reconstructed as a compatibility wrapper.

* C ABI

The production path is caller-owned and bulk-oriented:

#+begin_src c
poo_flow_status
poo_flow_proof_case_measure(
    poo_flow_proof_case_handle handle,
    poo_flow_proof_case_layout *out_layout);

poo_flow_status
poo_flow_proof_case_write(
    poo_flow_proof_case_handle handle,
    void *buffer,
    size_t capacity,
    size_t *written);
#+end_src

=measure= determines the exact layout and capacity. =write= performs one
validated bulk projection into caller-owned storage. A capacity failure returns
the required size without partially publishing a valid proof case.

A bounded scalar accessor surface remains for ABI smoke checks and diagnostics.
It is not used to reconstruct a vector field by field in the hot path.

The public error set distinguishes at least:

- success;
- invalid argument;
- buffer too small;
- stale or invalid handle;
- schema/ABI mismatch;
- malformed source evidence;
- unsupported obligation.

* First Proof Case

The first theorem set proves the complete AuthorizedEffectToken chain:

1. The token is bound to the declared policy revision.
2. The token effect digest equals the mediated effect digest.
3. The semantic root covers the normalized authorization inputs.
4. The execution root covers the actual mediation receipt.
5. Every required obligation appears in the present mask.
6. The nonce and epoch do not admit replay or reuse.
7. Diagnostic durability cannot authorize execution.
8. A proof receipt binds the vector digest, theorem-set digest, semantic root,
   execution root, and optional committed batch root.

An L3 result additionally requires verified mediation, a root-linked immutable
receipt, and verified inclusion. Signature presence is not signature
verification. A buffered item without a committed root-linked batch receipt
cannot exceed L1.

* Failure Model

Runtime, transport, and proof outcomes are different domains:

| Domain | Representative outcomes |
|--------+-------------------------|
| Runtime mediation | allow, deny, invalid-token |
| C ABI transport | success, buffer-too-small, stale-handle, schema-mismatch, invalid-argument |
| Lean proof | verified, refuted, unsupported-obligation, malformed-evidence |

The following invariants are mandatory:

- transport success does not imply proof verification;
- Cedar allow does not imply complete execution evidence;
- a refuted proof cannot be reduced to a warning and allowed to execute;
- an unsupported obligation fails closed;
- Diagnostic cannot generate executable authorization;
- partial vector writes cannot be interpreted as proof inputs.

* Delivery Slices

    S4 --> S6["S6 Negative, property, and performance gates"]

** S1 Canonical proof manifest

Define ProofCaseVector v1, the obligation registry, fixed widths, field order,
tags, reserved space, and schema fingerprint. Generate language projections and
freshness tests from one manifest.

** S2 Scheme POO projection

Add POO-native proof-case, obligation, and evidence projections. Raw alists,
record DSLs, ad hoc patch languages, and raw compute hooks are not the ordinary
public interface.

** S3 Kernel and bulk C ABI

Implement measure/write, caller-owned buffers, stable errors, layout checks,
pure-C conformance, fault tests, and an installed-consumer harness.

** S4 Lean typed proof

Decode the canonical vector, implement the AuthorizedEffectToken theorem set,
and emit a ProofReceipt bound to the vector and theorem set.

** S5 Python CFFI differential
The executable gate launches the Scheme encoder, compares its complete native
vector and SHA-256 with the manifest/Lean fixture, and then sends those same
bytes through the CFFI C ABI. A mismatch reports the two owners, first field,
and byte offset.
Use the Scheme caller-owned native encoder as the upstream vector owner, then
compare the C round trip, Python layout/status projection, and Lean canonical
fixture without introducing a JSON representation.

Consume the same bulk ABI from the production-capable Python runtime and compare
Scheme/kernel vectors at byte, field, and semantic levels.

** S6 Negative, property, and performance gates

Qualify replay, substitution, TOCTOU, omitted receipts, malformed vectors,
missing obligations, root mismatch, determinism, algorithmic complexity, and
the real bulk FFI path.

** S7 AC-09 qualification receipt

Aggregate Scheme, C, Python, Lean, installed-consumer, security, and performance
receipts. AC-09 completion still reports =abi-v1-frozen? #f=.

Each slice is committed only when it forms a complete validated feature. Files
are not committed merely because they were edited together.

* Qualification Matrix

** Schema and freshness

- Generated C, Lean, and Python projections share one schema fingerprint.
- Generated artifacts are reproducible and fresh.
- Reserved fields and unknown tags fail according to the declared version rule.

** Native conformance

- Pure-C consumers verify layout, alignment, endianness, sizing, zero-length,
  insufficient capacity, and invalid-handle behavior.
- The installed public header and library pass the same consumer harness.
- Fault and sanitizer gates run on the supported-host matrix.

** Lean

- Lake build and FFI smoke pass.
- Typed decoding rejects malformed or unsupported inputs.
- AuthorizedEffectToken positive theorems and all negative proof fixtures pass.
- ProofReceipt fields are derived from verified inputs, not copied assertions.

** Differential

- Scheme, kernel, Python, and Lean agree on shared digests, masks, roots, and
  vector bytes.
- Cedar decision evidence agrees with the semantic-root binding.
- Differential failures report the first divergent owner and field.

** Security

- Replay, fork, nonce reuse, token substitution, effect substitution, policy
  revision mismatch, root substitution, cross-runtime/session/lease/batch
  substitution, mutable-buffer TOCTOU, and receipt omission fail closed.

** Determinism

- Construction order does not change the semantic root.
- The same mediation sequence produces the same execution root and vector bytes.
- Batch inclusion and proof receipts are reproducible.

** Performance

- Per-item vector construction remains O(1).
- A k-item batch remains O(k).
- Root-history lookup remains O(log n) through an ordered/indexed route.
- The production bulk path does not cross FFI once per field.
- Caller-owned write does not allocate once per item.
- Strict and Batched use separate real-path gates.

Concrete time budgets are frozen only after a reproducible real-path baseline.
The RFC does not invent an unmeasured millisecond threshold.

** Regression

- AC-01 through AC-08 remain green.
- The production-capable Python runtime suite remains green.
- No old graph ABI compatibility surface reappears.

* Exit Criteria

1. One canonical proof manifest regenerates every language projection.
2. All generated-artifact freshness gates pass.
3. The public Scheme proof surface is POO-native.
4. The C ABI contains no JSON and exposes no Gerbil object layout.
5. The bulk path has no per-field FFI reconstruction or per-item allocation.
6. Lean build, FFI smoke, and the AuthorizedEffectToken theorem set pass.
7. Every security negative fixture fails closed.
8. Scheme, kernel, Python, and Lean agree on shared proof vectors.
9. ProofReceipt binds the vector, theorem set, semantic root, execution root,
   and any claimed batch root.
10. L3 is emitted only for a complete verified chain.
11. The installed pure-C consumer passes.
12. AC-01 through AC-08 regressions remain green.
13. Qualification reports =abi-v1-frozen? #f=.
14. No retired graph ABI compatibility wrapper exists.

* Non-Goals

- Final ABI v1 freeze; AC-10/11 own complete-chain qualification, canonical
  cutover, and final freeze.
- Implementing downstream Rust, TypeScript, or other language runtimes.
- Treating Python as a proof kernel.
- Proving arbitrary user code or external side effects.
- Replacing Cedar authorization with Lean theorem evaluation.
- Introducing a serialized document protocol into the native hot path.

* Delivery Boundary

The AC-09 implementation plan may change only:

- Scheme POO proof/evidence projections;
- canonical proof manifests and generated artifacts;
- runtime-v0 proof-case construction and C ABI accessors;
- the production Python CFFI proof projection;
- Lean typed decoders, theorems, receipts, and proof fixtures;
- pure-C/installed consumers, focused qualification tests, performance gates,
  owner/status documents, and receipts.

Unrelated CLI, build, CI, or downstream runtime work remains outside the AC-09
feature commits unless a focused qualification gate proves it is a direct owner.
