* 10.06.58 P0 Boundary Namespace Design
This document defines the P0 naming system for =poo-flow= boundaries and namespaces.

The goal is not cosmetic naming. Stable names are required because agents must know which layer they are allowed to edit, validators must know which layer owns rejection reasons, and runtime projection must never become the semantic owner.

** 10.06.58.10 Naming Thesis
Use three names for three different jobs:

1. User concept name. The word ordinary users see on a graph.
2. Boundary layer name. The architectural owner of meaning.
3. Gerbil namespace name. The concrete module or object-family namespace.

Do not collapse these names. A user may say "sandbox". The boundary layer may be =projection= or =observability=. The Gerbil namespace may be =poo-flow.sandbox.nono-sandbox=. Mixing those into one name makes agent repair unsafe.

The stable rule is:
#+begin_src text
User concepts explain the workflow.
Boundary layers own meaning.
Gerbil namespaces locate implementation.
POO object families carry extension behavior.
#+end_src

** 10.06.58.20 P0 Boundary Layers
These are the P0 boundary names. New documents and code should use these names consistently.

| Boundary | Owns | Does Not Own |
|----------+------+--------------|
| =intent= | natural-language or graph-shaped user request | Scheme expansion, validation, runtime execution |
| =author= | user and agent declarations such as =init.ss=, =use-module=, profiles, feature config | generated manifests, Lean facts, runtime execution |
| =graph= | user-visible agentic workflow graph: loop, step, model, module, gate, result path | POO method order, final ABI rows |
| =semantic= | Scheme/POO object graph, prototypes, C3 composition, functional workflow builders | external execution |
| =contract= | typed slots, structural validation, safety invariants, hard gates | graph rendering or runtime work |
| =observability= | events, spans, diagnostics, receipts, evidence provenance | semantic ownership or runtime execution |
| =projection= | manifest rows, graph previews, proof facts, doctor reports, runtime handoff payloads | source semantics |
| =runtime= | external execution boundary and handoff contract | Scheme-side policy authorship |

The strongest correction is =runtime=: in Scheme documents, =runtime= means the boundary that receives a validated projection. It does not mean Scheme executes heavy runtime work.

** 10.06.58.30 Layer Flow
#+begin_src mermaid
flowchart LR
  Intent["intent\nuser graph request"]
  Author["author\ninit.ss, use-module, profiles"]
  Graph["graph\nloop, step, model, module, gate"]
  Semantic["semantic\nScheme/POO + C3"]
  Contract["contract\nslot checks, hard gates"]
  Observability["observability\nevents, spans, diagnostics, receipts"]
  Projection["projection\nmanifest, proof facts, reports"]
  Runtime["runtime\nMarlin or external executor"]
  Intent --> Author --> Graph --> Semantic --> Contract --> Projection --> Runtime
  Contract --> Observability
  Observability --> Author
  Observability --> Graph
  Observability --> Projection
#+end_src

The repair loop is intentionally routed through =observability= back to =author= and =graph=. Agents should repair declarations or graph intent, not final projections.

** 10.06.58.40 Gerbil Namespace Families
Use the project brand prefix =poo-flow= everywhere. Do not introduce =funflow=, =poo*=, =popflow=, or generic =workflow= as top-level project namespaces.

P0 namespace families:

| Namespace Family | Purpose | Example |
|------------------+---------+---------|
| =poo-flow.intent.*= | graph intent helpers and guide-system input forms | =poo-flow.intent.graph= |
| =poo-flow.author.*= | authoring macros, =use-module= support, profile assembly | =poo-flow.author.init= |
| =poo-flow.graph.*= | graph data structures and algorithms | =poo-flow.graph.algorithms= |
| =poo-flow.semantic.*= | POO object graph, prototype composition, C3-facing helpers | =poo-flow.semantic.objects= |
| =poo-flow.contract.*= | slot contracts, structural validation, hard gates | =poo-flow.contract.resource= |
| =poo-flow.observability.*= | events, spans, diagnostics, receipts, evidence projections | =poo-flow.observability.receipt= |
| =poo-flow.projection.*= | manifest, report, proof, runtime handoff projections | =poo-flow.projection.manifest= |
| =poo-flow.runtime.*= | handoff adapters and runtime boundary contracts only | =poo-flow.runtime.marlin= |
| =poo-flow.<category>.<module>= | concrete user-selectable modules | =poo-flow.sandbox.nono-sandbox= |

The concrete module rule is important. User-facing modules are category-owned:
#+begin_src text
poo-flow.<category>.<module>
#+end_src

Examples:
- =poo-flow.workflow.cicd=
- =poo-flow.sandbox.nono-sandbox=
- =poo-flow.sandbox.cube-sandbox=
- =poo-flow.loop.engineering=
- =poo-flow.human.audit=

Avoid =poo-flow.modules.<category>.<module>= as the public namespace. =modules= may describe repository layout, but it should not leak into the public Gerbil namespace when the category already carries meaning.

Repository source folders should follow the canonical boundary layer when the implementation owns framework behavior:

| Source Folder | Public Import Role |
|---------------+--------------------|
| =src/intent/*= | graph intent helpers |
| =src/author/*= | authoring macros and profile assembly |
| =src/graph/*= | graph data structures and algorithms |
| =src/semantic/*= | POO object families and semantic descriptors |
| =src/contract/*= | typed validation and hard-gate receipts |
| =src/observability/*= | events, spans, diagnostics, receipts |
| =src/projection/*= | manifest, report, proof, and handoff projections |
| =src/runtime/*= | runtime handoff adapters only |

Do not create =src/boundary/*= as a framework owner. =boundary= is the subject of this design document, not a canonical implementation layer. For example, boundary namespace descriptors belong under =src/semantic/boundary-namespace.ss=, while their typed validation belongs under =src/contract/boundary-namespace.ss=.

** 10.06.58.50 Object Family Names
POO object families should use boundary-first names:

| Object Family | Meaning |
|---------------+---------|
| =intent/graph-request= | user or agent graph request before lowering |
| =author/module-selection= | selected category module and feature flags |
| =graph/workflow-graph= | graph-level workflow shape |
| =graph/step= | user-visible step node |
| =graph/gate= | user-visible gate node |
| =semantic/policy-object= | composed POO policy object |
| =semantic/profile= | POO profile composition object |
| =contract/validator= | typed validator object |
| =contract/hard-gate= | non-bypassable safety gate |
| =observability/event= | atomic evidence fact |
| =observability/span= | bounded operation with provenance |
| =observability/diagnostic= | validator-owned rejection or warning |
| =observability/receipt= | agent-facing feedback packet |
| =projection/manifest-row= | runtime handoff row |
| =runtime/handoff= | external executor boundary request |

Do not name a POO object after its final serialization shape. For example, a runtime manifest row is a projection of a validated object, so its object family should remain tied to =projection=, not =runtime= or a raw ABI name.

** 10.06.58.60 Agent Repair Namespace Rule
Every diagnostic and receipt must name the highest legal repair layer:

| Failure Kind | Repair Target Layer |
|--------------+---------------------|
| unclear user request | =intent= |
| wrong module or feature | =author= |
| missing step, loop exit, model, or gate | =graph= |
| invalid POO composition | =semantic= |
| permission widening, missing hard gate, bad slot type | =contract= |
| missing evidence, unknown validator owner, stale receipt | =observability= |
| malformed report, manifest, proof fact, or handoff payload | =projection= |
| external executor rejection after handoff | =runtime= |

Ordinary agent repair should stop at =author=, =graph=, =semantic=, or =contract=. It should not patch =projection= or =runtime= artifacts unless the task is explicitly boundary integration work.

** 10.06.58.70 Naming Anti-Patterns
Avoid these names in new public surfaces:

- =core= for business modules. Use the real boundary: =semantic=, =contract=, =graph=, or =observability=.
- =merge=. POO+C3 composition should name the actual operation: =compose=, =extend=, =linearize=, or =project=.
- =config= as a semantic owner. Use =author= for user declarations and =contract= for validation.
- =runtime= for Scheme-side construction. Use =projection= or =runtime/handoff=.
- =diagnostics= as a top-level framework. Use =observability=, with diagnostics as one projection.
- =modules= in public Gerbil namespaces when a category name is available.

** 10.06.58.80 Migration Policy
For P0 naming, do not preserve compatibility aliases in public docs or new code.

The accepted brand and namespace are:
#+begin_src text
poo-flow
poo-flow.<boundary>.*
poo-flow.<category>.<module>
#+end_src

Old names should be removed when touched:
- =funflow= as a project namespace;
- =poo*= or =pooFLow= spelling variants;
- =popflow=;
- =merge= as a layer name;
- public =poo-flow.modules.*= namespaces.

** 10.06.58.90 Acceptance Criteria
New documents, examples, and Scheme namespaces pass this naming check when:

- every user-facing concept maps to a boundary layer;
- every POO object family has a boundary-first name;
- every diagnostic names a repair target layer;
- every projection is described as a product, not an owner;
- =runtime= only means the external handoff boundary in Scheme-side docs;
- concrete user-selectable modules use =poo-flow.<category>.<module>=;
- observability owns events, spans, diagnostics, receipts, and evidence provenance.
