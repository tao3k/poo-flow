#+TITLE: AgentTeams Composition Replication Research
#+DATE: 2026-07-08

* Purpose

This note records the AgentTeams research lane after cloning
~agentscope-ai/AgentTeams~ into ~.data/AgentTeams~.

The question is not whether POO Flow should copy AgentTeams.  The useful
question is not whether POO Flow can be compatible with AgentTeams either.

AgentTeams is a capability-validation sample.  The question is whether a
community agent framework's core policy and flow mechanism can be expressed as a
Composition, reproduced from our control plane, then projected into a runtime.
For AgentTeams, that means checking whether its team/project/task mechanism can
be expressed as POO-native policy families plus functional strategy
combinators, then executed by our Python runtime and other runtime bindings.

The current evidence says yes.  AgentTeams has a strong operational shell, but
its core project/task workflow semantics are small enough to be reproduced by a
stable Composition surface and a Python runtime adapter.

* Kernel-First Extraction Principle

AgentTeams must not become an internal namespace that we copy.  That would be
the wrong lesson.

The long-term problem is that POO Flow will inspect hundreds or thousands of
agent projects.  Each project will have its own names: manager, worker, team,
crew, swarm, graph, node, run, thread, task, router, planner, executor, and many
others.  If every audit produces a new project-shaped module, the system becomes
a catalog of downstream frameworks instead of a control-plane kernel.

The correct extraction rule is:

- ignore surface names first;
- identify the invariant mechanism;
- express the mechanism as POO-native policy families;
- compose those families with functional strategy combinators;
- project the resulting kernel into runtime adapters and validation fixtures.

For AgentTeams, the surface names are ~Manager~, ~Worker~, ~Team~, ~Team
Leader~, and ~Human~.  The internal kernel is more general:

- actor substrate;
- role profile overlay;
- delegation topology;
- communication boundary;
- durable task lifecycle;
- graph readiness strategy;
- loop decision policy;
- artifact/result contract;
- runtime projection.

This means POO Flow should not grow an ~agentteams~ semantic core.  At most,
AgentTeams is an external validation projection: a way to prove that the kernel
can reproduce the behavior of this downstream framework.  The reusable owners
inside POO Flow should be names like:

- ~generic-worker/base~;
- ~role-profile~;
- ~role-topology-policy~;
- ~delegation-policy~;
- ~task-lifecycle-policy~;
- ~loop-policy~;
- ~durable-project-policy~;
- ~communication-boundary-policy~;
- ~runtime-projection~.

Project-specific labels belong in tests, fixtures, reports, or migration
projection layers.  They should not define the internal ontology.

** Relation To Prior Framework Audits

AutoGen, LangGraph, and CrewAI have already been studied as earlier downstream
samples.  They should be treated the same way as AgentTeams:

- AutoGen contributes lessons about conversational actor coordination,
  group-chat style routing, and multi-agent turn policy.
- LangGraph contributes lessons about graph state, checkpointing, durable
  execution, routers, and conditional edges.
- CrewAI contributes lessons about role/task/crew packaging and human-friendly
  task assignment surfaces.
- AgentTeams contributes lessons about declared organization topology,
  Manager/Leader/Worker boundaries, durable project/task flow, and loop policy.

These names are evidence labels, not ontology labels.  The reusable POO Flow
kernel should absorb the mechanisms behind them, then expose those mechanisms as
composition families.  We should not create one semantic namespace per audited
framework.

The accumulating cross-framework kernel is:

- generic actor substrate;
- role/profile overlay;
- graph and router strategy;
- conversation or channel policy;
- delegation topology;
- task lifecycle;
- durable checkpoint/project state;
- loop/replan/escalation policy;
- result and artifact contract;
- runtime projection.

** Codex Built-In Agent Role Lens

The local ~.data/codex~ checkout reinforces the same point from another angle.

ASP evidence found these relevant anchors:

- ~.data/codex/codex-rs/core/src/agent/role.rs~
  - owner for built-in role semantics;
  - lexical evidence for ~explorer~, ~awaiter~, and ~role~.
- ~.data/codex/codex-rs/core/src/agent/builtins/explorer.toml~
  - built-in role configuration surface.
- ~.data/codex/codex-rs/core/src/agent/builtins/awaiter.toml~
  - built-in role configuration surface.
- ~.data/codex/codex-rs/core/src/agent/agent_resolver.rs~
  - resolver anchor for mapping requested agents to role/config material.
- ~MultiAgentMode~
  - public protocol mode: ~none~, ~explicitRequestOnly~, ~proactive~.
- ~SubAgentActivityKind~
  - public activity lifecycle: ~started~, ~interacted~, ~interrupted~.

The evidence does not show a single public ontology enum named
~generic/default/explorer/worker~.  Instead, the stronger design reading is:

- there is a generic execution substrate;
- built-ins such as explorer/awaiter are role profiles over that substrate;
- the protocol exposes when multi-agent behavior is allowed;
- the protocol reports sub-agent activity as lifecycle events;
- the resolver binds a request to the concrete role/config material.

This is exactly the distinction POO Flow needs.  A Manager, Worker, Explorer,
Awaiter, Planner, or Reviewer should not be a separate hardcoded ontology root.
It should be:

- actor substrate;
- role profile;
- permission/communication policy;
- lifecycle policy;
- task/loop strategy;
- runtime projection.

So the AgentTeams ~Manager~ is best understood as a role capability overlay,
not a special species.  Likewise, ~Team Leader~ is a worker-like substrate with a
leader role profile and delegation-first policy.  This matches the Codex shape:
generic execution capacity becomes meaningful only after role instructions,
tool boundary, activity policy, and coordinator context are applied.

** Session Composition Kernel Gap Design

The current POO Flow gap is not that we lack another agent namespace.  The gap
is that session-as-flowing-value is not yet the obvious center of the user
surface.

OpenRath's angle makes the correction clear: one session flows through the
system, and Composition is the user-level way to transform, branch, delegate,
route, and project that session.  Agent roles are not a separate root.  They are
session-transform policy overlays.

The missing Composition kernel should be designed as five orthogonal policy
layers around the session:

1. ~session value substrate~
2. ~role transform profile~
3. ~multi-agent mode policy~
4. ~session activity lifecycle~
5. ~resolver projection~

These layers must be separate.  If they are collapsed into one object, every
downstream framework will pressure us to add its own names and special cases.

*** Layer 1: Session Value Substrate

The substrate is the session value that flows through Composition.  It should
not know whether the next transform will be called worker, executor, agent,
node, teammate, tool runner, or sandbox.

Its contract is:

- session identity;
- parent/child lineage;
- chunks and derived chunks;
- placement intent;
- role/transform bindings;
- visible context grants;
- artifact namespace;
- tool boundary;
- resource budget;
- receipt protocol.

This is the POO-native replacement for hardcoding "worker".  A worker is just
one session transform profile over the flowing session value.

*** Layer 2: Role Transform Profile

A role profile overlays behavior on a session transform.

Examples:

- explorer: read/search/inspect, no mutation by default;
- awaiter: wait/watch/report completion;
- worker: execute delegated task;
- manager: coordinate topology and delegate;
- team-leader: decompose, assign, aggregate;
- reviewer: validate result and request revision.

The profile should not own runtime execution or session storage.  It owns:

- instruction policy;
- allowed tools;
- delegation rights;
- communication permissions;
- default lifecycle hooks;
- result obligations;
- escalation behavior.

This makes a role cheap to define and easy to override:

#+begin_src scheme
(role-profile team-leader
  (:extends session-transform/generic-worker)
  (policy delegation-first)
  (can delegate-to team-worker)
  (can aggregate-results)
  (deny execute-domain-task-by-default)
  (activity lifecycle/leader))
#+end_src

*** Layer 3: Multi-Agent Mode Policy

Codex exposes multi-agent behavior as a mode.  POO Flow should generalize this
into a policy object rather than a global flag.

The minimal domain is:

- ~none~: do not spawn/delegate;
- ~explicit-request-only~: delegation only when the user or composition requests
  it;
- ~proactive~: policy may spawn/delegate when strategy conditions are met.

This policy should attach to a composition, session branch, stage, or transform
profile.

#+begin_src scheme
(multi-agent-mode explicit-request-only
  (allow-spawn? requested-by-composition)
  (allow-proactive? #f)
  (max-active 4))
#+end_src

For high-throughput runtimes, this can project into Python/AnyIO or Rust
scheduler limits.  The Scheme layer owns the policy; the runtime owns execution.

*** Layer 4: Activity Lifecycle

The lifecycle is the event vocabulary for session transformation.  Codex exposes
started, interacted, and interrupted for sub-agent activity.  POO Flow should
keep that minimal surface but make the session event stream the durable source.

Base lifecycle:

- created;
- started;
- interacted;
- yielded;
- completed;
- interrupted;
- failed;
- cancelled.

Task lifecycle overlay:

- assigned;
- acknowledged;
- in-progress;
- submitted;
- accepted;
- revision-needed;
- blocked.

Loop lifecycle overlay:

- iteration-started;
- iteration-ready;
- iteration-evaluating;
- replanning;
- waiting-user;
- stopped-success;
- stopped-blocked.

This should be a typed event stream, not a string log.  Human-readable history
can be rendered from events, but events are the durable source.

*** Layer 5: Resolver Projection

The resolver is the layer that turns a session composition request into a
concrete runtime handoff.

It answers:

- which session value is being transformed;
- which role transform profile overlays it;
- which mode policy governs delegation;
- which lifecycle vocabulary is active;
- which runtime binding executes it;
- which tool boundary and artifact namespace are assigned.

The resolver should be pure at the Scheme level.  It produces a normalized
session plan.  Runtime adapters consume the plan and materialize processes,
threads, workers, sub-agents, or C ABI calls.

#+begin_src scheme
(resolve-session-transform
  (session audit/root)
  (role team-leader)
  (mode explicit-request-only)
  (runtime python-anyio)
  (namespace project-audit))
#+end_src

*** Composition Shape

The user-facing surface should compose these layers without forcing users to
think in low-level POO object internals first.

OpenRath's useful UI lesson is concept compression.  A user starts from a
flowing session and applies agent/workflow layers; they do not manually assemble
every internal executor, memory, selector, and backend object at the call site.
POO Flow should be stricter than OpenRath internally, but the visible macro
surface should be at least as compact.

The old explicit style is intentionally rejected as too noisy.  It repeats the
internal module alias for every family and exposes the mechanism stack as if it
were the user's job to wire it by hand.  OpenRath's session-first lesson points
the other way: the user should see one compact concept, while the framework
expands it into profile/spec, capability checks, resolver output, and runtime
projection.

The normal surface should therefore use the existing ~use-module~ syntax and let
the module provide its kernel defaults implicitly:

#+begin_src scheme
(use-composition framework-audit
  (use-module funflow as ff)

  (session audit/root
    (transform coordinator
      (role manager)
      (mode explicit-request-only))

    (transform planner
      (role team-leader)
      (mode proactive)
      (delegates-to evidence-reader benchmark-runner))

    (transform evidence-reader
      (role explorer))

    (transform benchmark-runner
      (role worker))

    (routes
      (coordinator -> planner)
      (planner -> evidence-reader)
      (planner -> benchmark-runner))))
#+end_src

The FunFlow module should own its kernel profile internally.  Session value,
role transform, multi-agent mode, lifecycle, resolver, and runtime projection
are default inherited implementation families, not things the user should wire
by hand.  The user imports the module and writes the session composition.

The global kernel/factor/plugin/fixture boundary belongs to the module-system
design, not to this AgentTeams research note.  This section only relies on that
rule: FunFlow should own its kernel defaults internally, while plugins extend
declared slots from outside the kernel.

Plugin extension stays POO-native, but it is explicit and upward:

#+begin_src scheme
(use-composition framework-audit
  (use-module funflow as ff)
  (use-module audit-roles as audit
    (profiles audit-leader-role))

  (compose
    (profile audit audit-leader-role)))
#+end_src

This surface is still POO-native because profiles, session transforms, and
overrides lower into POO objects.  It is also functional because the resolver
produces a normalized session value that runtime adapters execute.

The plugin module owns the local POO object definition.  The composition only
selects that plugin profile.  This keeps the kernel hidden and stable while
still allowing upward extension.

*** Internal POO Object Families

The reusable owners should be:

- ~session-value/base~
- ~session-transform/base~
- ~role-transform-profile/base~
- ~role-topology-policy/base~
- ~multi-agent-mode-policy/base~
- ~activity-lifecycle/base~
- ~task-lifecycle/base~
- ~loop-lifecycle/base~
- ~resolver-projection/base~
- ~runtime-binding/base~
- ~artifact-namespace/base~

Framework labels such as AutoGen, LangGraph, CrewAI, AgentTeams, and Codex
should only appear in validation projections and fixtures.

*** Validation Targets

This design is complete only if it can reproduce the known samples without
creating project-specific ontology:

- Codex explorer/awaiter: session transform profiles with activity lifecycle
  and resolver projection.
- AgentTeams Manager/Team Leader/Worker: session role transforms plus
  delegation topology.
- LangGraph nodes/routers/checkpoints: session graph strategy plus durable
  lifecycle.
- CrewAI crew/role/task: role transform profile plus task lifecycle and
  assignment surface.
- AutoGen group chat: conversation/channel policy plus session turn strategy.

If a new framework requires a new profile, that is fine.  If it requires a new
ontology root, the kernel is missing an abstraction.

*** Implementation Direction

Implementation should proceed from stable contracts outward:

1. Define the normalized Scheme artifact for session value, transforms, roles,
   modes, lifecycle, resolver output, and runtime binding.
2. Add POO object families for session value, transform profile, mode,
   lifecycle, and resolver.
3. Add functional combinators for session routing, role topology, and
   delegation.
4. Project the normalized artifact into the Python runtime.
5. Add validation fixtures for Codex-style explorer/awaiter session transforms
   and AgentTeams-style Manager/Leader/Worker session flows.
6. Only then add runtime-specific accelerators in Python/Rust.

This fills the design gap without turning POO Flow into a copy of any one agent
framework.

* What AgentTeams Does Well

** Operational Shell

AgentTeams has a clear production story:

- Manager agent coordinates work.
- Worker agents execute delegated units.
- Matrix rooms provide human/agent collaboration and task isolation.
- Shared object storage provides artifact exchange.
- Gateway, Kubernetes, Helm, Docker Compose, and operator assets make the system
  deployable instead of just conceptual.

This is valuable because many agent frameworks stop at a local graph runner.
AgentTeams treats team execution as a distributed operational system.

This should not be overstated.  Kubernetes, Helm, nginx, object storage, and
gateway deployment are standard engineering pieces.  They are useful packaging,
not the semantic advantage.  The part worth absorbing is the declarative shape
around Manager, Worker, Team, Human, runtime, channel, artifact, and gateway
resources.

** Runtime Tool Boundary

The TeamHarness MCP server is organized around runtime tools:

- ~health~: environment and liveness.
- ~message~: external replies and Matrix routing.
- ~roomflow~: task-room lifecycle.
- ~filesync~: artifact synchronization.
- ~projectflow~: durable project planning and readiness.
- ~taskflow~: delegate, acknowledge, submit, and check task work.

This split is good.  It separates durable project semantics from transport,
file exchange, and room creation.  It is close to the boundary we want in POO
Flow: Scheme owns the policy/composition surface, while Python owns runtime I/O
adapters.

** Durable State Is Inspectable

The CoPaw task kernel uses an inspectable filesystem layout under shared project
and task directories.  Project metadata, task metadata, plans, specs, and
results are persisted as ordinary files.

This gives AgentTeams a useful property: users and operators can inspect or
repair the runtime without needing a private database client.

** Explicit Role And Transition Guards

AgentTeams does not treat every actor as equivalent.  It distinguishes manager,
worker, remote member, room identity, and actor identity in its taskflow
operations.

The test evidence covers important guards:

- delegation requires a task room;
- acknowledgement requires the expected worker;
- submission validates actor and task result shape;
- pending or idle runtime states are surfaced by active-task diagnostics.

This is a strong design choice because team execution failure is usually a
coordination failure, not only a graph scheduling failure.

** DAG And Loop Coverage

AgentTeams covers both:

- DAG work, where a pending node becomes ready after dependencies complete;
- loop work, where iteration tasks are produced and iteration history is
  recorded until the loop stops, blocks, or waits for user input.

The loop side matters for agent systems because many useful workflows are not a
single static DAG.

* Deployment Surface Analysis

AgentTeams' deployment story has two layers.

The outer layer is replaceable infrastructure:

- Kubernetes or a local embedded controller;
- Helm values or installer flags;
- Higress or another gateway;
- Tuwunel/Synapse-compatible Matrix service;
- MinIO/S3/OSS-compatible object storage;
- Element Web behind nginx;
- manager and worker containers.

This layer is operationally useful, but it is not a deep framework advantage.
POO Flow should not treat Kubernetes as the interesting part.  We can project
the same deployment contract into Kubernetes, nginx, a Scheme service, a local
supervisor, or another runtime-owned deployment lane.

The inner layer is the important one: AgentTeams makes deployment resources
declarative.

ASP evidence from ~docs/architecture.md~ and Helm/CRD searches shows these
resource families:

- ~Worker~: model, runtime, image, skills, MCP servers, exposed ports, channel
  policy, desired state, access entries.
- ~Manager~: model, runtime, image, soul/agent overrides, skills, MCP servers,
  heartbeat, worker idle timeout, notify channel, desired state, access
  entries.
- ~Team~: leader, workers, admin, peer mentions, channel policy, rooms, member
  readiness.
- ~Human~: display name, email, permission level, accessible teams/workers,
  Matrix user, initial credential state.
- Helm values: matrix, gateway, storage, controller, manager, worker,
  elementWeb, credentialProvider, runtime, default images, managed/existing
  service switches.

This is exactly the layer that JSON Schema to Scheme contracts can absorb.  The
deployment resource schema should become a POO Flow contract projection:

- schema fields become Scheme contract fields;
- enum/status fields become policy domains;
- nested resource specs become POO profiles;
- Kubernetes CRDs become one backend projection, not the owner;
- nginx/Scheme/local supervisor projections become alternative deployment
  backends.

The deployment conclusion is therefore:

- do not copy the Kubernetes implementation as architecture;
- extract the declarative deployment contract;
- express it as POO-native resource profiles;
- project it to any deployment backend the runtime needs.

* Agent Team Kernel

AgentTeams is most interesting when read as an agent-organization kernel, not as
a Kubernetes package or MCP demo.

ASP evidence from ~docs/declarative-resource-management.md~ and
~docs/k8s-native-agent-orch.md~ shows a clear organization model:

- ~Manager~ is the top-level coordinator.  It owns worker/team/human management,
  project delegation, and global resource policy.
- ~Team~ is a collaboration unit.  It has one leader and one or more workers.
- ~Team Leader~ is still a worker runtime class, but with a different role
  package: leader soul, leader skills, project-management, task-management, and
  delegation-first behavior.
- ~Worker~ is the execution unit.  A worker maps to a runtime instance, identity,
  communication account, storage namespace, and gateway consumer token.
- ~Human~ is a real principal with permission level and accessible teams/workers.

The important rule is that a role is not just a label.  A role is a bundle of:

- identity;
- runtime image/class;
- skills;
- communication permissions;
- storage namespace;
- lifecycle state;
- delegation rights;
- task/result obligations.

This is the part POO Flow should learn from.  A useful Agent Team is not created
by starting multiple agents.  It is created by declaring a topology of roles and
then projecting that topology into runtime identities, communication channels,
artifact namespaces, and task policies.

** Manager, Leader, Worker Boundary

AgentTeams' best role rule is the Manager/Team boundary:

- Manager delegates to the Team Leader.
- Team Leader decomposes work and assigns tasks to team workers.
- Team Room excludes the Manager.
- Manager does not directly coordinate in-team workers.

This is a good policy boundary.  It avoids turning the Manager into a global
bottleneck and gives the Team its own local planning loop.

For POO Flow, this should become a role-topology policy:

- ~manager -> team-leader~ is allowed for team-level delegation.
- ~team-leader -> team-worker~ is allowed for in-team task assignment.
- ~manager -> team-worker~ is denied by default.
- humans enter rooms based on permission policy, not ad hoc invitation.
- channel policy extends or restricts default communication edges.

The key design lesson is that communication permissions are part of the agent
kernel.  They should not be treated as chat configuration.

** Team Creation As Reconcile

AgentTeams creates a team by reconciling a declared desired state:

- resolve the declared leader and worker specs;
- create runtime identities;
- generate role-specific prompts/config;
- assign skills;
- create communication rooms;
- create storage namespaces;
- register room and worker metadata;
- start or update runtime instances.

This is not inherently Kubernetes-specific.  Kubernetes is one reconcile
backend.  A Scheme service, Python runtime supervisor, local process supervisor,
or nginx-fronted service can implement the same projection.

For POO Flow, the owner should be:

- Scheme contract: validate the team topology;
- POO resource profile: define defaults and overrides;
- Composition: connect role policies and flow policies;
- runtime projection: create the concrete processes, identities, channels, and
  namespaces.

** Role Families For POO Flow

The role model should be expressed as reusable POO families:

#+begin_src scheme
(use-composition research-team
  (use-module funflow as ff
    (profiles agent-team durable-projects))

  (compose
    (profile ff agent-team))

  (team audit
    (manager coordinator)
    (leader planner
      (policy delegation-first))
    (workers
      (worker evidence-reader)
      (worker benchmark-runner))
    (communication
      (allow coordinator -> planner)
      (allow planner -> evidence-reader)
      (allow planner -> benchmark-runner)
      (deny coordinator -> evidence-reader)
      (deny coordinator -> benchmark-runner))))
#+end_src

This is the capability-validation target: express the AgentTeams kernel as a
Composition, then project it into Python, Rust, or another runtime.

* Loop Policy Lessons

AgentTeams' loop policy is implemented in the CoPaw task kernel.  The important
owner is ~copaw/src/copaw_worker/task.py~.

The loop model contains:

- ~goal~;
- ~stop_condition~;
- ~iteration_template~;
- ~max_iterations~;
- ~current_iteration~;
- ~status~;
- current iteration tasks;
- history.

Loop readiness is not a plain while-loop.  Each iteration contains DAG-shaped
tasks.  ~ready_loop_nodes~ returns pending current-iteration tasks whose
dependencies are already completed.  This is a useful hybrid:

- outer loop: Leader decision policy;
- inner graph: dependency-ready task scheduling.

~record_loop_iteration~ is the Leader decision point.  It records one iteration
decision and maps it into loop state:

- ~continue~ keeps the loop running;
- ~replan~ keeps the loop running but signals graph/template revision;
- ~ask_user~ moves the loop to waiting for user input;
- ~stop_success~ completes the loop;
- ~stop_blocked~ blocks the loop.

That gives us three things worth preserving:

1. Loop progress is explicit and durable.
2. Stop conditions are named policy, not hidden control flow.
3. Human escalation is a first-class loop decision.

** Where Their Loop Policy Is Still Weak

The implementation is practical, but not policy-native enough:

- statuses and decisions are string domains;
- history is textual instead of typed event history;
- iteration template is stored but not a first-class strategy;
- replan is a decision value, but there is no reusable replan policy family;
- result acceptance is separate from loop advancement;
- stop condition is declarative text, not an executable/verifiable strategy;
- Leader evaluation is outside the formal policy model.

POO Flow can improve this by making Loop Policy an object family:

- iteration generator;
- readiness strategy;
- result acceptance strategy;
- stop-condition evaluator;
- replan strategy;
- human-escalation strategy;
- history/event recorder;
- repair strategy for blocked or idle iterations.

The AgentTeams lesson is not their exact implementation.  The lesson is the
shape: a durable leader-driven loop with a DAG inside each iteration.

** POO Flow Loop Policy Shape

A POO-native loop policy should look like a composition of strategies:

#+begin_src scheme
(loop-policy audit-loop
  (goal "produce verified framework audit")
  (stop-when evidence-complete-and-reviewed)
  (max-iterations 5)
  (iteration
    (generate-tasks from-open-gaps)
    (ready-by dag-dependencies)
    (accept-by effective-result-with-artifacts)
    (on-replan regenerate-from-findings)
    (on-user-needed request-human-decision)
    (on-blocked repair-or-escalate)))
#+end_src

This is stronger than AgentTeams because the loop behavior is not buried in a
runtime helper.  It is a reusable policy family that can be composed with role
topology, durable project state, and runtime projection.

* Where AgentTeams Is Weaker Than POO Flow

** Semantics Are Not Lifted Into Policy Families

The project/task semantics are implemented as runtime functions over files and
strings.  They are effective, but they are not first-class policy families.

In POO Flow, the same semantics should be represented as reusable objects:

- project lifecycle policy;
- task delegation policy;
- worker acknowledgement policy;
- result validation policy;
- DAG readiness strategy;
- loop iteration strategy;
- active-task repair strategy.

Once these are POO-native, users can override or extend them without rewriting a
runtime tool.

** Strategy Composition Is Shallow

AgentTeams has correct DAG readiness and loop readiness primitives, but the
composition layer is not the main abstraction.  Scheduling is mostly encoded by
the runtime helper functions.

POO Flow can make this stronger by using functional strategy combinators:

- ~dag-ready-by-dependencies~
- ~loop-ready-by-iteration~
- ~role-guarded-transition~
- ~accept-effective-result~
- ~detect-idle-worker~
- ~detect-pending-review~
- ~repair-or-escalate~

The important difference is that these combinators become normal composition
values, not hidden imperative branches in one runtime file.

** Runtime Tool Logic Risks Drift

The evidence shows a CoPaw projectflow tool and a TeamHarness MCP server that
both carry project/task runtime behavior.  Even if this is intentional for
packaging, it creates drift risk.

POO Flow should avoid that split.  The normalized Composition should own the
state machine once.  Runtime surfaces should project that state machine into
Python functions, C ABI calls, MCP tools, or other adapters.

** Types And Contracts Are Operational, Not Semantic

AgentTeams uses practical schemas, files, and tests.  That is enough for a
working system, but it is not the same as a typed policy contract.

POO Flow can put the contract one level higher:

- Composition validates the profile shape.
- POO objects hold reusable defaults and overrides.
- Runtime receives a normalized plan and stable ABI-friendly operations.
- Python tests verify projection behavior without becoming the source of truth
  for the policy semantics.

* Replication Thesis

AgentTeams can be reproduced by POO Flow through:

- a POO-native AgentTeams profile family;
- functional DAG and loop strategy combinators;
- a Python runtime adapter for MCP, Matrix, filesync, and execution;
- durable policy backends for project/task state;
- tests that mirror AgentTeams' behavior while preserving POO Flow's stronger
  composition model.

The core should not be implemented as a Python-only agent framework.  Python is
the runtime and connector layer.  Scheme Composition is the control plane.

* Replication Strategy Options

There are three plausible ways to reproduce AgentTeams.

** Option A: Python-Only Reimplementation

This route would implement projectflow, taskflow, roomflow, filesync, and MCP
projection directly in the Python runtime package.

It would be fast to start, but it gives up the main advantage of POO Flow.  The
state machine, transition guards, and DAG/loop policies would become Python
framework code.  Users would extend it with Python classes and callbacks instead
of reusable POO policy objects.

This option is useful only as a temporary capability spike.

** Option B: Scheme Composition Kernel, Python Runtime Adapter

This route declares the AgentTeams semantics as POO-native profile families and
functional strategy combinators.  Python loads the normalized artifact and
executes runtime operations through AnyIO, MCP, Matrix, and storage adapters.

This is the preferred route because it keeps the policy source of truth in the
control plane while still letting Python be a high-performance runtime boundary.

It also keeps the same story for Rust or another runtime language: the runtime
implements adapters and ABI-facing operations; it does not re-invent the policy
engine.

** Option C: AgentTeams Capability-Validation Projection

This route emulates the AgentTeams tool names and file layout first, then maps
them inward to POO Flow.

It is useful for validation because it lets us replay AgentTeams-shaped
project/task fixtures and prove that POO Flow can reproduce the same core flow
mechanism.  It should not become the internal architecture.  The AgentTeams
shape is a projection used for proof, not the semantic owner.

** Decision

Use Option B as the architecture, with Option C as an external
capability-validation projection for tests and migration.

Option A should be avoided except for short-lived experiments because it would
turn Python into another workflow policy engine.

* Mapping To POO Flow

| AgentTeams concept | POO Flow owner | Runtime owner |
|--------------------+----------------+---------------|
| projectflow tool | project lifecycle policy family | Python callable / MCP tool |
| taskflow tool | task transition policy family | Python callable / MCP tool |
| FileSystemTaskStore | durable backend profile | Python storage adapter |
| DAG plan | functional graph strategy | Python execution data |
| ready_nodes | DAG readiness combinator | fast runtime query |
| loop plan | loop policy family | Python iteration state |
| roomflow | collaboration-room profile | Matrix adapter |
| message | external-message profile | Matrix / gateway adapter |
| filesync | artifact-sync profile | storage adapter |
| check_active_tasks | observability and repair strategy | runtime probe |

This mapping is the key reason the project is a good downstream validation
case.  AgentTeams' runtime shell is broad, but its semantic core maps cleanly
onto POO Flow's intended abstractions.

* Proposed Composition Surface

The user-facing surface should stay POO-native first, but ordinary users should
not have to start from raw object internals.

A compact AgentTeams-style composition should look like:

#+begin_src scheme
(use-composition repo-ci
  (use-module funflow as ff
    (profiles github-ci agentteams))

  (compose
    (profile ff github-ci)
    (profile ff agentteams))

  (stage pull-request
    (step build
      (run "gxpkg" "build" "-g"))
    (step test
      (run "uv" "run" "pytest" "-q"))

    (edges
      (build -> test))))
#+end_src

For an AgentTeams-style project:

#+begin_src scheme
(use-composition research-team
  (use-module funflow as ff
    (profiles role-topology durable-projects loop-policy artifact-contract))

  (compose
    (profile ff role-topology)
    (profile ff durable-projects)
    (profile ff loop-policy))

  (project framework-audit
    (stage plan
      (step map-core
        (delegate worker-a))
      (step verify-runtime
        (delegate worker-b))
      (step report
        (delegate manager))

      (edges
        (map-core -> verify-runtime)
        (verify-runtime -> report)))))
#+end_src

The advanced native POO override remains available:

#+begin_src scheme
(def strict-task-result
  (.o (:extends task-result/base)
      accepted-status: '(SUCCESS SUCCESS_WITH_NOTES)
      require-room-id?: #t
      require-artifact-under-task-dir?: #t
      low-information-ack-policy: 'block))

(use-composition research-team
  (use-module funflow as ff
    (profiles role-topology durable-projects task-lifecycle))

  (compose
    (profile ff role-topology)
    (profile ff durable-projects)
    (profile ff task-lifecycle)
    (profile strict-task-result)))
#+end_src

The important rule is that ~use-module~ imports profiles, ~compose~ combines
profiles, and graph constructs such as ~stage~, ~step~, ~edges~, and routers are
owned by FunFlow instead of being repeatedly exposed as profiles.

* Minimal Behavior To Reproduce

The first reproduction target should cover the behavior AgentTeams already
tests:

- create a project;
- plan a DAG;
- compute ready nodes;
- delegate a task into a room;
- acknowledge the task as the expected worker;
- submit a result;
- check and accept effective results;
- unblock dependent DAG nodes after completion;
- plan a loop;
- compute current loop-ready tasks;
- record loop iteration history;
- pause, resume, and complete a project;
- report active-task diagnostics:
  - idle delegated task;
  - submitted result pending review;
  - invalid task result;
  - ready tasks waiting for delegation;
  - loop iteration waiting for decision;
  - project ready for completion.

This is enough to prove that POO Flow can reproduce the AgentTeams kernel
without Matrix, Kubernetes, or AgentScope being required for the semantic core.

* Runtime Design

Python should expose a stable runtime package, not a pile of example scripts.

The runtime package should provide:

- an AnyIO execution boundary;
- a CFFI or C ABI bridge where Scheme artifacts need native calls;
- a durable backend interface;
- a filesystem durable backend for parity with AgentTeams;
- optional Matrix/filesync adapters;
- an MCP projection layer;
- benchmark and compatibility tests.

Scheme should provide:

- profile declarations;
- normalized Composition artifacts;
- POO object families for project/task/result/room/artifact policy;
- functional combinators for DAG, loop, router, and repair strategies;
- AOT-generated artifacts for users who should not install a Gerbil compiler.

The runtime should not invent another policy language.  It should load the
normalized artifact, execute projected operations, and report receipts.

* Why This Is Dimensional Reduction

AgentTeams has a wide operational surface:

- team chat;
- room lifecycle;
- file sync;
- distributed workers;
- task delegation;
- durable project state;
- project diagnostics;
- deployment packaging.

But the semantic center is smaller:

- a project state machine;
- a task state machine;
- a DAG readiness strategy;
- a loop iteration strategy;
- role and room guards;
- result validation;
- repair diagnostics.

POO Flow is built exactly for this second layer.  Once these become POO-native
policy families and functional strategy combinators, AgentTeams-style behavior
is configuration plus runtime adapters.

That is the dimensional reduction:

- AgentTeams must implement the workflow semantics inside its runtime tools.
- POO Flow should declare those semantics once and project them into any
  runtime.
- Python, Rust, or other runtimes only need high-performance bindings and I/O
  adapters.

* Where We Still Need To Improve

This research also exposes gaps in POO Flow:

- The AgentTeams profile family is not yet modeled as a first-class module.
- The Python runtime package still needs a clean project/task API that mirrors
  normalized Composition instead of ad hoc fixtures.
- The durable policy in Scheme needs fuller project/task/result coverage.
- AOT artifact packaging must be part of the release lane so runtime users do
  not need to install Gerbil.
- Benchmarks must cover compile time, artifact load time, DAG readiness, loop
  iteration, task transition, and end-to-end runtime projection.
- The FunFlow user interface must keep graph primitives built in and expose
  domain profiles such as ~github-ci~ or ~agentteams~, not internal graph
  machinery.

* Validation Plan

Validation should happen in this order:

1. Recreate AgentTeams' projectflow/taskflow behavior in Python runtime tests.
2. Move the behavior source of truth to Scheme Composition artifacts.
3. Load the Scheme-generated artifact from Python.
4. Keep AgentTeams-style filesystem durability as one backend.
5. Add an MCP projection that exposes the same high-level tool families.
6. Add Matrix/filesync adapters only after the kernel is stable.
7. Benchmark against:
   - direct Python runtime calls;
   - AOT artifact load;
   - DAG readiness on large graphs;
   - loop iteration recording;
   - task transition throughput;
   - compile and test performance gates.

The benchmark target is not merely to match AgentTeams.  The target is to show
that POO Flow can reproduce the kernel and then compose stronger policies with
less duplicated runtime code.

* Research Conclusion

AgentTeams is a strong downstream validation case.  Its production shell is
serious, and its task/team abstractions are practical.  But its policy semantics
are still embedded in runtime-oriented Python tools and file protocols.

POO Flow should absorb the useful shape:

- durable project/task kernel;
- role-aware task transitions;
- DAG plus loop execution;
- runtime tool projection;
- human/agent collaboration adapters.

Then POO Flow should improve the layer where AgentTeams is weakest:

- policy reuse;
- native object extension;
- functional composition;
- typed normalized artifacts;
- ABI-friendly runtime projection;
- AOT distribution.

If this line is implemented correctly, downstream users should not write a new
AgentTeams-style workflow engine.  They should compose a profile, override the
policy objects they care about, and let the Python runtime execute the projected
artifact.
