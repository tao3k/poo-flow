#+title: Loop Engine Graph Proof
#+property: header-args :mkdirp yes

* Purpose

This proof note is the core direction after the LangChain/LangGraph profile
exercise. LangChain and LangGraph are useful as small user-interface examples,
but they are not the center of the formal model. The proof target is POO Flow's
own loop engine and graph: when a policy starts, which scopes are allowed, which
dependencies are ready, which guarded edge can advance, and why a loop step must
consume budget or terminate.

* Start Contract

The start contract is not a JSON manifest. It is a typed relation between
profile scope, session scope, sandbox scope, tool scope, dependency readiness,
and loop exit definition.

#+begin_src typst
$
"ScopeSubset"(A, B) = forall x,  x in A => x in B

"Tool" subset.eq "Sandbox" and "Sandbox" subset.eq "Session"
=> "Tool" subset.eq "Session"

"StartSound" =
"DepsReady" and "ExitDefined"
and "Profile" subset.eq "Session"
and "Tool" subset.eq "Session"
$
#+end_src

In Lean this appears as `LoopStartContract`: it carries the scope subset proofs
as fields, and `start_sound_from_contract` derives the start soundness fact used
by downstream gates.

* Guarded Graph

The graph proof is not a framework proof for LangGraph. It is the POO Flow
invariant that a runnable edge must be guarded and must advance in the ranked
control graph.

#+begin_src typst
$
"GuardedRankProgress"(e) =
"Guard"(e) and "rank"("from"(e)) < "rank"("to"(e))

"GuardedRankProgress"(e) => "from"(e) != "to"(e)
$
#+end_src

This is intentionally conservative. Real loop graphs can contain repeated
runtime iterations, but a single guarded transition in the control graph cannot
pretend to be progress when it is a self-loop without an explicit loop policy.

* Loop Fuel

Runtime loops need a separate progress argument. A transition is valid progress
only when its guard is satisfied and its fuel decreases.

#+begin_src typst
$
"LoopFuelProgress"(t) =
"GuardSatisfied"(t) and "fuelAfter"(t) < "fuelBefore"(t)

not "LoopFuelProgress"("fuelBefore" = 0)
$
#+end_src

This proof surface is the bridge between Scheme policy objects and Lean: Scheme
constructs the policy/profile/facts, while Lean states the mathematical contract
that the facts must satisfy before a loop step is considered sound.
