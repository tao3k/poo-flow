* 10.06.36 Session User Interface Ladder
** 10.06.36.10 Purpose
This note records the first OpenRath-inspired session tutorial slice that is implemented in POO Flow.
The goal is not runtime execution. The goal is a compact user-facing Scheme shape that builds POO session objects, resolves sandbox placement against the existing profile catalog, and presents graph and handoff receipts.
** 10.06.36.20 Delivered Shape
Users can now write:
#+begin_src scheme
(def custom-session-root
  (session custom/session-root
    (chunk request user
           "Run the POO Flow package checks and keep receipts.")
    (lineage root)
    (placement agent/nono)
    (metadata (source . user-interface)
              (case . openrath-session-first))))
(def custom-session-branch
  (session custom/session-branch
    (chunk branch-request user
           "Fork the package-check session for sandbox placement review.")
    (lineage fork custom/session-root)
    (placement agent/nono)
    (metadata (source . user-interface)
              (case . openrath-session-first)
              (branch . sandbox-placement))))
(def custom-session-presentation
  (session-graph custom-session-root custom-session-branch))
#+end_src
This expands to the existing POO object layer:
- =poo-flow-session-chunk= for receipt rows;
- =poo-flow-session-lineage= as a mixin prototype;
- =poo-flow-session-default-placement= for report-only sandbox catalog resolution;
- =poo-flow-session-value= for C3 composition of lineage and placement;
- =pooFlowSessionGraphPresentation= for graph receipts.
** 10.06.36.30 Boundary
The new syntax is intentionally small:
1. It does not execute sandbox runtime work.
2. It does not build Marlin runtime handlers.
3. It does not introduce a second session object model.
4. It does not hide placement diagnostics.
5. It keeps chunks and diagnostics as cheap receipt rows, not nested POO child objects.
** 10.06.36.40 Verification
The focused acceptance surface is:
#+begin_src sh
gxpkg env gxc src/modules/session/config.ss t/session-object-test.ss
gxpkg env gxc user-interface/custom/my-module/cases/session.ss user-interface/custom/my-module/config.ss
gxpkg env gxtest t/session-object-test.ss
gxpkg env gxtest t/unit-tests.ss
#+end_src
** 10.06.36.50 Session Transform Rung
The transform slice now has a concrete user-facing shape:
- input: one POO session value;
- transform: a functional flow arrow or workflow step;
- memory: POO memory intents attached to the transform;
- output: a derived session value plus transform and memory receipt metadata;
- runtime boundary: report-only handoff remains explicit.
Users can declare the memory side as an object and attach it to a transform:
#+begin_src scheme
(def custom-session-review-memory
  (session-memory-intent custom/review-memory
    (store session/memory)
    (scope project-workspace)
    (recall repository-summary review-notes)
    (commit commit-derived-session)))
(def custom-session-review-transform
  (session-transform custom-review-agent
    (intent review)
    (description "Review a custom session and derive a follow-up session.")
    (capabilities +provider-handoff +receipt-only +session-derivation)
    (memory-intents custom-session-review-memory)))
#+end_src
This still expands to POO objects and report-only receipts. It does not add an alist workflow DSL or a live memory backend call.
** 10.06.36.60 Next Ladder Rung
The next ladder rung should focus on lineage export and selector-style routing:
- present root, fork, detach, and merge-style session rows;
- keep export rows as receipts until the runtime bridge is explicit;
- model selectors as strategy objects over session receipts, not live provider calls.
