#+title: RFC 0147 - Adaptive Scheme Build Guard and Execution Windows
#+author: POO Flow
#+status: Accepted and Implemented; Full Cold Gate Blocked
#+language: en

* Decision

POO Flow keeps =std/make= as the only Scheme build executor and extends the
Building Framework with an optional POO-native adaptive execution-window
controller.  The controller does not create another scheduler.  It submits one
contiguous, dependency-order-preserving window to one =std/make= coordinator,
observes a Scheme-owned process-tree memory receipt, derives an immutable next
controller, and then submits the next window.

The existing topology projection remains the default.  When no adaptive
controller is supplied and =POO_FLOW_BUILD_MAX_RSS_BYTES= is not a positive
integer, the runtime and user-interface stages retain the established one-window
topology behavior.  Adaptive execution is activated only by an explicit POO
controller or by a positive =POO_FLOW_BUILD_MAX_RSS_BYTES= value.

The adaptive cold gate requires an explicit hard RSS cap.  A machine-derived
outer project-guard fallback is useful protection, but it is not authorization
for an adaptive full cold run.

* Invariants

1. =std/make= remains the only native executor.
2. At most one =std/make= coordinator session is active at a time.
3. Windows are contiguous prefixes of the remaining topology-ordered specs.
4. No spec is reordered, dropped, or duplicated.
5. There is no fixed global spec or chunk cap.
6. Window changes are derived from named POO observations.
7. Any invalid observation, non-completed outcome, cap mismatch, or cap breach
   fails closed.
8. One spec that cannot fit the hard cap fails closed rather than being retried
   without a bound.
9. Clean never receives an adaptive plan.
10. Portable receipts are canonical JSON, not text logs or printed alists.

The bounded comparisons use the same owners, source sets, and host.  The
optimized lane merges unnecessary Building Framework barriers while retaining
the single upstream executor.

* Cold-build truth and current gate

An empty output root produces 39 dependency-closed topology groups containing
376 specs.  The first 8 groups alone contain 223 specs.  The bounded baseline
for those first groups reached approximately 104.046505 seconds and was
terminated.  No complete cold receipt was produced, so a full cold build
remains blocked.

** Historical standardized baseline and regression interrupt

The adaptive work does not erase the earlier build-system evidence.  The
discarded per-subset child-=gxi= scheduler completed in 2,919,108 ms, about
48.7 minutes.  That design duplicated Building Framework planning and paid
process startup repeatedly, so it remains rejected evidence rather than a
performance baseline.

After the project was converted to one canonical Building Framework Gerbil
make project, the same host completed a cold build in 571.71 seconds, about
9.53 minutes, with a 2,617,507,840-byte peak process-tree RSS receipt.  A later
cold attempt crossed twelve minutes while still compiling and was interrupted;
its temporary progress stream contained 237 compile entries, but it produced
no accepted final timing receipt.  The twelve-minute observation is therefore
a regression stop, not a replacement baseline.

These historical receipts define the performance rule for the next authorized
cold run: it must use the canonical =build.ss= / Building Framework path, must
retain Scheme-owned guard receipts, and must not silently accept a result slower
than the 571.71-second host baseline.  The adaptive controller addresses the
memory risk exposed by larger sessions; it does not waive the elapsed-time
regression gate.

The evidence supports two separate conclusions:

- Barrier merging is real and useful, but it is a secondary gain.
- The dominant cost remains expansion and import serialization followed by a
  delayed flush of native compilation jobs.

The observed core increase from 1.733 to 1.989 is too small to indicate broad
phase overlap.  Merging barriers shortens coordinator overhead, while the main
pipeline still waits too long before native work becomes available.  A single
large session also raises memory pressure.  The local controller therefore
bounds memory and preserves order; it does not claim to solve upstream phase
serialization.

* Landed architecture

** Building Framework owner

The harness implementation is owned by these files:

- =src/building/std-builder.ss= defines the POO controller and observation
  contracts, adaptive plan and result objects, sequential window runner, and
  package-stage integration.
- =src/building/observability.ss= owns topology and adaptive canonical JSON.
- =t/std-builder-execution-window-test.ss= validates fake =make-proc=
  execution, order, observations, empty plans, and one-spec cap failure.
- =t/building-observability-test.ss= validates canonical adaptive JSON.

The principal public APIs are:

- =execution-window-controller?=
- =execution-window-controller-observe-run!=
- =execution-window-controller-next-state=
- =make-execution-window-observation=
- =make-adaptive-execution-window-plan=
- =std-builder-run-adaptive-plan!=
- =adaptive-execution-window-result-execution-windows=
- =adaptive-execution-window-result-window-observations=
- =build-adaptive-execution-windows->json-object=
- =build-adaptive-execution-windows->json-string=

The observation is a POO object with named slots for =result=, =outcome=,
=baseline-rss-bytes=, =peak-rss-bytes=, =max-rss-bytes=, and =elapsed-ms=.
=.next-state= receives the complete observation object and the actual spec
count.  Loose multiple values are not the protocol.

** POO Flow adapter and project activation

The downstream implementation is owned by these files:

- =src/build-api/process-memory-guard.ss= samples current-process-tree RSS,
  creates POO guard receipts, and emits canonical JSON.
- =src/build-api/adaptive-execution-window.ss= owns the immutable POO Flow
  controller constructor, live guard adapter, and pure sizing transition.
- =src/cli-support/project-build.ss= resolves optional caller or environment
  activation for build truth while preserving non-adaptive clean behavior.
- =t/build-api-adaptive-execution-window-test.ss= validates immutable sizing,
  fail-closed outcomes, guard JSON, stage activation, and zero headroom.

The principal POO Flow APIs are:

- =make-poo-flow-adaptive-execution-window-controller=
- =poo-flow-adaptive-execution-window-controller?=
- =poo-flow-adaptive-execution-window-controller-next-state/with-current-rss=
- =poo-flow-process-memory-guard-receipt->json-object=
- =poo-flow-process-memory-guard-receipt->json-string=
- =poo-flow-project-adaptive-cold-gate-available?=
- the optional controller arguments on =poo-flow-project-source-stages=,
  =poo-flow-project-build-requests=, =poo-flow-project-build-spec=, and
  =poo-flow-project-compile!=.

=poo-flow-project-build-requests=, =poo-flow-project-build-spec=, and
=poo-flow-project-compile!= resolve the same caller or environment controller,
so inspection and execution report the same build truth.  In contrast,
=poo-flow-project-clean!= calls =poo-flow-project-source-stages= with no
controller.  Clean therefore retains plain topology specs even when an RSS
environment variable exists in the caller.

* Adaptive sizing policy

The POO controller stores:

- worker count;
- hard maximum RSS bytes;
- caller headroom bytes;
- current window size;
- baseline RSS bytes;
- last estimated bytes per spec;
- window index;
- named =.observe-run!= and =.next-state= methods.

The initial window size is the worker count.  The caller supplies the hard RSS
cap.  =POO_FLOW_BUILD_RSS_HEADROOM_BYTES= is optional and nonnegative; its
default is exactly 0.  There is no percentage-derived default and no fixed
global spec cap.

The constructor fails early unless:

#+begin_example
hard-max-rss-bytes > baseline-rss-bytes + headroom-bytes
#+end_example

Each observation wraps the upstream thunk with the current-process Scheme
guard under =dynamic-wind=.  RSS is the complete process tree rooted at the
current =gxi=, including native compiler descendants.

After a completed receipt, the controller computes:

#+begin_example
observed-bytes-per-spec =
  max(1, ceil(max(0, peak-rss - start-baseline-rss) / actual-spec-count))

estimated-bytes-per-spec =
  max(previous-estimate, observed-bytes-per-spec)

available-bytes =
  hard-cap - current-process-tree-rss - caller-headroom

measured-capacity =
  max(1, floor(max(0, available-bytes) / estimated-bytes-per-spec))

next-window-size =
  max(1, min(measured-capacity, current-window-size + worker-count))
#+end_example

Growth is therefore limited to one worker-count quantum per successful
window.  Shrinkage is immediate.  The conservative maximum of the previous and
current estimates prevents an optimistic low-growth sample from erasing a
known higher memory cost.

Only =completed= and =ok= outcomes are accepted.  Invalid receipt shape,
non-integer measurements, hard-cap mismatch, guard failure, or peak RSS above
the controller cap raises an error before another window is attempted.

* Single-executor control flow

#+begin_src mermaid
flowchart TD
  C["POO immutable controller"] --> P["Building Framework adaptive plan"]
  P --> W["Next contiguous dependency-ordered window"]
  W --> M["One std/make coordinator session"]
  M --> G["Scheme process-tree guard and POO observation"]
  G --> Q{"Outcome completed and peak within hard cap?"}
  Q -->|"no"| F["Fail closed; do not attempt another window"]
  Q -->|"yes"| N["Derive immutable next controller"]
  N --> R{"Specs remain?"}
  R -->|"yes"| W
  R -->|"no"| J["Canonical adaptive JSON result"]
  X["No second executor and no concurrent coordinator"] -. invariant .-> M
#+end_src

The loop is a Building Framework extension around its existing effect
boundary.  It does not call =gsc= directly, create a per-spec =gxi= scheduler,
or overlap multiple =std/make= coordinators.

* Canonical receipt contracts

The established topology schema remains unchanged:

- =gslph.build-topology-execution-windows.v1=

It describes the default topology projection, including group count, upstream
session count, eliminated barriers, spec count, and dependency-order
preservation.

Adaptive execution uses a distinct schema:

- =gslph.build-adaptive-execution-windows.v1=

It projects only windows that were actually attempted and completed into the
result.  It records actual window specs and per-window spec counts, dependency
order preservation, worker count, hard RSS cap, and each observation's outcome,
baseline RSS, peak RSS, maximum RSS, and elapsed milliseconds.

The Scheme guard uses:

- =poo-flow.process-memory-guard.v1=

Its public JSON contains stable =kind= and =schema= strings, numeric =version:
1=, outcome, exit fields, process-tree peak and maximum RSS, elapsed
milliseconds, optional timeout, and a JSON-safe label.

All three contracts use Gerbil's native JSON support with sorted keys for the
canonical string.  A =.txt= progress stream is diagnostic only and is not a
receipt.  Printed alists remain internal or compatibility projections and are
not the public record format.

* Acceptance and full-cold gate

The landed code and focused receipts close the bounded implementation phase.
They do not authorize a full cold build.

A full cold run MUST NOT start until both conditions hold:

1. the caller supplies an explicit POO controller or a positive
   =POO_FLOW_BUILD_MAX_RSS_BYTES= hard cap; and
2. bounded receipts on the same host demonstrate valid canonical JSON,
   completed outcomes, preserved dependency order, and peak RSS below the cap.

If either condition is absent, =poo-flow-project-adaptive-cold-gate-available?=
must report the unavailable state and the project retains its default
one-window topology behavior.  The first authorized cold run must preserve all
completed per-window receipts and stop at the first failed boundary.

Phase-level overlap remains an upstream proposal.  The next performance path
is to change the upstream expansion/import and native-job flush behavior so
native compilation can overlap earlier with Scheme expansion.  That work must
remain inside the canonical =std/make= coordinator; it must not introduce a
second executor in POO Flow.

* Rejected alternatives

** Always one unbounded session

Rejected for adaptive cold execution.  It improves bounded wall time but the
runtime A/B increases allocation by 29.733% and RSS by 6.197%.

** Fixed chunk sizes or a global spec ceiling

Rejected.  They encode one host's behavior and cannot explain changes from
receipts.

** Concurrent std/make coordinators

Rejected.  They duplicate scheduling ownership and make the hard RSS cap more
difficult to enforce.

** Per-spec child gxi scheduling

Rejected.  Prior process multiplicity imposed dominant startup overhead and
duplicated Building Framework currentness, topology, and receipt ownership.

** Python-owned build policy

Rejected.  The Scheme build and its safety policy are Gerbil-owned.  Python may
remain a validation runtime below other interfaces, but it is not the owner of
this build controller.
