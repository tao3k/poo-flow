#+title: RFC 0147 - Adaptive Scheme Build Guard and Execution Windows
#+author: POO Flow
#+status: Accepted, Implemented, and Full-Cold Validated
#+language: en

* Decision

POO Flow keeps =std/make= as the only Scheme build executor and extends the
Building Framework with an optional POO-native adaptive execution-window
controller.  The controller does not create another scheduler.  It submits one
contiguous, dependency-order-preserving window to one =std/make= coordinator,
observes a Scheme-owned process-tree memory receipt, derives an immutable next
controller, and then submits the next window.

The existing topology projection remains the default.  When no adaptive
controller is supplied and =POO_FLOW_BUILD_MAX_RSS_BYTES= is not a positive
integer, the runtime and user-interface stages retain the established one-window
topology behavior.  Adaptive execution is activated only by an explicit POO
controller or by a positive =POO_FLOW_BUILD_MAX_RSS_BYTES= value.

The adaptive cold gate requires an explicit hard RSS cap.  A machine-derived
outer project-guard fallback is useful protection, but it is not authorization
for an adaptive full cold run.

* Invariants

1. =std/make= remains the only native executor.
2. At most one =std/make= coordinator session is active at a time.
3. Windows are contiguous prefixes of the remaining topology-ordered specs.
4. No spec is reordered, dropped, or duplicated.
5. There is no fixed global spec or chunk cap.
6. Window changes are derived from named POO observations.
7. Safe growth is geometric and capacity-capped, so coordinator-session count
   is logarithmic in the remaining spec count; native concurrency remains
   bounded independently by the detected worker count.
8. Any invalid observation, non-completed outcome, cap mismatch, or cap breach
   fails closed.
9. One spec that cannot fit the hard cap fails closed rather than being retried
   without a bound.
10. Clean never receives an adaptive plan.
11. Portable receipts are canonical JSON, not text logs or printed alists.
12. Runnable-process saturation is observable advisory evidence; only a memory
    safety-floor failure blocks admission before the guarded build starts.

** CI cache generations and pressure admission

GitHub's cache store is immutable, while =setup-bazel= derives its disk-cache
key from BUILD-file content.  A Scheme source change can therefore create a new
Bazel action result without changing the surrounding GitHub cache key.  An
exact restore then suppresses the post-job save and strands that new action
result on the ephemeral runner.

CI now materializes the pull-request merge commit as an ignored synthetic
=.ci/bazel-cache-generation/BUILD.bazel= before =setup-bazel=.  Each commit gets
a new exact generation, but the stable runner/toolchain prefix restores the
latest preceding disk cache.  Bazel reuses every valid CAS/action entry, runs
only actual misses, and the post step publishes the merged cache under the new
generation.  Reruns of the same commit remain exact hits.  This is a GitHub
cache-store adapter only; it does not enter =build.ss= or Building Framework.

CPU and memory pressure have different safety semantics.  The Scheme guard
records an instantaneous runnable-process saturation observation as an
=admission-advisories= value, but it does not treat that noisy system-wide
sample as a hard failure or override the detected/configured worker capability.
Exact physical memory is observed by Scheme only when the action executes; it
does not enter the Bazel action key because it changes scheduling and guard
limits, not the semantic compiled tree.  The process-tree RSS cap remains active
throughout compilation.  Admission still fails closed when available memory
cannot cover caller headroom plus the minimum guarded RSS floor.  A cached
Scheme receipt describes the producer execution of that cached artifact, while
the CI phase receipt describes the current cache consumer.

** Registered Gerbil toolchain convergence

The Scheme Bazel surface now requests a Gerbil capability through Bazel's
native toolchain resolution instead of binding project rules to labels inside
=@local_gerbil=.  =MODULE.bazel= registers one host-derived toolchain;
=local_gerbil_repository= emits a private implementation plus a public
=registered_toolchain= target constrained to the detected Darwin or Linux
execution platform; and both project compile and project test rules consume the
resolved =ToolchainInfo= through =ctx.toolchains=.

This separation makes the BUILD surface declarative: =scheme/BUILD.bazel=
declares source sets, output projections, and test roots, while the selected
execution platform supplies =gxi=, =gxc=, =gxpkg=, =gxtest=, native tools,
compiled external dependencies, physical memory, runfiles, and the canonical
host receipt.  No Scheme project rule names =@local_gerbil= or depends on its
internal target layout.

The current registered toolchain remains intentionally host-derived and local.
It normalizes an installed Gerbil and the canonical compiled dependency view;
it does not claim remote-execution hermeticity.  A future Nix or archive-backed
repository may register another implementation of the same toolchain type
without changing project BUILD declarations.  It must not replace =build.ss=:
the native Scheme project and Building Framework remain the only owners of
topology, scheduling, compilation policy, guards, and receipts.

** Runtime-C Gerbil capability convergence

*** Native ABI identity owns native cache compatibility

**** Bootstrap-safe dependency view

The registered Gerbil toolchain must remain analyzable before project
dependencies exist.  Otherwise toolchain resolution can block the
=//scheme:deps= target that is responsible for creating those dependencies.
The local repository therefore observes the =.gerbil/lib/clan= and
=.gerbil/lib/gslph= paths as Bazel repository inputs.  A missing path produces
an explicit empty dependency view instead of an analysis failure.  When the
dependency installer creates or changes either tree, Bazel invalidates and
re-evaluates the repository, then projects the compiled libraries into the
registered toolchain.

This is a dependency-state transition, not a compatibility fallback.  Scheme
compile and test actions still receive the complete dependency files as direct
Bazel inputs, and the toolchain receipt records the state of every projected
dependency.

**** Upstream-parallel toolchain bootstrap

The Gerbil source bootstrap must use the upstream build scheduler.  CI must not
reproduce the upstream dependency graph as a shell loop over individual
=build.sh= stages.  The Bazel-owned bootstrap capability invokes =make= with
the host's available online processor count and exports the same value through
=GERBIL_BUILD_CORES=.  Both the main acceptance workflow and the Python runtime
workflow consume this capability before the registered Gerbil toolchain exists.

The bootstrap does not overwrite the upstream-generated Gerbil version module.
The registered toolchain accepts both the release identity and the short
identity generated by the pinned source commit.  The JSON bootstrap receipt
records that observed Gerbil version together with the pinned source identity,
compiler, available build parallelism, elapsed time, installation prefix, and outcome.
This metric is separate from the POO Flow project cold-build baseline: a slow
toolchain bootstrap cannot be reported as a regression in the project compile,
and it cannot be hidden by that project's faster result.

**** Architecture profiles preserve performance and cache locality

The Gerbil toolchain declares an architecture profile instead of assuming that
every execution surface has the same cache-locality requirements.  The default
=native= profile retains the upstream highest-performance behavior and keys
artifacts by the compiler-selected CPU target.  The =portable= profile uses the
upstream-supported empty =--enable-march== value and keys artifacts by the host
OS, architecture, compiler, and profile without binding them to one transient
runner CPU model.

CI selects =portable= because its install cache must be reusable across
heterogeneous hosted runners.  Local and downstream production builds remain
=native= unless they explicitly request another profile.  Both choices flow
through the same Bazel-owned capability probe, repository receipt, and compile
action identity.

Gerbil and Gambit are configured with =--enable-march=native=.  A cache key
containing only the operating system, architecture, and Gerbil revision is
therefore unsound: two hosted runners can share those labels while selecting
different native instruction sets.  Reusing that installation or any compiled
Gerbil dependency can terminate a later code path with =SIGILL= (exit 132).

The native ABI identity is a Bazel toolchain capability, not a CI policy.  The
single probe at =tools/bazel/native_abi_fingerprint.sh= hashes the host system,
architecture, compiler version, and the compiler-selected =-march=native=
target.  =host_system.bzl= resolves or validates that identity and the local
Gerbil repository projects it into =native_abi.txt=, the toolchain receipt, and
every normalized Gerbil tool runfiles closure.  Consequently the fingerprint
participates in Bazel action identity even when a consumer only requests =gxi=,
=gxc=, =gxpkg=, or =gxtest=.

GitHub Actions remains a bootstrap consumer because its Gerbil installation
cache is restored before the Gerbil Bazel toolchain exists.  CI invokes the
same probe once, exports the result as =GERBIL_NATIVE_ABI=, and includes it in
the Gerbil installation, compiled dependency, Bazel disk, and ccache keys.  A
broad restore prefix must never cross a native ABI boundary.

** Declarative Just command surface

The repository maintains one root =justfile= as the human and CI command

** Historical standardized baseline and regression interrupt

The adaptive work does not erase the earlier build-system evidence.  The
discarded per-subset child-=gxi= scheduler completed in 2,919,108 ms, about
48.7 minutes.  That design duplicated Building Framework planning and paid
process startup repeatedly, so it remains rejected evidence rather than a
performance baseline.

After the project was converted to one canonical Building Framework Gerbil
make project, the same host completed a cold build in 571.71 seconds, about
9.53 minutes, with a 2,617,507,840-byte peak process-tree RSS receipt.  A later
cold attempt crossed twelve minutes while still compiling and was interrupted;
its temporary progress stream contained 237 compile entries, but it produced
no accepted final timing receipt.  The twelve-minute observation is therefore
a regression stop, not a replacement baseline.

These historical receipts define the continuing regression rule: cold builds
must use the canonical =build.ss= / Building Framework path, retain
Scheme-owned JSON guard receipts, and must not silently accept a result slower
than the 571.71-second historical baseline.  The new 202.87-second receipt is
the current host reference, while the old baseline remains the fail-closed
upper regression boundary until a separately approved policy changes it.

The evidence supports two separate conclusions:

- Barrier merging is real and useful, but it is a secondary gain.
- The dominant cost remains expansion and import serialization followed by a
  delayed flush of native compilation jobs.

The observed core increase from 1.733 to 1.989 is too small to indicate broad
phase overlap.  Merging barriers shortens coordinator overhead, while the main
pipeline still defers most native work until expansion has populated the
pending queue.  The 3.25% streaming experiment confirms that ad hoc polling is
not the missing order-of-magnitude optimization.  A single large session also
raises memory pressure.  The local controller therefore bounds memory and
preserves order; it does not claim to solve upstream phase serialization.

** 2026-07-16 host-pressure root cause and calibrated receipt

The 583-second interrupted run was not an acceptable cold-build result.  The
host had thirteen orphaned =devenv direnv-export= processes, a load average
above 160, 78 runnable processes, and no idle CPU.  The build guard still
started because its computed RSS ceiling was not connected to the default
adaptive execution-window controller and it had no host-pressure admission
decision.

The repaired Scheme control plane now makes admission and execution one
policy:

- logical CPU count and instantaneous runnable-process count determine the
  execution window;
- runnable count at or below logical CPU capacity preserves the configured
  full-speed worker count;
- pressure between one and two runnable processes per logical CPU shrinks the
  worker count by the excess over logical capacity;
- pressure above twice logical CPU capacity fails fast instead of converting
  host contention into a misleading build regression;
- physical memory, currently available memory, and an adaptive RSS headroom
  determine the effective RSS ceiling;
- insufficient memory headroom fails fast before =gxi= starts;
- the derived worker and RSS values are installed only across the explicit
  Building Framework handoff with =dynamic-wind= and are restored afterward;
- the native Scheme JSON receipt records every admission input and decision.

* Adaptive sizing policy

The POO controller stores:

- worker count;
- hard maximum RSS bytes;
- caller headroom bytes;
- current window size;
- baseline RSS bytes;
- last estimated bytes per spec;
- window index;
- named =.observe-run!= and =.next-state= methods.

The initial window size is the worker count.  The caller supplies the hard RSS
cap.  =POO_FLOW_BUILD_RSS_HEADROOM_BYTES= is optional and nonnegative; its
default is exactly 0.  There is no percentage-derived default and no fixed
global spec cap.

The constructor fails early unless:

#+begin_example
hard-max-rss-bytes > baseline-rss-bytes + headroom-bytes
#+end_example

Each observation wraps the upstream thunk with the current-process Scheme
guard under =dynamic-wind=.  RSS is the complete process tree rooted at the
current =gxi=, including native compiler descendants.

After a completed receipt, the controller computes:

#+begin_example
observed-bytes-per-spec =
  max(1, ceil(max(0, peak-rss - start-baseline-rss) / actual-spec-count))

estimated-bytes-per-spec =
  observed-bytes-per-spec

available-bytes =
  hard-cap - current-process-tree-rss - caller-headroom

measured-capacity =
  max(1, floor(max(0, available-bytes) / estimated-bytes-per-spec))

next-window-size =
  max(1,
      min(measured-capacity,
          max(current-window-size + worker-count,
              2 * current-window-size)))
#+end_example

Small windows therefore recover by at least one worker-count quantum, while
larger safe windows grow geometrically.  The measured capacity can shrink the
next window immediately.  Geometric window growth changes only the ordered
queue horizon visible to the single =std/make= coordinator; actual native
compiler concurrency remains bounded by =worker-count=.  The next decision
deliberately uses the latest completed window's observed bytes per spec rather
than a monotonic maximum.  This avoids treating one-time compiler residency
from an earlier window as permanent marginal per-spec cost, while the hard cap
remains the safety authority.  The immutable next state retains
=last-estimated-bytes-per-spec= as receipt evidence and history only; it is not
an input that can pin later windows to an older estimate.  Every window still
passes through the same pre-launch guard, and any missing, malformed,
non-completed, or cap-breaching receipt fails closed.

Only =completed= and =ok= outcomes are accepted.  Invalid receipt shape,
non-integer measurements, hard-cap mismatch, guard failure, or peak RSS above
the controller cap raises an error before another window is attempted.

* Rejected alternatives

** Always one unbounded session

Rejected for adaptive cold execution.  It improves bounded wall time but the
runtime A/B increases allocation by 29.733% and RSS by 6.197%.

** Fixed chunk sizes or a global spec ceiling

Rejected.  They encode one host's behavior and cannot explain changes from
receipts.

** Concurrent std/make coordinators

Rejected.  They duplicate scheduling ownership and make the hard RSS cap more
difficult to enforce.

** Per-spec child gxi scheduling

Rejected.  Prior process multiplicity imposed dominant startup overhead and
duplicated Building Framework currentness, topology, and receipt ownership.

** Python-owned build policy

Rejected.  The Scheme build and its safety policy are Gerbil-owned.  Python may
remain a validation runtime below other interfaces, but it is not the owner of
this build controller.

* Stable Gerbil project receipt v1 boundary

POO Flow pins gerbil-bazel revision
=ee0486d7f0845632967e31230c434ac6f68cac24=.  That revision defines
=gerbil-bazel.project-receipt.v1= as the stable outer envelope for every
=gerbil_project_compile= action.  Prefix-derived producer evidence does not
replace the envelope; POO Flow's =poo-flow.project-compile-guard.v1= receipt is
nested under =buildReceipt=.  Optional Gerbil-Bazel process-guard evidence is
nested independently under =resourceGuard=.

The same revision removes the former Bazel-owned package identity/revision
graph.  =gerbil.pkg= owns package metadata, =gxpkg= owns package lifecycle, and
=build.ss= with =std/make= owns compile membership, ordering, and Scheme-level
incrementality.  Bazel owns repository pins, declared inputs, dependency tree
artifacts, sandboxing, adaptive host capabilities, caching, guards, and
receipts.  The developer-only dependency target invokes one standard
=gxpkg deps --install= command; it is not build truth inside a compile action.

On Darwin, the registered toolchain leaves the installed =gambuild-C=
immutable.  =GERBIL_GSC= uses Gambit's public =-cc= and =-ld-options=
interfaces: native objects use the installed runtime's producer compiler,
dynamic modules receive the Darwin unresolved-symbol option, and =-exe= uses
the Xcode Clang driver through Gerbil's public =GERBIL_GCC= boundary.  Linux
retains the upstream host compiler and linker path without Darwin policy.
