#+TITLE: RFC 136 — Gerbil Module And POO Flow Feature Layering
#+STATUS: Active
#+DATE: 2026-07-13
#+OPTIONS: toc:3 num:nil

* Decision

POO Flow distinguishes Gerbil language modules from selectable product
features.  A Gerbil module is a static namespace, compilation, macro-phase,
import/export, and single-instantiation boundary.  A POO Flow =Feature= is a
POO-native declarative capability contribution selected by a profile.  Importing
a Gerbil module makes exported definitions available; it never enables a
Feature.

The public product vocabulary is =FeatureDescriptor=, =FeatureSelection=,
=FeatureProfile=, =FeatureActivationPlan=, and =FeatureInstance=.  The public
=use-module= vocabulary is retired rather than compatibility-wrapped.  Gerbil's
own =module=, =import=, and =export= vocabulary remains unchanged and is never
reimplemented by POO Flow.

* Four-layer architecture

#+begin_src mermaid
flowchart TB
    GM["GerbilModule: namespace, phase, import/export, AOT, single instantiation"]
    FD["FeatureDescriptor: POO-native static capability contribution"]
    FS["FeatureSelection and FeatureProfile: user intent, flags, dependencies"]
    AP["FeatureActivationPlan and FeatureInstance: explicit runtime handoff"]
    CC["CaseComponent"]
    DC["Closed DomainCase"]
    RT["Runtime Adapter"]

    GM -->|"exports runtime POO values and compiled macros"| FD
    FS -->|"selects and resolves"| FD
    FD -->|"contributes"| CC
    CC -->|"closes object, type, contract, policy, strategy, projection"| DC
    FS -->|"lowers to pure plan"| AP
    AP -->|"explicit effects only"| RT
    DC -->|"shared prototype"| AP
#+end_src

The four layers have different identities and different graphs.  None may
borrow another layer's edge semantics.

| Layer | Durable identity | Graph semantics | Mutation boundary |
|-------+------------------+-----------------+-------------------|
| GerbilModule | module path plus build provenance | code import/export DAG | module initialization only |
| FeatureDescriptor | feature id plus schema version | dependency, conflict, option schema | immutable |
| FeatureProfile | profile id plus canonical selections | selection and resolution graph | immutable |
| FeatureActivationPlan | plan key plus receipt schema | ordered runtime handoff | explicit adapter boundary |
| CaseComponent / DomainCase | component ids and canonical case key | POO C3 plus independent type/contract algebras | bounded cache and instance counters |

* Non-equivalence laws

1. =import FeatureOwner= does not select the Feature exported by
   =FeatureOwner=.
2. Selecting a Feature never mutates or re-instantiates its Gerbil module.
3. Feature dependency edges are not POO supers and never participate in C3.
4. Gerbil import edges are code availability edges and are not Feature
   dependency declarations.
5. A Feature conflict is a profile-resolution diagnostic, not a module loader
   failure.
6. A Gerbil module may export zero, one, or many FeatureDescriptors and
   CaseComponents.
7. Multiple FeatureProfiles may coexist in one process even though their owner
   Gerbil modules are single-instantiated.
8. Per-Agent, per-session, and per-organization mutable state never lives in a
   module-owned FeatureDescriptor.
9. Dynamic loading is an explicit adapter operation.  Macro expansion and
   =use-feature= never hide a loader call.

* Doom Emacs concepts translated without copying its runtime model

| Doom concept | POO Flow concept | Important difference |
|--------------+------------------+----------------------|
| module category | Feature namespace/category | semantic grouping only |
| module | FeatureDescriptor | immutable POO value, not a Gerbil module |
| module flag | FeatureOption selection | typed and schema-versioned |
| init.el selection | FeatureProfile | pure POO-native declaration |
| package declaration | build/package contribution | resolved by canonical build layer |
| init/config hooks | FeatureActivationPlan steps | effects occur only through runtime handoff |
| autoload/dynamic load | explicit plugin loader adapter | never implicit in selection macros |

Doom's directory and lifecycle conventions are design input for the product
feature layer.  They do not redefine Gerbil module instantiation or phase
semantics.

* Gerbil module capabilities POO Flow should exploit

** Single instantiation

A Gerbil module owns immutable FeatureDescriptors, CaseComponents, shared POO
prototypes, compiled option schemas, and optionally preclosed DomainCases.
Single instantiation ensures these values are constructed once per program
instance and reused across =N= Agent instances and multiple FeatureProfiles.

Single instantiation MUST NOT be used as a mutable global feature registry.
Only named bounded caches and observability counters may mutate at this layer.

** Compiled macros and the meta-syntactic tower

Static =defpoo-feature= and =defpoo-feature-profile= declarations are compiled
macros.  They validate identifiers, duplicate selections, option schemas,
dependency cycles, conflicts, component ownership, and projection selection at
expansion time.  They lower to ordinary POO constructor calls, immutable
constants, and functional resolver values.

Macros remain thin.  Graph traversal, canonicalization, diagnostics, and
runtime construction live in ordinary functions that can be tested without
macro expansion.

** Phase separation

Compile-time manifests contain syntax-safe immutable identifiers and schema
data.  Runtime POO objects are separate bindings.  A macro MUST NOT import a
runtime registry into its expansion phase, instantiate runtime adapters, or
capture mutable module state.

#+begin_src mermaid
flowchart LR
    SRC["Gerbil source module"] --> EXP["syntax-case expansion"]
    EXP --> CM["Compile-time Feature manifest"]
    EXP --> RV["Runtime POO FeatureDescriptor binding"]
    CM --> CHECK["Dependency, conflict, option and ownership validation"]
    CHECK --> PLAN["Compiled FeatureProfile plan"]
    PLAN --> REF["Direct runtime binding references"]
    RV --> REF
#+end_src

** Explicit import and export ownership

Gerbil =import= states which definitions are available to compile.  Gerbil
=export= states which macro and runtime bindings form an owner module's code
surface.  POO Flow uses this for provenance and static linking, not activation.

Every static FeatureDescriptor records an explicit semantic =feature-id=,
=schema-version=, and =owner-module-id=.  Module paths and pointer identity may
accelerate in-process lookup but are not durable protocol identity.

** Ahead-of-time compilation

The static fast lane resolves a FeatureProfile during expansion or build.  The
generated runtime program contains direct references to FeatureDescriptors,
CaseComponents, and precomputed plan values.  It performs no runtime string
registry search and does not rebuild the Feature dependency graph per Agent.

=build.ss= remains the canonical Scheme build truth.  Gerbil package metadata
describes code dependencies; Feature dependencies remain typed POO data and are
validated by the Feature resolver.

* Static fast lane

#+begin_src mermaid
flowchart LR
    DECL["defpoo-feature declarations"] --> AOT["Compiled Gerbil macros"]
    PROFILE["defpoo-feature-profile"] --> AOT
    AOT --> GRAPH["Canonical Feature graph"]
    GRAPH --> VALID["Compile-time validation"]
    VALID --> FPLAN["Immutable FeatureActivationPlan"]
    VALID --> DCASE["Shared closed DomainCases"]
    FPLAN --> RUN["Direct runtime references"]
    DCASE --> RUN
#+end_src

For =F= selected Features and =E= dependency edges, static graph validation is
=O(F + E)= at expansion/build time.  Runtime access to the compiled plan is a
direct binding reference.  DomainCase closure remains proportional to the =U=
unique demanded cases and is never repeated for all =N= Agents.

The static lane MUST NOT pre-materialize the Cartesian product of Feature
options, roles, policies, strategies, adapters, and DomainCases.

* Dynamic plugin lane

Dynamic plugins export the same POO FeatureDescriptor contract from a compiled
Gerbil module or foreign adapter.  A named loader performs the effect, records
module/build provenance, and hands the descriptor to the ordinary Feature
resolver.  Dynamic resolution uses a scoped bounded cache and the same
diagnostic/receipt schema as the static lane.

#+begin_src mermaid
flowchart LR
    REQUEST["Explicit plugin load request"] --> LOADER["Runtime loader adapter"]
    LOADER --> EXPORT["Precompiled exported FeatureDescriptor"]
    EXPORT --> VERIFY["Schema, provenance and dependency verification"]
    VERIFY --> CACHE["Scoped bounded plan cache"]
    CACHE --> PLAN["FeatureActivationPlan receipt"]
    VERIFY -->|"reject"| DIAG["POO diagnostic receipt"]
#+end_src

Dynamic loading MUST NOT execute during macro expansion, ordinary profile
selection, or DomainCase instantiation.  The loader is a runtime boundary, not
a second module system.

* POO-native public model

=FeatureDescriptor= contains at least:

- =kind=, =feature-id=, =schema-version=, and =owner-module-id=;
- =category= and human-facing responsibility metadata;
- required and optional Feature dependencies;
- explicit conflicts;
- typed FeatureOption schemas and defaults;
- CaseComponent contributions;
- policy and strategy contributions through named functional combinators;
- runtime adapter requirements as descriptors, never live adapter effects;
- projection declarations and receipt schema ids.

=FeatureSelection= binds one FeatureDescriptor to explicit option values.
=FeatureProfile= owns an ordered set of selections and profile-level contracts.
=FeatureActivationPlan= is the pure resolved result.  All are POO objects; raw
alists, record DSLs, mutable global registries, and ad hoc patch languages are
not ordinary public interfaces.

The intended static authoring shape is:

#+begin_src scheme
(defpoo-feature session-core-feature
  (feature-id 'session-core)
  (schema-version 1)
  (category 'agent-runtime)
  (options
   (multi-agent boolean default: #t)
   (durable-history boolean default: #f))
  (requires memory-core-feature)
  (components session-agent-component session-policy-component))

(defpoo-feature-profile default-agent-profile
  (select session-core-feature
          multi-agent: #t
          durable-history: #t))
#+end_src

The macro syntax is declarative, but its expansion is ordinary POO-native and
functional code.  The option forms are not runtime alists.

* Relationship to CaseComponent and DomainCase

A FeatureDescriptor is a selectable contribution bundle.  A CaseComponent is a
semantic reuse unit binding object behavior, type refinement, contracts, policy,
strategy, and projections.  A DomainCase is the sealed composition of selected
components plus local domain definitions.

Feature resolution decides which contributions are present.  DomainCase closure
decides whether those contributions form a valid constructible semantic case.
Neither operation is Gerbil module inheritance.

* FunFlow migration boundary

FunFlow remains a functional planning capability.  Its authoring macro lowers
to ordinary POO Feature objects, named functional combinators, and a compiled
runtime projection.  The canonical public shape is =use-feature funflow=; a
normalized internal form belongs to the FunFlow Feature owner and is never a
second public workflow language.

The lowering boundary is:

#+begin_src mermaid
flowchart LR
    UF["use-feature funflow"] --> FM["Compiled FunFlow feature macro"]
    FM --> PO["POO-native FeatureSelection and CaseComponents"]
    PO --> FP["Pure functional plan"]
    FP --> RP["Runtime projection"]
    RP --> RR["Runtime receipt"]
#+end_src

Source locations, declaration names, policies, and runtime contract ids remain
explicit in the projection.  Expanded forms do not expose raw alist IR, record
DSLs, mutable graph builders, or hidden runtime activation.

* Public vocabulary migration

The canonical migration is direct and does not preserve ambiguous legacy
syntax:

| Retired public term | Canonical term |
|---------------------+----------------|
| use-module | use-feature |
| module selection | FeatureSelection |
| module profile | FeatureProfile |
| module descriptor | FeatureDescriptor |
| module registry | FeatureCatalog or compiled owner exports |
| module lifecycle plan | FeatureActivationPlan |
| module flag | FeatureOption |

After the new surface is proven, project-owned semantic code moves out of
=src/module-system=.  Feature descriptors, selection, resolution, profiles, and
activation plans move to =src/feature-system=.  CaseComponent and DomainCase
owners move to =src/domain-case=.  Gerbil's module system remains upstream and
does not receive a POO Flow wrapper pretending to be another language module
system.

* Qualification gates

1. Importing a Feature owner module does not activate its Feature.
2. One Gerbil module exports two independent FeatureDescriptors.
3. Two FeatureProfiles coexist while sharing the same module-owned immutable
   descriptors and prototypes.
4. Static duplicate, missing dependency, cycle, conflict, and invalid option
   cases fail during expansion with syntax-owned diagnostics.
5. Macro expansion performs no runtime activation or dynamic load.
6. The static runtime path performs no FeatureCatalog string search.
7. Dynamic loading is observable only through the loader adapter and produces a
   provenance-bearing receipt.
8. Static and dynamic lanes produce equivalent canonical plan keys and
   diagnostics for equivalent inputs.
9. DomainCase keys include selected Feature identities and canonical option
   values without using pointer identity.
10. =N= Agents reuse =U= closed DomainCases and module-owned FeatureDescriptors.

* Delivery phases

** Phase 3 compiled declaration boundary

Phase 3A introduces hygienic =defrules= declarations.  A Feature declaration
has a fixed identity prefix followed by named clauses; a FeatureProfile owns an
explicit selections block.  Clauses lower only to the named POO fragments from
=model.ss=.  They never emit alists, mutate a compiler registry, or activate a
runtime adapter.

Feature option schemas, option values, and option-value collections are also
constant-slot POO objects.  A =select= form may attach one explicit
=FeatureOptionValues= object.  Phase 3A preserves this typed object boundary;
Phase 3B validates values against the selected Feature's schema manifest.

#+begin_src mermaid
flowchart LR
    DF["defpoo-feature"] --> CM["Compiled hygienic defrules"]
    DP["defpoo-feature-profile"] --> CM
    CM --> PF["Named constant-slot POO fragments"]
    PF --> FV["FeatureDescriptor and FeatureProfile"]
    FV --> PR["Pure Feature resolver"]
    PR --> AP["FeatureActivationPlan"]
#+end_src

Phase 3B remains explicit future work: cross-declaration dependency-cycle and
conflict validation at expansion/build time requires syntax-safe manifests and
must not be implemented as mutable expander-global state.  Until that manifest
owner exists, the same checks remain deterministic resolver diagnostics.

** Implementation status

- [X] Phase 1: canonical vocabulary, four-layer boundary, and legacy guidance
  removal.
- [X] Phase 2: POO-native Feature fragments, Descriptor, Selection, Profile,
  ActivationPlan, and pure resolver under =src/feature-system=.
- [X] Phase 3A: compiled declaration macros and hygienic POO lowering.
- [ ] Phase 3B: syntax-safe manifests and cross-declaration build diagnostics.
- [ ] Phase 4: replace the remaining project-owned =use-module= code surface.
- [ ] Phase 5: compile one real session FeatureProfile into DomainCases.
- [ ] Phase 6: explicit dynamic plugin loader and provenance receipts.

Phase 2 is split across three Gerbil modules: =model.ss= owns immutable POO
values and named functional fragments, =resolver.ss= owns graph validation and
ordering, and =interface.ss= is the explicit import/export facade.  POO object
construction uses constant-slot instance overlays; it does not run C3 for every
FeatureDescriptor, FeatureSelection, FeatureProfile, or Agent instance.

1. Land this vocabulary and layering decision; reduce the old =use-module=
   Macro IR path to a non-compatibility canonical-link page with no legacy
   examples or contract.
2. Introduce POO-native FeatureDescriptor, FeatureSelection, FeatureProfile, and
   pure resolver functions under =src/feature-system=.
3. Add thin =defpoo-feature= and =defpoo-feature-profile= compiled macros with
   phase-hygiene tests.
4. Replace public =use-module= with =use-feature= and remove old docs, exports,
   tests, and fixtures without a compatibility wrapper.
5. Compile one real session FeatureProfile into existing CaseComponents and
   DomainCases, then record build-time, runtime, and RSS receipts.
6. Add the explicit dynamic plugin loader lane only after the static fast lane
   and provenance receipts are stable.
