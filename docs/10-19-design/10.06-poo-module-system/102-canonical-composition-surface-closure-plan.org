#+title: Canonical Composition Surface Closure Implementation Plan

* Delivery rule

Implement in the current checkout and create one path-scoped commit only after
all qualification gates pass.  Preserve the pre-existing build and CLI-support
changes without staging or committing them.

* Step 1 — Parser contract

1. Add or update a focused parser test so the accepted module form contains the
   exact =use-module module-name as alias= prefix and optional profile clauses.
2. Repair the syntax-plan list deconstruction without changing profile, compose,
   or stage semantics.
3. Delete the legacy-specific parser branch and diagnostic text.
4. Prove invalid non-canonical forms report only
   =composition-invalid-module-form=.

* Step 2 — Accessor facade

1. Confirm =profile-composition-accessors.ss= as the accessor owner.
2. Re-export the public read-only composition accessors through
   =profile-composition.ss=.
3. Do not duplicate accessor implementations or expose mutable representation.

* Step 3 — Consumer closure

1. Remove repository-owned legacy-positive examples and assertions.
2. Keep LangChain, LangGraph, CrewAI, tool-calling, and smoke fixtures on the
   canonical grammar.
3. Update focused expectations only where the public accessor or canonical
   diagnostic contract changed.

* Step 4 — Artifact and regression gates

1. Compile the syntax plan, syntax lowering, accessor owner, and facade with the
   host Nix C include variables removed per command.
2. Run composition syntax smoke and profile composition tests.
3. Run LangChain/LangGraph, CrewAI, and tool-calling scenarios.
4. Run the AC-01 canonical organization Bundle scenario.
5. Parse the design, plan, and receipt through the Org provider.
6. Run =git diff --check=.

* Step 5 — Receipt and commit

Record the canonical grammar, removed public routes, owner/export closure,
commands, check counts, and any bounded environmental workaround.  Stage only
feature-owned paths and commit once with an explicit path scope.
