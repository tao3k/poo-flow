#+title: RFC 138: Ordered Feature Composition Plans for Multi-Agent Cases
#+status: implemented

* Decision

Phase 3C introduces an immutable POO-native =FeatureCompositionPlan=.  It is the
canonical bridge from a qualified =FeatureManifestBundle= to downstream Case,
agent-role, policy, strategy, adapter, and projection assembly.

The plan consumes facts that already belong to =FeatureDescriptor= and
=FeatureManifest= values.  It does not add another module registry, graph
resolver, patch DSL, or runtime executor.

* Problem

Feature bundles qualify identity, dependencies, conflicts, and option schemas,
but their contribution fields were not yet materialized into one resolver-ordered
value.  Every downstream assembler would otherwise need to repeat the same work:

1. reject an invalid bundle;
2. recover resolver activation order;
3. find each immutable manifest;
4. flatten components, policies, strategies, adapters, and projections;
5. preserve POO value identity and declaration order.

Repeating this logic across hundreds or thousands of agent roles would create
multiple composition semantics.  Phase 3C gives that transformation one owner.

* Public contract

=feature-composition-plan= accepts one =FeatureManifestBundle=.  It first passes
the bundle through =require-valid-feature-manifest-bundle= and therefore never
returns a partially accepted plan.  A successful constant-slot POO value exposes:

- =kind=, =schema-version=, =status=, and =accepted?=;
- the original bundle and semantic bundle id;
- resolver-ordered feature ids and manifests;
- ordered component values;
- ordered policy and strategy contribution values;
- ordered adapter requirement values;
- ordered projection values.

The declarative module-level spelling is:

#+begin_src scheme
(defpoo-feature-composition-plan agent-case-plan
  (from-bundle agent-feature-bundle))
#+end_src

=from-bundle= is intentionally the only clause spelling.  A generic =bundle=
literal can be shadowed by a same-named value binding under hygienic macro
comparison; no ambiguous legacy spelling is provided.

* Ordering and value semantics

Bundle declaration order is not authoritative.  The plan maps the resolver's
ordered feature ids through the bundle-owned manifest index, so required features
always contribute before their dependants.

Contribution values are preserved by identity and are not converted to raw
alists or records.  The composition plan does not deduplicate, execute, merge,
or override them.  Those operations belong to the domain Case assembler and its
explicit contract, policy, strategy, adapter, or projection owner.

* Functional implementation

The implementation has two pure control-plane stages:

1. =poo-flow-map= resolves ordered feature ids to manifest POO values through
   the accepted bundle index;
2. one tail-recursive fold accumulates all five contribution families in reverse
   order, followed by one final reverse per family.

No =append= occurs in the hot fold, so contribution growth is linear rather than
quadratic.

Let =F= be activated features and =C/P/S/A/R= the numbers of component, policy,
strategy, adapter, and projection values.  Expected time is:

#+begin_example
O(F + C + P + S + A + R)
#+end_example

Retained output space has the same linear bound.  Dependency and conflict
complexity remain owned by the existing =O(F + E)= resolver.

* Qualification gates

The implemented qualification suite proves:

- reverse bundle declaration order is normalized to dependency order;
- POO component values retain object identity;
- every contribution family preserves feature and declaration order;
- an empty bundle produces the identity composition;
- a 1024-feature chain with 5120 total contribution values remains ordered and
  completes within the Scheme-side memory guard;
- rejected bundles and malformed macro declarations fail instead of producing
  partial plans;
- source and isolated AOT execution require no stale local artifact.

* Next mainline stage

The next stage consumes =FeatureCompositionPlan= in the domain Case component
assembler.  That stage must bind object composition, contract/type projection,
policy composition, and strategy composition through explicit POO or functional
owners.  It must not move runtime execution, Cedar evaluation, or adapter I/O
into this pure plan layer.
