#+title: RFC 137: Immutable Feature Manifest Bundles and Explicit Build Qualification
#+status: implemented

* Decision

Phase 3B introduces immutable POO-native =FeatureManifest= values and explicit
=FeatureManifestBundle= qualification.  This RFC refines RFC 136 without
changing its Gerbil-module/POO-feature layering:

- Gerbil modules remain the language-level static ownership boundary.
- POO feature descriptors remain the public composition surface.
- Funflow-style pure mapping remains the ordinary control-plane route.
- =resolve-feature-profile= remains the only dependency graph resolver.
- no expander-global feature registry is introduced.

This document is implemented, not a draft.

* Problem

Separate feature declarations must be qualified as one deployable unit.  The
qualification needs stable identities, dependency order, conflicts, option
schemas, and structured diagnostics.  Gerbil single-instantiation modules can
carry those values efficiently, but single instantiation does not by itself
prove that unrelated macro expansions may discover each other through a hidden
compile-time table.

Treating that table as already available would mix three phases:

1. macro expansion of a local declaration;
2. module initialization of immutable feature values;
3. build qualification of a complete deployment bundle.

The implementation keeps those phases explicit.

* Public POO contract

** =FeatureManifest=

=feature-manifest= is a pure projection from a =FeatureDescriptor= POO object.
The resulting constant-slot POO object carries only static facts:

- schema version, feature id, owner module id, and category;
- required, optional, and conflicting feature ids;
- option schemas;
- component, policy, strategy, adapter, and projection contributions.

It carries neither a mutable registry callback nor a runtime compute hook.

** =FeatureManifestBundle=

=feature-manifest-bundle= receives a semantic bundle id and descriptor POO
values.  =defpoo-feature-manifest-bundle= is the declarative spelling for that
same POO-native value construction.  The bundle exposes:

- descriptors and their immutable manifests;
- the existing resolver activation plan;
- ordered feature ids;
- =ready= or =rejected= status;
- =accepted?= and structured diagnostics.

=require-valid-feature-manifest-bundle= is the named build gate.  It returns an
accepted bundle by object identity and raises with the bundle id plus diagnostic
POO objects for a rejected bundle.

Raw alist DSLs, record patch languages, and raw
=(lambda (self super) ...)= hooks are not public alternatives.

* Resolution and complexity ownership

The bundle does not duplicate dependency resolution.  It maps descriptors to
normal =FeatureSelection= POO values and invokes =resolve-feature-profile=.
Dependency, duplicate-selection, conflict, and cycle diagnostics therefore keep
one semantic owner and one =O(F + E)= graph pass.

Option identity validation is a separate structural pass.  A bundle-local hash
index rejects duplicate option ids within a feature in =O(O)= expected time,
where =O= is the number of option declarations.  The local index never escapes
as public mutable state.

* Phase truth

The bundle is constructed once through ordinary Gerbil module instantiation.
Qualification occurs when the explicit bundle value is constructed or passed
through the named build gate.  It is build qualification, not a claim that the
macro expander has already linked every declaration across modules.

Cross-module =syntax-local-value= lookup or an equivalent phase-bound manifest
binding may be considered only after Gerbil-owned evidence proves a stable
protocol.  If added, it must be an optimization of this contract; it may not
introduce a second resolver, compatibility syntax, or mutable global registry.

* Qualification gates

The implementation is complete only when all of the following hold:

- source and AOT compilation use Homebrew Gerbil with clean SDK variables;
- the public feature-system interface re-exports the manifest surface;
- a valid dependency bundle preserves resolver activation order;
- missing dependencies, duplicate feature identities, conflicts, and duplicate
  option identities produce rejected POO receipts;
- the named build gate accepts by identity and rejects with a non-zero Gerbil
  process result;
- no stale compiled artifact is required to run the compiled test suite.

* Implemented optimization: single-pass structural qualification

The initial implementation projected all manifests, mapped every manifest to a
separate diagnostic list, and then called =apply append= across the =F= lists.
Although asymptotically linear, that shape retained an avoidable list-of-lists
and expanded one procedure argument per feature.  The latter is unnecessary
pressure when bundles contain hundreds or thousands of agent capabilities.

The optimized implementation uses one tail-recursive functional fold over the
descriptor sequence.  Each step creates one immutable manifest and folds that
manifest's option schemas directly into a reverse diagnostic accumulator.  One
final reverse restores declaration order.  The existing =poo-flow-map= selection
projection and =resolve-feature-profile= graph owner remain unchanged.

The structural path is =O(F + O)= expected time and =O(F + D)= retained output,
where =O= is the number of option schemas and =D= is the number of diagnostics.
It has no =F=-wide =apply= call and no intermediate diagnostic list per feature.
The graph path remains =O(F + E)=; this optimization does not create a second
resolver or change diagnostic semantics.

The qualification suite includes a 1024-feature dependency chain.  It proves
manifest cardinality, activation order, accepted status, source-mode completion,
and bounded RSS under the same POO-native public interface.

* Follow-up

Phase 3C may investigate phase-bound manifest bindings.  It must begin from an
exact Gerbil expander owner/selector and a negative qualification fixture.  In
the absence of that evidence, explicit immutable bundles are the canonical and
only public route.
