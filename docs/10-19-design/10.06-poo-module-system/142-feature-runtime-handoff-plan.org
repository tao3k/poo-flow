#+title: RFC 142: Native Feature Runtime Handoff Planning
#+status: implemented

* Decision

Phase 3G introduces a module-owned =FeatureRuntimeHandoffManifest= and an
immutable =FeatureRuntimeHandoffPlan=.

The manifest declares how already-bound adapter requirements and projection
requests participate in three explicit handoff families:

- Runtime Bundle handoff, including Bundle v1 C ABI intent;
- Cedar input handoff;
- Lean/Evidence obligation handoff.

The plan resolves those declarations against an accepted
=FeatureAdapterProjectionBinding=.  It carries native Gerbil POO references to
the selected provider, adapter binding, and Domain Case projection binding.  It
does not serialize JSON, allocate a C request, call CFFI, import Python, evaluate
Cedar, invoke a projector, run Lean, or emit evidence.

* Existing owner boundary

Gerbil compiled-module metadata shows that
=src/module-system/loop-engine-runtime.ss= already owns user-intent runtime
envelopes and runtime-handoff facts.  It does not export a generic Feature
handoff planner.  Likewise, =src/core/policy.ss= and =src/core/strategy.ss= own
runner-facing execution snapshots and planners, not Feature composition.

Phase 3G therefore does not replace those owners.  It creates the missing pure
qualification layer before them:

1. Feature resolution and Domain Case closure remain unchanged;
2. policy/strategy and adapter/projection bindings remain authoritative;
3. the handoff plan resolves semantic references only;
4. a later runtime lowering boundary may translate the accepted plan into the
   existing loop-engine or C ABI representation.

* Native POO handoff declarations

Every handoff declaration contains:

1. semantic =handoff-id=;
2. bound =adapter-requirement-id=;
3. bound =projection-request-id=;
4. expected contract id;
5. expected contract version;
6. expected projection schema id.

Three distinct POO kinds keep downstream intent explicit while sharing the same
resolution algebra.

#+begin_src scheme
(defpoo-feature-runtime-bundle-handoff python-bundle-handoff
  (handoff-id 'python-bundle-handoff)
  (adapter-requirement-id 'python-runtime-requirement)
  (projection-request-id 'runtime-bundle-request)
  (bundle-contract-id 'poo-flow.runtime-c.bundle)
  (bundle-version +feature-runtime-c-bundle-version+)
  (schema-id 'runtime-bundle-schema-v1))

(defpoo-feature-cedar-input-handoff cedar-input-handoff
  (handoff-id 'cedar-input-handoff)
  (adapter-requirement-id 'cedar-input-requirement)
  (projection-request-id 'cedar-input-request)
  (input-contract-id 'poo-flow.cedar-input)
  (input-contract-version 1)
  (schema-id 'cedar-input-schema-v1))

(defpoo-feature-evidence-obligation lean-obligation
  (handoff-id 'lean-obligation)
  (adapter-requirement-id 'lean-evidence-requirement)
  (projection-request-id 'evidence-request)
  (evidence-contract-id 'poo-flow.lean-evidence)
  (evidence-contract-version 1)
  (schema-id 'lean-evidence-schema-v1))
#+end_src

=+feature-runtime-c-bundle-version+= is =1=.  This makes the accepted Bundle v1
contract visible in Scheme planning without inventing a JSON =runtime-v0=
envelope.  A future version remains a distinct contract/version value rather
than an ambiguous compatibility mode.

* Module-owned manifest

Gerbil modules declare handoff values once and compose them explicitly into a
manifest:

#+begin_src scheme
(defpoo-feature-runtime-handoff-manifest agent-runtime-handoffs
  (manifest-id 'agent-runtime-handoffs)
  (handoffs python-bundle-handoff cedar-input-handoff lean-obligation))
#+end_src

The manifest validates POO kinds, rejects duplicate semantic ids with an
equal-key hash index, and exposes immutable lookup.  No expander registry,
global runtime registry, or load-order side effect is introduced.

* Planning algorithm

The planner receives an accepted adapter/projection binding and an accepted
handoff manifest.

It builds two ephemeral indexes:

- adapter bindings keyed by =requirement-id=;
- projection bindings keyed by =request-id=.

For every handoff it then proves:

1. the adapter requirement was resolved;
2. the projection request was resolved;
3. contract id and version equal the bound adapter contract;
4. schema id equals the bound Domain Case projection schema.

A successful resolved handoff stores the original declaration, adapter binding,
projection binding, provider module id, contract, and schema.  Object identity
is preserved; no copying into alists or strings is required.

Planning complexity is O(A + P + H), where =A= is resolved adapter bindings,
=P= is resolved projection bindings, and =H= is handoff declarations.

* No-JSON and no-execution boundary

The handoff plan is not a transport envelope.  Its hot-path values are native
Scheme/POO references:

- provider module identity comes from the resolved adapter capability;
- contract identity/version comes from the adapter binding;
- schema identity and projector ownership come from the projection binding;
- policy and strategy remain reachable through the upstream accepted receipt.

Only the future runtime-c lowering owner may choose a C struct, pointer table,
arena, tagged union, or other ABI representation.  JSON is not an intermediate
requirement and must not be inserted into the native FFI path.

* Rejection semantics

Structured POO diagnostics distinguish:

- invalid or duplicate handoff declarations;
- invalid or rejected handoff manifests;
- invalid or rejected adapter/projection bindings;
- missing adapter requirement bindings;
- missing projection request bindings;
- adapter contract mismatch;
- projection schema mismatch.

The plan is ready only when every declared handoff resolves.  Partial results do
not authorize any downstream runtime action.

* Qualification gates

The implementation proves:

- Bundle v1, Cedar input, and Lean/Evidence declarations resolve through one
  shared native algebra while retaining distinct POO kinds;
- provider module, adapter binding, and projection binding object identities are
  preserved;
- invalid and semantic duplicate manifest entries are rejected;
- missing bindings, contract mismatch, and schema mismatch remain distinct;
- rejected upstream receipts stop planning;
- planning never invokes a Domain Case projector;
- 1024 ordered handoffs resolve through indexed bindings under the Scheme memory
  guard;
- empty-root AOT execution requires no repository source fallback.

* Next mainline stage

Phase 3H should audit and connect the accepted handoff plan to the existing
=bindings/runtime-c= Bundle v1 owner.  It should define native Scheme-to-C value
lowering, ownership/lifetime, zero-copy opportunities, error receipts, and ABI
conformance gates for Gerbil, =gerbil-mysql=, and Python CFFI.  It must not add a
JSON compatibility path or a second runtime protocol.
