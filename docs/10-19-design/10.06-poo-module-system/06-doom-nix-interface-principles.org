#+title: 10.06.06 Doom And Nix Interface Principles

* 10.06.06 Doom And Nix Interface Principles
** 10.06.06.10 Purpose
This note defines how =poo-flow= should learn from Doom Emacs and Nix modules without copying either system blindly.

Doom is useful because it is Lisp-family software. Its user interface shows how a large modular system can feel concise through macros, dynamic predicates, conditional forms, and high-level commands. Nix is useful because it makes module composition, options, defaults, and merge behavior explicit. =poo-flow= needs both lessons:

- Doom gives the authoring experience target.
- Nix gives the configuration semantics target.
- Gerbil Scheme, POO, and C3 give the host-language implementation substrate.

** 10.06.06.20 Doom Lessons To Borrow
*** Declarative Entry Macro
Doom's =doom!= shape is valuable because users declare grouped modules in one compact form. The important idea is not Emacs Lisp loading. The important idea is that the user sees a small Gerbil module authoring surface with groups, flags, conditional inclusion, and predictable order.

For =poo-flow=, the equivalent should eventually be a module-set macro that expands into Gerbil bindings which construct POO descriptor values and catalog values. The macro may organize syntax, but it must not load files, evaluate runtime tasks, query registries, or mutate global runtime state.

*** Active Module Predicate
Doom's =modulep!= pattern is useful because users can ask whether a module or flag is active at the point where conditional configuration is written.

=poo-flow= already absorbs this as explicit data checks:

- =poo-module-active?= checks a descriptor and flag requirements.
- =poo-flow-module-value-catalog-active?= checks a catalog by module id.

This is intentionally less magical than Doom. The active predicate receives data explicitly instead of guessing the current module from the source file.

*** Domain-Specific Macros
Doom's macros such as keybinding and package helpers are effective because each macro owns one domain and compiles compact user syntax into the host language's own extension points.

=poo-flow= should use the same rule: a macro is allowed only when it has one clear domain owner and expands to inspectable Gerbil/POO values. Good future candidates:

- module-set declaration;
- option schema declaration;
- hook payload declaration;
- extension contribution declaration;
- conditional module inclusion.

Bad candidates:

- a macro that activates modules and runs flows;
- a macro that performs source loading during expansion;
- a macro that hides option merge conflicts;
- a macro that silently changes task or flow registries outside descriptor activation.
For loop engineering this means Doom-style authoring forms may eventually declare a named loop bundle, but the expansion target must remain Gerbil values such as =loop-pattern-descriptor= or =loop-strategy-plan=. The macro should never wake a loop, allocate a worktree, call a connector, or bypass the Marlin handoff boundary.

*** High-Order Authoring Helpers
Doom's high-order helpers make small transformations concise. =poo-flow= should borrow the spirit, not the exact names. In Gerbil, this means hygienic higher-order macros, macro factories, small typed helpers, and expression-level composition around:

- descriptor projection;
- option receipt projection;
- import closure traversal;
- hook payload filtering;
- catalog selection.

When high-order style makes evidence harder to inspect, prefer a named helper with an adjacent typed contract. The project policy already enforces this preference.

** 10.06.06.30 Nix Lessons To Borrow
*** Options Are Declared Separately From Values
Nix modules separate option declarations from option values. That maps directly to the current =poo-flow= split:

- =poo-module-interface= declares schemas;
- module config carries values;
- validation receipts explain mismatches.

This is the right foundation. Future work should add richer merge semantics without moving validation into runtime execution.

*** Modules Compose Into A Result
Nix modules compose into a final configuration value. They do not rely on the user remembering hidden side effects. =poo-flow= should keep this shape:

- module descriptors compose into activation;
- activation lowers into =run-config=;
- eval and presentation return replayable receipts.

The user should be able to inspect what a module contributed before any runner or adapter is invoked.

*** Merge Policy Must Be Explicit
Nix is valuable because override and merge behavior are part of the module model. =poo-flow= now implements the first explicit merge layer as data receipts:

- option default;
- option required;
- option constant;
- option optional;
- option append;
- option override;
- option conflict.

These policies are schema vocabulary, not implicit alist order. =src/modules/merge.ss= projects them into receipts before runtime execution:
- append starts with the schema default and then dependency-to-root config values;
- override uses the root-most configured value after dependency traversal;
- conflict accepts identical values and returns an invalid receipt for divergent values;
- missing schemas and missing required values remain data findings.

** 10.06.06.40 Gerbil POO And C3 Shape
Gerbil POO and C3 are the reason this module system should stay in Scheme. They let the interface remain Lisp-friendly while preserving object composition semantics.

The design rule is:

- use POO objects for descriptor and interface values;
- use C3 composition for role and prototype inheritance;
- expose stable accessors for user-facing inspection;
- keep leaf owners split under =src/modules=;
- keep =:modules/module-system= as the only normal user import.

Macros should expand into Gerbil bindings and POO values in this system. They should not become an alternate runtime or a separate alist/record DSL.

** 10.06.06.50 Macro Design Contract
Every user-facing macro must satisfy all of these rules:

- Expansion is hygienic Gerbil code that constructs POO descriptors, catalogs, or boundary receipts.
- Expansion is inspectable through existing accessors.
- Expansion does not introduce an alist/record DSL as the semantic core.
- Expansion does not read files.
- Expansion does not resolve catalogs.
- Expansion does not activate modules.
- Expansion does not run tasks or flows.
- Expansion does not depend on hidden current-file module context.
- Expansion has a focused test that imports only =:modules/module-system=.

This contract lets =poo-flow= borrow Doom's friendly Lisp surface while staying a Gerbil-hosted POO module system.

** 10.06.06.60 Proposed User Interface Layers
*** Layer 1: Explicit POO API
Status: implemented.

Users can already declare interfaces, modules, imports, catalogs, active checks, eval receipts, and presentation receipts through =:modules/module-system=.

*** Layer 2: Focused Hygienic Authoring Forms
Status: partially implemented.

Implemented macros:

- =defpoo-module-set= for grouped module declarations;
- =poo-module-option-block= for schema and config POO slot blocks;
- =poo-module-hook-block= for hook POO slot blocks;
- =poo-module-when= for explicit conditional inclusion.

The =*-block= names avoid colliding with existing public accessors such as =poo-module-options= and =poo-module-hooks=. These macros only expand into layer-1 Gerbil/POO values.

*** Layer 3: Nix-Style Merge Policy
Status: implemented for first-pass projection semantics.
Schema vocabulary and receipts now cover default, override, append, and conflict semantics. Macro authoring remains constrained to POO behavior that is tested through =:modules/module-system=.
*** Layer 4: Loader Integration
Status: future.

Source refs may eventually resolve through loader backends. This must remain separate from macro expansion and descriptor activation.

** 10.06.06.70 Testing Requirement
Every new user-facing interface must add or update tests in =t/module-system-user-interface-test.ss=.

The required test shape is:

- import =:modules/module-system=;
- avoid importing leaf owners;
- construct values as a module author would;
- assert descriptor shape through public accessors;
- assert macro block values as POO slot objects before boundary projection;
- assert receipts before runtime execution;
- assert failure or validation behavior as data.

This test file is the guardrail that keeps implementation modular while allowing a Doom-like authoring experience.
** 10.06.06.80 Implemented Merge Receipt Layer
Status: implemented for first-pass data semantics.
Implemented helpers and projections:
- =poo-option-append=, =poo-option-override=, =poo-option-conflict=, and =poo-option-merge=;
- =poo-module-option-merge-receipts= for inspectable merge findings;
- =poo-module-merged-option-alist= for valid final option values.
The implemented merge owner is =src/modules/merge.ss=. It keeps append, override, conflict, constant, required, and missing-schema findings as inspectable data before runtime execution.
** 10.06.06.90 Implemented Doctor Presentation Layer
Status: implemented for first-pass user-facing summaries.
Implemented helpers and projections:
- =pooModuleDoctorPresentation= for value catalogs;
- =pooModuleSourceDoctorPresentation= for source refs and loader backends;
- doctor summaries for module ids, import graph, contribution counts, diagnostics, validation count, merge status, and merged options.
The implemented doctor owner is =src/modules/doctor.ss=. It borrows Doom's inspectable doctor experience while preserving the data-only, no-runtime-execution boundary.
** 10.06.06.100 Maintained User Interface
Status: implemented.
=user-interface/= is the root-level downstream configuration surface. It is intentionally thin: users choose modules in =modules.ss= and set flow, loop, sandbox, and strategy values in =config.ss=.
This is the POO Flow equivalent of a small user configuration shape:
- downstream owns module enablement and flags;
- downstream owns flow, loop, strategy, and sandbox option values;
- upstream =src/modules= owns descriptor schemas, import realization, loader manifests, catalogs, doctors, and merge semantics.
