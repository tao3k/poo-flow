* 10.06.07 Loop Engineering, C3 Now, C4 Later
** 10.06.07.10 Purpose
This note records the local research pass over =cobusgreyling/loop-engineering= and maps it onto =poo-flow='s Gerbil POO design.

The main design conclusion is narrow:

#+begin_src text
POO/C3 should rank loop policies, roles, and descriptor slots now.
C4 remains a forward-compatibility target after Gerbil 0.19 stabilizes.
Neither C3 nor C4 should schedule loop iterations or replace the execution DAG.
#+end_src

=loop-engineering= is useful for the operational shape of long-running agent loops. Gerbil POO/C3 is the current implementation substrate for deterministic composition of the policies that make such loops safe and reusable.

** 10.06.07.30 Loop Engineering Evidence
=loop-engineering= defines a loop as a recurring control system around agents, not as a one-shot workflow declaration.

Key upstream evidence:

- =README.md:103-120= shows the loop anatomy: schedule or automation, triage skill, durable state, isolated worktree, implementer sub-agent, verifier sub-agent, MCP/Git/ticket connector, then a human gate before commit, PR, action, or escalation.
- =docs/primitives.md:5-83= names the practical primitives: scheduling, worktrees, skills, MCP/connectors, maker/checker split, and durable memory/state.
- =docs/operating-loops.md:113-125= defines the upgrade path: L1 report-only, L2 small auto-wins with verifier/worktree/max attempts, L2+ connector writes, and L3 unattended only with denylist, budget, metrics, and human gates.
- =docs/safety.md:54-96= requires human gates for security/auth, payments/PII, production infrastructure, dependency upgrades, large changes, and repeated failed attempts. It also requires redaction, test safety, incident pause/revert, and a pre-flight checklist before L3.
- =docs/multi-loop.md:5-73= defines multi-loop coordination: one owner per branch, separate state files, triage reports while action loops execute, shared denylist, aggregate budget, explicit priority, =acting_on= collision checks, and a human inbox.
- =patterns/README.md= and =templates/pattern-template.md= make a loop pattern a reusable contract with goal, schedule, skills, state shape, cycle, verification strategy, human handoff, failure modes, and success metrics.

This is stronger than a normal workflow library because it treats time, memory, safety, cost, and human escalation as first-class design inputs.
** 10.06.07.40 C3 Baseline And C4 Evidence
The Gerbil upstream evidence is still useful, but it has two different statuses:

- C3/POO is the current baseline for =poo-flow= composition on Gerbil 0.18.2.
- C4 is forward-looking v0.19 evidence for how suffix-preserving composition can mature later.

- =doc/reference/gerbil/runtime/c3.md:1-90= describes C4 as a slight extension of C3 that preserves linearization, local precedence, monotonicity, shape determinism, extended precedence, and the C4 suffix property.
- =src/gerbil/runtime/c3.ss= exports =c4-linearize=, =c4-linearize*=, and =c4-compute-class-slots=.
- =src/gerbil/runtime/c3.ss:299-344= keeps the C3 candidate selection loop: choose a candidate with zero remaining ancestor count, remove it from tails, then continue building the precedence list.
- =src/gerbil/runtime/c3.ss:356-383= computes class slots from the precedence list, then applies direct slots.
- =src/gerbil/runtime/mop.ss:591-605= shows the MOP computing class precedence via =c4-linearize=.

The important property for =poo-flow= is not the algorithm's name. The property is that independently authored roles can be merged into a deterministic, monotonic precedence list before a run starts. Today that means POO/C3; later it may mean C4 when the runtime version is available.
** 10.06.07.50 Boundary Rule
Use POO/C3 where the question is:

#+begin_src text
When multiple policies or roles apply, which one wins and which inherited slots are visible?
#+end_src

Do not use POO/C3 or C4 where the question is:

#+begin_src text
Which task is ready to execute now?
Which loop should wake up next?
Which external action should be retried?
#+end_src

Those are runtime state and scheduling questions. In current =poo-flow=:

- =src/core/roles.ss= already owns POO role composition and safe slot probing.
- =src/core/plan.ss= owns flow lowering, dependency edges, root/terminal frontiers, and ready-node selection.
- =src/core/runner.ss= owns plan interpretation, ready-frontier projection, validation, execution receipts, and replay reports.
- =marlin-agent-core= should remain the owner for heavy scheduling, persistence, subprocesses, CAS, and concurrent execution.
** 10.06.07.60 Proposed Design: C3-Ranked Loop Pattern Descriptors
Add a =loop-pattern-descriptor= layer as a policy object, not as another task type.

A loop pattern descriptor should be a Gerbil POO descriptor composed from small roles:

| Role | Owns | POO/C3 Precedence Intent |
|------+------|----------------------|
| =loop-purpose-role= | goal, watched scope, non-goals | anchors the pattern contract |
| =loop-level-role= | L1, L2, L2+, L3 capability gates | limits what later roles may enable |
| =loop-schedule-role= | cadence, fire-immediately, pause window | proposes wake-up policy |
| =loop-state-role= | state store, run cursor, =acting_on= key | requires durable read/write |
| =loop-budget-role= | token cap, run cap, weekly cap | can veto scheduling or spawning |
| =loop-isolation-role= | worktree/session isolation | can require separate worktrees before writes |
| =loop-skill-role= | triage/action/verifier skills | supplies intent memory |
| =loop-connector-role= | MCP/Git/ticket scopes | exposes external surfaces with least privilege |
| =loop-maker-role= | implementer/explorer action agent | proposes changes |
| =loop-checker-role= | verifier, test runner, stop condition | decides whether maker output is accepted |
| =loop-safety-role= | denylist, human gates, max attempts | must override autonomy |
| =loop-observability-role= | receipts, run log, metrics | records decisions and cost |

POO/C3 matters because a concrete pattern often mixes several of these roles:

#+begin_src scheme
;; Shape only. Exact API belongs in the implementation milestone.
(define-loop-pattern pr-babysitter
  (:: @ loop-purpose-role
        loop-level-role
        loop-state-role
        loop-budget-role
        loop-isolation-role
        loop-maker-role
        loop-checker-role
        loop-safety-role
        loop-observability-role))
#+end_src

The descriptor composition order becomes reviewable policy. For example, =loop-safety-role= and =loop-budget-role= must be later inspected as hard gates before =loop-maker-role= can request a write-capable runtime.
** 10.06.07.70 Proposed Run Protocol
The loop run protocol should be explicit data that can be described and validated by Scheme, then executed by =marlin-agent-core=:

1. Read the loop descriptor and its POO/C3-ranked policy slots.
2. Read durable state and prior run log.
3. Apply level gates, safety denylist, budget gates, and collision checks.
4. Run triage or planning in report-only mode when L1.
5. If action is allowed, allocate isolated worktree/session and invoke maker.
6. Invoke checker with a fresh verification context.
7. Accept, reject, retry, or escalate based on checker output and max attempts.
8. Write state, receipt, and run log before returning.

This run protocol is a loop envelope around flows. If the maker executes a =poo-flow= flow, that flow is still lowered by =src/core/plan.ss= and interpreted by =src/core/runner.ss=.
** 10.06.07.80 Multi-Loop Governor
The multi-loop problem should be a first-class descriptor bundle:

- each loop has a state file or external state key;
- each action loop writes =acting_on= before mutating;
- the governor reads all loop states before scheduling action;
- static priority is a descriptor slot, not a hard-coded special case;
- human inbox is a shared state projection;
- budget is aggregate across loops, not only per-pattern.

C3 can compose default priority policies with repository-specific overrides. Runtime readiness still depends on current state, branch ownership, and budget counters, so the governor remains a policy interpreter rather than an inheritance algorithm. C4 can later strengthen this layer where suffix-preserving descriptor families matter.
** 10.06.07.90 Rust Boundary
The heavy implementation should stay outside Scheme and land in =marlin-agent-core= when it is operational:

- durable timers and wake-up queues;
- state storage beyond simple files;
- worktree creation and cleanup;
- process execution and cancellation;
- connector IO and credential handling;
- concurrency, backoff, and retry accounting.

Scheme should own:

- loop pattern descriptors;
- policy composition and validation;
- level gates and human-gate definitions;
- conversion from descriptor policy to runtime requests;
- receipts and audit-friendly explanations.

This preserves the current project direction: Gerbil is the policy/control plane, Rust or other runtimes do the heavy execution.
** 10.06.07.100 Milestone Plan
The next implementation path should be incremental:

1. Add a design-only fixture for =loop-pattern-descriptor= and the role vocabulary above.
2. Add Gerbil POO descriptor constructors and projection helpers, keeping state as plain data.
3. Add L1 report-only loop receipts: read descriptor, read state, produce triage/handoff receipt, no writes except explicit state/log updates.
4. Add budget and safety gates as verifier-visible facts.
5. Add multi-loop governor descriptors with static priority and =acting_on= collision detection.
6. Add Rust/runtime request projection after the Scheme descriptor contract is stable.

The first code milestone should not attempt L3 unattended behavior. It should make L1/L2 policy surfaces inspectable and testable first.

** 10.06.07.120 Next Slice
The next useful slice is not a scheduler. It is a small =loop-governor= policy descriptor that composes:
- a static priority table;
- a shared denylist;
- an aggregate budget envelope;
- a state-key convention for =acting_on= collision checks;
- a human inbox projection.
The governor should still lower to data. The runtime owner for polling, locking, persistence, and concurrent execution remains =marlin-agent-core=.
** 10.06.07.130 Governor Code Landing
The governor slice is now represented in code:
- =src/loops/governor.ss= composes priority, state-key, aggregate-budget, collision, inbox, and handoff roles.
- =loop-governor-open-patterns= filters actionable strategy patterns through shared denylist, supplied =acting_on= state facts, and =max-actionable=.

#+end_src
** 10.06.07.140 Marlin Request Projection Landing
The Rust-facing request projection is now represented in code:
- =+loop-governor-marlin-request-schema+= names the ABI request wrapper.
- =loop-governor->marlin-request-envelope= wraps =loop-governor->contract= for =marlin-agent-core=.
- =validate-loop-governor-marlin-request-envelope= rejects malformed wrappers before they leave Scheme.
- =t/loop-governor-test.ss= asserts schema, operation, request id, target, transport, open patterns, blocked patterns, and runtime boundary.
The request envelope remains inert data:
#+begin_src text
Scheme builds the envelope.
Marlin consumes the envelope.
Neither side is simulated by the Scheme projection.
#+end_src
This gives the Rust execution layer a stable shape to bind through the Scheme ABI while keeping =poo-flow= focused on policy composition.
** 10.06.07.150 Marlin ABI Manifest Landing
The request projection now publishes a separate ABI discovery manifest:
- =+loop-governor-marlin-abi-schema+= names the manifest itself.
- =loop-governor-marlin-abi-manifest= lists the request schema, governor schema, operation, required fields, optional fields, producer, consumer, transport, and runtime boundary.
- =loop-governor->marlin-request-envelope= includes the manifest as =abi-manifest= so a Marlin binding can inspect the contract shape without reverse-engineering request instances.

This keeps three layers distinct:
#+begin_src text
ABI manifest: what Rust should expect.
Request envelope: one concrete govern-loop request.
Governor contract: policy facts derived from descriptors and supplied state.
#+end_src

The manifest does not submit work, allocate worktrees, lock state, or persist receipts. It is only the Scheme-side contract publication needed before =marlin-agent-core= implements the execution owner.
** 10.06.07.160 L1 Receipt And Runtime Manifest Landing
The loop governor now has two more data-only projections:
- =loop-governor->l1-run-receipt= emits a report-only handoff receipt for L1 validation.
- =loop-governor->marlin-runtime-manifest= publishes the concrete =govern-loop= request envelope through a runtime-manifest discovery surface.

The L1 receipt explicitly records:
#+begin_src text
state-writes = ()
effects      = ()
schedules    = ()
#+end_src

That is the local safety boundary. Scheme can prove the handoff shape, open
patterns, blocked patterns, human inbox items, and Marlin target, but it still
does not schedule the loop or mutate durable state.

The runtime manifest keeps Marlin from guessing entry points. It carries the
request schema, ABI manifest, request envelope, operation, target, transport,
and receipt schema as one inspectable alist.
