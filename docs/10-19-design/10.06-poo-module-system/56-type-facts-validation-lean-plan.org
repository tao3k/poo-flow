* 10.06.56 Type Facts Validation And Lean Plan
This document defines the first implementation slice for the =poo-flow= type validation layer.

Correction: "not user-facing" means "not a required first concept for ordinary workflow authors." It does not mean users cannot write Scheme. Scheme/POO is the programmable authoring surface; ordinary users may start from a natural-language graph, while advanced users and agents may work directly with Scheme/POO, type facts, Lean facts, and projections.

The design choice is direct: type validation needs first-class Scheme structures before final alist projection. POO remains the semantic object model, but validation receipts, type facts, and Lean fact contracts should use =defstruct= so later proof tooling can rely on stable fields instead of anonymous HashTable or raw alist shapes. Type facts, contracts, schemas, and Lean facts are internal strength that agents, validators, advanced users, and proof tooling use to make the graph safe and explainable.
** 10.06.56.10 Purpose
The type validation layer must prove that a composed POO policy can legally cross a runtime boundary.

It owns:
- slot type facts;
- contract validation receipts;
- harness validation receipts;
- safety and monotonicity diagnostics;
- Lean fact contracts;
- final report-only projections.

It does not own:
- user-facing module syntax;
- user-facing workflow graph vocabulary;
- runtime execution;
- Marlin scheduling, IO, or sandbox realization;
- direct mutation of profiles or manifests.
** 10.06.56.15 User Boundary
The type validation layer is powerful, but it is not the required first concept for ordinary workflow authors.

The ordinary user should be able to reason about:
- workflow graph;
- loop;
- step;
- model;
- module;
- gate;
- runtime handoff;
- result path.

The agent, advanced user, and control plane may reason about:
- Scheme and POO objects;
- C3 composition;
- slot contracts;
- harness validation;
- graph diagnostics;
- type facts;
- Lean fact contracts;
- final manifest projection.

The product rule is:
#+begin_src text
Show ordinary users the graph.
Let agents generate or repair Scheme/POO from that graph.
Let validators explain why the graph is valid or invalid.
Keep schema, Lean, and manifest details available for advanced users, agent
repair, proof, and runtime integration, not as required first concepts.
#+end_src
** 10.06.56.20 Layer Model
#+begin_src mermaid
flowchart LR
  UserGraph["User graph\nworkflow, loop, step, model, module, gate"]
  POO["POO object graph\nslots, prototypes, C3 order"]
  Facts["Type facts\ndefstruct records"]
  Harness["Harness validation\nstructured POO object checks"]
  Validators["Local validators\nhard gates and monotonic rules"]
  Receipt["Type validation receipt\ndefstruct"]
  Lean["Lean fact contracts\nproof-facing names"]
  Projection["Final projection\nalist / manifest / report"]
  UserGraph --> POO --> Facts --> Harness --> Validators --> Receipt --> Lean --> Projection
  Validators -->|"reject invalid graph"| POO
  Validators -->|"explain invalid graph"| UserGraph
#+end_src

The critical boundary is between =Receipt= and =Projection=. Before that boundary, data should be structured Scheme values. After that boundary, alists are allowed because the payload is a final report or runtime handoff encoding.
** 10.06.56.30 First Code Slice
The first implementation slice adds:
- =src/type-facts/objects.ss=;
- =src/type-facts.ss= facade;
- sandbox resource contract integration;
- a focused scenario test for sandbox resource validation receipts.

The new structures are:
- =poo-flow-type-fact-contract=;
- =poo-flow-lean-fact-contract=;
- =poo-flow-type-validation-receipt=.

The sandbox resource contract now returns a =poo-flow-type-validation-receipt= from =poo-flow-sandbox-resources-prototype-contract-validation=. Its final projection still emits alist rows, but the semantic owner is now a structured validation receipt.
** 10.06.56.40 Why defstruct Here
=defstruct= is appropriate at this layer because validation receipts are not user extension objects. They are stable internal facts with fixed accessors.

=defstruct= is not a weaker substitute for BAML-style DSL types. It is a Scheme-native way to give fixed shape to proof and validation receipts while keeping the semantic model in POO. Combined with Lean-facing contracts, this gives =poo-flow= a stronger validation ceiling than a standalone DSL type system, because it can validate whole-system policy composition rather than only function input/output shapes.

Use POO when:
- users or modules extend the object;
- C3 composition matters;
- profiles inherit or override slots;
- the value participates in semantic policy composition.

Use =defstruct= when:
- the value is a closed receipt;
- fields need stable accessors;
- tests, projection, and Lean extraction need fixed names;
- mutation and inheritance would weaken the proof boundary.

This keeps POO strong where it matters and gives the proof layer stronger shape guarantees.
** 10.06.56.50 Lean Boundary
Lean should not consume arbitrary runtime manifests as semantic truth.

The intended path is:
#+begin_src mermaid
flowchart TD
  SlotContract["Slot contract\nfield, kind, merge, default"]
  TypeFact["Type fact\nScheme defstruct"]
  Validation["Validation receipt\nvalid, diagnostics, source"]
  LeanContract["Lean fact contract\nowner, name, polarity"]
  LeanProjection["Lean input projection\nstable fact rows"]
  Theorem["Lean theorem / obligation"]
  SlotContract --> TypeFact --> Validation --> LeanContract --> LeanProjection --> Theorem
#+end_src

Lean fact contracts should include:
- fact key;
- Lean owner/module;
- Lean theorem or constructor name;
- source slot;
- polarity;
- metadata for project scope.

This gives agents a legal path to add proof-facing facts without editing Lean-facing output directly.
** 10.06.56.60 Next Implementation Milestones
Phase 1: sandbox resource closure.
- Keep sandbox resources as the first type-fact owner.
- Add tests for missing required slots, unstructured filesystem projection, and optional slots.
- Ensure every validation receipt has =runtime-executed = #f=.

Phase 2: session policy migration.
- Move existing session Lean fact key contracts toward shared =src/type-facts= structures.
- Preserve session-specific projections as final ABI/report surfaces.
- Avoid duplicating generic Lean contract structs in session handoff code.

Phase 3: graph and governor facts.
- Add type facts for denylist union, budget tightening, permission narrowing, and human-gate monotonicity.
- Expose them as validation receipts before runtime manifest projection.

Phase 4: Lean extraction.
- Add a projection function that emits only declared Lean facts.
- Reject unknown fact keys before Lean handoff.
- Treat missing required fact contracts as validation failures, not proof-time surprises.
** 10.06.56.70 Acceptance Rules
- Type validation receipts are structured Scheme records before final projection.
- POO object graphs remain the source of policy semantics.
- C3 order is never treated as proof of safety.
- Validators own legality.
- Final alists are projection products only.
- Lean facts are declared through contracts before they are emitted.
