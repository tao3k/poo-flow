* 10.06.52 Three Loop Human Audit Runtime Plan
** 10.06.52.10 Purpose
This document aligns Andrew Ng's June 2026 three-loop product-development
framing with =poo-flow='s Scheme/POO control plane.

The local design goal is not to copy a loop-engineering slogan. The goal is to
make the boundary precise:

#+begin_src text
Scheme expresses policy, composition, validation, graph explanation, and
runtime handoff manifests.

Marlin/Rust executes heavy runtime work: scheduling, IO, persistence, tool
connectors, sandbox operations, concurrent execution, and telemetry.
#+end_src

The three loops give us a scientific ordering for implementation:
1. Agentic Coding Loop;
2. Developer Feedback Loop;
3. External Feedback Loop.

** 10.06.52.20 Source Evidence
Andrew Ng describes loop engineering as more than a single coding-agent retry
loop. The recent public framing names three loops:
- Agentic Coding Loop;
- Developer Feedback Loop;
- External Feedback Loop.

Primary/near-primary evidence:
- Andrew Ng's X post on loop engineering:
  =https://x.com/AndrewYNg/status/2071988145667928442=.
- DeepLearning.AI The Batch issue 359:
  =https://www.deeplearning.ai/the-batch/issue-359=.

Local interpretation:
- The Agentic Coding Loop is the fastest inner loop.
- The Developer Feedback Loop changes spec, eval, product judgment, and policy.
- The External Feedback Loop supplies evidence from users, production signals,
  alpha tests, or other outside sources.

** 10.06.52.30 Three Loop Model
The loops are nested by time scale and authority.

#+begin_src mermaid
flowchart TD
  ExternalLoop["External Feedback Loop\nhours to weeks\nusers, production, market, alpha tests"]
  DeveloperLoop["Developer Feedback Loop\nminutes to hours\nproduct judgement, spec and eval deltas"]
  HumanAudit["Human Audit Loop\nlegal approval boundary"]
  CheckedInterface["Checked mutation interface\nvalidated spec/profile/eval/policy changes"]
  ControlPlane["Scheme control plane\nS-expression + POO + C3 + validators + graph"]
  Manifest["Control-plane runtime manifest\nreport-only, runtime-executed=false"]
  Marlin["Marlin/Rust runtime\nheavy execution"]
  CodingLoop["Agentic Coding Loop\ncode, test, observe, repair"]

  ExternalLoop --> Evidence["feedback evidence receipt"]
  Evidence --> DeveloperLoop
  DeveloperLoop --> Proposal["spec/eval/policy delta proposal"]
  Proposal --> HumanAudit
  HumanAudit -->|approved| CheckedInterface
  HumanAudit -->|changes requested| DeveloperLoop
  HumanAudit -->|rejected or escalated| Evidence
  CheckedInterface --> ControlPlane
  ControlPlane --> Manifest
  Manifest --> Marlin
  Marlin --> CodingLoop
  CodingLoop --> DeveloperLoop
#+end_src

This model deliberately separates feedback from mutation. External feedback
does not edit the spec. Developer judgement does not bypass validation. Human
Audit decides whether a proposed mutation can enter the checked interface.

** 10.06.52.40 Human Audit As Spec Evolution Boundary
Human Audit is the legal mutation boundary for spec evolution.

It owns:
- review items derived from external feedback, developer judgement, graph
  diagnostics, failed runs, safety signals, or product deltas;
- decisions: pending, approved, rejected, escalated, changes-requested;
- evidence that a human approved a spec/profile/eval/policy mutation;
- monotonic safety constraints: human gates can be added or escalated, but not
  removed by ordinary task policy.

It does not own:
- direct config mutation;
- workflow execution;
- runtime scheduling;
- file, process, network, connector, or persistence side effects.

The current implementation direction already matches this boundary:
=src/loops/human-audit.ss= is review-only, governor-derived, human-owned, and
runtime-executed false. The next step is to let it review explicit spec
evolution proposals, not only loop pattern facts.

** 10.06.52.50 Control-Plane Runtime Surface
The word "runtime" in this layer means a runtime surface, not Scheme doing the
heavy execution.

#+begin_src mermaid
sequenceDiagram
  participant User as User or Agent
  participant Scheme as poo-flow Scheme control plane
  participant Audit as Human Audit Loop
  participant Marlin as Marlin/Rust runtime

  User->>Scheme: author S-expression POO policy/profile/spec
  Scheme->>Scheme: compose POO objects with C3
  Scheme->>Scheme: validate slots, safety, budget, graph gates
  Scheme->>Audit: submit spec/profile/eval/policy delta when mutation is needed
  Audit-->>Scheme: approved / rejected / escalated / changes-requested
  Scheme->>Marlin: emit validated control-plane runtime manifest
  Marlin->>Marlin: execute heavy runtime work
  Marlin-->>Scheme: receipts, telemetry, diagnostics
#+end_src

The manifest should expose:
- resolved POO object graph identity;
- C3-derived role/slot precedence;
- selected modules and features;
- workflow DAG or loop pattern facts;
- sandbox profile and capability constraints;
- session, memory, and durability policy refs;
- budget ceilings and denylist unions;
- human gates and approval evidence;
- graph diagnostics: reachability, dependency cones, cycle checks, and gate
  coverage;
- schema, producer, consumer, request id, and =runtime-executed = #f=.

** 10.06.52.60 POO Policy Composition Requirement
An agentic OS runs multiple modules at once. A real runtime policy is not owned
by only workflow, only sandbox, or only session.

The policy graph composes facts from:
- workflow/funflow: task DAG, evals, result protocols, runtime handoff shape;
- sandbox: filesystem, resources, capabilities, backend intent;
- session: lineage, memory, durability, communication, transform receipts;
- loop-engine: cadence, maker/checker, autonomy ceiling, retry policy;
- loop-governor: multi-loop collision, ownership, budget, denylist;
- human-audit: approved spec/profile/eval/policy mutation evidence;
- external feedback: evidence receipts and product direction signals.

POO and C3 make these fragments composable. Validators make them legal. Graph
analysis makes them explainable. Marlin executes only after the handoff bundle
is valid.

** 10.06.52.70 Scientific Implementation Plan
The implementation should advance in small, falsifiable slices.

Phase 1: documentation and contract closure.
- Add this document as the canonical three-loop plan.
- Cross-link =07= loop engineering, =11= Marlin runtime handoff, =33= graph
  algorithms, and =44= documentation contract.
- Ensure framework and loop design docs include Mermaid diagrams.

Acceptance:
- Org documents pass format checks.
- The three-loop model has one Mermaid flowchart and one protocol sequence
  diagram.

Phase 2: Spec Evolution Proposal objects.
- Add Scheme-side POO objects for =external-feedback-receipt=,
  =spec-change-proposal=, and =spec-evolution-review-item=.
- Keep them report-only and side-effect-free.
- Projection payloads are final alists only.

Acceptance:
- Focused tests prove proposals cannot mutate config directly.
- Invalid proposal shape fails before manifest handoff.

Phase 3: Human Audit integration.
- Extend =src/loops/human-audit.ss= from loop-pattern review to explicit
  spec-evolution review items.
- The review item must be a POO object whose slots describe proposal identity,
  source evidence, requested mutation kind, required approver, decision state,
  and runtime handoff eligibility.
- The object must remain review-only. It cannot write config, schedule work,
  mutate profile state, or call Marlin.
- Ordinary policy may add stricter human gates. It may not remove a human gate
  required by safety, budget, external feedback, or downstream Marlin consumer
  contracts.
Acceptance:
- Focused tests prove approved, rejected, escalated, and changes-requested
  decisions are represented as values.
- Tests prove review items keep =runtime-executed = #f=.
- Invalid review decisions fail before manifest projection.

Phase 4: Runtime manifest bridge.
- Put the three-loop facts into the existing runtime manifest discovery plane.
- The manifest should expose report-only handoff facts, not a scheduler.
- =poo-flow= owns the Scheme control-plane projection.
- =marlin-agent-core= owns the Gerbil ABI consumer and Rust execution bridge
  after it depends on =poo-flow= through =gerbil.pkg=.
Acceptance:
- A focused manifest test proves loop, audit, graph, sandbox, session, and
  budget facts are discoverable from one manifest object.
- The manifest names the consumer contract without pretending that =poo-flow=
  owns the Marlin runtime ABI.
- Projection payloads are final alists only at the boundary; user-facing
  construction remains POO-native and functional.

Phase 5: Graph and validator gates.
- Add graph diagnostics for proposal source paths, affected policy cones,
  required human gates, and downstream runtime handoff coverage.
- Keep finite workflow DAG cycle checks hard.
- Allow explanation cycles only in feedback and review graphs, where a rejected
  or changes-requested proposal can return to developer feedback.
Acceptance:
- Tests distinguish workflow DAG cycles from feedback explanation cycles.
- Missing Human Audit coverage blocks mutation-capable manifest projection.
- Graph receipts identify the validator that blocked the proposal.

Phase 6: Documentation and downstream user closure.
- Add user-interface scenarios that show real use, not introspection-only toys:
  CI/CD policy change, nono/cube sandbox capability escalation, funflow eval
  update, session memory durability escalation, and loop-governor budget change.
- Keep examples as POO objects and functional combinators. Do not introduce
  alist DSLs or record DSLs as user-facing syntax.
- Document the downstream path for Marlin:
  =marlin-agent-core/crates/*/gerbil/gerbil.pkg= depends on =poo-flow=;
  Marlin's Gerbil side consumes the manifest and owns ABI tests.
Acceptance:
- User-interface scenarios compile through focused =gxtest= files.
- Org docs pass lint.
- The handoff docs explicitly say Scheme is report-only and Marlin executes.

** 10.06.52.80 Non-Goals
- Do not implement heavy runtime scheduling in Scheme.
- Do not let external feedback directly mutate specs.
- Do not make a raw alist DSL for spec evolution.
- Do not bypass Human Audit with ordinary agent policy.
- Do not make Mermaid decorative; diagrams are part of the design contract.

** 10.06.52.90 Next Code Slice
The next code slice should be narrow:

1. Add POO-native spec evolution proposal objects.
2. Add a proposal-to-human-audit review projection.
