* 10.06.02 Marlin Gerbil Modules Audit
** 10.06.02.05 Current Migration Status
Status: implemented for the shared module facade.

=poo-flow= now carries the Marlin-inspired surface locally in =src/modules/module-system.ss=: =poo-module-interface=, =pooModules=, =poo-import=, =poo-imports=, =poo-local-source=, =poo-source-ref=, =pooEvalModules=, and =pooModuleSystemPresentation=. Direct symbol/string imports remain closed-world dependencies. Import specs with inline module profiles join the module closure before doctor or activation.

This deliberately does not copy Marlin's concrete policy-pack furniture or deck-runtime dependencies. =poo-flow= keeps task registries, flow registries, activation, and run-config lowering as local ownership boundaries.

Marlin's Gerbil module layer is useful because it models user-facing module
objects in the same language family. It is not the owner of =poo-flow='s core.

** 10.06.02.20 Observed Shape
The Marlin module core exposes:
- module interface objects carrying schemas and metadata;
- module config objects with imports, extensions, scripts, options, and
  metadata;
- source references for local or future package sources;
- defaulting behavior where config fields fall back to interface defaults.

The important pattern is that the user API mirrors typed config records while
keeping the module object inspectable.

** 10.06.02.30 Useful Ideas To Borrow
*** Interface Defaults
Marlin separates an interface object from a user config object. Missing config
fields can inherit interface defaults.

For =poo-flow=, this maps to:
- descriptor prototype defaults through Gerbil POO;
- module descriptor slots for imports, registries, and options;
- future schema descriptors for module options.

*** Structured Imports
Marlin has source-reference helpers rather than treating imports as plain file
paths everywhere.

For =poo-flow= first slice, imports are symbolic and explicit. Later slices can
add structured source refs without changing the activation contract.

*** User-Facing Object Shape
Marlin returns POO objects for module configuration. =poo-flow= should keep
this pattern because Gerbil POO is the reason to use Scheme here.

** 10.06.02.40 What Not To Copy
- Do not copy Marlin runtime ownership into =poo-flow=.
- Do not let module config objects hide execution behavior.
- Do not bind =poo-flow= modules to Marlin scripts or backend profiles.
- Do not make imports load code implicitly in the first slice.

** 10.06.02.50 Design Consequence
=poo-flow= modules use Marlin as a vocabulary reference:
- =interface= becomes future schema/UX metadata;
- =imports= becomes explicit symbolic dependencies;
- =extensions= becomes descriptor registries;
- =options= becomes run-config options.

The actual implementation is local: =poo-module-descriptor= wraps existing
=TaskFamilyRegistry= and =FlowDeclarationRegistry= values.
