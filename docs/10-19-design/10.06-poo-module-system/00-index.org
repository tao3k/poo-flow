* 10.06 poo-flow Module System
** 10.06.10 Purpose
This document pack defines the module-system layer for =poo-flow=. The package is not a port of Doom Emacs and not a copy of Marlin's Gerbil module API. =poo-flow= remains the core control plane. External systems are reference material for interface shape, diagnostics, and modular organization.

The implementation lives in =src/modules/module-system.ss=. It depends on core registries and config APIs, but core does not import or re-export it.

** 10.06.30 Local Evidence
The module-system design builds on existing =poo-flow= contracts:
- =src/core/roles.ss= owns Gerbil POO role composition and C3-facing slot probing.
- =src/core/task.ss= owns task-family descriptors and immutable task-family registries.
- =src/core/flow.ss= owns flow-declaration descriptors and immutable flow-declaration registries.
- =src/core/config.ss= accepts task and flow registries at the =run-config= boundary.
- =src/modules/module-system.ss= owns source refs, catalog resolution, module activation, and import validation.
- =src/extensions/*= demonstrates descriptor bundles as extension modules.
** 10.06.40 Delivered Surface
The implemented module-system surface includes:
- source refs: =local=, =package=, =registry=, and =generated=;
- immutable module catalogs;
- source resolution from catalog refs to already-built descriptors;
- POO module descriptors;
- the thin =poo-module= macro;
- module doctor reports and alist projection;
- descriptor activation into run-config registries;
- typed failures for missing imports and missing catalog sources;
- duplicate and empty-contribution diagnostics for agent/CLI inspection.
** 10.06.50 Non-Goals
- Do not vendor Doom Emacs, Marlin, or backend runtime code into =src=.
- Do not let core import or re-export the module-system layer.
- Do not execute tasks during module activation.
- Do not let macro DSLs perform loading, validation, or runtime work.
- Do not replace existing task/flow registries with a parallel registry type.
