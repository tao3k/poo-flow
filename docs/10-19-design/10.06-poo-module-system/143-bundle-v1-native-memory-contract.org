#+title: RFC 0143: Bundle v1 Native Memory Contract
#+status: accepted
#+phase: 3H

* 结论

Bundle v1 是 POO Flow 控制面向 runtime handoff 的正式 native memory
contract。它不是 JSON schema，也不是把旧 =runtime_v0= 改名后的兼容层。

本 RFC 固定以下边界：

- Scheme/POO 侧仍然拥有对象、类型、合约、角色、能力、policy、strategy、
  adapter 与 projection 的组合语义；
- lowering 一次性把这些 POO-native value 投影到一块连续 foreign arena；
- C 边界只接收固定宽度的 descriptor、typed region 与 compact id；
- runtime 和下游语言不能持有 Scheme、Python 或其他语言 GC heap pointer；
- Bundle 打开前执行一次 =O(n)= 验证，执行期 component 查询为
  =O(log n)=，且返回 arena 内的只读行，不复制 payload；
- 当前 draft =runtime_v0= 不因此成为稳定 v1，也不增加任何 compatibility
  wrapper。只有 Runtime、Python adapter、Cedar 与 Lean/Evidence 全部切到新
  边界后，才能一次性删除旧 ABI 并冻结 Runtime v1。

* 为什么必须先分离 Bundle v1

编译器和构建系统审计确认，当前 runtime C ABI 已经使用 native struct、arena、
handle 和 batch API，并不是 JSON FFI；但是它仍明确声明
=ABI_MAJOR=0=、=organization-bundle.draft.3=，Gerbil benchmark 也明确标记
=payload-zero-copy=false= 与 =abi-v1-frozen=false=。

因此问题不是“把 JSON 换成 struct”，而是旧 ABI 把稳定 Bundle、runtime
lifecycle、批处理与 evidence mediation 混在一个 draft namespace。直接将宏从
0 改成 1 会制造一个不可证明的假稳定版本。Phase 3H 先固定 Bundle v1 的内存
事实，Runtime v1 再直接消费它。

* 架构边界

#+begin_src mermaid
flowchart LR
  POO["POO-native Domain Case<br/>object + type + contract"]
  ALG["Functional composition<br/>role + capability + policy + strategy"]
  LOWER["Scheme lowering<br/>pure plan + one arena materialization"]
  ARENA["Registered foreign arena<br/>immutable typed regions"]
  BUNDLE["Bundle v1 descriptor<br/>256 bytes"]
  RUNTIME["Future Runtime v1<br/>Rust kernel or downstream runtime"]
  PY["Python production adapter<br/>CFFI"]
  OTHER["Rust / TypeScript adapters<br/>downstream-owned"]
  CEDAR["Cedar input projection"]
  LEAN["Lean / Evidence obligations"]

  POO --> ALG --> LOWER
  LOWER --> ARENA
  LOWER --> BUNDLE
  BUNDLE --> RUNTIME
  ARENA --> RUNTIME
  RUNTIME --> PY
  RUNTIME --> OTHER
  RUNTIME --> CEDAR
  RUNTIME --> LEAN
#+end_src

重要的是：C struct 不是新的用户 DSL。正常用户路径仍然只能通过 POO object、
prototype composition、slot operator、functional combinator 和 declarative macro
得到 lowering plan。C layout 只存在于 runtime handoff boundary。

* 固定布局

| 类型 | 大小 | 用途 |
|------+--------+--------------------------------------------------|
| =compact_id= | 16 B | 跨语言稳定身份，不传 symbol/string pointer |
| =region= | 24 B | arena offset、length、stride、alignment |
| =symbol_entry= | 32 B | interned id 到 UTF-8 metadata slice |
| =component_entry= | 200 B | Domain Case 的完整组合身份列 |
| =edge_entry= | 80 B | case 内 component 关系与顺序 |
| =evidence_entry= | 96 B | contract 到 proof/evidence obligation |
| =descriptor= | 256 B | Bundle identity、digest、epoch 与五个 region |

布局只使用 =uint16_t/uint32_t/uint64_t= 和固定数组；禁止 =size_t=、C bitfield、
语言对象 pointer、隐含 enum width 与 flexible array member。header 内的
=_Static_assert= 是 ABI gate，不允许编译器静默改变布局。

* Scheme type 到 native type

| Scheme / POO value | Bundle v1 native projection |
|--------------------+---------------------------------------------|
| object identity | 128-bit =compact_id= |
| symbol | interned =compact_id=；可选 UTF-8 只进入 metadata region |
| fixnum / integer | lowering 时 range-check 后写入固定宽度 unsigned integer |
| boolean / option | 已声明的 flag bit；未知 bit fail closed |
| object/type/contract binding | 一个 =component_entry= 的独立 identity column |
| role/capability | 同一 component row 的固定 column |
| policy/strategy | 同一 component row 的固定 column |
| adapter/projection | 同一 component row 的固定 column |
| composition relation | 按 case/source/order 排序的 =edge_entry= |
| proof obligation | 按 case/obligation 排序的 =evidence_entry= |

这不是逐字段跨 FFI 调用。Scheme 先用 functional traversal 计算有序 immutable
rows，再一次性写入 foreign arena。跨边界只有 descriptor 和 arena view。

* 所有权与生命周期

#+begin_src mermaid
stateDiagram-v2
  [*] --> Planned: "POO-native lowering plan"
  Planned --> Materialized: "allocate foreign arena once"
  Materialized --> Registered: "register arena and generation"
  Registered --> Open: "validate descriptor and sorted tables"
  Open --> Open: "O(log n) read-only lookup"
  Open --> Released: "last Bundle/session handle released"
  Released --> Recycled: "arena generation advances"
  Recycled --> [*]

  Open --> Rejected: "bounds / layout / sort / duplicate failure"
  Rejected --> Released
#+end_src

- arena 由 adapter 在 foreign/native memory 中分配；不得直接暴露移动 GC heap；
- descriptor 和 region 不拥有内存，只描述已注册 arena；
- runtime 在 Bundle/session handle 存活期间只读借用 arena；
- release 完成前 adapter 不得 free、resize 或 mutate；
- recycle 必须推进 generation，旧 generation 的任何访问都 fail closed；
- 推荐 arena base 64-byte aligned；每个 region 仍声明并验证其实际 type alignment；
- callback 不得从任意 runtime worker thread 反向进入 Gerbil GC。释放由显式
  lifecycle API 驱动，而不是隐式 finalizer。

* 索引与复杂度

=component_entry= 按 =(case_id, component_id)= 严格升序；重复 key 不是覆盖，而是
Bundle 无效。打开时线性验证排序，之后使用 lower-bound binary search。

#+begin_src mermaid
flowchart TD
  KEY["case_id + component_id"] --> RANGE["components typed region"]
  RANGE --> MID["lower-bound midpoint"]
  MID -->|"key is smaller"| LEFT["left half"]
  MID -->|"key is larger"| RIGHT["right half"]
  MID -->|"equal"| VIEW["const arena-backed row"]
  LEFT --> MID
  RIGHT --> MID
#+end_src

这使成百上千 agent、不同 role/capability/policy/strategy/adapter 的组合不依赖
语言 hash table，也不在热路径重新解释 POO object。当前基准在 65,536 rows 上
执行 1,000,000 次 lookup，实测约 0.425 秒；这个数字是本机工程门禁证据，不是
跨机器性能承诺。

* Gerbil 与 gerbil-mysql 审计带来的约束

本地研究 checkout：

- =.data/gerbil= at =0618730d4e30bf3f48538d6b09722ad3b6056414=；
- =.data/gerbil-mysql= at =847b4b61747152e505dfc72acee4438009302c5f=。

=gerbil-mysql/db/_mysql.scm= 是 Gambit FFI compilation unit，而不是可直接 import
的普通 Gerbil module。审计探针也证明：错误地交给 =gxc= 时，即使 stderr 已有
=*** ERROR=，进程仍可能返回 0；真正的 Gambit C generation 使用 =gsc -c=。
因此后续 Scheme lowering/build macro 必须同时检查：退出码、stderr error marker、
预期 C/object artifact 三个条件，不能只信 exit code。

Bundle v1 不复用数据库 binding 常见的逐次 string conversion 模式。symbol 和
string 只在 materialization 阶段 intern；hot path 只访问 compact id 和 typed row。
这也避免了 Gerbil FFI 每次 crossing 的 char-string 转换成本。

* Python 生产验证边界

Python 继续是生产级 runtime adapter 验证，不是第二个 runtime owner。Phase 3H
使用 CFFI ABI mode 直接 =dlopen= Bazel 构建的 Bundle v1 dylib，并证明：

- Python 构造的 typed rows 可被 C validator 接收；
- lookup 返回的地址就是原 CFFI arena 中的 row；
- missing key 返回稳定 status；
- 全路径不使用 =ctypes=，也不编码/解析 JSON。

现有 runtime-v0 wheel 仍把 =POO_FLOW_RUNTIME_V0_LIBRARY=、
=_runtime_v0_cffi= 和 =libpoo_flow_runtime_v0= 固化在安装面。Runtime v1 迁移必须
一次性重写并移除这些名字，不能保留 alias。

* Fail-closed 条件

以下任一条件立即拒绝 Bundle：

- schema major 不是 1，或 minor 高于实现支持版本；
- descriptor size 不足、未知 flags、reserved field 非零；
- digest 全零；
- arena length 与 descriptor 不一致；
- region overflow、stride 错误、alignment 非二次幂或 offset 未对齐；
- symbol metadata 越界；
- component table 未排序或 key 重复；
- lookup 前发现 component region 本身不满足固定布局。

* Phase 3H 落地与后续冻结门禁

Phase 3H 已落地：

- =poo_flow/bundle_v1.h= 的正式 native layout；
- descriptor validator；
- =O(log n)= component lookup；
- C harness、benchmark 与 Python CFFI validation；
- Bazel 独立 package，不污染 Scheme build truth。

Runtime v1 冻结前仍必须完成：

1. POO-native Scheme lowering object 与 functional arena materializer；
2. Runtime v1 lifecycle 直接消费 Bundle v1，不增加 v0 wrapper；
3. Cedar typed input 与 Lean/Evidence obligation table 的端到端消费；
4. Python wheel、loader、CFFI module 与环境变量一次性迁移；
5. 删除全部 runtime-v0 public headers、symbols、tests、docs 与安装 artifact；
6. Gerbil AOT/RSS、ASan/UBSan、symbol surface、wheel install 与 benchmark gates 全绿。

只有六项同时满足，Runtime 才能声明 ABI v1 frozen。
