* 10.06.54 Runtime ABI Pressure Relief Interface

** 10.06.54.10 Purpose

This document turns the LoopX pressure test and Marlin ABI evidence into a
runtime-facing interface plan for =poo-flow=.

The goal is not to make every runtime language understand Gerbil, POO objects,
C3 slot precedence, functional strategy combinators, or downstream UI concepts.
The goal is to make =poo-flow= a pressure-relief layer:

#+begin_src text
POO-native policy families
  + functional strategy combinators
  + validation and readiness gates
  + compact packet projection
  -> runtime language consumes stable packet protocol
  -> runtime executes with its own scheduler, IO, memory model, and telemetry
#+end_src

Marlin is the reference runtime case. Other runtimes should be able to follow
the same packet contract without copying Marlin internals.

** 10.06.54.20 Terms

- Author UI :: the user-facing =poo-flow= surface: POO-native policy families,
  profile fragments, and functional strategy combinators.
- Runtime UI :: the runtime-author surface: capability negotiation, packet
  validation, action batches, and receipt batches.
- Native adapter :: an optional high-performance binding layer such as C ABI,
  Rust =cdylib=, or language-specific generated binding.
- Pressure relief :: the act of resolving rich authoring semantics before the
  runtime boundary, so the runtime receives normalized facts rather than a
  second user-interface language.

** 10.06.54.30 Boundary Ownership

=poo-flow= owns:
- POO object construction, inheritance, slot precedence, and role composition;
- functional strategy composition;
- eligibility, gate, capability, fallback, and readiness decisions;
- user-facing diagnostics and conflict reasons;
- policy packet projection;
- public/private projection rules;
- conformance fixtures for runtime authors.

Runtime owners own:
- thread pools, async runtimes, shared queues, and blocking bridges;
- process, filesystem, connector, network, and sandbox IO;
- durable append, checkpoint, repair, index, and compaction work;
- compare-and-swap writes, leases, and conflict resolution;
- telemetry ingestion and high-volume persistence;
- runtime-owned execution receipts.

Language SDKs own:
- binding ergonomics for a host language;
- packet encoding and decoding;
- memory/allocator rules for native adapters;
- transport selection;
- conformance test invocation.

** 10.06.54.40 Runtime Access Levels

Every runtime should consume the same semantic packet families. The transport
can vary by performance need.

- L0 manifest transport :: file/stdout manifests for CLI, debugging, and
  low-frequency handoff. This has the highest boundary overhead and the lowest
  integration cost.
- L1 service transport :: local process, socket, HTTP, or gRPC transport for a
  service runtime, daemon, or remote worker. This has moderate overhead and easy
  deployment.
- L2 native transport :: C ABI or Rust native library transport for in-process
  runtime integration. This has low overhead and requires an explicit memory
  contract.
- L3 optimized native transport :: C ABI plus epoch backing and batch packets
  for high-frequency policy projection. This has the lowest repeated
  marshalling cost.

The public author UI must not change across levels. A user writes the same POO
policy families. Runtime authors choose a transport level.

** 10.06.54.50 Minimal Runtime Interface

A runtime language should only need to implement this narrow surface:

*** 10.06.54.50.10 =describe-capabilities=

Return runtime id, language, transport class, ABI id/version, exported symbols,
supported policy families, supported action kinds, batch limits, and receipt
contracts.

*** 10.06.54.50.20 =prepare-epoch=

Accept stable profile, graph, policy-scope, and root ids. Return an epoch handle
or receipt proving which scalar backing can be reused across repeated calls.

*** 10.06.54.50.30 =validate-packet=

Accept one policy/action packet or batch. Return schema, readiness, capability,
privacy, and conflict diagnostics. This operation must not execute runtime work.

*** 10.06.54.50.40 =execute-action-batch=

Accept validated action packets. Execute only runtime-owned effects. Return
compact receipt refs and diagnostics.

*** 10.06.54.50.50 =return-receipt-batch=

Return monotonic receipt updates, telemetry refs, final status, lease state, new
watermarks, and runtime-owned failure data.

All names here are semantic operations. A transport may spell them as functions,
commands, request types, messages, or native symbols.

** 10.06.54.60 Packet Families

*** 10.06.54.60.10 Runtime Capability Descriptor

=runtime-capability-descriptor.v1= declares what a runtime can accept.

Required fields:
- =runtime-id=
- =runtime-language=
- =runtime-owner=
- =transport-class=
- =abi-id=
- =abi-version=
- =exported-symbols=
- =supported-policy-families=
- =supported-action-kinds=
- =supported-receipt-contracts=
- =max-action-batch-size=
- =supports-epoch-backing?=
- =supports-borrowed-bytes?=
- =supports-durable-leases?=
- =supports-readiness-gates?=
- =runtime-executed?=

For descriptors produced by =poo-flow=, =runtime-executed?= is always =#f=.
Runtime-returned descriptors may include runtime-owned probe receipts.

*** 10.06.54.60.20 Policy Profile Packet

=policy-profile-packet.v1= is the normalized form of POO policy families.

Required fields:
- =profile-id=
- =policy-epoch=
- =source-refs=
- =policy-families=
- =queue-policy=
- =continuation-policy=
- =tool-batch-policy=
- =model-route-policy=
- =evidence-policy=
- =failure-policy=
- =memory-policy=
- =human-gate-policy=
- =self-evolution-policy=
- =projection-policy=
- =privacy-class=
- =diagnostics=
- =runtime-executed?=

This packet is where =poo-flow= removes pressure from runtime authors. It
contains final policy facts. It does not contain POO objects, slot resolution
rules, macros, closures, or user-facing DSL fragments.

*** 10.06.54.60.30 Runtime Action Packet

=runtime-action-packet.v1= is the executable intent handed to a runtime.

Required fields:
- =packet-id=
- =request-id=
- =profile-id=
- =policy-epoch=
- =action-kind=
- =graph-ref=
- =policy-scope=
- =root-ref=
- =candidate-refs=
- =gate-state=
- =required-capabilities=
- =readiness-requirements=
- =readiness-receipts=
- =evidence-refs=
- =idempotency-key=
- =watermark=
- =lease-ref=
- =conflict-policy=
- =budget-ref=
- =privacy-class=
- =fallback-policy=
- =diagnostics=
- =runtime-executed?=

Allowed =action-kind= values for the first closure:
- =run=
- =observe=
- =quiet-skip=
- =ask-owner=
- =self-repair=
- =fallback=
- =handoff=
- =validate-only=

=poo-flow= may emit non-executing action packets such as =quiet-skip= and
=ask-owner=. This keeps readiness failures, missing symbols, unsupported
capabilities, and human gates visible before a runtime attempts execution.

*** 10.06.54.60.40 Runtime Receipt Batch

=runtime-receipt-batch.v1= is returned by the runtime.

Required fields:
- =runtime-id=
- =batch-id=
- =received-packet-ids=
- =accepted-packet-ids=
- =rejected-packet-ids=
- =action-receipts=
- =capability-receipts=
- =readiness-receipts=
- =lease-receipts=
- =telemetry-refs=
- =watermark=
- =status=
- =diagnostics=
- =runtime-executed?=

Runtime-returned receipt batches may set =runtime-executed?= to =#t=. Scheme
side manifests and handoff packets remain =#f= until a runtime-owned receipt
states otherwise.

** 10.06.54.70 Pressure-Relief Rules

1. No POO objects cross the runtime boundary.
2. No Scheme closures, continuations, or macro artifacts cross the runtime
   boundary.
3. No private transcript body crosses the runtime boundary unless represented
   by an explicit handle or evidence ref.
4. Slot precedence and role composition are resolved before packet projection.
5. Functional combinators emit decision receipts before packet projection.
6. Native ABI readiness is a strategy gate, not a late runtime surprise.
7. Missing symbols, unsupported capabilities, expired leases, and privacy
   violations produce diagnostic packets or receipts before execution.
8. Stable epoch keys should be reusable across repeated runtime calls.
9. Per-decision payloads should stay small and batchable.
10. Receipt updates must be monotonic and replayable.

** 10.06.54.80 Marlin Reference Mapping

Marlin gives the first concrete mapping for this interface:

| =poo-flow= packet concept | Marlin reference concept |
|---------------------------+--------------------------|
| =runtime-capability-descriptor.v1= | native ABI id/version/exported-symbols and readiness plan |
| =policy-profile-packet.v1= | =LoopPolicyProfile= plus governance/evidence policy |
| native readiness requirement | =GraphNativeAbiRequirement= |
| native readiness result | =GraphNativeAbiReadinessReceipt= |
| policy proposal | =GraphPolicyProposal= and validation receipt |
| executable run intent | =GraphLoopRunRequest= / =GraphLoopExecutionRequest= |
| continuation action | =GraphLoopContinuationAction= and continuation receipt |
| queue lanes | =GraphLoopInputLane= and input queue receipt |
| tool batch policy | =GraphToolBatchExecutionMode= and batch receipt |
| runtime snapshot | =RuntimePlanSnapshot= and =SessionRuntimeSnapshot= |
| runtime concurrency policy | =TokioRuntimePolicy= and =TokioRuntimePolicyReceipt= |
| high-frequency native calls | Gerbil native request epoch backing and conversion profile |

The mapping rule is strict: =poo-flow= should align with these shapes where
they are already semantically right, but Marlin remains the runtime owner.
=poo-flow= should not import Marlin's Tokio policy into the author UI.

** 10.06.54.90 Other Runtime Language SDK Pattern

Other languages should not build one-off integrations against Gerbil internals.
They should implement a small SDK around the packet protocol.

SDK minimum:
- generated packet types or schema bindings;
- a capability descriptor implementation;
- a validator for schema version, required fields, action kinds, and privacy
  class;
- a receipt-batch encoder;
- conformance fixtures;
- optional C ABI/native adapter for L2/L3 performance.

The lowest-cost SDK can start with L0 or L1. The high-performance SDK can add
L2/L3 without changing the packet semantics.

** 10.06.54.100 Implementation Roadmap

M1. Specify packet schemas as Org contracts and Scheme schema constants.

M2. Define POO-native policy families for:
- work lane;
- monitor;
- user gate;
- capability gate;
- claim/lease;
- handoff;
- self-repair;
- projection/frontstage.

M3. Define functional strategy combinators for:
- eligibility;
- gate blocking;
- capability matching;
- ABI readiness;
- monitor cadence;
- fallback routing;
- outcome floors;
- self-repair;
- no-op backoff.

M4. Add packet projectors:
- POO profile -> =policy-profile-packet.v1=;
- strategy decision -> =runtime-action-packet.v1=;
- runtime response -> =runtime-receipt-batch.v1= normalization.

M5. Add Marlin reference fixtures:
- one accepted readiness packet;
- one missing-symbol packet;
- one human-gate =ask-owner= packet;
- one =quiet-skip= packet;
- one batched =run= handoff packet;
- one runtime receipt batch.

M6. Add a conformance suite for runtime authors:
- schema round trip;
- capability descriptor validation;
- readiness gate behavior;
- batch idempotency behavior;
- monotonic receipt update behavior;
- public/private projection behavior.

M7. Add performance receipts:
- packet size;
- batch size;
- repeated epoch scalar reuse;
- per-decision payload marshalling count;
- conversion diagnostics comparable to Marlin's native conversion profile.

** 10.06.54.110 Acceptance Criteria

This design is closed when:

1. A downstream user can author LoopX-shaped behavior as POO policies and
   combinators without writing a new control-plane engine.
2. Marlin can consume the projected packets without reconstructing POO
   semantics.
3. A second runtime language can implement the conformance suite without
   importing Gerbil internals.
4. The native path can optimize ABI transport through batching and epoch reuse
   without changing author UI.
5. Runtime failures return receipts that can be replayed, audited, projected,
   and rendered in public/private surfaces.

** 10.06.54.120 Immediate Next Slice

The next code slice should be narrow:

1. Add Scheme constants and row projectors for the three packet families.
2. Add POO policy prototypes for work lanes, gates, monitors, claim/lease, and
   handoff.
3. Add one user-interface scenario that projects a LoopX-shaped lifecycle into
   =policy-profile-packet.v1= and =runtime-action-packet.v1=.
4. Add focused tests proving the projected packet is report-only and
   =runtime-executed?= remains =#f=.

This slice should not execute Marlin and should not introduce a runtime
scheduler in Scheme.

** 10.06.54.140 Remaining Gaps

The next implementation slice should not add another runtime language yet. It
should first strengthen the packet interface around strategy facts:

1. Move work-lane, gate, monitor, claim/lease, and handoff policy rows into
   POO-native policy families.
2. Add functional strategy combinators that produce explicit decision receipts
   before packet projection.
3. Add conformance fixtures for accepted readiness, missing-symbol readiness,
   =quiet-skip=, batched =run=, and runtime receipt batches.
4. Decide whether high-frequency native transports need epoch-backed scalar
   tables in Scheme, or whether that remains entirely runtime-owned.
5. Split or justify large test owners before requiring the full scoped policy
   gate as a closure condition for this area.
