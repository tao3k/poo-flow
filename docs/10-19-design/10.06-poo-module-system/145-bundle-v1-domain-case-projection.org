#+title: RFC 0145 - Domain Case Projection into Bundle v1
#+startup: overview

* Status

Accepted and implemented.

This RFC defines the Phase 3J boundary that projects an accepted POO-native
Domain Case runtime handoff plan into the canonical Bundle v1 lowering model.
It is the final semantic projection before Phase 3K writes a foreign-memory
arena through the native C ABI.

* Decision

The public projection operation is:

#+begin_src scheme
(feature-bundle-v1-project-domain-case bundle-epoch runtime-handoff-plan)
#+end_src

The operation accepts one POO =FeatureRuntimeHandoffPlan=. It does not accept a
raw alist, record DSL, JSON document, positional tuple, or an independent set of
partially related arguments.

The runtime handoff plan remains the owner of the resolved runtime boundary.
The projection derives its adapter binding, policy binding, accepted Domain
Case assembly, Bundle identifier, Runtime Bundle handoffs, and Evidence
handoffs from that one owner.

The result is a POO-native projection object. Rejected projections are also
POO-native values and retain sufficient owner context for diagnostics.

* Architectural boundary

The projection does not create another runtime. POO Flow owns the Scheme
control plane, Bundle v1 lowering, and the native C binding contract. Python is
a production-grade validation runtime for the complete interface. Rust,
TypeScript, and other downstream runtimes adapt the same C interface without
changing the Scheme semantic owners.

The projection is a semantic bridge. It does not serialize the graph and does
not perform foreign-memory allocation.

* Owner graph

The input owners have distinct responsibilities:

- =FeatureDomainCaseAssembly= owns the accepted Domain Case, its components,
  the composition plan, the Domain Case identity, adapter requirements,
  projection requests, and selected projections.
- =FeatureAdapterProjectionBinding= owns resolved adapter and projection
  bindings.
- =FeaturePolicyStrategyBinding= owns effective policy and strategy algebra
  selections.
- =FeatureRuntimeHandoffPlan= owns the resolved Runtime Bundle, Cedar, and
  Evidence handoffs presented to a runtime boundary.
- Bundle v1 lowering owns compact identifiers, deterministic layout, edge
  lowering, evidence lowering, and arena-size planning.

These owners are connected by resolved semantic identifiers. They are not
joined by list position.

* Public POO surface

The module exports the projection operation, projection predicates and
accessors, and named diagnostics. Repeated internal object families are
generated by bounded declarative macros that expand to ordinary POO objects.

Raw alists are used only as an internal construction detail behind the
POO-native surface. Callers neither construct nor inspect an alist DSL.

The accepted projection retains:

- the source runtime handoff plan;
- the accepted Domain Case assembly;
- the canonical Bundle v1 lowering plan;
- the owner-indexed Runtime Bundle handoffs;
- the resolved Evidence obligations;
- projection diagnostics.

A rejected projection retains the available runtime owner, assembly, and
lowering context instead of collapsing failure into a string.

* Canonical mapping

The projection uses separate compact-identifier domains even when two domains
currently derive from the same semantic source. This prevents accidental type
confusion and keeps future identity refinements ABI-compatible within Bundle
v1.

| Bundle v1 domain | Semantic source |
|------------------+-----------------|
| Bundle | Runtime handoff plan Bundle identifier |
| Case | Domain Case assembly identifier |
| Component | Canonical component identifier |
| Object | Component identifier in the object domain |
| Role | Component identifier in the role domain |
| Type | Component type identifier in the type domain |
| Contract | Component type identifier in the contract domain |
| Capability | Resolved Runtime Bundle handoff capability |
| Adapter | Resolved Runtime Bundle handoff provider module |
| Projection | Resolved Runtime Bundle handoff projection schema |
| Policy | Effective policy algebra identifier |
| Strategy | Effective strategy algebra identifier |
| Evidence obligation | Evidence handoff identifier |
| Evidence type | Evidence projection schema |
| Proof system | Evidence provider module |

Object and role identifiers deliberately use the component identifier in
different compact-ID domains. An arbitrary role prototype does not yet expose
a stable semantic identity that can safely become an ABI identifier.

Missing optional runtime, policy, or strategy values use versioned Bundle v1
sentinels. Sentinels remain domain-specific and cannot alias a real value from
another compact-ID domain.

* Composition and inheritance

Accepted component order becomes Bundle v1 composition order. Component
parent relations become directed Bundle v1 edges:

- source: child component;
- target: parent component;
- relation: =poo-flow.bundle-v1.component-parent=.

No legacy graph ABI or compatibility wrapper is introduced.

* Matching algorithm

Runtime Bundle handoffs must be matched to their component owners without
quadratic lookup.

The implementation:

1. computes compact owner identifiers;
2. performs a stable functional sort of components;
3. performs a stable functional sort of Runtime Bundle handoffs;
4. joins both sorted streams with one pure linear merge;
5. restores canonical composition order for lowering.

For =n= components and =m= runtime handoffs, owner matching costs
=O(n log n + m log m)= time and =O(n + m)= auxiliary space. The merge itself
is linear. The implementation uses functional combinators and does not require
a mutable hash table in the Scheme control plane.

* Evidence projection

Only resolved Evidence handoffs enter Bundle v1 evidence lowering. Each
obligation maps:

- obligation identifier from the Evidence handoff identifier;
- contract identifier from the resolved contract;
- evidence type from the resolved projection schema;
- proof system identifier from the provider module.

Runtime Bundle and Cedar handoffs do not become evidence obligations merely
because they are present in the same handoff plan.

* Fail-closed rules

Projection is rejected when any of these conditions holds:

- the Bundle epoch is invalid;
- the runtime handoff plan is rejected;
- the Domain Case assembly is unavailable or rejected;
- a Runtime Bundle handoff refers to an unknown component owner;
- more than one Runtime Bundle handoff claims the same component owner;
- canonical Bundle v1 lowering rejects the derived values.

Rejection is deterministic and returns structured POO diagnostics. It never
falls back to positional pairing, JSON coercion, or legacy graph semantics.

* Native ABI implications

Phase 3J produces the exact semantic values required by the Bundle v1 memory
contract. Phase 3K may therefore focus on allocation, alignment, checked
offsets, slice descriptors, lifetime, and C ownership without rediscovering
Domain Case meaning.

The hot path remains:

#+begin_example
Scheme values -> Bundle v1 native layout -> C ABI values -> runtime values
#+end_example

JSON may exist as an observability or interchange format outside the native
hot path, but it is not the runtime FFI representation.

* Verification

The implementation is verified by:

- 35 checks in the Phase 3J integration suite;
- 123 checks across the Phase 3G, Phase 3I, and Phase 3J combined regression;
- independent AOT compilation producing 9 artifacts;
- a guarded full build with no authoritative =*** ERROR= output;
- a 1,000-component benchmark with 1,000 Runtime Bundle handoffs,
  1,000 evidence obligations, 999 inheritance edges, and 2,000 resolved
  handoffs.

* Consequences

This decision establishes one canonical semantic route from a composed Domain
Case to Bundle v1. It increases reuse of POO components, contracts, types,
policies, strategies, adapters, and projections without merging their distinct
owners.

The next phase is Phase 3K: implement the checked foreign arena writer and
expose the resulting Bundle v1 slices through the native C binding surface.
