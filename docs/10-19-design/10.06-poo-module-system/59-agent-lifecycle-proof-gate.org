* 10.06.59.10 Purpose
This document records the proof gate for AI strategy/policy lifecycle safety.
It covers the path from a created session to sandbox placement, loop policy,
subagent linkage, tool permission containment, receipt evidence, and
counterexample rejection.

The target is not to prove the Scheme interpreter. The target is to prove that
an agent-authored policy is safe enough to become a reusable public policy
surface.
* 10.06.59.20 Lifecycle Axes
The lifecycle gate has four runtime policy axes and three evidence axes.

| Axis | Required evidence |
| Session | session created, parent session bound |
| Sandbox | sandbox attached, scope contained |
| Tool | tool permissions contained by sandbox/profile policy |
| Loop | start guarded, exit defined, handoff guarded |
| Subagent | every subagent is parented by the session |
| Receipt | tests, proof receipts, observability receipts |
| Counterexample | unsafe cases rejected before publication |

#+begin_src mermaid
flowchart LR
  Session["Session evidence"] --> Sandbox["Sandbox scope"]
  Sandbox --> Tools["Tool permissions"]
  Session --> Loop["Loop policy"]
  Session --> Subagent["Subagent topology"]
  Tools --> Gate["Lifecycle gate"]
  Loop --> Gate
  Subagent --> Gate
  Receipt["Test / proof / observability receipts"] --> Gate
  Counterexample["Rejected bad cases"] --> Gate
  Gate --> Surface["Reusable public policy surface"]
#+end_src
* 10.06.59.30 Mathematical Contract
The gate is intentionally stronger than a successful happy-path scenario. A
policy can only enter the reusable public surface when all lifecycle facts hold.
#+begin_src latex
\operatorname{AcceptedLifecycle}(a)
  \Rightarrow
    \operatorname{SessionCreated}(a)
    \land \operatorname{ParentSessionBound}(a)
    \land \operatorname{SandboxAttached}(a)

\operatorname{AcceptedLifecycle}(a)
  \Rightarrow
    \operatorname{SandboxScopeContained}(a)
    \land \operatorname{ToolPermissionsContained}(a)

\operatorname{AcceptedLifecycle}(a)
  \Rightarrow
    \operatorname{LoopStartGuarded}(a)
    \land \operatorname{LoopExitDefined}(a)
    \land \operatorname{LoopHandoffGuarded}(a)

\operatorname{AcceptedLifecycle}(a)
  \Rightarrow
    \operatorname{SubagentsParented}(a)
    \land \operatorname{DependencyClosed}(a)
    \land \operatorname{CounterexampleRejected}(a)
#+end_src

This is the policy-level version of the profile library gate. Profile proof
answers whether a shared profile is reusable. Lifecycle proof answers whether
an AI action graph can safely run under a session, sandbox, loop strategy, and
subagent topology.
* 10.06.59.40 Rejected Cases
The proof layer should discover missing evidence by proving contradictions for
accepted gates with unsafe facts.
#+begin_src latex
\operatorname{AcceptedLifecycle}(a)
  \land \neg \operatorname{SubagentsParented}(a)
  \Rightarrow \bot

\operatorname{AcceptedLifecycle}(a)
  \land \neg \operatorname{SandboxScopeContained}(a)
  \Rightarrow \bot

\operatorname{AcceptedLifecycle}(a)
  \land \neg \operatorname{ToolPermissionsContained}(a)
  \Rightarrow \bot

\operatorname{AcceptedLifecycle}(a)
  \land \neg \operatorname{LoopExitDefined}(a)
  \Rightarrow \bot
#+end_src

The Scheme scenario tests mirror these bad cases. A public policy with proof
receipts still fails the gate if it has an unparented subagent, sandbox scope
leak, tool permission overflow, or loop without exit and handoff guards.
* 10.06.59.45 Topology Proof Layer
The lifecycle gate first checks whether required evidence is present. The
topology proof layer gives that evidence mathematical structure.

Scope containment is not a string comparison. It is a preorder over scope
objects.
#+begin_src latex
\operatorname{ScopeOrder} = (S,\leq)

\forall s \in S,\; s \leq s

a \leq b \land b \leq c \Rightarrow a \leq c
#+end_src

Tool permission is scoped by that order. A requested tool is valid only when it
is allowed in the child scope admitted by the lifecycle.
#+begin_src latex
\operatorname{RequestedTool}(a,t)
  \Rightarrow
    \operatorname{AllowedInScope}(t,\operatorname{ChildScope}(a))
#+end_src

Loop policy is a transition obligation. A start edge must imply both an exit
edge and a handoff edge.
#+begin_src latex
\operatorname{LoopStart}(a,l)
  \Rightarrow
    \operatorname{LoopExit}(a,l)
    \land \operatorname{LoopHandoff}(a,l)
#+end_src

Subagent topology is a session relation, not just a list of names.
#+begin_src latex
\operatorname{SubagentOf}(a,c)
  \Rightarrow
    \operatorname{Session}(c) = \operatorname{Session}(a)
#+end_src

The Lean module =PooFlowProof.PooC3.AgentLifecycleTopology= owns these
relations. The Scheme gate projects compact topology facts so bad cases remain
observable in tests, while Lean owns the deeper relation laws.
* 10.06.59.50 Implementation Boundary
The Scheme owner is =src/module-system/agent-lifecycle-gate.ss=. It creates POO
receipt and gate objects, then projects final Lean facts at the boundary.

The Lean owner is =PooFlowProof.PooC3.AgentLifecycle=. It treats the Scheme
facts as propositions and proves that accepted public lifecycle gates imply the
session/sandbox/loop/subagent safety conditions.

The scenario owner is =t/scenarios/agent-lifecycle-gate-test.ss=. It produces
test receipts for both accepted and rejected lifecycle cases without becoming a
package compile target.
