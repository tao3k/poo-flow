* 10.06.83 FunFlow use-module Macro IR
This document fixes the FunFlow macro-expansion contract after the Python runtime and LangGraph-alignment work exposed an interface risk: the expanded form must not become a second public workflow language.

The public user interface remains =use-module funflow=. Internal source IR, plan IR, and runtime projections exist only under that facade.

** 10.06.83.10 Decision
FunFlow authoring is POO-native first and =use-module= first.

A user-facing flow should look like a module declaration that names domain facts:

#+begin_src scheme
(use-module funflow
  (flow custom-ci
    (policy ci-policy)
    (stage source
      (checkout commit))
    (stage checks
      (parallel
        (lint source)
        (unit source #:fanout 4)))
    (stage gate
      (prove sandbox-policy))
    (stage artifact
      (package source))))
#+end_src

The macro expansion target is not a public =(funflow-ir ...)= language. The first stable lowering target is a normalized =use-module funflow= form that the module system owns.

#+begin_src scheme
(use-module funflow
  (normalized-flow
    (name custom-ci)
    (policy ci-policy)
    (source-map ...)
    (plan ...)
    (runtime-contract anyio-runtime-v1)))
#+end_src

The normalized form is still part of the FunFlow module facade. It is not a user extension surface unless the module system explicitly exposes a named helper for it.

** 10.06.83.20 Why This Is The Right Boundary
Prefect is useful as an engineering reference because it keeps a stable authoring facade while hiding its engine graph, orchestration metadata, retries, timeouts, state, and receipts behind that facade. The lesson for this project is not to copy Python decorators. The lesson is to keep a durable user-level authoring contract and make the engine plan an internal product of that contract.

For =poo-flow=, the durable facade is not =@flow=. It is =use-module= plus POO-native module declarations and functional strategy combinators.

This preserves three project invariants:

1. Users compose flows through POO objects, policy families, and named functional helpers.
2. Macro expansion lowers into ordinary module-system structures, not a parallel alist, record, or graph DSL.
3. Runtime languages receive a compiled projection. They do not own strategy, policy semantics, or user syntax.

** 10.06.83.30 Layer Contract
The FunFlow path has five layers.

#+begin_src text
user form
  -> normalized use-module funflow form
  -> POO-native module object
  -> FunFlow plan/projection
  -> runtime receipt
#+end_src

The layers have different ownership.

| Layer | Owner | Public? | Responsibility |
|-------+-------+---------+----------------|
| User form | =src/modules/funflow/*= and module facade | yes | Pleasant flow authoring, stage names, policy references, combinators |
| Normalized =use-module funflow= form | module-system macro layer | no | Expansion-time validation, defaults, source map, static shape |
| POO-native module object | module-system runtime layer | project internal | Slot ownership, profile integration, policy-family attachment |
| Plan/projection | FunFlow projection layer | runtime boundary | Stable node table, edge table, policy table, runtime contract |
| Runtime receipt | Python/AnyIO or other runtime | boundary output | Execution status, timing, trace, durable checkpoint reference |

The internal plan may have source IR and plan IR names in code, but those names must remain implementation names. They are not the public user interface.

** 10.06.83.40 Macro Expansion Rules
The macro layer should follow these rules:

- Expand public forms into normalized =use-module funflow= declarations.
- Keep source locations and declaration names in a source map.
- Validate fixed syntax at expansion time when possible.
- Generate POO-native objects or functional builders, not raw alist DSLs.
- Avoid repeated large object literals in generated code.
- Avoid cons-heavy hot paths in the runtime projection.
- Intern or cache repeated identifiers when plan construction is hot.
- Keep policy references as references until projection time; do not copy policy families into every node.
- Make runtime contracts explicit at the projection boundary.

The macro should own only the repeated structural frame. Domain decisions stay visible at the call site.

** 10.06.83.50 Plan Shape
The runtime-friendly plan should be a POO-native object or a POO-owned projection with table-oriented slots:

#+begin_src scheme
(.o
  (kind 'funflow-plan)
  (name 'custom-ci)
  (version 1)
  (node-table '#(...))
  (edge-table '#(...))
  (policy-table '#(...))
  (effect-table '#(...))
  (runtime-contract 'poo-flow.anyio.v1)
  (source-map '#(...)))
#+end_src

The important property is not the exact constructor spelling. The important property is that the plan is built from the normalized =use-module funflow= form and is cheap for runtimes to consume.

For Python, the existing dataclass DAG should be treated as a runtime projection object. It is not the canonical user API and should not drive Scheme surface design.

** 10.06.83.60 Error And Debug Contract
FunFlow expansion must preserve enough source evidence to make failures actionable:

- expansion errors point at the user clause, not the generated plan;
- duplicate stage names are reported before runtime execution;
- missing dependencies report the producing stage name and the consuming stage name;
- runtime receipts include the flow name, node id, stage name, runtime contract, elapsed time, and source-map key;
- compile/debug modes can print the normalized =use-module funflow= form and the final plan projection separately.

This also supports the existing performance-gate concern: when compile time or memory usage regresses, the user should be able to see which declaration family produced the large expansion or expensive projection.

** 10.06.83.70 Non-Goals
This design does not introduce a public =funflow-ir= DSL.

It does not move policy strategy, validation semantics, or workflow composition into Python.

It does not require JSON as the Scheme-to-runtime path. JSON-like data may appear only as a final projection format when a specific runtime boundary requires it.

It does not replace POO-native policy families with Python validation libraries.

** 10.06.83.80 Acceptance Criteria
The design is satisfied when:

- a custom flow can be authored through =use-module funflow=;
- macro expansion lowers to a normalized =use-module funflow= form before runtime projection;
- the normalized form constructs or references POO-native module objects and policy families;
- Python/AnyIO receives only the compiled runtime projection;
- tests cover user-form authoring, normalized-form diagnostics, projection shape, and runtime execution;
- benchmarks compare authoring/projection overhead and runtime execution overhead separately.
