* 10.06.31 Profile Object User Interface
** 10.06.31.10 Decision
The primary =user-interface/custom/*/profiles/*.ss= surface is Gerbil POO
syntax, not a second profile DSL.

=use-module= remains the module/package wrapper. Inside it, users write POO
profile prototypes directly:

#+begin_src scheme
(use-module nono-sandbox
  (.def (agent/session @ nono-sandbox-profile
                       network capabilities resources metadata)
    network: (allowlisted-network "github.com" "crates.io")
    capabilities: '(process-run filesystem-read tmpdir)
    resources: =>.+ runtime-volume-resources
    metadata: => (lambda (super-metadata)
                   (append super-metadata
                           '((intent . coding-agent)
                             (stage . interactive)))))

  (.def (agent/session-cache @ agent/session
                             capabilities resources metadata)
    capabilities: => (lambda (super-capabilities)
                       (append super-capabilities '(cache-mount)))
    resources: =>.+ runtime-volume-resources
    metadata: => (lambda (super-metadata)
                   (append super-metadata
                           '((cache . cargo))))))
#+end_src

This is concise because it is exactly the POO model:
- =agent/session= is a prototype object.
- =@= names the self object in Gerbil POO.
- =nono-sandbox-profile= is the super prototype.
- =agent/session-cache= extends =agent/session= by naming it as super.
- =network capabilities resources metadata= are self-slot accessors.
- =slot:= overrides or defines a slot.
- ~=>~ transforms an inherited slot value with one value expression.
- ~=>.+~ extends an inherited nested POO object slot.

Do not introduce a public shorthand that only hides =.def= or default slot
accessors. That kind of macro makes simple cases shorter, but it weakens the
advanced teaching story because users no longer see the actual POO relationship
they are extending.

This replaces the old row DSL. The public interface should not preserve
compatibility for:
- =(use-module nono-sandbox :config (profiles ...))=
- row operators such as =:override=, =:append=, =:remove=
- raw resource-policy alists as an authoring language
- raw =(lambda (self super) ...)= hooks in user profiles

Rows and alists may remain only as internal materialized values at runtime,
manifest, bridge, or Marlin ABI boundaries.

** 10.06.31.20 Evidence
=.data/grafonnix/README.md= states that most functions return POP objects and
that those objects are extended by methods on the object. In =poo-flow= the
Gerbil POO syntax already gives the better version of that idea: the object is
the profile, and extension is ordinary prototype inheritance.

=.data/gerbil-poo/object.ss= and =.data/gerbil-poo/doc/poo.md= give the matching
Gerbil mechanism:
- =.def= defines a named prototype object.
- in =(.def (child @ parent slot ...) ...)=, =@= is self and =parent= is super.
- extra names after the super are self-slot accessors.
- =.mix= composes objects/prototypes.
- ~=>~ transforms inherited slot values.
- ~=>.+~ extends an inherited child POO object slot.

Therefore the user interface should expose POO declarations directly. Wrapper
functions such as =with-network= or =with-resources= are less faithful and less
concise than slot definitions. Shorthand macros that only remove =.def= or
default slot accessors have the same problem at a smaller scale: they hide the
real object relationship without adding new POO capability.

** 10.06.31.30 Public Vocabulary
The upstream public library provides prototype values and slot helper values,
not a new chain API.

Backend super profiles:
#+begin_src scheme
nono-sandbox-profile
docker-sandbox-profile
cubeSandbox-profile
sandbox-profile
#+end_src

Reusable resource prototypes:
#+begin_src scheme
runtime-volume-filesystem
runtime-volume-resources
readonly-project-workspace-resources
readwrite-project-workspace-resources
#+end_src

Network values:
#+begin_src scheme
(deny-network)
(allowlisted-network "github.com" "crates.io")
#+end_src

Common profile slots:
#+begin_src scheme
network:
capabilities:
resources:
metadata:
#+end_src

The important rule is: these are POO slots, not row names. Agents should prefer
slot operators over helper verbs:

#+begin_src scheme
capabilities: => (lambda (super-capabilities)
                   (append super-capabilities '(cache-mount)))
resources: =>.+ runtime-volume-resources
metadata: => (lambda (super-metadata)
               (append super-metadata '((cache . cargo))))
#+end_src

For a root resource object, define the child object directly:

#+begin_src scheme
(.def runtime-volume-resources
  filesystem: runtime-volume-filesystem
  cpu: 2
  memory: "4Gi")
#+end_src

Do not use =filesystem: =>.+= on a root resources object; ==>.+= needs an
inherited parent slot.

** 10.06.31.40 Profile-To-Profile Extension
Extending one profile into another is a direct POO declaration:

#+begin_src scheme
(use-module nono-sandbox
  (.def (agent/session @ nono-sandbox-profile
                       network capabilities resources metadata)
    network: (allowlisted-network "github.com" "crates.io")
    capabilities: '(process-run filesystem-read tmpdir)
    resources: =>.+ runtime-volume-resources
    metadata: => (lambda (super-metadata)
                   (append super-metadata
                           '((intent . coding-agent)
                             (stage . interactive)))))

  (.def (agent/session-cache @ agent/session
                             capabilities resources metadata)
    capabilities: => (lambda (super-capabilities)
                       (append super-capabilities '(cache-mount)))
    resources: =>.+ runtime-volume-resources
    metadata: => (lambda (super-metadata)
                   (append super-metadata '((cache . cargo))))))
#+end_src

No separate =(extends ...)= form is needed. POO already has the relationship:

#+begin_src scheme
(.def (child @ parent slot ...) ...)
#+end_src

That is the core user story. The parent profile is the super prototype.

** 10.06.31.50 Self/Super In The User Interface
The user-facing self/super mapping is the Gerbil POO mapping:

#+begin_src scheme
(.def (agent/session-cache @ agent/session
                           capabilities metadata)
  capabilities: => (lambda (super-capabilities)
                     (append super-capabilities '(cache-mount)))
  (metadata (super-metadata)
    (append (super-metadata)
            `((cache . cargo)
              (effective-capabilities . ,capabilities)))))
#+end_src

Read this as:
- =agent/session-cache= is the object being defined.
- =@= names self.
- =agent/session= is the super prototype.
- =capabilities= and =metadata= are self-slot accessors.
- =capabilities: => (lambda ...)= transforms the inherited capabilities slot.
- =(metadata (super-metadata) ...)= names the inherited metadata computation.
- the body can combine =super-metadata= with the self slot =capabilities=.

This is both the advanced extension story and the teaching story. There is no
need to show a raw =(lambda (self super) ...)= in user files.

** 10.06.31.90 Scenario Coverage
The root =user-interface/custom/my-module/config.ss= should expose representative
fragments so agents can learn the current surface by reading real code:
- =profiles/session.ss=: root nono profile and interactive session policy.
- =profiles/task.ss=: task profile derived from a backend profile, plus a
  cache-enabled child profile.
- =profiles/cicd.ss=: CI/CD check/build/release/promotion profiles.
- =profiles/agent-sandbox-audit.ss=: advanced profile derivation metadata using
  explicit inherited slot computation.
- =profiles/object-extension.ss=: backend object extension and slot-operator
  behavior for reusable sandbox policy.
- =cases/funflow-cicd.ss=: functional flow/pipeline scenario that consumes the
  profile references without executing runtime work.
- =cases/poo-introspection.ss=: formal Funflow CI/CD gate that carries POO
  authoring observability in check metadata.

These files are not examples outside the product; they are the public teaching
surface for users and agents.

** 10.06.31.100 Agent-Facing Writing Rules
Agents should write this interface by following these rules:
- Start with =(use-module <backend> ...)= in each profile fragment.
- Write profile objects with =.def=.
- Use =(.def (name @ super slot ...) ...)= for every profile.
- Use the backend super profile for roots, for example =nono-sandbox-profile=.
- Use a named profile as the super when deriving, for example =agent/task=.
- Use =slot:= for direct slot values.
- Use ~=>~ with one value expression when transforming an inherited slot.
- Use a lambda for captured append/prepend behavior, for example
  =metadata: => (lambda (super-metadata) ...)=.
- Use ~=>.+~ only when extending an inherited nested POO object.
- Use =(slot (super-slot) ...)= only when the slot needs explicit inherited
  computation.
- Keep POO primitive names out of user slot names: do not use =object?=,
  =.o=, =.def=, =.ref=, or =.slot?= as slots.
- Keep slot initializers from shadowing their slots. Prefer names such as
  =slot-name=, =slot-present?=, and =slot-current-value= over =slot=,
  =present?=, and =value= when writing helper objects.
- Keep raw alists at ABI/materialization boundaries, not in the authoring
  surface.
- Do not write =with-*= wrapper methods, row operators, =:derive=, =:compute=,
  raw alists, or nested =.call=.

** 10.06.31.105 POO Authoring Observability
POO observability is a strict, non-POO edge. It must not dereference the object
being diagnosed. The concrete adapter is
=poo-flow-poo-slot-authoring-observations= from
=src/module-system/observability.ss=.

It catches two classes that otherwise become expensive runtime debugging:

- =self-referential-slot-initializer=: source shape like =(.o slot: slot)=.
- =primitive-shadow-slot=: slot names that shadow POO primitives such as
  =object?=, =.ref=, or =.slot?=.

User-interface scenarios must expose this through formal module configuration,
not by constructing standalone receipt objects. A Funflow authoring gate carries
the observability request as check metadata:

#+begin_src scheme
(use-module funflow
  :config
  (.def (funflow/profile-authoring @ funflow-check
                                   check-name profile-ref command-vector
                                   artifact-outputs cache-intents
                                   result-protocol runtime-mode
                                   observability observes guards report)
    check-name: 'profile-authoring
    profile-ref: 'ci/build
    command-vector: '("gxtest"
                      "t/sandbox-core-profile-authoring-diagnostics-test.ss")
    artifact-outputs: '(poo-authoring-report)
    cache-intents: '(gerbil-build-cache)
    result-protocol: '(read :lines)
    runtime-mode: 'manifest-handoff
    observability: 'poo-slot-authoring-summary
    observes: '(profile-slots object-extension funflow-check-metadata)
    guards: '(self-referential-slot-initializer primitive-shadow-slot)
    report: 'authoring-report)

  (.def (funflow/readiness @ funflow-check
                           check-name profile-ref command-vector
                           artifact-outputs result-protocol runtime-mode
                           dependency-refs)
    check-name: 'funflow-readiness
    profile-ref: 'ci/check
    command-vector: '("gxtest"
                      "t/user-interface-cicd-pipeline-run-test.ss")
    artifact-outputs: '(runtime-manifest-readiness)
    result-protocol: '(read :lines)
    runtime-mode: 'manifest-handoff
    dependency-refs: '(profile-authoring))

  (.def (funflow/poo-authoring-gate @ funflow-pipeline
                                    pipeline-name checks metadata)
    pipeline-name: 'poo-authoring-gate
    checks: (list funflow/profile-authoring funflow/readiness)
    metadata: '((scenario . poo-authoring-gate)
                (authoring-style . gerbil-poo-native))))
#+end_src

The observation and summary output is plain alist data by design. It remains
readable even when the problem being diagnosed is a lazy POO slot recursion.
That alist output belongs to adapters and tests. The user-facing config carries
the stable request surface as POO slots: =observability:=, =observes:=,
=guards:=, and =report:=.
