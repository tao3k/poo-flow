#+title: AC-10 Integrated Qualification And Release Assurance Manifest Design
#+date: 2026-07-12
#+status: approved-design

* Decision

AC-10 joins the already qualified AC-01 through AC-09 owners into one
reproducible vertical-slice qualification.  It introduces a canonical release
assurance manifest and an orchestration harness; it does not introduce another
semantic model, runtime, proof kernel, or compatibility ABI.

The manifest is a POO-native Scheme value projected into a deterministic
machine-readable receipt.  It references evidence produced by existing owners
and never converts “test command succeeded” into a stronger semantic claim.

Every arrow is backed by a typed identity or receipt.  The runner rejects a
path whose neighboring owners cannot prove that they refer to the same Bundle,
epoch, token/effect binding, semantic root, execution root, and proof vector.

* Ownership

| Owner | Responsibility |
|-------+----------------|
| Existing AC-01–09 semantic owners | Produce canonical facts and focused receipts |
| Scheme assurance manifest | Declare claims, TCBs, levels, evidence references, exclusions, and qualification state |
| Qualification runner | Execute owner commands and assemble receipts without reinterpreting them |
| Native C harness | Prove installed ABI, lifecycle, fault, memory, and hot-path properties |
| Python runtime harness | Prove installed production CFFI behavior and cross-language conformance |
| Lean package | Check only the named formal claims and their bound identities |
| Release reviewer | Decide ABI freeze and cutover from the complete receipt set |

Downstream Rust, TypeScript, and other adapters remain consumers of the C ABI.
They are not AC-10 runtime implementations.

* ReleaseAssuranceManifest v1

The canonical manifest contains:

- release candidate identity and source revision;
- toolchain and supported-host identity;
- canonical Bundle, schema, epoch, vector, and theorem-set identities;
- an ordered claim registry;
- the owner and immutable evidence reference for every claim;
- the claim-specific TCB and assurance level;
- exclusions, nondeterminism, and unsupported-host declarations;
- functional, security, fault, performance, and hygiene gate results;
- ABI version and explicit freeze decision.

Public construction uses POO objects, prototype extension, slot operators, and
functional combinators.  Raw alists, record DSLs, patch languages, and shell
fragments are not the ordinary manifest API.

The serialized receipt may be JSON for tooling interchange, but JSON is not a
native runtime ABI and is never passed through the C hot path.

* Claim model

Each claim is fail-closed and declares exactly one assurance level:

| Level | Meaning |
|-------+---------|
| L0 Interface | Version, layout, and negotiation only |
| L1 Conformant | State-machine, lifecycle, ownership, and trace conformance |
| L2 Evidenced | Authorization, receipt, replay, checkpoint, and evidence-chain binding |
| L3 Mediated | Declared effects are constrained by isolation or independent observation |
| L4 Formally Linked | A named Lean theorem binds the declared semantic identities |

Levels are cumulative but claim-specific.  A Lean proof does not promote an
actual-effect claim to L3, and a production runtime does not promote an
unproved statement to L4.

Required TCB families are Bundle semantic, authorization, actual effect,
evidence history, and Lean claim.  A global “trust everything” TCB is invalid.

* Qualification fixture

The first canonical fixture exercises one allow path and its negative family:

1. construct and validate a canonical five-facet Bundle;
2. negotiate runtime-v0 and publish a deterministic batch;
3. consume through an isolated installed Python wheel;
4. authorize a normalized effect with a one-shot token;
5. mediate and seal Strict or Batched evidence;
6. verify roots, inclusion, replay state, and durability profile;
7. encode the exact ProofCaseVector v1;
8. check the AuthorizedEffectToken theorem set;
9. assemble the release assurance manifest from immutable receipts.

The negative family covers deny, replay, revocation, substitution, crash,
timeout, cancellation, retry, restore, indeterminate outcome, fork, truncation,
missing obligation, malformed vector, and stale proof identity.

The runner owns process orchestration only.  It may normalize exit status,
duration, environment identity, and artifact digests; it may not synthesize a
semantic pass when an owner did not emit the required evidence.

* Performance qualification

The complete matrix retains batch sizes 1, 8, 32, 128, and 1024 and payloads
empty, 1 KiB, 64 KiB, and 1 MiB across allow, deny, success, runtime error,
timeout, cancellation, and restore where the owner supports that dimension.

Receipts bind crossings, allocations by runtime domain, bytes copied, GC time,
p50/p99, throughput, resident memory, batch utilization, and evidence sealing
overhead.  Unsupported dimensions are explicit, never silently omitted.

The current single-host numbers remain baselines.  ABI v1 cannot freeze until a
recorded budget exists and the real hot path has no unexplained regression.

* Private-bypass audit

Qualification fails if production Python can execute without a negotiated
native context, if an effect can bypass authorization/mediation, if a proof
digest is caller-injected, if the retired graph ABI reappears, or if checkout
imports substitute for the installed consumer.

Diagnostic routes remain explicit and cannot produce executable authorization
or a release-qualified L3 claim.

* Exit criteria

1. One canonical fixture traverses every accepted boundary from a clean build.
2. All focused AC-01–09 owners remain green.
3. Every release claim binds a declared owner, TCB, level, evidence identity,
   exclusions, and failure state.
4. Functional, security, fault, differential, determinism, performance,
   sanitizer, leak, installation, and hygiene receipts are complete.
5. No private runtime, authorization, evidence, or proof bypass exists.
6. The ABI freeze decision is explicit and independently reviewable.
7. AC-11 remains blocked until this receipt and migration approval exist.

* Non-goals

- implementing a second Rust or TypeScript runtime;
- replacing Python as the production-capable validation runtime;
- moving semantic or theorem ownership into the runner;
- adding a graph ABI compatibility wrapper;
- freezing latency budgets from one machine;
- treating L4 as a global product badge.
