* 10.06.49 Scenario Test Owner Matrix
This document is P0 for the execution plan in =10.06.48=. It makes implementation measurable by tying each real scenario to a user-facing surface, source owner, validation surface, status, and next work package.
** 10.06.49.10 Ground Rules
1. Scenarios must be real user workflows. Introspection examples are not sufficient as acceptance surfaces.
2. Public configuration must stay 100% POO-native. Use =.def=, POO objects, prototype extension, slot operators, and functional constructors as the visible path.
3. Functional flow and loop composition should be value-in/value-out until an explicit report-only runtime or Marlin handoff boundary.
4. Alists are allowed only as final manifest, ABI, receipt, snapshot, or report payloads.
5. Validation surfaces name files for the current testing framework and native =gxtest= single-file selection. This document does not prescribe legacy wrapper commands.
6. Missing scenario coverage is a planning fact, not a blocker to documenting the gap. Later packages must close the gap before changing the affected public contract.
7. Every implementation slice that introduces or changes a public scenario must land the matching scenario test and performance scenario fixture in the same slice when performance is part of the contract.
8. Scenario roots must be registered through =src/testing/project.ss= as harness-native =gxtest= roots. Do not build scenario aggregates by importing source-only test files as library modules.
9. The performance aggregate root =t/performance-tests.ss= must explicitly execute each scenario performance suite with =run-tests!=; importing performance modules is not enough evidence that their gates ran.

** 10.06.49.50 Boundary Notes
- This repository owns Scheme-side strategy, composition, validation, POO-native extension surfaces, and report-only receipts.
- =marlin-agent-core= owns scheduling, backend runtime, LLM/tool execution, ABI consumer integration, and heavy execution.
- The bridge between them should be runtime manifest discovery plus named request/receipt contracts, not hidden direct calls from user code.
- The downstream Gerbil package should depend on =poo-flow= through =gerbil.pkg= and adapt ABI boundaries there.
