#+title: RFC 141: Feature Adapter Capability and Projection Binding
#+status: implemented

* Decision

Phase 3F introduces three POO-native declaration values and one immutable
binding receipt:

- =FeatureAdapterCapability= declares what a provider module can satisfy;
- =FeatureAdapterRequirement= declares what a Feature requires;
- =FeatureProjectionRequest= declares which Domain Case projection and schema a
  Feature needs;
- =FeatureAdapterProjectionBinding= records the resolved capability and
  projection pairs.

The binding phase performs no adapter loading, C call, Python invocation, Cedar
evaluation, projection execution, or evidence emission.  It is a pure Scheme
control-plane qualification step.

* Owner audit and repaired closure gap

Gerbil compiled-module metadata proves that the existing Domain Case owner
exports:

- =poo-flow-case-projection= and its predicate;
- =domain-case-select-projections=;
- =poo-flow-domain-case-close=;
- =poo-flow-domain-case-project=;
- the closed =projection-catalog= and =selected-projection-ids= slots.

A runtime object probe established an important boundary fact: closing a Domain
Case with the default empty selected-id list produces an empty projection
catalog, even when components declare projections.  Phase 3D previously kept
Feature projection values beside the assembly but did not pass selection ids to
the Domain Case owner.  A later binder could therefore discover a declaration
but could not use =poo-flow-domain-case-project= on the closed Case.

Phase 3F repairs that gap at the correct boundary.  The assembly now accepts
only POO =FeatureProjectionRequest= values, extracts their projection ids, and
passes them to the original Domain Case close operation.  There is still one
closure, one canonical key, one selected catalog, and one cache owner.

Raw projection symbols are not compatibility aliases.  They are rejected before
Domain Case closure with an =invalid-feature-projection-request= diagnostic.

* POO capability model

An adapter capability contains:

1. =capability-id=;
2. =provider-module-id=;
3. =contract-id=;
4. =contract-version=.

An adapter requirement contains:

1. =requirement-id=;
2. the required =capability-id=;
3. the exact =contract-id=;
4. the exact =contract-version=.

A projection request contains:

1. =request-id=;
2. the Domain Case =projection-id=;
3. the required =schema-id=.

The ordinary declaration surface is fully POO-native:

#+begin_src scheme
(defpoo-feature-adapter-capability python-runtime-capability
  (capability-id 'python-runtime-v1)
  (provider-module-id 'poo-flow/python-runtime)
  (contract-id 'poo-flow.runtime-c)
  (contract-version 1))

(defpoo-feature-adapter-requirement python-runtime-requirement
  (requirement-id 'python-runtime-requirement)
  (capability-id 'python-runtime-v1)
  (contract-id 'poo-flow.runtime-c)
  (contract-version 1))

(defpoo-feature-projection-request runtime-envelope-request
  (request-id 'runtime-envelope-request)
  (projection-id 'runtime-envelope-view)
  (schema-id 'runtime-envelope-v1))
#+end_src

The functional constructors produce the same immutable POO values and support
generated Feature families.  Raw alists and record DSLs are not public input
forms.

* Module-owned adapter capability catalog

The audited Feature and Domain Case control-plane exports do not provide a
generic adapter capability catalog.  Phase 3F therefore introduces an explicit,
module-owned POO catalog instead of inventing a global registry.

#+begin_src scheme
(defpoo-feature-adapter-capability-catalog runtime-adapter-catalog
  (catalog-id 'runtime-adapter-catalog)
  (capabilities python-runtime-capability cedar-policy-capability))
#+end_src

Catalog construction validates every POO capability, rejects semantic duplicate
ids, and builds an immutable-after-construction hash index.  Lookup is expected
O(1); the catalog remains an explicit argument to the binder and respects
Gerbil's single-instantiation module reuse.

* Binding algorithm

=feature-adapter-projection-binding= consumes:

1. an accepted =FeaturePolicyStrategyBinding=;
2. an accepted module-owned adapter capability catalog.

It follows these steps:

1. recover the accepted assembly and its closed Domain Case;
2. validate adapter requirements and reject duplicate requirement ids;
3. resolve every capability id through the catalog index;
4. require exact adapter contract id and version equality;
5. index the already selected Domain Case projection catalog once;
6. validate projection requests and reject duplicate request ids;
7. resolve every projection id and require schema id equality;
8. produce ordered POO binding objects and one immutable top-level receipt.

The control-plane lookup work is O(C + A + P + R), where =C= is catalog size,
=A= is adapter requirement count, =P= is selected Domain Case projection count,
and =R= is projection request count.  No requirement-by-catalog or
request-by-projection quadratic scan is introduced.

* Projection non-execution invariant

Binding stores the original =poo-flow-case-projection= object next to its POO
request.  It never reads or invokes the projection's projector procedure.

Only a later, explicit runtime handoff may call
=poo-flow-domain-case-project=.  This ensures projection execution remains under
the existing Domain Case diagnostic and receipt boundary rather than becoming a
hidden Feature-system side effect.

* Rejection semantics

Structured POO diagnostics distinguish:

- invalid or duplicate adapter capability declarations;
- invalid or duplicate adapter requirements;
- missing adapter capabilities;
- adapter contract id/version mismatch;
- invalid or duplicate projection requests;
- a projection absent from the closed Domain Case catalog;
- projection schema mismatch;
- rejected upstream policy/strategy binding;
- rejected adapter capability catalog.

Partial success never changes the top-level result: the binding is ready only
when every adapter requirement and projection request resolves.

* Qualification gates

The implementation proves:

- projection requests are selected during the authoritative Domain Case close;
- raw projection syntax is rejected before closure;
- module-owned adapter catalog lookup preserves object identity;
- capability, contract, projection, and schema mismatches remain distinct;
- semantic duplicate strings are detected by equal-key hash indexes;
- upstream rejection prevents all resolution;
- binding does not invoke any projector;
- 256 adapter requirements and 256 projection requests resolve in dependency
  order under the Scheme memory guard;
- empty-root AOT execution does not use repository source fallback.

* Next mainline stage

policy/strategy and adapter/projection bindings.  The plan may name Bundle v1 C
ABI contracts, Python runtime providers, Cedar inputs, and Lean/Evidence
