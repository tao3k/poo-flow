#+TITLE: 10.04 Funflow Tutorial Result Ladder

* 10.04 Funflow Tutorial Result Ladder

** 10.04.00 Purpose

This document changes the alignment driver from source-shape parity to
tutorial-result parity. Funflow's tutorials show what users actually observe:
values printed by minimal flows, composed flows, custom task extensions,
word-count workflows, external configuration checks, Docker execution, and
runtime-backed artifacts.

Poo-flow implementation should now advance by result gates. Each gate must
state:
- the Funflow tutorial output being matched;
- the poo-flow behavior that produces an equivalent observable result;
- the command that proves the result in this repository.

** 10.04.10 Audited Tutorial Surface

The reference checkout under =.data/funflow= exposes these result sources:
- =funflow-tutorial/notebooks/Tutorial1.ipynb=: minimal pure flow, saved output
  =2= for running =(+1)= over input =1=.
- =funflow-tutorial/notebooks/QuickReference.ipynb=: minimal greeting output
  ="Hello Watson !"=, composition output ="Hello world"=, and cache-oriented
  repeated =Increment!= output.
- =funflow-tutorial/notebooks/Tutorial2.ipynb=: custom task extension output
  ="Kangaroo goes woop!woop!woop!woop!woop!woop!woop!"=.

  prints counts such as =a: 3=, =and: 3=, =lets: 2=, and =file: 1=.
- =funflow-tutorial/notebooks/ExternalConfig/ExternalConfig.ipynb=: config
  values from env/file/placeholder feed Docker args; missing config fails before
  expensive runtime work.
- =funflow-tutorial/notebooks/ErrorHandling.ipynb=: =tryE= catches Docker and
  local failures and routes them to handled branches.

- =funflow-examples/makefile-tool/README.md=: runnable example builds
  =makefile-tool=, enters =funflow-examples/makefile-tool/test=, runs the tool,

*** 10.04.27 Stage 7: Docker/CAS Real Runtime

Reference result:
- CCompilation prints =15= after Docker GCC execution.
- ExternalConfig prints configured Docker echo output.
- makefile-tool builds a C example and then =./hello= runs.

Poo-flow target result:
- Rust/runtime owns Docker execution, CAS materialization, filesystem movement,
  and process output.
- Scheme core owns descriptor registries, config preflight, request envelopes,
  response normalization, and receipts.
- Scheme core exposes a =runtime-command= descriptor boundary so procedure
  commands, process-backed commands, and later Rust commands share one
  invocation shape.
- Process-backed commands may use the stdout protocol: print one
  =poo-flow.runtime-response.v1= s-expression and let Scheme normalize it.
- Docker-specific declarations belong to =src/modules/docker.ss=.
- CAS/store lifecycle interpretation belongs to a store/CAS extension or Rust
  runtime, not to =src/core/runner.ss=.
- Core receipts may carry generic =adapter-result= evidence: task, kind,
  adapter operation, status, request id, and artifact handle.

Poo-flow precursor landed:
- =src/modules/docker.ss= owns =docker-task-family-descriptor=.
- =src/modules/docker.ss= owns =docker-flow= and =make-docker-task=.
- The extension installs its own registry, strategy capability, and adapter
  capability instead of changing core defaults.
- Docker request data carries image, command, args, volume bindings, and output
  policy as adapter-routed request data.
- CCompilation-style payloads can carry =gcc:9.3.0=, =gcc= arguments, mount
  bindings, and process-handle output policy.
- ExternalConfig-style payloads can carry rendered file/env/placeholder args
  without doing Docker execution in Scheme.
- =src/core/runner.ss= no longer emits =cache-hit= or =cache-request-only= for
  store operations; it emits generic adapter evidence so CAS policy can be
  layered outside core.
- =src/workflow/store.ss= owns =store-task-family-descriptor=,
  =store-flow=, =make-store-task=, and store-enabled run configs.
- Core defaults no longer install store task families or store capabilities;
  Store/CAS users opt in through the extension registry and strategy wrappers.
- Stage 7 now uses =make-stdout-runtime-command=, so the visible-result proof
  crosses a real process boundary before decoding the runtime response.
- The preferred process path is =make-stdout-runtime-command=, which removes
  per-call custom decoders from tutorial proof code.

Proof:
- =gxtest t/docker-descriptor-test.ss=, cases ="captures CCompilation-style
  docker payload"= and ="captures ExternalConfig-style rendered arguments"=.
- =gxtest t/tutorial-result-test.ss=, case ="stage 7 docker runtime command
  returns CCompilation visible result"=.

Remaining proof target:
- A future Rust-backed runtime command descriptor must replace the test process
  command, print the same stdout runtime-response schema, and produce visible
  output equivalent to =15= or =hello= without embedding Docker/CAS
  implementation in Scheme.
