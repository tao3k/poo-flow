#+title: Runtime Benchmark Surface

* Status

The graph-domain benchmarks remain historical/domain evidence.  Old C ABI hot
path claims are retired and do not qualify runtime v0.1 or the future ABI v1.

* Intent

Benchmarks are package module surfaces, not pyproject script expansion.

The benchmark layer must prove two things:

- LangGraph-compatible runtime semantics are covered by executable tests.
- POO Flow can go beyond parity through stronger functional composition, durable policy integration, nested subgraph projection, and measured C ABI hot paths.

* Composition suite

The composition suite currently emits line-oriented receipts for:

- strategy-combinator-pipeline
- durable-fanout-strategy
- nested-subgraph-stream-projection
- cabi-status-hot-call
- validation-workflow-round-trip

The C ABI scenarios intentionally separate the loaded binding hot path from the full validation workflow.  The hot path measures the low-level ABI call after the dylib is loaded.  The workflow path measures manifest validation and graph handoff as a higher-level runtime operation.

* AnyIO micro qualification

The current candidate validates and prepares one executor at each public
=abatch= boundary, reuses that prepared executor only within the batch, and
refreshes preparation for the next batch.  There is no permanent program
cache.  The immediately preceding same-semantics failing receipt is retained
below as the before baseline.

reference with the same capacity and action as the

* Large-state Program-to-executor copy qualification
** Scope and invariants
This qualification isolates the Python runtime's shallow root-dictionary copy at the async Program-to-executor boundary. It is not a C ABI qualification, does not define a JSON ABI or JSON handoff protocol, and does not make the line-oriented benchmark receipt part of any runtime ABI.

Each timing pair starts from equivalent fresh user states. The candidate transfers its already-owned internal state dictionary to the executor, while the reference uses the public defensive-copy route. Both sides use the same executor, compiled plan, and action, and their public state projection, trace, and events must be equal. The suite reports =copy-boundary: program-executor=, =copy-kind: shallow-root-dict=, =phase: steady-fresh-state=, and =semantics-verified: true= when these invariants hold.

#+begin_src mermaid
flowchart LR
    U["Equivalent fresh user states"] --> IA["Program internal state A"]
    U --> IB["Program internal state B"]
    IA --> C["Candidate: owned transfer"]
    IB --> R["Reference: defensive root copy"]
    C --> E["Same executor, plan, and action"]
    R --> E
    E --> P["Same public projection"]
    P --> S["State, trace, and event equality"]
    S --> Q["Qualification receipt"]
#+end_src

** Memory model
Let =F= be the payload-field count, =B= the logical bytes per payload field, and =I= the items per timing pair. The template's payload byte contents estimate is =F * B= and excludes Python object, dictionary, key, and reference overhead. A shallow =dict= copy duplicates root references; it does not copy the =F * B= retained payload byte contents.

Each pair creates =2 * I= input root dictionaries. Each root contains =F + 2= references, so the input-root reference count is =2 * I * (F + 2)=. In the canonical runs, =I= is 24, which yields 48 input roots and the receipt counts 3,168, 12,384, 49,248, and 196,704. Additional runtime, result, and event roots are a constant-factor =O(I * F)=. The resident-memory model is per timing pair, not =O(total observations * F)=, because completed pairs do not retain every observation's state.

** Interpretation and limits
- =passed: true= is a same-run no-regression result under =gate-purpose: same-run-no-regression= and the receipt's threshold ratio. It is not itself a speedup claim.
- =candidate-faster: true= and each positive =absolute-saved-ns-per-item= report that the candidate median was lower in these four runs.
- Tail latency is not uniformly better. At 4,096 fields, candidate p95 is 1,737,168.375 ns/item and reference p95 is 1,540,557.292 ns/item.
- The saved median is not cleanly linear at every point. From 1,024 to 4,096 fields it increases by about 5.39 times for a four-times field increase. The signal generally grows with field count, but the matrix is noisy and does not establish a universal scaling coefficient.
- The measured root copy is =O(F)= in field count, while the full timed path includes shared graph, action, projection, trace, and event costs.
- =payload-field-bytes= describes logical retained payload shape only. Python's shallow dictionary copy copies references rather than the referenced byte contents.
- This suite does not measure the pre-action optimization surface. It qualifies only the named Program-to-executor ownership-transfer boundary under the steady fresh-state workload.
- Results apply to these commands, runtime, host, and process conditions. They do not establish cross-runtime, cross-host, C ABI, serialization, or end-to-end application performance.

* Open issue: burst auto-capacity calibration observability
This issue is independent of the large-state owned-transfer qualification. It is not an owned-state optimization result and must not be used as evidence for that optimization.

Two auto-capacity runs exposed different selections with the same visible calibration inputs: 12 available CPUs, a calibration population of 1,000, candidates 12, 24, 48, 96, 192, 384, 768, and 1,000, and topology =serial_steps=3, =parallel_fanout=4, =parallel_steps=1, =interval=0. The full auto owned-state run selected capacity 768; the subsequent minimal =--population 1000= auto run selected 48.

The canonical receipt does not report per-candidate calibration throughput. The selection gap is therefore unobservable and cannot be attributed to system load, the owned-state change, or any other cause from these receipts.

Repeated fixed-capacity 100,000-item runs at capacity 48 also show material run-to-run jitter:

| Run | Makespan (ms) | Throughput (/s) | Startup p99 (ms) | Wall (s) | User (s) | Sys (s) | Change from historical baseline |
|-----+---------------+-----------------+------------------+----------+----------+---------+---------------------------------|
| First current run | 34,871.279 | 2,867.690 | 34,494.595 | 39.66 | 9.79 | 0.34 | makespan +18.211%; throughput -15.405%; startup p99 +18.002% |

These receipts qualify scoped Python runtime behavior only.  They do not claim
a globally green repository; Scheme and module-system blockers remain separate
acceptance work.
