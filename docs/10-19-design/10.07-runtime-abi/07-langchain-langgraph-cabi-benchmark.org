* Scope

The benchmark compares native LangChain/LangGraph/CrewAI orchestration with
semantically aligned POO Flow Python executor surfaces. It does not benchmark
real LLM latency, network tools, database IO, or another language's production
runtime. The deterministic actions model control-plane costs for realistic AI
Agent Flow shapes.

The suite covers:

- LangChain prompt/parser/runnable sequence.
- LangChain branch routing for RAG/tool/handoff selection.
- LangGraph StateGraph conditional routing.
- LangGraph dynamic =Send= fanout and reducer aggregation.
- LangGraph interrupt/resume through an in-memory checkpointer.
- LangGraph 32-node long graph scale cost.
- LangGraph stream update consumption.
- POO Flow executor-only graph routing, handoff/reducer, and interrupt/resume.
- POO Flow executor-only 32-node long graph and event stream consumption.
- POO Flow executor-only CrewAI-style sequential composition.
- No legacy graph C ABI case or gate.

* Runtime Language Implication

The Python runtime is the concrete runtime-language target for the current control-plane work. The benchmark is not only a LangGraph comparison; it validates that POO Flow plans can be lowered into Python runtime graph execution at low cost.

This matters for more than LangGraph:

- CrewAI-style crews and flows can lower from profile composition into Python graph/process execution.
- Pi Agent core-style behavior can lower from session/tool/agent-loop policy facts into Python runtime bindings.
- Scheme stays the control planner and proof surface; Python owns the operational runtime behavior.

The desired closure path is:

#+begin_src mermaid
flowchart LR
    S["Scheme user composition"]
    P["Python RuntimeGraphProgram"]
    E["Runtime execution receipt"]
    L["Generated Lean facts"]
    C["runtime-v0 C ABI interface"]
    S --> P --> E --> L
    S --> C
    C -. "downstream runtime adapters" .-> E
#+end_src

If a scenario requires new operational behavior, the first question is whether Python or another runtime language needs a new executor capability. It should not become a Scheme-side runtime clone unless the behavior is actually a control-plane proof or composition primitive.
