#+title: Python Runtime Acceleration Dependencies

* Boundary

The Python runtime package uses acceleration dependencies only below the POO
Flow strategy boundary.  POO-native policy families and functional strategy
combinators remain the source of strategy semantics.

* AnyIO

AnyIO is used as the Python runtime concurrency adapter.  It gives downstream
runtime authors async entry points without changing the graph semantics:

- =RuntimeGraphExecutor.ainvoke=
- =RuntimeGraphExecutor.abatch=
- =RuntimeGraphExecutor.astream=
- =RuntimeGraphExecutor.astream_events=
- =RuntimeGraphProgram.ainvoke=
- =RuntimeGraphProgram.abatch=
- =RuntimeGraphProgram.astream=
- =RuntimeGraphProgram.ainvoke_thread=
- =RuntimeGraphProgram.aresume_thread=
- =RuntimeGraphProgram.aresume_interrupted=

The synchronous graph executor remains the semantic owner.  AnyIO runs those
entry points through the runtime adapter layer and preserves ordered batch
results.

Async graph entry points await async node actions and async conditional routers.
This lets Python agent runtimes attach native async tools or model calls without
changing POO Flow graph semantics.

Threaded runtime execution is also async-aware.  =ainvoke_thread= stores
interrupt checkpoints through an async checkpointer when one is available, and
=aresume_thread= reloads and clears those checkpoints through the same durable
async facade.

* CFFI

CFFI is used as an alternate C ABI backend beside the existing ctypes backend.
The C header under =bindings/runtime-c/include= remains the ABI source of truth.
The Python CFFI loader reads that header and does not duplicate the ABI as an
inline Python string.

The public CFFI entry point is =PooFlowRuntimeCffiBinding=.  It exposes the same
context and graph-plan shape as =PooFlowRuntimeBinding= so runtime authors can
choose the backend that best fits their integration constraints.

* Turso

Turso is used as the durable runtime substrate.  The Python package exposes:

- =TursoRuntimeGraphStore=
- =TursoRuntimeGraphCheckpointer=

The store and checkpointer mirror the existing memory/file contracts.  Runtime
state and checkpoints remain POO Flow objects; Turso only provides durable
storage and high-concurrency database behavior.

The Turso store and checkpointer also expose async facades such as =aput=,
=aget=, =asave=, =aload=, and =aupdate_state=.  These facades use the runtime
AnyIO adapter and keep the synchronous durable contract as the semantic owner.

Durable policy is still owned by Scheme.  Python accepts
=RuntimeDurablePolicyManifest= as runtime input and applies only the parts that
belong to the substrate adapter, such as checkpoint id ownership and history
retention.  Python must not become the durable policy engine.

* Non-goals

The Python runtime package does not introduce LangGraph or Pydantic as strategy
or validation dependencies.  Graph semantics, validation policy, and durable
policy stay in the POO Flow layer.
