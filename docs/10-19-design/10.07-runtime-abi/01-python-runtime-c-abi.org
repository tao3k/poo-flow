* Purpose

POO Flow Scheme is the policy and control-plane layer. Runtime languages
should not need to reimplement policy families, strategy combinators, or POO
object semantics. They should receive a narrow, stable handoff surface and keep
their own scheduling, thread, memory, and device-specific work inside the
runtime.

The first runtime-language proof target is Python. Python is intentionally not
the performance target; it is the integration-cost target. If Python can call a
small stable C ABI with predictable ownership and status behavior, Rust, Marlin,
and other runtime languages can use the same ABI shape with lower overhead and
stronger native type discipline.

* Boundary

The ABI boundary is not a POO object boundary.

Do not expose Gerbil POO objects, Gambit objects, closures, green-thread state,
or GC-managed Scheme values to Python. The runtime ABI should use only:

- opaque context handles;
- byte pointer plus length payloads;
- integer status codes;
- returned byte buffers with explicit free;
- optional last-error or diagnostic receipt calls.

This lets Scheme remain the POO-native policy authoring surface while runtime
languages see a compact native contract.

* Gerbil And SSI Direction

Gerbil FFI is a compile-time surface. A direct interpreter probe with raw
Gambit C forms is not the right model. The working direction is:

- define C-facing functions through Gerbil's foreign interface layer;
- compile modules so Gerbil emits static interface metadata, including SSI;
- let Scheme imports see stable Scheme bindings;
- keep C ABI details behind a small compiled module boundary.

The =gerbil-mysql= style package is the useful model: C details live in a
binding module, while the public Scheme API remains idiomatic. For POO Flow,
the public Scheme API must stay POO-native and functional; the C ABI is a
runtime handoff artifact, not the user interface.

This is a control-plane ABI. Payload bytes may later become canonical GSON,
canonical JSON, or another stable binary receipt format. The ABI should not
depend on that choice.

* Python Runtime Role

The Python runtime adapter should call the C ABI with =ctypes= or =cffi=.
For the first smoke layer, =ctypes= is enough because it is in the standard
library. =cffi= is preferred once the header is stable because it reads C
declarations directly and reduces Python-side structure drift.

Python responsibilities:

- load the shared library;
- pass canonical request bytes;
- free returned buffers exactly once;
- translate status codes into runtime-level exceptions or receipts;
- never reach into Scheme object memory.

The repository should maintain Python as a formal =uv= package under
=packages/python-runtime=. ALALEX and other validation-runtime work should plug
into that package and consume the same binding wrapper instead of creating
independent =ctypes= entry points.

This package must not depend on downstream orchestration frameworks such as
LangGraph or LangChain to express POO Flow semantics. Flow, graph, policy
families, and strategy composition remain owned by the Scheme layer. Python is a
runtime-language consumer used to validate the C ABI handoff and exercise
runtime integration behavior.

The Python runtime may still provide a dependency-free compiled-plan executor.
That executor mirrors the runtime capabilities needed to cover LangGraph-like
downstream scenarios: START/END, node actions, ordinary edges, conditional
routes, reducers, and bounded execution. It is a runtime target for Scheme
handoff plans, not the policy-authoring interface. Cross-language transfer of
plans, receipts, and runtime handoff data remains owned by the C ABI, including
opaque runtime graph plan handles. The ABI should reject invalid graph plans at
the boundary, including duplicate nodes, empty plans, and dangling ordinary or
conditional route endpoints. A compiled runtime graph must also have START and
END boundaries, every node must be reachable from START, and every node must be
able to reach END. Nodes should be bound to runtime action symbols so runtime
languages can resolve actions through their local registries without inventing a
second graph protocol. State keys can also be bound to reducer symbols through
the same C ABI so runtime languages can preserve LangGraph-like merge behavior
without adopting LangGraph itself.

Graph validation and graph handoff receipts include a stable plan digest. This
gives runtimes a compact identity for cache keys and receipt comparison while
keeping the graph representation inside the ABI-owned handle.

The package-level verification entry points are:

* Landing Plan

1. Land a self-contained C ABI probe and Python runtime smoke test.
2. Replace the C probe implementation with a Gerbil/Gambit compiled module once
   the project-owned ABI module location is selected.
3. Add Scheme tests that assert the POO-native policy layer produces the same
   manifest and handoff receipts consumed by Python.
4. Add runtime-language conformance tests so Rust, Python, and Marlin exercise
   the same ABI contract.

* Repository Layout

Runtime bindings are first-class upstream artifacts. They belong under
=bindings=, not under =examples=.

- =bindings/runtime-c= owns the stable C ABI header and the current ABI probe
  implementation;
- =packages/python-runtime= is a =uv=-managed runtime-language consumer and
  validation package;
- later runtime packages should consume the same =bindings/runtime-c= contract
  instead of forking their own ABI surface.

* Current Probe

The initial C binding probe is under =bindings/runtime-c=. The Python consumer
test is under =packages/python-runtime=. The probe is deliberately small and
does not enter the main Scheme package build graph yet. It validates these
contract points first:

- context allocation and release;
- status code translation;
- byte-buffer return and explicit free;
- runtime handoff receipt shape;
- invalid argument behavior.

Once this probe is stable, the implementation body can move from plain C into
the compiled Gerbil binding layer without changing the runtime-facing ABI.
