#+TITLE: LangGraph Alignment Reflection

* Alignment Snapshot

The Python runtime is now functionally aligned with the LangGraph runtime
surface that matters for POO Flow downstream adapters:

- Graph construction: nodes, edges, entry points, finish points, sequences, and
  conditional routing.
- Control flow: =RuntimeGraphCommand=, direct target routing, and dynamic
  =RuntimeGraphSend= fan-out from nodes or routers.
- Streaming: =values=, =updates=, =messages=, =custom=, =checkpoints=, =tasks=,
  =events=, and =debug= modes.
- Persistence: checkpointers, stores, thread invocation, successful final
  checkpoints, interrupted checkpoints, history, =load_at=, and =update_state=.
- Human-in-the-loop: dynamic interrupts, persisted interrupted runs, and resume
  APIs.
- Subgraphs: reusable graph-as-node adapters and stream projections.
- Runtime packaging: ABI surface manifest, validation receipts, and executable
  self-test gates.

Within the intended POO Flow scope, this is roughly 90-95% aligned with
LangGraph's local runtime semantics. The remaining 5-10% is intentionally
outside this layer: LangGraph Cloud, hosted SDK semantics, visualization,
LangChain model integrations, and Python-specific schema ecosystems. Those do
not belong in the Scheme-owned policy/runtime boundary.

* Boundary Difference

LangGraph owns a Python-first application framework. POO Flow owns a
Scheme-first policy and strategy layer with runtime adapters. Therefore the
alignment target is not API mimicry for its own sake. The target is that a
runtime-language adapter can express the same execution patterns without
inventing policy logic:

- Scheme owns policy families, durable strategy contracts, and functional
  strategy combinators.
- Python owns runtime execution, async/concurrency integration, C ABI access,
  and package-level self-tests.
- Other runtime languages should only need to implement the same manifest,
  receipt, and validation-entrypoint shape.

* Completed Gap Closure

The final local-runtime gaps closed here are:

- =RuntimeGraphBuilder.add_sequence= for linear graph construction.
- =compile(..., checkpointer=..., store=..., thread_id=..., metadata=...)= and
  the same options for =compile_program=.
- Conditional routers may return direct node names, lists of node names,
  =RuntimeGraphSend=, or lists of =RuntimeGraphSend=.
- Successful =invoke_thread= calls now save a final checkpoint, so history and
  time-travel tooling work after normal completion.
- =RuntimeGraphProgram= exposes =get_state=, =get_state_history=, =get_state_at=,
  and =update_state= plus async equivalents, so downstream adapters do not need
  to call checkpointer internals for LangGraph-style state access.
- C ABI force rebuild compiles to a temporary library and atomically replaces
  the stable output, avoiding slow direct writes to a loaded dylib path.

* Remaining Policy

Do not add LangGraph as a dependency. If a downstream scenario appears to need
more LangGraph surface area, first classify it:

- Runtime execution shape: implement in Python runtime or another runtime
  adapter.
- Policy or durable strategy semantics: implement in Scheme POO-native policy
  families and functional combinators.
- Hosted ecosystem feature: document as out of scope for POO Flow runtime
  alignment unless it exposes a portable ABI/receipt requirement.
