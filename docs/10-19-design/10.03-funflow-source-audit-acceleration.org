#+TITLE: 10.03 Funflow Source Audit And Acceleration Plan

* 10.03 Funflow Source Audit And Acceleration Plan
** 10.03.00 Document Status
This document is the source-backed acceleration plan after auditing the cloned
Funflow tree under =.data/funflow=.

The working conclusion is unchanged from 10.01 and 10.02:
- Gerbil Scheme owns declaration, POO descriptors, planning policy, extension
  strategy, lightweight local execution, request normalization, and receipts.
- Rust or another host runtime owns content-addressed storage, Docker or
  container execution, durable cache state, process control, large filesystem
  operations, external config loading at scale, and parallel scheduling.

This document should be used as the implementation queue for the next slices.
It is intentionally more operational than 10.01 and 10.02.

** 10.03.10 Audited Source Map
The useful reference surface is concentrated in these files:

- =.data/funflow/funflow/src/Funflow/Flow.hs=: public flow API, required
  strands, extended strands, and smart constructors.
- =.data/funflow/funflow/src/Funflow/Run.hs=: configured runner, strand
  weaving, store/cache setup, Docker preflight, config validation, and
  =performP= execution.
- =.data/funflow/funflow/src/Funflow/Config.hs=: file/env config model,
  configurable values, required key collection, and missing-config checks.
- =.data/funflow/funflow/src/Funflow/Tasks/*.hs=: simple, store, and Docker
  task family data shapes.
- =.data/funflow/cas/store/src/Data/CAS/ContentStore.hs=: durable store state,
  pending/complete lifecycle, aliasing, lookup/wait, and cache computation.
- =.data/funflow/cas/hashable/src/Data/CAS/ContentHashable.hs=: content hash
  semantics for files, directories, store items, and external references.
- =.data/funflow/docker-client/src/Docker/API/Client*.hs=: Docker manager,
  container spec, pull/run/archive/log operations, and typed Docker errors.

** 10.03.20 Reference Feature Inventory
*** 10.03.21 Flow And Extension Shape
Funflow's public flow layer is a typed declaration surface.

Source facts:
- =RequiredStrands= includes =simple=, =store=, and =docker= in
  =Funflow/Flow.hs:45=.
- =Flow input output= is an =ExtendedFlow '[] input output= in
  =Funflow/Flow.hs:73=.
- =ExtendedFlow= allows additional strands before interpretation in
  =Funflow/Flow.hs:78=.
- Smart constructors include =pureFlow=, =ioFlow=, =dockerFlow=,
  =putDirFlow=, =getDirFlow=, =throwStringFlow=, and =returnFlow= in
  =Funflow/Flow.hs:99= through =Funflow/Flow.hs:130=.

Poo-flow interpretation:
- Scheme should not copy the type-level rope encoding.
- Scheme should keep POO descriptors and registries as the extension mechanism.
- The immediate missing behavior is not another flow record. It is making
  configured execution carry custom task and flow descriptor registries.

*** 10.03.22 Runner And Configured Execution
Funflow's runner is the integration point where task families become a runnable
core.

Source facts:
- =RunFlowConfig= carries =storePath= and optional =configFile= in
  =Funflow/Run.hs:92=.
- =runFlowWithConfig= is the explicit configured entrypoint in
  =Funflow/Run.hs:105=.
- The runner weaves =docker=, =store=, and =simple= strands in
  =Funflow/Run.hs:123= through =Funflow/Run.hs:129=.
- It collects Docker images and external config requirements before execution
  in =Funflow/Run.hs:136= through =Funflow/Run.hs:149=.
- It installs cache/store context with =localStoreWithId= in
  =Funflow/Run.hs:168=.
- It executes with =performP= in =Funflow/Run.hs:190= through
  =Funflow/Run.hs:192=.

Poo-flow interpretation:
- Scheme's runner must remain the policy interpreter, not the heavy runtime.
- A run config should snapshot strategy, adapter, task descriptor registry,
  flow descriptor registry, external config policy, and runtime options.
- Validation should happen before adapter submission so unsupported descriptors
  produce typed control-plane failures.

*** 10.03.23 Config And Missing-Input Diagnostics
Funflow treats external config as a first-class preflight surface for Docker
tasks.

Source facts:
- =ExternalConfig= has =fileConfig= and =envConfig= in =Funflow/Config.hs:35=.
- =Configurable= supports file, environment, and literal values in
  =Funflow/Config.hs:44=.
- =render= converts a configurable value against external config and reports
  errors in =Funflow/Config.hs:61=.
- =ConfigKeysBySource= separates file and environment keys in
  =Funflow/Config.hs:100=.
- =missing= reports absent config keys in =Funflow/Config.hs:151=.

Poo-flow interpretation:
- Scheme should model config requirements and missing keys as inspectable
  policy data.
- Loading and secret-handling should be owned by the runtime adapter when it is
  environment-specific or process-specific.
- Receipts should record required config keys and missing keys without leaking
  secret values.

  =Funflow/Tasks/Store.hs:15= through =Funflow/Tasks/Store.hs:17=.
- =DockerTaskConfig= carries =image=, =command=, and =args= in
  =Funflow/Tasks/Docker.hs:29=.

  =Funflow/Tasks/Docker.hs:39=.

  =Funflow/Tasks/Docker.hs:63=.

Poo-flow interpretation:

- Docker should not remain a generic external task forever. It needs a

*** 10.03.25 Store, Cache, And Content Hashing
Funflow's CAS is a real runtime subsystem, not a small helper.

Source facts:
- Store state is persisted in the filesystem, and pending/complete items have
  distinct paths in =Data/CAS/ContentStore.hs:24= through
  =Data/CAS/ContentStore.hs:48=.
- =lookupOrWait= returns completed items or waits on pending items in
  =Data/CAS/ContentStore.hs:476=.
- =constructIfMissing= atomically queries state and marks missing items pending
  in =Data/CAS/ContentStore.hs:502=.
- =markPending=, =markComplete=, and =removeFailed= define item lifecycle in
  =Data/CAS/ContentStore.hs:563=, =Data/CAS/ContentStore.hs:575=, and
  =Data/CAS/ContentStore.hs:609=.
- =cacheComputation= and =cacheKleisliIO= are the core cache execution helpers
  in =Data/CAS/ContentStore.hs:1164= and =Data/CAS/ContentStore.hs:1189=.
- =putInStore= hashes content and stores the computed item in
  =Data/CAS/ContentStore.hs:1223=.
- =DirectoryContent= hashing ignores the absolute directory path and hashes
  relative contents in =Data/CAS/ContentHashable.hs:512=.

Poo-flow interpretation:
- Rust should own durable store state, file locks, pending/complete lifecycle,
  hash materialization, archive movement, and remote cache.
- Scheme should own store task declarations, cache policy decisions, request
  envelopes, and receipts that can be replayed or audited.

*** 10.03.26 Docker Runtime

  =Docker/API/Client/Internal/Requests.hs:44=.

  =Docker/API/Client/Internal/Requests.hs:69=.
- =pullImage= is explicit preflight in
  =Docker/API/Client/Internal/Requests.hs:110=.

Poo-flow interpretation:
- Docker descriptor and request schema belong in Scheme.
- Docker manager, image pulling, container lifecycle, archive extraction,
  UID/GID handling, and Docker error conversion belong in Rust or a host
  runtime.

** 10.03.35 Gerbil POO Implementation Correction
The first descriptor implementation used POO objects but kept too much slot
construction policy inside =task.ss= and =flow.ss=. The corrected shape follows
the =gerbil-poo= source model more closely.

Evidence:
- =asp gerbil-scheme search extension gerbil-poo usage --view seeds= confirms
  =gerbil-poo://= is registered knowledge and not project activation.
- =asp gerbil-scheme search pattern gerbil-poo usage --view seeds= points to
  the object slot-resolution model and warns against direct slot hash guesses.
- =.data/gerbil-poo/object.ss= shows =.mix= delegates to =make-object= with
  =slots:= and =defaults:=, and slot functions are computed from slot specs.

Implementation decision:
- Descriptor and registry policy stay as POO objects.
- Raw descriptor values are converted to =$constant-slot-spec= before entering
  =.mix slots:=.
- That conversion lives in =role-constant-slots= under =src/core/roles.ss=, so
  task and flow modules do not each carry private POO slot machinery.
- Tests assert =.slot?= and =.all-slots= over descriptors and registries, not
  only accessor results. This keeps the POO behavior observable.

** 10.03.40 Acceleration Lanes
*** 10.03.41 Lane A: Config-Bound Descriptor Registries
Core Scheme path landed.

Landed:
- Run config carries task-family and flow-declaration registries.
- Configured runners are assembled from those registries.
- Strategy planning consumes the configured flow declaration registry.
- Runner validation, adapter operation selection, and receipt metadata consume
  the configured task-family registry.
- Tests install custom registries through =run-flow-with-config= and observe
  planning, validation, dispatch, and receipt behavior.

Exit gate:
- A custom task descriptor or flow descriptor can be installed through a run
  config and observed in plan, validation, dispatch, and receipt output.

*** 10.03.42 Lane B: Typed Control-Plane Failures
First Scheme slice landed.

Landed:
- Added =execution-failure= with owner, code, message, detail, and
  recoverability.
- Replaced registry, strategy, task, and runner generic error boundaries with
  typed failure construction and direct structured raises.
- Converted adapter-result error fields into receipt failures before audit
  persistence.
- Added tests for unknown task family, unsupported planner, and adapter
  failure receipt wrapping.

Remaining:
- Normalize runtime response failures once the Rust bridge returns real error
  payloads.

Exit gate:
- Failing plans and failing adapter submissions return or raise structured
  failures that are inspectable by tests.

*** 10.03.43 Lane C: External Config Policy
First Scheme preflight slice landed.

Landed:
- Added config requirement descriptors with source, key, and secret metadata.
- Added config preflight results with status, requirements, and missing keys.
- Added run-config accessors for requirements and caller-supplied source keys.
- =run-flow-with-config= validates preflight before runner execution.
- Missing keys raise typed =execution-failure= under owner =config= without
  exposing secret values.
- Added configurable argument descriptors for literal, env, file, and
  placeholder arguments.
- Added rendering helpers that preserve placeholders as runtime references and
  redact secret env/file values from Scheme request data.

Remaining:
- Let Rust/runtime own real file/env loading and secret materialization.

Exit gate:
- A Docker-style task can declare required environment and file keys, and
  missing keys are reported before runtime submission.

*** 10.03.44 Lane D: Rust Runtime Bridge
Implement:
- Stabilize a request/response schema for submit, fetch, store-put, store-get,
  docker-run, and cache-query.
- Add process invocation or IPC behind the Rust adapter.
- Preserve request-only adapter behavior as a deterministic test double.
- Record runtime response ids and artifact handles in receipts.

Implementation receipt:
- Landed the first Scheme-side bridge schema in =src/core/runtime-adapter.ss=
  with =poo-flow.runtime-request.v1= and =poo-flow.runtime-response.v1=
  constants.
- Rust submit and store-put now emit alist-shaped envelopes with schema,
  runtime, operation, request id, artifact handle, policy, plan id, node id,
  and frontier data.
- Adapter results now project into a durable runtime response shape so receipts
  and future Rust IPC can share the same response contract.
- =make-rust-run-config= now accepts a =runtime-command= option that receives
  one request envelope and returns a normalizable runtime response.
- Runtime command responses normalize from runtime-response alists,
  =runtime-response= structs, or internal adapter results; invalid command
  output becomes a failed adapter result.
- Added =t/runtime-bridge-test.ss= coverage for request schema fields,
  store-put operation tags, adapter-result response projection, configured
  command responses, and invalid command responses.

Remaining:
- Add concrete process invocation or IPC behind the configured runtime command.
- Wire fetch and store-get through the same request/response schema instead of
  placeholder adapter results.
- Add docker-run and cache-query request operations after the Docker and CAS
  descriptors land.
- Record runtime response ids and artifact handles in receipts once the runtime
  can return real ids.

Exit gate:
- A non-local task can cross the Rust boundary and return a real response or a
  typed adapter failure.

*** 10.03.45 Lane E: CAS Store And Cache Runtime
Implement in Rust/runtime:
- Content hash model and item handles.
- Put/get directory operations.
- Pending/complete/failed lifecycle.
- Cache key lookup and wait semantics.
- Receipt-compatible cache evidence.

Scheme scope:
- Store task descriptors.
- Cache policy snapshot.
- Request envelope and receipt projection.

Exit gate:
- Store put/get flows produce stable artifact handles and replayable cache

*** 10.03.46 Lane F: Docker Task Runtime
Implement:
- First-class Docker task descriptor in Scheme.
- Docker request schema for image, command, args, config requirements, volume
  bindings, and output policy.
- Rust/runtime container execution, image pull, logs, archive extraction, and
  CAS output registration.

Exit gate:
- A Docker-style workflow can be planned in Scheme and executed by the host
  runtime without Docker logic inside Scheme.

*** 10.03.47 Lane G: Parallel Scheduler
Implement:
- Strategy-level scheduling policy for ready frontier selection.
- Runtime-level concurrent execution of independent ready nodes.
- Receipt events that preserve node ids, dependencies, frontier, start/end
  timestamps, and runtime response ids.

Exit gate:
- Branch nodes can execute concurrently in the runtime while Scheme still owns
  the plan and policy evidence.

** 10.03.60 Implementation Priorities
The fastest route is:

1. Finish M9 first. The current code already has descriptor registries, but
   configured execution does not own them. Without M9, extension remains a
   helper-level feature instead of a workflow-level feature.
2. Finish M10 next. Typed failures are needed before the Rust bridge, config
   preflight, Docker, and CAS work can produce useful diagnostics.
3. Finish M11 before Docker. Docker argument rendering and missing config
   checks should be control-plane behavior, while actual values and execution
   remain runtime-owned.
4. Finish M12 before implementing heavy store or Docker behavior. Multiple
   Scheme stubs would create churn; one bridge protocol should carry store,
   docker, and cache requests.
5. Then split M13 and M14 by owner. Store/cache and Docker are mostly runtime
   work; Scheme should only add descriptors, request schemas, and receipts.
6. Defer M15 until the request/response protocol exists. Parallel scheduling
   needs runtime response ids and artifact handles to be useful.

** 10.03.70 Verification Gates
Every milestone should satisfy these checks before commit:

- Provider-aware source discovery remains valid through ASP or =rg=.
- Gerbil tests run through the project harness, not only through ad hoc Scheme
  commands.
- Descriptor registries are exercised through configured execution where
  relevant.
- Receipts expose the policy owner and runtime owner for new behavior.
- Heavy behavior is either runtime-backed or explicitly request-only; Scheme
  must not quietly implement a partial Docker, CAS, or scheduler clone.
- Documentation updates cite the source audit area and the owner boundary.

** 10.03.80 Next Action
M9 through the M12 Scheme command-boundary slice are implemented. Continue under
the tutorial-result ladder in =10.04-funflow-tutorial-result-ladder.org=:

- Keep Stage 1 through Stage 6 green as the lightweight result gate.
- Add Docker task descriptors using the ExternalConfig and CCompilation
  tutorial payload shapes.
- Add process invocation or IPC behind the configured runtime command boundary
  only when it can produce a visible tutorial-shaped result.
- Preserve typed control-plane failures and config preflight before runtime
  submission.

This resets implementation priority from abstract feature parity to visible
tutorial-result parity without moving Docker, CAS, or scheduling logic into
Scheme.
