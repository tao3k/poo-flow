* 10.05 Agent Sandbox Extension Design
** 10.05.00 Document Status
This design defines the Scheme-side =agent-sandbox= extension for POO Flow.
It does not implement the real agent runtime, LLM integration, API/R support,
or C bindings. Those are planned integration surfaces for
=marlin-agent-core=.

** 10.05.10 Context
POO Flow currently has a Docker extension that models Docker as a task family
and an adapter capability. The Scheme side declares task request data,
strategy capabilities, and run configuration. It does not directly own image
pulls, mounts, process control, CAS, or runtime execution.

That split is the right pattern for agent sandbox support. Modern agent
sandbox backends such as =nono= and =CubeSandbox= are useful reference systems,
but their real execution authority should stay outside this Scheme control
plane. POO Flow should express a stable, backend-neutral request contract that
Marlin can later interpret through its runtime, API/R surfaces, and C binding
support.

** 10.05.20 Goals
- Add a first-class =agent-sandbox= task family.
- Keep Scheme responsible for declaration, validation, request shape, and
  receipt-friendly metadata.
- Preserve the existing runtime-adapter handoff model.
- Make the request shape rich enough for future =nono=-style local sandboxing
  and =CubeSandbox=-style remote or clustered sandboxing.
- Avoid making Docker, nono, CubeSandbox, LLM calls, or Marlin runtime code
  direct dependencies of POO Flow.

** 10.05.30 Non-Goals
- Do not call a real sandbox backend from Scheme.
- Do not add live LLM execution.
- Do not embed Marlin runtime code in this repository.
- Do not replace the Docker extension in the first slice.
- Do not vendor =nono= or =CubeSandbox= code into =src=.

** 10.05.40 Design
*** 10.05.41 Extension Module
Create =src/extensions/agent-sandbox.ss= beside the existing Docker and store
extensions.

The module should export:
- =agent-sandbox-task-family-descriptor=
- =make-agent-sandbox-task-family-registry=
- =make-agent-sandbox-enabled-strategy=
- =make-agent-sandbox-enabled-adapter=
- =make-agent-sandbox-run-config=
- =make-agent-sandbox-task=
- =task-agent-sandbox-config=
- field accessors for the request contract
- =agent-sandbox-flow=

The shape should mirror =src/extensions/docker.ss= so agents can understand the
new extension by comparing it with the existing Docker extension.

*** 10.05.42 Task Family Descriptor
The descriptor should identify:
- family: =agent-sandbox=
- task kind: =agent-sandbox=
- execution owner: =adapter=
- runtime owner: =marlin-or-external-runtime=
- default operation: =submit=

This is deliberately stronger than naming a concrete backend. It says the
Scheme task must be lowered into an adapter request, while leaving backend
selection to the runtime side.

*** 10.05.43 Adapter Capability
The extension should add an =agent-sandbox= capability to strategies and
runtime adapters. Missing capability validation should fail before execution,
matching the existing Docker extension pattern.

*** 10.05.44 Request Contract
An =agent-sandbox= task request should be alist-shaped and stable across
language boundaries. The first slice should include these fields:
- =backend-kind=: symbolic backend family, for example =local=, =remote=,
  =nono=, =cube=, =e2b-compatible=, or =docker-compatible=.
- =backend-ref=: optional endpoint, profile name, template id, or runtime
  selector.
- =command=: executable or agent entrypoint.
- =args=: argument list.
- =env=: environment alist.
- =workdir=: optional working directory.
- =mounts=: mount or path capability declarations.
- =network-policy=: symbolic network mode and optional allow/proxy metadata.
- =capabilities=: backend-neutral capability declarations.
- =resource-policy=: CPU, memory, timeout, snapshot, or concurrency hints.
- =output-policy=: artifact capture policy.
- =metadata=: receipt-facing annotations for agents and debuggers.

The contract should stay data-only. It should not depend on a Marlin Rust type,
C ABI layout, R API surface, or backend SDK. Marlin can translate the alist
into its typed request later.

*** 10.05.45 Run Config
=make-agent-sandbox-run-config= should install:
- =agent-sandbox= strategy capability
- =agent-sandbox= adapter capability
- the agent-sandbox task family registry
- options containing at least =((extension . agent-sandbox))=

It may wrap =make-rust-adapter= with an optional =runtime-command= exactly like
the Docker extension. The command remains a test or bridge hook, not a real
backend implementation.

** 10.05.50 Backend Reference Mapping
The reference repositories guide the contract, not the implementation.

=nono= maps to:
- path and filesystem capabilities
- network blocked, allow-all, or proxy-only modes
- local process sandbox application

=CubeSandbox= maps to:
- remote sandbox creation and resume
- template or image references
- E2B-compatible API selection
- stronger isolation and snapshot hints

Both should be representable by =backend-kind=, =backend-ref=,
=capabilities=, =network-policy=, and =resource-policy= without adding direct
backend code.

** 10.05.60 Testing Strategy
- Descriptor tests should prove the =agent-sandbox= family is registered.
- Extension tests should prove strategy and adapter capabilities are added.
- Task tests should prove request fields round trip through accessors.
- Runner or validation tests should prove missing =agent-sandbox= adapter
  capability fails before execution if the core validation path supports that.
- The first slice should use request-only or command-backed adapter receipts,
  not a live sandbox backend.

** 10.05.70 Implementation Slices
*** 10.05.71 Slice 1: Scheme Extension Contract
Add =src/extensions/agent-sandbox.ss= and tests for descriptor, run config,
task request shape, and accessors.

*** 10.05.72 Slice 2: Docker Compatibility Surface
Optionally add a compatibility helper that can express Docker-like tasks as
=agent-sandbox= backend requests, without removing =src/extensions/docker.ss=.

*** 10.05.73 Slice 3: Marlin Runtime Bridge
In a later Marlin-owned slice, map the alist request into typed Marlin runtime
requests and backend-specific adapters. That work belongs in
=marlin-agent-core=, not in this repository.

** 10.05.80 Acceptance Criteria
- A Scheme user can declare an =agent-sandbox-flow= without importing Docker.
- The generated task has kind =agent-sandbox= and a stable alist request.
- Capability validation can distinguish =agent-sandbox= from generic
  =external= execution.
- No live backend, LLM, API/R, or C binding dependency is required to run the
  POO Flow tests for this slice.
- The request contract is explicit enough for Marlin to implement backend
  selection later without changing the Scheme task API.
** 10.05.90 Native Binding Build Audit
The =gerbil-mysql= reference package shows the correct Gerbil pattern for a
real native FFI module:
- =build.ss= imports =:std/build-script= and =:std/make=.
- The build spec uses =gsc:= for a raw Gambit FFI file such as =db/_mysql.scm=.
- The same target passes =-cc-options= and =-ld-options= through =cppflags= and
  =ldflags= so package build owns native compiler and linker flags.
- An =ssi:= target exposes the raw FFI module to Gerbil code.

POO Flow should not copy that pattern for the current =nono= surface. The
=nono= adapter in this repository is not a Scheme-owned FFI module; it is a
host compile probe and a runtime handoff contract. Therefore:
- =build.ss= remains responsible for Gerbil package lifecycle through
  =clan/building=.
- =src/modules/nono-sandbox/c-binding-descriptor.ss= owns the ABI inventory.
- =src/modules/nono-sandbox/c-binding-build.ss= owns the POO build/probe object
  and direct argv command for syntax-only host checks.
- =src/modules/nono-sandbox/c-binding-runtime.ss= consumes that build object
  when it needs a smoke-test command, but it still never links, dlopens, or
  applies the native sandbox.

If POO Flow later adds a real Gerbil FFI module, that slice should add a
dedicated =gsc:= + =ssi:= pair and keep C flags in the package build spec, as
=gerbil-mysql= does. Until then, modeling the nono probe as =gsc:= would be a
false owner boundary.
** 10.05.91 Nono C Probe Input Contract
The =nono= C probe remains a syntax-only host check, but its input contract is
now explicit:
- package-owned adapter includes live under =bindings/nono-c=;
- upstream generated includes default to =.data/nono/bindings/c/include= for
  local development;
- both include groups are POO build object slots and can be overridden by a
  user or installer;
- required build inputs are serialized into the build contract so receipts can
  show missing paths before invoking =clang=;
- probe command generation validates required inputs first and raises a typed
  Scheme failure instead of relying on compiler diagnostics.

This keeps the current slice below the full =gsc:= + =ssi:= FFI line while
removing the hidden dependency on a particular =.data= checkout layout.
** 10.05.92 Always-Further Nono Diagnostic ABI
The =always-further/nono= C binding source currently exposes diagnostic helpers
from =bindings/c/src/diagnostic.rs=:
- =nono_last_diagnostic_code=
- =nono_last_remediation_json=
- =nono_session_diagnostic_report_to_json=
- =nono_merge_diagnostic_report_json=

POO Flow now includes these symbols in the nono C ABI descriptor, adapter
header, compile probe, and runtime =diagnostic-plan=. This lets Marlin consume
denial diagnostics and remediation JSON through the same handoff manifest as
capability, query, apply, and state functions.

The local generated =.data/nono/bindings/c/include/nono.h= may lag the Rust
source until =cbindgen= is rerun. The POO Flow adapter header declares the
diagnostic prototypes compatibly so the syntax-only probe stays tied to the
source ABI while still including =nono.h= for shared types and constants.
** 10.05.93 PATH-Gated Nono Live Test
The nono live test is optional and environment-discovered:
- the default live probe command is =nono --version=;
- if the executable can be spawned from =PATH=, the receipt is enabled and the
  command exit status determines success;
- if the executable cannot be spawned, the receipt is skipped with
  =nono-not-found-on-path=;
- live probes never call =nono_sandbox_apply= and keep =would-apply?= false.

This lets developer machines or CI images with =nono= installed exercise the
real CLI presence path automatically while keeping ordinary package tests green
on machines that only have the C headers and compiler probe inputs.
